# Erratic Test Failure

## 要約

コードを変えていないのにテストが通ったり落ちたりする場合、多くはテスト間で共有されるデータが正しく初期化されていないことが原因です。テストデータをできるだけ分離し、断続的な失敗が起きたら、共有状態や初期化漏れを疑って調べるべきだと述べています。

## 読むときの観点

- テスト失敗をIDEやツールの問題と決めつけず、自分のコードやテストの前提を疑う。
- テストの実行順や実行方法によって結果が変わるなら、共有状態が残っていないかを見る。
- 高速なテスト実行とのトレードオフを意識しつつ、できる限り新鮮なテストデータを作る。

## 原文の翻訳

先日、自分の本に載せるサンプルコードを触っていた。いくつか変更を加え、すべて動くようにし、テストを走らせてから個人用のリポジトリにコミットした。その後、別の領域に移って少し変更したところ、前の領域の予期しないテストが壊れた。自動テストを走らせる目的の一部は予期しない破壊を見つけることにあるが、この本のコードでは各領域は完全に独立している。妙なことだった。

問題をデバッグしようとする代わりに、私は `DiffDebugging` を使った。コミット後に大したことはしていなかったので、`svn revert` を実行した。テストを再実行すると、失敗した。しかし、コミット前にテストを走らせたことは確かだった。そこで IntelliJ ではなく ant からテストを走らせることにした。ant のテストはすべて通った。どちらも同じテストで、ディレクトリ内のすべての JUnit クラスを走らせている。では、なぜ ant では通り、IntelliJ では失敗するのだろうか。

この時点で、自分が次に考えたことを認めるのは恥ずかしい。「IntelliJ の何かがおかしいに違いない。何らかのキャッシュがあって、subversion の revert で混乱しているのかもしれない」。プログラミングを始めて間もないころ、年長のプログラマからデバッグの第一原則を教わった。バグは常に自分のコードにあり、コンパイラにあるのではない。だが愚かさに影響された私は IntelliJ を再起動し、するとテストはまたすべて通った。

問題解決、ではなかった。幸い、この奇妙な振る舞いが二度目に起きたとき、私は Sergey とペアを組んでいた。彼は私のような愚かな考え方をせずに問題に向き合い、バグを見つけた。

こうした問題の答えを見つける助けとして、屋外に出て、高さ6フィート半の文字で単語を作るとよい。塗装しなくて済むように杉で作る。ただし、さくらんぼで飾るのを忘れてはいけない。その単語はこうだ。

isolation.

コードを変えていないのに、テストがある実行では通り、別の実行では失敗する。あるいは、あるテストスイートで走らせると通るのに、別のスイートでは失敗する。そういう場合、九分九厘、原因はテスト間で共有される何らかのデータが正しく再初期化されていないことにある。その状態では、あるテストを走らせること自体が、他のテストが通るか失敗するかを左右してしまう。結果として断続的な失敗になる。これは確実に再現できないため、常に最悪の部類に入る。

私は JUnit を使っていた。JUnit は分離に強い。だからこそ `JunitNewInstance` の振る舞いを使っている。したがって、私の問題は何らかの static データから来ているはずだった。この場合、それは現在の日付を取得する呼び出しだった。私は `ClockWrapper` を使っていたが、一部のテストでそれを初期化し損ねていた。そのため、最後にそれを初期化したテストがどれだったかによって、いくつかのテストが失敗していた。

ここには二つの教訓がある。第一に、テストデータを分離するためにできることはすべて行うことだ。毎回新しいデータを作るようにする。ただし、ここにはテスト実行を速くすることとのトレードオフがある。**よいテスト分離を実践すればするほど、この種の問題に出くわす可能性は低くなる。**

第二に、この種の断続的なテスト失敗に遭遇したら、テスト間で共有されるデータを疑うことだ。断続的に失敗するテストが、テスト実行時に必要なデータを完全に初期化しているかを確認する。初期化されていないものがあれば、それがどこで作られたのか、そして変更されることがあるのかを必ず把握する。
