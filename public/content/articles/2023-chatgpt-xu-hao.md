# An example of LLM prompting for programming

## 要約

Martin Fowler が、Xu Hao との社内 Zoom で見た ChatGPT の使い方を紹介する記事です。Xu Hao は、ChatGPT にいきなりコードを書かせるのではなく、まず技術スタック、アーキテクチャ上の方針、テストの考え方、実装戦略をプロンプトで与えます。そのうえで、コードではなく実装計画を作らせ、以後の会話でその計画を参照しながらコード生成を進めます。

記事の中心は、LLM を「完成品を一度で出す機械」として扱うのではなく、設計方針を共有したうえで段階的に出力を調整する相手として扱うことです。そこで使われている技法として、Generated Knowledge prompting と Chain of Thought prompting が説明されています。

また、ChatGPT の出力はそのまま動くとは限らず、プロジェクトの技術スタックやチームの慣習に合わせて書き換えが必要になることも示されています。特に、コンテキストウィンドウの制限により、長い会話では初期の情報を忘れ始めるため、マスタープランを作って作業を分割することが重要になります。

## 読むときの観点

- LLM にコードを書かせる前に、設計方針、技術スタック、テスト方針を明確に伝えている。
- 最初のプロンプトでは「コードを書かないで」と指示し、まず実装計画を作らせている。
- Generated Knowledge prompting は、最終成果物の前に、問題を解くための有用な情報を LLM に生成させる使い方として説明されている。
- Chain of Thought prompting は、LLM に推論の筋道を示させるための技法として紹介されている。
- 生成されたコードは出発点であり、**人間が評価し、書き換え、チームの文脈へ合わせる**必要がある。
- コンテキスト制限に備えて、マスタープランと実装戦略を再利用できる形にしておくことが役に立つ。

## 原文の翻訳

Xu Hao は、自己テストするコードを書くときに、ChatGPT に対して Chain of Thought と Generated Knowledge のプロンプトを使っている。

この記事は、Xu Hao との社内チャットを Martin Fowler が記録したものです。Xu Hao は、ChatGPT を使って有用な自己テストコードを生成する方法を示しました。最初のプロンプトでは、実装戦略を与えて LLM を準備します。これは Chain of Thought prompting にあたります。また、そのプロンプトでは、いきなりコードではなく実装計画を求めています。これは Generated Knowledge prompting にあたります。計画が得られたら、それを使って実装を洗練し、有用なコード片を生成していきます。

Xu Hao は Thoughtworks 中国の Head of Technology です。彼は、ChatGPT を使って Self Testing Code の構築を助ける方法を説明しました。まず、アプリケーションの文脈と、コードをどのように構成したいかをプロンプトで設定します。

例では、オンラインホワイトボードシステムを題材にしています。技術スタックは TypeScript、React、Redux、KonvaJS、react-konva、Vitest、React Testing Library、Cypress component tests です。要件は MVVM アーキテクチャの React コンポーネントとして実装する、共有 view model は Redux store slice として実装する、ローカル view model は主に props や state で表す、といった設計方針が示されます。

プロンプトには、実装とテストで従うべきパターンも含まれています。テストでは `test` ではなく `describe` を使うこと、データ駆動テストを好むこと、view component のテストでは view model interface を通じて fake view model を使うことなどです。

ここでの要求は、ホワイトボード上に他ユーザーの awareness 情報、つまりカーソル、名前、オンライン情報を表示することです。受け入れ条件として、ローカルユーザーは表示しないこと、リモートユーザーのカーソル位置が変わったらアニメーションで変化を表示することが挙げられます。Xu Hao は ChatGPT に、awareness 情報を Konva layer に置き、cursor と name を描画する awareness info component を使う方針を伝えます。

重要なのは、この段階で Xu Hao が「コードを生成しないで」と明示していることです。最終的にはコードとテストを生成したいのですが、まずはアプローチを説明させます。これは Generated Knowledge と呼ばれるプロンプト技法です。LLM に直接最終成果物を求めるのではなく、最初に問題について有用な情報を生成させ、その情報を次の入力として使います。

プロンプトの中で最も興味深い部分は、implementation strategy です。ここでは、LLM に従わせる指示を与える Instruction prompting と、推論の道筋を促す Chain of Thought prompting が組み合わされています。Xu Hao は、望ましいアーキテクチャパターンを ChatGPT が従うべき思考の手順として記述し、そのうえで、その手順に基づくタスクリストを作らせました。

ChatGPT は、AwarenessInfo コンポーネントの作成、AwarenessLayer の作成、メインホワイトボードへの追加、Redux slice の作成、AwarenessViewModel interface の作成、カーソル位置変更時の更新、アニメーション、テストの作成といったタスクリストを返します。このタスクリストは、以後の会話で参照できる「マスタープラン」になります。

LLM の応答は非決定的なので、同じプロンプトを入れても毎回同じ結果になるとは限りません。それでも、番号付きのタスクリストが得られることには価値があります。Xu Hao は、その番号を使って、以後の会話で特定の作業項目を指し示せます。

マスタープランを見て問題があれば、ChatGPT に計画を書き直させることもできます。たとえば、タスクを model、view model、view に分けて整理するよう求めれば、アーキテクチャ上の意図と計画が合っているかを確認できます。実際、ある設計指示は最初から入っていたものではなく、後の反復で追加されました。

マスタープランが妥当に見えたら、Xu Hao は次に、コンポーネント名、メソッド名、props を含めた詳細な計画に書き直すよう求めます。これにより、AwarenessInfo、AwarenessLayer、awareness slice、AwarenessViewModel、updateUserAwarenessInfo などの具体名が計画に現れます。

ここまで来て、Xu Hao はようやく計画の一部についてコードを求めます。awareness slice と AwarenessViewModel のテストと実装例を、テストファーストで出すよう依頼します。ChatGPT はテストと実装を返しますが、そのコードはそのままでは動きません。`configureMockStore` が欠けており、技術スタックに含まれていない `redux-mock-store` を前提にしているためです。

Xu Hao は、この問題を一度に直そうとするのではなく、LLM に段階的な書き換えを求めます。たとえば `redux-mock-store` を使わない形に書き直すよう依頼し、さらにチームのテスト慣習に合わせるよう調整します。少しの接着コードとパス修正を加えれば、生成されたコードは実際に動く形へ近づきます。

このやり方で問題になるのが、ChatGPT のコンテキストウィンドウ、つまり token limit です。会話が長くなると、初期の情報を忘れ始めることがあります。短い会話では目立ちませんが、大量の文脈を扱う場合には重要です。

Xu Hao は、token limit には主に三つの影響があると見ています。第一に、ChatGPT が途中で生成を止めることがあります。これは「続けて」と促せば直せることが多いです。第二に、プロンプト全体が大きすぎると致命的なエラーになり、新しい会話を始める必要があります。第三に、より厄介な問題として、ChatGPT が文脈を忘れ始めます。

文脈をリセットする必要があるとき、マスタープランを作り、作業を分割しておくことが助けになります。新しい会話に元の戦略とマスタープランを渡し、計画の別項目についてコードを生成させられるからです。Xu Hao は、別々の会話で生成したコードをうまく合わせるには、文脈内の Chain of Thought が重要だと感じています。うまくまとまらないときには、単発のヒントを足すよりも、思考の流れそのものを見直すほうがよい結果になりました。

Martin Fowler の結論は、Chain of Thought と Generated Knowledge のプロンプト技法は、プログラミングにおいてかなり有用な道具になり得るというものです。LLM をうまく使うには、よい結果を得るためのプロンプトの組み立て方を学ぶ必要があります。この経験は、LLM をジュニアな相棒のように扱うとよいことを示しています。つまり、アーキテクチャ上の指針を与え、考え方を示させ、出力を見ながら調整していくのです。
