# Temporal Patterns

## 要約

時間に関わるモデルでは、「今の状態」だけでなく、「過去のある時点の状態」や「過去の時点で、さらに別の過去について何を知っていたか」を答えたいことがあります。
この記事は、そのための `Audit Log`、`Effectivity`、`Temporal Property`、`Snapshot`、`Temporal Object` といったパターンを概観します。

特に重要なのは、実際に物事が起きた時点である actual time と、それを記録として知った時点である record time を分けることです。
この二つの時間軸を理解すると、bi-temporal な情報を扱う難しさと、どこを単純化できるかが見えやすくなります。

## 読むときの観点

- 過去の状態を知りたいだけなのか、過去に何を知っていたかも知りたいのかを分ける。
- `Effectivity` の明示性と、`Temporal Property` や `Snapshot` による隠蔽の違いを見る。
- actual time と record time の二つの時間軸を区別する。
- 更新操作を additive update や current update に制限することで複雑さを下げる考え方を押さえる。

## 原文の翻訳

物事は変わります。
世界についての情報を保存しているだけなら、それは問題ではないかもしれません。
結局のところ、何かが変わったとき、コンピュータ化された記録システムの大きな価値のひとつは、修正液を使ったり何ページもの情報を打ち直したりせずに、記録を簡単に更新できることです。

しかし、変更の履歴を記録する必要が出てくると、話はおもしろくなります。
私たちは世界の状態を知りたいだけでなく、6か月前の世界の状態を知りたいのです。
さらに厄介なことに、2か月前の私たちが、6か月前の世界の状態をどう考えていたかを知りたいこともあります。

こうした問いは、temporal patterns という魅力的な領域へ私たちを導きます。
これらはすべて、ドメインモデルを完全にもつれさせることなく、こうした問いへの答えを簡単に見つけられるオブジェクトをどう整理するかに関わっています。
オブジェクトモデリングの課題の中でも、これは**最も一般的で、しかも最も複雑なもののひとつ**です。

この問題を解く最も単純な方法は、`Audit Log` を使うことです。
ここでは変更記録を残すことに関心がありますが、それを頻繁に戻って使うことは期待していません。
そのため、作成が簡単で、通常の作業にできるだけ割り込まないことを望みます。
誰かがそれを見る必要が出たときには、その情報を掘り出すためにかなりの作業が必要になると想定します。
それを頻繁に行う必要がなく、結果もすぐに必要でないなら、これで十分です。
実際、データベースを使っているなら、これはただで手に入ります。

情報にもっとアクセスしやすくする必要があるなら、もう少し取り組まなければなりません。
この状況で最もよく使われるパターンは `Effectivity` です。
これは単に、オブジェクトにそれが有効とみなされる期間を印づけします。
そしてオブジェクトを操作するとき、その期間を使って、適切な時点の適切なオブジェクトを得ます。

図1: 明示的な date range を使って effectivity を示す。

`Effectivity` の問題は、それが明白で明示的だということです。
あらゆる文脈でそのオブジェクトを使う人全員が、時間的な側面を意識しなければなりません。
これはモデルをかなり複雑にすることがあります。
そのため、時間に関する問題が広く行き渡っているなら、不要なときにはそれを隠し、必要なときには扱いやすくするほうが理にかなうことがよくあります。

簡単にアクセスできる時間情報を必要とするプロパティが、オブジェクトに少ししかないなら、そのオブジェクトで `Temporal Property` を使えます。
`Temporal Property` の要点は、明白な effectivity date を取り除くことです。
代わりに、通常のプロパティのように見えるものを使いますが、そのアクセサは日付を引数に取ります。
これにより、「2000年4月1日時点で、このプロパティの値は何か」と尋ねられます。

図2: address に temporal property を使う。

`Temporal Property` の本質は、日付でパラメータ化されたアクセサを持つという考え方です。
これにより、そのプロパティの利用者は、`Effectivity` を持つ多数のオブジェクトをたどらずに済みます。
したがって、`Temporal Property` と `Effectivity` は排他的なパターンではありません。
`Effectivity` を使った address usage オブジェクトで、`Temporal Property` のインターフェースを実装してもよいのです。

address usage オブジェクトがほかの責務も担っているなら、その責務を必要とするクライアントには address usage へアクセスするメソッドを提供し、より便利だと感じる人には `Temporal Property` のインターフェースを提供できます。

`Temporal Property` は、時間に基づくインデックスで物事を参照するという考え方を導入します。
ただし、それは単一のプロパティまでに留まります。
ひとつのオブジェクトに多くの temporal properties があると、扱いがぎこちなくなります。
これに対処する方法は二つあります。
第一の方法は `Snapshot` を使うことです。
`Snapshot` は、ある時点における実体オブジェクトの状態を参照するオブジェクトを与えます。
そのため、2000年4月1日時点の customer について多くの質問をしたいなら、その customer の2000年4月1日の `Snapshot` を求め、その後は日付を繰り返し渡さずにプロパティの値を尋ねます。

図3: snapshot は、ある日付時点のオブジェクトのビューを示す。

`Snapshot` は、ある日付におけるオブジェクトのビューを持たせます。
しかし、ときには、時間とともに変更が行われるにつれて、オブジェクトが明示的なバージョンを通っていくものとして考えたい場合があります。
この欲求は `Temporal Object` につながります。
`Temporal Object` には多くの形がありますが、最も単純な形は二つのクラスを持ちます。
contract クラスは、時間を通じた contract を表すため、通常ほかのオブジェクトから参照されるものです。
それは、各変更時点における contract の状態を記録する versions のコレクションを含みます。
これにより、人々は変わらない contract という概念を参照することも、特定の時点の特定バージョンを参照することもできます。

図4: Temporal Objects は明示的なバージョン履歴を持つため、それぞれの変更が新しい version につながる。

実際には `Effectivity` が広く使われているのを見ます。
しかしそれは、多くの場合、人々が `Temporal Property` やほかのより洗練されたパターンに十分慣れていないからです。
`Temporal Property` や `Temporal Object` のようなパターンは、時間的な振る舞いの仕組みの多くを隠すため、うまく働きます。
とはいえ `Effectivity` も重要です。
特定の期間だけ有効な明示的オブジェクトが欲しい場合には、それが正しい選択です。

## 時間の次元

ここまで説明したように、時間はモデリングにおいてかなり手ごわい概念です。
しかし私は、temporal models の最も厄介な側面を飛ばしてきました。
私たちは、悪いSF小説からであれ何であれ、時間は第四の次元だと学んできました。
困ったことに、これは間違っています。

この問題を説明する最善の方法は、例を使うことだと思います。
給与システムがあり、ある従業員のレートが1月1日から1日100ドルだと知っているとします。
2月25日に、このレートで給与処理を実行します。
3月15日に、その従業員のレートが2月15日付で1日211ドルに変わっていたことを知ります。
では、2月25日のレートは何だったのかと尋ねられたとき、何と答えるべきでしょうか。

ある意味では、答えは211ドルです。
今の私たちは、それがその時点のレートだったと知っているからです。
しかし、2月25日時点で私たちがレートを100ドルだと考えていたことを無視できない場合もよくあります。
結局、その時点で給与処理を実行したのです。
小切手を印刷し、彼に送り、彼はそれを現金化しました。
これらはすべて、そのときの彼のレートに基づいて起きました。
税務当局が2月25日のレートを尋ねてきたなら、このことは重要になります。

実際、Dinsdale の給与レートについて、私たちにとって重要な履歴は二つあると考えられます。
今知っている履歴と、2月25日に知っていた履歴です。
一般化すれば、Dinsdale の給与レートには過去の各日についての履歴があるだけでなく、Dinsdale の給与レートについての**履歴の履歴**があると言えます。
時間は第四の次元ではなく、第四と第五の次元なのです。

私は第一の次元を actual time、つまり何かが起きた時刻として考えます。
第二の次元は record time、つまり私たちがそれを知った時刻です。
何かが起きるときには、常にこの二つの時刻が付いてきます。
Dinsdale の昇給には、2月15日という actual date と、3月15日という record date がありました。
同様に、Dinsdale の給与レートが何だったかを尋ねるとき、本当は二つの日付、record date と actual date を与える必要があります。

| record date | actual date | Dinsdale の rate |
| --- | --- | --- |
| Jan 1 | Jan 1 | $100/day |
| Feb 25 | Feb 25 | $100/day |
| Mar 14 | Feb 25 | $100/day |
| Mar 15 | Jan 1 | $100/day |
| Mar 15 | Feb 25 | $211/day |

二つの次元は次のように考えられます。
actual history は actual time を振り返るものです。
現在の actual history を見るなら、Dinsdale の給与は2月15日まで100ドルで、その時点で211ドルに上がったことが分かります。
しかし、それは record time における今日の actual history です。
2月25日の actual history を見るなら、Dinsdale は1月1日以降100ドルで支払われており、211ドルはそこには登場しません。
record time の毎日、厳密にはすべての `Time Point` が、actual history を持ちます。

かつて真だと思っていたことが、もはや真ではないと分かるにつれて、これらの履歴は異なっていきます。

別の角度から言えば、actual history の各日は record history を持っています。
record history は、その日に関する私たちの知識が時間とともにどう変わったかを示します。
そのため actual time における2月25日は、3月15日までは Dinsdale の給与が100ドルであり、その時点で211ドルに達したという record history を持ちます。

この例をもう少し進めましょう。
3月26日の給与処理で、対応する調整を行ったとします。
4月4日に、以前の情報が間違っており、レートは実際には2月15日に255ドルへ変わっていたと知らされます。
では、「2月25日の従業員のレートは何だったのか」という問いに、どう答えるでしょうか。

私は、この種のものに直面して頭を抱える経験豊かな開発者を見てきました。
しかし、すべてがこの二つの次元の考え方に帰着するのだと分かれば、物事はかなり単純になります。
これを視覚化するひとつの方法は、先ほどの表を拡張することです。

| record date | actual date | employee の rate |
| --- | --- | --- |
| Jan 1 | Jan 1 | $100/day |
| Feb 25 | Feb 25 | $100/day |
| Mar 14 | Feb 25 | $100/day |
| Mar 15 | Jan 1 | $100/day |
| Mar 15 | Feb 25 | $211/day |
| Mar 26 | Feb 25 | $211/day |
| Apr 4 | Jan 1 | $100/day |
| Apr 4 | Feb 25 | $255/day |

現在の actual history、つまり record date が今日である actual history を見るなら、Dinsdale の給与は1月1日から100ドルで、2月15日に255ドルへ上がったと言うでしょう。
現在の actual history では、211ドルのレートはまったく発生しません。
それは真ではなかったからです。
3月26日の actual history を見るなら、Dinsdale の給与は2月15日まで100ドルで、そこで211ドルへ上がったことになります。
3月26日の actual history では、255ドルのレートはまだ知っていなかったため、発生しません。

2月25日の record history について考えることもできます。
この record history では、その日のレートは3月15日までは100ドルで、その時点で211ドルへ変わります。
そして4月4日に再び255ドルへ変わります。

二つの次元について分かると、問題について考えるのはずっと簡単になります。
しかし、この種のものを実装しなければならないと考えると怖くなります。
幸い、実装に関しては、問題を単純化するためにできることがいくつかあります。

第一の単純化は、こうした変更に対処するために `Audit Log` を使うことは難しくないという点です。
必要なのは、各エントリに record date と actual date の両方を記録することだけです。
この単純な作業だけで、ログは両方の次元に対して有効になります。
そして、たとえ片方だけを気にしている場合でも、これはやる価値があると私は考えています。

第二の単純化は、モデルで必ずしも両方の次元を扱いたくないことが多いという点です。
ここで重要なのは、モデルの中にどちらを持ち、どちらを `Audit Log` に任せるかを知ることです。

物事が時間とともにどう変わったかを知りたいが、その変化をいつ知ったかには関心がない履歴を保持したいなら、それは actual-temporal だと言えます。
たとえば従業員の住所の記録を保持するなら、actual-temporal property として保持することを選ぶかもしれません。
オンライン問い合わせを支援する情報システムでは、これはうまく働きます。
データベースにアクセスするとき、通常知りたいのは actual-history だからです。

record-temporal facts は、オブジェクトの状態に基づいて請求書を作るようなシステムで現れます。
こうしたものは、請求書がどのように計算されたかという問いにつながり、その請求書を計算した時点で、オブジェクトの状態をどう考えていたかを知る必要を生みます。
record-temporal facts は、ソフトウェアのバージョン管理システムにたとえられることがよくあります。
そこでは戻って、「4月1日時点でこのファイルはどんなものだったか」と言えます。

もちろん、両方の次元を同時に必要とする場合もあります。
これらは bi-temporal facts と呼ばれます。
本質的に、bi-temporal な情報は常に両方の日付を必要とします。

bi-temporality は完全な解ですが、それを避ける方法を考える価値は常にあります。
例は請求計算にあります。
請求額がなぜその値になったのかを知りたい場合、ひとつの可能性は完全に bi-temporal なデータベースを持つことです。
しかし、請求書を計算したときに計算の詳細なトレースを保存しておくほうが、しばしばよい方法です。
それは bi-temporal なオブジェクトモデルより、ずっと単純な方法で要求を満たします。

## Temporal Record の更新

ここまでは、temporal information を情報へのアクセスという観点だけで話してきました。
更新という観点では話していません。
更新をどう許すかによって、さらに多くの判断が生じます。
ただし、その多くには有用な単純化が伴います。

一般に、時間の記録を変更するのはかなり面倒です。
ある従業員のレートが2月15日から4月15日まで211ドルだったとします。
完全に一般的な更新では、開始日、終了日、値の任意の組み合わせを変更できるでしょう。
このためのインターフェースを提供するのは厄介です。
クライアントが temporal information の仕組みについて多くを知っている必要があるからです。

第一の単純化は、追加的な変更だけを持つことです。
追加的な変更は、常に記録の末尾に加わります。
追加的な変更の例は、「2月15日付で従業員のレートを211ドルにする」です。
一見すると、これは混ざっているデータからひとつの情報だけを取り除いているように見えます。
しかし結果としては、更新を大きく単純化します。
実際に起こる更新の大半は追加的なので、これによりクライアントをかなり単純にできます。

さらに、追加的更新の組み合わせでどんな変更でも行えます。
複雑な履歴を持つオブジェクトではひどく面倒になるでしょうが、単純な履歴を持つオブジェクトでは、この性質により追加的なインターフェースだけをサポートすればよくなります。

第二の単純化は、現在の更新だけを許すことです。
current update は、有効日が今日である場合にだけ記録の変更を許します。
一般には、追加的な変更であっても過去や未来の任意の日付に起こりえます。
current change では日付情報がまったく不要になるため、更新用のインターフェースを完全に非時間的にできます。

current update は、うますぎる話に見えます。
current update でしか更新しないなら、なぜ temporal information を持つのでしょうか。
ここでのよい知らせは、record-temporal information は current update でしか更新できないということです。
record-temporal information に対する遡及的変更は記録の完全性を壊すだけではありません。
actual-time の次元で扱えないような、record-temporal information への遡及的変更を必要とする状況はありません。
ただし、不正行為が要件のひとつである場合は別です。

これは大きな一歩です。
ひとつの次元全体について、更新が見えなくなるからです。
record time を気にする必要があるのは問い合わせのときだけであり、更新のときではありません。
