# Package Customization

## 要約

Package Customization は、パッケージ製品を購入したあと、自社のやり方に合わせるために大きくカスタマイズしてしまう危険を論じた記事です。
ユーティリティ的な業務であれば、パッケージに合わせて業務を調整するほうがよい場合が多いにもかかわらず、実際には大規模なカスタマイズが行われがちです。

この記事は、パッケージのカスタマイズ環境はしばしばソフトウェア開発基盤として貧弱であり、アップグレード性、採用、保守、統合のコストを過小評価しないよう警告しています。

## 読むときの観点

- 競争優位に関わる業務は自作し、そうでない業務はパッケージに合わせるという考え方を確認する。
- パッケージの大規模カスタマイズが、通常のカスタム開発と同じ道具やプロセスを必要とする点を見る。
- アップグレード性を、後回しではなく横断的な要求として扱う意味を読む。
- パッケージ固有のカスタマイズ言語や環境が、採用や保守のリスクになる点を考える。
- カスタマイズではなく API 連携や単一ベンダー選択で負担を減らせる場面を検討する。

## 原文の翻訳

IT 部門でよく出る問いに、ある機能をカスタムソフトウェアとして作るべきか、パッケージを買って提供すべきか、というものがあります。私がプログラミングを始めるより前から、その選択をどう行うべきかについて議論が続いてきました。私の基本的な立場は UtilityVsStrategicDichotomy に基づいています。簡単に言えば、支援している業務プロセスが競争優位の一部であるならカスタムソフトウェアを作るべきであり、そうでないならパッケージを買い、そのパッケージの動きに合うように業務プロセスを調整すべきだ、ということです。

私の意見が明らかに優れているにもかかわらず、そうしている企業は多くないようです。二分法そのものを見落としていることもよくあります。それも問題ですが、ここで焦点を当てたいのは、パッケージを買ったときによくはまる罠です。

上で私が「パッケージを買い、それに合うように業務プロセスを調整する」と言ったことに気づいたでしょう。そう言う理由は二つあります。第一に、ユーティリティ的な業務プロセスを支えるためにパッケージを買うのであれば、その業務プロセスには差別化要素がありません。したがって、パッケージに合うやり方でその業務プロセスを行えばよいのです。もちろん、これは非常にソフトウェア寄りの世界観です。

チームが差別化につながる仕事をしていないとしても、どこかの愚かなソフトウェアパッケージが求めるやり方ではなく、自分たちのやり方で進めたいと思うでしょう。プロセスより人を重んじる者として、私は当然その見方に大いに共感します。

しかし、この自然な行動の結果として、企業はパッケージにかなり大きなカスタマイズを加え始めます。そして、ここから問題が始まります。実際のところ、ほとんどのパッケージは、少なくとも大規模には、カスタマイズが本当に成り立つようには設計されていません。典型的には、私の同僚 Scott Shaw が「deliverability」と呼ぶもの、つまりバージョン管理、テスト、デプロイメントパイプラインへの対応などを欠いています。そのため、変更は壊れやすく、制御しにくくなります。

カスタマイズが小さいなら、それに耐えることもできます。しかし多くの場合、そうではありません。最近、ある同僚が 300KLOC のカスタマイズコードに達したパッケージカスタマイズに出会いました。これは私たちの Studios 製品群全体のコードベースよりも多く、戦略的な業務を 10 年にわたって支えてきた、私たちの大きめのクライアントプロジェクトのコードベースの 2 倍です。その規模になれば、**カスタムソフトウェアで使う道具やプロセスなしに管理できるとは期待できません**。

この問題は、ベンダーがパッケージのアップグレードをリリースしたときに最も表面化しがちです。アップグレードによってカスタマイズが壊れるため、その作業に法外な量の労力がかかることに気づくからです。Gartner は最近、企業システムを最新バージョンへ上げるには 5000 億ドルかかる、2015 年までには 1 兆ドルに増える、と見積もりました。

これは大きな数字ですが、本当のコストは、価値のなかったカスタマイズや、純粋なカスタム開発を選んだほうが安く済んだかもしれないカスタマイズに、どれほどのお金が浪費されたかにあります。

では、何ができるでしょうか。まず、ユーティリティ的な業務機能に対するパッケージカスタマイズについては、少し厳しい姿勢を取ることが重要だと思います。ソフトウェア側のコストは本当にそれに見合うのでしょうか。ユーティリティ機能ではアジャイルなアプローチの重要性は相対的に下がりますが、小さく進むという考え方には価値があります。最初はカスタマイズなしでパッケージを使い、実際にどれほどうまく機能するかを見られないでしょうか。人々は当然その変化に不安を覚えます。人とはそういうものだからです。

しかし時間が経てば、重要だと思っていたことが、今ではそれほど重要でないと分かるかもしれません。

カスタマイズがどう行われるかを、より真剣に見ることもできます。提供しやすいアプローチもあれば、そうでないものもあります。同じような規模のカスタマイズを古いバージョンのパッケージで行った人を探し、アップグレードに何が必要だったかを調べるとよいでしょう。それによってコストのより本当らしい姿が見えてくるかもしれません。一般に、パッケージでは **upgradeability を第一級の横断的要求として扱うべき**です。

ベンダーにパッケージをカスタマイズしてもらう場合、顧客側のデリバリープラクティスに従ってもらうのは非常に困難です。彼らがそのプラクティスに従わないリスクは高く、リスク計画の一部として考慮すべきです。

多くの組織は、使う言語やフレームワークの数を制限しようとします。ここで忘れてはならないのは、カスタマイズ可能なパッケージの多くは、実質的には別の言語やプラットフォームであるということです。その結果、別の言語を採用することに反対する議論は、パッケージカスタマイズの取り組みにも同じように当てはめるべきです。

実際、新しい言語の導入に反対する一般的な議論は、その言語の開発者を見つけにくくなるというものです。この問題は、通常パッケージでは特に強く現れます。そうしたパッケージが提供する雇用機会は、狭い範囲に限られがちだからです。さらに、多くのパッケージカスタマイズ作業の性質は有能な人を遠ざけるため、polyglot programming に抵抗のない優れた人材を見つけるのはいっそう難しくなります。

人材採用の難しさに伴うコストは、主流のプログラミングプラットフォームでカスタム開発する場合に比べて、パッケージを使うことで得られるどんなコスト削減よりも大きくなるかもしれません。

パッケージをカスタマイズするのではなく、パッケージが有効な API を通じてデータや機能を公開できないかを見て、カスタム機能についてはカスタムアプリケーションを書くことを検討します。これを嫌がる人はよくいます。同じワークフローの異なる部分で、別々のアプリケーションを使わなければならないからです。しかし、それは受け入れられる負担かもしれませんし、Web インターフェースを使えば負担を小さくできます。

ベンダーはこれを難しくするかもしれません。ロックインが弱まるからです。しかし連携のしやすさは、どのベンダーを選ぶかを決めるうえで重要な要素であるべきです。

ただし、ここにも罠があります。カスタマイズの大きな発生源のひとつは、異なるベンダーのパッケージ同士を統合することです。これは、best-of-breed を選ぶよりも、複数のパッケージを単一ベンダーから選ぶほうを好む大きな理由です。単一ベンダーを選べば、統合を正しく行うことがそのベンダーの利益にもなるため、統合しやすくなります。ユーティリティ的な業務機能であれば、best-of-breed パッケージの価値はいずれにしても限られています。

結局のところ、パッケージ環境はたいてい、ソフトウェア開発のプラットフォームとしては非常に貧弱です。そのようなパッケージをカスタマイズし、最新状態に保つコストは、多くの人が考えがちな額よりもずっと高くつきます。
