# Demo Front-End

## 要約

Demo Front-Endは、APIをデモし、試し、調査するための小さなUIである。APIだけのプロダクトでは、JSONレスポンスを見せるだけでは非技術者に伝わりにくく、開発者にとっても正しいpayloadやheaderを毎回組み立てるのは面倒になる。

この記事は、専用の重いフロントエンドを作るのではなく、APIと一緒に配布できる軽量なHTML、CSS、JavaScriptの画面を用意することを勧めている。最初はPostmanの代替程度でよく、必要に応じて入力生成、レスポンス整形、ワークフロー支援へ育てられる。

## 読むときの観点

- Demo Front-Endを本番UIではなく、APIを理解し試すための道具として見る。
- build stepを増やさず、APIと同じ環境で即座に使えることを重視する。
- JSONをそのまま見せるだけでなく、入力補助と出力整形で認知負荷を下げる。
- Swagger UIとの違いを、ワークフロー、動的payload、非技術者への分かりやすさで考える。

## 原文の翻訳

以下は、原文の構成に沿って主要な主張を日本語でたどる抄訳です。

開発チームでは、最新の改善を定期的にdemoすることが重要である。プロダクトにUIがあれば、そのUIを通じて自然にdemoでき、参加者が直接触ることもできる。しかしプロダクトがAPIだけの場合、事情は変わる。JSONレスポンスを画面に出しても、非技術者には意味が伝わりにくい。ビジネス関係者が自分で試すことも難しい。

このような状況で役立つのがDemo Front-Endである。これは、APIの使い方を示し、素早く試せるようにするための簡単なUIである。見た目が洗練されている必要はなく、専用buildを用意する必要もない。目的は、API利用の摩擦を下げ、demoや検証をすぐに始められるようにすることだ。

Demo Front-Endは、関連するAPIが使える場所ならどこでも同時に使えるのがよい。Spring Bootなら`src/main/resources/public/testdrive`のような場所に静的HTML、CSS、JavaScriptを置き、APIサーバーからそのまま配信できる。最初の画面は、HTTP method、path、JSON payload、responseを扱うだけでもよい。

初期版では、サンプルpayloadを静的ファイルからtextareaへ読み込み、ボタンを押すとAPIを呼び出し、成功時は成功表示、失敗時はエラー表示としてresponseを整形して見せる。まだJSONを入力しJSONを出力しているだけでも、Postmanより便利な点がある。JavaScriptでIDや日付などを自動生成でき、ユーザーが正しいpayloadを毎回覚える必要がなくなる。

著者は、Demo Front-Endには専用のnpm buildを設けない方がよいと述べる。build stepが増えると、APIを動かして試すまでのループが遅くなる。現在のJavaScriptは十分強力であり、必要ならjQuery、htmx、inline Reactのような軽い選択肢で多くを実現できる。**試したいと思ってから実際に試すまでの遅延を増やさないこと**が重要である。

Demo Front-Endは自然に育てられる。まず、さまざまな入力を生成するためのHTML formやdropdownを追加できる。旅行APIの例では、出発空港、到着空港、日付、人数、通貨などをJSONで直接編集するのは面倒で間違いやすい。dropdownや日付の自動調整でpayloadを更新すれば、専門知識が少ない人でも正しいリクエストを作りやすくなる。

次に、レスポンスもJSONのままではなく、人間が理解しやすい形に整形できる。航空券価格のmatrixのようなJSONは、そのままでは読みにくい。EJSのようなtemplate libraryを使ってHTML tableへ変換すれば、技術者にも非技術者にも結果が分かりやすくなる。

Swagger UIとの違いも述べられている。Swagger UIはAPIと一緒に配布しやすく、同じリポジトリで管理できる。しかし、入出力はJSON中心で、非技術者には親しみにくい。動的なIDや現在日付をpayloadに入れたい場合、利用者が手で直す必要がある。また、複数APIを順に呼ぶワークフローを自然に案内するのは得意ではない。Demo Front-Endなら、前のAPIの出力を次のpayloadへ入れるような流れも表現できる。

著者の経験では、Demo Front-Endは単なるおまけではなく、トラブルシューティングの強力な道具になった。あるAPI製品では、1つのAPI呼び出しが多数の下流サービス呼び出しにつながり、失敗の原因を追うのが難しかった。Demo Front-Endに下流サービスとのやりとりを表示する機能を加えたことで、テスターや内部ユーザーが数分で原因を調べられるようになった。

最後に、API endpointは単独で使われるだけでなく、何らかの判断や自動処理のworkflowとして連続して使われることが多い。Demo Front-Endは、そのworkflowを明示するドキュメントにもなり、将来の本格的なフロントエンド実装のprototypeにもなる。
