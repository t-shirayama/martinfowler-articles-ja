# Code Ownership

## 要約

この記事は、コード所有権のあり方を、強いコード所有、弱いコード所有、共同コード所有の三つに分けて整理した短い記事です。Fowler 氏は、特に強いコード所有に批判的です。コードベースをモジュール単位に分け、各モジュールを一人の開発者だけが変更できるようにすると、簡単な変更であっても所有者を待たなければならず、開発の遅れや設計上の歪みにつながりやすいからです。

弱いコード所有では、担当者は存在するものの、他の開発者も必要に応じて変更できます。大きな変更をするときには所有者と相談するのが礼儀ですが、変更そのものは止められません。共同コード所有では、モジュールごとの個人所有という考えを捨て、チーム全体がコードベースを所有します。

Fowler 氏は、弱い所有と共同所有のどちらがよいかはチームの社会的な力学に左右されると見ています。どちらも機能することも失敗することもありますが、個人的には Extreme Programming の文脈での共同コード所有を好むと述べています。

## 読むときの観点

- コード所有権を、責任の所在だけでなく、変更の流れをどう制御するかとして読む。
- 強い所有が、単純なリファクタリングにも不必要な調整コストを生む点に注目する。
- 弱い所有と共同所有の違いを、制度よりもチームの信頼関係やレビュー文化の違いとして見る。
- XP の共同コード所有が、個人の無責任ではなく、チームとしての責任を強める考え方である点を見る。

## 原文の翻訳

Fowler 氏は、これまで見てきたコード所有権の形を三つの大きなカテゴリに分けています。第一は強いコード所有です。コードベースをクラス、関数、ファイルなどのモジュールに分け、それぞれを一人の開発者に割り当てます。開発者は自分が所有するモジュールだけを変更でき、他人のモジュールを変える必要があるときは、その所有者に依頼しなければなりません。パッチを書いて渡せば多少は早くできますが、最終的な変更は所有者に委ねられます。

第二は弱いコード所有です。こちらもモジュールごとの所有者はいますが、他の開発者がそのモジュールを変更することは許されます。所有者は自分の担当モジュールに責任を持ち、他の人が加えた変更にも目を配ります。大きな変更を加える場合には、先に所有者と話し合うのが望ましいとされています。

第三は共同コード所有です。これは、モジュールを個人に所有させる考えを捨てます。コードベースはチーム全体のものであり、誰でもどこでも変更できます。これは「所有権がない」と見ることもできますが、この考え方を支持する人たちは、個人ではなくチームが所有するという点を重視します。共同コード所有という言葉は Extreme Programming に由来し、後の版では Shared Code と呼ばれています。

Fowler 氏が三つの中でもっとも好まないのは、強いコード所有です。実際の開発では、自分が必要とする変更が他人のコードに及ぶ場面が数多くあります。そのたびに相手を説得し、変更を待たなければならないと、遅れやより深い問題が生まれます。特に、変更そのものが単純な場合には、この待ち時間は大きな負担になります。**強いコード所有は、モジュール境界を越える変更を過度に重くします。**

典型的な例は、公開メソッドの名前を変えるようなリファクタリングです。現代のリファクタリングツールなら、広く使われている公開メソッドの名前変更も安全に行えます。しかし、変更がモジュール境界を越えると、強いコード所有の規則に触れてしまいます。結果として、開発者間のすべてのインターフェースを、外部に公開されたインターフェースのように扱うことになり、変更に大きな手間がかかります。

さらに悪い状況もあります。必要な実装変更をすぐに取り込めないため、他人のモジュールにあるコードを自分のモジュールへコピーし、そのコピーを呼び出して変更するような場合です。本人は後で片づけるつもりでいても、重複や不整合が残りやすくなります。

弱いコード所有は、こうした問題を和らげるよい方法です。開発者は自由に変更でき、所有者は自分の担当範囲で何が起きているかを見守ります。所有者がいることで責任感は保たれますが、変更の必要性があるたびに待ち行列が生まれるわけではありません。

弱い所有と共同所有のどちらを選ぶかは、チームの社会的な力学に大きく依存します。どちらも同じくらいうまく機能することがあり、同じくらい失敗することもあります。Fowler 氏自身は、特に Extreme Programming の文脈では、共同コード所有のチームが生む力学を好むと述べています。
