# How to manage a program in a product-mode organization

## 要約

プロダクトモードの組織では、長寿命のプロダクトチームが成果に責任を持ちます。
一方、大きな変更や横断的な取り組みでは、複数チームをまたぐ program が必要になることがあります。

この記事は、プロダクトモードを壊さずに program を扱うため、program を一時的な調整構造として使い、プロダクトチームの所有権を尊重し、成果、依存関係、意思決定、資金配分、進捗の見方を変える必要があると説明します。

## 読むときの観点

- program を、プロダクトチームを置き換えるものではなく、横断的な調整の仕組みとして読む。
- プロジェクト型の計画、固定スコープ、中央管理が、プロダクトモードと衝突する理由を見る。
- チームへの仕事の「割り当て」ではなく、成果と依存関係の調整に注目する。
- program が終わった後も、プロダクトチームが学習と所有権を持ち続ける点を確認する。

## 原文の翻訳

多くの組織が、プロジェクト中心の働き方からプロダクト中心の働き方へ移ろうとしています。プロダクトモードでは、チームは一時的なプロジェクトのために組成され、終わると解散するのではありません。長寿命のチームがプロダクトや能力を所有し、顧客や利用者の成果に責任を持ち、継続的に改善します。

この働き方は、チームの文脈を保ち、学習を積み上げ、短期的な納品ではなく長期的な価値に集中できるようにします。しかし、組織には依然として、複数のプロダクトチームをまたぐ大きな取り組みが必要になります。規制対応、大規模なプラットフォーム変更、新しい市場への参入、複数システムにまたがる顧客体験の改善などです。こうした取り組みを program と呼ぶことがあります。

課題は、program をどう扱うかです。従来の program management は、中央でスコープ、計画、依存関係、予算、進捗を管理し、チームへ作業を割り当てる傾向があります。このやり方をそのまま持ち込むと、プロダクトモードの良さを壊してしまいます。**program はプロダクトチームの上に乗る命令系統ではなく、横断的な成果を助ける一時的な調整構造**であるべきです。

### プロダクトモードと program の緊張

プロダクトモードのチームは、自分たちのプロダクトの方向性、バックログ、技術的健全性、利用者成果に責任を持ちます。彼らは、何を作るかだけでなく、なぜ作るか、どう測るか、どの順番で進めるかを理解しています。

一方、program は複数チームが関わる成果を求めます。1つのチームだけでは完了できないため、依存関係、同期、意思決定、リスク管理が必要になります。この必要性自体は正当です。しかし、program がチームの優先順位を上から置き換え始めると、長寿命チームの所有権が弱まります。チームは自分たちのプロダクトではなく、program のタスクをこなす部品になってしまいます。

プロダクトモードの組織で program を扱うには、この緊張を認める必要があります。program はチームに代わって価値を届けることはできません。価値を届けるのはプロダクトチームです。program の役割は、チームがそれぞれの所有権を保ちながら、共通の成果へ向かえるようにすることです。

### 成果から始める

従来の program は、しばしば大きなスコープ文書と計画から始まります。しかしプロダクトモードでは、最初に明確にすべきなのは成果です。どの顧客、ユーザー、事業成果を変えたいのか。どう測るのか。なぜ複数チームが必要なのか。何が変われば program は成功したと言えるのか。

成果が明確であれば、各プロダクトチームは自分たちの文脈で貢献方法を考えられます。中央の program がすべてのタスクを分解して割り当てる必要はありません。むしろ、各チームが成果に対してどの能力を提供し、どの制約を持ち、どの順番で進めるのがよいかを一緒に考えます。

このとき重要なのは、program の成果と各プロダクトの成果をつなげることです。あるチームにとって program の仕事が単なる外部からの割り込みに見えるなら、優先順位の衝突が起きます。チームのプロダクト目標と program の成果がどう重なるかを明確にする必要があります。

### チームの所有権を尊重する

program が仕事を進めるために、チームへ「この機能をこの日までに作ってください」と割り当てたくなることがあります。しかし、プロダクトモードでは、チームは自分たちのプロダクトの設計と優先順位に責任を持っています。program はチームのバックログへ直接命令するのではなく、成果、制約、依存関係を共有し、チームと合意を形成します。

これは、program に力がないという意味ではありません。組織として重要な成果なら、優先順位を上げる必要があります。ただし、その調整はチームの所有権を壊さない形で行うべきです。チームは、自分たちのプロダクト内でどの変更が必要か、どの技術的負債を先に解くべきか、どのリリース順が安全かを最もよく知っています。

program manager や program lead の役割は、指揮官よりもファシリテーターに近くなります。チーム間の合意を助け、未解決の依存を見える化し、必要な意思決定を適切な場所へ持ち込み、進捗とリスクを成果の観点で伝えます。

### 依存関係を管理する

program が必要になる理由の多くは、依存関係です。あるチームがAPIを変えなければ別のチームが進めない。データモデルの変更が複数サービスに影響する。共通プラットフォームの能力が整わなければ顧客体験が完成しない。こうした依存を放置すると、チームは待ち時間と手戻りに苦しみます。

依存関係を管理する第一歩は、見える化です。どのチームがどの成果に貢献し、どの順番で何が必要かを明らかにします。ただし、依存関係マップを作ること自体が目的ではありません。依存を減らすことが重要です。

プロダクトモードの program では、依存を細かくスケジュールで管理するだけでなく、アーキテクチャやチーム境界を見直して依存を減らせないかを考えます。API契約を明確にする、フィーチャートグルを使う、データ所有権を整理する、共通能力をセルフサービス化する、リリースを分離する。依存を減らす取り組みは、program の成功だけでなく、組織全体の将来の速度にも効きます。

### 計画は学習のために使う

program には計画が必要です。しかし、計画を「約束された未来」として扱うと問題が起きます。複数チーム、複数システム、変化する顧客理解が関わる取り組みでは、最初の計画は必ず不完全です。計画は、現在の理解を共有し、仮説を明らかにし、学習のきっかけにするためのものです。

そのため、program の計画は定期的に見直されるべきです。何を学んだか。成果指標は動いているか。依存は変わったか。あるチームの発見が、他のチームの計画を変えるか。スコープを固定して進捗率を報告するのではなく、成果に近づくために計画を更新します。

これは無計画という意味ではありません。むしろ、計画を生きたものとして扱うということです。固定されたガントチャートより、短いフィードバックループ、明確なマイルストーン、リスクの可視化、意思決定の記録が重要になります。

### 資金配分を考え直す

プロジェクト型の組織では、program はしばしば独自の予算を持ち、その予算で人を集めます。プロダクトモードでは、チームは長寿命であり、資金もプロダクトや能力に紐づくことが多くなります。ここで program 予算を従来通りに扱うと、チームを一時的な作業単位として引き剥がす力が働きます。

よりよい方法は、program の成果がどのプロダクト能力に関わるかを見て、既存のプロダクトチームの投資と結びつけることです。必要なら追加資金を提供しますが、それはチームを一時的に買うためではなく、チームが自分たちのプロダクト内で必要な変更を行えるようにするためです。

この考え方では、program の終了後も能力はプロダクトチームに残ります。作られたものは「program の成果物」ではなく、プロダクトの一部になります。保守、改善、計測も同じチームが続けます。

### 進捗の見方を変える

従来の program では、進捗はしばしばタスク完了率、マイルストーン達成、予算消化、スケジュール差異で測られます。これらは管理上役立つこともありますが、プロダクトモードでは十分ではありません。重要なのは、望む成果に近づいているかです。

そのため、program はアウトプットだけでなくアウトカムを追う必要があります。顧客の完了率が上がったか。手作業が減ったか。リードタイムが短くなったか。規制リスクが下がったか。運用負荷が下がったか。これらの指標は、チームが届けた機能の価値を理解する助けになります。

もちろん、成果指標だけでは日々の調整はできません。依存、リスク、意思決定、作業の流れも見なければなりません。しかし、ステータス報告が成果から切り離されると、program は忙しさを管理するだけになります。

### 意思決定を近くに置く

複数チームが関わると、意思決定が中央へ集まりやすくなります。中央の program board や steering committee があらゆる判断を行うと、スピードが落ち、現場の文脈が失われます。プロダクトモードでは、意思決定はできるだけ情報の近くに置くべきです。

program レベルで決めるべきことはあります。成果の優先順位、組織横断のトレードオフ、重大なリスク、予算上の判断などです。しかし、実装方法、ローカルな優先順位、技術的詳細は、関係するプロダクトチームが決めるべきです。

良い program governance は、すべてを承認することではありません。どの判断を誰が行うかを明確にし、判断が詰まったときに解消する仕組みを用意することです。意思決定の記録を残し、後からなぜその道を選んだかを理解できるようにします。

### program を終わらせる

program は一時的な構造です。永続化すると、プロダクト組織の上にもう1つの組織ができてしまいます。program は、横断的成果を達成し、必要な能力がプロダクトチームへ吸収され、残りの改善が通常のプロダクト運営で扱えるようになったら終わるべきです。

終わらせるには、最初から出口を考えておく必要があります。どの成果が達成されたら program は閉じられるか。どの責任がどのチームへ残るか。どの指標を誰が見続けるか。どの未完了作業は通常のバックログへ入るか。これらを明確にしないと、program は調整会議と報告のために生き残り続けます。

program の終了は失敗ではありません。むしろ、プロダクトモードの組織では健全なことです。横断的な集中が必要な期間が終わり、長寿命チームが学んだことを持って前へ進むのです。

### program lead のふるまい

この環境での program lead は、従来の計画管理者とは違うふるまいをします。チームに作業を押し込むのではなく、成果を明確にし、チーム間の会話を促し、依存を減らし、意思決定を早め、リスクを可視化します。彼らはチームの所有権を尊重しながら、組織として必要な横断的整合を作ります。

そのためには、プロダクト、技術、組織政治、財務のすべてをある程度理解する必要があります。program lead は、詳細な実装をすべて知る必要はありません。しかし、どの依存が本当に危険か、どの判断が遅れているか、どのメトリクスが成果を示しているかを理解できなければなりません。

### まとめ

プロダクトモードの組織でも program は必要です。しかし、program を従来のプロジェクト管理の拡大版として扱うと、プロダクトモードの価値を損ないます。program は、長寿命のプロダクトチームを置き換えるものではなく、複数チームが共通の成果へ向かうための一時的な支援構造です。

成果から始め、チームの所有権を尊重し、依存を減らし、計画を学習に使い、意思決定を情報の近くに置き、最後には program を終わらせる。そうすれば、大きな横断的取り組みを進めながら、プロダクトモードの組織が持つ学習と所有権を保てます。
