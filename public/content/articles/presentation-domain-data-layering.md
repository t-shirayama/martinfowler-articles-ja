# Presentation Domain Data Layering

## 要約

情報量の多いプログラムを分割する代表的な方法は、presentation、domain logic、data access の3つの層に分けることです。
この分離は、置き換え可能性やテスト容易性にも効きますが、Fowler が最も重視する利点は、考える範囲を狭められることです。

ただし、この3層を組織構造そのものにしてしまうと、層をまたぐ学習や協力が妨げられます。
開発者全員が full-stack である必要はなくても、チームとしては full-stack であるべきだ、という主張がこの記事の大事な結論です。

## 読むときの観点

- 3層分離の主な価値を「置換」だけでなく「考える範囲の縮小」として読む。
- 論理的な layer と物理的な tier を混同しない。
- 小さな粒度では有効でも、大規模システムの最上位分割には向かない点を見る。
- レイヤー別チーム分割がなぜ摩擦を生むのかに注目する。

## 原文の翻訳

情報量の多いプログラムをモジュール化する最も一般的な方法の1つは、それを3つの大きな層に分けることです。presentation、つまり UI、domain logic、別名 business logic、そして data access です。そのため Web アプリケーションでは、HTTP リクエストを扱い HTML を描画することを知っている web layer、検証や計算を含む business logic layer、データベースやリモートサービスにある永続データをどう管理するかを整理する data access layer に分けられているのをよく見ます。

全体として、私はこれを多くのアプリケーションにとって効果的なモジュール化の形だと感じてきましたし、定期的に使い、勧めてもいます。私にとって最大の利点は、3つの話題を比較的独立して考えられるようにすることで、注意を向ける範囲を狭められることです。domain logic のコードに取り組んでいるとき、私は UI をほとんど無視でき、データソースとのやり取りは、自分に必要なデータを渡し、望むように更新してくれる抽象的な関数群として扱えます。

data access layer に取り組んでいるときは、データを自分のインターフェースが必要とする形へ整える細部に集中します。presentation に取り組んでいるときは、UI の振る舞いに集中でき、表示または更新するデータは、関数呼び出しによって魔法のように現れるものとして扱えます。これらの要素を分けることで、それぞれの部分で考える範囲を狭められます。そのため、自分がすべきことを追いやすくなります。

この範囲の縮小は、それらをプログラムする順序を意味するものではありません。私はたいてい、層の間を行き来して反復する必要があると感じます。UX についての最初の理解をもとに data layer と domain layer を作るかもしれませんが、UX を洗練するときには domain を変える必要が生じ、その結果 data layer も変える必要が出ます。しかし、そのような層をまたぐ反復があっても、変更を進めるときには一度に1つの層へ集中するほうが楽だと感じます。

これは、リファクタリングの2つの帽子で起こる思考モードの切り替えに似ています。

モジュール化するもう1つの理由は、モジュールの実装を別のものへ置き換えられるようにすることです。この分離により、同じ domain logic の上に複数の presentation を、重複なしに構築できます。複数の presentation とは、Web アプリケーション内の別々のページかもしれませんし、Web アプリケーションに加えてモバイルのネイティブアプリを持つこと、スクリプティング用の API、さらには昔ながらのコマンドラインインターフェースかもしれません。

データソースをモジュール化すれば、データベース技術の変更にうまく対応したり、ほとんど予告なく変わる可能性のある永続化サービスを支えたりできます。ただし、データソース層を分ける動機として data access の置き換え可能性の話はよく聞く一方で、実際にそれをやった人の話はほとんど聞かない、ということは述べておかなければなりません。

モジュール性はテスト容易性も支えます。Self Testing Code の大ファンである私には、これは自然と魅力的に映ります。モジュール境界は、テストにとってよい手がかりになる境界を露出させます。UI コードはテストが難しいことが多いので、できるだけ多くのロジックを domain layer に置き、UI 経由でプログラムへアクセスするための面倒な工夫なしに簡単にテストできるようにするのはよいことです。UI 周りのテストを助ける重要な道具としては Page Object もあります。

data access は遅くて扱いにくいことが多いため、data layer の周りで Test Doubles を使うと、domain logic のテストがずっと簡単で反応のよいものになることがよくあります。

置き換え可能性とテスト容易性は確かにこの layered architecture の利点ですが、強調しておきたいのは、それらの理由がどちらもなかったとしても、私はやはりこのように layer へ分けるだろうということです。**注意を向ける範囲を小さくできる**という理由だけで十分です。

この話をするとき、私たちはそれを1つのパターン、つまり presentation-domain-data として見ることもできますし、2つのパターン、つまり presentation-domain と domain-data に分けて見ることもできます。どちらの見方も有用です。私は presentation-domain-data を、presentation-domain と domain-data の合成と考えています。

私はこれらの layer を module の一形態だと考えています。module とは、ソフトウェアを比較的独立した部品へまとめる方法を指すために私が使う一般的な言葉です。ソフトウェアシステムを変更するときに、そのうちの明確に定義された一部だけを理解すればよいようにする、その明確に定義された一部が module です。これがコードにどう対応するかは、私たちがいるプログラミング環境によって異なります。通常、最も低いレベルには何らかの subroutine や function があります。オブジェクト指向言語なら、関数とデータ構造を集める class という概念を持ちます。

ほとんどの言語には、package や namespace と呼ばれる、より高いレベルの何らかの仕組みがあり、それらは階層にできることが多いです。module は、個別にデプロイできる単位、つまり library や service に対応するかもしれませんが、そうでなくてもかまいません。

layering は、これらのどのレベルでも起こりえます。小さなプログラムでは、各 layer の別々の関数を異なるファイルに置くだけかもしれません。より大きなシステムでは、各 layer が namespace に対応し、その中に多くの class があるかもしれません。

ここでは3つの layer について述べましたが、3つより多い layer を持つアーキテクチャもよく見ます。よくある変種は、domain と presentation の間に service layer を置くこと、あるいは Presentation Model のようなものを使って presentation layer を別々の layer に分けることです。私は、layer が増えてもこのパターンの本質が壊れるとは思いません。中核となる分離は残っているからです。

依存関係は一般に、layer stack の上から下へ流れます。presentation は domain に依存し、domain は data source に依存します。よくある変種は、domain と data source layer の間に mapper を導入して、domain が data source に依存しないように構成することです。このアプローチは、よく Hexagonal Architecture と呼ばれます。

これらの layer は論理的な layer であって、物理的な tier ではありません。3つの layer をすべて自分の laptop 上で動かすこともできます。presentation と domain model を desktop 上で動かし、database を server 上で動かすこともできます。presentation を、browser 上の rich client と server 上の Backends For Frontends に分けることもできます。その場合、BFF は特定の presentation の選択肢を支えることに焦点を当てているので、私はそれを presentation layer として扱います。

presentation-domain-data の分離は一般的なアプローチですが、比較的小さな粒度にだけ適用すべきです。アプリケーションが成長すると、各 layer はそれだけで十分に複雑になり、さらにモジュール化する必要が出てきます。そうなったとき、より高いレベルの module として presentation-domain-data を使うのは、たいてい最善ではありません。

フレームワークは、view-model-data のようなものを最上位 namespace にすることを勧めることがよくあります。小さなシステムではそれで構いません。しかし、これらの layer のどれかが大きくなりすぎたら、最上位は domain 指向の module に分け、その内部を layered にするべきです。

開発者が full-stack である必要はありませんが、チームは full-stack であるべきです。

この layering が組織を誤らせるよくある道筋として、開発チームをこれらの layer ごとに分ける AntiPattern を見てきました。これは魅力的に見えます。front-end と back-end の開発には異なるフレームワーク、場合によっては異なる言語が必要であり、開発者がどちらかに専門化しやすいからです。共通のスキルを持つ人々を一緒に置くと、スキル共有が支えられ、組織はそのチームを単一の、はっきり区切られた種類の仕事の提供者として扱えるようになります。

同じように、database の専門家を全員一緒に置くことは、database や schema の中央集権化という一般的な流れに合っています。しかし、これらの layer の間には豊かな相互作用があり、頻繁に行き来することが必要になります。同じチーム内に専門家がいて気軽に協力できるなら、それほど難しくありません。しかしチーム境界は大きな摩擦を加えます。それだけでなく、システムについての重要な層横断的理解を育てようとする個人の動機も弱めます。

さらに悪いことに、layer をチームに分けると、開発者とユーザーの距離が広がります。開発者が full-stack である必要はありません。そうであれば称賛すべきことではあります。しかし、**チームは full-stack であるべきです**。
