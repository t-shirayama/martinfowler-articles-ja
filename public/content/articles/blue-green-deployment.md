# Blue Green Deployment

## 要約

Blue Green Deployment は、できるだけ同一にした2つの本番環境を用意し、一方を稼働中、もう一方を次リリースの最終確認用に使うデプロイ手法です。新しい環境で問題がなければ、ルーターの向き先を切り替えて本番化します。

この方法の価値は、停止時間を短くできることと、問題が起きたときに素早く元の環境へ戻せることです。ただし、データベース変更は難所になりやすく、スキーマ変更とアプリケーション更新を分けるなどの工夫が必要です。

## 読むときの観点

- 2つの本番環境を、できるだけ同じ状態に保つ理由を押さえる。
- 切り替えの速さだけでなく、ロールバックの速さにも注目する。
- データベース変更はアプリケーションの切り替えと別に考える。
- 災害復旧の練習を、通常リリースのたびに行える点を見る。

## 原文の翻訳

私の同僚たちと私は、クライアントに完全に自動化されたデプロイプロセスを目指すよう勧めています。デプロイを自動化すると、ソフトウェアが「完成」してから価値を発揮するまでの間に生じる摩擦や遅れを減らせます。Dave Farley と Jez Humble は、この話題について『Continuous Delivery』という本を仕上げているところです。

この本は Continuous Integration によく結びつけられる多くの考え方を土台にし、ソフトウェアをすばやく本番に出し、実際に動かす能力へ向かっています。その中の blue-green deployment の節が目に留まりました。十分に使われていない技法のひとつだと思ったので、ここで短く概観します。

デプロイ自動化の難しさのひとつは、最終テスト段階から本番稼働へ移す、切り替えそのものです。停止時間を最小にするため、通常はこれを素早く行う必要があります。blue-green deployment は、**できるだけ同一の2つの本番環境**を持つことでこれを実現します。ある時点では、その一方、ここでは blue と呼びます、が本番稼働しています。新しいリリースを準備するときには、green 環境で最終段階のテストを行います。

green 環境でソフトウェアが動くようになったら、ルーターを切り替え、すべての受信リクエストが green 環境へ向かうようにします。この時点で blue は待機状態になります。

blue-green deployment は、素早いロールバック手段にもなります。何か問題があれば、ルーターを blue 環境へ戻せばよいからです。ただし、green 環境が稼働していた間に取りこぼしたトランザクションをどう扱うかという問題は残ります。設計によっては、green が稼働している間も blue をバックアップとして保てるよう、両方の環境へトランザクションを流せるかもしれません。

あるいは、切り替え前にアプリケーションを読み取り専用モードにし、しばらく読み取り専用で動かしてから、読み書き可能なモードへ切り替えることもできます。それだけで、未解決の問題をかなり洗い出せる場合があります。

2つの環境は別々である必要がありますが、できるだけ同一でなければなりません。状況によっては別々のハードウェアでもよいですし、同じ、または別々のハードウェア上で動く別々の仮想マシンでもかまいません。1つの運用環境を別々の zone に分け、それぞれに別の IP アドレスを与える形もありえます。

green 環境を本番化し、その安定性に満足したら、次のデプロイに向けた最終テストのステージング環境として blue 環境を使います。次のリリース準備ができたら、以前 blue から green に切り替えたのと同じように、green から blue に切り替えます。こうして blue と green の両環境は、本番、ロールバック用の前バージョン、次バージョンのステージングという役割を定期的に循環します。

この方法の利点は、hot standby を機能させるために必要な基本的な仕組みと同じであることです。したがって、リリースのたびに災害復旧手順をテストできます。災害よりも頻繁にリリースしていることを願います。

基本的な考え方は、簡単に切り替えられる2つの環境を持つことです。細部には多くのバリエーションがあります。あるプロジェクトでは、ルーターを操作する代わりに Web サーバーを再起動して切り替えました。別のバリエーションとして、同じデータベースを使い、Web 層とドメイン層だけを blue-green で切り替えることもできます。

データベースは、この技法では難所になりがちです。特に、新しいバージョンのソフトウェアを支えるためにスキーマ変更が必要な場合がそうです。コツは、**スキーマ変更のデプロイとアプリケーション更新を分離する**ことです。まず database refactoring を適用して、新旧両方のアプリケーションを支えられるようスキーマを変更し、それをデプロイします。すべて正常に動くことを確認してロールバック地点を作ってから、新しいバージョンのアプリケーションをデプロイします。

そしてアップグレードが落ち着いたら、古いバージョンを支えるためのデータベース上の対応を取り除きます。

この技法はずっと以前から存在していますが、使われるべき頻度ほどには使われていないように思います。名前は、Daniel Terhorst-North と Jez Humble のどこか霧のかかった組み合わせから出てきたものです。
