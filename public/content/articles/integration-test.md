# Integration Test

## 要約

Integration Test は、独立して開発されたソフトウェア単位が接続されたときに正しく動くかを確認するテストです。ただし、この言葉は非常に曖昧になっており、多くの人は広い範囲のエンドツーエンド的なテストを想像します。一方で、実際にはより狭い範囲で行うほうが効果的なことがあります。

この記事では、1980年代的な統合テストの歴史を踏まえ、複数モジュールが一緒に動くことのテストと、複数モジュールからなるシステム全体のテストが混同されてきたことを整理します。Martin Fowler は、特にサービス間連携では narrow integration test と contract test を組み合わせることで、すべての実サービスを起動する重いテストを大幅に減らせると説明しています。

## 読むときの観点

- 「統合テスト」という言葉が、narrow integration test と broad integration test の両方を指し得る点に注意する。
- 外部サービスとの結合を、実サービスではなく忠実なテストダブルで検証できる条件を考える。
- 広い統合テストをすべて否定する話ではなく、速く保守しやすいフィードバックをどこで得るかの話として読む。

## 原文の翻訳

Integration Test は、独立して開発されたソフトウェアの単位が、互いに接続されたときに正しく動くかを判断するためのものです。この用語は、ソフトウェア業界の曖昧な標準から見てもなお曖昧になっているため、私は自分の文章で使うことに慎重でした。特に、多くの人は Integration Test が必然的に広い範囲のものだと考えていますが、実際にはより狭い範囲で行うほうが効果的な場合があります。

こうしたことではよくあるように、少し歴史から始めるのがよいでしょう。私が最初に integration testing を学んだのは1980年代で、ソフトウェア開発の考え方にはウォーターフォールが強く影響していました。大きめのプロジェクトでは、設計フェーズでシステム内のさまざまなモジュールのインターフェースと振る舞いを仕様化しました。その後、それぞれのモジュールが開発者に割り当てられてプログラムされました。

ひとりのプログラマーがひとつのモジュールを担当することも珍しくありませんでしたが、そのモジュールは数か月かけて作るほど大きいものでした。この作業はすべて分離された状態で行われ、プログラマーが完成したと思うと、テストのためにQAへ引き渡されました。

テストの最初の部分は unit testing でした。これは、設計フェーズで作られた仕様に照らして、そのモジュール単体をテストするものです。それが完了すると integration testing に進み、さまざまなモジュールを結合しました。結合の対象はシステム全体の場合もあれば、重要なサブシステムの場合もありました。

名前が示すように、integration testing の目的は、別々に開発された多くのモジュールが期待どおりに一緒に動くかをテストすることです。多くのモジュールを起動し、それら全体に対してより高レベルのテストを実行することで、一緒に動作することを確認しました。これらのモジュールは、単一の実行ファイルの一部であることもあれば、別々のものとして存在することもありました。

2010年代の視点から見ると、ここでは2つの異なることが混同されていました。

- 別々に開発されたモジュールが、正しく連携して動くことをテストする。
- 複数のモジュールからなるシステムが、期待どおりに動くことをテストする。

この2つは混同しやすいものでした。結局のところ、ショッピングカートモジュールとカタログモジュールを両方とも単一の環境で起動し、その両方を通るテストを実行する以外に、どうやってそれらをテストできるでしょうか。

2010年代の視点は、1980年代にはほとんど考えられていなかった別の選択肢を提示します。その選択肢では、ショッピングカートとカタログの統合を、ショッピングカートのコードのうちカタログとやり取りする部分を動かし、カタログの `TestDouble` に対して実行することでテストします。そのテストダブルがカタログに忠実なダブルであれば、完全なカタログインスタンスを起動しなくても、カタログとのやり取りに関する振る舞いをすべてテストできます。

それらがモノリシックアプリケーション内の別々のモジュールであれば、これはそれほど大きな問題ではないかもしれません。しかし、カタログが別サービスであり、独自のビルドツール、環境、ネットワーク接続を必要とするなら、大きな問題になります。サービスの場合、このようなテストはプロセス内のテストダブルに対して実行することも、`mountebank` のようなものを使ってネットワーク越しのダブルに対して実行することもできます。

ダブルに対して integration testing を行うときの明らかな落とし穴は、そのダブルが本当に忠実かどうかです。しかし、それは `ContractTests` を使って別にテストできます。

狭い integration test と contract test を組み合わせれば、外部サービスの実インスタンスに対してテストを実行しなくても、その外部サービスとの統合に自信を持てます。これにより、ビルドプロセスは大きく楽になります。このやり方をするチームでも、すべての実サービスを使った何らかの end-to-end system test を行うことはあります。ただし、その場合でも、テストするパスを非常に限定した最後の smoke test にとどめます。

成熟した `QA in Production` の能力があることも助けになります。それが十分に成熟していれば、end-to-end の system testing をまったく行わないことさえあります。

問題は、何が integration test を構成するのかについて、少なくとも2つの異なる考え方があることです。

narrow integration tests は、次のようなものです。

- 自分のサービスのうち、別サービスと会話する部分のコードだけを実行する。
- そのサービスのテストダブルを使う。ダブルはプロセス内にあっても、リモートにあってもよい。
- したがって、多くの狭い範囲のテストで構成される。多くの場合、その範囲は unit test と同じくらい小さく、unit test と同じテストフレームワークで実行されることが多い。

broad integration tests は、次のようなものです。

- すべてのサービスの稼働中のバージョンを必要とし、かなりのテスト環境とネットワークアクセスを必要とする。
- 相互作用を担うコードだけではなく、すべてのサービスを通るコードパスを実行する。

そして、ソフトウェア開発者の大きな集団にとって、「integration test」は broad integration tests だけを意味します。そのため、狭いアプローチを使う人たちに出会うと、多くの混乱が生じます。

もし手元にある integration test が broad なものだけなら、narrow なスタイルを試してみる価値があります。テスト速度、使いやすさ、回復力が大きく改善する可能性が高いからです。**narrow integration tests は範囲が限られているため非常に速く実行でき、`DeploymentPipeline` の早い段階で走らせて、失敗時に素早いフィードバックを得られます。**

この用語上の混乱だけでも十分ややこしいのですが、2010年代後半には「integration test」のさらに別の使い方が現れ、状況は悪化しました。これは `Unit Test` の意味が分岐したことから来ています。ある人たちは unit test を、私が solitary unit test と呼ぶもの、つまりテスト対象以外のすべてのプログラム要素をテストダブルで置き換えるテストとして定義します。この狭い定義を前提に、一部の書き手は sociable unit tests を意味する言葉として「integration test」を定義するようになりました。

こうした理由から、私は「integration test」という言葉に慎重です。この言葉を読むときは、著者が本当はどの種類を指しているのかを知るために、より多くの文脈を探します。broad integration tests について話す場合、私は「system test」または「end-to-end test」という言葉を使うほうを好みます。narrow integration tests については、よりよい名前を持っていないのでその言葉を使います。ただし、読者にテストの性質が伝わるように「narrow」を付けます。私は今後も、unit test という言葉は両方の種類に使い、区別が必要なときには solitary と sociable を使います。
