# Evans Classification

## 要約

Evans Classification は、Eric Evans が『Domain-Driven Design』で示した、ドメインオブジェクトを Entity、Value Object、Service に分けて考える分類です。
それぞれは、同一性を持つもの、属性の組み合わせとして意味を持つもの、ドメイン内の独立した操作として捉えられるものです。

この分類は実務感覚によく合う一方で、厳密に定義しきるのは難しく、文脈なしに使うと他の意味と混同されやすい用語でもあります。
Domain Model の文脈で、Eric Evans の本における意味として読んでいることを明確にするのが重要です。

## 読むときの観点

- Entity、Value Object、Service を、実装技法ではなくドメインモデル上の役割として見る。
- Entity と Value Object の違いを、同一性と等価性の扱いから考える。
- Value Object は不変にする、という実装上の注意を押さえる。
- Service という言葉が SOA や Service Layer と混同されやすい点に注意する。

## 原文の翻訳

Eric Evans は、すばらしい著書『Domain-Driven Design』の中で、出会うことになりそうなさまざまな種類のドメインオブジェクトについて、ひとつの分類を作っています。

- Entity: 時間の経過や表現の違いをまたいで続く、はっきりした同一性を持つオブジェクトです。これらは「reference object」と呼ばれることもあります。
- Value Object: 属性の組み合わせとしてだけ意味を持つオブジェクトです。すべての属性の値が同じである 2 つの value object は、等しいものとみなされます。私は『Patterns of Enterprise Application Architecture』でも value object について説明しています。
- Service: ドメインの文脈における、独立した操作です。Service Object は、ひとつまたは複数の service をオブジェクトとしてまとめます。通常、実行コンテキストの中では、各 service object の型につきインスタンスは 1 つだけです。

この分類は、ドメインモデルに必要なものについての私の経験とよく響き合います。
難しいのは、この 3 つを正確に定義するのが難しく、**「見ればわかる」種類のもの**だという点です。

そのため、いくつか例があると助けになります。
Entity は通常、Customer、Ship、Rental Agreement のような大きなものです。
Value は通常、Date、Money、Database Query のような小さなものです。
Service は通常、Database Connection、Messaging Gateway、Repository、Product Factory のような、外部リソースへのアクセスです。

Entity と value の明確な分かれ目のひとつは、value は等価性を判定するメソッドを上書きするのに対し、entity は通常そうしない、という点です。
したがって、value は hash も上書きします。
これは、処理コンテキスト内で、同じ概念上の entity を表すオブジェクトが複数あることを通常は望まない一方で、複数の "5.0" オブジェクトがあっても気にしないからです。
Value は、区別のある言語では primitive であってもよく、.NET のように言語上の特別なサポートを持っていてもよいですが、**必ずしもそうである必要はありません**。

従うべき重要なルールのひとつは、**value object は不変であるべき**だということです。
そうでないと、aliasing bug によるさまざまな問題に入り込んでしまいます。
値を変更する場合、たとえば私の身長を変える場合、height オブジェクトを変更するのではなく、新しいものに置き換えます。

Service object は、グローバル変数、クラスフィールド、Robert Martin の用語で言う monostate、あるいは singleton を使って実装されることがよくあります。
確かに、それらは通常 singular です。
つまり、1 つだけ存在するということです。
ただし、それをどう実現するかはもっと多様です。
通常、その単一性は処理コンテキストの中にあります。
したがって、マルチスレッド環境ではスレッドごとに 1 つ、という形になります。
いずれにしても、実装の仕組みは他のドメインオブジェクトから隠されるようにして、**あとから簡単に変更できる**ようにしておくべきです。

Eric は本の中で、service は stateless であるべきだと述べています。
ただし、私たちはその点について話し合ったことがあり、彼は今では、それが必要だとは考えていません。
もちろん、**そうできるなら望ましい**ことではあります。

この領域の問題のひとつは、これらの用語が示唆に富んでいる一方で、他の考え方とひどく混ざり合ってしまうことです。
Entity は、データベーステーブル、あるいはデータベーステーブルに対応するオブジェクトを表すために使われることがよくあります。
Service には、Service Oriented Architecture 全体の話がありますし、アプリケーションアーキテクチャにおける service layer もあります。
したがって、私がこれらの用語を使うなら、**Domain Model の文脈で、Eric の本における意味として使っている**ことを明確にしなければなりません。

ですから、人々がこれらの言葉をこの意味で使っていると決めつけることには注意してください。
これらの言葉は、かなり多くの意味を背負わされています。
残念ながら、代替になる言葉はあまりありません。
