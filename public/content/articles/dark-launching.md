# Dark Launching

## 要約

Dark Launching は、新しいバックエンドの振る舞いを既存ユーザーのリクエストから実行しつつ、ユーザーには見せないまま負荷や性能への影響を調べる手法です。
公開発表やUI公開の前に、実運用に近い条件で挙動を観察できます。

Feature Flag と組み合わせると、本番環境でオン・オフを切り替えられます。
ただし、ユーザーの選択に依存する機能の検証には Canary Release の方が向いています。

## 読むときの観点

- Dark Launching は「見せないが実行する」リリース手法である。
- 主な狙いは、負荷、性能、運用上の影響を公開前に測ることにある。
- Keystone Interface や Feature Flag と相性がよい。
- 再実装した機能を旧実装と並行実行して結果を比較する用途にも使える。
- 部分公開一般を指して使われることもあるが、意味が広がりすぎている点に注意する。

## 原文の翻訳

Dark Launching とは、新しい、または変更されたバックエンドの振る舞いを、既存ユーザーの通常の利用から呼び出すが、ユーザーにはそれが呼ばれているとわからないようにすることを指す。目的は、新機能を公に発表する前に、システムへ追加される負荷や性能への影響を評価することにある。

例として、小売サイトのチェックアウトにクロスセルのおすすめ機能を追加する場合を考える。おすすめを計算すれば、チェックアウト処理に時間と負荷が加わり、場合によってはサイト全体を遅くするかもしれない。Dark Launching を使えば、機能を公開する前にその影響を調べられる。

このような機能は、Keystone Interface を使って構築し、本番リリースに統合しておくことができる。ユーザーインターフェースはまだ追加しないため、ユーザーには機能の存在が見えない。さらに、Keystone を加える前にチェックアウトの流れからおすすめエンジンを本番と同じように呼び出し、結果だけをUIに表示しない。こうすれば、公開後と同じ処理は走るが、誰にも見えない。

Feature Flag を使えば、本番環境でおすすめ機能を簡単にオン・オフできる。性能に不安な影響が見えた場合は、ユーザーが気づく前に切り戻せる。その後、性能が許容範囲に入るまで調整し、最後に Keystone を追加して機能を公開する。

Dark Launching は、再実装した機能の並行実行にも使える。旧コードと新コードの両方を呼び出して結果を比較し、UIには片方の答えだけを返す。これにより、**新しいアルゴリズムの違いを本番の入力で検出**できる。

この手法が最も向いているのは、既存のユーザー操作に付随して実行され、ユーザーが自分で選んで使う種類ではない処理である。ユーザーの選択に依存するものを試したいなら、Canary Release を使う方が適している。

なお、Dark Launch という言葉は、登場してから意味が拡散してきた。Canary Release やその他の部分的なリリース戦略を指して使われることもあるため、会話の中では具体的に何を意味しているのか確認するとよい。
