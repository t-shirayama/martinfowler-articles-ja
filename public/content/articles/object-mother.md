# Object Mother

## 要約

Object Mother は、テストで使う例示オブジェクトを作るためのファクトリです。複雑なフィクスチャを毎回組み立てる代わりに、チーム内で馴染みのある標準的なデータを提供し、必要に応じて少し調整して使えるようにします。

## 読むときの観点

- テストフィクスチャを共有することで、テストの読みやすさがどう変わるか
- 標準データを再利用する利点と、変更時の結合の強さ
- `CustomerMother` や `ProductMother` のように対象ごとに分ける設計

## 原文の翻訳

Object Mother は、テストで使う例示オブジェクトを作るためのクラスの一種です。

ある程度の大きさのシステムでテストを書くと、多くの例示データを作る必要が出てきます。たとえば、従業員の傷病手当の計算をテストしたいなら、従業員が必要です。しかしそれは単純なオブジェクトではありません。婚姻状況、扶養家族の人数、雇用履歴や給与履歴なども必要になるでしょう。作らなければならないオブジェクトはかなり多くなる可能性があります。この一連のデータは、一般にテストフィクスチャと呼ばれます。

最初の一手は、xUnit テストのセットアップメソッドでフィクスチャを作ることです。そうすれば、複数のテストで再利用できます。しかし、よくある問題は、似たデータが複数のテストクラスで必要になることです。この段階では、標準的なフィクスチャを返すファクトリオブジェクトを用意するのが自然です。たとえば「先週入社した John」や「10年勤めている Heather」のような従業員を返すものです。

Object Mother は、そのようなファクトリにつけられた覚えやすい名前にすぎません。この名前は、世紀の変わり目のころに Thoughtworks のプロジェクトで生まれ、印象に残りやすかったため、そのまま定着しました。

Object Mother が生み出す定番オブジェクトは、チームにとって馴染みのあるものになります。ときにはユーザーとの会話にまで入り込むことがあります。この点では、ペルソナの考え方に似ています。ただし、常に人物であるとは限りません。保険契約、供給契約、あるいはテストフレームワークが必要とするどんなデータでもかまいません。複数のテストで似たデータを使うと、そこで使われている例を人々が理解しやすくなります。

こうした定番オブジェクトは、特定のテストにそのままぴったり合うとは限りません。しかし多くの場合、追加のセットアップで目的に合わせられます。「John を使って、2か月前から病欠していることにしよう」という具合です。ときには新しい定番オブジェクトを Object Mother に追加する必要もありますが、可能なら既存のものを少し調整して使うほうがよいでしょう。読み手が既存の定番オブジェクトに慣れていれば、テストの理解が早くなるからです。

通常は、生成したいオブジェクトの種類が複数あります。そのため、クラスごとに別々の Mother を作ると便利です。たとえば `CustomerMother`、`ProductMother` といった形です。

Object Mother にも欠点はあります。特に、多くのテストが Mother の中にある正確なデータに依存するため、**強い結合が生まれます**。その結果、何らかの理由で標準データを変えたくなったときに扱いが難しくなります。クラス自体に変更があればテストの移行も必要になりますが、それはいずれにしても避けにくい問題です。
