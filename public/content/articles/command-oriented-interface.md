# Command Oriented Interface

## 要約

Command Oriented Interfaceは、モジュールの操作をメソッド呼び出しではなく、操作ごとのコマンドクラスとして表すインターフェイスのスタイルです。メソッド呼び出しより手間は増えますが、実行器をdecoratorで包むことでトランザクション、ログ、キューイング、ネットワーク越しの受け渡しなどを扱いやすくなります。

一方で、コマンドクラスへロジックを詰め込みすぎると重複が増えます。Fowlerは、これはコマンドそのものの問題というより、Transaction Scriptに多くのロジックを持たせる一般的な問題だと見ています。

## 読むときの観点

- 通常のメソッド指向インターフェイスとの違いを見る。
- コマンド実行器を分けることで共通処理を入れやすくなる点に注目する。
- コマンドクラスは実装本体ではなく、クライアント向けインターフェイスとして考える。
- 戻り値をどう扱うかが設計上の論点になる。

## 原文の翻訳

モジュールへのインターフェイスとして最も一般的なスタイルは、手続き、またはオブジェクトのメソッドを使うものです。たとえば、契約について一連の料金を計算するモジュールが欲しいなら、計算を行うメソッドを持つ`BillingService`クラスを用意し、次のように呼び出すでしょう。

```text
aBillingService.calculateCharges(aContract)
```

Command Oriented Interfaceでは、操作ごとにコマンドクラスを用意し、次のように呼び出します。

```text
CalculateChargeCommand.new(aContract).run()
```

本質的には、メソッド指向インターフェイスで持っていたそれぞれのメソッドに対して、1つのコマンドクラスを持つことになります。

よくある変形は、コマンドの実行そのものを担う別のcommand executorオブジェクトを持つ形です。

```text
aCommandExecutor.run(CalculateChargeCommand.new(aContract))
```

Strutsのようなフレームワークを使ったことがあれば、actionクラスがこの操作スタイルに従っていることに気づくでしょう。

では、なぜ人々はこのアプローチを好んだり嫌ったりするのでしょうか。まず言えるのは、Command Oriented Interfaceはメソッド指向のものよりかなり手が込んでいるということです。コマンドをインスタンス化し、それを実行へ渡さなければなりません。単にメソッドを呼ぶより複雑なので、このアプローチの支持者であっても、重要なインターフェイス、たとえばservice layer、サーバー側ロジック、大きなサブシステムのインターフェイスにだけ使うのが普通です。

Command Oriented Interfaceにはいくつもの利点があります。主要な利点の1つは、command executorをdecoratorで包むことで、コマンドへ共通の振る舞いを追加しやすいことです。これはトランザクション、ログ記録などを扱うときにとても便利です。コマンドは後で実行するためにキューへ入れられますし、コマンドとそのデータがシリアライズ可能なら、ネットワーク越しに渡すこともできます。コマンド名と引数から合成したキーに結果を結びつけておけば、コマンド結果をキャッシュすることもできます。

コマンドに対する不満を見聞きした中で、最大の問題は、コマンド内に重複したロジックが多いことに由来しています。これは、多くのロジックを含むTransaction Scriptとしてコマンドが作られているときに最も起きがちです。これは必ずしも、メソッドではなくコマンドを使うこと自体の問題ではありません。この問題はTransaction Script一般の問題です。

ただし、コマンド指向の構造はこの傾向を強めるかもしれません。というのも、多くの人はクラスが存在する価値を持つには1、2ページ分のコードが必要だと感じ、その結果、コマンドに入れるべき以上のコードを入れてしまうからです。

このページで私がinterfaceという言葉を使っていることに気づくでしょう。これは、コマンドを使うかどうかの選択が、主にコマンドロジックの実装そのものではなく、クライアントに対するインターフェイスの問題であることを反映しています。`run`メソッドが別のメソッドを1行で呼び出すだけのコマンドクラスがあっても、まったく妥当です。そうすれば、**コマンドの利点を得ながら、ロジック自体をコマンドクラスから外に保てます**。

このようなコマンドクラスは、ごく少ない行数のコードになります。

コマンドについてよく出る質問は、何を返すかです。汎用的な`run`メソッドには、`Object`や`CommandResult`のような一般的な戻り値型が必要です。しかし、コマンドを実行した結果を得るには、より具体的な型が欲しくなります。1つの方法は、各コマンドクラスに対して結果クラスを定義し、命名規約を使うことです。たとえば`CalculateChargeCommand`なら戻り値型を`CalculateChargeResult`にします。別の方法は、コマンド自身に結果を保持させることです。

この場合は、まずコマンドを実行し、その後でコマンドオブジェクトへ結果を問い合わせます。
