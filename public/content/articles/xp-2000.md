# The XP 2000 Conference

## 要約

2000年6月にサルデーニャ島で開かれた XP2000 conference の参加記です。Extreme Programming と軽量な方法論をめぐる発表、事例、議論の様子を通じて、XP が熱狂や宣伝だけではなく、実践から学ぶ場として語られていたことが分かります。

記事では、Chrysler Payroll project、Object Mentor の経験、Refactoring Browser、OTI の事例などが紹介されます。Fowler 氏は、軽量な方法論は万能薬ではないものの、成功に向かうための原則を与えると見ています。また、XP と RUP の関係、XP コミュニティの開かれた態度、組織が技術者に技術判断を任せるかどうかという問題にも触れています。

## 読むときの観点

- 2000年時点の XP が、まだ形を固めつつある運動だったことを意識して読む。
- 事例紹介を、成功談だけでなく失敗やキャンセルの要因も含む経験共有として見る。
- XP と RUP の関係を、手法の名前よりも人と組織のあり方の問題として読む。
- Jack Bolles 氏の補足から、XP の実践を個人で始められるものとチーム全体で必要になるものに分けて考える。

## 原文の翻訳

6月下旬、100人を超える人々が地中海のサルデーニャ島に集まり、Extreme Programming とその他の柔軟な方法論について議論する XP2000 conference に参加しました。会場は国際的なソフトウェア会議の場所としては便利とは言えませんでしたが、主催者は温かく、食事はすばらしく、会議そのものもよいものでした。

参加者数はちょうどよい規模でした。当初は60人ほどを見込んでいたところ、実際には160人が集まりました。会場には十分な活気がありながら、話したい相手に近づけないほど大きすぎるわけではありません。Ron Jeffries、Robert Martin、Don Wells、Kent Beck といった XP の中心人物に加え、Erich Gamma、Dave Thomas、Ralph Johnson、Alistair Cockburn らも参加していました。

最初の本格的な日には、適応的なプロセスを使ったプロジェクトやチームについての招待講演が四つありました。完全に XP を使った事例は、その中では有名な Chrysler Payroll project、いわゆる C3 でした。Ron Jeffries 氏は、その成功だけでなく、キャンセルに至った要因も含めて説明しました。

C3 は最初の1年ほどは非常にうまく進み、Chrysler の給与支払いに使われるシステムを届けました。しかしその後、給与部門と IS 部門の目標のずれ、チームと管理側とのコミュニケーション不全などが問題として現れました。この報告は、XP の事例を勝利の物語としてだけでなく、現実の組織問題を含む経験として見るものでした。

Robert Martin 氏は、教育テスト企業でのプロジェクトについて話しました。要求は曖昧で変わりやすかったため、1週間単位の短いイテレーションと顧客との頻繁な接触で対応しました。XP のすべての実践を使ったわけではありませんが、短いイテレーションは C++ 環境でも機能し得ることを示しました。

Ralph Johnson 氏は、Smalltalk の有名な Refactoring Browser の開発を例に挙げました。このツールの開発でも、短いイテレーション、進化的な設計、そして設計の中でリファクタリングを積極的に使うことが中心にありました。

Dave Thomas 氏は、Object Technology International の経験を紹介しました。OTI は Smalltalk と仮想マシン技術を使った難しい組み込みシステムのプロジェクトで知られていました。彼が強調したのは、人に焦点を当てること、自動テスト、短いイテレーションでした。また、固定価格のプロジェクトでもこれらの技法を使えると述べました。

これらの講演を通じて見えてきたのは、軽量な方法論で成功する道は一つではないということです。軽量なアプローチも他の方法論と同じく万能薬ではありませんが、人々を成功へ向かわせる原則を与えてくれます。

会議で繰り返し話題になったのは、XP と Rational Unified Process の関係でした。RUP が XP を取り込むこと、あるいは XP を RUP に見えるようにすることには、業界で XP を採用しやすくする利点があります。一方で、RUP-XP のような形にすると、XP の重要な実践が薄まるのではないかという懸念もありました。

Fowler 氏は、この点については大きな違いはないと見ています。RUP が XP を含むことは止められないし、XP を部分的に使う問題は RUP の有無にかかわらず起こります。より重要なのは、適応的なプロセスが本質的に人を中心にしたものであり、人を抽象的な資源として扱う環境では機能しないという点です。XP が RUP の旗の下で採用されるかどうかよりも、**人と技術判断をどう扱うか**のほうが決定的です。

この会議でよかったことの一つは、XP を唯一の正しい道として押しつける雰囲気がほとんどなかったことです。インターネット上では、XP 支持者が逸脱を許さない熱狂的な集団のように見えることがあります。しかし会議では、XP をよい基盤としつつ、変化や部分的な採用も認められていました。

Fowler 氏自身も、この開かれた態度に安心したと述べています。XP の強い姿勢は人々の注意を引きましたが、同時に抵抗も生みました。実践には大きな価値があるものの、方法論は人々に合わせて選ばれ、適応されるべきであり、人々が方法論に合わせて形を変えられるべきではありません。

最後のパネルでは、XP をソフトウェア業界でどう採用するかが話題になりました。多くの人にとっての問題は、自分たちの組織で XP、あるいは XP の一部を使えるかどうかでした。参加者自身が使いたいと思っても、上司を説得できるのかという問いです。

Fowler 氏は、この問いは少し的を外していると考えます。適応的な方法の強いテーマの一つは、技術的な責任を技術者に置き、技術者が技術的な意思決定をすべきだということです。問題は、会社が XP を許すかどうかではなく、会社が技術判断を技術者に任せるかどうかです。それがなければ、XP かどうかは重要ではありません。

もし会社がそうではないなら、次の問いは会社を変えられるかどうかになります。変えられないなら、そこに残るべきかを考えなければなりません。Fowler 氏はパネルで、即興の言い回しとして「組織を変えられないなら、組織を変えよ」と述べました。これは、働く組織のやり方を変えるという意味にも、働く組織そのものを変えるという意味にも読めます。

Jack Bolles 氏の補足では、XP2000 は話題そのものよりも、議論と参加者の質の高さが印象的だったとされています。XP のように強い名前を持つテーマなら、宣伝や旗振りが多くなると予想していましたが、実際にはそうではありませんでした。

会場には Smalltalk の経験者が多く、現在は Java に取り組んでいる人も多くいました。また、C を使う通信系の人々もかなりいました。こうした人々に共通していたのは、開発プロセスに関わる全員のよいコミュニケーションが不可欠だという認識です。XP の12の実践はいずれも、開発者同士、開発者と顧客、あるいは顧客同士のコミュニケーションを促します。

Bolles 氏は、XP の12の実践を二つのグループに分けられると考えました。一つは、テスト、リファクタリング、シンプルな設計、ペアプログラミング、持続可能な労働時間、コーディング標準のように、個人でもかなり始めやすい実践です。もう一つは、小さなリリース、継続的インテグレーション、計画ゲーム、オンサイト顧客、共同所有のように、チーム全体を必要とする実践です。

個人向けの実践は、開発者が自分で始めることができます。テストを書いたりリファクタリングしたりするよい例は、周囲に広がることがあります。一方で、開発チームを超えたチーム実践を変えるのは難しくなります。顧客や管理者が慣れていないことをする必要があり、彼らにとっては制御を失うように感じられるからです。

XP の文献や伝承に残る大きな穴として、顧客や管理者に「変化を受け入れてもらう」方法が十分に語られていないことがあります。XP2001 までに、その点が変わることを期待して記事は締めくくられています。
