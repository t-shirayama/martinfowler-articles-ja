# Rigorous Agile

## 要約

Agile methods には厳密な定義がない、という批判はよくあります。厳密な定義がなければ、あるチームが本当に agile なのかを判定しにくく、教えるためのカリキュラムも作りにくい、という不満です。

この記事は、その不満を理解しつつも、agile methods の核心には、状況に応じてチーム自身がプロセスを選び、継続的に変えていくという考え方があると述べています。適合性を検査できるような厳密なプロセス定義は、その哲学と相いれません。

## 読むときの観点

- Agile を、決まった手順への準拠ではなく、チームが自分たちの働き方を調整し続ける考え方として読む。
- 厳密な定義がないことは弱点であると同時に、agile の人間中心の性質から来るものだと捉える。
- Scrum や Extreme Programming の有効性を測る難しさは、そもそも何を「その方法」と呼ぶかの難しさにつながっている。
- 「見ればわかる」という判断には経験が必要で、経験者の間でも意見が割れうる点に注意する。

## 原文の翻訳

私は、agile methods には厳密な定義がない、という不満によく出会います。不満を言う人は、そのせいで、あるチームが agile method を使っているのかどうかを判断できない、と言うかもしれません。また、agile methods を人に教えるのが難しくなる、とも言うかもしれません。カリキュラムはどうなるのか、というわけです。

ある程度まで、私はこの不満の痛みを感じます。しかし、治療法はないと受け止めています。この厳密さの欠如は、agile methods の性質を定めるものの一部であり、**その中核にある哲学の一部**です。

一般的な思考プロセス、そして特にソフトウェア開発における根本的な問題の一つは、置かれる状況が非常に多様であることです。異なる種類のシステムには、異なる種類の圧力や力学があります。そのため、それらを十分に覆える「何をすべきか」の厳密な記述を作るのは、とても難しくなります。

この影響は、ソフトウェア開発が非常に人間中心の活動であり、人間は本来、一貫性に欠け、とても変わりやすいという事実によって、さらに強まります。Agile を支持する人たちがここから導いた結論は、ソフトウェア開発を厳密なプロセスに縛ろうとしても効果的ではない、ということでした。なぜなら、それはそのプロセスを実行する主要な構成要素、つまり人間の本質を無視することになるからです。

おそらく私たちの職業がコンピュータを扱うものだからこそ、私たちは人間のシステムを、コンピュータをプログラムするのと同じようにプログラムしたくなるのでしょう。両者はこれほど違っているにもかかわらず、です。

このすべてからの帰結は、agile methods が根本的に、どのプロセスに従うかをチーム自身が決めることを期待しており、さらにチームが自分たちのプロセスを積極的かつ定期的に変えることを期待している、という点です。適合性を検査できるような厳密なプロセスを定義しようとする試みは、**この哲学に反します**。

これがもどかしいことは否定できません。そもそも Scrum とは何かについて明確な定義を得られないのに、agile methods が代替手段より効果的かどうか、あるいは Extreme Programming が Scrum より効果的かどうかを、どうやって調査すればよいのでしょうか。クライアントが Extreme Programming を使ってシステムを作ってほしいと望むとき、それが本当に行われているとどうやって判断できるのでしょうか。

そこには「見ればわかる」という感覚があります。しかし、その感覚を持つには経験ある実践者が必要ですし、それでもなお、そうした実践者たちの間で意見が大きく分かれる余地があります。

この難問に対する答えを、私は持っていません。実のところ、答えはないと思っています。それは活動そのものから生じる残念な結果です。泳げば濡れることを避けられないのと同じです。
