# How platform teams get stuff done

## 要約

内部プラットフォームの成功は、どれだけ多くのチームに採用されるかに強く左右される。そのため、プラットフォームチームは、自分たちのコードベースだけで完結する仕事ではなく、他のプロダクトチームのコードベースに変更を入れてもらう、または一緒に変更するための協働を避けて通れない。

この記事は、プラットフォーム採用の場面を「プラットフォーム移行」「プラットフォーム消費」「プラットフォーム進化」の3つに分け、それぞれでどのチームが作業を駆動し、どのチームが変更対象のコードベースを所有しているかを手がかりに、適した協働パターンを整理している。結論として、若いプラットフォームでは密な協働が必要になる一方、採用が広がるほど、よりスケールし、利用側の認知負荷を下げる as-a-service 的な関わり方へ移行することが重要だと述べている。

## 読むときの観点

- いまの作業が「移行」「消費」「進化」のどれに当たるか。
- 作業を駆動しているチームと、変更対象のコードベースを所有しているチームはどちらか。
- チケット依頼、埋め込み、信頼された外部貢献、社内オープンソースのどれが現実的か。
- 密な協働をいつまで続け、いつセルフサービスや as-a-service に寄せるべきか。

## 原文の翻訳

内部プラットフォームの成功は、それを採用するチームの数によって決まる。つまり、プラットフォームチームの成功は、他チームと協働する力、とりわけ他チームのコードベースに必要な変更を入れる力にかかっている。

この記事では、プラットフォームチームが他チームと関わるときに現れる協働の段階を見ていく。扱うのは、プラットフォーム移行、プラットフォーム消費、プラットフォーム進化の3つである。それぞれで何が違うのか、プラットフォームチームとプロダクトデリバリーチームがどのような運用モデルを取れるのか、どの協働パターンが合うのかを考える。

ここでいうプロダクトデリバリーチームは、会社の顧客、つまりプロダクトの利用者に向けて機能を作るチームである。フィーチャーチーム、プロダクトチーム、垂直チームと呼ばれることもある。この記事では短く「プロダクトチーム」と呼ぶ。

一方、プラットフォームデリバリーチームは、社内の他チームに向けたプロダクトを作る。プラットフォームチームのプロダクトのエンドユーザーは、同じ会社の中にいる他チームである。**プラットフォームチームは、社内の他チームを顧客とするプロダクトを作っている。**

ここでいうプラットフォームは、最近よく語られる Internal Developer Platform だけに限らない。データプラットフォーム、フロントエンドのデザインシステム、実験基盤のような内部プロダクトも含む。さらに、技術的なプラットフォームに限らず、フィンテック企業の送金サービスや、EC企業の商品カタログサービスのように、共有された業務能力を提供する内部プロダクトにも同じ考え方を適用できる。

3つの段階を見るうえで重要なのは、誰が作業を駆動しているのか、そして変更が行われるコードベースを誰が所有しているのかである。この2つの答えによって、どの協働パターンが意味を持つかが大きく変わる。

### プラットフォーム移行

プラットフォーム移行では、新しいプラットフォーム能力に切り替えるために、プロダクトチームのコードベースへ変更を加える。ここでは、変更を進めたいのはプラットフォームチームだが、変更対象のコードベースを所有しているのはプロダクトチームである。そのため、チーム間の協働が必要になる。

移行作業にはいくつかの種類がある。比較的単純なものとしては、共有コンポーネントライブラリのバージョンアップや、サービスの言語ランタイムの更新がある。より大きな移行としては、サードパーティシステムへの直接統合を、プラットフォームチームが保守する社内ラッパーやゲートウェイに置き換える作業がある。ログ、分析、観測性の計装を共有ライブラリに寄せることや、決済処理を内部ゲートウェイ経由にすることが例である。

廃止予定の社内サービスから新しいサービスへ移る移行もある。たとえば古い `User` サービスから新しい `Account Profile` サービスへ移る、あるいは複数の信用情報関連サービスを統合されたゲートウェイに置き換える、といったものだ。インフラレベルの再プラットフォーム化もあり、プロダクトチーム所有のサービスを Docker 化する、サービスメッシュを導入する、データベースを MySQL から Postgres に切り替える、といった作業が含まれる。

移行では、プロダクトチームがその変更に強く動機づけられていないことが多い。新しいプラットフォームが魅力的な能力を提供する場合は別だが、多くの場合、より大きなアーキテクチャ上の取り組みの一部として移行を求められており、プロダクトチーム自身がすぐ大きな価値を得るわけではない。

一つの方法は、プラットフォームチームがプロダクトチームのバックログにチケットを作り、必要な変更をプロダクトチーム自身に依頼することだ。この方法はスケールしやすい。対象となるすべてのプロダクトチームに実装作業を分配でき、進捗も追跡しやすい。チケット作成をプログラムマネージャーなどが担うこともできる。

しかし、この方法は遅くなりやすい。プロダクトチームが作業を始めるまで長いリードタイムが発生する。また、優先順位をめぐる綱引きも起きる。依頼された側のチームは直接の利益を感じにくいため、より緊急で影響の大きい作業を優先したくなるのは自然である。

別の方法は、プラットフォームチームがプロダクトチームのコードベースに自ら変更を入れることだ。このとき、プラットフォームエンジニアが一時的にプロダクトチームに加わる Tour of Duty、信頼された外部貢献者として変更する Trusted Outsider、社内オープンソースとしてプルリクエストを出す Internal Open Source といったパターンがある。

この方法の利点は、変更までのリードタイムを短くしやすいことである。作業を必要としているプラットフォームチーム自身が実装するため、インセンティブがそろっている。一方で、プロダクトチーム側がそれを受け入れられる必要がある。プラットフォームエンジニアを一時的に迎え入れられるか、外部からの変更を任せられるだけの信頼関係があるか、社内オープンソースを支える準備があるかが問われる。

この自力実装の戦略は、スケールしにくい。通常、プラットフォームチームのエンジニアリングキャパシティは、プロダクトチーム全体のキャパシティより小さい。すべてをプラットフォームチームが担うと、プロダクトチーム側の実装能力を活用できない。

実際には、これらの組み合わせになることが多い。まず多数のプロダクトチームにチケットを作り、しばらく働きかける。進めるチームもあれば、忙しさや意欲の低さから残るチームも出る。そこでプラットフォームチームが袖をまくり、よりスケールしないが直接的な方法で変更を入れる。

### プラットフォーム消費

プラットフォーム消費は、プロダクトチームが日々の機能開発の中でプラットフォーム能力を利用している状態である。たとえば、インフラプラットフォームチームが保守する service chassis を使って新しいサービスを立ち上げる、社内の顧客分析プラットフォームを使い始める、機密データ用ストアに個人情報を保存する、共有 UI コンポーネントライブラリを使い始める、といった作業がある。

移行との大きな違いは、プラットフォーム消費では、作業を駆動しているのもコードベースを所有しているのもプロダクトチームだという点である。プロダクトチームには自分たちのより大きな目的があり、それを達成するためにプラットフォームの機能を利用する。

この状況では、チーム間の協働は不要に見えるかもしれない。理想的には、Stripe や Auth0 のように、よく文書化され、使いやすく、プロダクトエンジニアがプラットフォームチームの直接支援なしに使える完全セルフサービスのプラットフォームを作ることが目標になる。

しかし現実には、特に初期の内部プラットフォームはそこまで整っていない。文書が弱く、エラーメッセージがわかりにくく、混乱を招くバグもある。プロダクトチームは途中で行き詰まり、プラットフォームチームに手伝ってほしいと頼むことになる。

プロダクトチームがプラットフォームチームに手厚い支援を求めると、再びチーム間の協働が必要になる。ここでの一つの形は、プラットフォームチームが小さなプロフェッショナルサービス部門のように振る舞い、顧客であるプロダクトチームのシステムに自分たちのプロダクトを統合してあげることである。

このモデルでは、プロダクトチームがチケットを作ってプラットフォームチームに依頼し、プラットフォームチームが Trusted Outsider や Internal Open Source の形で実装する。インフラ領域ではよく見られる。新しいサービスを立ち上げる、API ゲートウェイに外部エンドポイントを登録する、設定値を更新する、といった変更を、従来どおりチケットで依頼する習慣が残りやすいからである。

初期段階のプラットフォームでは、ホワイトグローブ型のオンボーディングも有効である。文書がまだ十分でないとき、プラットフォームチームが初期利用チームと横に並んで作業し、利用開始を支援する。これは初期採用の負担を下げるだけでなく、最初の顧客がプラットフォーム機能をどう使うかを学ぶ貴重な機会にもなる。

プラットフォームが成熟し、使いやすくなるにつれて、プラットフォームチームは手を動かして統合作業をする立場から、助言する立場へ移れる。オフィスアワーを設けて質問を受ける、計画や設計セッションで助言する、といった関わり方である。大規模で豊かなプラットフォームでは、利用者同士が助け合う Community of Practice に投資することもある。専用のチャネル、定例の会合、事例共有の場などがその例である。

ただし、手厚い支援はスケールしない。初期には、プラットフォームチーム自身が多くのエネルギーを使って消費を支えるのが合理的である。開発者体験はまだ粗く、機能も作りかけで、利用チームも試験台になることに慎重だからだ。さらに、横に並んで働くことは、プラットフォームチームが顧客を理解するよい方法でもある。

それでも、広い採用を目指すなら、プラットフォームチームは開発者体験に投資し、実装支援に溺れないようにしなければならない。また、利用者に対して、どのような支援モデルを期待すべきかを明確に伝える必要がある。初期にホワイトグローブ型の支援を受けたチームは、説明がなければ将来も同じ体験を期待する。

### プラットフォーム進化

プラットフォーム進化は、プラットフォームを使っているチームが、プラットフォーム自体に変更を必要とする場面である。UI コンポーネントライブラリを使っているチームが新しい種類の `<Button>` コンポーネントを必要とする、既存の `<Button>` に設定項目を追加したい、service chassis により詳しい観測性情報を出してほしい、新しいシリアライズ形式に対応してほしい、といった例がある。

ここでは、プラットフォーム移行とは役割が反対になる。作業を駆動しているのはプロダクトチームだが、変更が必要なのはプラットフォームチームのコードベースである。

単純な方法は、プロダクトチームがプラットフォームチームにチケットを作り、必要な変更を依頼することだ。しかしこれはしばしば大きな不満を生む。プロダクトチームは必要になった瞬間にプラットフォームの欠けに気づくことが多く、プラットフォームチームが優先順位をつけて実装するまでの時間が長すぎることがある。その結果、プラットフォームチームがボトルネックとなり、プロダクトチームの進捗を止めてしまう。

十分な予告がある場合は、必要なプラットフォーム改善に取り組むため、一時的にエンジニアを移すことができる。プロダクトエンジニアがプラットフォームチームで Tour of Duty を行うこともできるし、プラットフォームエンジニアが Embedded Expert としてプロダクトチームに入ることもできる。

エンジニアを移すと短期的には生産性に影響が出る。それでも、埋め込まれたエンジニアはチーム間の伝言ゲームを減らし、長期的には効率を高めることがある。この方法は、最初に固定的なコストがかかり、その後に継続的な利点が出るため、大きめのプラットフォーム改善に向いている。数週間だけ誰かを動かす程度では、助けになるより混乱のほうが大きいかもしれない。

こうした一時配置には、比較的成熟したマネジメント構造も必要である。埋め込まれたエンジニアが孤立しないようにするためだ。また、プラットフォームエンジニアがプロダクトチームに入る場合、本来必要だったプラットフォーム改善ではなく、単なる「追加の手」としてプラットフォーム消費作業を手伝うだけになってしまう危険もある。

プラットフォームチームが Internal Open Source を受け入れているなら、プロダクトチームが必要なプラットフォーム変更を自分たちで実装する選択肢もある。この場合、プラットフォームチームは主に相談役となり、設計上の助言をし、プロダクトチームからのプルリクエストをレビューする。何度か貢献を重ねれば、プロダクトエンジニアが十分な信頼を得て Trusted Outsider になることもある。

多くのプラットフォームチームはこの状態を望む。顧客が自分たちに必要な改善を自分で実装でき、プラットフォームチームがすべてを抱え込まずに済むからである。しかし、社内オープンソースの現実は一般のオープンソースと似ている。外部貢献を支えるには驚くほど多くの投資が必要であり、大多数の利用者が意味のある貢献者になるわけではない。

そのため、プラットフォームチームは、外部貢献を支えるための準備なしにコードベースを開かないほうがよい。コードベースは共有資源だと宣言したのに、実際には他チームからの貢献に対して「そうではない」と言い続けることになれば、双方に深い不満が残る。

### 結論

プラットフォーム移行、消費、進化を見ていくと、プラットフォームをめぐるチーム協働には多様な形があることがわかる。正しい形が一つだけあるわけではない。どの関わり方が最適かは、プラットフォーム採用の段階だけでなく、チーム間やシステム間のインターフェースがどれだけ成熟しているかにも依存する。

新しい内部プラットフォームを、成熟した外部サービスと同じように、手放しの as-a-service モードで統合できると期待するのは危険である。良い境界を設計するには、最初は焦点を絞った濃い協働が役に立つ。Embedded Expert や Tour of Duty のようなパターンを使い、システム間、チーム間の適切な境界とインターフェースを探ることができる。

ただし、その協働モードに長くとどまりすぎてはいけない。プラットフォームチームはインターフェースを定義し、File a Ticket や Internal Open Source のような、より as-a-service に近い形へすばやく移ることを目指すべきである。密な協働はプラットフォームチーム側でスケールしないだけでなく、利用するプロダクトチーム側にも大きな認知負荷をかける。

若いプラットフォームチームにとって、この密な協働から as-a-service への移行は大きな難題である。顧客は手厚い体験に慣れる。優れた文書を書くのは難しい。断ることも難しい。それでも、スケールの課題が見え始めたら、プラットフォームチームは関わり方がどう変わるのか、なぜ変わるのかを早めに伝え始めるべきである。

移行は痛みを伴うが、方針が揺れるとさらに悪くなる。プロダクトチームは、プラットフォーム提供者がどのように支援するのかについて、明確なルールを歓迎する。また、手厚い協働という支えを外すことは、セルフサービスのインターフェースや文書を改善する強い動機にもなる。

プラットフォームチームは他チームとの協働によって成功する。その協働には多くの形がある。どの形を選ぶかは、相手チームがどの種類のプラットフォーム作業をしているのか、そしてチームとシステムの現在地がどうなっているのかを現実的に見て判断する必要がある。**採用が広がるほど、プラットフォームチームはより手離れがよく、スケールし、利用者の認知負荷を下げる協働モードへ意図的に移る必要がある。**
