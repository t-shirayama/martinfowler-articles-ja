# Asset Capture

## 要約

Asset Capture は、Strangler Fig Application を進めるための戦略です。
旧システムが管理している資産の一部を選び、新システム側で扱い始めることで、段階的な移行を可能にします。

うまく進めるには、旧システムと新システムの間で資産を双方向に移せること、捕捉した資産に関するイベントを確実に新システムへ届けることが重要です。

## 読むときの観点

- ここでの asset は、従業員、取引、リース契約のように、アプリケーションが管理する中核的な対象として読む。
- 移行対象を「機能」だけでなく「資産の集合」で切り出す考え方に注目する。
- 逆方向の移行が、リスク低減と頻繁なリリースにどう効くかを見る。
- Event Interception や Content-Based Router との関係を押さえる。

## 原文の翻訳

Asset Capture は、Strangler Fig Application を開発するための戦略です。多くのアプリケーションは、重要な資産の集合を管理しているものだと考えられます。給与計算システムは従業員を扱い、取引システムは取引を扱い、リースシステムはリース契約を扱います。新しいシステムへ段階的に切り替えるには、まず新システムで扱い始める資産の一部を特定します。最初に扱うのに適しているのは、すぐに始められる単純な資産か、旧システムでは特に扱いにくいニーズを持つ資産であることが多いです。

これを滑らかに機能させるには、旧アプリケーションから strangler へ、そして strangler から旧アプリケーションへ、資産を移行する仕組みが必要です。逆方向の移行は不要に聞こえるかもしれません。しかし、リスクを下げるうえでも、資産が動的に変化して新アプリケーションでは扱えなくなる場合に対処するうえでも、とても効果的です。たとえば初期の給与計算システムが病気休暇中の従業員を扱えないとしましょう。**双方向の移行**があれば、従業員が病気休暇に入ったときに、その従業員を旧システムへ戻せます。

資産を管理するには、捕捉した資産に必要なイベントをすべて確実に取得するために、Event Interception を使う必要があります。すべてのイベントを横取りする必要はありません。捕捉した資産に関するイベントだけで十分です。また、旧システムがもはや管理していない資産のイベントを受け取らないようにする必要があるかもしれません。そのためには、Content-Based Router のようなものを使います。

私が関わった書き換えの多くは、何らかの形で asset capture を使っていました。唯一の後悔は、もっと細かい粒度で捕捉して、より頻繁なリリースを可能にしなかったことです。多くの場合、障害になっていたのは双方向の移行がなかったことでした。これが整うと、頻繁なリリースはずっと容易になります。
