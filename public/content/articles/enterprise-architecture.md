# Enterprise Architecture

## 要約

Enterprise Architecture は、単一アプリケーションの内部設計ではなく、企業内の複数アプリケーションをどのように一貫した全体へ組み合わせるかを扱います。
Fowler は、中央のアーキテクチャ部門が大きな構想を掲げても、各アプリケーションチームのビジネス価値と衝突して失敗しがちな循環を指摘しています。

この記事は、Enterprise Architecture の取り組みが成功するには、政治的現実を認識し、短期的なビジネス価値を積み重ね、既存アプリケーションへの負担を最小化する必要があると述べています。

## 読むときの観点

- Enterprise Application Architecture と Enterprise Architecture を区別する。
- 中央集権的な計画が、ビジネススポンサーを持つアプリケーションチームに負ける構造を見る。
- 「全体計画」よりも、低負担で疎結合な統合技術を重視する理由を考える。
- 個々のアプリケーションの良いレイヤリングが、企業全体の統合にも効いてくる点を押さえる。

## 原文の翻訳

最近、『P of EAA』について、Amazon でいくつか悪いレビューを見かけました。その理由は、その本には Enterprise Architecture について何も書かれていない、というものでした。もちろん、それには十分な理由があります。この本は Enterprise Application Architecture、つまり Enterprise Application をどう設計するかについての本だからです。Enterprise Architecture は別の話題であり、企業内の複数のアプリケーションを、どのように一貫した全体として組織するかを扱います。

実のところ、私は Enterprise Architecture についてかなり皮肉っぽくなることがあります。この皮肉は、Enterprise Architecture の取り組みによく見られるライフサイクルから来ています。たいていの場合、それは華々しく注目を集めながら始まります。IT 部門が大きな取り組みを立ち上げ、アプリケーションの孤島という煙突を打ち壊すことで得られる相乗効果、再利用、その他あらゆる利益をもたらす、というわけです。2、3年後には、あまり多くのことは成し遂げられておらず、Enterprise Architecture グループの電話には誰も折り返さなくなっています。そのさらに1、2年後に、その取り組みは静かに死にます。しかしすぐにまた別の取り組みが始まり、好況と破綻の循環が再び始まります。

では、なぜこの循環はこれほど規則的に起こるのでしょうか。私は、こうした取り組みに関わる多くの人は、失敗の主な理由を政治だと言うだろうと思います。しかし彼らが見落としがちなのは、その政治的な力が避けられないものだという点です。こうしたことに成功するには、まず **政治的な力の強さを認識する** 必要があります。

中央のアーキテクチャグループにとっての問題は、彼らが IT 管理層によって動かされている一方で、彼らが組織しようとしているアプリケーションはビジネス上のニーズによって動かされていることです。あるアプリケーションチームが、自分たちのアプリケーションには直接利益をもたらさないが、アーキテクチャに合わせやすくする作業を求められれば、それに気が進まないのは自然です。さらに彼らには切り札があります。ビジネススポンサーです。Enterprise Architecture の計画に従うためにアプリケーションの出荷が4か月遅れると言われれば、ビジネススポンサーは、アプリケーションチームが「後でやります」と言って断るときに、そのチームを後押しする動機を持ちます。アプリケーションはビジネス価値の提供に直接結びついており、中央アーキテクチャチームはそうではないため、アプリケーションチームが勝ちます。こうした勝利が、Enterprise Architecture の取り組みを破綻させます。

これを避けるには、Enterprise Architecture の取り組みは政治的現実を認識し、それに従わなければなりません。

- Enterprise Architecture の取り組みが持つビジネス価値を理解する。
- どの作業も、短期的で段階的なビジネス価値によって支えられているようにする。
- アプリケーションへのコストを最小化する。

これを考えるよい方法は、こうした取り組みを、アプリケーションの包括的な計画を作るものとしてではなく、アプリケーションがどのように組み合わされていても統合できる技術を見つけるものとして見ることです。結局のところ、Application Boundary は主に社会的構成物であり、誰かの将来計画に従うとは限りません。この統合アーキテクチャは、アプリケーションチームへの影響を最小限にして機能すべきです。そうすれば、チームはビジネス価値が正当化する範囲で、小さな機能片を提供できます。また、より密結合な方法のほうが効率的に見える場合であっても、アプリケーション間の結合を最小にするアプローチに集中する必要があると思います。

こうした理由から、私は統合にはメッセージングのアプローチへ向かいがちです。欠点はありますが、既存アプリケーションへの影響を最小限にして適用できるものだからです。

ちなみに、Enterprise Application Architecture は Enterprise Integration に大きな影響を与えることがあります。適切にレイヤ化されたアプリケーション、特によい Presentation Domain Separation を持つアプリケーションは、ずっとつなぎ合わせやすくなります。アプリケーションの機能をサービスとして公開しやすいからです。これはアプリケーションにとってのコストではありません。よいレイヤリングは、そのアプリケーション自体も保守しやすくするからです。しかし、Presentation Domain Separation をどう行うかを理解しているアプリケーション開発者は少なすぎます。統合グループができる最善のことのひとつは、開発者がそれをできるよう教育とトレーニングを支援することです。このアプローチは、彼らが Architectus Reloadus ではなく Architectus Oryzus のように振る舞うときに最もよく支えられます。その意味では、私の本は Enterprise Architecture と大いに関係があります。
