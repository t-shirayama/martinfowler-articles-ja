# Vibe Coding

## 要約

martinfowler.com の [Vibe Coding](https://www.martinfowler.com/bliki/VibeCoding.html) は、LLM に作りたいものを指示し、動かしてみて、さらに変更を頼むことでソフトウェアを作る方法についての短い記事です。特徴は、LLM が生成したコードを読まないことにあります。そのため、プログラミング経験がない人でも自分用の小さなアプリケーションを作れる可能性があります。

一方で、コードを見ないことは利点であると同時に限界でもあります。保守性、正しさ、セキュリティに問題を抱えやすく、広い利用者や重要なデータを扱うソフトウェアには向きません。Fowler は、vibe coding と agentic programming を区別し、後者は生成されたコードをレビューし構造にも注意を払う別の活動だと説明しています。

## 読むときの観点

- 「コードの存在を忘れる」ことが、vibe coding の定義の中心にある点を見る。
- LLM にすべてのコードを書かせることと、生成されたコードを無視することを分けて考える。
- 使い捨ての個人用ツールや試作品では有用でも、長く使うソフトウェアではリスクが大きくなる。
- セキュリティ、保守性、正しさのリスクは、コードを読まないほど見えにくくなる。
- 生成 AI の能力が上がっても、構造の良いソフトウェアの価値が消えるわけではない。

## 原文の翻訳

Vibe coding は、LLM に指示してソフトウェアアプリケーションを作るやり方です。何を作るかを伝え、動かして試し、変更を頼みます。ただし、LLM が生成したコードは見ません。この方法は、プログラミングの知識がない人にも使えます。しかし、できあがったソフトウェアは保守性、正しさ、セキュリティに問題を抱えることが多いため、限られた利用者向けの使い捨てソフトウェアに向いています。

この言葉は、2025年2月に Andrej Karpathy が X の投稿で使ったものです。彼は、LLM の能力が高くなったことで、コードの存在を忘れ、見たものを言い、実行し、コピー＆ペーストしながら、だいたい動くものを作るような開発を指していました。Fowler にとって、この言葉の核心は **コードの存在を忘れる** という点にあります。そこに有用性があり、同時に限界もあります。

November Inflection 以降、多くのプログラマーは LLM にすべてのコードを書かせるようになり、自分ではもう直接コードを書かないかもしれないと語っています。しかし、その人たちは生成されたコードを気にかけ、レビューし、内部構造にも注意を払います。その場合はコードの存在を忘れていないので、vibe coding ではなく、Fowler が agentic programming と呼ぶ別の活動です。残念ながら「vibe coding」という言葉は広く流行し、agentic programming の意味でも使われるようになりました。

それでも Fowler は、vibe coding と agentic programming を分けて考える価値があると述べます。両者は使い方も結果も違うからです。Vibe coder はコードを見ないため、プログラミング能力を必要としません。そのため、プログラミングを知らない人が自分のためにアプリケーションを作るには非常に適しています。経験豊かなプログラマーにとっても、使い捨てのソフトウェアや試作品をすばやく作る場面では便利です。

Vibe coding はまだ新しいため、その限界は探られている途中です。そして、その限界はモデルや周辺ツールの進歩によって変化します。ただし、vibe で作ったソフトウェアが広く使われたり、機密情報にアクセスしたりする場合には、大きなリスクが生じます。

最も深刻なリスクはセキュリティです。LLM は本質的に大きな攻撃面を持ち、攻撃者に狙われやすい性質があります。Vibe coding で作られたアプリケーションは、機密情報を漏らしたり、さらに組織内の深いシステムへ攻撃するための credential を露出させたりすることがあります。プログラマーでない人であっても、Lethal Trifecta の危険を知っておく必要があります。

コードにほとんど注意を払わないと、vibe で作られたソフトウェアは低品質なコードを大量に生み出しやすくなります。そのようなコードは、将来の変更や拡張を難しくします。将来の LLM が巨大なスパゲッティコードでも扱えるようになる可能性はありますが、現時点では、よく構造化されたソフトウェアのほうが LLM にとっても扱いやすいことは明らかです。

LLM は、誤った事実を自信ありげに提示する hallucination で知られています。同じ性質はソフトウェアにも現れ、正しく振る舞わないプログラムを作ることがあります。しかも、その誤りは利用者に見えないかもしれません。さらに LLM の非決定性により、ある機能追加を頼んだだけで、変更する必要のない部分にまで誤りが入り込む可能性があります。

そのため、LLM が生成したソフトウェアには懐疑的に向き合うべきです。有用であることは確かですが、リスクを理解する必要があります。全体として、vibe coding で作ったソフトウェアは、作者本人か、リスクを理解し受け入れる近い協力者だけが使う、使い捨てのものに最も向いています。より複雑で、より広く使われ、失敗の影響が大きいコードについては、コードの存在を忘れてはいけません。
