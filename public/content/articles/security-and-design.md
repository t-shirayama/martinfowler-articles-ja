# Security And Design

## 要約

Security And Designは、セキュリティを後付けの機能ではなく、設計の性質として扱うべきだという短い記事です。システムを作った後で「安全にする」のではなく、境界、責務、データの扱い、アクセス制御を設計の中で考える必要があります。

セキュリティは、単独の部品やチェックリストだけで保証されません。複雑な設計、曖昧な責務、広すぎる権限、理解しづらいデータフローは、脆弱性を見つけにくくし、修正もしづらくします。

## 読むときの観点

- セキュリティを、機能要件ではなく設計品質として読む。
- 境界、責務、権限、単純さが安全性にどう影響するかを見る。
- 「あとでセキュリティを足す」発想の危うさを考える。
- 設計レビューと脅威の見方を結びつける。

## 原文の翻訳

セキュリティは、ソフトウェア設計における重要な関心事です。しかししばしば、機能を作り終えた後に追加するものとして扱われます。この見方は危険です。安全なシステムを作るには、設計の早い段階からセキュリティを考える必要があります。

優れた設計は、システムを理解しやすくします。何がどこで起きるのか、どのデータがどの境界を越えるのか、誰が何をできるのかが見えやすくなります。これはセキュリティにも直接効きます。理解しにくいシステムでは、攻撃可能な経路も、守るべき境界も、見落とされやすくなるからです。

セキュリティのためには、境界を明確にすることが大切です。外部から来る入力、内部サービス間の通信、永続化されたデータ、利用者の権限は、それぞれ異なる信頼の前提を持ちます。これらを曖昧に混ぜると、検証や権限チェックが抜け落ちやすくなります。

単純さも重要です。複雑な設計は、セキュリティ上の判断を難しくします。誰がどの状態を変更できるのか、どの値が信頼できるのか、どの経路でデータが外へ出るのかを説明できなければ、安全性を評価することも難しくなります。**理解できない設計は守りにくい**のです。

また、権限は必要最小限に保つべきです。広すぎる権限は、ひとつの欠陥が大きな侵害へ広がる原因になります。設計の中で責務を分け、アクセスできる範囲を限定し、失敗したときの影響を小さくすることが大切です。

セキュリティを設計に組み込むとは、すべてを完璧に予測するという意味ではありません。むしろ、システムの境界と前提を明確にし、脅威を考えやすくし、変更に合わせて安全性を見直せる構造を作ることです。セキュリティは最後に塗る保護膜ではなく、設計を形づくる制約のひとつです。
