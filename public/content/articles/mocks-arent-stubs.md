# Mocks Aren't Stubs

## 要約

「Mock Object」は、テストで本物のオブジェクトをまねる特別なオブジェクトとしてよく使われる言葉です。
しかし Martin Fowler は、モックはテストダブルの一種にすぎず、スタブとは目的も検証の仕方も異なると説明します。

この記事は、状態検証と振る舞い検証、クラシック派 TDD とモック派 TDD という2つの違いを整理します。
モックを使うかどうかは単なるツール選択ではなく、テストの粒度、設計の進め方、実装との結合、オブジェクト設計のスタイルに関わる判断です。

## 読むときの観点

- Dummy、Fake、Stub、Spy、Mock というテストダブルの役割を区別する。
- 状態検証と振る舞い検証の違いを押さえる。
- クラシック派 TDD とモック派 TDD が、設計の進め方にどう影響するかを見る。
- モックがテストを実装詳細へ結合させる利点と危うさを読む。

## 原文の翻訳

「Mock Object」という言葉は、テストのために本物のオブジェクトをまねる特殊なオブジェクトを表す言葉として広く使われるようになりました。いまでは多くの言語環境に、モックオブジェクトを簡単に作れるフレームワークがあります。しかし、しばしば見落とされているのは、モックオブジェクトは特殊なテスト用オブジェクトの一形態にすぎず、**別のテストスタイルを可能にするもの**だという点です。

この記事では、モックオブジェクトがどのように機能するのか、どのように振る舞いの検証に基づくテストを促すのか、そしてそれを取り巻くコミュニティがどのように別のテストスタイルを発展させてきたのかを説明します。

私は数年前、Extreme Programming (XP) のコミュニティで「mock object」という言葉に初めて出会いました。それ以来、モックオブジェクトに触れる機会はどんどん増えています。その理由の一部は、モックオブジェクトの主要な開発者の多くが、さまざまな時期に Thoughtworks で私の同僚だったからです。また別の理由として、XP の影響を受けたテスト関連の文献で、モックオブジェクトを目にする機会が増えていることもあります。

しかし、モックオブジェクトはかなり頻繁に、うまく説明されていない形で見かけます。特に、テスト環境でよく使われる補助であるスタブと混同されていることがよくあります。この混乱は理解できます。私自身もしばらくは似たものだと思っていました。しかしモックの開発者たちとの会話を通じて、少しずつではありますが、私の硬い頭蓋にもモックへの理解がしみ込んできました。

この違いは、実際には2つの別々の違いです。一方には、テスト結果をどのように検証するかという違いがあります。つまり、状態検証と振る舞い検証の区別です。もう一方には、テストと設計がどのように関係し合うかについてのまったく異なる哲学があります。この記事ではそれを、Test Driven Development のクラシック派スタイルとモック派スタイルと呼びます。

### Regular Tests

まず、単純な例を使って2つのスタイルを説明します。この例は Java ですが、原則はどのオブジェクト指向言語にも当てはまります。注文オブジェクトを取り、倉庫オブジェクトからその注文を満たしたいとします。注文はとても単純で、1つの商品と数量だけを持ちます。倉庫はさまざまな商品の在庫を持っています。注文に対して、倉庫から自分自身を満たすよう依頼すると、2つの反応があり得ます。

倉庫に注文を満たすだけの商品があれば、注文は満たされた状態になり、倉庫内のその商品の数量は適切な分だけ減ります。倉庫に十分な商品がなければ、注文は満たされず、倉庫では何も起きません。

この2つの振る舞いから、いくつかのテストが導かれます。かなり典型的な JUnit テストに見えます。

```java
public class OrderStateTester extends TestCase {
  private static String TALISKER = "Talisker";
  private static String HIGHLAND_PARK = "Highland Park";
  private Warehouse warehouse = new WarehouseImpl();
  protected void setUp() throws Exception {
    warehouse.add(TALISKER, 50);
    warehouse.add(HIGHLAND_PARK, 25);
  }
  public void testOrderIsFilledIfEnoughInWarehouse() {
    Order order = new Order(TALISKER, 50);
    order.fill(warehouse);
    assertTrue(order.isFilled());
    assertEquals(0, warehouse.getInventory(TALISKER));
  }
  public void testOrderDoesNotRemoveIfNotEnough() {
    Order order = new Order(TALISKER, 51);
    order.fill(warehouse);
    assertFalse(order.isFilled());
    assertEquals(50, warehouse.getInventory(TALISKER));
  }
```

xUnit のテストは、典型的には4つの段階をたどります。セットアップ、実行、検証、後片付けです。この場合、セットアップ段階は一部が `setUp` メソッドで行われ、倉庫が準備されます。残りはテストメソッドで行われ、注文が準備されます。`order.fill` の呼び出しが実行段階です。ここで、テストしたいことをオブジェクトに行わせます。その後の assert 文が検証段階であり、実行されたメソッドが正しく仕事をしたかを確認します。

この場合、明示的な後片付け段階はありません。ガベージコレクタが暗黙にそれを行ってくれます。

セットアップ中には、2種類のオブジェクトを組み合わせています。Order はテスト対象のクラスですが、`Order.fill` が動作するには Warehouse のインスタンスも必要です。この状況では、Order がテストの焦点となるオブジェクトです。テストに関心のある人たちは、このようなものを object-under-test や system-under-test と呼びたがります。どちらの言葉も口に出すには不格好ですが、広く受け入れられている用語なので、鼻をつまんで使うことにします。

Meszaros にならい、私は System Under Test、より正確にはその略語である SUT を使います。

したがって、このテストには SUT (`Order`) と1つの協調者 (`warehouse`) が必要です。倉庫が必要な理由は2つあります。1つは、テスト対象の振る舞いをそもそも動かすためです。`Order.fill` が倉庫のメソッドを呼ぶからです。もう1つは検証のためです。`Order.fill` の結果の1つとして、倉庫の状態が変わる可能性があるからです。この話題をさらに見ていくと、SUT と協調者の区別を何度も行うことになります。

この記事の以前の版では、SUT を「primary object」、協調者を「secondary object」と呼んでいました。

このテストスタイルは状態検証を使います。つまり、実行されたメソッドが正しく動いたかどうかを、メソッドの実行後に SUT とその協調者の状態を調べることで判断します。これから見るように、モックオブジェクトは**別の検証アプローチ**を可能にします。

### Tests with Mock Objects

今度は、同じ振る舞いをモックオブジェクトで扱います。このコードでは、モックを定義するために jMock ライブラリを使っています。jMock は Java のモックオブジェクトライブラリです。他にもモックオブジェクトライブラリはありますが、これはこの技法の創始者たちが書いた比較的新しいライブラリなので、出発点としてよいものです。

```java
public class OrderInteractionTester extends MockObjectTestCase {
  private static String TALISKER = "Talisker";
  public void testFillingRemovesInventoryIfInStock() {
    // セットアップ - データ
    Order order = new Order(TALISKER, 50);
    Mock warehouseMock = new Mock(Warehouse.class);

    // セットアップ - 期待
    warehouseMock.expects(once()).method("hasInventory")
      .with(eq(TALISKER),eq(50))
      .will(returnValue(true));
    warehouseMock.expects(once()).method("remove")
      .with(eq(TALISKER), eq(50))
      .after("hasInventory");
    // 実行
    order.fill((Warehouse) warehouseMock.proxy());

    // 検証
    warehouseMock.verify();
    assertTrue(order.isFilled());
  }

  public void testFillingDoesNotRemoveIfNotEnoughInStock() {
    Order order = new Order(TALISKER, 51);
    Mock warehouse = mock(Warehouse.class);

    warehouse.expects(once()).method("hasInventory")
      .withAnyArguments()
      .will(returnValue(false));
    order.fill((Warehouse) warehouse.proxy());

    assertFalse(order.isFilled());
  }
```

まず `testFillingRemovesInventoryIfInStock` に注目してください。後のテストでは、いくつか省略した書き方をしています。

最初に、セットアップ段階が大きく違います。まず、データと期待の2つの部分に分かれています。データの部分では、扱いたいオブジェクトを準備します。その意味では従来のセットアップに似ています。違いは、作られるオブジェクトにあります。SUT は同じで、注文です。しかし協調者は倉庫オブジェクトではありません。代わりにモック倉庫であり、技術的には `Mock` クラスのインスタンスです。

セットアップの2つ目の部分では、モックオブジェクトに期待を作ります。期待は、SUT が実行されたときにモック上でどのメソッドが呼ばれるべきかを示します。

すべての期待を置いたら、SUT を実行します。実行後には検証を行います。検証には2つの側面があります。以前と同じように SUT に対して assert を実行します。しかしそれに加えて、モックも検証します。つまり、期待どおりに呼ばれたかを確認するのです。

ここでの重要な違いは、注文が倉庫との相互作用で正しいことをしたかどうかを、どのように検証するかです。状態検証では、倉庫の状態に対して assert を行います。モックは振る舞い検証を使い、代わりに、注文が倉庫に対して正しい呼び出しをしたかどうかを確認します。この確認は、セットアップ時にモックへ期待を伝え、検証時にモック自身へ確認させることで行います。

assert で確認されるのは注文だけです。メソッドが注文の状態を変えない場合、assert はまったく存在しないこともあります。

2つ目のテストでは、いくつか違うことをしています。まず、モックの作り方が違います。コンストラクタではなく、`MockObjectTestCase` の `mock` メソッドを使っています。これは jMock ライブラリの便利メソッドで、後で明示的に verify を呼ぶ必要がなくなります。この便利メソッドで作られたモックは、テストの最後に自動的に検証されます。

最初のテストでもこれを使えましたが、モックを使ったテストがどのように機能するかを示すために、検証をより明示的に見せたかったのです。

2つ目のテストケースでのもう1つの違いは、`withAnyArguments` を使って期待の制約を緩めていることです。その理由は、最初のテストが数量が倉庫に渡されることを確認しているため、2つ目のテストでその要素を繰り返す必要がないからです。後で注文のロジックを変える必要が出ても、失敗するテストは1つだけになり、テストの移行作業が楽になります。実際には、`withAnyArguments` はデフォルトなので、完全に省略することもできました。

#### Using EasyMock

モックオブジェクトライブラリはいくつもあります。私がかなりよく見かけるものの1つが EasyMock で、Java 版と .NET 版の両方があります。EasyMock も振る舞い検証を可能にしますが、jMock とはスタイルにいくつか違いがあり、議論する価値があります。おなじみのテストをもう一度示します。

```java
public class OrderEasyTester extends TestCase {
  private static String TALISKER = "Talisker";

  private MockControl warehouseControl;
  private Warehouse warehouseMock;
  public void setUp() {
    warehouseControl = MockControl.createControl(Warehouse.class);
    warehouseMock = (Warehouse) warehouseControl.getMock();
  }

  public void testFillingRemovesInventoryIfInStock() {
    // セットアップ - データ
    Order order = new Order(TALISKER, 50);
    // セットアップ - 期待
    warehouseMock.hasInventory(TALISKER, 50);
    warehouseControl.setReturnValue(true);
    warehouseMock.remove(TALISKER, 50);
    warehouseControl.replay();

    // 実行
    order.fill(warehouseMock);

    // 検証
    warehouseControl.verify();
    assertTrue(order.isFilled());
  }

  public void testFillingDoesNotRemoveIfNotEnoughInStock() {
    Order order = new Order(TALISKER, 51);
    warehouseMock.hasInventory(TALISKER, 51);
    warehouseControl.setReturnValue(false);
    warehouseControl.replay();

    order.fill((Warehouse) warehouseMock);

    assertFalse(order.isFilled());
    warehouseControl.verify();
  }
}
```

EasyMock は、期待を設定するために record/replay という比喩を使います。モックしたいオブジェクトごとに、control と mock object を作ります。モックは二次オブジェクトのインターフェイスを満たし、control は追加の機能を提供します。期待を示すには、期待する引数でモック上のメソッドを呼び出します。戻り値が必要な場合は、その後に control への呼び出しを続けます。

期待の設定を終えたら、control で replay を呼びます。この時点でモックは記録を終え、primary object に応答する準備ができます。完了したら control で verify を呼びます。

record/replay という比喩を最初に見たとき、人々はしばしば戸惑うようですが、すぐに慣れるようです。jMock の制約指定に対する利点は、文字列でメソッド名を指定するのではなく、モックに対して実際のメソッド呼び出しを行う点にあります。つまり、IDE のコード補完を使え、メソッド名のリファクタリングも自動的にテストを更新してくれます。欠点は、より緩い制約を持てないことです。

jMock の開発者たちは、実際のメソッド呼び出しを使えるようにするため、別の技法を使った新しい版に取り組んでいます。

### The Difference Between Mocks and Stubs

モックオブジェクトが最初に導入されたとき、多くの人は、テストでスタブを使うという一般的な考え方とモックオブジェクトを簡単に混同しました。それ以来、人々はその違いをよりよく理解するようになったようです。この論文の以前の版がその助けになったことを願っています。しかし、人々がモックをどのように使うかを十分に理解するには、モックと他の種類のテストダブルを理解することが重要です。「ダブル」という言葉が初めてでも心配いりません。数段落待てば、すべてはっきりします。

このようなテストを行うとき、私たちはソフトウェアの1つの要素に一度に焦点を当てています。だから unit testing という一般的な用語があるのです。問題は、1つの unit を動作させるには、しばしば他の unit が必要になることです。先ほどの例で何らかの倉庫が必要だったのはそのためです。

上で示した2つのテストスタイルでは、最初のケースは本物の倉庫オブジェクトを使い、2つ目のケースはモック倉庫を使いました。もちろんモック倉庫は本物の倉庫オブジェクトではありません。モックを使うことは、テストで本物の倉庫を使わない1つの方法ですが、このようなテストで使われる本物ではないオブジェクトには、他にもいくつかの形があります。

これについて話す語彙は、すぐに混乱してきます。stub、mock、fake、dummy など、さまざまな言葉が使われます。この記事では、Gerard Meszaros の本の語彙に従うことにします。すべての人が使う語彙ではありませんが、よい語彙だと思いますし、これは私のエッセイなので、どの言葉を使うかは私が選べます。

Meszaros は、テスト目的で本物のオブジェクトの代わりに使われるあらゆる種類の見せかけのオブジェクトを表す総称として Test Double という用語を使います。この名前は、映画の Stunt Double という考え方から来ています。彼の狙いの1つは、すでに広く使われている名前を避けることでした。Meszaros はそのうえで、5つの特定の double を定義しました。

- Dummy オブジェクトは受け渡されるだけで、実際には使われません。通常は、引数リストを埋めるためだけに使われます。
- Fake オブジェクトは実際に動く実装を持ちますが、たいていは何らかの近道をしているため、本番には適しません。in-memory database はよい例です。
- Stub は、テスト中に行われる呼び出しに対して決められた答えを返します。通常、テスト用にプログラムされた範囲外のものにはまったく応答しません。
- Spy は、呼び出され方に基づいて何らかの情報も記録する stub です。たとえば、送信されたメッセージ数を記録するメールサービスがその一形態です。
- Mock はここで話しているものです。受け取ると期待される呼び出しの仕様を形作る期待で、あらかじめプログラムされたオブジェクトです。

これらの double のうち、振る舞い検証を要求するのは mock だけです。他の double は状態検証を使うことができ、通常はそうします。実行段階では、mock も実際には他の double と同じように振る舞います。SUT に対して、本物の協調者と話していると思わせる必要があるからです。しかし mock は、セットアップ段階と検証段階で異なります。

テストダブルをもう少し掘り下げるために、例を拡張する必要があります。多くの人は、本物のオブジェクトが扱いにくい場合にだけテストダブルを使います。テストダブルがより一般的に登場するのは、注文を満たせなかった場合にメールメッセージを送信したい、というような場合でしょう。問題は、テスト中に顧客へ実際のメールメッセージを送りたくないことです。そこで、メールシステムのテストダブルを作り、自分たちで制御し操作できるようにします。

ここで、モックとスタブの違いが見え始めます。このメール送信の振る舞いに対するテストを書くなら、次のような単純なスタブを書くかもしれません。

```java
public interface MailService {
  public void send (Message msg);
}
public class MailServiceStub implements MailService {
  private List<Message> messages = new ArrayList<Message>();
  public void send (Message msg) {
    messages.add(msg);
  }
  public int numberSent() {
    return messages.size();
  }
}
```

すると、このようにスタブに対して状態検証を使えます。

```java
class OrderStateTester...
      public void testOrderSendsMailIfUnfilled() {
        Order order = new Order(TALISKER, 51);
        MailServiceStub mailer = new MailServiceStub();
        order.setMailer(mailer);
        order.fill(warehouse);
        assertEquals(1, mailer.numberSent());
      }
```

もちろん、これはとても単純なテストです。メッセージが1通送られたことだけを確認しています。正しい相手に送られたか、正しい内容だったかはテストしていません。しかし、要点を説明するには十分です。

モックを使うと、このテストはかなり違って見えます。

```java
class OrderInteractionTester...

      public void testOrderSendsMailIfUnfilled() {
        Order order = new Order(TALISKER, 51);
        Mock warehouse = mock(Warehouse.class);
        Mock mailer = mock(MailService.class);
        order.setMailer((MailService) mailer.proxy());

        mailer.expects(once()).method("send");
        warehouse.expects(once()).method("hasInventory")
          .withAnyArguments()
          .will(returnValue(false));
        order.fill((Warehouse) warehouse.proxy());
      }
    }
```

どちらの場合も、本物のメールサービスの代わりにテストダブルを使っています。違いは、スタブが状態検証を使い、モックが振る舞い検証を使う点です。

スタブで状態検証を使うには、検証を助けるためにスタブへいくつか追加メソッドを作る必要があります。その結果、スタブは `MailService` を実装しつつ、テスト用の追加メソッドも持つことになります。

モックオブジェクトは常に振る舞い検証を使いますが、スタブはどちらにも進めます。Meszaros は、振る舞い検証を使うスタブを Test Spy と呼んでいます。違いは、double がどのように実行され、どのように検証されるかという細部にあります。そこは、あなた自身でさらに探ってみてください。

### Classical and Mockist Testing

これで、2つ目の二分法を検討できるところまで来ました。クラシック派 TDD とモック派 TDD の違いです。ここでの大きな問題は、いつモック、または他の double を使うかです。

クラシックな TDD のスタイルでは、可能なら本物のオブジェクトを使い、本物を使うのが扱いにくい場合に double を使います。したがってクラシック派の TDD 実践者なら、本物の倉庫を使い、メールサービスには double を使うでしょう。double の種類は、それほど重要ではありません。

一方、モック派 TDD の実践者は、興味深い振る舞いを持つ任意のオブジェクトに対して、常にモックを使います。この場合なら、倉庫とメールサービスの両方に対してです。

さまざまなモックフレームワークは、モック派テストを念頭に置いて設計されていますが、多くのクラシック派の人たちも、double を作るためにそれらが便利だと感じています。

モック派スタイルの重要な派生として、Behavior Driven Development (BDD) があります。BDD はもともと、私の同僚である Daniel Terhorst-North が、TDD を学ぶ人をよりよく助ける技法として開発しました。特に、TDD が設計技法としてどのように働くかに焦点を当てるためです。そこから、テストを behavior と呼び替えることにつながりました。オブジェクトが何をする必要があるかを考える際に、TDD がどこで役立つのかをよりよく探るためです。

BDD はモック派のアプローチを取りますが、命名スタイルと、分析を技法に統合したいという志向の両面で、それを拡張しています。ここではこれ以上踏み込みません。この記事に関係するのは、BDD が TDD のもう1つの変種であり、モック派テストを使う傾向があるという点だけだからです。詳しく知りたい場合は、リンクをたどってください。

「classical」を「Detroit」スタイル、「mockist」を「London」スタイルと呼ぶのを見かけることもあります。これは、XP がもともと Detroit の C3 プロジェクトで開発され、モック派スタイルが London の初期 XP 採用者たちによって開発されたという事実を示唆しています。また、多くのモック派 TDD 実践者がこの用語を好まず、実際にはクラシック派テストとモック派テストの間に異なるスタイルがあることを示唆するあらゆる用語を嫌う、ということも言っておくべきでしょう。彼らは、2つのスタイルを区別することに有用性があるとは考えていません。

### Choosing Between the Differences

この記事では、2組の違いを説明してきました。状態検証か振る舞い検証か、クラシック派 TDD かモック派 TDD かです。それらを選ぶとき、どんな議論を念頭に置くべきでしょうか。まずは、状態検証と振る舞い検証の選択から始めます。

最初に考えるべきなのは文脈です。注文と倉庫のような扱いやすい協調を考えているのでしょうか。それとも、注文とメールサービスのような扱いにくい協調でしょうか。

扱いやすい協調なら、選択は単純です。私がクラシック派 TDD 実践者なら、モックもスタブも、どんな double も使いません。本物のオブジェクトと状態検証を使います。私がモック派 TDD 実践者なら、モックと振る舞い検証を使います。決めることは何もありません。

扱いにくい協調なら、私がモック派であればやはり決めることはありません。ただモックと振る舞い検証を使います。私がクラシック派なら選択はありますが、どちらを使うかは大きな問題ではありません。通常、クラシック派はケースごとに、一番楽な道を選びます。

このように、状態検証か振る舞い検証かは、たいてい大きな判断ではありません。本当の論点は、クラシック派 TDD とモック派 TDD の間にあります。とはいえ、状態検証と振る舞い検証の特徴はその議論に影響します。そこで、これからの多くの労力はそこに向けます。

その前に、境界的なケースを1つ投げ込ませてください。扱いにくい協調ではないにもかかわらず、状態検証を使うのが本当に難しいものに、ときどき出くわします。よい例がキャッシュです。キャッシュの要点は、キャッシュヒットだったのかミスだったのかを、その状態からは判別できないことにあります。これは、筋金入りのクラシック派 TDD 実践者にとっても、振る舞い検証を選ぶのが賢明なケースです。逆方向も含め、他にも例外はあるはずです。

クラシック派とモック派の選択を掘り下げるには、考えるべき要素がたくさんあります。そこで、おおまかなグループに分けてみます。

#### Driving TDD

モックオブジェクトは XP コミュニティから生まれました。そして XP の主要な特徴の1つは Test Driven Development を重視することです。TDD では、テストを書くことに駆動される反復によって、システム設計を進化させます。

したがって、モック派の人たちが、モック派テストが設計に与える影響について特に語るのは驚くことではありません。彼らは特に、need-driven development と呼ばれるスタイルを支持します。このスタイルでは、user story の開発を、システムの外側に対する最初のテストを書くことから始め、何らかのインターフェイスオブジェクトを SUT にします。協調者に対する期待を考えることで、SUT とその隣接者の間の相互作用を探ります。実質的には、SUT の外向きインターフェイスを設計しているのです。

最初のテストが動くようになると、モックに対する期待が次のステップの仕様となり、テストの出発点になります。各期待を協調者に対するテストへ変換し、SUT を1つずつシステムの内部へ進みながら、このプロセスを繰り返します。このスタイルは outside-in とも呼ばれます。とても説明的な名前です。これはレイヤー化されたシステムでうまく機能します。まず、下にあるレイヤーをモックにして UI をプログラムするところから始めます。

次に下位レイヤーのテストを書き、レイヤーを1つずつ進みながら、徐々にシステムをたどります。これは非常に構造化され、制御されたアプローチであり、OO と TDD の新参者を導くうえで役立つと多くの人が考えています。

クラシック派 TDD は、まったく同じ種類の指針を提供するわけではありません。モックの代わりにスタブ化されたメソッドを使って、似たような段階的アプローチを取ることはできます。そのためには、協調者から何かが必要になったとき、SUT が動くためにテストが必要とする応答を、そのままハードコードします。そしてそれで green になったら、そのハードコードされた応答を適切なコードに置き換えます。

しかしクラシック派 TDD では、他のこともできます。一般的なスタイルの1つは middle-out です。このスタイルでは、機能を1つ取り、その機能が動作するためにドメインで何が必要かを決めます。ドメインオブジェクトに必要なことをさせ、それらが動くようになったら、その上に UI を重ねます。このやり方では、何かを fake する必要がまったくないかもしれません。多くの人がこれを好むのは、まずドメインモデルへ注意を向けられるからです。それにより、ドメインロジックが UI に漏れ出すのを防ぎやすくなります。

ここで強調しておきたいのは、モック派もクラシック派も、これを1つの story ずつ行うということです。あるレイヤーが完成するまで次のレイヤーを始めない、というように、アプリケーションをレイヤーごとに構築する考え方もあります。しかしクラシック派もモック派もアジャイルの背景を持つ傾向があり、細かい反復を好みます。その結果、レイヤーごとではなく、機能ごとに作業します。

#### Fixture Setup

クラシック派 TDD では、SUT だけでなく、テストに応じて SUT が必要とするすべての協調者も作らなければなりません。例ではオブジェクトが数個だけでしたが、実際のテストでは多くの二次オブジェクトが関わることがよくあります。通常、これらのオブジェクトはテストの実行ごとに作られ、破棄されます。

一方、モック派のテストで作る必要があるのは、SUT と、その直接の隣接者に対するモックだけです。これにより、複雑な fixture を組み立てるための込み入った作業を避けられる可能性があります。少なくとも理論上はそうです。かなり複雑なモックセットアップの話を耳にしたことはありますが、それはツールをうまく使えていないためかもしれません。

実際には、クラシック派のテスターは複雑な fixture をできるだけ再利用する傾向があります。最も単純な方法は、fixture セットアップコードを xUnit の setup メソッドに置くことです。より複雑な fixture は複数のテストクラスで使う必要があるため、この場合は特別な fixture 生成クラスを作ります。私は通常これを Object Mothers と呼んでいます。これは、初期の Thoughtworks XP プロジェクトで使われた命名規約に基づいています。

大規模なクラシック派テストでは mothers を使うことが不可欠です。しかし mothers は保守すべき追加コードであり、mothers への変更はテスト全体に大きな波及効果をもたらす可能性があります。また、fixture のセットアップには性能上のコストがあるかもしれません。もっとも、適切に行われている場合にこれが深刻な問題だと聞いたことはありません。ほとんどの fixture オブジェクトは作成コストが低く、そうでないものは通常 double にされます。

その結果、私は両方のスタイルが互いに「手間がかかりすぎる」と非難するのを聞いてきました。モック派は fixture を作るのに多くの労力がかかると言います。一方クラシック派は、それは再利用されるが、モックは各テストで作らなければならないと言います。

#### Test Isolation

モック派テストのシステムにバグを入れると、通常は、そのバグを含む SUT のテストだけが失敗します。しかしクラシックなアプローチでは、クライアントオブジェクトのテストも失敗する可能性があります。その結果、バグを持つオブジェクトが別のオブジェクトのテストで協調者として使われる場所でも失敗が起きます。よく使われるオブジェクトの失敗は、システム全体に失敗テストの波及を引き起こします。

モック派のテスターは、これを大きな問題だと考えます。エラーの根本原因を見つけて修正するために、多くのデバッグが必要になるからです。しかしクラシック派は、これを問題の源としては表現しません。通常、失敗しているテストを見れば原因は比較的見つけやすく、開発者は他の失敗が根本的な障害から派生したものだと判断できます。

さらに、定期的にテストしているなら、そうすべきですが、壊れた原因が最後に編集したものだとわかります。したがって、障害を見つけるのは難しくありません。

ここで重要になり得る要素の1つは、テストの粒度です。クラシック派のテストは複数の本物のオブジェクトを実行するため、1つのテストが1つのオブジェクトだけではなく、オブジェクト群に対する主要なテストになっていることがよくあります。その群が多くのオブジェクトにまたがると、バグの本当の発生源を見つけるのがかなり難しくなります。ここで起きているのは、テストが粗すぎるということです。

モック派のテストは、この問題に悩まされる可能性が低いでしょう。なぜなら、primary を越えるすべてのオブジェクトをモック化する慣習があり、それによって協調者にはより細かい粒度のテストが必要だと明確になるからです。とはいえ、過度に粗い粒度のテストを使うことは、必ずしもクラシック派テストという技法の失敗ではありません。むしろ、クラシック派テストを適切に行っていないことの失敗です。よい経験則は、各クラスに対して細かい粒度のテストを分けて用意することです。

クラスタが妥当なこともありますが、それはごく少数のオブジェクト、せいぜい半ダース以下に限るべきです。さらに、過度に粗い粒度のテストのためにデバッグ問題に直面しているなら、テスト駆動でデバッグすべきです。進めながら、より細かい粒度のテストを作っていきます。

本質的に、クラシックな xUnit テストは unit test であるだけでなく、mini-integration test でもあります。その結果、多くの人は、クライアントのテストが、あるオブジェクトの主要なテストでは見逃されたかもしれないエラーを捕まえることがある点を好みます。特に、クラス同士が相互作用する領域を探る場合です。モック派のテストではこの性質が失われます。加えて、モック派テストの期待が誤っているリスクもあります。その場合、ユニットテストは green でありながら、内在するエラーを覆い隠してしまいます。

ここで強調しておくべきなのは、どちらのテストスタイルを使うとしても、システム全体にまたがって動作する、より粗い粒度の acceptance test と組み合わせなければならないということです。私は、acceptance test の導入が遅れ、それを後悔したプロジェクトに何度も出会ってきました。

#### Coupling Tests to Implementations

モック派のテストを書くとき、SUT が供給者と適切に話していることを確かめるために、SUT の外向きの呼び出しをテストしています。クラシックなテストが気にするのは最終状態だけであり、その状態がどのように導かれたかではありません。したがってモック派のテストは、メソッドの実装により強く結合します。協調者への呼び出しの性質を変えると、通常、モック派のテストは壊れます。

この結合はいくつかの懸念につながります。最も重要なのは、Test Driven Development への影響です。モック派テストでは、テストを書くことで振る舞いの実装について考えるようになります。実際、モック派のテスターはこれを利点と見なします。しかしクラシック派は、外部インターフェイスから見て何が起きるかだけを考え、実装についてのあらゆる考慮はテストを書き終えるまで残しておくことが重要だと考えます。

実装への結合はリファクタリングも妨げます。クラシック派テストに比べて、実装変更がテストを壊す可能性がずっと高いからです。

これは、モックツールキットの性質によって悪化することがあります。多くの場合、モックツールは、この特定のテストには関係がない場合でも、非常に具体的なメソッド呼び出しとパラメータ一致を指定します。jMock ツールキットの狙いの1つは、期待の指定をより柔軟にして、重要でない領域では期待を緩くできるようにすることです。ただし、その代償として文字列を使うため、リファクタリングが少し難しくなることがあります。

#### Design Style

これらのテストスタイルで私にとって最も興味深い側面の1つは、それらが設計判断にどう影響するかです。両方のタイプのテスターと話す中で、それぞれのスタイルが促す設計の違いにいくつか気づくようになりました。ただし、私はまだ表面を少しなでただけだと思っています。

レイヤーへの取り組み方の違いについてはすでに述べました。モック派テストは outside-in のアプローチを支えます。一方、domain model out を好む開発者は、クラシック派テストを好む傾向があります。

より小さなレベルでは、モック派のテスターは、値を返すメソッドから少し離れ、収集オブジェクトに対して作用するメソッドを好む傾向があることに気づきました。複数のオブジェクトから情報を集めてレポート文字列を作る振る舞いの例を考えてみましょう。よくある方法は、レポート作成メソッドがさまざまなオブジェクト上の文字列を返すメソッドを呼び、その結果の文字列を一時変数に組み立てることです。

モック派のテスターなら、さまざまなオブジェクトへ文字列バッファを渡し、それぞれにさまざまな文字列をバッファへ追加させる可能性が高いでしょう。つまり、文字列バッファを collecting parameter として扱うのです。

モック派のテスターは、「train wreck」を避けることについてより多く語ります。これは `getThis().getThat().getTheOther()` のような形のメソッドチェーンです。メソッドチェーンを避けることは、Law of Demeter に従うこととしても知られています。メソッドチェーンは smell ですが、逆方向の問題、つまり転送メソッドで膨れ上がった middle man オブジェクトもまた smell です。私は昔から、Law of Demeter が Suggestion of Demeter と呼ばれていたら、もっと受け入れやすかっただろうと感じています。

OO 設計で人々が理解するのが最も難しいことの1つは、「Tell Don't Ask」の原則です。これは、オブジェクトからデータを引きはがしてクライアントコードで処理するのではなく、オブジェクトに何かをするよう伝えることを促します。モック派は、モック派テストを使うことでこれを促進し、最近のあまりに多くのコードにはびこる getter の紙吹雪を避けられると言います。クラシック派は、これを行う方法は他にもたくさんあると主張します。

状態ベースの検証で認識されている問題は、検証を支えるためだけに query メソッドを作ることにつながり得ることです。テストのためだけにオブジェクトの API へメソッドを追加するのは、決して気持ちのよいものではありません。振る舞い検証を使えば、この問題を避けられます。これに対する反論は、実際にはそのような変更はたいてい小さい、というものです。

モック派は role interface を好み、このテストスタイルを使うことで、より多くの role interface が促されると主張します。各協調が別々にモック化されるため、role interface に変わる可能性が高くなるからです。したがって、上のレポート生成で文字列バッファを使う例なら、モック派は、そのドメインで意味を持つ特定の role を作り出す可能性が高いでしょう。その role は文字列バッファによって実装されるかもしれません。

この設計スタイルの違いが、ほとんどのモック派にとって重要な動機であることを忘れないことが大切です。TDD の出発点には、進化的設計を支える強力な自動回帰テストを得たいという願いがありました。その過程で実践者たちは、先にテストを書くことが設計プロセスを大きく改善することを発見しました。モック派は、どのような設計がよい設計なのかについて強い考えを持っており、主として人々がその設計スタイルを発展させるのを助けるためにモックライブラリを開発してきました。

### So should I be a classicist or a mockist?

これは、自信を持って答えるのが難しい質問です。個人的には、私はずっと古風なクラシック派 TDD 実践者であり、今のところ変える理由は見当たりません。モック派 TDD に説得力のある利点は見えておらず、テストを実装に結合させる結果を心配しています。

これは、モック派のプログラマを観察したときに特に強く感じました。私は、テストを書いている間に、どのように行われるかではなく、振る舞いの結果に焦点を当てられるという事実がとても好きです。モック派は、期待を書くために、SUT がどのように実装されるかを常に考えています。これは私には本当に不自然に感じます。

また私は、玩具以上のものに対してモック派 TDD を試したことがないという不利も抱えています。Test Driven Development そのものから学んだように、技法は真剣に試してみなければ判断が難しいことがよくあります。優れた開発者の多くが、モック派であることにとても満足し、確信を持っていることも知っています。だから、私は今でも納得したクラシック派ではありますが、あなたが自分で判断できるように、両方の議論をできるだけ公平に示したいと思います。

もしモック派テストが魅力的に思えるなら、試してみることを勧めます。モック派 TDD が改善しようとしている領域で問題を抱えているなら、特に試す価値があります。ここでは主に2つの領域が見えます。1つは、テストがきれいに壊れず、問題の場所を教えてくれないために、失敗時のデバッグに多くの時間を費やしている場合です。

これは、クラシック派 TDD をより細かい粒度のクラスタに対して使うことでも改善できます。2つ目の領域は、オブジェクトに十分な振る舞いが含まれていない場合です。モック派テストは、開発チームがより振る舞い豊かなオブジェクトを作るよう促すかもしれません。

### Final Thoughts

ユニットテスト、xUnit フレームワーク、Test Driven Development への関心が高まるにつれ、ますます多くの人がモックオブジェクトに出会うようになっています。多くの場合、人々はモックオブジェクトフレームワークについて少し学びますが、その背後にあるモック派とクラシック派の分断を十分には理解していません。その分断のどちら側に寄るとしても、この見方の違いを理解することは有用だと思います。

モックフレームワークを便利だと感じるために、モック派である必要はありません。しかし、そのソフトウェアの設計判断の多くを導く考え方を理解しておくことは有用です。

この記事の目的は、かつても今も、これらの違いを指摘し、それらのトレードオフを並べることです。モック派の考え方には、ここで扱う時間がなかったことがさらにあります。特に、設計スタイルへの帰結です。今後数年のうちに、この点についてさらに多くの文章が書かれ、コードより先にテストを書くことがもたらす興味深い帰結への理解が深まることを願っています。

モック派のテストスタイルについてもっと知るには、全体として最もよい資料は Freeman と Pryce の本です。著者たちは mockobjects.com を運営しています。特に、すばらしい OOPSLA 論文を読んでください。TDD から派生した別の流れであり、スタイルとして非常にモック派的である Behavior Driven Development については、Daniel Terhorst-North の入門から始めるとよいでしょう。

これらの技法については、jMock、nMock、EasyMock、.NET EasyMock のツールサイトを見ることでも、さらに学べます。他にもモックツールはあります。この一覧が完全だとは考えないでください。

XP2000 では、最初の mock objects 論文が発表されましたが、いまとなってはかなり古くなっています。
