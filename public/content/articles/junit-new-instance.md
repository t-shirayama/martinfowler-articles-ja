# Junit New Instance

## 要約

JUnit が各テストメソッドの実行ごとに新しいテストケースオブジェクトを作るのは、偶然ではなく意図的な設計判断です。その目的は、テスト同士が状態を共有して互いに影響し合うのを防ぎ、どの順序でも同じ結果になるようにすることです。

## 読むときの観点

- JUnit の「テストごとに新しいインスタンス」という振る舞いが、テストの独立性をどう支えているか。
- `setUp` や `tearDown` と、新しいテストケースインスタンスを作る設計判断の関係。
- 状態を共有したい場合に、独立性を破るリスクをどう扱うべきか。

## 原文の翻訳

JUnit テスティングフレームワークの設計判断の一つについて、よく質問を受けます。それは、テストメソッドを実行するたびに新しいオブジェクトを作る、という判断です。短い bliki エントリとして書いておく価値があるくらいには、よく話題になります。

とはいえ、JUnit について書いているからといって、他の形のテストが重要でないと言いたいわけではありません。役に立つテスト活動はたくさんあります。JUnit とその仲間たちは、その多くで価値を持ちますが、すべてを解決するものではありません。xUnit テストについて書くことが、リファクタリングやユースケースなどの重要性を否定する、という意味でもありません。

次のような小さな Java のテストクラスを考えます。

```java
import junit.framework.*;
import java.util.*;

public class Tester extends TestCase {
  public Tester(String name) {super(name);}
  private List list = new ArrayList();
  public void testFirst() {
    list.add("one");
    assertEquals(1, list.size());
  }
  public void testSecond() {
    assertEquals(0, list.size());
  }
}
```

意外に思う人もいるかもしれませんが、この二つのテストはどちらも成功します。実行順序がどうであっても成功します。JUnit はこのテストを実行するとき、`testXXX` メソッドごとに `Tester` のインスタンスを一つずつ作るからです。そのため、`list` フィールドは各テストメソッドの実行ごとに新しく初期化されます。これは JUnit のバグではなく、**意識的な設計判断**です。

JUnit の基本設計は、Kent Beck が Smalltalk で作ったテスティングの仕組みに由来しています。その重要な原則の一つが分離、つまり isolation です。これは、あるテストが他のテストを失敗させるようなことをしてはならない、という考え方です。

分離にはいくつかの利点があります。任意の組み合わせのテストを、任意の順序で実行しても同じ結果になります。あるテストが失敗したときに、その原因が別のテストの書き方にあるのではないかと悩まずにすみます。また、一つのテストが失敗しても、その後始末の残骸が他のテストを失敗させる心配が減ります。これは、本当のバグを隠してしまう連鎖的なエラーを防ぐ助けになります。

JUnit には、分離を支える別の仕組みもあります。特に `setUp` と `tearDown` は、各テストメソッドの開始時と終了時に実行されます。先ほどの例なら、次のように `setUp` を使えます。

```java
public void setUp() {
  list = new ArrayList();
}
```

多くの場合、必要な再初期化は `setUp` でできるため、`tearDown` を使う必要はありません。

テストメソッドを分離するには、すべての状態をローカル変数に置き、フィールドを使わないという方法もあります。しかしそうすると、セットアップ用のコードを各テストに重複して書くことになります。重複は避けたいものです。

JUnit の方式に批判的な人は、`setUp` と `tearDown` があるなら、毎回新しいオブジェクトを作る必要はない、と主張します。必要なフィールドをそれらのメソッドで初期化し直せばよい、というわけです。一方、この方式を支持する人は、それは正しいとしても、多くの人がフィールドで初期化を行うのだから、より強い分離を提供した方がよい、と考えます。フレームワーク設計の大切な役割は、**正しいことを簡単にし、問題を起こすことを難しくする**ことです。

この方式のコストについての主な議論は、余分なオブジェクトが作られることにあります。JUnit のテストケースオブジェクトだけでなく、セットアップやフィールド初期化で作られるその他のオブジェクトも含まれます。しかし多くの場合、この心配は的外れです。オブジェクト生成に対する恐れの多くは、オブジェクトの割り当てと回収の仕組みに関する古い感覚に基づいています。もちろん、オブジェクト生成が問題になりうる環境はありますし、初期の Java ではそうした面もありました。しかし現代の Java では、たいていの場合、オブジェクト生成を過度に心配する必要はありません。

ただし、「たいてい」は「常に」ではありません。頻繁に作りたくないオブジェクトのよい例はデータベース接続です。これは共有する意味がありますが、単一のテストケースクラス内の全テストメソッドで共有するだけでは足りないことが多いでしょう。もっと広い範囲で共有したくなるはずです。簡単で粗い方法として static 変数があります。一般には static を避けるのが賢明ですが、テスト実行の文脈では問題ない場合もあります。それでも、できれば避けたいところです。

JUnit には、テストフィクスチャのオブジェクトを共有するための柔軟な仕組みとして `TestSetup` デコレータがあります。これを使うと、任意のテストスイートに対して共通の状態を設定できます。単一のテストケースクラス内のメソッド間で共有するより、ずっと柔軟にテストのグループ間で状態を共有できます。

ただし、`TestSetup` の大きな問題は情報を見つけにくいことです。そしてもっと重要なのは、これを使うと分離を破ることになる点です。**壊れた分離は、見つけにくいバグにつながりやすい**ものです。本当に必要なとき以外は使うべきではありません。

各テストメソッドをそれぞれ別のクラスにすればよいのではないか、と思うかもしれません。実際、JUnit の最初期の形では、フィクスチャを持つテストケースをサブクラス化した内部クラスを使っていました。その方が設計としては明らかでしたが、テストを書くのが難しくなりました。そこで、より分かりにくいものの、テストを書きやすくする pluggable selector のパターンを使う方向へ進みました。

JUnit の方式に対するもう一つの反論は、直感的ではないというものです。この仕組みを実現するメカニズムが理解しづらい、という指摘には共感できます。Pluggable Selector は広く知られたパターンではありませんし、馴染みのないパターンを使った設計スタイルには居心地の悪さがあります。それでも全体としては、テストの分離とテストの書きやすさが、その実装の分かりにくさを上回ると考えています。

もちろん、この判断に同意しない優れた人たちもいます。TestNG はこの方式を採っていませんし、広く使われている NUnit も同じです。NUnit で同様のテストを書くと、共有されたインスタンス状態のために失敗が起きます。

```csharp
[TestFixture]
public class ServerTester
{
  private IList list = new ArrayList();
  [Test]
  public void first() {
    list.Add(1);
    Assert.AreEqual(1, list.Count);
  }
  [Test]
  public void second() {
    Assert.AreEqual(0, list.Count);
  }
}
```

このような方式のフレームワークを使っているなら、すべてのインスタンス変数をセットアップメソッドで初期化することを強く勧めます。そうすればテストの分離を保ち、デバッグで髪をむしりたくなるような状況を避けやすくなります。

私はテストケースインスタンスを再利用する設計には同意しません。しかし、その判断をした人たちが愚かだとも、何か悪意を持っているとも思いません。彼らは設計上のトレードオフを別の形で評価したのです。ソフトウェア設計には流動的な面があり、そこについて敬意を持って意見を違えられる方がよいと思います。
