# What do you mean by “Event-Driven”?

## 要約

イベント駆動という言葉は、文脈によってかなり異なる意味で使われます。
通知、状態変化の記録、非同期メッセージング、イベントソーシングなどが同じ言葉で語られるため、議論が混乱しやすくなります。

この記事は、イベント駆動という表現を見たときに、具体的にどのパターンや設計を指しているのかを切り分ける助けになります。
アーキテクチャ会話で言葉を揃えるために読みたい記事です。

## 読むときの観点

- イベントが「通知」なのか「事実の記録」なのかを区別する。
- 非同期化とイベントソーシングを混同しない。
- 言葉の定義を揃えること自体が設計上のリスク低減になる。
- イベントの利用目的から設計を選ぶ。

## 原文の翻訳

昨年の終わりごろ、私は Thoughtworks の同僚たちと「イベント駆動」アプリケーションの性質について議論するワークショップに参加した。ここ数年、私たちはイベントを多用するシステムを数多く作ってきた。それらはしばしば称賛され、またしばしば非難もされてきた。北米オフィスがサミットを企画し、世界各地から Thoughtworks のシニア開発者が集まって考えを共有した。

このサミットで得られた最大の成果は、人々が「イベント」について話すとき、実際にはかなり異なるものを意味しているのだと認識したことだった。そこで私たちは、有用なパターンとして何を取り出せるのかを整理するため、多くの時間を費やした。このメモは、私たちが特定した主なパターンの短いまとめである。

### イベント通知

これは、あるシステムが自分のドメイン内で起きた変化を他のシステムへ知らせるために、イベントメッセージを送る場合に起きる。イベント通知の重要な要素は、発信元のシステムが応答をそれほど気にしないことだ。多くの場合、まったく返答を期待しない。あるいは発信元が気にする応答があるとしても、それは間接的なものになる。イベントを送るロジックの流れと、そのイベントへの何らかの反応に応答するロジックの流れとの間には、はっきりした分離がある。

私はこの話題について、2017年4月の GOTO Chicago のオープニング基調講演で、より深く話した。

イベント通知のよいところは、結合度が低いことを意味し、設定もかなり簡単な点だ。しかし、複数のイベント通知にまたがって実際にはひとつの論理的な流れが存在している場合、問題になりうる。そのような流れは、どのプログラムテキストにも明示されていないため、見えにくいからだ。しばしば、その流れを把握する唯一の方法は、稼働中のシステムを監視することになる。これにより、その流れのデバッグや変更が難しくなる。危険なのは、イベント通知を使うと、うまく疎結合に見えるシステムをとても簡単に作れてしまう一方で、**より大きなスケールの流れを見失っていることに気づかない**まま、将来の問題を招き入れてしまうことだ。このパターンは今でもとても有用だが、その罠には注意しなければならない。

この罠の単純な例は、イベントが受動攻撃的なコマンドとして使われる場合である。これは、発信元システムが受信者に何かのアクションを実行してほしいと期待していて、本来ならその意図を示すためにコマンドメッセージを使うべきなのに、そのメッセージをイベントとして装っている場合に起きる。

イベントは多くのデータを運ぶ必要はない。多くの場合、何らかの ID 情報と、追加情報を問い合わせるための送信者へのリンクだけで十分である。受信者は、何かが変わったことを知り、その変化の性質について最小限の情報を得るかもしれない。そのうえで次に何をするかを決めるため、送信者へリクエストを返す。

### イベント搬送状態転送

このパターンは、あるシステムのクライアントを更新するとき、クライアントがその後の作業のために発信元システムへ問い合わせなくてよいようにしたい場合に現れる。顧客管理システムは、顧客が住所などの詳細情報を変更するたびにイベントを発火し、そのイベントに変更されたデータの詳細を含めるかもしれない。受信者はその変更を使って自分自身の顧客データのコピーを更新できるため、将来の作業で主要な顧客システムとやり取りする必要がなくなる。

このパターンの明らかな欠点は、大量のデータがあちこちへ運ばれ、多数のコピーが生まれることだ。しかし、ストレージが豊富な時代には、それは以前ほど大きな問題ではない。代わりに得られるのは、より高い耐障害性である。顧客システムが利用できなくなっても、受信側システムは機能し続けられる。顧客情報へアクセスするためのリモート呼び出しが不要になるため、レイテンシも下がる。すべての利用側システムからの問い合わせを満たすために、顧客システムへかかる負荷を心配しなくてもよい。ただし、受信側の複雑さは増す。必要になったときに送信者を呼び出して追加情報を得るほうが普通は簡単だが、このパターンでは受信側がすべての状態を維持する方法を整理しなければならないからだ。

### イベントソーシング

イベントソーシングの中核にある考えは、システムの状態を変更するたびに、その状態変更をイベントとして記録するというものだ。そして将来のどの時点でも、イベントを再処理することでシステム状態を確実に再構築できる。イベントストアが主要な真実の源となり、システム状態はそこから純粋に導出される。プログラマにとって、この最良の例はバージョン管理システムだ。すべてのコミットのログがイベントストアであり、ソースツリーのワーキングコピーがシステム状態である。

イベントソーシングは多くの論点を持ち込む。ここではそれらには踏み込まないが、よくある誤解をいくつか強調しておきたい。イベント処理が非同期である必要はない。ローカルの git リポジトリを更新する場合を考えてみればよい。それは完全に同期的な操作であり、Subversion のような集中型バージョン管理システムを更新する場合も同じである。もちろん、これらのコミットがすべてあることで、さまざまな興味深い振る舞いが可能になる。git はその優れた例だ。しかし、中核にあるコミットは、基本的には単純なアクションである。

もうひとつのよくある誤りは、イベントソーシングされたシステムを使うすべての人が、有用なデータを判断するためにイベントログを理解し、アクセスすべきだと考えることだ。しかし、イベントログに関する知識は限定できる。私はこれを、ソースツリー内のすべてのコミットを知らないエディタで書いている。そのエディタは、ディスク上にファイルがあると仮定しているだけだ。イベントソーシングされたシステム内の処理の多くは、有用なワーキングコピーに基づいて行える。イベントログ内の情報を本当に必要とする要素だけが、それを操作すればよい。役に立つなら、異なるスキーマを持つ複数のワーキングコピーを持つこともできる。しかし通常は、ドメイン処理と、イベントログからワーキングコピーを導出する処理との間に、**明確な分離**があるべきだ。

イベントログを扱うときは、ワーキングコピーのスナップショットを作ることがよく役に立つ。そうすれば、ワーキングコピーが必要になるたびに、すべてのイベントを最初から処理しなくて済む。実際ここには二重性がある。イベントログは、変更のリストとして見ることも、状態のリストとして見ることもできる。片方からもう片方を導出できるのだ。バージョン管理システムは、最高の性能を得るため、イベントログ内でスナップショットと差分を混在させることが多い[^1]。

イベントソーシングには、興味深い利点がたくさんある。バージョン管理システムの価値を考えれば、容易に思い浮かぶものだ。イベントログは強力な監査能力を提供する。会計取引は、勘定残高に対するイベントソースである。イベントログをある時点まで再生することで、過去の状態を再作成できる。再生時に仮想的なイベントを注入することで、別の履歴を探索することもできる。イベントソーシングにより、Memory Image のような永続化されないワーキングコピーも現実的になる。

イベントソーシングにも問題はある。結果が外部システムとのやり取りに依存する場合、イベントの再生は問題を抱える。時間の経過とともにイベントのスキーマが変化することへどう対処するかも考えなければならない。イベント処理がアプリケーションに多くの複雑さを加えると感じる人も多い。ただし私は、それがワーキングコピーを導出するコンポーネントとドメイン処理を行うコンポーネントの分離が不十分であることに、より強く起因しているのではないかとも思っている。

### CQRS

Command Query Responsibility Segregation、つまり CQRS は、情報の読み取りと書き込みに別々のデータ構造を持つという考え方である。厳密に言えば、CQRS は実際にはイベントに関するものではない。設計にイベントがまったく存在しなくても CQRS は使えるからだ。しかし、人々はよく CQRS をここまでに挙げたパターンと組み合わせる。そのため、サミットにも CQRS が登場した。

CQRS を正当化する理由は、複雑なドメインでは読み取りと書き込みの両方を扱う単一のモデルが複雑になりすぎるため、モデルを分離することで単純化できる、というものだ。読み取りが大量にあり、書き込みが非常に少ない場合のように、アクセスパターンに違いがあるときには特に魅力的である。しかし、CQRS を使って得られる利点は、別々のモデルを持つことによる追加の複雑さと釣り合っていなければならない。私の同僚の多くは CQRS の利用にかなり慎重で、しばしば誤用されていると見ている。

### これらのパターンを理解する

標本集めに熱心なソフトウェア植物学者のような立場から見ると、これは扱いにくい地形だと思う。中心的な問題は、異なるパターンを混同してしまうことにある。あるプロジェクトで、有能で経験豊富なプロジェクトマネージャが、イベントソーシングは大失敗だったと私に語った。どんな変更にも、読み取りモデルと書き込みモデルの両方を更新するため、2倍の作業が必要になったという。その言い方だけでも、私はイベントソーシングと CQRS の混同が起きている可能性を感じ取れる。では、どちらが原因だったのかをどうやって見極めればよいのだろうか。そのプロジェクトの技術リードは、主な問題は大量の非同期通信だったと主張した。確かに非同期通信は複雑さを増すことで知られているが、イベントソーシングにも CQRS にも必須の要素ではない。さらに、これらのパターンはいずれも、適切な場所ではよいものであり、間違った地形に置かれれば悪いものになることにも注意しなければならない。しかし、パターンをひとまとめにして混同していると、何が適切な地形なのかを見極めるのは難しい。

この混乱を整理し、それぞれのパターンをどううまく実践するか、いつ使うべきかについて確かな指針を与える、決定版の論考を書けたらよいと思う。残念ながら、私にはそれを書く時間がない。このメモが役に立つことを願って書いているが、本当に必要なものにはかなり届いていないこともよくわかっている。

### さらに読むために

私はこの話題について講演を用意し、GOTO Chicago 2017 の基調講演で話した。

2006年に、私は『Patterns of Enterprise Application Architecture』の続巻を作るつもりで、いくつものプロトパターンを書いた。残念ながら、10年以上たってもこの作業を続ける時間を持てていない。しかし当時書いたものは、今でも読める状態にある。イベントについては、まず Focusing on Events から読むとよい。これは当時の私の、イベント利用についての考えをまとめたものだ。少し前の文章ではあるが、そこで書いたことの大部分は今でも妥当だと思う。

これらの記事の中で最も影響力があるのは Event Sourcing に関する記事である。そこでは主に、再生を使って過去の状態や代替的な状態を作ることの価値について述べている。

Event Collaboration に関する記事では、Event Notification と Event-Carried State Transfer のパターンに触れているが、それらのパターンを混同している。私がそれらを別々のものとして考え始めたのは、このワークショップの最中だった。

私は CQRS についての bliki 記事も書いている。

これらの話題については、Web 上にさらに多くの資料がある。探索を楽しんでほしい。ここではそれらについてコメントしない。私は時間をかけて調べ、推薦できるものを選び出していないからだ。

### 脚注

[^1]: git は `.git/objects` にファイルやツリーの状態を保存しているのでイベントソーシングの例ではない、と言う人の話を聞くことがある。しかし、システムが内部ストレージで変更を使うかスナップショットを使うかは、それがイベントソーシングされているかどうかには影響しない。git は必要に応じて変更のリストを喜んで取り出してくれる。さらに、packfile へデータを圧縮するときには、性能上の理由で選ばれた組み合わせとして、スナップショットと変更の両方を使う。
