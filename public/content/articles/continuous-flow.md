# Continuous Flow

## 要約

Continuous Flowは、作業を固定長のイテレーションへ割り当てるのではなく、優先順位づけされたUser Storyのリストから、完了するたびに次の作業を引き取って進める計画方式です。

この記事の中心は、Continuous FlowにはWIP Limitが重要であり、予測しにくい作業の到着には向いている一方で、作業内容そのものが予測しにくい場合や、経験の浅いチームには難しくなる、という点です。

## 読むときの観点

- Continuous Flowを、Timeboxed Iterationsの単なる省略版ではなく、別のフィードバック設計として読む。
- WIP Limitが、作業を始める量ではなく終わらせることへチームを集中させる点に注目する。
- 常に忙しいContinuous Flowチームは、健全ではなく危険信号かもしれない。
- 作業の到着が予測不能なのか、作業の性質が予測不能なのかを区別する。

## 原文の翻訳

Continuous Flowは、作業をスケジュールするためのアプローチであり、agile software developmentと結びつけて語られることがよくあります。チームはソフトウェアの機能をUser Storyに分解します。次に、それらのstoryを大まかなリストとして優先順位づけします。チームはそのUser Storyのいくつかに取り組み、ひとつを完了したら、リストから次のひとつを引き取ります。

Continuous Flowで作業するときは、チームが同時に取り組めるstoryの数について、work-in-progress limit、つまりWIP Limitを設定すると役に立ちます。WIP Limitに達したら、すでに進行中のstoryをひとつ完了するまで、それ以上のstoryを始めることはできません。WIP Limitが重要なのは、チームの意識をstoryを終わらせることに向け続けるからです。そうしないと、「ほとんど終わっている」作業の山がたまりやすくなります。

Continuous Flowは、bug fixingやmaintenance tasksのように、予測しにくい作業の流れに向いています。ただしそのような場合は、急増にすばやく対応できるだけの人数をチームに確保するよう注意が必要です。これは、静かな時期にはチームが作業環境を改善するために使えるSlackを持つ、ということでもあります。**Continuous Flowのチームが常に忙しいなら、それはたいてい危険信号です。**

Continuous FlowはTimeboxed Iterationsの代替です。利点は、storyをiterationへ割り当てたり、storyを見積もったり、iteration capacityを計算したりする作業をチームが行わずにすむことです。しかし、そのようなチームはしばしば困難にぶつかります。iterationの規則的なcadenceは、コードベースにcruftがたまっていることや、予想よりはるかに大きなstoryに時間を沈めていることなど、チームが問題に気づくためのフィードバックループを提供しているからです。したがってContinuous Flowは、仕事のceremonyを減らしたい熟練したチームには有効ですが、経験の浅いチームにはiterationのほうが向いています。

私の同僚であるKennedy Collinsは、Continuous Flowは作業がいつ到着するかを予測しにくい場合にはよいが、作業の性質が予測しにくい、あるいは十分に理解されていない場合にはそれほど向いていない、と指摘しました。
