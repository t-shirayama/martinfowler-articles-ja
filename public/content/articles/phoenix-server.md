# Phoenix Server

## 要約

Phoenix Serverは、サーバーを定期的に破棄して、既知の構成から再構築できるようにする運用の考え方です。
サーバーを長く生かして手作業の変更を積み重ねると、構成のずれが生まれ、Snowflake Serverへ近づきます。

定期的に作り直せるサーバーは、構成がコードや自動化に正しく表れているかを検証する手段にもなります。
PuppetやChefのような構成管理ツールによる再同期と組み合わせることで、構成ドリフトへの耐性を高められます。

## 読むときの観点

- サーバーを「大事に修理するもの」ではなく「再生成できるもの」として見る。
- Configuration DriftとSnowflake Serverの関係を押さえる。
- 構成管理ツールによる再同期と、破棄して再構築する戦略の違いを見る。
- Phoenix ServerはImmutable Serverの前段にある発想としても読める。

## 原文の翻訳

著者はかつて、運用チーム向けの認定サービスを始めるという空想をしたことがあります。その評価では、同僚と一緒に企業のデータセンターへ現れ、重要な本番サーバーを壊しにかかります。評価基準は、運用チームがすべてのアプリケーションを復旧させるまでにどれだけ時間がかかるかです。

もちろん、これは馬鹿げた空想です。しかし、その中には知恵のかけらがあります。実際にサーバーを壊す必要はありませんが、仮想的には定期的にサーバーを焼き払うのがよい考えです。サーバーはPhoenixのように、灰の中から繰り返し立ち上がるべきです。

Phoenix Serverを使う主な利点は、Configuration Driftを避けることです。Configuration Driftとは、記録されない場当たり的な変更がシステム構成に入り込むことです。ドリフトはSnowflake Serverへ続く道であり、そこへ進みたくはありません。

ドリフトに対抗する方法のひとつは、既知のベースラインへサーバーを自動的に再同期するソフトウェアを使うことです。PuppetやChefのようなツールには、定義済みの構成を自動的に再適用する機能があります。これらのツールはPhoenix Serverを作るためにも有用です。

ただし、このような再適用には限界があります。構成の再適用で検出できるのは、ツールが管理すると定義した領域のドリフトだけです。その外側で起きた構成ドリフトは修正されません。一方、Phoenix Serverは最初から作り直すため、元の構成に含まれていないずれを持ち越しません。

だからといって、構成の再適用が役に立たないわけではありません。通常、再適用はサーバーを焼き払うより速く、影響も小さい方法です。しかしSnowflake Serverを遠ざけるには、再同期と再構築の両方を使う価値があります。

Phoenix Serverの要点は、サーバーを壊れないように保存することではありません。**いつでも作り直せることを前提に運用し、構成の真実を自動化された定義へ置く**ことです。
