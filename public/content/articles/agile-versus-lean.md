# Agile Versus Lean

## 要約

AgileとLeanは、ソフトウェア開発において互いに競合する選択肢ではありません。Leanはもともとトヨタ生産方式に由来する製造業の考え方であり、Agileは1990年代に生まれた複数のソフトウェア開発手法を束ねる広い概念です。

この記事は、Agileの初期からLeanの影響があり、Mary PoppendieckとTom Poppendieckによってそのつながりがより明確になったと説明します。結論として、AgileかLeanかを選ぶのではなく、Agileの中でLean由来の考え方をどれほど明示的に使うかが問題だと述べています。

## 読むときの観点

- AgileとLeanを、導入候補として二者択一にしない。
- Leanの由来はソフトウェアではなく製造業、とくにトヨタ生産方式にある。
- Agileは特定の単一プロセスではなく、価値観と原則を共有する広い傘として読む。
- Value Stream Map、Last Responsible Moment、cycle timeなど、Agile内にあるLean由来の語彙に注目する。
- Lean Softwareは、ScrumやXPのような公開プロセスというより、Agileに混ぜ込める思考道具として捉える。

## 原文の翻訳

Agile software developmentを使おうと思っているが、代わりにLean software developmentを使うべきだろうか。

この問いには、最近何度か出会っている。すぐには答えられない問いだ。というのも、この問いはLeanとAgileの関係についての誤った前提に基づいているからだ。そこでまず、その関係を説明するために、少し歴史をたどる必要がある。

「Lean」は、基本的には1950年代にトヨタが開発した、製造業の世界におけるアプローチを指す。当時の日本の産業は、第二次世界大戦の荒廃から立ち直りつつあった。このアプローチは、しばしばトヨタ生産方式と呼ばれ、さまざまな西洋の思想家、とくにDemingの影響を受けてはいるものの、主に大野耐一の功績とされている。

トヨタ生産方式は、1990年代に、なぜ日本企業が多くの産業で米国を上回っているのかを説明する本を西洋の著者たちが書き始めたことで、世界のほかの地域にもよく知られるようになった。西洋の著者たちは、このアプローチをLean Manufacturingと呼んだ。その後、日本の産業全体はより難しい時期にも直面したが、トヨタは多くの西洋の自動車会社を上回り続けている。

Agile software developmentは、1990年代に開発された複数のソフトウェア開発手法、たとえばExtreme ProgrammingやScrumを含む包括的な用語である。これらの手法は共通する哲学を持っており、それは`Manifesto for Agile Software Development`で価値と原則として述べられたものだ。私のエッセイ「The New Methodology」では、この点をより詳しく扱っている。

Lean ManufacturingとAgile softwareの間には、最初からつながりがあった。さまざまなAgile手法の開発者の多くが、Lean Manufacturingの考え方から影響を受けていたからだ。このつながりをより明確にしたのが、Mary PoppendieckとTom Poppendieckである。MaryはLean Manufacturingを採用した製造工場で働いていたことがあり、夫のTomは経験豊富なソフトウェア開発者である。

二人は、Leanの考え方をソフトウェアの世界へ適用する本をいくつか書いている。人々がLean Softwareについて話すとき、たいていはこれらの本にある考え方を指している。ただし、同じような結びつけ方をしている人はほかにもいる。

Lean ManufacturingとAgile software methodsは、とてもよく似た哲学を持っている。どちらも、適応的な計画と、人に焦点を当てたアプローチを重視する。そのため、Leanの考え方はAgile softwareの物語と非常によく合う。MaryとTomはどちらもAgileコミュニティで非常に活発に活動してきた。実際、MaryはAgile Allianceの形成に重要な役割を果たしたと私は考えている。

彼女は私と同じくAgile Allianceの創設時の理事だったが、私よりはるかに有能に働いていた。

Lean Manufacturingが最初からAgilistたちに影響を与えていたことは、すでに述べた。この数年で、明らかにLean Manufacturingの系譜を持つ考え方が、Agileの世界にさらに現れている。具体的な技法であるValue Stream Mapから、設計判断を下すLast Responsible Momentのように、既存のAgile概念を言語化したものまで幅広い。分析や設計のドキュメントを在庫として考えるという発想は、Poppendieck夫妻から来ている。

私の知る何人かのAgilistは、cycle timeを短くする重要性を強調している。これもLeanの影響が強い考え方だ。私の同僚であるRichard Durnallは、Leanの知識がAgileの考え方とどのように溶け合うかについて、よい要約を書いている。

Leanの考え方が多くの人に強く訴える理由の一つは、それがAgile software developmentを人に説明する方法を与えてくれることだ。とくにIT部門の上級者や顧客側の上級者に説明するときに役立つ。その説明は、トヨタをまねようという粗い訴えから、Lean Manufacturingの利点がAgile software developmentの考え方へどのように対応するかを詳しく論じるものまで幅がある。

ここまで見てきたように、ソフトウェアの世界ではLeanとAgileは深く絡み合っている。両者を代替案として語ることは、実際にはできない。Agileを実践しているならLeanも実践しており、その逆もまた同じだ。Agileはもともと非常に広い概念として意図されていた。表面的には異なって見えるプロセスが共有する、価値と原則の中核である。**AgileかLeanかを行うのではなく、AgileとLeanを行う**のだ。唯一の問いは、Lean Manufacturingから直接引き出した考え方を、どれほど明示的に使うかである。

Poppendieck夫妻は、Leanを独立した別個の考え方として紹介したわけではない。また、ScrumやXPのような公開されたプロセスとしてLeanを紹介したわけでもない。むしろ二人は、どんなAgileアプローチにも容易に混ぜ込める思考道具の集合としてLeanを紹介した。私はLeanを、Agileコミュニティ内の思考の一本の糸のように考えている。豊かな絨毯の中の模様のようなものだ。

Leanを名乗るメーリングリストや、自分をLean thinkerと呼ぶ人たちがいるように、独自のLean softwareコミュニティは存在する。しかしこれは、強いXP、Scrum、その他のコミュニティが存在することと変わらない。これらのコミュニティの多くの人は、自分たちをより広いAgileムーブメントの一部だと考えており、複数のAgileコミュニティで活動している人も多い。

そもそも「Agile」という言葉を作った要点は、私たちが価値と原則の中核を共有していると認識したことにある。そして、この共通の中核があるからこそ、私たちに共通するものは、違いよりも大きいのである。
