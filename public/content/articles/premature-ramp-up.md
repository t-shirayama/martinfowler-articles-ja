# Premature Ramp Up

## 要約

Premature Ramp Up は、うまく進んでいるソフトウェアチームに人を急いで増やしすぎると、かえってコードベースとチームのコミュニケーションが崩れやすくなる、という警告です。

人数を増やしても生産性は線形には上がりません。新しい人がコードやドメイン、人間関係に馴染むまでには時間がかかり、既存メンバーもその支援に時間を使います。特にプロジェクトの初期に急拡大すると、真似されるべきでない設計まで広まり、後の開発基盤を傷める危険があります。

## 読むときの観点

- ramp-up は「採用人数」ではなく、チームが安全に人を受け入れられる速度として読む。
- コードベースの一貫性と、人間同士の関係形成の両方が崩れる点に注目する。
- 大規模化の前に、小さなチームで設計の核を安定させる意味を考える。
- 「agile をスケールさせる」前に、本当に人数を増やす価値があるのかを問い直す。

## 原文の翻訳

ソフトウェアのよいところのひとつは、人々がそれを欲しがり、しかも早く欲しがることです。組織がチームにソフトウェアの生産を速めるよう求めるのはよくあることです。そして時折、組織は自分たちの本気度をいかにも示す方法で支援しようとします。つまり、チームに人を増やすためにお金を使うのです。

ソフトウェアチームに人を加えることの本当の効果については、これまで多くの議論がありました。私には、線形の効果は得られないことが明らかに思えます。チームを倍にしても生産性は倍になりません。コミュニケーションと調整のコストが効き始め、その増加分を鈍らせるからです。私のまったく科学的ではない経験則では、生産性の増加は人数増加の平方根に比例しそうです。つまり、チームを倍にすると、生産性はおよそ50%増える、という程度です。

しかし実際には、個別の要因によって大きく変わります。私の知っている人の中には、かなり大きなチームでさえ、その人ひとりで生産性を倍にしそうな人もいます。一方で、チームの生産性を下げてしまう人に出会ったこともあります。

ここで私が取り上げたい問題は、ramp-up です。うまく機能している小さなチームがあり、より多くのソフトウェアを求めていて、そのためにお金を使う用意もあるとします。進捗速度を倍にするためなら、4倍、場合によっては6倍の費用を払ってもよいと思っているかもしれません。重要でありながら十分に理解されていない要因は、**チームに安全に人を追加できる速度**です。

私はこれまでに、あまりに多くの人をあまりに急いで追加したプロジェクトに何度も出会ってきました。それは、コードベースそのものの凝集性が崩れるという形で現れます。重複がはびこります。私の友人の何人かは、ひとつのアプリケーションの中にオブジェクトリレーショナルマッピングのフレームワークが3つあったプロジェクトを知っています。この崩壊が起こるのは、新しい人たちが現在のコードベースの動き方を理解しておらず、競合するフレームワークを追加するような、既存のあり方と食い違うことをしてしまうからです。

新しい人が多すぎると、チームのリーダーシップはそのすべてを追いきれず、コードベースには問題が芽を出します。すると、誰も一貫したやり方を見つけられないため、それらの問題が互いに補強し合います。Broken Windows 症候群が働き、正のフィードバックループが生まれます。そして、正のフィードバックループはたいてい悪いものです。

これに加えて、あまりに急な ramp-up は、人間同士のコミュニケーションの仕組みも壊します。人々が互いに一緒に働くことに慣れるには時間がかかります。急激な ramp-up は、大きなチームが成功に必要な関係を形成するのを妨げてしまうことがあります。

では、安全にできる ramp-up はどれくらいなのでしょうか。ここで具体的な助言をするのは難しいです。経験豊かなプロジェクトマネージャーに尋ねると、考慮すべき変数がたくさんあると、誰もが正しく指摘するからです。私は、最も信頼している PM 情報源のひとりである Joe Zenevitch に突っ込んで聞いてみました。彼は、一度にチームを倍より大きく増やすことは決して望まない、と答えてくれました。

倍にすることでさえリスクのある判断です。既存チームがすでに12人以上だったり、ジュニアがかなり多かったりする場合、そのリスクはさらに高くなります。

大きく人数を増やしたなら、新しく加わった人たちがチームに馴染むまでは、さらに人を追加すべきではありません。これには数週間かかります。開発者はコードベースとドメインを知る必要があります。BA はドメインエキスパートに慣れる必要があります。全員が互いを知る必要があります。Thoughtworks では、人々が早く立ち上がることを期待しています。私たちは優秀で、意欲が高く、学習の速い人たちを採用しているからです。それでも、1週間か2週間はかかります。

ほとんどのチームでは、もっと長い期間を見ておくべきです。

チームに人を加えても、能力がすぐに増えるわけではありません。新しいプロジェクトで生産的になるには時間がかかります。さらに悪いことに、既存メンバーは新しい人たちが立ち上がるのを助けるために時間を使わなければなりません。そのため、velocity は最初は下がるかもしれません。Joe Z の Thoughtworks チームに関する観察では、最初の数週間は、新しい人による増加分と既存メンバーへの負荷が打ち消し合うため、正味の効果はありません。多くの ThoughtWorker は、pair programming が人をより早くオンボーディングする大きな助けになると指摘します。Pat Kua も、新しい人をチームへ効果的に迎え入れる方法について、よい助言をしています。

もうひとつ注意すべきなのは、プロジェクトのあまり早い段階で ramp-up しないことです。大規模プロジェクト、つまり50人を超えるようなプロジェクトに携わる人たちから聞いた最も確かな助言のひとつは、プロジェクトは小さく始めるべきだというものです。おそらく、開発者は十数人ほどでよいでしょう。彼らは、システムの初期部分を作ることで、主要な設計要素と相互作用を見極めるべきです。その設計が落ち着いてから初めて、チーム規模を本来の大きさまで増やすことを考えるべきです。

その設計を落ち着かせる一環として、真似されるべきではない設計要素を取り除くための時間を取りましょう。人は、すでにそこにあるものを自然に真似します。だから、そこにあるものが今後の開発にとってよい基盤になるようにしなければなりません。ここでは、**コードの清潔さに過剰なくらい注意を払う**側に倒すべきです。

最後に、ramp-up を考えるときは、それに価値があるのかを非常に慎重に考えてください。私は、大きなチームで、チームの生産性を落とさずにかなり人数を減らせるのではないかという感覚がない場面に、めったに出会ったことがありません。かつて私が言ったように、「agile をスケールさせることは、最後にやりたいこと」です。

### 注記

この「数週間」という規則は ThoughtWorker 向けのものです。そのため、Thoughtworks の採用基準に合わない人たちでは、もっと時間がかかるだろうと考えます。
