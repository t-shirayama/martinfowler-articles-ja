# Database And Build Time

## 要約

同規模のエンタープライズアプリケーションでも、フルビルドとテストに1時間かかるプロジェクトと、2から3分で終わるプロジェクトがありました。
大きな差を生んでいた要因として、テストがデータベースにどれだけ依存しているかが挙げられます。

データベースアクセスを隔離し、多くのテストをインメモリで実行できる設計にすると、ビルド時間を大きく短縮できます。
速いフィードバックを得るには、永続化層との境界設計も重要です。

## 読むときの観点

- ビルド時間の遅さは、テスト量だけでなく外部資源への依存で生まれる。
- データベースを通るテストは価値があるが、すべてのテストが通る必要はない。
- ORM や永続化層の分離は、設計上の関心だけでなくフィードバック速度にも効く。
- インメモリで実行できるテストを増やすと、開発者の反復が速くなる。
- テストピラミッドや継続的インテグレーションを考える入口として読める。

## 原文の翻訳

同じくらいの規模のエンタープライズアプリケーションプロジェクトを比べると、興味深い対比が見える。どちらもおよそ10万行のコードで、Java と .NET という似た環境にある。一方はフルビルドとテストに1時間かかり、もう一方は2から3分で終わる。

何が違うのか。ここまでの分析では、差を生んでいるのはデータベースアクセスだと見られている。どちらのプロジェクトにも大きなテストスイートがあり、長いビルド時間の大部分はテストに費やされている。特に時間を使っているのは、データベースにアクセスするテストであり、全体のおよそ半分を占めていた。

ビルド時間が短いプロジェクトでは、オブジェクトリレーショナルマッピング層を使って、アプリケーションをデータベースから隔離している。データベースに実際に触れるのはそのマッピング層のテストだけであり、多くのテストはインメモリで実行される。そのため、テスト全体がずっと速くなる。

この話の要点は、データベーステストをなくすことではない。永続化の境界で本当に必要な検証は残すべきである。しかし、アプリケーションロジックの多くを毎回データベース越しに動かすと、フィードバックが遅くなり、開発者の反復も重くなる。

速いビルドは、継続的インテグレーションにおいて重要な資産である。変更してすぐ結果が返ってくれば、問題を小さいうちに直せる。逆に、テストが遅いと実行頻度が下がり、統合の遅れや不安定さを招きやすい。

したがって、データベースに依存するテストをどこに置くかは、単なる実装の詳細ではない。**設計上の分離がフィードバックの速さを左右する**。永続化層をうまく隔離し、インメモリで検証できる領域を広げることは、ビルド時間を短くする強力な手段になる。
