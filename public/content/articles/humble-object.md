# Humble Object

## 要約

テストしにくい要素に複雑なロジックを置くと、バグを抱え込みやすく、変更もしづらくなります。Humble Object は、そうした要素をできるだけ薄く保ち、ロジックをテストしやすい場所へ移すための考え方です。

## 読むときの観点

- テストできない部分を完全になくすのではなく、そこに残す責務をどこまで減らせるか。
- UI コントロールのように自動テストが難しい場所から、どのように振る舞いを外へ逃がすか。
- Presentation Model や Passive View のようなパターンが、この考え方をどう支えるか。

## 原文の翻訳

プログラムの中には、本質的にテストが難しい、あるいはほとんど不可能な要素があります。そのような要素にロジックが入っていると、バグが入り込みやすく、進化させるのも難しくなります。

この問題を和らげるには、テストしにくい要素からできるだけ多くのロジックを取り出し、コードベースの中でもっと扱いやすい場所へ移します。**テストできないオブジェクトを「控えめ」にする**ことで、厄介なバグを抱え込む可能性を下げられます。「humble」という言葉の使い方は、Michael Feathers の記事に由来します。

よくある例はユーザーインターフェイスです。プラットフォームによっては、UI コントロールに対して自動テストを走らせるための仕組みが用意されていないことがあります。たとえ仕組みがあっても、複雑なセットアップ、特別なフレームワーク、遅いテスト実行が必要になり、扱いづらくなることが少なくありません。

しかし多くの場合、UI コントロールに持たせる振る舞いを最小限にすることで、効果的にテストできます。Presentation Model、つまり MVVM や Passive View のようなパターンは、このために使える代表的な手段です。**難しい場所には判断を置かず、判断はテストしやすいモデル側へ寄せる**、というのが中心にある考え方です。

このアプローチを詳しく知るには、Gerard Meszaros の『xUnit Test Patterns』が重要な情報源です。同書の Humble Object の項目はオンラインでも読め、バリエーションや例を含めてより深く説明されています。
