# Feature Flag

## 要約

Feature Flag は、コードを変更せずに機能の表示や振る舞いを切り替えるための仕組みです。
未完成の機能を mainline に統合しながら、リリースでは見せないようにできます。

便利な一方で、フラグは複雑さを増やします。
使う場所を最小限にし、不要になったリリースフラグはすぐに削除することが重要です。

## 読むときの観点

- Feature Flag は Feature Branch の代替として語られることが多い。
- 主な用途は、未完成機能を隠す release toggle だけではない。
- 実験、運用制御、権限による公開範囲制御にも使われる。
- フラグ判定は、機能への入口に集中させる。
- リリースフラグは最後の手段であり、小さなリリースや Keystone Interface を先に考える。

## 原文の翻訳

Feature Branch を支持するよくある理由のひとつは、ひとつのリリースサイクルより長くかかる未完成機能を扱う仕組みを提供することだ。たとえば2週間ごとに本番リリースしているが、完成まで3か月かかる機能を作る必要があるとする。Continuous Integration を使って全員が mainline で作業しながら、半分だけ実装された機能をリリースで見せないようにするにはどうすればよいか。Feature Flag は、その問題に対処する便利な道具である。

基本的な考え方は、保留中の機能に対するフラグを構成ファイルなどで定義し、実行中のアプリケーションがそのフラグを見て新機能を表示するかどうかを決める、というものだ。多くの場合、この判断はアプリケーションのユーザーインターフェースで行われる。UI技術によって詳細は異なるが、未公開の要素をフラグで包み、オンのときだけ通すという発想は同じである。

新しい価格計算アルゴリズムのように、UI要素を持たない機能もある。この場合、フラグの判定はアプリケーションコード内で行われる。単純な条件分岐でもよいし、依存性注入で差し替えられた strategy のような、より洗練された実装でもよい。

フラグの判定は、新機能を適切に隠すために必要な最小限の Toggle Point にだけ置くべきである。たとえば複数画面からなる機能でも、ホーム画面のひとつのリンクからしか入れないなら、その入口だけを守ればよい。新機能のすべてのコードパスをフラグで守ろうとするのではなく、ユーザーがそこへ到達する入口を切り替えることに集中する。

フラグの作成、保守、削除に大きな時間がかかるなら、フラグ判定が多すぎる兆候である。単純な条件分岐は最も簡単な実装だが、多態的な差し替えなどを使って、フラグを評価する場所を減らすことを考えるべきだ。

ここまでの Feature Flag は、部分的に作られた機能を隠す release toggle として説明できる。しかし用途はそれだけではない。A/Bテストのための experiment toggle、運用担当者に制御手段を与える ops toggle、ユーザーの部分集合に機能アクセスを与える permissioning toggle などもある。

Feature Flag は実行時に設定されることが多いが、release toggle をビルド時に設定する例もある。ビルド時トグルの小さな利点は、新機能のコードがリリースされる実行ファイルにコンパイルされないことである。

Feature Toggle の危険のひとつは、意図しない露出である。誰かがUI要素をトグルで包むのを忘れると、隠すべきものが見えてしまう。これはテストしづらい。隠すべき個々の要素を列挙しないと「何も見えていない」ことを確認しにくく、忘れられる要素はテストでも忘れられやすいからだ。

Feature Flag を使うとテストの組み合わせ爆発が起きるのか、という質問もよくある。一般には、すべての組み合わせをテストする必要はない。release flag では、次のリリースでオンになる予定のフラグをオンにした組み合わせと、すべてのフラグをオンにした組み合わせを走らせれば十分なことが多い。

リリースフラグは、保留中の機能が本番で落ち着いたら必ず廃止することが非常に重要である。構成ファイルの定義と、それを使うコードを取り除く。そうしなければ、誰も使い方を覚えていないトグルが山のように残る。

Release Flag は有用で、多くのチームが使っている。しかし、本番へ機能を出すための手段としては最後の選択肢にするべきである。第一の選択肢は、機能を小さく分け、安全にプロダクトへ導入できるようにすることだ。小さく頻繁なリリースは、失敗のリスクを減らし、ユーザーが実際にどう使うかについて価値あるフィードバックを与えてくれる。

どうしても部分的に作られた機能を隠す必要があるなら、Keystone Interface を使うのがよい。UIの入口以外を作り、最後のリリースサイクルでその入口を追加する。こうすれば非UIコードは他のコードと統合済みになるが、最後の要素を追加するまでユーザーには見えず使われない。小さなリリースや Keystone Interface ができない場合にだけ、**release flag を使う**と考えるのがよい。
