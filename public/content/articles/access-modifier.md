# Access Modifier

## 要約

Access Modifierは、クラスの機能へ他のクラスがどのようにアクセスできるかを制御する言語機能です。C++、Java、C#、Smalltalkでは、その考え方や単位が少しずつ異なります。

この記事の中心は、アクセス制御は絶対的な防壁ではなく、クラスが何を自分の内部に保ちたいかを伝えるための仕組みだ、という点です。さらに、言語上の`public`よりも、外部チームや外部利用者に公開されたPublished Interfaceのほうが、実務では重要な境界になりうると述べています。

## 読むときの観点

- `public`、`protected`、`private`の意味は言語ごとに微妙に異なる。
- JavaのpackageやC#のassemblyのように、アクセス制御の単位にも注目する。
- Smalltalkでは、テキスト上のスコープではなくオブジェクト単位の見方が強い。
- アクセス制御は侵入を完全に防ぐものではなく、設計上の意図を伝える手段として読む。

## 原文の翻訳

オブジェクト指向言語は、プログラムをクラスと呼ばれるモジュールに分割します。それぞれのクラスはfeatureを持ち、それはデータ、つまりフィールドと、メソッドから成ります。すべての言語がこの用語を使うわけではありませんが、ここではこれで十分です。言語には、他のクラスがあるクラスのfeatureへどのようにアクセスできるかについて、さまざまな規則があります。多くの場合、それらはクラスに適用されるaccess modifierに基づいています。

### C++の選択

おそらく最も影響力のあるaccess modifierの組み合わせはC++から始まりました。C++には3つあります。

- public: どのクラスでもfeatureにアクセスできる
- protected: どのサブクラスでもfeatureにアクセスできる
- private: 他のクラスはfeatureにアクセスできない

クラスは`friend`キーワードを使って、別のクラスやメソッドにアクセス権を与えることもできます。そのため、C++ではfriend同士が互いのprivateな部分に触れられる、という冗談が生まれました。

### Java

JavaはC++を基礎にしました。Javaは言語にpackageという概念を追加し、それが振る舞いに影響しました。

- public: どのクラスでもアクセスできる
- package: デフォルトで、コード上ではキーワードを使わない。同じpackageのどのクラスでもアクセスできる
- protected: どのサブクラス、または同じpackageのどのクラスでもアクセスできる
- private: 他のクラスはアクセスできない

JavaのprotectedとC++のprotectedの間には、微妙な違いがあることに注意してください。話をややこしくするためにそうなっているようなものです。

### C#

C#もC++のモデルを基礎にしています。

- public: どのクラスでもアクセスできる
- internal: 同じassembly内のどのクラスでもアクセスできる。メソッドとクラスのデフォルトだが、明示することもできる
- protected: どのサブクラスでもアクセスできる
- protected internal: どのサブクラス、または同じassembly内のどのクラスでもアクセスできる
- private: 他のクラスはアクセスできない。フィールドのデフォルト

C#ではassemblyは物理的な構成単位であり、dll、jar、binaryに相当します。C#にはJavaのpackageに似た論理的な単位、namespaceもありますが、access modifierには関与しません。

### Smalltalk

Smalltalkはしばしば最も純粋なOO言語と考えられており、C++、Java、C#よりも古い言語です。アクセスを制御するためにキーワードを使わず、基本的な方針を使いました。Smalltalkerは、フィールドはprivateで、メソッドはpublicだと言うでしょう。

しかし、このprivateなフィールドは、C++系の言語で言うprivateと同じ意味ではありません。C++などでは、アクセスはテキスト上のスコープとして考えられます。PersonクラスのサブクラスであるProgrammerクラスがあり、MartinとKentという2つのインスタンスがある例を考えてみましょう。C++では、どちらのインスタンスも同じクラスなので、MartinはKentのprivateなfeatureにアクセスできます。

Smalltalkの世界観では、アクセスはオブジェクトに基づきます。したがってMartinとKentは異なるオブジェクトなので、MartinがKentのフィールドに手を出す筋合いはありません。しかし同時に、すべてがオブジェクトに基づいているため、Martinは、たとえPersonクラスで宣言されたフィールドであっても、自分自身のすべてのフィールドにはアクセスできます。つまりSmalltalkのデータはprivateよりprotectedに近いのですが、いずれにしてもオブジェクトスコープによって事情は異なります。

### アクセス制御はアクセスを制御しない

privateなフィールドがあるなら、他のクラスはそれにアクセスできない、という意味になるでしょうか。違います。本当にそうしたければ、ほとんどどの言語でもアクセス制御の仕組みを迂回できます。通常の抜け道はreflectionです。デバッガやその他のシステムツールはprivateデータを見る必要があることが多いため、たいていreflection interfaceはそれを許します。

C++にはこの種のreflectionはありませんが、C++は根本的にメモリが開かれているため、直接メモリ操作を使えば済みます。

アクセス制御の要点は、アクセスを防ぐことではありません。むしろ、クラスがいくつかのものを自分の内側に保ちたいと示すことにあります。access modifierを使うことは、プログラミングにおける多くのことと同じく、**主としてコミュニケーション**なのです。

### Published Methods

私は、もうひとつのaccess type、PublishedInterfaceが入る余地が本当にあると論じてきました。プロジェクトチーム内の他のクラスへ公開するfeatureと、他のチーム、たとえばAPIとして公開するfeatureとの間には、根本的な違いがあると思います。こうしたpublished featureはpublic featureの一部であり、異なる扱いを必要とします。そのため私は、publishedとpublicの区別は、publicとprivateの区別よりも重要だと考えるほどです。
