# Microservices and the First Law of Distributed Objects

## 要約

分散オブジェクトの第一法則は「オブジェクトを分散させてはならない」という警告です。
これは、ローカル呼び出しとリモート呼び出しを同じように扱えるという発想への批判から生まれました。

この記事は、マイクロサービスがこの法則と矛盾するのかを検討します。
マイクロサービスは分散を伴いますが、分散オブジェクトが目指した「プロセス内とリモートの透明性」を求めるものではありません。
それでも分散は複雑さを増やすため、モノリスを基本に考えつつ、実証的な成功例と未解決の不確実性を見て判断すべきだと述べています。

## 読むときの観点

- 「分散オブジェクト」が何を意味していたのかを確認する。
- ローカル呼び出しとリモート呼び出しの違いを隠せない理由を見る。
- マイクロサービスが第一法則の「文字」ではなく「精神」に反するのかを考える。
- 分散の複雑さと、実際の成功事例から得られる証拠の両方を読む。

## 原文の翻訳

『Patterns of Enterprise Application Architecture』を書いたとき、私は「分散オブジェクト設計の第一法則」と呼ぶものを作りました。
それは「**オブジェクトを分散させてはならない**」というものです。
ここ数か月、マイクロサービスへの関心が高まり、そのため何人かから、マイクロサービスはこの法則に反しているのではないか、もしそうならなぜ私がマイクロサービスを支持しているのか、と尋ねられました。

この第一法則の文で、私が「分散オブジェクト」という言葉を使っている点に注意することが重要です。
これは1990年代後半から2000年代初頭にかなり流行し、その後は正しくも人気を失った考え方を指しています。
分散オブジェクトの発想は、オブジェクトを設計し、その同じオブジェクトをプロセス内でもリモートでも使えるようにする、というものでした。
ここでリモートとは、同じマシン上の別プロセスかもしれませんし、別のマシン上のプロセスかもしれません。

DCOM や CORBA 実装のような賢いミドルウェアが、プロセス内かリモートかの違いを処理します。
そうすれば、システムを書いた後で、アプリケーションの設計とは独立してプロセスへ分割できる、という考えでした。

私が分散オブジェクトという考えに反対した理由は、オブジェクト境界の背後に多くのものをカプセル化できるとしても、**リモートかプロセス内かという違いはカプセル化できない**からです。
プロセス内の関数呼び出しは速く、常に成功します。
もちろん例外は起こりえますが、それはアプリケーションの都合によるものであり、呼び出しを行ったという事実そのものに由来するものではありません。

一方、リモート呼び出しは桁違いに遅く、リモートプロセスや接続の失敗によって呼び出しが失敗する可能性が常にあります。

この違いの結果として、API の指針も変わります。
プロセス内呼び出しなら細粒度でもかまいません。
100個の商品の価格と在庫状況が欲しければ、商品価格関数を100回、在庫状況関数をさらに100回、気軽に呼び出せます。
しかしその関数がリモート呼び出しなら、通常は100個すべての価格と在庫状況を一度に求める単一の呼び出しへまとめた方がよいでしょう。
その結果、商品オブジェクトのインターフェースはまったく違ったものになります。

したがって、同じクラスを、プロセス内でもリモートでも透明に使うことはできません。
クラスとは主にインターフェースに関わるものだからです。

私が話してきたマイクロサービス支持者たちは、この違いをよく理解しています。
彼らがプロセス内とリモートの透明性について語るのを聞いたことはありません。
つまり彼らは、分散オブジェクトがやろうとしていたことをしようとしていないので、第一法則には違反していません。
むしろ彼らは、HTTP 上の文書や軽量メッセージングによる、**粗粒度のやり取り**を支持しています。

要するに、分散オブジェクトに関する私の見解と、マイクロサービス支持者の見解との間に矛盾はありません。
しかし、この本質的な非衝突にもかかわらず、今や別の問いを避けて通れません。
マイクロサービスは、モノリスよりずっと多く、リモート接続で通信する小さな分散単位を意味します。
たとえ法則の字面には反していないとしても、第一法則の精神には反しているのではないでしょうか。

多くのシステムに分散設計を採る正当な理由があることは認めます。
しかし私は、分散は複雑さを増幅するものだと考えています。
粗粒度の API は、細粒度の API より扱いにくいものです。
リモート呼び出しの失敗にどう対処するか、そしてそれが一貫性と可用性へどのような結果をもたらすかを決めなければなりません。
プロトコル設計によってリモート呼び出しを最小化したとしても、それらをめぐる性能問題について、より多く考える必要があります。

モノリスを設計するときは、モジュール間の責務配分を気にしなければなりません。
分散システムでは、モジュール間の責務配分に加えて、**分散に関する要因**も気にしなければなりません。

小さなマイクロサービスは確かに理解しやすいものです。
しかし私は、そのことがサービス間の接続へ複雑さを押し出してしまうのではないかと心配しています。
そこでは複雑さがあまり明示的でなく、何かが壊れたときに原因を突き止めにくくなります。
リモート境界をまたいでリファクタリングしなければならない場合、リファクタリングはずっと難しくなります。
マイクロサービス支持者は、非同期通信によって結合が減ることを強調します。
しかし非同期性もまた、複雑さを増幅するものです。

Cookie-cutter scaling を使えば、分散の複雑さを増やさずに大量のトラフィックを扱えます。

したがって、私は分散に用心深く、基本的な傾向としてはモノリシックな設計を好みます。
では、それにもかかわらず、なぜ私はマイクロサービスについて多くの労力をかけて説明し、それを支持する同僚たちを支援してきたのでしょうか。
答えは、私の直感がいつも正しいとは限らないと知っているからです。

多くのチームがマイクロサービスのアプローチを採り、それで成功を見いだしていることは否定できません。
Netflix や、おそらく Amazon のようなよく知られた公の事例もありますし、Thoughtworks の内外で私が話をしたさまざまなチームもあります。
私は本来、経験主義者です。
理論が私の直感よりずっとよく練られたものであったとしても、**経験的な証拠は理論に勝る**と信じています。

もっとも、この件に決着がついたと思っているわけではありません。
ソフトウェアデリバリーにおいて、成功を見極めるのは非常に滑りやすいことです。
Netflix や Spotify のような組織が、マイクロサービスによる初期の成功を大きく語っている一方で、Etsy や Facebook のように、よりモノリシックなアーキテクチャで成功した例もあります。

あるチームがマイクロサービスで成功したとどれだけ考えていたとしても、本当に比較すべき相手は反事実です。
つまり、もしモノリスでやっていたら、もっと良い結果になっていただろうか、という問いです。
マイクロサービスのアプローチは比較的短い期間しか存在していないため、私たちが嫌っている古いモノリスと比較できる、10年もののレガシーなマイクロサービスアーキテクチャからの証拠はあまりありません。

さらに、まだ特定できていない要因によって、ある状況ではモノリスの方がよく、別の状況ではマイクロサービスが有利になる、ということもありえます。
ソフトウェア開発で証拠を集める難しさを考えると、何年も経った後でさえ、どちらか一方を強く支持する決定的な材料は得られない可能性の方が高いでしょう。

この不確実性を前提にすると、私のような書き手にできる最も重要なことは、私たちが学んだと思っている教訓を、たとえそれらが互いに矛盾していたとしても、できるだけ明確に伝えることです。
読者は自分で判断します。
アーキテクチャ上の線のどちら側に立つにせよ、その判断が十分な情報に基づいたものになるようにするのが、書き手としての私たちの仕事です。

### さらに読む

同僚の James Lewis と書いた記事は、マイクロサービスアーキテクチャスタイルの定義を準備しようとしたものです。
私は『Patterns of Enterprise Application Architecture』で分散オブジェクトの第一法則を作りました。
これは第7章「Distribution Strategies」に登場します。
その章は、Dr. Dobb's によって「Errant Architectures」という題名でオンラインにも無料公開されていました。

「Fallacies of Distributed Computing」は、分散を透明に、あるいは簡単に行えるという考えに注意すべき理由を示す古典的な声明です。
Waldo らの「Note on Distributed Computing」も、分散オブジェクトの根本的な問題を述べた優れた文章でした。
