# Infrastructure As Code

## 要約

Infrastructure as Codeは、コンピューティング基盤やネットワーク基盤の定義をソースコードとして扱う方法です。
コードとして管理することで、バージョン管理、レビュー、テスト、監査、再現可能な構築、Continuous Deliveryの実践をインフラにも適用できます。

クラウド時代の動的なインフラでは、サーバーの作成、構成、破棄をソフトウェアの命令で行えます。
この能力を安全に使うには、手作業の調整を避け、定義ファイル、テスト、小さな変更、継続的な可用性を重視する必要があります。

## 読むときの観点

- インフラを「作業手順」ではなく「実行可能な定義」として読む。
- Snowflake Serverを避けるために、手作業の変更をどう制限するかを見る。
- バージョン管理、テスト、監査、再現性が互いに支え合う構造を押さえる。
- Phoenix ServerやImmutable Serverとの相性に注目する。

## 原文の翻訳

Infrastructure as Codeは、コンピューティングやネットワークのインフラをソースコードで定義するアプローチです。そのコードは、他のソフトウェアシステムと同じように扱えます。ソース管理に置けば監査しやすくなり、Reproducible Buildを助け、テストの対象にでき、Continuous Deliveryの規律を適用できます。

この考え方は、成長するクラウド基盤を扱うために使われてきました。これからの時代には、コンピューティングインフラを扱う主要な方法になっていくでしょう。

かつて新しいサーバーアプリケーションをリリースするには、物理ハードウェアを探し、そのハードウェアをアプリケーションの必要に合わせて構成し、そこへアプリケーションをデプロイする必要がありました。ハードウェアの調達は高価で、時間もかかり、多くの場合は数か月単位の仕事でした。現在のクラウド時代では、新しいサーバーの起動は数秒で済み、必要なのはインターネット接続とクレジットカード程度です。

これは動的なインフラです。サーバーの作成、プロビジョニング、破棄を、ドライバーに触れることなくソフトウェアコマンドで行えます。対象は仮想マシンであることが多いですが、ベアメタルへのインストールにも適用できます。

### 実践

Infrastructure as Codeはいくつかの実践に基づきます。

まず、定義ファイルを使います。すべての構成は、シェルスクリプト、Ansible playbook、Chef recipe、Puppet manifestのような実行可能な構成定義ファイルに書きます。サーバーへログインして、その場限りの調整をしてはいけません。そのような手作業はSnowflake Serverを生みやすく、永続的な定義となるコードを開発している間だけに限定すべきです。コードを適用すれば更新が速く行える状態にします。

次に、システムとプロセスを自己文書化します。人間が手順書を読んで実行するより、コードは正確で一貫して実行されます。必要なら、人間向けの文書をそのコードから生成できます。

また、すべてをバージョン管理します。構成コードをソース管理に置けば、あらゆる構成と変更が記録され、監査できます。問題の診断に役立つ再現可能な構築も可能になります。

さらに、システムとプロセスを継続的にテストします。テストは、インフラ構成の多くの誤りをコンピュータに素早く見つけさせます。現代的なソフトウェアシステムと同じように、インフラコードにもDeployment Pipelineを用意でき、インフラ変更にもContinuous Deliveryを実践できます。

大きな一括変更ではなく、小さな変更を行います。インフラ更新が大きいほど誤りを含みやすく、複数の誤りが絡むと検出も難しくなります。小さな更新は誤りを見つけやすく、戻しやすいものです。インフラ変更でも、頻度を上げることで難しさを下げられます。

最後に、サービスを継続的に利用可能に保ちます。多くのシステムでは、アップグレードや修正のための停止を許容できません。Blue-Green DeploymentやParallel Changeのような技法を使えば、可用性を失わずに小さな更新を行えます。

### 利点

これらの実践により、動的インフラの利点を活かせます。新しいサーバーを簡単に起動し、より新しい構成へ置き換えられたサーバーや負荷低下で不要になったサーバーを安全に破棄できます。必要な数だけサーバーインスタンスを作るには、スクリプトを実行すればよいのです。このアプローチはPhoenix ServerやImmutable Serverと相性がよいものです。

サーバー構成をコードで定義すると、サーバー間の一貫性が高まります。手作業のプロビジョニングでは、曖昧な指示の解釈違いや単純なミスにより、微妙に異なる構成のSnowflake Serverが生まれます。その違いは、デバッグしにくい障害につながります。監視設定も不一致になりやすいですが、コードを使えば監視も一貫させられます。

何より重要なのは、構成コードによって変更が安全になることです。アプリケーションやシステムソフトウェアを、より低いリスクで更新できます。障害は素早く見つけて修正でき、最悪の場合でも最後に動いていた構成へ戻せます。

インフラをバージョン管理されたコードとして定義すると、コンプライアンスや監査にも役立ちます。構成へのすべての変更を記録でき、人間の記録ミスに左右されません。

扱うサーバーが増えるほど、この重要性は増します。Microservicesを本格的に採用するなら、Infrastructure as Codeは必要な能力になります。サーバーの構成だけでなく、サーバー同士がどう関わるかを指定する点でも、大きなクラスタを効果的に管理できます。

Infrastructure as Codeの核は、インフラを手作業で育てることではなく、**ソフトウェアと同じ規律で変更し続けられる対象にする**ことです。
