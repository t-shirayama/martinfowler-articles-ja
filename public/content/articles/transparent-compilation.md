# Transparent Compilation

## 要約

Transparent Compilation は、CoffeeScript や SCSS のように、別のテキスト形式のソースへ変換される言語で、生成された出力コードを開発者が読み、デバッグ時にも対応関係を追えることを前提にしたコンパイルのあり方です。

## 読むときの観点

- ソース言語とターゲット言語の対応関係が、デバッグ体験にどう影響するかを見る。
- 透明性を保つために、ソース言語の表現力へどんな制約がかかるかを考える。
- SCSS や TypeScript のようなスーパーセット型の言語と、CoffeeScript のように構文を大きく変える言語の違いを比べる。

## 原文の翻訳

Web 開発者は、CoffeeScript や SCSS のように、ブラウザで実行される別のテキスト形式のソース言語へコンパイルされる言語を使うことが増えています。こうした source-to-source compiler、つまり transpiler は新しいものではありません。C++ の初期には、Cfront が C++ から C のコードを生成するために広く使われていました。ただし、CoffeeScript や SCSS には、それらを透明なコンパイラとして区別したくなる特徴があります。

多くのコンパイラでは、下流で生成されるものをそれほど気にしません。ソース言語の意味に従っているかぎり、それは実質的に大きなビット列のかたまりです。しかし、ブラウザ用の JavaScript を生成する場合、その無関心は保ちにくくなります。デバッグ環境はずいぶん洗練されてきましたが、それらは基本的に HTML、CSS、JavaScript の三点セットを前提にしています。そのため、入力言語が実行されるターゲットへどう変換されるかを理解していることが重要になります。

この制約は、ソース言語に大きな影響を与えます。出力がソースにかなり明確に対応している必要があるからです。CoffeeScript で短いイベントハンドラを書いたとき、ブラウザのデバッガに現れる JavaScript を容易に認識できるなら、その対応関係は保たれています。より複雑な変換、たとえばメソッド定義や `switch`、存在チェックを JavaScript のプロトタイプや一時変数を使ったコードへ変える場合でも、元の CoffeeScript と生成された JavaScript の対応が見えるなら、デバッグ時に迷いにくくなります。

ここに、コンパイルプロセスを透明にする本質があります。**出力言語でも作業することを意図している**、という点です。

もちろん、不透明なコンパイルでも出力を調べることはあります。奇妙な振る舞いやバグを追うために、コンパイラの出力へ潜る必要がある場合もありますし、コンパイラが何をしているかを理解したがるプログラマもいます。しかし、そのような活動は例外です。

これに対して、出力言語で作業することを期待していない source-to-source compiler もあります。あるいは、出力言語が見えてしまうのは一時的で残念な仕組みだと考えるものもあります。Dart、GWT、ClojureScript のような JavaScript 世界の言語は、この方向でも有用です。透明な transpilation とより一般的なアプローチを分けるのは、この意図の違いです。

透明性を維持しなければならないという事実は、ソース言語でできることに制限をかけます。より制約の少ないコンパイルに比べると、言語構文の自由度は小さくなります。ターゲット言語の基本的な意味論に従い、おおむね同じプログラム構造を保たなければなりません。こうした制約は、言語設計の特徴としてはあまり語られてきませんでした。

CoffeeScript は、こうした制約があっても、ソース言語とターゲット言語の構文をかなり違うものにできることを示しています。CoffeeScript の構文は、C 系の JavaScript よりも Python に近い感触があります。ただし、すべての透明なコンパイル言語が大きく構文を変えるわけではありません。重要な一群として、ターゲット言語のスーパーセットであろうとするものがあります。SCSS や TypeScript はこの分類に入ります。CSS の式は SCSS でも有効です。スーパーセットの言語を使うと対応関係はいっそう明確になり、CSS のように構文自体はうまく機能しているが便利な機能が欠けている場合には、特によく合います。

透明なコンパイルを使う意味は小さい、という意見もあります。デバッグのためにターゲットコードを理解しなければならないなら、別のソース言語を使う価値はどこにあるのか、というわけです。私にとって、その価値はいくつかの方向にあります。まず、ターゲット言語に欠けている便利な言語機能を得る手段になります。SCSS なら、色の値を直接何度も書く代わりに変数を使えます。調整したいときも一箇所を変えれば済みます。

CoffeeScript のように構文を大きく変える場合には、より強い理由が必要です。ある同僚は、プロジェクトを終えたあとにこの点をうまく表現していました。彼は経験豊富な JavaScript プログラマで、そのプロジェクトでは最初からよく規律づけられた JavaScript を書いていました。その結果、JavaScript コードベースの品質には満足していました。それでもなお、CoffeeScript で作業したほうがよかったと結論づけました。生成された JavaScript をデバッグしているときでさえ、CoffeeScript を読むほうが何が起きているかを理解しやすいからです。

小さな断片だけを見ると、この変換は大した違いに見えないかもしれません。しかし、数百行、ましてそれ以上のコードになれば大きな違いになります。**透明なコンパイルの価値は、出力を理解できるまま、入力側の読みやすさや便利さを得ること**にあります。
