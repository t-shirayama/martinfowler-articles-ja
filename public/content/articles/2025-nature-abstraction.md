# LLMs bring new nature of abstraction

## 要約

LLM は、ソフトウェア開発における抽象化の性質を大きく変えつつあります。アセンブラから高水準言語への移行が抽象化レベルを大きく引き上げたように、prompt によって機械へ指示する開発は、Ruby などの高水準言語からさらに別の段階へ進む変化です。

ただし今回の変化は、抽象化レベルが上がるだけではありません。LLM は非決定的な抽象化を導入します。prompt を git に保存しても、同じ振る舞いが毎回得られるとは限りません。私たちは、抽象化の上方向だけでなく、非決定性という横方向への移動にも向き合う必要があります。

## 読むときの観点

- LLM を単なる新しいツールではなく、プログラミングの抽象化を変えるものとして読む。
- 高水準言語がアセンブラから何を隠したのかを思い出し、prompt が何を隠すのかを考える。
- 従来の言語やフレームワークの進化と、LLM による非決定性の導入を分けて捉える。
- prompt をコードのように保存しても、同じ再現性を期待できない点に注意する。

## 原文の翻訳

この分野で声の大きい人の多くと同じように、私も生成 AI システムがソフトウェア開発で果たす役割に大きな関心を寄せています。LLM の登場は、アセンブラから最初の高水準プログラミング言語へ移ったときと同じくらい、ソフトウェア開発を変えるだろうと考えています。その後の言語やフレームワークの発展は、抽象化レベルと生産性を高めました。しかし、それはプログラミングの性質そのものを変えるほどではありませんでした。LLM はその規模の影響をもたらしつつあります。ただし違いは、抽象化レベルを上げるだけでなく、非決定的な道具でプログラムするとはどういうことかを考えさせる点にあります。

高水準言語は、根本的に新しい抽象化レベルをもたらしました。アセンブラでは、特定の機械の命令セットについて考えます。単純な動作でさえ、適切な register にデータを移し、特定の命令を呼び出す方法を考えなければなりません。高水準言語によって、私は statement の並び、選択のための conditional、データ値の集まりへ繰り返し statement を適用する iteration という形で考えられるようになりました。コードの多くの場所に名前を導入し、値が何を表すのかを明確にできます。

初期の言語にはもちろん制約がありました。私が最初に職業として使った言語は Fortran IV で、`IF` 文には `ELSE` 節がなく、整数変数の名前は `I` から `N` で始めることを覚えておく必要がありました。

こうした制約が緩み、block structure が得られるようになったことは、プログラミングを楽にし、楽しくもしました。しかしそれは同じ種類の改善でした。今では私はほとんど loop を書かず、関数をデータとして渡すことも自然に行います。それでも、Dorset の荒野で Fortran を使っていた頃と、機械への話しかけ方は似ています。Ruby は Fortran よりはるかに洗練された言語ですが、雰囲気としては同じ系統にあります。Fortran と PDP-11 の機械語命令との違いほどには、断絶していません。

私はまだ最高の生成 AI ツールを本格的に使い込む機会はなく、触ってみる程度にとどまっています。それでも、友人や同僚が経験を共有するのを聞くと強く引きつけられます。これはまた別の根本的な変化だと確信しています。prompt で機械に話しかけることは、Ruby と比べて、Fortran がアセンブラと違っていたのと同じくらい違います。

しかしこれは、抽象化の大きな飛躍以上のものです。Fortran の関数を書いたとき、私はそれを100回コンパイルしても、まったく同じバグを持った結果が現れると分かっていました。Large Language Model は **非決定的な抽象化** を導入します。そのため、prompt を git に保存しておけば毎回同じ振る舞いが得られる、と考えることはできません。同僚の Birgitta が言ったように、私たちは抽象化レベルを上へ移動しているだけではなく、同時に非決定性のほうへ横にも移動しているのです。

仕事で LLM を使うことを学ぶにつれて、私たちはこの非決定性とどう暮らすかを考えなければなりません。この変化は劇的で、私はかなりわくわくしています。失うものの中には残念に思うものもあるでしょうが、まだほとんどの人が理解していない得るものもあるはずです。非決定性へ向かうこの進化は、私たちの職業の歴史に前例のないものです。
