# Organizing Presentation Logic

## 要約

プレゼンテーション層を設計するとき、Separated Presentationによってドメインと表示を分けたあとには、プレゼンテーションロジック自体をどう整理するかが問題になります。単純な画面なら1つのクラスで十分なこともありますが、複雑な画面ではAutonomous View、Supervising Controller、Passive View、Presentation Modelなどの選択肢が出てきます。

この記事は、テストしやすさ、複数viewの支援、画面状態とセッション状態とレコード状態の同期、複数画面間の同期、Observer利用時の注意点を整理しています。特に、同期の深さと頻度、Flow SynchronizationとObserver Synchronizationの使い分けが中心です。

対応範囲: 原文本文を翻訳しています。原文がdraft formである旨は本文に含めています。謝辞、重要な改訂、改訂履歴は省略しています。

## 読むときの観点

- プレゼンテーションロジックを分ける主な理由は、GUIを直接動かさずにテストすることにある。
- 画面の同期は、screen state、session state、record stateという複数の状態のコピーをどう扱うかとして読む。
- 同期には頻度、深さ、粒度があり、それぞれが設計とユーザー操作に影響する。
- Observerは複雑なUIでは有力だが、暗黙の制御フローとメモリ管理の問題を持つ。

## 原文の翻訳

プレゼンテーションのロジックを分割する方法はいくつかあります。

これは、私が2000年代半ばに取り組んでいたFurther Enterprise Application Architecture developmentの一部です。残念ながら、その後あまりにも多くのことが私の注意を奪い、これ以上進める時間を取れませんでした。近い将来に時間が取れる見込みもあまりありません。そのため、この資料はかなりdraft formのままであり、再び取り組む時間を見つけるまでは、修正や更新を行う予定はありません。

どのようなpresentation layerを設計する場合でも、役に立つことのひとつはSeparated Presentationを徹底することです。それを行ったら、次の段階は、presentation logic自体をどのように整理するかを考えることです。単純なwindowなら、1つのclassで十分かもしれません。しかし、より複雑なlogicでは、より幅広い分割方法が必要になります。

もっとも一般的なアプローチは、アプリケーション内の各windowに対して1つのclassを設計することです。このclassは通常、GUI libraryのwindow classを継承し、そのwindowを扱うために必要なすべてのcodeを含みます。windowが複雑なpanelを含む場合、そのpanel用に別のclassを持つこともあります。これはcomposite構造につながります。ここでは、その種のcompositeな構成には踏み込みません。それはかなりわかりやすいからです。代わりに、1つのwindow内の基本的な振る舞いを整理する方法に焦点を当てます。

### Viewからpresentation logicを分ける

すべてのpresentation logicをAutonomous Viewに置くことは、一般的であり実用にも耐えますが、欠点もあります。今日Autonomous Viewについてもっともよく語られる欠点は、テストに関するものです。GUI windowを通してpresentationをテストするのは、しばしば扱いにくく、場合によっては不可能です。GUIを操作する何らかのUI driverを作らなければなりません。

生のmouse eventやkeyboard eventを模倣するGUI toolを使う人もいます。しかし、そうしたtoolはたいてい脆いtestを作り、presentationに小さな変更があるたびに誤検知を起こします。より詳細なtoolはUI controlをより直接扱うため、脆さは小さくなりますが、それでも厄介です。また、GUI frameworkがAPIを通じて直接controlへアクセスする十分な支援を持っていることにも依存します。そして、すべてがそうではありません。

その結果、programmer-based testingを支持する多くの人は、humble dialog、あるいはultra-thin GUIを勧めます。ここでの中心的な考えは、すべてのlogicを他のpresentation layer classへ移すことで、UI controlを含むclassをできるだけ小さく、単純にすることです。このGUI control classは、理由はすぐ述べますが、通常viewと呼ばれます。

そうすれば、必要ならhumble viewをstubし、GUI controlを使わずに、知的なclassに対してtestを実行できます。viewはとても単純なので、問題が起きる余地は少なく、ほとんどのbugは知的なclassに取り組むことで見つけられます。私はこのstyleのtestingをsubcutaneous testingと呼ぶのが好きです。testがapplicationの皮膚のすぐ下で動作するからです。

現在、presentation classを分割する主な理由はsubcutaneous testingですが、そのような分割を検討する価値がある理由は他にもいくつかあります。知的なclassは、controlの選択、controlのlayout、場合によっては正確なUI frameworkそのものといったviewのいくつかの側面から独立させられます。これにより、同じ論理的振る舞いに対して、複数の異なるviewを支援できます。

これはapplicationの複数のskinを支える場合には有用ですが、humble viewを置き換えるだけで実現できるvariationは限られます。

ある面では、presentation logicを分離するとpresentationのprogrammingが楽になります。振る舞いを書いている間、view layoutの詳細を無視できるからです。実質的には、viewのcontrolに対してより扱いやすいAPIを得ることになります。

一方で、presentation logicを分離すると、その分離を支える追加の仕組みが必要になります。その性質は、どのpatternを使うかによって変わります。その結果、分離はpresentationを単純にするとも、複雑さを増すとも言え、どちらにも合理的な議論があります。

この分割には歴史的な前例があります。Model View Controller、つまりMVCの一部だったのです。『Patterns of Enterprise Application Architecture』で議論したように、MVC approachは2つの分離を行いました。もっとも重要な分離はSeparated Presentationであり、modelをview/controllerから分けることでした。もうひとつの分離、つまりviewとcontrollerの分離は、rich client GUI frameworkではあまり人気がありませんでした。ただし、web-based user interfaceでは復活しました。

MVCでは、viewはmodel内の情報を単純に表示し、controllerがさまざまなuser input eventを扱いました。これは多くのGUI frameworkとはうまく合いません。GUI frameworkは、UI controlが表示とuser input eventの受け取りの両方を行うよう設計されているからです。

humble viewを作るには、設計上、すべての振る舞いをviewの外へ移さなければなりません。user eventの処理も、domain informationのdisplay logicもです。これには大きく2つの方法があります。ひとつはModel-View-Presenter styleで、振る舞いをpresenterへ移します。presenterはcontrollerの一種と考えられます。presenterはuser eventを扱い、viewの更新にも一定の役割を持ちます。Supervising ControllerとPassive Viewは、このapproachの2つのstyleです。

Supervising Controllerは単純なview logicをviewに置き、Passive Viewはすべてのview logicをcontrollerに置きます。もうひとつのstyleはPresentation Modelです。これは、すべてのdataとbehaviorを捉える一種のmodelを作り、viewには単純な同期だけを必要とさせます。

どちらのstyleでも、viewはuser eventの最初のhandlerですが、すぐに制御をcontrollerへ引き渡します。

これら3つのpatternはいずれも、追加のclassを導入するため、議論の余地はありますが、より複雑な設計になります。多くを行いすぎるclassを、それぞれの責務を満たす別々のclassに分けるのはよいpracticeです。しかし問題は、Autonomous Viewが本当に複雑すぎるのかどうかです。確かに他のpatternは、別のtesting optionと、複数viewを支援する能力を提供します。

複数viewを必要とせず、viewを通したtestingで満足できるなら、特にwindowがあまり複雑でない場合、Autonomous Viewで十分かもしれません。

Presentation Model、Supervising Controller、Passive Viewのどれを選ぶかは、より恣意的です。結局のところ、自分のGUI環境でそのpatternをどれだけ簡単に実現できるか、そして個人的な好みによります。

MVC由来の用語を使うと、modelとは何かという問いが避けられません。classic Smalltalk MVCでは、modelはDomain Modelでした。今日のMVCの議論では、modelは一般にdomain layerへのinterfaceを意味します。それはclassic domain modelかもしれませんし、Service Layer、Transaction Script、Table Module、その他どんなdomain表現かもしれません。独立したdomain layerがない場合、modelはdatabaseへのinterfaceであることさえあります。

### 画面、layer、data

ほとんどのenterprise applicationはdataの編集を伴います。このdataは通常、applicationの複数layerの間でcopyされ、おそらく同じsystemを使う複数userの間でも扱われます。enterprise applicationの振る舞いの多くは、このdataの変更をどのように調整し、layer間でどのように同期するかに依存します。これについて考えるための一般に受け入れられた用語はありません。そこで、この本では次の用語を使うことにします。

### dataのlayer

最初に考えるべき問題は、異なるlayerにあるdataの異なるcopyです。物理的に考えると、通常はmemory上のdataとdatabaseやfile内のdataの違いがあります。これはtransient dataとpersistent dataの違いと考えられます。しかし、たいていはそれ以上のものがあります。memory上のdataでさえ、しばしば2か所に存在します。画面上のdataと、その画面を支える何らかのin-memory store内のdataとの違いが見つかることがよくあります。

これはdatabaseから取得したRecord Set、ただしまだdatabaseへcommitしていないものかもしれませんし、Domain Model内のdataかもしれません。

word processorでdocumentを扱う場合を考えてください。documentはdisk上にあり、あなたはそれをword processorで開き、documentのtextを編集しました。すると、memory上のtextはdisk上のものと異なります。次に、一部のtextのformattingを変更するdialog boxを開きます。多くの場合、dialog box内ではformattingを変更できますが、apply buttonを押すまでは基礎にあるtextは変わりません。

dialog box内のformatting dataはin-memory dataですが、mainのin-memory document内のdataとは異なるdataです。

この本で使う用語は、screen state、session state、record stateです。Screen Stateはuser interfaceに表示されるdataです。Session Stateは、userが現在作業しているdataです。session stateはuserにとってやや一時的なものに見え、通常、作業を保存することも破棄することもできます。Record Stateは、より永続的なdata、つまりsessionをまたいで残ることが期待されるdataを指します。

Session stateは主にmemory上で操作されますが、diskへ保存されることもよくあります。現代のword processorは、停電やsystem crashによる作業喪失を避けるため、rescue fileを保存することがよくあります。enterprise applicationでは、session stateをlocalのcheckpoint fileに保存したり、request間でserverがstateをdiskに保存したりできます。

enterprise applicationでsession stateとrecord stateを特に区別する点は、record stateがsystemの複数userの間で共有される一方、session stateはそれを作業しているuserだけに見えるprivate stateであることです。したがってuserは、変更をより永続的な形に保存するかどうかを決めているだけではありません。それを同僚と共有するかどうかも決めています。

Session stateはしばしば単一のbusiness transactionと対応しますが、多くの場合、複数のsystem transactionにまたがります。これはしばしばoffline concurrencyを必要とする状況です。

すべてのapplicationがsession stateを持つわけではありません。screen stateとrecord stateだけを持つものもあります。dataに対する変更は、保存されると直接record stateへ行きます。このような場合、session stateをまったく持たないか、変更が即座にrecord stateへ書き込まれるため、session stateは常にrecord stateと完全に同期しています。session stateを持たないことは、session stateの管理を気にしなくてよくなるため、applicationを大きく単純化します。

多くのapplicationでは、userが作業を失う心配をしなくてよいため、userにも好まれます。しかしsession stateを省くことは、よいことばかりではありません。userは作業中のstateであるscenarioを試し、気に入らなければ破棄する能力を失います。また、multi-user applicationで人々が独立して作業することも妨げます。

stateのlayerはさらに増えることもあります。たとえばsession stateがclient tierとserver tierの両方に保存される場合です。これらのstateは独立して変化できますが、通常は同期方法についてかなり制約的な規則があり、そのおかげで管理はずっと単純になります。

追加layerの例として、developerがteamで作業する方法があります。この場合、record stateは共有source code repositoryの状態です。developerのlocal machine上のworking copyは、disk上にあるsession stateの一形態です。この場合、それ自体もややtransientです。さらにIDE内に他の表現があります。現代的なIDEでは、syntax treeをmemory内に保持し継続的に更新するので、それに加えて画面に表示されるtextがあります。

この場合は3つを超えるlayerがありますが、それでもdeveloperが画面上で見ているもの、privateなsession data、共有されたpermanent dataを考えることは有用です。効果的に推論するために、私は各data集合に名前を付け、それらを別々のlayerとして話すでしょう。個々のapplicationは常に独自の集合を持ちますが、ここでの議論では、screen、session、record stateという一般的な三つ組みに焦点を当てます。

たいていの場合、userは一度にひとつのsessionで作業します。ときには、userが複数のsessionを同時に操作することもあります。これはしばしば混乱を招きます。片方のsessionでの変更は、両方がrecord stateと同期されるまで、もう片方のsessionには現れないからです。2つのsessionを同期すれば回避できますが、たいてい面倒です。

こうした複数のstateは、enterprise applicationのさまざまなlayerに対応することがよくあります。presentation、domain、data sourceのlayerを使う理想的なapplicationでは、domain logicはsession stateに対してだけ動作します。実際には、この区別は濁りがちです。たいていはよくない理由からですが、ときにはよい理由からそうなることもあります。

domain layerがpresentationとは別processにあるapplicationでは、applicationを適切に応答的にするため、domain logicの一部をpresentation processで実行したいことがあります。そのようなlogicでは、mainのdomain processからsession stateの一部をcopyしたり、presentationのcontrol内のdataに対してdomain logicを実行したりしなければならないかもしれません。

同様に、大量のdataを操作する必要がある場合、stored procedureのようなものを通じて、domain logicをdatabaseへ埋め込む必要があるかもしれません。そのようなlogicはrecord stateに対して動作します。しかし、ほとんどの場合、domain logicはsession stateに対して動作させたいものです。

### layer間の同期

こうした異なるcontext間でdataを同期することは、applicationを構築するうえで重要な部分です。user interfaceで作業しているとき、screen stateは2つの深さに同期できます。session stateかrecord stateです。session stateへ同期するなら、userがsession stateをrecord stateへ保存できるようにするcontrolが必要です。

同期はさまざまな頻度で起こります。私は次の3つが有用だと考えています。Key Synchronizationは、key pressやmouse clickごとに同期することを意味します。Field Synchronizationは、fieldの編集を終えたときに同期することを意味します。Screen Synchronizationは、画面いっぱいの情報を扱い終えたときに、UI上の特別なbutton、通常はapply、OK、cancel、submitなどと書かれたbuttonで同期することを意味します。

同期が必要になったら、次の問いはどれだけ同期するかです。screen dataとsession dataを同期する場合、私は主に2つの方式を見ています。Coarse-grained Synchronizationでは、UI上で何かを変更するたびにUI全体を同期します。つまりartist fieldを変更すると、他に変える必要が何もなくても、window全体を同期します。Fine-grained Synchronizationでは、本当に更新が必要なfieldだけを変更します。

たとえばtitle fieldを変更すると、title field、window title、list boxを同期しますが、それ以外は同期しません。

session dataとrecord dataの同期では、通常は異なるapproachを使います。session dataは普通、複数人に同時利用されるものではないので、concurrency issueを心配する必要はありません。sessionとrecord dataの同期は通常screen synchronizationで起こり、たいてい時間も長くかかります。その結果、data elementをdirtyとmarkしたり、Unit of Workを使ったりします。

これらの側面はすべて、UIの内部設計とinteraction designの両方でtrade-offになります。もっとも明らかなtrade-offは、同期の頻度と深さの間で起こります。key synchronizationをrecord stateまで行うと、許容できない性能、その他の問題につながります。そのため、その深さではほとんどの場合screen synchronizationが使われます。実際、sessionの深さでもscreen synchronizationがもっとも一般的です。

通常それがもっとも簡単で、多くのapplicationがそのように動くため、userも慣れています。しかし、interaction designが本当にfield synchronizationを必要とすることもかなりあります。domain logicがpresentationと同じprocess内にあるならfield synchronizationはかなり簡単ですが、別processにある場合に良い性能を得るのはずっと厄介です。したがってdomain layerについては、screen synchronizationを妥当なdefaultとしつつ、field synchronizationもかなり頻繁に必要になると見込むべきです。

Key synchronizationはよりまれなようですが、domainが同じprocess内にあるならかなり簡単に実現できます。

timingの選択は深さやapplication designによって変わりますが、screen stateとsession stateの間では、私はほとんど常にcoarse-grained synchronizationを好みます。多くの人は性能への影響を心配して、coarse-grained synchronizationを避けます。しかしfine-grained synchronizationは保守がとても大変です。codeが多くなり、重複も頻繁に生じるからです。そうした中のbugは見つけにくく、したがって修正もしにくくなります。

ほとんどの場合、coarse-grained synchronizationで十分に高性能です。したがって私の助言は、まず常にそれを使うことです。性能問題にぶつかり、profileによって本当に同期が原因だと確認したなら、そのときは問題を直すために少量のfine-grained synchronizationを導入しなければなりません。その時点では、性能問題に対処するために必要な最小限だけを行います。

同期の必要性はあまりに一般的なので、人々がそれを処理しようとするframeworkを開発するのは避けられません。多くの注目を集めているもののひとつが.NETのdata binding frameworkで、screen stateとsession stateを自動的に同期します。data bindingには多くの良い性質があり、理論上は同期をうまく扱えるはずです。これまでのところ、version 1.0まででは、単純なcaseではうまく動くものの、中程度に複雑なcaseでは崩れると私は見ています。

data bindingから始めたprojectの人々と話すと、bindingの動作を制御する方法が十分になかったため、しばらくして使うのをやめたという例がありました。そのため、必要が非常に単純でない限り、慎重に扱うよう助言します。ただし、後のversionでは再評価してください。これが同期問題に対する非常に有効な解へ変わることは、十分にあり得ます。

### 同期と複数画面

同期の一部はstateのlayer間を同期することです。もう一部は、同じlayerにある複数の流れを同期することです。単一のrecord stateの上に複数のsessionがあり、各sessionの上に複数のscreenがあることはよくあります。それぞれが別々のcontextであり、あるcontextでの変更が他のcontextへどのように伝播するかを考えなければなりません。

ここではpresentationについて話しているので、複数sessionの同期についてはあまり述べません。いずれにしても、それはよりよく理解されていて比較的単純な話題です。ほとんどの場合、sessionは互いに分離されており、record stateとだけ同期します。そのためにtransactionや何らかのoffline concurrency controlを使います。

presentationはより複雑です。userはより少ない分離と、より素早い同期を期待するからです。

複数screenをどのように同期するのが最善かは、screenがどのように構成され、screen間のflowがどのように構造化されているかに大きく依存します。2つの極端な例として、wizardと、file system explorerのような完全にnon-modalなinterfaceを対比して考えられます。

wizard user-interfaceでは、systemがuserを非常に制御されたscreenのflowに沿って導きます。どの時点でも見えるscreenは1つだけで、通常userは各screenから戻るか進むかしかできません。この状況では、screenのdesignerは、どのdataが表示されているか、screenがいつ開かれ閉じられるかを正確に知っています。

file system explorerでは、userはscreen間を自由に移動できます。さらに重要なことに、userは同じfileを表示する複数のexplorer windowを開けます。あるwindowでfolder名を変更したら、他のwindowも更新されるべきです。UI programmerは、windowがいつ開かれるか、同じdataが複数windowに表示されているかどうかを、実際には決して確信できません。

この2つの極端な例は、screen間で情報を調整する2つの異なる方法を示唆します。Flow Synchronizationでは、各screenがapplicationのflowに基づいて、screen stateを基礎となるsession stateといつ同期するかを決めます。たとえばwizardでは、screenから別のscreenへ移るときに同期するのが典型です。古いscreenを書き出し、新しいscreen用のdataを読み込みます。

Flow Synchronizationは、screen間のflowが単純で、screen stateからsession stateへdataを保存したり読み込んだりする明確な点がある場合にもっともよく機能します。

file explorerではFlow Synchronizationは難しいでしょう。あるscreenは、別のscreenが基礎dataを変更したかどうかを本当には知ることができません。この場合、screen同士は互いを意識せず、基礎dataが変わるたびに同期する必要があります。Observer Synchronizationでは、基礎となるscreen stateがdataのmaster sourceとして振る舞います。

それが変わるたびに、表示しているscreenへ通知され、screen stateを更新できます。通常はObserver patternを使います。この形では、Observer SynchronizationはModel View Controller styleの根本的な部分です。

Observer Synchronizationの良いところは、すべてのscreenが常に完全に互いから独立していることです。同期のために互いを知る必要も、同期eventを互いに知らせる必要もありません。そのため、application内に非常にad-hocで複雑なflowを持つことが簡単になります。

Observer Synchronizationの欠点は、Observerの利用に依存することです。そしてそれは暗黙の振る舞いを導入します。制御不能にしてしまうと、非常に厄介になり得ます。

全体としては、Observer Synchronizationが複雑なUIでは支配的な選択です。Flow Synchronizationが本当に使いやすいのは、application flowが非常に単純な場合だけです。通常は一度に1つのactive screenで、screen間のflowも単純な場合です。それでも、Observer Synchronizationに慣れると、こうした単純なcaseでもそれを使いたくなるかもしれません。

### Observerの落とし穴

rich client presentationにおける多くのinteractionは、Observer patternを利用します。Observerは有用なpatternですが、知っておくべき重要な問題もいくつか伴います。

observerの大きな強みであり弱みでもあるのは、制御がsubjectからobserverへ暗黙に渡ることです。codeを読んでもobserverが発火することはわかりません。何が起きているかを見る唯一の方法はdebuggerを使うことです。その結果、複雑なobserver chainは、理解、変更、debugが悪夢のようになり得ます。actionが別のactionをtriggerしても、なぜそうなるのかの手がかりがほとんどないからです。そのため私は、observerは非常に単純な形でだけ使うことを強く勧めます。

- objectが別のobjectをobserveし、それがさらに別のobjectをobserveするようなchainを作らない。observer関係は1 layerが最善です。ただしEvent Aggregatorを使う場合は別です。
- 同じlayer内のobject同士でobserver関係を持たない。domain objectは他のdomain objectをobserveすべきではなく、presentationは他のpresentationをobserveすべきではありません。observerはlayer境界をまたいで使うのが最善であり、典型的な使い方はpresentationがdomainをobserveすることです。

observerに関するもうひとつの問題はmemory managementにあります。いくつかのscreenがいくつかのdomain objectをobserveしているとします。screenを閉じたら、それを削除したいものです。しかしdomain objectは、observer関係を通じて実際にはscreenへのreferenceを持っています。memory-managed environmentでは、長生きするdomain objectが大量のzombie screenを保持し、大きなmemory leakを起こすことがあります。そのため、observerを削除したいときには、subjectからde-registerすることが重要です。

domain objectを削除したい場合にも、似た問題がよく起こります。domain object間のlinkをすべて切れば十分だと考えていても、screenがdomainをobserveしている可能性があるため、それでは足りないかもしれません。実際には、screenは去っていき、domain objectのlifetimeは通常data source layerを通じて制御されるため、この問題はそれほど頻繁には起こりません。しかし一般には、observer relationshipは忘れられたまま残ることが多く、zombieのよくある原因であることを覚えておく価値があります。

Event Aggregatorを使うと、これらの関係はしばしば単純になります。万能薬ではありませんが、扱いやすくしてくれます。
