# More Version Control

## 要約

バージョン管理をソフトウェア開発以外の一般的な作業にも広げられないか、という短い考察です。文書や図面など、日常的なファイル作業でも diff や merge が使えるようになれば、共同作業の質を高められる可能性があると述べています。

## 読むときの観点

- バージョン管理はソースコードだけでなく、より広いコンピュータ利用に応用できる
- 一般アプリケーション側が diff や merge を前提に設計されていないことが制約になる
- 図や文書の差分には、単なるテキスト差分ではなく Semantic Diff のような考え方が必要になる
- オープンソースコミュニティには、既存アプリケーションを協調作業向けに発展させる余地がある

## 原文の翻訳

私はいつもバージョン管理を使っているので、これはコンピュータ利用のもっと多くの領域に広がっていけるものだと思っている。ソフトウェア開発者以外でバージョン管理を使っているコンピュータ利用者は少ない。しかしソフトウェア開発者が知っているように、バージョン管理は共同作業のための優れた仕組みであり、複数の人がひとつのソフトウェアシステムに一緒に取り組むことを可能にする。では、バージョン管理がもっと広く使われるようになったら、どんな利点があるだろうか。

私たちは、誰もが自分の作業でバージョン管理システムを使えるだけの実用的な段階に来ている。Subversion は自由に入手できるシステムで、バイナリ形式を扱いやすく、CVS の多くの制約を取り除いている。ディスク容量も十分に安くなったので、人々の作業ディレクトリ全体をバージョン管理下に置くこともできる。

現時点での大きな制約は、バージョン管理を意識しているアプリケーションやツールが十分に多くないことだ。Word には長い間、変更履歴を追跡する機能があるが、バージョン管理を前提に作られているわけではない。一般的なアプリケーションで、diff や merge を含むバージョン管理機能はどのように見えるだろうか。そうしたアプリケーションは、この種の考え方をどのように活用できるだろうか。

私は Visio のようなツールで模式図を描く。図面についても、バージョン間で何が変わったのか、あるいは他の人が自分の図にどんな変更を加えたのかを diff で見られると便利だろう。こうしたことの価値を本当に引き出すには、**Semantic Diff を支えるツール**が必要になるかもしれない。

これは、オープンソースコミュニティにとって本当の機会になるかもしれない。誰もが Subversion を簡単に入手して使えるという事実を土台に、従来型のアプリケーションをこの方向へ進められるからだ。このあたりでよいアイデアが出てくれば、共同作業を大きく高められるだろう。
