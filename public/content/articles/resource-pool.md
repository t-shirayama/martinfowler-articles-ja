# Resource Pool

## 要約

Resource Poolは、作成や維持にコストのかかるリソースをまとめて管理する仕組みです。
代表例は、データベース接続やスレッドです。

クライアントはリソースを直接作るのではなく、プールから借り、使い終わったら返します。
この記事では、プールサイズ、枯渇時の振る舞い、動的な拡張と縮小、そして返却漏れによるリソースリークの検出について説明しています。

## 読むときの観点

- 作成コストや維持コストが高いリソースを、なぜ共有管理するのかを見る。
- プール上限に達したとき、例外、待機、拡張のどれを選ぶかを考える。
- 動的に拡張するプールでも、上限や縮小の仕組みが必要になる。
- テスト時にプールサイズを1にすることで、返却漏れを見つけやすくなる。

## 原文の翻訳

多くのプログラムは、作成や維持にコストのかかるリソースを使う必要があります。その例には、データベース接続やスレッドがあります。Resource Poolは、これらのリソースを管理する良い方法を提供します。

クライアントがデータベース接続を使う必要があるとき、自分で接続を作るのではなく、プールにひとつ渡してくれるよう依頼します。クライアントが接続を使い終えたら、それをプールに返します。多くの場合、プールは起動時に接続をいくつかまとめて作ります。プールのサイズは通常、設定で指定でき、クライアントからの需要やリソースを維持するコストに応じて調整できます。

クライアントが新しいリソースを求めたとき、すべてのリソースが使用中だった場合には、いくつかの応答が考えられます。ひとつは例外を投げることです。もうひとつは、リソースが利用可能になるまでクライアントを待たせることです。可能であれば、多くの場合最も良い応答は、さらに接続を作ってプールを大きくすることです。そうすれば、プールは小さな初期サイズに設定しておき、必要に応じて成長できます。これを行うなら、多くのリソースがアイドル状態になっていないかを確認し、それらを保持することが問題になる場合にはプールを縮小する仕組みが必要になるかもしれません。動的に成長させる場合でも、おそらくプールサイズには何らかの上限が必要です。その時点を超えてプールを大きくしたくはないからです。

Resource Poolでよく遭遇する問題のひとつは、クライアントがリソースを返さないことです。これはresource leakです。これに対抗する良い方法は、テスト中はプール上限を1に設定し、上限に達しているときにリソースを要求したら例外が発生するようにプールを設定することです。そうすれば、**リークしているクライアントを見つけやすくなります**。

Resource Poolについてさらに知るための良い情報源は、POSA 3のPoolingパターンです。
