# User Story

## 要約

User Storyは、ソフトウェアシステムに求めるふるまいを小さな単位に分け、計画や優先順位づけに使うための考え方です。長い仕様書よりも、会話を通じて要件を掘り下げるための軽量な入口として扱います。

よいストーリーには、独立していること、交渉可能であること、価値があること、見積もれること、小さいこと、テスト可能であることが求められます。また「As a ... I want ... So that ...」という形では、誰が、何を、なぜ欲しいのかを明らかにします。

## 読むときの観点

- User Storyは詳細仕様ではなく、会話と優先順位づけのための小さな単位として読む。
- INVESTの各条件が、計画しやすさと実装前の理解の深さにどう効くかを見る。
- 「so that」が、顧客の要求を実際の必要へつなげる文脈を与える点に注目する。
- Use Caseとの違いや、XPにおける背景を意識して読む。

## 原文の翻訳

User Storyとは、ソフトウェアシステムに望まれるふるまいのまとまりです。アジャイルなソフトウェア手法では広く使われており、大量の機能を、計画のためにより小さな部分へ分割するのに役立ちます。同じ概念をfeatureと呼ぶこともありますが、最近のアジャイル界隈では「story」あるいは「user story」という言葉が広く使われるようになっています。

Kent Beckは、Extreme Programmingの一部としてこの用語を最初に導入しました。長い文章で書かれた仕様よりも、もっと形式ばらず、会話を重視した要件の引き出し方を促すためです。ストーリーの本質は、一枚のノートカードに書けます。Kentと私は3インチ×5インチのカードを好んでいます。ストーリーは、開発に入る準備ができるまでは意図的に詳細化しません。他のストーリーと優先順位を比べられるだけの理解があれば十分です。

Bill Wakeは、よいストーリーの特徴を説明するために、INVESTというニーモニックを考案しました。

- Independent: ストーリーはどの順序でも提供できる。
- Negotiable: ストーリーに何が含まれるかの詳細は、開発中にプログラマーと顧客が共同で作り上げる。
- Valuable: その機能は、ソフトウェアの顧客またはユーザーにとって価値があると見なされる。
- Estimable: プログラマーが、そのストーリーを作るための妥当な見積もりを出せる。
- Small: ストーリーは短い時間、通常は人日単位で作れるべきである。少なくとも、1イテレーションの中で複数のストーリーを作れるようにすべきである。
- Testable: そのストーリーのソフトウェアが正しく動くことを検証するテストを書けるべきである。

ストーリーを表現する一般的な方法に、「As a ... I want ... So that ...」という形式があります。「As a」の節は、誰がそのストーリーを望んでいるのかを示します。「I want」は、機能が何であるかを説明します。「so that」は、その人がなぜこの機能を望んでいるのかを説明します。「so that」の部分は、顧客が欲しいと思っているものから、実際に必要としているものへたどり着く助けになる、**重要な文脈**を与えます。

Mike Cohnは、User Storyの書き方について、今では標準的な本となっているものを書きました。User StoryがXPに持つルーツを理解するには、white book、あるいは趣味のよいgreen bookを検討するとよいでしょう。以前のblikiエントリでは、Use Cases and Storiesがなぜ異なるのかを論じています。
