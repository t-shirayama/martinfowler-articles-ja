# Segregated DOM

## 要約

Single-page applicationでは、DOM操作、アプリケーションロジック、サーバーアクセスが混ざり合い、いわゆるjQuery soupになりがちです。Segregated DOMは、DOM操作を専用のJavaScriptオブジェクトに閉じ込め、アプリケーションロジックをDOMの詳細から切り離すためのモジュール化の手法です。

## 読むときの観点

- DOMを直接触るコードと、アプリケーション判断を行うコードを分ける。
- DOM用オブジェクトをGatewayとして見れば、アプリケーションとHTML DOMという二つの文脈をつなぐ境界になる。
- DOMからロジックを離すほど、ブラウザなしでテストできる範囲が広がる。

## 原文の翻訳

Single-page web applicationは、しばしばjQuery soupになります。アプリケーションロジック、DOM操作、サーバーアクセスが混ざり合い、本来よりも理解しにくく、テストしにくいものになってしまいます。Segregated DOMは、DOMに対する操作を専用のJavaScriptオブジェクトに分離するモジュール化の戦術です。

例として、Fowlerがユーロゲームのカタログを扱うために作ったフィルターページを考えます。このページでは、左側のフィルターパネルをクリックすると、右側の一覧に表示されるゲームが絞り込まれます。この振る舞いはJavaScriptで実装されていますが、フィルターパネルと一覧にはそれぞれJavaScriptクラスがあり、そのクラスが該当部分のDOMアクセスをカプセル化します。どのゲームを表示するかを決めるロジックはコントローラークラスに置き、コントローラーはDOM操作クラスと、DOMに依存しないインターフェースを通じてやり取りします。コントローラー自身はjQueryを使いません。

初期化時、コントローラーは更新用の関数をフィルターDOMに渡します。フィルターDOMは、チェックボックスのクリックイベントにその関数を設定します。フィルター表示が変わると、コントローラーの更新関数がフィルターDOMクラスに現在有効なフィルターを問い合わせます。フィルターDOMクラスはjQueryを使って該当するHTML要素を探し、チェックされているボックスを抽出し、各グループでどのボックスが選ばれているかを示す単純なデータ構造を返します。

コントローラーはこのフィルター状態を使って、どのゲームを表示すべきかを判断します。そして左右のカラムに分け、一覧に表示すべき要素IDをEntry List DOMへ渡します。Entry List DOMはjQueryを使ってHTML項目を適切な`div`へ移動し、画面上の表示を整えます。

このやり方の主な利点は、**各クラスを一つの仕事に集中させ、考えやすくする**ことです。Filter DOMクラスを扱っているときは、HTML要素に対するDOM操作に集中できます。コントローラーを扱っているときは、HTML構造やCSSクラス名の細部を無視できます。

より抽象的なパターンとして見れば、これらのDOMオブジェクトはGatewayとして働きます。つまり、アプリケーションとHTML DOMという二つのBounded Contextの間を通る通路です。Gatewayと同じように、DOMオブジェクトはクライアントであるアプリケーションの語彙で話すインターフェースを持ち、その実装ではHTML DOM側の詳細へ変換します。同じ分離はサーバーアクセスにも使えます。これはGatewayの古典的な使い方です。

良いモジュール設計は、テストしやすさとも関係します。各クラスは比較的独立してテストできます。Segregated DOMの特に大きな利点は、コントローラーをブラウザやPhantomJSのような模擬ブラウザなしでテストできることです。コントローラーのテストにDOMが不要なら、Node上で動かし、DOM Gatewayに対して単純なテストダブルを渡せばよくなります。これにより、コントローラーの変更は速く簡単になります。

DOM Gatewayからできるだけ多くのロジックを外へ出すほど、PhantomJSのような仕組みに頼らずにテストできる範囲を広げられます。これはHumble Objectパターンの応用でもあります。扱いにくい外部環境に近い部分を薄くし、判断や振る舞いの中心を通常のコードとしてテストしやすくする、という考え方です。
