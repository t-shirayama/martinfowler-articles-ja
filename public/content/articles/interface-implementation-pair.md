# Interface Implementation Pair

## 要約

Interface Implementation Pair は、すべてのクラスに対応する interface を作り、`ICustomer` と `Customer`、あるいは `Customer` と `CustomerImpl` のようなペアにする慣習です。
どこでも代替実装を差し込める利点はありますが、実装が1つしかない場面では同期の手間が増え、複数実装が本当にある箇所も見えにくくなります。

この記事は、アプリケーションコードと公開ライブラリで interface の判断が違うことを示します。
公開 interface は利用者の必要に合わせて設計すべきであり、実装クラスをそのまま写すことが最善とは限りません。

## 読むときの観点

- interface を作る理由が、実際の代替実装なのか習慣なのかを見る。
- アプリケーションコードでは後から Extract Interface できる点に注目する。
- 公開ライブラリでは利用者のフィードバックが遅いため、判断が変わることを押さえる。
- 公開 interface を実装クラスの単なる複製にしないという注意点を読む。

## 原文の翻訳

Interface Implementation Pair とは、すべてのクラスを取り上げ、それに interface を組み合わせる慣習のことです。その結果、`ICustomer` と `Customer`、あるいは `Customer` と `CustomerImpl` のようなペアを目にすることになります。多くの点で、各クラスにヘッダーファイルを用意する C/C++ の習慣を思わせます。ただしこの場合、interface と実装は実際に別々の型です。

このアプローチの利点は、interface の別実装を提供することで、どの時点でも何でも完全に置き換えられることです。

しかし、これは私があまり好んできた技法ではありません。複数の実装を持つつもりがないところで interface を使うと、すべてを同期させ続けるための余分な手間が生じます。もっとも、よい IDE は助けになります。さらに、実際に複数の実装を提供している箇所が見えにくくなります。

よくあることですが、トレードオフは、アプリケーションクラスを書いているのか、ライブラリを書いているのかによって変わります。アプリケーションでは、interface が必要になったのにまだ存在しないなら、Extract Interface を行えばそれで済みます。公開ライブラリでは、利用者からそのような速いフィードバックを得られないため、公開する型を interface にしておくことはずっと有用です。しかし、Published Interface の中で実装クラスをただまねることが最善手であることはめったにありません。interface はクライアントの必要を中心に設計されるべきであり、**その必要は実装と一致しないことが多い**のです。
