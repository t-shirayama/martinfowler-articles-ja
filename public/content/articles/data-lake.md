# Data Lake

## 要約

Data Lakeは、組織内の分析者が必要とする可能性のあるraw dataを、元の形のまま集める単一の保存場所です。Data Warehouseが統一されたスキーマや事前のクレンジングを重視するのに対し、Data Lakeはスキーマの解釈を利用者側に委ね、分析者が目的に応じて意味づけることを前提にします。

この自由さは、データ品質のばらつきやプライバシー、セキュリティの難しさを伴います。Data Lakeは運用システム間の連携に使うものではなく、明確な由来と時点を持つ履歴的な分析データとして扱い、必要に応じてlakeshore martへ整えるのがよい、という主張です。

## 読むときの観点

- Data LakeとData Warehouseの違いを、raw dataとschemaの扱いから見る。
- 「data swamp」という批判が、なぜ妥当でありながら本質を外しているとされるのかを考える。
- Data Lakeを直接多人数で使うのではなく、curationとdata martを挟む構造に注目する。
- provenance、immutability、privacyがData Lakeの設計上の重要条件になる点を見る。

## 原文の翻訳

Data Lakeは、Big Dataの世界におけるデータ分析パイプラインの重要な構成要素を表す言葉として、この10年ほどで現れました。考え方としては、組織内の誰かが分析に必要とするかもしれないすべてのraw dataを、ひとつの保存場所に置くというものです。Data Lake内のデータを扱うためにHadoopがよく使われますが、この概念はHadoopだけに限られません。

組織が分析したいすべてのデータをひとつに集める場所と聞くと、すぐにData WarehouseやData Martを思い浮かべます。しかし、Data LakeとData Warehouseの間には重要な違いがあります。Data Lakeは、データソースが提供するどんな形であれ、raw dataを保存します。データのスキーマについて仮定は置きません。各データソースは好きなスキーマを使えます。そのデータを自分の目的に合わせて理解する責任は、データの利用者側にあります。

これは重要な一歩です。多くのData Warehouseの取り組みは、スキーマの問題であまり先に進めませんでした。Data Warehouseは、すべての分析ニーズに対して単一のスキーマを持つという考え方に向かいがちです。しかし私は、単一の統一データモデルは、ごく小さな組織を除いて現実的ではないと考えています。少しでも複雑なドメインをモデル化するには、それぞれ独自のデータモデルを持つ複数のBounded Contextが必要です。

分析の観点では、それぞれの分析利用者が、自分の分析に合ったモデルを使う必要があります。raw dataだけを保存する方向へ切り替えることで、責任は明確にデータ分析者へ移ります。

Data Warehouseの取り組みで問題になりやすいもうひとつの要因は、データ品質の確保です。信頼できる単一のデータ源を作ろうとすると、異なるシステムでデータがどのように取得され、使われているかを多く分析しなければなりません。あるデータにはシステムAがよく、別のデータにはシステムBがよい、ということがあります。最近の注文ならシステムAがよいが、1か月以上前の注文はシステムBがよい。ただし返品が絡む場合は別、という規則に出会うこともあります。さらに、データ品質は主観的な問題であることも多く、分析ごとに品質問題への許容度が異なったり、何をよい品質とするかの考え方が違ったりします。

このため、Data Lakeは品質のばらばらなデータを放り込むだけの場所であり、Data Swampと呼ぶほうがふさわしい、という批判がよく出ます。この批判は妥当であり、同時に本質的ではありません。New Analyticsで注目される肩書きはData Scientistです。この肩書きは乱用されていますが、その多くは科学のしっかりした背景を持っています。そして真剣な科学者なら、データ品質の問題をよく知っています。時間とともに変化する気温の読み取りを分析するという、一見単純な問題を考えてみてください。観測値に微妙な影響を与える形で気象観測所が移設されること、装置の問題による異常値、センサーが動いていない期間の欠落などを考慮しなければなりません。

高度な統計手法の多くは、データ品質の問題を整理するために作られています。科学者は常にデータ品質に懐疑的で、疑わしいデータを扱うことに慣れています。だからこそ、彼らにとってData Lakeは重要です。raw dataを扱い、何らかの不透明なデータクレンジング機構に任せるのではなく、データを理解するための手法を意識的に適用できるからです。そうした不透明な仕組みは、よいことより悪いことをする場合があります。

Data Warehouseは、通常、データをクレンジングするだけでなく、分析しやすい形へ集約もします。しかし科学者は、ここにも反対しがちです。集約はデータを捨てることを意味するからです。Data Lakeにはすべてのデータを入れておくべきです。今日、あるいは数年後に、何が価値を持つかは分からないからです。

同僚のひとりが、この考え方を最近の例で説明してくれました。会社の契約マネージャーが手作業で行った予測と、自動化された予測モデルを比較しようとしたそうです。そのために、1年前のデータでモデルを訓練し、当時マネージャーが行った予測と比較することにしました。今では正解が分かっているので、精度を公平に試せるはずでした。

ところが始めてみると、マネージャーの予測はひどく、2週間で作った単純なモデルでさえ圧倒しているように見えました。あまりに良すぎる結果なので疑ったところ、調査の末、マネージャーの予測に紐づいたtimestampが間違っていることが分かりました。月末処理のレポートによって更新されていたのです。つまり、Data Warehouse内の値はこの比較には使えませんでした。しかしさらに調べると、当時のレポート自体が保存されており、そこから実際の予測を取り出せることが分かりました。再びモデルが勝っているものの、そこにたどり着くまでに何か月もかかった、という例です。

raw dataは複雑なので、より扱いやすい構造へcurateするものが必要です。大量のデータ量を減らす役割もあります。Data Lakeは、あまり直接アクセスされるべきではありません。raw dataを理解するには高度な技量が必要だからです。Data Lakeで作業する人は比較的少数にし、その人たちがData Lake内の一般に有用な見方を見つけたら、単一のBounded Contextに特化したモデルを持つ複数のData Martを作れます。より多くの下流利用者は、こうしたlakeshore martを、その文脈における信頼できる情報源として扱えます。

ここまでは、Data Lakeを企業全体にまたがってデータを統合する単一地点として説明してきました。しかし、もともとの意図はそうではありませんでした。この言葉はJames Dixonが2010年に作ったもので、彼はData Lakeを単一のデータソースに使うものと考えていました。複数のデータソースがある場合は、むしろwater gardenになるという考えです。当初の定式化とは異なり、現在の一般的な用法では、Data Lakeは多くのソースをまとめるものとして扱われています。ここで重要なのは、Lakeが大量のデータを自然な状態で保存するという点であり、流れ込む川の数は大きな問題ではありません。

Data Lakeは分析目的に使うべきであり、運用システム同士の協調に使うべきではありません。運用システムが協調する場合は、RESTful HTTP呼び出しや非同期messagingのように、その目的のために設計されたサービスを通じて行うべきです。Lakeは運用上の通信を探し回るには複雑すぎます。Lakeの分析から新しい運用上の通信経路が見つかることはあるかもしれませんが、その経路はLake経由ではなく直接構築されるべきです。

Lakeに入るすべてのデータには、場所と時点における明確なprovenanceが必要です。すべてのデータ項目は、どのシステムから来たのか、いつ作られたのかを明確に追跡できなければなりません。したがってData Lakeは履歴記録を含みます。これは、Domain EventをLakeへ流し込むことで実現でき、Event Sourcedなシステムとは自然に合います。あるいは、ソースシステムが現在状態を定期的にdumpすることでも実現できます。ソースシステムに時間を扱う能力がないが、そのデータを時間軸で分析したい場合には有用な方法です。この帰結として、Lakeに入れられたデータはimmutableになります。いったん述べられた観測は削除できません。後で反証されることはありえますし、Contradictory Observationsも想定すべきです。

Data Lakeはschemalessです。どのスキーマを使うかはソースシステムが決め、その結果として生じる混沌をどう扱うかは利用者が考えます。さらに、ソースシステムは流入データのスキーマを自由に変えられます。その場合も利用者が対処しなければなりません。もちろん、そうした変更はできるだけ破壊的でないことが望ましいものです。しかし科学者は、データを失うよりも、乱雑なデータを好みます。

Data Lakeは非常に大きくなります。保存の多くは、大きなschemaless構造という考え方を中心に組み立てられます。そのため、Data LakeにはHadoopやHDFSがよく使われます。lakeshore martの重要な役割のひとつは、扱う必要のあるデータ量を減らすことです。そうすることで、Big Data analyticsが巨大なデータ量をいつも直接扱わずに済みます。

大量のraw dataを飲み込もうとするData Lakeの性質は、プライバシーとセキュリティについて厄介な問いを生みます。Datensparsamkeitの原則は、今すべてのデータを取り込みたいというData Scientistの欲求と強い緊張関係にあります。Data Lakeは攻撃者にとって魅力的な標的になります。直接Lakeにアクセスできる人を小さなData Scienceグループに制限すれば、この脅威は減るかもしれません。しかし、そのグループが扱うデータのプライバシーに対して、どのように説明責任を負うのかという問いは残ります。
