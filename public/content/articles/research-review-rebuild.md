# Research, Review, Rebuild

## 要約

この記事は、Bahmniというオープンソースの病院管理・電子カルテシステムで、AngularJSとOpenMRS REST APIに依存していた表示コントロールを、React、TypeScript、FHIR APIへ移行する試みを紹介している。

中心にあるのは、LLMにいきなりコードを書かせるのではなく、MCPを使って既存機能を調査し、人間がその調査結果をレビューし、そのうえで再構築するという「Research, Review, Rebuild」の流れである。AIによる速度向上は大きいが、ドメイン知識、既存仕様の取捨選択、生成コードのレビューは人間が担う必要がある。

## 読むときの観点

- AIをコード生成器ではなく、既存システム理解を助ける分析者として使う。
- レガシー移行では、旧機能をすべて移すのではなく、残すもの、捨てるもの、後回しにするものを分ける。
- MCPで参照できる情報源を増やしても、最終判断はドメイン専門家と開発者が行う。
- 生成コードは、プロジェクト文脈を知らない優秀な開発者の成果物としてレビューする。

## 原文の翻訳

以下は、原文の構成に沿って主要な主張を日本語でたどる抄訳です。

著者は、かつて生成AIは新規開発に向いており、長く使われてきた複雑なシステムの近代化には向かないと考えていた。しかしModel Context Protocol、つまりMCPの登場により、その見方は変わった。MCPを使うことで、LLMはドキュメント、設定、ソースコードといった文脈へアクセスしやすくなり、レガシー移行の調査を支援できる。

対象になったBahmniは、低リソース環境の医療提供を支えるオープンソースシステムである。フロントエンドはAngularJS 1.xで作られており、AngularJSの長期サポート終了後も、多くの主要ワークフローがそれに依存していた。これは、セキュリティ、採用・オンボーディング、現代的なツールとの互換性、保守性の面で大きなリスクになる。医療システムでは、古いソフトウェアの継続利用が臨床ワークフローや患者データの安全にも影響しうる。

移行対象として選ばれたのは、Treatment Details Controlという表示コントロールだった。これは患者の処方や薬剤オーダーを、訪問履歴とともに表形式で表示する。既存実装はOpenMRS REST APIのデータ構造に強く結びついており、FHIRの`MedicationRequest`とは構造も表現も大きく異なる。移行は単なるUIの書き換えではなく、データモデル、表示意図、業務上の意味を読み解く作業だった。

Researchフェーズでは、MCPで拡張されたLLMに、対象機能の分析を行わせる。ここで重要なのは、旧コードの振る舞いを理解し、将来の実装に必要な情報を整理することだ。分析では、機能の目的、データモデル、データフロー、FHIRとの対応、旧実装と新実装の差分、性能上の観点、引き継ぐべき機能かどうかを確認する。**近代化は古い技術を置き換えるだけではなく、システムの未来像を再設計する作業**でもある。

著者は、Atlassian MCP serverで既存ドキュメントを参照し、filesystem MCP serverでレガシーのフロントエンドコードと設定を読ませた。ドキュメントは古くなっていることがあるため、ソースコードを真実の基準とし、ドキュメントは補助情報として扱うよう指示した。

Reviewフェーズでは、人間がAIの分析を検証する。AIの分析はそのまま採用するものではなく、ドメイン専門家とアーキテクトが、何を採用し、何を捨て、何を次のイテレーションへ送るかを判断する。たとえば、処方箋印刷や細かな表示設定など、一部の機能は今回の移行範囲から外された。

また、LLMが生成するコードを既存プロジェクトへ合わせるために、プロジェクト固有の規約、再利用可能な関数、データモデル、サービス、atomic componentなどを明示する必要がある。これが曖昧なままだと、反復ごとにバラバラなコードが生まれ、技術的負債が増える。

Rebuildフェーズでは、調査結果と実装ガイドを材料に、LLMへReact、TypeScript、FHIRベースの表示コントロールを作らせる。ここでも生成結果は終点ではない。開発者とドメイン専門家がコードレビューし、必要に応じてリファクタリングする。LLMは言語やフレームワークには強くても、プロジェクト特有の意図や業務文脈を完全には知らない。

著者のケースでは、調査、実装、ユニットテスト、統合テストを含めて、Claude 3.5 Sonnetにより短時間・低コストで移行できたという。ただし、成果を支えたのはAIそのものだけではない。事前に十分な調査とレビューを行い、実装前にスコープと意図を明確にしたことが大きい。

Reviewフェーズの価値は、AIが余計なものを作り始めるのを防ぐ点にもある。曖昧な性能要件から不要な`useMemo`や意味の薄いテストが生成された例が示されている。途中でAIへ細かく割り込んで軌道修正するより、最初に意図と制約を明確に与え、ひとまとまりの作業を完了させる方がよい結果につながりやすい。

結論として、この手法は人間の専門性を置き換えるものではない。むしろ、**ドメイン知識を持つ人間がAIを導くことで、レガシー移行の調査と実装を加速する**ためのワークフローである。AIは速度をもたらすが、何が正しい移行なのかを判断する責任は、なお人間の側に残る。
