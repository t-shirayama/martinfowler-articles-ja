# Focusing on Events

## 要約

エンタープライズアプリケーションを、外部世界で起きた出来事に反応するシステムとして捉えると、入力や連携、監査、履歴再生の設計が見えやすくなります。
この記事は、イベント駆動アーキテクチャを複数システムの統合だけでなく、単一アプリケーション内部の設計視点として扱っています。

特に、Domain Event、Event Collaboration、Event Sourcing、Parallel Models、イベントとコマンドの違いをつなげて、イベントを中心に据えるとどのような可能性と難しさが生まれるのかを整理しています。
ドラフト段階の文章ですが、イベントソーシングやイベント連携を考えるうえでの初期の問題意識を読み取れます。

## 読むときの観点

- イベントを「外部世界で起きたことの記録」として扱う視点に注目する。
- Event Collaboration が結合を緩める一方で、処理の流れを見えにくくする点を押さえる。
- Event Sourcing が監査、過去状態の再現、代替シナリオの検討を可能にする理由を見る。
- イベントとコマンドは似ていても、名前が設計者の発想に影響する点を意識する。

## 原文の翻訳

これは、私が2000年代半ばに書いていた Further Enterprise Application Architecture development の一部である。残念ながら、その後あまりにも多くの別件に注意を奪われてしまい、これらをさらに進める時間は取れていないし、近い将来に時間が取れる見込みもあまりない。そのため、この素材はかなりドラフトに近い形であり、再び取り組む時間を見つけられるまでは、修正や更新を行うつもりはない。

エンタープライズアプリケーションを考えるうえで長く使われてきた見方のひとつは、外部世界から来るイベントに反応するシステムとして捉えることだ。この考え方は、1980年代後半の構造化設計コミュニティで定着した。現在では「Event-Driven Architecture」という看板のもとで耳にする。

この考え方では、システムとその外部世界との相互作用を、イベントの伝達として見ることに集中する。入力は、システムへの可能な各入力を外部世界のイベントに結びつけたイベントリストを作ることで理解される。同様に、システムは自身に起きた重要な変化を、外部システムへイベントとして合図することで知らせる。

このようにシステムイベントを見ることは、システムの設計プロセスの一部になりうる。よい例は、構造化分析の後期の形で使われたイベント分割という手法だ。また実装の中に焼き込むこともできる。つまり、イベントオブジェクトを明示的に作り、それらを何らかのハンドラへ渡して処理させるのである。

このようにイベントへ目を向けることは、複数システムの統合に大きな関心を生み出してきた。これがイベント駆動アーキテクチャの中心的な動機である。Event Message を使えば、送信者と受信者を、識別の面でも時間の面でも簡単に分離できる。識別の面では、誰が反応するかを気にせずイベントをブロードキャストできる。時間の面では、イベントをキューに入れ、受信者が処理できるようになったときに転送できる。

このようなアーキテクチャは、疎結合によって、スケーラビリティと変更しやすさの面で多くのものをもたらす。

しかし、この節で私が関心を持っているのは、それらの組み合わせそのものではない。もちろん、それらは価値がある。むしろ私が関心を持っているのは、**イベントに焦点を当てることが単一のアプリケーションに何をもたらすか**である。そうしたものを見ているうちに、非常に興味深い機会を生み出す、いくつかの興味深い特徴に気づいた。その多くはかなり断片的だったため、頭の中で整理するのにとても苦労してきた。

それでもここには、通常のアプリケーションではあまり提供されないような、非常に興味深い機能の機会がある。

### イベントを表現する

ここまではイベントについてやや大まかに話してきたが、もう少し深く掘り下げるには、イベントをもっと注意深く見る必要がある。本質的に、Domain Event は、アプリケーションにとって関心のある何かが外部世界で起きたことを合図する。それは何らかのデータ構造でアプリケーションへ伝えられ、イベントを説明するデータを伴っている。私はこれをソースデータと呼ぶ。イベントは、メッセージングシステム、ユーザーインターフェイス、データベーステーブル上のトリガーなど、さまざまな場所からやって来る。

イベントは非常に異なるソースデータを持つ。多くの異なるものを表現できるからだ。しかし、イベントが持つべき重要なものがふたつあり、それは Time Point である。常に考えるべき、区別されたふたつの Time Point がある。イベントが発生した時刻と、システムがそのイベントに気づいた時刻である。複雑なシステムでは、気づいた時刻が複数あることさえある。これらの Time Point は、私が時間パターンで話している actual と record の概念にかなり近く対応する。

イベントは、ソースデータに加えて、そのイベントに対して何を行ったかを説明する処理データを持つこともある。このふたつを区別することが重要だ。特に、イベントのソースデータは不変である。それは、何が起きたと私たちが知っているかを表すものであり、簡単には変えられない。もっとも、ここには落とし穴がある。誤ったソースデータを受け取る場合があるからだ。この点にはあとで触れる。

Domain Event はオブジェクトなので、記録しやすい。Domain Event をログに残し、記録として保管するのはたいてい意味がある。少なくとも、それは優れた監査証跡になる。

### イベントで協調する

私たちは、プログラムを協調して動く複数のコンポーネントへ分けることに慣れている。ここで私は意図的に「コンポーネント」という曖昧な言葉を使っている。この文脈では、プログラム内のオブジェクトや、ネットワーク越しに通信する複数プロセスなど、多くのものを意味しているからだ。それらを協調させるもっとも一般的な方法は、リクエスト/レスポンスのスタイルである。顧客オブジェクトが販売員オブジェクトから何らかのデータを欲しければ、そのデータを求めて販売員オブジェクトのメソッドを呼び出す。

もうひとつの協調スタイルが Event Collaboration である。このスタイルでは、あるコンポーネントが別のコンポーネントへ何かをするよう依頼することはない。代わりに、各コンポーネントは何かが変わったときにイベントを合図する。他のコンポーネントはそのイベントを聞き、望むように反応する。よく知られた Observer パターンは Event Collaboration の例である。

Event Collaboration は、オブジェクトが責務を果たす方法についてのいくつかの前提を変える。特に、状態の保存をめぐる責務が変わる。リクエスト/レスポンスの協調では、特定のデータを保存するコンポーネントをひとつだけにしようとする。他のコンポーネントは、そのデータが必要になったとき、そのコンポーネントへ問い合わせる。Event Collaboration では、各コンポーネントが必要なデータをすべて保存し、そのデータの更新イベントを聞く。

リクエスト/レスポンスの協調では、データを保存しているコンポーネントが通常その更新にも責任を持つ。Event Collaboration では、あるデータの更新に責任を持つコンポーネントが、そのデータを保存している必要はまったくない。更新時にイベントが発生するよう保証するだけでよい。

アプリケーションでイベントを使うからといって、Event Collaboration が必須になるわけではない。また Event Collaboration とリクエスト/レスポンスの選択は、排他的な選択でもない。よく見かけるのはふたつのスタイルの混在であり、たいていはリクエスト/レスポンスが優勢である。

Event Collaboration は非常に疎な結合をもたらすため、既存コンポーネントを変更せずに新しいコンポーネントをシステムへ追加するのが特に容易になる。Event Collaboration の欠点は、協調の理解がとても難しいことだ。リクエスト/レスポンスの協調は、全体の流れを示す何らかのコードとして指定される。Event Collaboration ははるかに暗黙的であり、予期しないことが起きたときのデバッグがずっと難しくなる。

### イベントソーシング

しかし、監査証跡を超えて、さらに興味深い領域へ進むことができる。それを可能にするのは、システムへのすべての変更がイベントによって引き起こされる場合である。私はこのアプローチを Event Sourcing と呼ぶ。別の見方をすれば、Domain Event のログを処理することでアプリケーションの状態を完全に導出できるとき、Event Sourcing が成立している。

この状況は、いくつかの興味深い機会を開く。機能面の機会もあれば、実装面の機会もある。すぐに見える実装上の機会は、アプリケーションの状態全体をメモリ内に保持し、永続データベースを完全に取り払うことだ。アプリケーションがクラッシュしたら、イベントを再実行する。場合によっては、定期的に取得したアプリケーション状態のスナップショットから再実行する。

システムは、性能上の理由や単純さの理由から、この方法を選ぶかもしれない。システムから永続化ロジックを取り除くと、構築と保守の労力を大幅に減らせることがある。もちろん、すべてのシステムがこの道を進めるわけではないが、一部のシステムにとっては有用な選択肢である。

アプリケーション状態が、処理済みイベントによって完全に定義されているなら、Domain Event の別のリストを処理することで、別のアプリケーション状態を構築できる。私はこれを Parallel Models と呼ぶ。特に、これにより過去へ戻り、誰かがなぜそうしたのかを調べるために過去時点のアプリケーション状態を構築したり、過去と現在の変化を比較したりできる。

過去を調査できるだけでなく、別のイベントストリームを探索することで、起きなかったことを検討することもできる。木曜日にあの嵐が O'Hare 空港を2時間ではなく1時間だけ止めていたら、何が起きていただろうか。配送サイクルを1日1回から2回に変えたら、私たちの業務はどう違っていただろうか。

Parallel Models はエンタープライズアプリケーションでは一般的ではないが、エンタープライズアプリケーションの開発者には非常になじみ深い。開発者の道具箱に欠かせないソースコード管理システムは、Event Sourcing を使って、必要に応じて Parallel Models を生成するシステムである。これらの Parallel Models は履歴上のものかもしれないし、ブランチによって複数の現実を許すものかもしれない。

そのため、ソースコード管理システムは、Parallel Models の利点と問題を探るための有用なメタファーを提供できる。

イベントストリームを操作することでできる、もうひとつ興味深いことは、誤った情報の結果に対処することだ。先に述べたように、イベントのソースデータは不変である。受け取り、処理したものを変えることはできない。

では、受け取ったものが間違っていたらどうするのか。Event Sourcing を使っているなら、イベントが発生した時点へ戻り、本来起きるべきだったことを捉えた新しい Parallel Model を構築できる。要するに、その誤ったイベントを新しい Retroactive Event で置き換えるのだ。実際には、通常なら込み入って高価な手作業を要するこのプロセスを、完全に自動化するのが驚くほど簡単な場合もある。

こうして見ると Event Sourcing はすばらしく聞こえる。Safeways にも置いてあるのだろうか。残念ながら落とし穴がある。しかもいくつもある。もっとも単純な落とし穴はプログラミングモデルだ。システムへのあらゆる更新を永続化可能なイベントの形にするのは扱いにくい。特に、インタラクションの多いシステムではそうだ。それは自然なやり方でもないため、慣れるまで少し時間がかかる。

しかし Event Sourcing の本当の難しさは、イベントを中心にしていない外部システムとの接続にある。イベントを再生するには、外部システムの過去の状態を問い合わせられる必要がある。下流システムへ重複した更新を送らないようにしなければならない。Parallel Models の典型であるソースコード管理システムは、この種の動的な統合を行う必要がないため、この問題を避けている。

しかし多くのエンタープライズアプリケーションは、自分自身の仕事と同じくらい統合に関わっている。そのため、これらの問題は大きく立ちはだかる。

問題は乗り越えられないものではない。どれほど多くの代替処理を行うか、外部システムとの統合の度合いがどれほどか、その外部システム自身がどれほどイベントや Event Collaboration を使っているかによっては、解く価値があるかもしれない。Event Sourcing のより凝った帰結を使わないとしても、そのようにシステムを設計すれば、優れた監査能力が得られ、後でより洗練された方向へ進むための土台にもなる。

Event Sourcing を後付けするのは非常に厄介そうに見える。そう言うのは、実際にそのように後付けしたシステムを私は見たことがないからだ。

もうひとつ考えるべきことは、システム全体が Event Sourcing を使う必要はないという点だ。実際、私が見てきた Event Sourcing の多くは、システムの一部、たいていは会計側で使われていた。実を言うと、以前これらのパターンを書こうとしたときは、会計の言葉で書いていた。私がそれを見ていた場所が会計だったからだ。会計は、Event Sourcing がもたらす強い監査を必要とするだけでなく、Event Sourcing の問題の一部を和らげる助けにもなる。

### イベントを処理する

イベントがシステムのすべての更新を運ぶ場合でも、一部だけを運ぶ場合でも、それらを処理するロジックをどう編成するかについて、いくつかの興味深い選択肢を提示する。

特に興味深いスタイルのひとつは、イベントのソースデータを問い合わせ、どの実際の performer オブジェクトがイベントを処理するかを決める dispatcher オブジェクトを使うことだ。これにより、performer オブジェクトは単純なままにでき、dispatcher は正しい performer を見つけるために必要なもの以外のビジネスロジックを持たない。

概念上、このような dispatcher を編成する方法はたくさんあるが、私が繰り返し見てきたスタイルのひとつが Agreement Dispatcher である。ここでは、dispatch 機構の中心的な整理軸は、イベントの文脈を支配する契約上の合意である。顧客への販売は、継続中の契約によって支配されることもあれば、ときどき取引する顧客との共通販売ポリシーを示す暗黙の合意によって支配されることもある。

このスタイルは、dispatcher がイベントを問い合わせてそれをどの agreement へ送るかを見つけ、agreement オブジェクトがさらに一連の条件確認を行い、最終的な performer に到達するまで連鎖が続く、という委譲の連なりを促す。

強みは、performer と event の対応づけロジックの多くを、オブジェクト間の関係、実質的にはデータとして設定できることだ。これにより、システムは大きな設定可能性を得る。特に、このオブジェクト関係の網に時間を含めることで、イベント処理でよくあるもうひとつの問題、つまりビジネスルールの更新への対応を扱える。

### イベントとコマンド

この議論では、アプリケーション状態へのすべての変更をイベントによってカプセル化すると述べているが、同じことを「Command」という言葉とパターンを使って述べることも容易にできる。イベントは明らかにコマンドの性質の大部分を共有している。独自の寿命を持てること、キューに入れられて実行できること、イベントの取り消しがコマンドの undo と同じであることなどだ。

イベントという言葉を使う理由のひとつは、この種の振る舞いについて、この用語が分野で広く使われていることだ。event driven architecture や Event Message といった用語はよく耳にする。

最終的に私がイベントを選んだのは、それに伴う微妙だが重要な連想の集合があると思うからだ。人々はコマンドを、要求をカプセル化したものとして考える。コマンドでは、システムに X をせよと告げる。これに対してイベントは、何かが起きたことを伝えるだけである。イベントでは、Y が起きたことをシステムに知らせる。もうひとつの違いは、イベントは関心を持つかもしれないすべての人へブロードキャストするものと考えるのに対し、コマンドは特定の受信者へ送るものと考える点だ。

実際の振る舞いに関しては、それほど大きな違いはない。システムが X コマンドに多態的に反応することは、Y イベントへの反応と違う必要はない。それでも私は、**名前づけが、人々がイベントについてどう考え、どのようなイベントを作るかに影響する**と思っている。

Command は Gang of Four に記述された古典的なパターンである。Command Message と Event Message の対比については、hohpe-woolf も見るとよい。
