# Output Build Target

## 要約

ビルドターゲットを`compile`や`unit-test`のような作業名で名付けると、依存関係ベースの計算モデルではなく、手続き的な順序としてビルドを考えやすくなります。
この記事は、ターゲットを作業ではなく成果物に基づいて名付けることで、依存関係をより自然に表現できると述べます。

また、ターゲットは不要な作業をしないようにすべきです。
出力を意識すると、何が更新済みで、何を再実行すべきかを考えやすくなります。

## 読むときの観点

- ターゲット名は、ビルドをどう考えるかに影響する。
- 依存関係は後続ターゲットの列挙ではなく、各成果物の直接依存として表す。
- 不要な作業を避けるには、出力が最新かどうかを見る。
- ビルドシステムに命令列を手で書くのではなく、依存関係から順序を導かせる。

## 原文の翻訳

ここ数日、同僚のJulian Simpsonが書いている、`ant`ファイルのリファクタリングに関する記事をレビューしていました。Julianは私たちの「デプロイメント担当」の一人で、アジャイル指向の作業習慣をシステムのデプロイに適用してきた人です。その中で彼は、厄介な`ant`ビルドスクリプトにいやというほど出会ってきました。彼の記事は、その混乱を片づけるためのお気に入りの方法をうまく説明しています。

彼の観察の中で、特に私の興味を引いたものがあります。`ant`ファイルは、`compile`や`unit-test`のように、作業名をターゲット名にして分解されがちだという指摘です。そうしたタスクは、しばしば手続き的なターゲット呼び出しとして組み合わされます。

```xml
<target name="cruise" depends="clean, compile, copy-static-files,
unit-test, publish, javadoc, tag"/>
```

このようなスタイルには嫌なにおいがあります。ビルドファイルが基づいている依存関係ベースの計算モデルと合っていないからです。このモデルでは、それぞれのターゲットを見て、それが何に依存しているかを尋ね、その依存関係をターゲットに入れます。したがって、ここでの`unit-test`タスクは`compile`と`copy-static-files`ターゲットに直接依存すべきであり、後続のターゲットで言及されるべきではありません。

一見ばかげたものに出会ったとき、「テストをコンパイルなしに実行できるほど単純なシステムなのかもしれない」と軽口を叩くのは簡単です。しかし、もう少し深く掘り下げる価値があることもよくあります。

Julianも私もUnixの背景を持っているので、`make`というビルド言語にはなじみがあります。`Ant`と同じく、`make`も依存関係ベースのプログラミングを使います。どちらのシステムも作業をターゲットに分解し、依存関係で結びます。しかし、微妙な違いがあります。`ant`ではターゲットに名前を付け、どのターゲットに依存するかを示します。一方`make`では、出力ファイル、それが依存する入力ファイル、そして入力から出力を得る方法を述べます。

たとえばCで2ファイルのhello worldをコンパイルする`make`は、次のようになります。

```make
hello : hello.o greet.o
  gcc -o hello hello.o greet.o

hello.o : hello.c
  gcc -c hello.c

greet.o: greet.c
  gcc -c greet.c
```

最初の行は、ターゲットファイル`hello`が`hello.o`と`greet.o`に依存することを示しています。2行目は、ソースファイルからターゲットをどう作るかを示します。実際の`make`ファイルでは、多くの場合これらは一般規則になっているので、重複して書く必要はありません。その後、最上位ルールへの入力をどう作るかを同様に示します。

Javaで`ant`を使うと、次のようになります。

```xml
<project name = "hello" default = "test">
  <target name = "compile">
    <javac
      srcdir = "src"
      destdir = "build"
    />
  </target>
  <target name = "test" depends = "compile">
    <junit printsummary="on">
      <classpath location = "build"/>
      <test name = "Tester"/>
    </junit>
  </target>
</project>
```

ここでは、`test`タスクが`compile`タスクに依存すると言っています。つまり、`test`タスクを実行する前のどこかで`compile`タスクを実行する必要があるということです。

どちらの断片も依存関係を示していますが、この名前付けの違いは、私たちの考え方に影響します。その命名規約が、依存関係について考えることから、手続き的な規則について考える方向へ頭を引っ張ってしまうのではないかと思います。

もう1つの側面は、不要な作業をどう扱うかです。ビルドはいつも、望むより長くかかるように見えます。そのため、不要な作業を避けるために多くの努力をします。そもそも依存関係ベースのプログラミングを使う目的はそこにあります。ターゲットが何度言及されても、実行されるのは一度だけです。

しかし、不要な作業を避けたいという欲求は、ターゲット自身の実行にも関わります。`make`が作業を避ける基本的な仕組みは、出力ファイルと入力ファイルの更新時刻を比較することです。出力が最も新しい入力より新しければ、それは明らかに最新であり、作り直す必要はありません。

`Ant`はこれを行いません。代わりに、`ant`タスク自身が何を実行すべきかを分析することに頼ります。コンパイラが、変更後にどの出力クラスを再ビルドすべきかを判断する必要がある場合には、これは理にかなっています。しかし、人々が自分でターゲットを書く場合、そのターゲットが何らかのチェックを行わないことがよくあります。どんなターゲットでも、何かが変わった可能性がある場合を除き、大きな作業をしないようにすべきです。**出力に注目することは、そのための良い方法**です。

出力が明らかでないこともあります。ユニットテストタスクの出力とは何でしょうか。`make`の帽子をかぶるなら、私は「何らかのファイル」と答えるでしょう。テストレポートを結果にして、そのレポートファイルが入力のどれよりも古いかをタスクで確認できます。もっとも、それは望むほど簡単ではないかもしれません。レポートは何かを含んでいる必要すらありません。単なるTouch Fileでもよいのです。

私のUnixと`make`の根が行き過ぎているのでしょうか。そうかもしれません。しかし、ビルドファイルを見るときには、次のことを考えてみるよう勧めます。

- 各ターゲットには、それが行う作業ではなく、その成果物に基づいて名前を付ける。
- ターゲットは、必要がなければ大きな作業をしないようにする。
- 各ターゲットの依存関係を直接述べ、実行すべき手続き的な順序はビルドシステムに導かせる。
