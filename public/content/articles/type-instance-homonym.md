# Type Instance Homonym

## 要約

型インスタンス同音異義語は、同じドメイン用語が「型」と「そのインスタンス」の両方を指して使われることで起きる曖昧さです。
たとえば「本」は、文学作品としての本も、手に持てる物理的な一冊も指します。

会話では人間が文脈から自然に補正できるため大きな問題になりにくい一方、ソフトウェアでは別々の概念として表現しなければなりません。
ユビキタス言語では、精密さが必要な場面で区別できる名前を用意しつつ、日常会話から同音異義語を完全に追い出そうとしない姿勢が重要です。

## 読むときの観点

- 同じ言葉が型とインスタンスの両方を指していないかを見る。
- 会話で自然に通じる曖昧さと、コードで明示すべき区別を分ける。
- 区別用の名前は、ドメイン専門家に理解できるものにする。
- ユビキタス言語で精密さを求める場面と、軽く流してよい場面を見極める。

## 原文の翻訳

> 「『戦争と平和』はすばらしい本だ」
>
> 「どれどれ......この本は表紙がずいぶん傷んでいるね」

この二つの文は、どちらも「本」という言葉を使っています。
私たちは日々、こうした組み合わせを何気なく読み流し、それぞれの文で「本」という言葉がまったく違うものを意味していることを、ほとんど意識しません。

最初の文で「本」は文学作品を指しています。
それは百年以上にわたって存在してきたものです。
二つ目の文で「本」は物理的な物体を指しており、おそらくそれほど長く存在してはいないでしょう。
後者は暖炉にくべれば簡単に壊せますが、前者は炎に触れられることなく、間違いなく私より長く残るでしょう。
前者はいくつもハードディスクに入れられ、それらを合わせても後者より軽いのです。

この種の曖昧さは、自然言語では驚くほどよく見られます。
「車」「オーク」「プログラマ」のような言葉を考えてみてください。
これをきちんと整理しようとすると、哲学における意味論の暗い一角へ、すぐに足を踏み入れることになります。
ここでは当面、私がこの問題をどう考えているかだけを説明します。

「本」という言葉は、複数の概念を指しうる用語です。
上の二つの文は、「本」が指す二つの概念、つまり文学作品と物理的な一冊を示しています。
読者の多くはプログラマなので、プログラミングの類推を使って言えば、どちらも「本」と呼ばれる二つの異なるクラスがある、ということです。
これらは `LiteraryWork` と `PhysicalCopy` というクラスとして呼び分ければ区別できます。
最初の文で「『戦争と平和』は本だ」と言うとき、私は「『戦争と平和』は `LiteraryWork` クラスのインスタンスである」と言っています。
二つ目の文は、`PhysicalCopy` のインスタンスの性質に言及しています。

この二つの概念は独立していません。
「右」と「権利」のような同音異義語とは違い、この二つの概念には密接な関係があります[^1]。
本 [`PhysicalCopy`] には、本 [`LiteraryWork`] の表現が載っています。
だから私は、自分の『戦争と平和』は表紙が傷んでいる、と言うかもしれません。
同じように、私のスバル・レガシィ [物理的な車のインスタンス] は、スバル・レガシィ [車種のインスタンス] と関係を持っています。
実際、私の車の型はスバル・レガシィ [車種] だ、と言うこともできるでしょう。

この種の関係はとても一般的で、オブジェクトモデリングやデータモデリングをしたことのある人にはなじみ深いものです。
`Product` と `Product Type` のようなものに、似た構造をよく見かけます。
私は `Book` と `Book Type` という用語は使いませんが、**根本的には同じ関係**です。

ほとんどどんなドメインでも、モデリングやプログラミングに時間を使っていれば、やがてそのドメインの一般的な用語が二つの別々の概念を表していることに気づく時点に達します。
そしてそれは、コード上では異なる名前を持つ別々のクラス、あるいはテーブルを意味します。
ドメイン言語に同音異義語が現れる場合、多くはその用語が異なるコンテキストで違うように使われているからです。
これは Eric Evans が境界づけられたコンテキストと呼ぶものです。
Jim Odell が好んで指摘していたように、「Mary had a little lamb」は、獣医にとっては「メアリーは小さな子羊を飼っていた」という意味になり、レストラン経営者にとってはまったく違う意味になりえます。
型インスタンス同音異義語の厄介な点は、**とても狭い境界づけられたコンテキストの中でさえ起きる**ことです。

ここでの私の目的は、この種の状況をどうモデリングするかについて深く論じることではありません。
それだけでひとつの大きな話題になります。
むしろ、このよくある同音異義語が生む曖昧さについて考えたいのです。
この曖昧さについて本当に興味深いのは、それがほとんど問題を起こさないことです。
人間は複数の概念の間を難なく切り替えられるようで、その過程でつまずくことはめったにありません。

さらに困るのは、日常会話の中で意識的に別々の用語を使おうとしても、それを続けるのはほとんど不可能だということです。
私自身、会話でも文章でも試してみましたが、すぐに見込みのない試みだと気づきました。
単に私にその規律がなかっただけではありません。
そうすると、何かよくわからない精神状態の人のように聞こえてしまうのです。
この性質を持つのは型インスタンスの場合だけではありません。
「先週塗ったドアを通って行って」という表現も、それを示しています。

私自身の専門であるオブジェクト指向設計には、特に興味深い例があります。
私たちはよく「オブジェクトにメソッドを追加する」のように話します。
しかしもちろん、実際にはオブジェクトにメソッドを追加しているのではなく、クラスに追加しています。
キャリアのどこかで、あなたも会議中にその点を指摘する細かい人になったことがあるでしょう。
そしてその後の数え切れない会話では、同じ点を何の苦もなく読み流してきたはずです。

そこで私は、この同音異義語を必ず意識しなければならない、という結論に至りました。
ソフトウェアでは、概念を別々に表現し、異なる名前を付ける必要があります。
しかし、実際に混乱する危険がある文脈でない限り、会話の中でそれらを区別し続けることを期待すべきではありません。
たいていの場合、**私たちの脳が曖昧さを自動的に解決してくれる**からです。

では、これはユビキタス言語の文脈では何を意味するのでしょうか。
第一に、私たちはモデリングを行い、異なる概念があることを認識しなければなりません。
つまり同音異義語に気づく必要があります。
次に、それらの異なる概念について、ユビキタス言語で、したがってソフトウェア上の表現でも使う名前を考えます。
名前を考えるとき、私は裸の同音異義語を避けるのが好きです。
たとえば図書館の文脈では、「本」は使わず、`Literary Work` や `Physical Copy` のような名前を選びます。

こうした曖昧さを取り除く用語は、ユビキタス言語のほかの要素と同じく、ドメイン専門家に理解できるものでなければなりません。
したがって、ドメイン専門家も同音異義語を認識し、新しい用語を考えることに関わる必要があります。
そのうえで、得られる精密さが必要なときには新しい用語を使います。
しかし、同音異義語を完全に追い払おうとすることには慎重であるべきです。
それは実質的に、新しい用語それぞれの同義語になります。

たとえばドメイン側の人が「本には著者がいる」と言ったなら、私たちは「そこで言っているのは文学作品のことですよね」と確認するかもしれません。
同音異義語を使わないようにすることは、誰にとっても不自然に感じられる、ということを覚えておきましょう。
だから精密さは軽い手つきで適用します。
幸いなことに、ここでの戦いの90パーセントは、**同音異義語に気づくこと**です。

この同音異義語からさらに導かれることがあります。
特に新しいドメインで作業を始めた初期には、常にアンテナを張ってそれを探すべきです。
たいていの場合、それはそのドメインのど真ん中にある用語に現れます。
本と図書館の例がまさにそうです。
私はユビキタス言語をモデリングするために、非公式なクラス図を使うのが好きです。
そうすると、**この種の同音異義語を見つけ出す**うえでよい働きをしてくれることがわかっています。

### 脚注

[^1]: Jim White はメールで、言語学では、異なるが関連した意味を持つ語を示すために「多義語（polysemy）」という用語が使われることがある、と教えてくれました。
