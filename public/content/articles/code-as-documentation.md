# Code As Documentation

## 要約

アジャイル手法では、コードをソフトウェアシステムの主要なドキュメントとして扱う考え方がよく現れます。
ただし、それはコード以外のドキュメントが不要だという意味ではありません。

コードが主要なドキュメントになり得るのは、十分に詳細で正確だからです。
その代わり、コードを本当にドキュメントとして機能させるには、プログラマが読みやすさに責任を持たなければなりません。

## 読むときの観点

- 「コードがドキュメントである」は「コードだけで十分」と同義ではない。
- コードは詳細で正確だが、自然に読みやすくなるわけではない。
- 明快なコードを書くには、レビューとチームの合意が重要になる。
- 個人の好みよりも、チームが理解しやすいスタイルを優先する。

## 原文の翻訳

アジャイル手法に共通する要素の1つは、ソフトウェア開発におけるプログラミングの役割を中心に据えることです。これは、通常のソフトウェア工学のコミュニティが考えるよりも、ずっと大きな役割です。その一部として、コードをソフトウェアシステムの主要な、場合によっては最も重要なドキュメントとして分類します。

すぐに、よくある誤解に反論しておく必要を感じます。この原則は、コードが唯一のドキュメントだと言っているわけではありません。Extreme Programmingについてそう言われるのをよく耳にしますが、Extreme Programming運動のリーダーたちがそう言うのを聞いたことはありません。たいていの場合、コードを補うための追加ドキュメントは必要です。

コードが主要なドキュメント源である理由は、その役割を果たせるほど詳細で正確なのはコードだけだからです。この点は、Jack Reevesの有名なエッセイ「What is Software Design?」が雄弁に述べています。

この原則には重要な帰結があります。プログラマは、コードが明確で読みやすいものになるよう努力しなければならない、ということです。コードがドキュメントだと言うことは、特定のコードベースが良いドキュメントだと言うことではありません。他のドキュメントと同じように、コードも明確なこともあれば、意味不明なこともあります。コードは、他の形式のドキュメントより本質的に明確なわけではありません。

もちろん、他の形式のドキュメントも絶望的に不明瞭なことがあります。よくある例を挙げれば、意味不明なUML図をたくさん見てきました。

たしかに、多くのコードベースはあまり良いドキュメントではないように見えます。しかし、コードをドキュメントと宣言することが他の形式を排除する、という結論が誤りであるのと同じように、コードがしばしば悪いドキュメントだから必然的に悪い、という結論も誤りです。明確なコードを書くことは可能です。実際、ほとんどのコードベースはずっと明確にできると私は確信しています。

コードが読みにくい理由の一部は、人々がコードをドキュメントとして真剣に扱っていないことにあると思います。コードを明確にしようという意志がなければ、コードが勝手に明確になる見込みはほとんどありません。だから明確なコードへの第一歩は、**コードはドキュメントである**と受け入れ、そのうえで明確にする努力を払うことです。これは、多くのプログラマがプログラミングを学び始めたときに何を教わったかに関係していると思います。

私の教師たちは、コードを明確にすることをあまり重視しませんでした。彼らはそれを価値あるものとして扱っていないようでしたし、どうすれば明確にできるかについても話しませんでした。私たち業界全体は、コードの明快さにもっと大きな価値を置く必要があります。

次のステップは、その方法を学ぶことです。ここで、ベストセラー技術書の著者として助言をするなら、レビューほど役に立つものはありません。私は、多くの人に読んでもらいフィードバックを受けずに本を出版しようとは思いません。同じように、明確なコードにとって、他の人から何が分かりやすく、何が分かりにくいかのフィードバックを得ることほど重要なものはありません。だから、他の人に自分のコードを読んでもらう機会をできるだけ見つけてください。

何が理解しやすく、何が混乱を招くのかを知ってください。もちろん、ペアプログラミングはそのための優れた方法です。

より具体的な助言としては、プログラミングスタイルに関する良い本を読むことを勧めます。最初に見るべきなのは `Code Complete` です。私としては当然 `Refactoring` も勧めます。リファクタリングの多くは、コードを明確にすることに関わっているからです。`Refactoring` の後なら、`Refactoring to Patterns` も明らかな候補です。

さまざまな点で意見が分かれることは必ずあります。コードベースは主にチームの所有物であることを忘れないでください。たとえ一部について個人所有を実践しているとしてもです。プロのプログラマは、チームの必要に合わせて自分の個人的なスタイルを曲げる用意があります。だから、たとえ三項演算子が好きでも、チームがそれを理解しやすいと思わないなら使わないでください。個人プロジェクトなら自分のスタイルでプログラムできます。しかしチームで行うものは、チームの必要に従うべきです。
