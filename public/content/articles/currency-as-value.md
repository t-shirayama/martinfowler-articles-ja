# Currency As Value

## 要約

通貨は多くのシステムで、通貨コードを中心にした Value Object として扱えます。
ただし、通貨に付随する値が変化する場合には、単純な不変オブジェクトだけでは表現しにくくなります。

この記事では、通貨コードのような不変の値と、pip value のように変わりうる参照情報を分ける設計を紹介しています。
通貨を常に複雑に扱うべきだという話ではなく、ほとんどのシステムでは単純な Value Object で十分だという注意も重要です。

## 読むときの観点

- 通貨を値として扱うとき、何を不変の属性とみなすかを考える。
- Value Object の不変性が、変化する付随情報と衝突しないかを見る。
- 値オブジェクトと参照オブジェクトを分ける判断が、どのような場合に必要になるかを考える。
- 珍しい設計例として読み、一般化しすぎないようにする。

## 原文の翻訳

Value Object のよくある例はたくさんあります。私のお気に入りは Money で、Money と密接に結びついているもののひとつが通貨です。

多くのシステムでは、通貨は値としてうまく機能します。必要になる中心的な要素は、USD が米ドルを表すような、**国際的に認められた通貨コード**です。

ただし、以前私が関わったシステムでは、もう少し話が面白くなりました。記憶が正しければ、そのシステムで通貨に求められていたもののひとつに `pip value` がありました。UI には、通貨の値を上下に少し動かすための微調整ボタンがありました。それぞれの通貨には固有の pip value があり、この pip value は変更されることがありました。頻繁に変わるわけではありませんでしたが、それでも変わります。これは、Value Object の非常に有用な規則、つまり**不変であること**に反します。

私たちが取った解決策は、通貨を表すクラスを二つ持つことでした。ひとつは Value Object で、通貨コードと、おそらく他のいくつかの不変なものを保持していました。もうひとつは Reference Object でした。たしか CurrencyValue と CurrencyReference のような名前で呼んでいたと思います。ほとんどの場合は Value Object を受け渡していましたが、pip value のような Value Object 上の一部のメソッドは、主に静的なデータである Reference Object に委譲していました。

Reference Object は、通貨コードをキーにしたルックアップテーブルに保持されていました。もちろん、**それが唯一の方法というわけではありません**。

この二重の意味を持つ振る舞いは珍しいものです。私が見たのはこの一度だけです。たしかに、ほとんどのシステムでは、通貨については単純な Value Object で十分です。
