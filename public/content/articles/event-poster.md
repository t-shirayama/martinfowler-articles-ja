# Event Poster

## 要約

Event Poster は、イベントソーシングされた読み取り中心のアプリケーションで、表示に必要な状態をメモリ上に保持する設計です。
初期状態を読み込んだあと、イベントストリームで状態を更新し、障害時には初期状態とイベントキューから再構築します。

永続化用データベースを使わないことで高速化し、メモリ上のモデルとデータベースの対応づけを避けられます。
一方で、メモリに収まる範囲やイベントソーシング由来の制約を受けます。

## 読むときの観点

- 読み取り中心の画面で、表示用状態をどこまでメモリに置けるかを見る。
- イベントストリームが状態再構築の根拠になる点を押さえる。
- 永続化を避ける利点と、メモリ容量やイベントソーシングの制約を比較する。
- 更新要求を直接反映せず、イベントの発生源に渡してからイベントを待つ流れに注目する。

## 原文の翻訳

これは、私が何度か見かけたことのあるアプリケーションのスタイルである。アプリケーションは主に、何らかの状態についてリアルタイム情報をユーザーに提供するレポーティングアプリケーションである。ユーザーは何を見ているかについて多くの制御を持ち、特定の領域へドリルダウンしたり、表示を一般的に操作したりできるという意味では能動的なアプリケーションだ。しかしそれでも、少なくとも主としては読み取り専用のアプリケーションである。

この種のアプリケーションのもうひとつの特徴は、イベントソーシングされたアプリケーションであることだ。つまり、**表示状態の更新はすべて、保存やキューイングができるイベントオブジェクトを通じて行われる**。これらのイベントはアプリケーション専用に作られたものでもよいし、外部のメッセージストリームでもよい。Chris Stevenson は、イベントポスターの例として Inkblot の話をしてくれた。これは別のアプリケーション向けに書き込まれるデータベーステーブルを使い、行の挿入をイベントとして扱っていた。

この2つの特徴を合わせると、表示のメモリ上の状態を保存するためにデータベースは不要だとわかる。多くの場合、システムに読み込む初期状態がある。たとえば、その日の開始時点の世界である。その後の変更はすべて、イベントストリームを通じてイベントポスターのメモリ上の状態へ反映される。アプリケーションが失敗した場合は、初期状態を読み込み直し、キュー内のすべてのイベントを再生するだけでよい。

アプリケーション状態を永続化しないことには、主に2つの利点がある。第一に、ユーザーがシステムを操作している間にディスクアクセスが発生しないため、アプリケーションは非常に高速になる。Marcel Weiher は、BBC のニュースフィードアプリケーションについて話してくれた。以前のバージョンは本番環境で1つのリクエストを処理するのに数秒かかっていたが、置き換えられたイベントポスターアプリケーションは、彼のラップトップ上で1秒あたり数百件を容易に処理できた。

第二に、メモリ上の表現とデータベースとの間のマッピングにまつわる複雑さがすべて取り除かれる。そのため、表示の必要性に合わせたよいドメインモデルを構築でき、永続化について心配しなくてよい。（表示の振る舞いがたまたまリレーショナルな振る舞いに合っているなら、これはそれほど大きな利点ではないかもしれない。その場合は SQL が利点になるからだ。）

もちろん、この種のアプリケーションには限界がある。扱えるデータはメモリに収まるものに限られる。ただし現在では、メインメモリには非常に多くのものを保持できる。ギガバイト級のデータベースがかなり大きいと考えられていたのは、それほど昔のことではない。また、イベントソーシングに伴う他の制約にも直面する。

この種のシステムをクラスタ化するのは、ほとんど自明なほど簡単である。すべきことは、イベントがすべてのコピーへブロードキャストされるようにすることだけだ。ひとつが停止しても、状態は常にほぼ同期しているはずなので、別のものに置き換えるのは容易である。

私はこれらのアプリケーションを読み取り専用だと言ったが、厳密にはそうではない。更新を行う場合、ポスターは情報を捕捉し、イベントの発生源であるバックエンドアプリケーションへ送る。**自分自身のデータを直接更新するのではなく**、発生源アプリケーションから適切なイベントがストリームを通じて届くのを待つ。これにより、複数のポスターが同じデータを表示し、発生源アプリケーションが変更に対して必要な処理を行えるようになる。

私たちは、この種のアプリケーションによい名前をつけるのに苦労した。名前から想起できるとよさそうな重要な特徴がいくつかある。イベントストリーム、一時的なデータ、表示専用であることだ。私は「poster」がよいと思った。毎朝はがされて取り替えられるポスターに、その日のニュースを載せている掲示板を思い浮かべたからである。

イベントコラボレーションを伴う CQRS を使うシステムでは、Event Poster は自然な選択肢である。
