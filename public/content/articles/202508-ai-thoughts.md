# Some thoughts on LLMs and Software Development

## 要約

Martin Fowler は、LLM とソフトウェア開発をめぐる現状について、確定的な予測ではなく、いくつかの考えを短い断片として共有しています。AI が開発を速くするのか、品質を上げるのか下げるのかを測る調査はありますが、LLM の使い方の違いを見ない調査は誤った結論へ導きかねない、と警告しています。

将来のプログラミングや職業としてのソフトウェア開発については、Fowler は分からないと明言します。だからこそ、他人の事例を読むだけでなく、具体的なワークフローに注目し、自分でも実験して経験を共有することが大切だと述べています。

LLM は非決定的な存在であり、幻覚は単なる欠陥ではなく性質そのものです。そのため、同じ問いを複数回投げ、回答の揺れを比較すること、決定的に計算できることをLLMに直接計算させないこと、そしてセキュリティ上の攻撃面が大きく広がることを理解する必要があります。

## 読むときの観点

- 「AIで速くなるか」という問いを、どのワークフローで使ったのかという問いと切り離さない。
- LLM をジュニア開発者の比喩だけで理解せず、非決定的なシステムとして扱う。
- 幻覚を例外的な故障ではなく、LLM の出力の基本的な性質として読む。
- 数値や検証可能な結果は、LLM の回答ではなく、決定的な計算やテストで確かめる。
- AI agent の利便性と、機密データ、信頼できない入力、外部通信が組み合わさるリスクを一緒に見る。

## 原文の翻訳

Fowler は、しばらくサイトの更新から離れる前に、LLM とAIの現状について散らばった考えを書き残したい、と切り出します。

AI がソフトウェア開発に与える影響については、開発を速くするのか、コード品質を良くするのか悪くするのかを調べる初期の調査がいくつかあります。しかし大きな問題は、そうした調査が人々の LLM の使い方を十分に区別していないことです。多くの利用は、Copilot のような高度なオートコンプリートに近いものです。一方で、LLM から最も大きな価値を得ている人たちは、ソースコードを直接読んで編集し、タスクを進められるワークフローを好むことが多いようです。使い方の違いを無視した調査は、人々を誤った方向へ導くデータを生むおそれがあります。

Fowler は、プログラミングの未来、これからソフトウェア開発に入るべきか、ジュニアエンジニアの仕事がなくなるのか、シニアエンジニアは退くべきなのか、といった質問をよく受けます。彼の答えは、分からない、です。今はまだ LLM の使い方を学んでいる段階であり、特にモデルが大きく進歩するなら、うまい使い方を理解するまでには時間がかかります。だからこそ、少なくとも他人の実践を読み、できれば自分でも試し、経験を共有することが勧められます。

AI はバブルなのか、という問いに対しては、Fowler はもちろんバブルだと見ています。大きな技術的進歩には、運河、鉄道、インターネットのように経済的なバブルが伴ってきました。バブルが弾け、多くの投資が消えることはかなり確かですが、いつ弾けるのか、弾ける前にどれだけ大きくなり、どれだけ実質的な価値を生むのかは分かりません。ドットコムバブルが多くの企業を消した一方で、Amazon は残った、という点も重要です。

Fowler は数年前に公の講演から退きましたが、業界の友人たちと過ごす時間は恋しいと述べ、GOTO Copenhagen で多くの人と再会することを楽しみにしています。

Rebecca Parsons は以前から、幻覚は LLM のバグではなく特徴だ、と言ってきました。Fowler も、LLM がしていることは幻覚を生成することであり、その一部を私たちが有用だと感じているのだ、と捉えます。このため、LLM には同じ質問を一度だけではなく、表現を少し変えて複数回尋ねることを考えるべきです。回答の違いは、回答そのものと同じくらい有用なことがあります。数値的な答えを求めるなら、少なくとも3回は尋ね、ばらつきを見るべきです。ただし、決定的に計算できる答えを LLM に計算させるべきではありません。計算するコードを書かせるのはよいとしても、それも複数回試すほうがよいでしょう。

他の工学分野では、世界のばらつきを前提にします。構造エンジニアは測れない要因に対して余裕を組み込み、プロセスエンジニアは人間が時に忘れたり不注意になることを考えます。ソフトウェア工学は、決定的な機械を相手にする点で珍しい分野でした。LLM は、ソフトウェア開発が他の工学と同じように、**非決定性を扱う世界へ入る契機**なのかもしれません。

LLM はよくジュニアの同僚にたとえられます。しかし、LLM はテストがすべて通ったと言いながら、実際に走らせると失敗していることがあります。もしジュニアエンジニアが同じ振る舞いをしたら、すぐに問題になるでしょう。この比喩には限界があります。

LLM はソフトウェアシステムの攻撃面を大きく広げます。Simon Willison は AI agent における Lethal Trifecta として、機密データへのアクセス、信頼できないコンテンツへの接触、外部へ通信する手段の組み合わせを説明しています。信頼できないコンテンツは、ウェブページを読むだけでも入り込む可能性があります。攻撃者は、見えにくい文字で指示を埋め込み、LLM に機密情報を取得させようとするかもしれません。これは、ブラウザ上で動くエージェントでは特に深刻です。攻撃者のページを読んだエージェントが、別タブの銀行サイトなどで危険な操作を行うよう誘導される可能性があるからです。
