# Refinement Code Review

## 要約

Refinement Code Reviewは、コードが追加される前に一度だけレビューするのではなく、チーム全体が既存コードを読むたびに少しずつ改善していくレビューの考え方です。
Pull Request上のPre-Integration Reviewだけをcode reviewと見なす狭い見方に対する、別の視点として説明されています。

ソフトウェアは建物のように完成して固定されるものではなく、継続的に変化します。
そのため、実際にコードを使う文脈で見つかった理解や改善点を、コードへ戻していくことが重要になります。

## 読むときの観点

- Code Reviewを「工程」ではなく「継続的な改善行為」として読む。
- 後から読む人の負荷を下げるために、理解をコードへ移すという考えに注目する。
- Pull Requestがないとreviewできない、という誤解への反論として読む。
- セキュリティのような低頻度・高影響の問題には、別の注意が必要だと押さえる。

## 原文の翻訳

人々がcode reviewについて考えるとき、多くは開発チームのワークフローにある明示的な工程を思い浮かべます。現在では、Pull Request上で行われるPre-Integration Reviewが最も一般的な仕組みです。そのため、Pull Requestを使わなければcode reviewの機会がなくなると考える人までいます。

しかし、この見方は狭すぎます。明示的なreviewのさまざまな仕組みを無視するだけでなく、さらに重要なこととして、おそらく最も強力なcode reviewの技法を見落としています。それは、チーム全体によって絶えず行われるrefinementです。

ソフトウェアについて広く見られる考え方のひとつは、ソフトウェアを建てて完成させるものだと見ることです。このため、建築や建設の比喩が繰り返し使われます。しかしソフトウェアの重要な性質は、それがsoftであり、リリース後でも、最初にプログラマのエディタで書かれたときと同じように変更できることです。Erik Dörnenburgが、architectureよりtown planningのほうがよい比喩だと述べるのはこのためです。

価値あるソフトウェアは、通常、絶えず変化しています。私たちは、ソフトウェアがもたらせる価値をよりよく理解するにつれて機能を追加します。しかし機会は機能追加だけではありません。そのソフトウェアを洗練させる機会もあります。チームが学び続ける知見を取り込み、将来の変更をよりうまく支えられる形へ変えていくのです。

適切な環境があれば、著者は6か月前に書かれたコードを見て、書き方の問題に気づき、素早く直せます。そのコードに最初から欠陥があった場合もあれば、その後のコードベースの変化によって、もはや適切ではなくなった場合もあります。原因がどちらであれ、重要なのは、その問題が邪魔になり始めたらすぐに直すことです。

コードを読んだだけではすぐにわからなかった理解が自分の中に生まれたら、その理解を頭の中から取り出してコードへ入れる責任があります。Ward Cunninghamの言葉で言えば、理解を自分の頭の中に閉じ込めず、コードに移すのです。そうすれば、次に読む人は同じ苦労をしなくて済みます。

このrefinementのプロセスは、code reviewで起きていることと本質的に同じです。ただし、コードが追加されるタイミングではなく、コードが読まれるたびに起動します。これは著者にとって重要な洞察でした。Code reviewが直そうとする多くの問題は、将来そのコードが読まれるときに初めて問題になるからです。その時まで心配しすぎないほうがよい、という強い議論もあります。

たとえば大きな集合住宅が交通パターンを変えるように、6か月後にはコードの文脈が変わっており、そのコードに必要な修正の種類も変わっているかもしれません。この方式では、より多くの人も関わります。コードを読むすべての開発者がreviewerであり、抽象的で曖昧な指針ではなく、実際にそのコードを使う経験に基づいてreviewできます。

ある実践の妥当性を考える方法のひとつは、それが独占的な仕組みだったらどうなるかを考えることです。もしcode reviewの仕組みが、後から来たプログラマによる反復だけだったらどうでしょうか。ひとつの結果は、reviewの注意がよく読まれるコードへ集中することです。そして多くの場合、そこは注意を向けるべき場所です。

懸念は、まったく読まれないコードがreviewされないことです。しかし多くの場合、それでも問題ありません。よいテスト実践を持つチームなら、そのコードが動くことには自信を持てます。性能テストは性能上の問題を見つけられます。そのうえで、そのコードを二度と見る必要がないなら、理解しやすくするための努力は不要です。実際にはそのようなケースは非常に稀でしょうが、考える価値のある思考実験です。

もちろん、ほとんどの場合とすべての場合は違います。明らかな例外はセキュリティ問題です。コードは何年も正しく動いているように見えて、攻撃者が脆弱性を見つけた瞬間にreview不足を悔やむことがあります。これは、低頻度だが影響の大きい安全上の懸念であり、特別な注意に値します。

しかしそれは、Refinementをcode reviewの仕組みとして意識的に使うべきではない、という意味ではありません。むしろ、稀だが影響の大きい懸念を認識し、自分たちの状況で必要な程度に、その種の問題を見つけるワークフローを調整すべきだということです。

Refinement Code Reviewの要点は、レビューをひとつの門番工程へ閉じ込めないことです。**コードを読むたびに、そこで得た理解をコードへ戻す**ことで、チーム全体が継続的にコードベースをよくしていきます。
