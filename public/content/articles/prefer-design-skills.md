# Prefer Design Skills

## 要約

採用で、特定プラットフォームに詳しい人と、広い設計スキルを持つ人のどちらを選ぶか。Fowler は、多くの場合で広い設計スキルを優先すると述べます。プラットフォームの知識は比較的早く学べる一方、保守しやすいコードを書く感覚や設計の美意識は身につけるのが難しく、別の環境にも持ち運びやすいからです。

ただし、設計スキルが完全に移植可能なわけではなく、学ぶ意欲やチーム内の深いプラットフォーム知識も必要です。理想は広い設計力、プラットフォーム経験、ドメイン理解を兼ね備えることですが、現実には不足を補い合う判断が求められます。

## 読むときの観点

- 「すぐ使える技術経験」と「長く効く設計力」の違いを意識する。
- チームに不足している深い知識が何かを見極めながら読む。
- 顧客が求める経験年数と、専門家としての判断が衝突したときの倫理的な難しさに注目する。

## 原文の翻訳

採用の場面を想像してほしい。数年の経験を持つ候補者が二人いる。青コーナーには、あなたが好む設計スタイルについて、広く優れた設計スキルを持つ人がいる。私の場合なら DRY、慎重なパターンの利用、TDD、伝わりやすいコードなどが含まれるが、具体的な一覧は重要ではない。重要なのは、それがあなたの重視するものだということだ。ただし、その人はあなたが使っている特定のプラットフォーム技術をまったく知らない。

赤コーナーには、それらの設計上の問題についてはほとんど知識も関心もないが、あなたのプラットフォームを非常によく知っている人がいる。言語のエッジケース、利用可能なライブラリ、手に馴染んだツール操作まで把握している。二人のその他の条件はすべて同じだとしよう。もちろん、思考実験でもなければそんなことはないのだが。また、その候補者で埋めるべき大きな穴がチームにあるわけでもないとする。あなたならどちらを選ぶだろうか。

私の答えは単純だ。広い設計スキルを持つ人を選ぶ。よいプログラマなら、新しいプラットフォームは比較的すばやく身につけられる、というのが私の一貫した考えだ。基本的な設計の美意識を学ぶほうが難しく、しかも新しいプラットフォームへ持ち越しやすい。Java で重要なよい設計プラクティスは .NET でも同じように価値がある。

プラットフォームに馴染みがないことは、たしかに速度を落とす。C# でリテラルなクラス名を取るにはどうするのだったか、という具合に。しかし、**本当に差を生むのは、よく設計されたコードを生み出せること**だ。

広い設計スキルが完全に移植可能なわけではない。Java と .NET は言語としてはおおむね同等だが、Ruby に移ると事情はもう少し変わる。関数型言語のように大きく異なるものへ移るなら、さらに大きな変化になる。いずれにせよ、新しい環境で過去の設計習慣をただ盲目的に再現することはできない。しかし新しい世界に目を向けていれば、かなり多くのことは持ち越せる。

この原則が有効であることを、私たちは Thoughtworks で見てきた。Java を始めた初期には、経験豊富な開発者たちが Forte で学んだスキルが、Java に取り組む上で優れた勘を与えてくれた。私たちは早い段階で EJB 中心の考え方から離れたが、それを導いたのは他のプラットフォームでの経験だったと思う。.NET ではさらに強くそれを見た。

Java の背景を持つよい開発者が、設計スキルを欠いた長い .NET や Microsoft の経験者よりも、急速に効果を発揮する場面を何度も見た。その差は数か月ではなく数週間で、時には数日で見えた。

現時点でこの変化が最も目立つのは Ruby だ。今年はかなり多くの Ruby プロジェクトがあり、しばしば波括弧系言語で長い経験を持つ人たちに参加してもらっている。ここでも、広い設計スキルの価値を見てきた。

もちろん、いつもうまくいくとは限らない。別のプラットフォームで経験を積んだ人が、新しいものに入り込んで学ぶ意欲を持たないケースも見てきた。ここでは学ぶ意欲が必要な要素だ。広い設計力を持つ人が新しいプラットフォームを学ぶ気がなく、単一プラットフォームの専門家が広い設計を学ぶ気があるなら、私はその専門家を選ぶ。また、チームの中にそのプラットフォームをよく知る人がいることも不可欠だ。

Thoughtworks の多くの人は、プラットフォーム知識より設計スキルを好むと思う。多くの顧客はその見方を共有しない。そのため、実務的にも倫理的にも難しい選択が生じることがある。

強い設計スキルを持つがプラットフォーム経験のない人をチームに入れたいとする。しかし顧客は、そのプラットフォームで少なくとも二年の経験を求めている。あなたの専門的判断では、その広い力を持つ候補者が、他に得られる誰よりも生産的になる。顧客には正直である必要がある。一方で、顧客はあなたの専門的判断に対して対価を払っている。顧客があなたにプロジェクトのデリバリー責任を与えているなら、この判断は変わるだろうか。

私たちにとって、この問いは金銭的利害が絡むため、より重い。ThoughtWorker をチームに加えれば、私たちはその人の分を請求する。顧客がプラットフォーム専門の契約者を雇えば、その収入は得られない。多くの人にとってこれは状況の重要な事実だが、私たちのプロジェクトマネージャーは、間違った人を加えるリスクのほうが一人分の請求収入よりはるかに重要だと分かっているはずだ。

別のケースを考えよう。あなたは顧客に率直に説明したが、顧客はその広い設計力を持つ人にはプラットフォーム知識がなく、仕事をしながら学ぶことになるのだから、単価を下げるべきだと要求する。あなたは、その不足にもかかわらず、設計スキルのおかげで競合するプラットフォーム専門家よりも生産的になると確信している。値下げを受け入れるべきだろうか。

私たちの職業、そして多くの職業の性質として、人は仕事をしながら学ぶ。プラットフォーム専門家も、保守可能なコードを生み出すには広い設計スキルを学ばなければならない。ここで重要なのは、設計を学ぶほうがプラットフォームを学ぶよりたいてい難しいだけでなく、より不確実でもあるということだ。意欲ある広い設計者なら、時間が経てばプラットフォームを身につけるだろうとかなり確信できる。しかし逆方向には保証がない。

プラットフォームの詳細を学ぶのは得意でも、明快なコードを書く方法をいつまでもつかめない人もいる。

ここでは広い設計スキルについて話してきた。技術軸では、それが差を生むと私は信じている。しかし広さには他の次元もある。ソフトウェアプロジェクトのリスクの多くは、ビジネス側の人々とプログラマのコミュニケーションにある。そのため、ユーザーとうまくコミュニケーションできる候補者は、チームに大きな価値をもたらす。

問題領域の知識にも似た話がある。顧客はしばしば、自分たちのビジネスをすでに知っている人を求める。しかし誰かがすばやく十分な理解を得て役に立つようになると、驚くことがある。ここで中心になるのは、他者と協働する能力だと私は長く考えている。ドメイン専門家やプラットフォーム専門家と協働することで、広いスキルを持つ人は非常にすばやく有効になれる。

他のドメインの知識は、プロジェクトに意外な洞察を持ち込むことが多い。類似性は、驚くような場所に現れることも多い。表面上は会計に見えない場所に、会計の中核的なパターンが現れることがどれほど多いかは注目に値する。結局のところ、**他者と働けることと、速く学べることが決定的なスキル**なのだ。

ここで深いプラットフォーム知識を軽視しているわけではない。理想の世界では、すべてのチームメンバーが優れた広いプログラマであり、数年のプラットフォーム経験を持ち、問題領域にも十分馴染みがあり、似たシステムを少なくとも二度は書いたことがあるだろう。しかし現実がその理想からどれほど遠いかは、誰もが知っている。チームにはある程度のプラットフォーム知識が必要であり、そこが穴なら、私はその穴を埋めるためにプラットフォーム専門家を選ぶ。

それでも、ほとんどの場合に広い設計スキルを優先するという私の基本姿勢は変わらない。
