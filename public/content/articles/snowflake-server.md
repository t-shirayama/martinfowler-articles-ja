# Snowflake Server

## 要約

Snowflake Serverとは、手作業の設定変更を重ねた結果、再現しにくく壊れやすくなった本番サーバーのことです。問題は、障害時に同じ環境をすばやく作れないこと、クラスタ内の構成を揃えにくいこと、テスト環境を本番に近づけられないことにあります。

この状態を避けるには、OSやミドルウェア、アプリケーションの設定を自動化されたrecipeとして表現し、version controlで管理します。サーバーへの直接変更を避け、すべての変更をrecipe経由にすれば、再現性だけでなく監査性も高まります。

## 読むときの観点

- 手作業の運用変更が、なぜ長期的なリスクになるのか。
- disk imageによる複製と、構成をrecipeとして管理することの違い。
- Phoenix ServerやContinuous Deliveryの考え方と、構成管理がどうつながるのか。

## 原文の翻訳

本番サーバーを動かし続けるには、細かな作業が多く必要になる。OSや依存ソフトウェアにpatchを当て、ホストしているアプリケーションを更新し、環境が効率よく動き、他システムと適切に通信できるように設定を調整する。こうした作業では、command line、GUI、テキストファイルの編集が混ざり合う。

その結果として生まれるのが、ひとつだけの雪片のようなサーバーである。スキー場ではよいものでも、data centerでは困りものだ。

Snowflake Serverの最初の問題は、**同じものを再現しにくい**ことである。hardwareに問題が起きたとき、同じ役割を持つ別のサーバーを立ち上げるのが難しい。clusterを組む場合も、すべてのinstanceを同じ状態に保つのが難しくなる。本番環境をテスト用に複製することも簡単ではないし、本番障害を開発環境で再現して調べることも難しくなる。

disk imageを作ることで、ある程度は助けになる。しかしimageには不要な設定や過去の失敗も残りやすく、それらがそのまま引き継がれてしまう。

Snowflakeの本当の脆さは、変更が必要になったときに現れる。どの設定が重要で、どれが昔の初期状態の名残なのか分からなくなる。あるsoftwareをupgradeすると、予期しない影響が別の場所に出る。結果として、長く緊張を強いられるdebuggingが必要になる。監査要件を満たすためにも、手順書や手作業の記録が必要になりやすい。重要なsoftwareが古いOS上で動き続ける理由の一つはここにある。

Snowflakeを避けるよい方法は、サーバーの運用構成全体を、自動化されたrecipeとして保持することだ。PuppetやChefのようなtoolは、運用環境をDomain-Specific Languageに近い形で定義し、それを対象systemに適用できる。

recipeを使う目的は、単にサーバーを再構築しやすくすることだけではない。構成を理解しやすくし、変更しやすくすることにもある。さらに構成がテキストとして表現されるため、version controlで管理できる。

serverへの直接のshell accessを無効にし、すべての構成変更をversion control上のrecipeから適用するようにすれば、環境への変更が記録される。これは、規制の強い環境でも有用な監査の仕組みになる。

application deploymentにも同じ考え方を適用すべきである。完全に自動化し、すべての変更をversion controlで管理する。Snowflakeを避ければ、test environmentをproductionの真のcloneに近づけやすくなり、構成差異に由来する本番bugを減らせる。

Snowflakeを避けられているかを確かめる方法の一つは、Phoenix Serverを使うことである。server configurationをversion-controlled recipeで定義することは、Continuous Deliveryの重要な一部である。
