# Technical Debt Quadrant

## 要約

Technical Debt Quadrantは、技術的負債を「慎重か無謀か」と「意図的か偶発的か」の二軸で整理する考え方です。設計上の欠陥を負債と呼べるかどうかを争うより、負債の性質を見分け、どう対処するかを考えるために役立ちます。

Fowlerは、乱雑なコードも無謀な負債として扱えると考えます。また、優れたチームでも、作りながら学ぶことで後からよりよい設計が見えるため、慎重だが偶発的な負債が生まれると述べます。

## 読むときの観点

- 「これは負債かどうか」より「どんな負債か」に注目する。
- 慎重な負債と無謀な負債を分けて考える。
- 意図的な判断と、学習によって後から見える負債を区別する。
- 優れたチームでも負債が生まれることを前提に、返済判断を見る。

## 原文の翻訳

著作権上の理由により、このファイルには原文本文の全文翻訳を収録していません。以下は、記事の主張を日本語で読むための要旨です。

Technical Debtをめぐって、どのような設計上の欠陥を負債と呼ぶべきかという議論があります。たとえば、よい設計実践を知らない人が作った乱雑なコードは負債ではなく、短期的利益のために持続しない設計を意識的に選んだ場合だけを負債と呼ぶべきだ、という立場があります。

Fowlerは、この問いの立て方自体があまり有益ではないと考えます。Technical Debtは比喩なので、大事なのは、負債という比喩が設計問題への対処やコミュニケーションに役立つかどうかです。特にこの比喩は、非技術者と話すときに便利です。

乱雑なコードも、無謀な負債として考えれば比喩は役に立ちます。そのような負債は、重い利息を生んだり、元本返済に長い期間を必要としたりします。一方で、リリースのために慎重に引き受けた負債は、利息が十分に小さいならすぐ返済しない判断もありえます。

したがって、有用な区別は「負債か負債でないか」ではなく、「慎重な負債か無謀な負債か」です。さらに、負債には「意図的か偶発的か」という軸もあります。チームが負債を取っていることを理解し、早期リリースの利益と返済コストを比べて判断しているなら、それは意図的です。

設計実践を知らずに悪いコードを書いているチームは、どれほど負債を抱えているか気づかないまま無謀な負債を取っています。また、よい設計を知っているチームでも、「速く汚く」進めることを選ぶ場合があります。Fowlerは多くの場合、これも無謀な負債だと見ます。よい設計やクリーンなコードの目的は、速度を落とすことではなく、速く進むためにあるからです。

では、慎重だが偶発的な負債はあるのでしょうか。Fowlerは、それは存在し、優れた設計者のチームでは避けがたいと述べます。チームはよい仕事をし、顧客に価値あるソフトウェアを届け、コードも清潔に保っていたとしても、作り終えた後で「本当はこう設計すべきだった」と気づくことがあります。

これは、プログラミングが学習の過程だからです。作っている間に問題領域への理解が深まり、最適な設計が後から見えてくることがあります。その瞬間、チームは偶発的な負債に気づきます。この種の負債も、利息を払い続けるか元本を返済するかという判断には負債の比喩が使えます。

ただし、慎重で偶発的な負債は金融上の負債に対応物を見つけにくく、マネージャーに説明しにくい面があります。それでも、優れたチームであってもプロジェクトが進むにつれて負債を抱えるという事実は重要です。だからこそ、無謀なコードで最初から負債を過剰に積み上げるべきではありません。

Technical Debt Quadrantは、負債という言葉を雑に使わないための整理です。**慎重か無謀か、意図的か偶発的か**を見分けることで、責めるためではなく、返済と開発速度の判断に使えるようになります。
