# Use Case

## 要約

Use Case は、要求を整理し、引き出すための技法です。1980年代後半から1990年代前半にかけて Ivar Jacobson によって広められました。

UML には Use Case 図がありますが、Fowler は、図そのものの価値は小さく、本当に重要なのは UML で標準化されていないテキストだと述べています。Use Case は短く読みやすいときに最も有用であり、開発前に何週間も何か月もかけて文書を作るべきではありません。

## 読むときの観点

- Use Case を、図ではなく要求を引き出して整理するための文章技法として読む。
- Alistair Cockburn の本が、Use Case に関する主要な参照先として挙げられている点を見る。
- UML の Use Case 図よりも、利用者の目的とシステムとのやり取りを記述するテキストに価値があることを押さえる。
- Use Case 文書は、短く読みやすく、開発を遅らせないものとして扱う。

## 原文の翻訳

Use Case は、要求を整理し、引き出すための技法です。もともとは1980年代後半から1990年代前半にかけて、Ivar Jacobson によって広く知られるようになりました。

人々は Use Case についてよく質問します。そのほとんどは、Alistair Cockburn の本で答えられます。この本は、Use Case についての本として群を抜いて優れています。Use Case を使ったり、Use Case について質問したりする前に、この本を読むべきです。

Use Case は UML の中では Use Case 図という形で現れます。しかし、これらの図にはあまり価値がありません。**Use Case の重要な価値は、UML では標準化されていないテキストにあります**。したがって Use Case を使うなら、力を注ぐべきなのはテキストです。

Use Case は、短く読みやすいときに最も力を発揮します。開発を始める前に、Use Case 文書を作るために何週間も、まして何か月も費やすべきではありません。
