# Patterns in Enterprise Software

## 要約

この記事は、エンタープライズソフトウェア開発に関する主要なパターンカタログを、Martin Fowler の視点で整理した個人的な概観です。
各カタログを完全に網羅するのではなく、Fowler がよく知っていて有用だと考えるものを取り上げ、それらが扱う領域の関係を示しています。

中心になる観点は、Enterprise Application Architecture、Enterprise Integration、Domain Logic です。
特に、Enterprise Architecture と Enterprise Application Architecture を混同しないこと、そしてドメインロジックを重要な設計対象として見ることが強調されています。

## 読むときの観点

- パターンカタログを「どの問題領域を扱うものか」で分類して読む。
- Enterprise Architecture と Enterprise Application Architecture の違いを押さえる。
- 統合、ドメインロジック、レイヤリング、データアクセスがそれぞれ別の関心事であることを見る。
- ここで挙げられる本やカタログが、互いに補完し合う関係にあることを意識する。

## 原文の翻訳

近年、エンタープライズシステム開発のためのパターンを記述する動きが、小さいながらも有益に広がってきました。
このページでは、そうしたパターンのうち特に重要なカタログを一覧にし、それらの大まかな相互関係についていくつか考えを述べます。

これらの著者たちを結びつける正式な組織はありません。
しかし、私たちの間には、お互いの仕事を頻繁にレビューし合うという強い非公式のつながりがあります。
もっと組織立ったグループを作るべきかと考えたことも何度かありますが、実際に動き出すほどのエネルギーは集まりませんでした。
自分たちの仕事を書くだけでも、十分に大変なのです。

パターンが何に役立つのか、なぜ興味深いのかについては、人によって期待が異なります。
私自身の見方は、IEEE Software のコラムで述べました。

ここにカタログを挙げるのは、少なくとも私がかなりよく知っており、安心して紹介できるものだからです。
この領域にあるパターンカタログを、**完全に列挙するつもりはありません**。

### カタログ

ここでは、私が有用だと考える主要なカタログを挙げます。

**Patterns of Enterprise Application Architecture** は、レイヤ化アーキテクチャの文脈で Enterprise Application Architecture に集中しています。
主な章は、ドメインロジック、Web プレゼンテーション、データベースとのやりとり、オフライン並行性、分散です。
オフライン並行性は David Rice によるものです。
最も大きいのはデータベースとのやりとりに関する章で、オブジェクトリレーショナルマッピングの問題を扱う多くのパターンがあります。
著者は Fowler です。

**Core J2EE Patterns** は、Java J2EE プラットフォームの文脈で Enterprise Application Architecture のパターンを扱います。
パターンは J2EE プラットフォームを中心にしていますが、たいていは他のエンタープライズアプリケーションプラットフォームにも、少し形を変えれば同じように適用できます。
著者は Alur、Crupi、Malks です。

**Enterprise Integration Patterns** は、異なるエンタープライズアプリケーションを統合するうえで、非同期メッセージングが最も有効な方法のひとつだという私の見方を強めてくれました。
EIP は、このアプローチの基礎となるパターン集です。
著者は Hohpe と Woolf です。

**Microsoft Enterprise Solution Patterns** は、Microsoft による最初のエンタープライズソフトウェアパターン集です。
Web プレゼンテーション、デプロイメント、分散システムに関するパターンを含みます。
著者は Trowbridge、Mancini、Quick、Hohpe、Newkirk、Lavigne です。

**Microsoft Data Patterns** は、データの移動、つまり複製と同期に関するパターン集です。
著者は Teale、Etx、Kiel、Zeitz です。

**Microsoft Integration Patterns** は、Microsoft による統合パターンの見方です。
統合レイヤの戦略、システム接続の方法、統合のトポロジーを扱います。
著者は Trowbridge、Roxburgh、Hohpe、Manolescu、Nadhan です。

**Domain Driven Design** では、オブジェクト指向の Domain Model を構築することが、ドメインロジックを整理する一般的な方法として扱われます。
これは複雑なドメインに特によく向いています。
一方で、うまく行うのは難しいという弱点もあります。
この本のパターンは、豊かなドメインモデルをどう考え、どう構造化するか、また現実世界でモデリング原則の適用を妨げがちな障害をどう認識し、乗り越えるかを説明します。
著者は Evans です。

**Analysis Patterns** は、十分な数のドメインモデルを見ると繰り返し現れる構造を、パターンとして捉えようとした本です。
これは、そうした共通性をパターンの形で記録しようとした私の試みでした。
多くの点で更新が必要になっていますが、基本的な考え方は今でもかなりしっかりしています。
この資料が役に立つと感じるなら、私の Web サイトに置いた新しい補足資料もぜひ見てください。
著者は Fowler です。

**Data Model Patterns** は、データモデルにおける共通パターンを扱います。
非常に概念的なアプローチから作られているため、これらのパターンはデータモデリングだけでなく、オブジェクトモデリングにも役立ちます。
著者は Hay です。

**Gang of Four** は、最初の、そして最もよく使われているパターン本です。
これらは主に基礎的なパターンであり、エンタープライズソフトウェア開発に特化したものではありません。
しかし、エンタープライズパターンはそれらを広く参照しています。
著者は Gamma、Helm、Johnson、Vlissides です。

**POSA** は、アーキテクチャパターンに関する仕事で特に大きな影響を与えました。
レイヤ、つまりエンタープライズアプリケーションにおける Layers と、メッセージングにおける Pipes and Filters は、多くのエンタープライズパターン作業の基礎になっています。
著者は Buschmann、Meunier、Rohnert、Sommerlad、Stal です。

### エンタープライズソフトウェアの側面

これらのカタログは、エンタープライズソフトウェア開発のさまざまな側面を扱っています。
ここでは、そうした側面を起点に、カタログを別の見方で整理します。

### Enterprise Application Architecture

Enterprise Application とは、私がある種のソフトウェアシステムにつけた名前です。
多くの企業が日々の業務を動かすために使っている、データ集約的なソフトウェアシステムを指します。
別の、そしておそらくより適切な名前は Information Systems です。
なぜなら、これらは情報を処理し操作するシステムだからです。

EAA に関する本の多くは、まずエンタープライズアプリケーションを論理レイヤへ分けるところから始めます。
このレイヤ構造が、レイヤ内部およびレイヤ間の他の設計判断を導きます。
したがって、パターンも同じようにレイヤを通して整理されることが多いのは不思議ではありません。
著者ごとに独自のレイヤ構造がありますが、それらの間には認識できる類似点があります。

驚くほどよくあるのが、Enterprise Architecture と Enterprise Application Architecture という用語の混同です。
2つ目の A、つまり Application という言葉が非常に重要です。
EAA は、**単一のアプリケーションを構築すること**に関するものです。
Enterprise Architecture は、かなり違う対象です。

**Patterns of Enterprise Application Architecture** は、技術に依存しない視点から EAA を見ます。

**Core J2EE Patterns** は、EAA に集中した最初の本であり、J2EE プラットフォームの文脈からそれを扱います。

**Microsoft Enterprise Solution Patterns** は、.NET の角度から EAA を見ます。

### Enterprise Integration

Enterprise Application はある程度独立した存在ですが、機能するためには互いに連携する必要があります。
独立して開発された EA をつなぎ合わせることが統合の仕事です。
統合をまったく意識して設計されていないアプリケーションや、あなたが使っているものとは違う技術で統合することを想定しているアプリケーションを、統合しなければならないこともよくあります。

**Enterprise Integration Patterns** は、メッセージングのためのパターンです。
著者たちと私は、メッセージングを統合のための最も有望な方法だと見ています。

**Microsoft Integration Patterns** は、Microsoft の技術を使って統合を行うための戦略です。

**Microsoft Data Patterns** は、データの複製と同期のためのパターンです。
これらは統合にとって価値ある2つの技法です。

### Domain Logic

エンタープライズアプリケーションの中で最も重要でありながら、しばしば忘れられる側面のひとつがドメインロジックです。
これは、情報システムへデータが取り込まれるとき、あるいは表示されるときに、そのデータに対して働く業務ルール、検証、計算です。
単純なデータベース記録システムでは、ドメインロジックがほとんど、あるいはまったくないこともよくあります。
しかし多くのシステムでは、かなり複雑なドメインロジックが存在し、このロジックはビジネス状況の変化に応じて定期的に変わります。

**Patterns of Enterprise Application Architecture** の一部の章は、ドメインロジックを整理する主要なパターンを概説しています。

**Domain Driven Design** は、Domain Model の利用を詳しく扱います。
Domain Model は、最も洗練されたドメインロジックのパターンであり、複雑なロジックに最も適したものです。

**Analysis Patterns** は、ドメインモデルの例を示すパターンです。

**Data Model Patterns** も、ドメインの例を示すパターンです。
こちらはデータモデリングのアプローチから扱います。
