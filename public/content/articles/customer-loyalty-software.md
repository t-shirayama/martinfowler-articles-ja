# Customer Loyalty Software

## 要約

顧客ロイヤルティシステムの中心は、ポイントやマイルを扱う会計システムとして捉えると理解しやすくなります。
ポイント残高、調整、潜在的なマイル、複数種類のポイントは、会計の考え方を適用することで扱いやすくなります。

一方で、ロイヤルティシステムには顧客体験を追跡する側面もあります。
会計処理と顧客分析は同じイベントを使いながら内部構造が大きく異なるため、別々のシステムとして設計する余地があります。

## 読むときの観点

- ポイントやマイルを、金銭ではない会計単位として見る。
- 手作業の調整や将来発生するマイルを、会計上の仕訳や買掛のように考える。
- ロイヤルティプログラムのルール変更を、イベントと日付つきのルールで扱う。
- 会計処理と顧客体験分析を、同じイベントから派生する別の関心事として分けて考える。

## 原文の翻訳

先週カルガリーのオフィスにいて、私たちが最も信頼している技術リードの一人である John Kordyback とよい話をしました。
彼は旅行系のロイヤルティソフトウェアシステム、つまり frequent flyer や sleeper などにいくつも関わり、深く掘り下げてきました。
私たちは、こうしたものの性質と、それをより実りある形で考えるにはどうすればよいかについて話しました。

ロイヤルティシステムの中核は、ポイント、あるいはマイルを追跡するシステムです。
これにより、顧客は自分のポイントを確認でき、企業は未使用ポイントを管理できます。
多くの人はそう見ていないようですが、これは本質的には、ドルの代わりにポイントを扱う**会計システム**です。
John の観察では、人々が難しい問題だと見なしているものに彼は何度も出会ってきましたが、それらは会計の眼鏡をかけるとずっと扱いやすくなります。

その例が、場当たり的な変更への対応です。
自動化されたルール処理がどれほど優れていても、何か奇妙なことが起きて手作業で介入しなければならない場合は必ずあります。
多くのシステムでは、その結果として合計値に対する場当たり的な変更が行われ、誤りやすく、監査もされません。
しかし会計の考え方をすると、こうした変更を**会計上の調整**として見ます。
そして、このためのパターンはよく理解されています。

ロイヤルティプログラムと多くの会計システムの目立つ違いは、ロイヤルティプログラムが資産の管理よりも、負債の管理に重きを置いていることです。
そのため、税や収益報告といった共通のテーマに加えて、リスク管理のようなものにより多くの注意が向けられます。

多くのロイヤルティシステムには、通常のマイルとエリート資格判定用マイルのように、複数種類のポイントがあります。
これはよくある複雑さの原因です。
しかし会計の視点を使えば、これらを**複数通貨**として簡単に追跡できます。

この話における興味深いひねりが、潜在的なポイントです。
私が来月のフライトを予約した場合、航空会社は、私が来月搭乗したときに獲得するマイル、つまり潜在的なマイルがあることを知る必要があります。
これらの潜在的なマイルは航空会社の負債に影響します。
しかし、それが実際のマイルに変わるのは、私が搭乗したときだけです。
ここでも会計の考え方が役に立ちます。
再び複数通貨を使うこともできますし、買掛金のような考え方を使うこともできます。
仕組みはすでに存在し、よく理解されています。
必要なのは、**その状況にモデルを適用すること**だけです。

私たちはこれを実践の中で具体化しました。
その際、Test Driven Development を使うことが本当に役立つこともわかりました。
あるグループは、潜在的なマイルを計画的な設計で整理しようとして数週間を費やしましたが、TDD を使うと中核の問題は数日で解けました。
そこで決定的に重要だったのは、問題を具体的にするために**例に集中すること**でした。

会計の類推は、活動に対してどのようにマイルを付与するかを決める場合にも当てはまります。
ただし、こちらは部分的に少し直接性が弱くなります。
どのプログラムにも活動ルールがあり、それらは非常に柔軟である必要があり、ロイヤルティプログラムの絶え間ない変更に対応しなければなりません。
これを、Agreement Dispatcher を使ってドメインイベントが会計エントリを引き起こすモデルに従うものとして見ることができます。
これは John と私が何度も使ってきたパターンで、この種の変化するルールにうまく機能します。
基本的には、参加者のクラスに対するプログラム全体のルールを表す agreement があります。
各 agreement は、イベントの種類と日付範囲をキーにした posting rule の集合で構成されます。
ドメインイベント、たとえばホテル宿泊が発生すると、その顧客の agreement dispatcher を調べ、そのイベントを使って適切な posting rule を探します。
次に posting rule を実行し、そのイベントに対するマイルを表す適切な会計エントリを作ります。
イベントに時点を持たせることで、posting rule を時間とともに変更しながら、古いイベントも扱え、調整の自動処理も正しく行えます。
いつかこれらのパターンについて書き上げるつもりですが、今ウェブにあるものでも、何かしらの着想を得るには十分であることを願っています。

ロイヤルティシステムの第二の側面は、顧客体験を追跡することです。
会計処理では顧客の活動を記録する必要があるため、ロイヤルティシステムは、顧客と企業の相互作用から学ぶための自然な基盤になります。
この多くはデータマイニングです。
つまり、顧客行動のパターンを探し、それが新しい商品やプロモーションにつながることがあります。
また、この活動履歴を使ってプロモーションの成功を評価することもできます。
ある路線に搭乗するとマイルボーナスを付与する場合、反応はどうだったのか、というようにです。

私と同じく、John は Reporting Database を使うことの強い支持者です。
そしてこれは、この種の問題によく合っています。
会計側は、非常に異なるデータ構造の集合を必要とし、活動が発生するたびに定期的な更新を行います。
顧客体験の分析はすべて読み取り専用なので、会計側からの定期的な、ただし必ずしもリアルタイムではないフィードを受けて、**正規化を緩めた構造**を使えます。

さらに進めると、会計システムと顧客体験システムを完全に分離するのは妥当に思えます。
どちらも同じイベントを追跡するため、通常は一つの顧客ロイヤルティシステムとして一緒に置かれています。
しかし内部が大きく異なるため、同じイベントストリームを取り込む**二つの別システム**として扱うほうが理にかなっているかもしれません。
会計側は、おそらく顧客体験側のためにいくつかのイベントも生成するでしょう。

顧客体験追跡の習慣のひとつは、新しい種類の分析を支えるためにシステムへ頻繁な変更を加えることです。
私たちは、顧客活動の単一の保存済みイベントログを持ち、そこに比較的独立した miner を差し込むアプローチを試せるのではないかと考えました。
miner は、ログから選択した情報を、さまざまな分析を行うためのより具体的なデータ構造へ変換します。
miner 同士は比較的独立させられるため、構築しやすくなるでしょう。

見てのとおり、私たちの議論は、John の経験を見るところから、このようなシステムを将来どのように作れるかについての共同の推測へと移っていきました。
私たちにはっきりしているのは、この領域には新しい考えを探る余地が大きくあるということです。
それにより、この業務活動をよりよく支援できるシステムにつながる**新しい抽象化の集合**を導入できるかもしれません。
最近、この領域にはますます多くの注意が払われているため、私たちが取り組むには実りの多い場所に思えます。
