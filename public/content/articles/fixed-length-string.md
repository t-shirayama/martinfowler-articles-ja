# Fixed Length String

## 要約

Fixed Length String は、文字列の値だけでなく許容される長さも知っている値型です。
データベースの `char` や `varchar` には長さ制約がありますが、多くのプログラミング言語の文字列型にはその制約がないため、更新時に長すぎる文字列を渡してしまう危険があります。

この記事は、単純な値型を導入することで、データベース制約、UI の入力制限、検証ロジックの重複回避をより扱いやすくできると説明しています。

## 読むときの観点

- データストアを隠蔽しても、その制約から自由になるわけではない点を見る。
- 単純な値型が、検証と UI の両方を改善する場面を考える。
- UI に検証を置くことと、検証ロジックを重複させないことのバランスを読む。
- 長さチェックを、より一般的な値オブジェクト設計の例として捉える。

## 原文の翻訳

アプリケーションプログラミング言語とリレーショナルデータベースの間でやり取りするライブラリの多くを見ると、データベースの文字列型、つまり `char` や `varchar` を、プログラミング言語の文字列型へマッピングしていることに気づくでしょう。単純で明らかですが、おそらくそれは間違っているのかもしれません。

私がこれを疑問に思う理由は、SQL の varchar は固定長である一方、多くのアプリケーション言語の文字列はどんな長さでも扱うからです。そのため、データベースを更新するときには、長すぎる文字列で更新してしまう危険が常にあり、多くの潜在的な問題が生じます。

これに対処するひとつの方法は、Fixed Length String 型を作ることです。そのインスタンスは、長さと値の両方を知っています。そうすれば、文字列値を代入するたびに、その長さと照合できます。そのようなオブジェクトは、例外を投げたり、切り詰めたり、その他の振る舞いを実行したりできます。実際、長すぎる入力文字列に対する振る舞いを、インスタンスデータに追加することもできます。

このような型を使うと、UI をよくできます。UI のフィールドウィジェットが Fixed Length String にサイズを問い合わせ、そのサイズの入力だけを受け付けるウィジェットを作れるからです。これは長すぎる入力文字列に対処する方法として、はるかに最も気持ちのよいやり方です。しかしそれが可能なのは、情報がデータベースから前方へ持ち出されている場合だけです。通常、そのような情報は失われてしまいます。

この考え方は、より一般的な設計上の問題の具体例です。

- データストアを隔離するのはよいことですが、それでもデータストアが課す制約の中で生きなければなりません。
- 単純な値型を作れば物事がずっと簡単になることが多いにもかかわらず、人々はしばしばそれを作ることに消極的です。詳しくは When To Make a Type を参照してください。
- 単純な検証を UI に置くと、よりよいインターフェースになります。こつは、検証ロジックを重複させずにそれを行う方法を見つけることです。

文字列を保持して検証するオブジェクトは、長さチェック以上のこともできます。正規表現に照合するような検査を含めることもできます。しかし長さチェックは、データベースを扱いやすくする単純な例です。
