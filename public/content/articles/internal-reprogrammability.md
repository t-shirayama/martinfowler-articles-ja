# Internal Reprogrammability

## 要約

Internal Reprogrammability は、ツールの内部から、そのツール自身と同じ仕組みで機能を追加・変更できる性質です。
Emacs や Smalltalk のような環境では、別のツールチェーンや再起動なしに、小さな変更を素早く加えられます。

この記事は、プラグインやマクロとは異なる深い拡張性の価値を述べつつ、個人ごとにソフトウェアのインスタンスが分岐する問題や共有・アップグレードの課題も認めています。

## 読むときの観点

- 拡張可能性と、内部から再プログラムできることの違いを見る。
- Emacs、Smalltalk、Lisp、Unix シェルが示す「道具との関係」の変化を考える。
- カスタマイズの自由が、断片化やアップグレードの難しさも生む点を押さえる。
- 非プログラマー向けツールにも、この性質を広げるべきかという問いを読む。

## 原文の翻訳

私はプログラミングをしていて、今入力している場所の上に空行を追加したくなりました。使っていたエディタにはその機能が組み込まれておらず、その欲求を十分に何度も感じていたので、本当に欲しくなっていました。Google で素早く検索し、数行のコードを見つけ、それを自分の起動ファイルに貼り付けて実行しました。すると、1つのキーストロークで上に空行を作れるようになりました。かかった時間はほんの数分で、プラグインをインストールする必要も、エディタを再起動する必要もありませんでした。これは Emacs ユーザーにとっては、ごく日常的なことです。

Emacs は古いソフトウェアで、1970年代半ばにさかのぼります。実行中の環境を変更することで、人々がそれを簡単に拡張できるようにするという哲学は、Lisp machine や Smalltalk のような、古いけれど画期的だったいくつかのシステムにも共有されています。

その哲学は、今ではより珍しくなったように見えます。確かに、拡張可能なシステムはたくさんあります。Firefox のようなブラウザや Eclipse のような編集スイートには、プラグインをインストールできます。自由ソフトウェアやオープンソースの運動全体も、あなたのマシンで動くコードにアクセスできるようにし、理論上は好きなように調整できるようにするものです。しかし、こうした環境の多くにおける拡張と、Emacs や Smalltalk で行う種類の再プログラミングとの間には、明らかな違いがあります。上で私が追加した新しいコマンドのように、小さな変更を素早く簡単に行えることに何かがあります。また、環境を離れずに行えることにも意味があります。私は Emacs の関数を追加するために別のツールチェーンを起動するのではなく、Emacs 自身の中で作業します。

これは、何らかの macro capability を追加するツールとも異なります。新しい elisp 関数を追加することは、Emacs 自身がプログラムされている方法そのものです。小さな拡張をプログラムする方法と、ソフトウェアの中核をプログラムする方法の間に違いはありません。この一体性により、エディタの奥深くまで手を伸ばせます。また、自分の変更がどこかの scripts メニューへ追いやられることもありません。それらはツールの他の部分と見分けがつかないものになります。

この能力は、道具とどのように関わるかについての哲学でもあります。多くの人にとって、使うソフトウェアは比較的固定された製品です。プラグインでさえ、比較的限られた選択肢のメニューを追加するだけです。内部から再プログラム可能なツールでは、ソフトウェアのどの部分でも追加したり変更したりできます。そのため、自分の道具を、比喩的な意味で自分の手にぴったり合うように作り込めます。

この考え方は、プログラミング言語そのものにも当てはまります。Lisp と Smalltalk はどちらも最小限の言語であり、拡張が中核部分と同じに見えるように言語を拡張しやすくしています。どちらの言語にも、条件分岐のような基本的な言語機能のための特別な構文はありません。この柔軟性により、Smalltalk は言語を変更せずに例外処理を追加できました。

Internal Reprogrammability の最大の問題のひとつは、その結果としてソフトウェアのインスタンスが断片化することです。私が自分の Emacs インスタンスを多くの個人的な関数で変更すると、自分のマシン上の Emacs 設定に密結合した、私専用の Emacs を作っていることになります。必然的に、これは中核アプリケーションのアップグレードをどう扱うか、そして自分の関数を他の人とどれだけ簡単に共有できるかという問いを生みます。

プラグインアーキテクチャやマクロ言語を持つシステムは、カスタマイズできる表面積を減らすことでこの問題に対処します。しかし Nic Ferrier がうまく言ったように、「再プログラム可能なシステムは信じられないほど強力だ。その力を濫用することは常に可能であり、再プログラム可能なシステムでは、人々がそれを濫用できなければならないというのが原則上の要点である」。

Emacs コミュニティはもちろん、これが実際にどう進展してきたかを示すよい例です。Emacs は十分に安定しており、定期的な更新があるにもかかわらず、ほとんどの人は深刻な頭痛なしにアップグレードできます。Emacs はパッケージ管理システムを使い、共有可能な変更の配布を助けてきました。元の Emacs や Smalltalk の時代から、コードを共有する方法については大きな進歩がありました。分散バージョン管理ツールの台頭も、共有コードを管理するためのさらなるアイデアを加えています。

ほとんどの開発者にとって、Internal Reprogrammability の哲学に最も近いものは、Unix のコマンドラインシェルかもしれません。私はこれを、その単純な例だと考えています。誰でも簡単に新しいコマンドを作成し、組み込みのものと見分けがつかない形でシステムへ追加できるからです。パッケージにより、人々はさらに多くのコマンドを追加し、自分固有のニーズに合わせてコマンドライン環境を調整できます。コマンドラインの対話モデルは Emacs や Smalltalk ほど豊かではないため、それらの環境が提供するものの淡い反映にすぎません。しかし、その経験の手がかりは与えてくれます。

プログラマー向けのツールでさえ Internal Reprogrammability が珍しいなら、非プログラマー向けのツールではさらに珍しくなります。私はよく、それは変わるべきなのではないかと考えてきました。より多くのツールがこの性質を示すようになったら、何が起こるでしょうか。多くの時間を過ごす環境をよりよく制御するために、より多くの人がプログラミングについて学ぶよう促されるでしょうか。これは確かに、Alan Kay の dynabook 構想の一部でした。彼は子どもたちをメディアの受動的な消費者ではなく、自分たちの環境を能動的にプログラムする存在として見ていました。

プログラミングは簡単ではありませんし、私はプログラマーが日々直面する課題を軽く見せようとは思いません。しかしだからといって、Internal Reprogrammability を1970年代の未来像へ追いやるべきだということにはなりません。現代の dynabook が Kay の構想にあった Internal Reprogrammability を欠いている大きな理由は、それが十分に高い優先順位に置かれてこなかったからです。おそらく、それは私たちがもっと考えるべきことなのでしょう。
