# Building your own CLI Coding Agent with Pydantic-AI

## 要約

この記事は、Pydantic-AI を使って、自分用の CLI コーディングエージェントを作る過程を紹介している。題材は、既製のコーディング支援ツールを使うだけでなく、ツール呼び出し、会話の流れ、ファイル操作、モデル選択を自分で組み合わせることで、エージェントの仕組みを理解することにある。

本文では、Amazon Bedrock 経由のモデル利用、Python による CLI 実装、ファイル読み書きやコマンド実行のためのツール、MCP サーバーとの接続などが扱われる。完成品そのものよりも、エージェントを構成する部品を小さく作りながら、どこに注意が必要かを学ぶ記事として読むとよい。

## 読むときの観点

- コーディングエージェントを、モデル単体ではなく「モデル、ツール、状態、権限、実行環境」の組み合わせとして読む。
- サンプル実装の便利さだけでなく、ファイル操作やコマンド実行に伴う安全性も確認する。
- Pydantic-AI の使い方を追いながら、エージェントにどの情報を渡し、どの操作を許すかという設計判断を見る。
- MCP は追加機能ではなく、外部ツールや文脈を接続するためのインターフェースとして捉える。

## 原文の翻訳

この記事では、自分で CLI ベースのコーディングエージェントを作る。目的は、既存の AI コーディングツールを再発明することではなく、エージェントがどのような部品で動いているのかを具体的に理解することにある。

出発点になるのは、Pydantic-AI を使った小さな Python プログラムである。LLM にプロンプトを送り、返答を受け取り、ユーザーとの対話を続ける。最初の段階では、これは通常のチャットに近い。しかし、コードベースを扱うエージェントにするには、モデルが外部の情報を読めるようにし、必要な操作をツールとして呼び出せるようにする必要がある。

記事では、Amazon Bedrock を通じてモデルを利用する構成が説明される。モデルの選択や接続方法は実行環境に依存するが、重要なのは、エージェントの中心が「モデルへの単発の質問」ではなく、ユーザーの要求、過去の会話、利用可能なツールを含む実行ループになる点である。

次に、エージェントにコードベースを読ませるためのツールを用意する。ファイル一覧を調べる、ファイル内容を読む、必要に応じて変更する、といった操作は、コーディングエージェントにとって基本的な能力である。ただし、これらは便利であると同時に危険でもある。**ツールは能力であると同時に権限でもある**ため、何を許し、どの範囲に限定するかを設計しなければならない。

CLI エージェントは、ユーザーからの指示を受け、モデルの応答に従ってツールを呼び出し、その結果を再びモデルに渡す。この反復によって、エージェントは調査、編集、確認を段階的に進められる。人間がコードを読むときと同じように、最初に全体を眺め、関係するファイルを絞り、仮説を立て、変更後に検証する流れが必要になる。

記事の後半では、MCP サーバーとの連携も扱われる。MCP を使うと、エージェントはファイルシステム以外のツールや外部コンテキストに接続できる。これは、エージェントの能力を広げる方法である一方、接続先が増えるほど境界と安全性の確認も重要になる。

この実装から学べる中心的なことは、コーディングエージェントが魔法のような単体の存在ではないということだ。実際には、モデル、プロンプト、ツール定義、実行結果、会話履歴、権限管理が組み合わさっている。小さく作ることで、それぞれの部品がどのように影響し合うかが見えやすくなる。

最後に、この種のエージェントを実用に近づけるには、操作のログ、失敗時の扱い、ユーザー確認、テスト実行、変更範囲の制限が欠かせない。エージェントに任せるほど、作業を観察し、必要なところで止められる仕組みが重要になる。
