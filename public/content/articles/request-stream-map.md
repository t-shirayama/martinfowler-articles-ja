# Request Stream Map

## 要約

Request Stream Mapは、ひとつのリクエストを取り上げ、そのリクエストとレスポンスのデータがアプリケーションの各レイヤーをどう通るかを可視化する方法です。
レイヤーの数や変換の流れを見えるようにすることで、そのレイヤーにどれだけのコストと価値があるかを考えやすくします。

この記事では、過剰なレイヤリングを見つめ直すための道具として、この地図を紹介しています。
ただし、必要なデータ変換まで無理にひとつへ押しつぶすべきではなく、小さな変換を適切に保つほうがよい場合もあると注意しています。

## 読むときの観点

- ひとつのリクエストがシステム内でどれだけのコードや層を通るかを見る。
- レイヤリングは有用だが、過剰になると理解や変更のコストが増す。
- Value Stream Mappingになぞらえ、リクエストの流れから無駄を見つける。
- 不要な変換を減らすことと、必要な変換を乱暴にまとめることを区別する。

## 原文の翻訳

Thoughtworksの同僚たちの近くにいると、ほどなくして、良いEnterprise Service Bus、つまりESBとは、死んだESBだけだという印象を受けるようになります。Jim WebberはそれらをErroneous Spaghetti Boxesと呼んでいます。そのため、必要のないシステムからESBを取り除こうとした話を聞くのは珍しくありません。

ある顧客先で戦いが始まり、それは私に、若いころにD&Dで遊んでいた日々を思い出させました。Webberが剣を振るが、ESBのACは2なので外れる。Evanが命中させ、2d8を振って6ダメージ。Erikが最後に「Summon Request Stream Map」を唱えて、それを仕留める。

では、Erik Dörnenburgの決定的な呪文とは何だったのでしょうか。要するに、その考え方は、単純なリクエストをひとつ取り、そのリクエストとレスポンスのデータがアプリケーションのレイヤーをどのように進んでいくかを示すことでした。Erikは、この仕組みを理解するために読む必要のあるすべてのコードを印刷しました。それは数ページに及びました。さらに、彼は図も作りました。

現在、アジャイルの世界では、ソフトウェア開発プロセスの無駄を明らかにする方法としてValue Stream Mappingが流行しています。私はこれをRequest Stream Mapだと考えています。なぜなら、同じようにリクエストを取り上げ、それがレイヤーをどう移動するかを示し、何が起きているのかを可視化し、レイヤーのコストと価値について考えられるようにするからです。

レイヤリングは、ソフトウェアアプリケーションを作るための不可欠な道具です。しかし、人生のほとんどの不可欠なものと同じように、**多すぎることは少なすぎることと同じくらい問題になりえます**。このような可視化、あるいは数ページにわたるコードは、「ちょうど十分」がどこにあるのかを見つける助けになります。

ただし、ひとつ危険があります。データをある形から別の形へ変換する必要があるなら、通常は大きな変換をひとつ置くより、小さな変換をいくつか置くほうがよいものです。避けたいのは不要な変換であり、必要な変換を圧縮することではありません。
