# Protected Data

## 要約

protected なデータをクラスに置くことがよい設計かどうかは、継承階層を自分の制御下で使っているのか、外部の利用者に拡張される framework を作っているのかで変わります。
後者では protected な要素も公開インターフェースの一部として扱う必要があり、フィールドを公開するのは避けるべきです。

この記事は、protected を常に善悪で判断せず、どの範囲に対して変更の自由を保ちたいのかを考えるための短いメモです。

## 読むときの観点

- protected の意味は言語ごとに微妙に違う点を押さえる。
- 自分のチーム内の継承と、外部に公開する framework の継承を分けて考える。
- protected feature が published interface になりうることに注意する。
- fragile base class problem との関係を見る。

## 原文の翻訳

私のクラスに protected な Access Modifier を持つデータを置くのは、よいオブジェクト指向設計でしょうか。

オブジェクト指向設計者の間では、すべてのデータを private にすべきか、それとも一部を public にしてもよいかについて意見が分かれます。

この濁った池に飛び込む前に、protected という Access Modifier の意味は、言語ごとに微妙ながら重要な違いを持つことを指摘しておきます。

この話題が難しくなる理由は、人々がそれをチーム内開発の観点か、チームをまたぐ開発の観点のどちらかから見ており、その見方が議論を支配するからだと思います。

一方で、私が subsystem を書いていて、ある振る舞いを階層によって実装しようと決めた場合、subclass が superclass の内部へ深くアクセスできることにはかなり満足します。結局それはすべて私のコードです。私は subclass を、多態的な振る舞いを提供するために使っているのであって、モジュール性を提供するために使っているわけではありません。

他方で、framework を作っていて、上書きされることを期待する hook class を提供している場合、将来、人々の subclass を壊さずに変更できるよう、データを保護したいと思うかもしれません。実際にはこれはさらに厄介です。注意しなければ subclass は簡単に superclass を壊せるからです。これは fragile base class problem と呼ばれます。

したがって、データを protected にすることが常によい、あるいは常に悪いとは言えないと思います。自分の制御外の client が特定の class を override することを意図しているなら、protected feature を **published interface の一部**として扱うべきです。その結果、より注意深く扱う必要があります。field を公開するのはよくない考えなので、その場合、私はそうしません。

しかし、それが意図ではなく、自分が書く subclass からデータへアクセスできることが役立つなら、私にはそれは妥当に思えます。とはいえ、私の既定は field を private にすることであり、それを protected に変えることはめったにない、ということは認めておきます。

関連する問いとして、Self Encapsulation を使うべきかどうかがあります。
