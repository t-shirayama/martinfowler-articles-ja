# Specification By Example

## 要約

Specification By Example は、テストが XP において果たす役割の一つを表す考え方です。仕様は本来、すべての場合を一般的に覆うものだと考えられがちですが、例は具体的で、人が理解しやすく、特に利用者側の人々が考えやすいという強みがあります。

ただし、例だけで要求を完全に表すことはできません。例は設計チームに抽象化を促しつつ、その抽象化を現実に結びつけますが、会話、Domain Driven Design、契約的なインターフェイスの考え方など、ほかの手段と組み合わせる必要があります。

## 読むときの観点

- Specification By Example を、仕様の完全な代替ではなく、要求を具体化する強力な技法として読む。
- pre-condition / post-condition と例による仕様の、書きやすさと検証しやすさの違いを見る。
- TDD のテストが、コードとは別の形で同じ内容を表す「二重チェック」になる点に注目する。
- 例だけでは不完全だからこそ、継続的な会話と協力関係が必要になる、という結論を押さえる。

## 原文の翻訳

2002年の XP/Agile Universe でワークショップに参加していたとき、`Specification By Example` という言葉が、XP におけるテストの役割の一つを説明する方法として私の心に響きました。

（最近では、Test Driven Development（TDD）がテストと何か関係があると語るのは、ひどく流行遅れです。しかし Jon と同じく、私は包括的な自動テストスイートを持つことは、`side-effect` という言葉が示すよりも価値があると思っています。これは、山登りをしたら100万ドルをくれると言われるようなものです。私は、その山登りの主な目的は自然を楽しむことだと言うかもしれませんが、財布への副作用は決して些細ではありません。）

Specification By Example は、私たちの多くが教わってきた仕様の考え方とは違います。仕様は、あらゆる場合を覆う一般的なものであるべきだとされます。例は、いくつかの点を強調するだけで、一般化は自分で推論しなければなりません。つまり、Specification By Example は、使う要求技法の唯一のものにはなれません。けれども、**主導的な役割を担えないという意味ではありません**。

これまで、厳密な仕様、つまり合格か不合格かを明確に判断できる仕様で支配的だった考え方は、pre-condition と post-condition を使うことでした。これらの技法は形式手法で中心的な位置を占めており、Design by Contract の土台にもなっています。こうした技法には出番がありますが、理想的というわけではありません。pre-condition と post-condition は、状況によってはとても簡単に書けますが、別の状況では非常に難しくなります。

私はエンタープライズアプリケーションの文脈で、それらを何度か使おうとしてきました。そして多くの場合、pre-condition と post-condition を書くのは、解決策を書くのと同じくらい難しいと感じました。Specification By Example の大きな利点の一つは、たいていの場合、例のほうがはるかに思いつきやすいことです。特に、私たちがソフトウェアを書く相手である、技術者ではない人々にとってそうです。

（Timothy Budd は、スタックの振る舞いの多くは pre-condition と post-condition で示せるものの、LIFO という性質を示す pre-condition と post-condition を書くのは非常に難しい、と指摘しました。）

TDD のテストの重要な性質は、二重チェックを含んでいることです。実のところ、これは XP コミュニティのちょっと面白い小さな嘘を浮き彫りにします。彼らは `Once and Only Once` と言って、物事を一度だけ語ることを強く主張します。しかし実際には、彼らはいつも二度語っています。コードで一度、テストでもう一度です。Kent は、この二重チェックの原則が重要な原則だと指摘してきましたし、それは人間がいつも使っている原則でもあります。

二重チェックの価値は、その両側で異なる方法を使うことと強く結びついています。Specification By Example を本番コードと組み合わせると、同じことをかなり異なる形で表すことになります。**その違いが、誤りを見つける力を高めます**。

形式仕様のコミュニティは、設計が仕様を満たしていることを検証する際に、特に誤りやすい人間に頼らずにそれを行う点で、ずっと苦労してきました。Specification By Example では、これは簡単です。理論上の価値は低く見えるが、実務上の価値は高い、という Specification By Example のもう一つの例です。

Design by Contract の支持者の一人は、テストとして仕様を書くと、供給者は特定のテスト入力に対してハードコードした応答を返すだけで仕様を満たせてしまう、と指摘しました。これに対する私の軽い返答は、もしそれを心配しているなら、テスト対 Design by Contract という問題は、あなたの心配事の中では最も小さいものだ、というものです。しかし、ここには真面目な論点があります。テストは常に不完全です。したがって、テストは常にほかの仕組みによって支えられなければなりません。

ひねくれた考え方をする私は、実はこれを長所だと見ています。Specification By Example だけでは十分でないことが明らかなので、すべてを適切に伝えるには、もっと多くのことをしなければならないことも明らかになります。従来型の要求仕様で最も危険なことの一つは、人々がそれを書いた時点で、もうコミュニケーションは終わったと考えてしまうことです。

Specification By Example は、双方が敵対せずに協力している作業関係の中でだけ機能します。例は、設計チームの中で抽象化を促しながら、その抽象化を現実に根づかせます。さらに多くのものが必要です。定期的な会話、Domain Driven Design のような技法、実際には Design by Contract の要素すら必要です。私は Design by Contract を完全な形で、つまり Eiffel とともに使う機会はありませんでしたが、インターフェイスについて契約的な観点から考えることはよくあります。Specification By Example は強力な道具であり、おそらく私が最もよく使う道具ですが、**決して唯一の道具ではありません**。

（Specification By Example について、たとえその名前では呼んでいなくても、さらに考えるには Brian Marick の文章を見るとよいでしょう。彼のサイトのどこかには、おそらく彼の見方をまとめたすばらしいページが一つあるはずです。もちろん、それを探すことには、探している間にそこにある文章を全部読むことほどの価値はありません。）
