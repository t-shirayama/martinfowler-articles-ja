# Minimal Interface

## 要約

Minimal Interfaceは、クライアントが必要なことをすべて実現できるようにしつつ、APIのメソッド集合を合理的に最小に保つ設計スタイルです。
便利メソッドをすべて否定するわけではありませんが、既存のメソッドで実現できるなら追加する基準は高くなります。

この記事は、Humane Interfaceとの対比として、学習しやすさ、実装者の負担、公開後に削れないAPIの性質を理由に、最初は小さく始めることの利点を説明しています。

## 読むときの観点

- 便利さとAPIの小ささのどちらを優先する設計かを見る。
- クラスの責務を絞ることと、メソッド数を絞ることの関係を考える。
- 公開ライブラリとアプリケーション内部コードでは判断基準が違う点に注意する。
- 一度公開したAPIから機能を削る難しさを意識する。

## 原文の翻訳

Minimal InterfaceはAPI設計のスタイルであり、ここではHumane Interfaceと対比して説明します。Minimal Interfaceの考え方は、クライアントが必要なことをすべてできるAPIを設計しつつ、その能力を、仕事をこなせる合理的に最小のメソッド集合へ絞り込むことです。（違いのよい例についてはHumane Interfaceを参照してください。）

Humane Interfaceで使ったRubyの配列とJavaのリストの例を借りると、すでにインデクサと長さを得るメソッドがあるリストクラスには、`first` や `last` メソッドを入れないことになります。既存のインターフェースだけで `first` と `last` を実現できるからです。そのため `first` と `last` は便利メソッドです。Minimalistは便利メソッドをすべて避けるわけではありませんが、**便利メソッドが採用されるには高い基準を越える必要があります**。

Humane Interfaceを支持する議論はそちらにあります。ここではMinimal Interfaceの理由を述べます。

インターフェースを学ぶには時間がかかります。巨大なインターフェースを持つクラスは、うまく使われにくいでしょうし、そもそも使う気をそぐかもしれません。小さく焦点の合ったメソッド集合に保つことで、クライアントはそのクラスが何であり、何ができるのかを見つけやすくなります。

この焦点は、クラス設計者にとっても重要です。クラス設計でよくある問題は、クラスに多くをやらせすぎることです。本質に焦点を合わせると、余分なものをクラスから締め出し、ひとつの仕事に集中してそれをうまく行えるようにできます。

Humaneなアプローチに従うとして、どこで止めればよいのでしょうか。誰かが欲しがるかもしれないという理由でメソッドを追加し続けるなら、メソッドは際限なく増えていきます。したがって、このメソッド爆発を避けるための指針が必要です。Humaneな指針、つまり有用なものを提供するという指針は、恣意的で難しいものです。Minimalistの指針は単純です。クライアントが既存のメソッドでそれを実現できるなら、追加のメソッドは不要です。

なお、何が有用かを見極める問題は、アプリケーションコードよりも、公開されたクラスライブラリを書く人にとって大きな問題です。アプリケーションコードでは、用途が分かっています。閉じたシステムだからです。

JavaやC#の `interface` キーワードのように、静的型付けされた純粋なインターフェースを使っている場合、メソッド数を少なく保つもうひとつの理由があります。実装者の負担を減らせることです。多数のメソッドはすべて実装されなければならず、それは大きな作業量になります。（abstract classをmixinとして使うと、この負担を減らせることがあります。）

最小限のクラスにさらに機能が欲しければ、他のクラスを使えば実現できます。たとえばJavaでリストを反転またはソートしたい場合、RubyのArrayでは通常のメソッドである操作ですが、Javaでは `Collections` ユーティリティクラスを使います。

ライブラリに取り組んでいるとき、一度それを公開すると、何かを取り除くのは非常に困難です。そのため、大きすぎるものを作って削れなくなるより、**小さすぎるものから始めて後で追加するほうがよい**のです。
