# Seedwork

## 要約

Seedwork は、完成された再利用フレームワークではなく、必要に応じて自由に変更して育てるための最小限の機能を指します。

再利用可能なフレームワークは理想的に見えますが、作るのも使うのも難しく、アプリケーション境界をまたぐ再利用には社会的な難しさもあります。
Seedwork は完璧ではありませんが、より作りやすく使いやすい代替として価値があるかもしれません。

## 読むときの観点

- framework と seedwork の違いを、拡張するものか、改変して育てるものかで捉える。
- コピーアンドペースト再利用への違和感と、それでも有用かもしれないという再評価を読む。
- 共有ライブラリのバージョン管理問題が、再利用の難しさとして挙げられている点を見る。
- アプリケーション境界が技術だけでなく社会的構築物である、という主張に注目する。

## 原文の翻訳

オブジェクト指向のごく初期のころ、私のような OO 支持者は、再利用を擁護する議論に多くの注意を払っていました。初めのころは、クラスの再利用について話していました。その後、個々のクラスを再利用することは、うまくいく場合もあるものの、別の場面ではあまりうまくいかないことに気づきました。そこで再利用可能なフレームワークへ向かい、機能の一部が組み立てられたアプリケーションを得るようになりました。

技術面では、この種の再利用は成功してきました。Java や .NET のような環境で利用できる大規模なライブラリを見ればわかります。OO だけではありません。CPAN もそれを示しています。しかし、特にビジネス面では、そうした再利用はそれほど早く現れていません。また技術面についても、多くの人は、自分たちが扱うフレームワークの多くが目的に対して複雑すぎ、その複雑さが有用性の妨げになっていると感じています。

Michael Feathers の最近の weblog がこの問いを掘り下げ、その結果としての議論から、別の考え方である seedwork が出てきました。フレームワークは、必要なものを提供するために、制御された方法で拡張することを想定した、半焼けのアプリケーションであるはずです。Seedwork は、必要なものを得るために好きなように変更する最小限の機能です。もちろんこれは、seedwork に対する共通の更新を受け取る方法がなくなることを意味します。育てた後は、**それはあなたのものになる**のです。

これは、多くの人が、私も含めて、軽蔑してきた種類のコピーアンドペースト再利用です。

もしかすると、私はそこまで軽蔑すべきではないのかもしれません。フレームワークやライブラリは、十分に熟成していればとてもうまく機能します。しかし、良いフレームワークを得るのは非常に難しいことです。Seedwork は良いフレームワークほど有用ではありませんが、作るのも使うのも容易です。重要なのは、それが理想的かどうかではなく、**有用かどうか**です。

成熟した再利用でさえ、しばしば問題になります。私たちは、異なるスケジュールでアップグレードされる共有ライブラリにどう対処すべきかを、まだ本当に理解できていません。私たちは皆、Microsoft の DLL hell について不満を言ってきました。ちょうど今週、私の RedHat システムは、あるソフトウェアをインストールしようとしたときに詰まりました。バージョン依存関係がすっかり壊れていたのです。そのせいで半日が消えました。.NET のバージョン管理システムがこれを解決するのかもしれませんが、今のところ、優秀な人でさえ簡単に痛い目に遭います。

私は、アプリケーション内での再利用、あるいは重複の回避は不可欠だと考えています。しかしアプリケーションをまたぐ再利用はずっと難しいものです。主な理由は、Application Boundary が主に社会的な構築物だからです。これは、再利用可能なフレームワークが私たちの考える以上にずっと難しいという、さらなる証拠です。そして、seedwork のような、より完全ではない代替を検討すべきさらなる理由でもあります。
