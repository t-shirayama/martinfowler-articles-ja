# Technical Debt

## 要約

Technical Debtは、内部品質の欠陥であるcruftを、金融上の負債になぞらえて考える比喩です。cruftがあると、新しい機能を追加するたびに余計な時間がかかります。この余計な時間が利息であり、cruftを取り除く作業が元本の返済に当たります。

この比喩の価値は、どのcruftに取り組むべきかを判断し、非技術者と話しやすくする点にあります。ただし、金融の負債と違って時間が経つだけで利息が発生するわけではなく、その部分を変更するときに利息を払うという点に注意が必要です。

## 読むときの観点

- 技術的負債を、悪いコード全般のラベルではなく判断の比喩として読む。
- 利息は、変更時に余計にかかる時間として考える。
- あまり触らないcruftと、頻繁に変更するcruftを区別する。
- 負債を理由に品質を下げる判断が、実際には速度を落としていないかを見る。

## 原文の翻訳

著作権上の理由により、このファイルには原文本文の全文翻訳を収録していません。以下は、記事の主張を日本語で読むための要旨です。

ソフトウェアシステムにはcruftが蓄積しがちです。cruftとは、理想的な状態よりも変更や拡張を難しくしている内部品質上の欠陥です。Technical Debtは、Ward Cunninghamが作った比喩で、このcruftを金融上の負債のように考えるための枠組みです。

たとえば、モジュール構造が混乱しているコードベースに新機能を追加するとします。構造が明確なら4日で済む機能が、cruftのために6日かかるなら、余計な2日が負債の利息です。混乱した構造を取り除く作業は、元本を返済することに相当します。

この比喩が有用なのは、cruftへの対処を判断しやすくする点です。ひとつの機能だけを見ると、先に5日かけて構造を直すより、そのまま6日で機能を入れるほうが速いかもしれません。しかし同じ領域に似た変更が続くなら、先に元本を返済したほうが総時間は短くなる可能性があります。

ただし、これを単純なスプレッドシート問題にすることはできません。生産性や内部品質は客観的に測りにくく、見積もりの精度も限られます。そのため、多くの場合は金融の負債と同じように、少しずつ元本を返済するのが現実的です。機能追加のついでに周辺のcruftを少し取り除くと、将来の変更コストを下げられます。

この漸進的な改善には重要な利点があります。頻繁に変更する領域ほど自然に手を入れる機会が増え、まさにcruftを減らす必要が高い場所に時間が使われます。逆に、ひどいコードであってもほとんど変更しない領域なら、すぐ返済しなくてもよい場合があります。金融負債と違い、技術的負債の利息は時間の経過だけではなく、その部分を変更するときに発生します。

一方で、活動が多い領域ではcruftへの許容度を低くすべきです。変更が多いほど利息の支払いは大きくなり、内部品質に注意しない変更が重なるほどcruftも増えます。

技術的負債の比喩は、ときに内部品質の軽視を正当化するために使われます。急ぎの機能があるから意図的に負債を取る、という判断です。しかし多くの場合、この分析は甘くなります。cruftはすぐに新機能の開発を遅らせるため、結果として高品質に取り組んだ場合より遅くなることがあります。

Technical Debtは、すべての悪い設計を同じように扱う言葉ではありません。意図的か偶発的か、慎重か無謀かによって性質は異なります。それでも中心にある考えは同じです。**変更時に払う利息と、元本を返す費用を比べる**ことで、どのcruftにいつ取り組むかを判断します。
