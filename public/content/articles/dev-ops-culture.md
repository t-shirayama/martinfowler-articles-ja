# Dev Ops Culture

## 要約

DevOps は、開発と運用の間にあるサイロを取り除き、協働を促す動きです。
新しい運用ツールやアジャイルな技術プラクティスは助けになりますが、それだけでは DevOps の価値は実現できません。

中心にあるのは文化です。
共有責任、自律したチーム、品質の作り込み、フィードバック、そして自動化が、開発と運用の協働を支えます。

## 読むときの観点

- DevOps はツール名でも職種名でもなく、協働を促す文化的な変化である。
- 開発と運用が成功と失敗の両方に共同で責任を持つ。
- ハンドオーバーや承認の壁は、責任共有よりも責任分断を強めやすい。
- 自律したチームには、信頼と失敗への恐れが少ない環境が必要である。
- 自動化は、人を置き換えるだけでなく、協働と学習を助ける。

## 原文の翻訳

アジャイルソフトウェア開発は、要求分析、テスト、開発の間にあったサイロを壊してきた。デプロイ、運用、保守もまた、ソフトウェア開発プロセスの他の部分から切り離されがちな活動だった。DevOps の動きは、こうしたサイロを取り除き、開発と運用の協働を促すことを目指している。

DevOps が可能になった背景には、仮想化、クラウドコンピューティング、自動構成管理といった運用ツールと、継続的インテグレーション、進化的設計、クリーンコードのようなアジャイルなエンジニアリングプラクティスがある。しかし、それだけでは不十分である。どれほどよいツールがあっても、適切な文化がなければ DevOps は単なる流行語になる。

DevOps 文化の第一の特徴は、開発と運用の役割間で協働が増えることだ。この協働を支える文化的な変化は、チーム内にも組織全体にも必要になる。

共有責任の態度は、DevOps 文化の重要な側面である。開発チームがシステムを別チームへ引き渡してしまうと、運用や保守への関心を失いやすい。開発チームがシステムのライフタイムを通じて面倒を見る責任を共有すれば、運用担当者の痛みを理解し、デプロイや保守を簡単にする方法を見つけやすくなる。

同時に、運用担当者がシステムのビジネス目標に責任を共有すれば、開発者とより密に働き、システムの運用上のニーズを理解して満たしやすくなる。実践では、開発者がデプロイや監視などの運用上の関心を理解し、運用担当者が新しい自動化ツールやプラクティスを採用するところから協働が始まることが多い。

共有責任の文化を支えるには、組織上の変化も必要である。開発と運用の間にサイロを置くべきではない。ハンドオーバー期間やドキュメントは、最初から一緒に解決策を作ることの代替にはならない。運用担当者が早い段階からチームに関われるよう、リソース構造を調整することが役に立つ。

ハンドオーバーやサインオフは、人々が責任を共有することを妨げ、非難の文化を助長しやすい。開発者と運用担当者は、システムの成功にも失敗にも共に責任を持つべきである。DevOps 文化は、開発者と運用担当者の役割の境界をぼかし、最終的には区別を不要にするかもしれない。

DevOps 導入時によくあるアンチパターンは、誰かに「DevOps」という役割を割り当てたり、あるチームを「DevOps チーム」と呼んだりすることだ。これは DevOps が壊そうとしているサイロを温存し、文化やプラクティスが組織全体に広がるのを妨げる。

もうひとつ重要な組織的変化は、自律したチームを支えることだ。開発者と運用担当者が効果的に協働するには、複雑な意思決定プロセスなしに判断し、変更を適用できる必要がある。そのためには、チームを信頼し、リスク管理の方法を変え、失敗への恐れが少ない環境を作ることが求められる。

本番へ新しいコードを出しやすくなると、さらに別の文化的変化も必要になる。変更が健全であることを確かめるため、チームは開発プロセスの中に品質を作り込む価値を重視しなければならない。性能やセキュリティのような横断的関心も含まれる。

フィードバックを重視することも重要である。開発者と運用担当者の働き方、そしてシステムそのものを継続的に改善するには、フィードバックループが必要だ。本番監視は、問題を診断し改善の可能性を見つける有用なループになる。

自動化は DevOps の基礎であり、協働を促進する。テスト、構成、デプロイなどを自動化すれば、人はより価値の高い活動に集中でき、人為的ミスも減らせる。さらに、自動化されたスクリプトやテストは、システムについての常に新しいドキュメントとしても役立つ。

たとえばサーバー構成を自動化すれば、Snowflake Server にありがちな推測を取り除ける。開発者も運用担当者も、サーバーがどのように構成されているかを同じように知り、変更できるようになる。DevOps 文化では、**ツールは文化を支える手段**であり、文化そのものではない。
