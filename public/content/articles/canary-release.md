# Canary Release

## 要約

Canary Release は、新しいソフトウェアバージョンをいきなり全体へ展開せず、少数のユーザーや一部のインフラに段階的に流して、本番でのリスクを下げる技法です。問題があれば、ユーザーを旧バージョンへ戻すことでロールバックできます。

この技法は phased rollout や incremental rollout とも呼ばれます。A/B testing と実装は似ることがありますが、目的は異なります。Canary Release は問題や回帰の早期検出を狙い、A/B testing は仮説検証を狙います。

## 読むときの観点

- 新バージョンを本番の一部だけに流し、徐々に信頼を高める流れを見る。
- ルーティング対象を、ランダム、社内ユーザー、地域、ブランドなどで選べる。
- A/B testing と混同せず、目的の違いを押さえる。
- 複数バージョン管理、配布型クライアント、DB 変更が難所になる。

## 原文の翻訳

Canary release は、新しいソフトウェアバージョンを本番へ導入するリスクを下げる技法です。変更をインフラ全体へ展開し、すべての人に使えるようにする前に、少数のユーザーへゆっくり展開します。

BlueGreenDeployment と同じように、まずはユーザーがルーティングされていないインフラの一部へ、新しいバージョンのソフトウェアをデプロイします。

新しいバージョンに満足できたら、選ばれた少数のユーザーをそこへルーティングし始めます。どのユーザーが新バージョンを見るかには、いくつかの戦略があります。単純な方法はランダムサンプルを使うことです。世界へ出す前に社内ユーザーや従業員へリリースする会社もあります。より高度な方法として、ユーザーの profile やその他の demographics に基づいて選ぶこともできます。

新しいバージョンへの信頼が高まるにつれて、インフラ内のより多くのサーバーへリリースし、より多くのユーザーをそこへルーティングし始めます。新バージョンを rollout する良い実践は、PhoenixServers を使って既存インフラを再利用すること、または ImmutableServers を使って新しいインフラを provision し、古いものを廃止することです。

Canary release は ParallelChange の応用であり、migrate phase はすべてのユーザーが新バージョンへルーティングされるまで続きます。その時点で、古いインフラを廃止できます。新バージョンに問題が見つかった場合の rollback strategy は単純で、問題を修正するまでユーザーを旧バージョンへ戻すことです。

Canary release の利点のひとつは、安全な rollback strategy を持ったまま、本番環境で新バージョンの capacity testing ができることです。負荷を徐々に高めることで、新バージョンが本番環境に与える影響を監視し、metrics を集められます。これは完全に別の capacity testing environment を作る代替になります。なぜなら、その環境は可能な限り本番に近いものになるからです。

この技法の名前は馴染みがないかもしれませんが、canary releasing の実践は以前から採用されてきました。phased rollout や incremental rollout と呼ばれることもあります。この名前は、鉱夫が鳥かごに入れた canary を炭鉱へ連れていったことに由来します。有毒ガスが漏れた場合、鉱夫より先に canary が倒れます。Canary release も同じように、**本番インフラやユーザー全体へ影響が及ぶ前の早期警告**を与えます。

大規模で分散したシナリオでは、どのユーザーを新バージョンへリダイレクトするかを router で決める代わりに、別の partitioning strategy を使うことも一般的です。たとえば、地理的に分散したユーザーがいるなら、新バージョンをまず特定の region や location へ rollout できます。複数の brand があるなら、最初に1つの brand へ rollout できます。Facebook は複数の canary を使う戦略を選んでおり、最初の canary は社内従業員だけに見え、すべての FeatureFlags をオンにします。これにより、新機能の問題を早く見つけられます。

Canary release は、技術的な実装が似ているため、A/B testing を実装する方法として使われることがあります。しかし、この2つの関心事を混同しないほうがよいです。Canary release は問題や regression を検出する良い方法であり、A/B testing は variant implementation を使って仮説を検証する方法です。canary で business metrics を監視して regression を検出している場合、同時に A/B testing にも使うと結果に干渉する可能性があります。

より実務的には、A/B test で統計的有意性を示すのに十分なデータを集めるには数日かかることがあります。一方で、canary rollout は数分から数時間で完了させたいものです。

Canary release の欠点のひとつは、複数バージョンのソフトウェアを同時に管理しなければならないことです。本番で3つ以上のバージョンを同時に動かすこともできますが、同時に存在するバージョン数は最小限に保つのが最善です。

Canary release が難しい別のシナリオは、ユーザーのコンピューターやモバイルデバイスにインストールされるソフトウェアを配布する場合です。この場合、新バージョンへの upgrade がいつ起きるかを制御しにくくなります。その配布ソフトウェアが backend と通信するなら、ParallelChange を使って両方のバージョンを支え、どの client version が使われているかを監視できます。利用数がある水準まで下がったら、backend を contract して新バージョンだけを支えるようにできます。

Canary release でデータベース変更を扱う場合も注意が必要です。ここでも ParallelChange は問題を緩和する技法です。rollout phase の間、データベースがアプリケーションの両方のバージョンを支えられるようにします。
