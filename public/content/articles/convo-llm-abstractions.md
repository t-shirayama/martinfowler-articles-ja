# Conversation: LLMs and Building Abstractions

## 要約

Unmesh Joshi と Martin Fowler は、LLM がソフトウェア開発にどう役立つかを、抽象化を作る営みとの関係から議論しています。LLM は、設定作業、定型コード、既に安定したパターンの反復など、偶発的複雑さを減らす場面で強力です。しかし、ソフトウェアの本質的な難しさは、名前を選び、責務を分け、問題領域と解決領域の間で意味のある抽象を育てるところにあります。

この記事の中心は、プログラミングには「抽象を発見し安定させる活動」と「安定した抽象を適用する活動」がある、という区別です。前者では開発者が主導権を持ち、LLM は代替案を素早く見せる会話相手として使うのがよい。後者では、チームが育てた語彙に従って、LLM に反復的な実装を任せやすくなります。

## 読むときの観点

- LLM の価値を「コード生成量」ではなく、偶発的複雑さと本質的複雑さのどちらに効いているかで見る。
- プログラミングを、単に構文を打ち込む行為ではなく、問題を表す語彙を育てる行為として読む。
- 抽象を発見する段階では、プロンプトで一気に決めるより、コードとの往復で理解を深める必要がある点に注目する。
- 安定した抽象があるほど、LLM はより信頼できる定型実装の担い手になる。
- ドメインモデルは業務知識だけでなく、アーキテクチャや技術選択という solution domain と交わるところで育つ。

## 原文の翻訳

Unmesh Joshi と Martin Fowler は、プログラマが LLM とどう働くかについてメールで考えを交わし、それを公開向けの会話として整えました。出発点の一つは、Fred Brooks の「銀の弾などない」にある、偶発的複雑さと本質的複雑さの区別です。

Unmesh は、この区別が LLM の使いどころを考えるうえで役立つと言います。言語の構文、フレームワーク統合、ボイラープレート、プロジェクトの初期設定などは、主に偶発的複雑さです。LLM はこの種の作業を減らすのが得意です。たとえば、Spring Boot、Kafka、OpenTelemetry を組み合わせたサービスのひな形を素早く作ることはできます。

しかし、ソフトウェアを書くことは、コンパイルできるコードを出すことだけではありません。問題を分け、関連するデータと振る舞いを束ね、意図が伝わる名前を選び、解決策の構造を少しずつ形にしていく活動です。良い名前は複雑さを切り開き、コードを読み手がたどれる図式に変えます。

最初の日に全体設計が見えているわけではありません。少し考え、少し書き、立ち止まり、見えてきたものを磨く。この往復の中で、コードと理解の両方が鋭くなります。Unmesh は、最初の仕様はせいぜい仮説であり、ウォーターフォールはこの制約と戦い、アジャイルはそれを受け入れた、と述べます。

Fowler は、この話を Domain-Driven Design の Ubiquitous Language に結びつけます。LLM とのやり取りも、ある意味では言語を作っていくことです。Unmesh はさらに、Guy Steele の講演を踏まえ、「言語を作る」というより「言語を育てる」と言うほうがよいと述べます。

プログラミングとは、働く語彙を育てることです。Repository、Controller、Factory のような抽象は、プログラミング共同体の中で共有された言葉です。関数型、オブジェクト指向、特定の言語やパラダイムも、プログラマが問題をどう捉えるかに影響します。

コードを書きながら、私たちは解決策の構造を学んでいます。クラスを分けるべきか、関数名は何がよいか、階層を別の形にすべきか。こうした細かな判断は、コードとの往復の中で起こります。LLM は、すでに決まった具体的な判断に従ってコードを生成・変更するのは得意です。しかし、その判断に至るには、開発者自身がコードを通じて考える必要があります。

すべてをプロンプトで進めようとすると、そこには大きな upfront design の気配があります。コードと反復せずに設計構造を推測し、抽象化の判断を LLM に渡してしまうからです。後からレビューすればよいという反論はありますが、判断がすでにコード化された後のレビューは、進行中の対話ほど効果的ではありません。これは、受け身のレビューよりペアプログラミングを好む理由にも通じます。

Fowler は、プルリクエスト中心の文化では「作ったものを見せてレビューしてもらう」形に慣れやすい一方、ペアプログラミングではより高頻度で、役割も流動的に入れ替わると指摘します。ペアリングに慣れた人は、LLM とも、完成案を投げるより、考えを一緒に育てるように付き合いやすいでしょう。

Unmesh は、プログラミングには二つの活動が深く絡み合っていると言います。一つは、抽象を発見し安定させる活動です。複数の選択肢を試し、共通するものと将来変わりそうなものを見極め、安定した構造へ収束していく創造的な段階です。もう一つは、安定した抽象を使って新しいユースケースや変種を実装する活動です。

この二つでは、LLM との働き方が違います。抽象が安定していれば、それに基づくコード生成はより機械的に扱えます。チームが言語を定義していれば、LLM はその言語を使って実装できます。プロジェクト設定、ビルドパイプライン、CI/CD、確立済みパターンの反復などは、LLM に任せやすい領域です。

一方で、抽象を発見する段階でも LLM は役に立ちます。小さなスニペットや別案を素早く出し、名前や境界を試す助けになります。このとき Unmesh は、既存実装を壊す心配の少ない chat mode を好みます。別の言語に書き換える language projection も、実装上の選択肢を短時間で比較する方法として有効です。

Fowler は、自分がコンパイラとの短い反復を通じてコードを改善してきた経験を重ねます。フィードバックループが速くなると、事前にすべてを点検するより、反復するほうが有効になります。抽象が作った言語で振る舞いをうまく表現できないなら、その抽象をさらに磨く必要があるという合図です。コンパイラとテストは、抽象の限界を突きつけてくれます。

LLM は、そのコンピュータとの協働の延長にあるものとして見られます。曖昧な考えに対して、より豊かなフィードバックを返せる新しい仕組みです。ただし、このプロセスは静的なプロンプトには還元できません。LLM が生成したコードを読むだけでは、自分でコードを書くときに起こる深い思考を逃しやすいからです。

抽象の発見段階では、LLM はブレインストーミング相手として最も有効です。代替設計や構造の案を出してもらうことはできますが、核となる実装をそのまま任せることには慎重であるべきです。**抽象を発見し安定させる段階の主導権は開発者にある**、というのがこの記事の重要な線引きです。

最後に Unmesh は、ドメインモデルの語彙は機能的な業務領域だけから生まれるわけではないと述べます。Web システムかイベント駆動か、リレーショナルデータベースかデータグリッドか、同期 API か非同期メッセージングか。こうした solution domain の選択も抽象を形作ります。

そのため、よい語彙は、業務に詳しい人と技術に詳しい人が密に協力するところから生まれます。LLM が登場しても、この事実は変わりません。むしろ共有されたドメイン語彙を作れないことのコストは高くなります。語彙がなければ、LLM は具体的な参照点を持てず、意味を推測し、的外れな結果を出しやすくなるからです。
