# Testing Strategies in a Microservice Architecture

## 要約

マイクロサービスアーキテクチャでは、システムが小さく独立してデプロイできるサービス群に分かれるため、テストの境界も増えます。この記事は、単体テスト、統合テスト、コンポーネントテスト、契約テスト、エンドツーエンドテストをどう組み合わせ、どのリスクをどの粒度で扱うかを整理したインフォデッキです。

中心にある考え方は、すべてを大きなエンドツーエンドテストで確かめるのではなく、できるだけ細かい粒度で早いフィードバックを得ながら、サービス間の契約やシステム全体の振る舞いも必要な範囲で検証することです。テストピラミッドを守り、重く壊れやすいテストを増やしすぎないことが、マイクロサービスでも重要になります。

## 読むときの観点

- サービス内部、サービス境界、システム全体のどこで何を保証するのかを分けて読む。
- 外部サービスやデータストアとの通信は、ネットワーク境界をまたぐリスクとして扱う。
- 契約テストを、利用側と提供側の変更を安全にする仕組みとして捉える。
- エンドツーエンドテストは数を絞り、重要なユーザージャーニーに集中させる。
- テスト戦略はアーキテクチャ設計やチーム境界と切り離せないものとして読む。

### 概要

ここ数年、サービスベースのアーキテクチャでは、より小さく、より焦点を絞った「マイクロ」サービスへ向かう変化が起きています。このアプローチには、各コンポーネントを独立してデプロイ、スケール、保守できることや、複数チームで並行して開発を進められることなど、多くの利点があります。しかし、こうした追加のネットワーク分断が導入されると、モノリシックなインプロセスアプリケーションに適用していたテスト戦略を見直す必要があります。

ここでは、独立してデプロイできる複数コンポーネントによって増えるテストの複雑さを管理するためのいくつかのアプローチを取り上げます。また、複数のチームがそれぞれ異なるサービスの守り手として振る舞う状況で、テストとアプリケーションをどのように正しい状態に保つかも説明します。

このインフォデッキは、マイクロサービスアーキテクチャのスタイルが効果的なテストの構成に突きつける課題を扱い、必要になるテストの種類と、それらをどう組み合わせるかを概説します。

### マイクロサービスとは何か

マイクロサービスアーキテクチャは、協調するサービス群としてソフトウェアを構築します。

マイクロサービスアーキテクチャは、単一責任原則をアーキテクチャレベルに適用した自然な帰結です。その結果、従来のモノリシックアーキテクチャに比べて、独立したデプロイ可能性、コンポーネントごとの言語・プラットフォーム・技術の独立性、異なる軸でのスケーラビリティ、そして高いアーキテクチャ上の柔軟性といった利点が得られます。

大きさについて、厳密な決まりはありません。一般に、マイクロサービスは数百行程度であることが多いものの、内包する責任によっては数十行にも数千行にもなり得ます。具体的ではないものの有用な経験則は、所有するドメイン概念を表すために、**可能な限り小さく、必要なだけ大きく**することです。「How big should a micro-service be?」には、さらに詳しい説明があります。

マイクロサービスは、多くの場合 HTTP 上の REST を使って統合されます。この方法では、ビジネスドメインの概念をリソースとしてモデル化し、各サービスがそのうち1つ以上を管理します。成熟した RESTful システムでは、リソースはハイパーメディアコントロールによってリンクされ、各リソースの場所はサービス利用者から見て不透明になります。詳細は Richardson Maturity Model を参照してください。

軽量なメッセージングプロトコル、publish-subscribe モデル、Protobuf や Thrift のような別の転送方式など、代替の統合機構が使われることもあります。

各マイクロサービスは、何らかのユーザーインターフェースを提供する場合もあれば、提供しない場合もあります。

### マイクロサービスの内部構造

マイクロサービスは、通常、似た種類のモジュールに分けられます。

多くの場合、マイクロサービスは、表示されている層の一部または全部からなる似た内部構造を持っています。採用するテスト戦略は、軽量さを保ちつつ、サービスの各層と層の間をカバーすることを目指すべきです。

リソースは、サービスが公開するアプリケーションプロトコルと、ドメインを表すオブジェクトへのメッセージとの間を対応づける mapper として機能します。通常、リソースは薄く、リクエストの妥当性を確認し、ビジネストランザクションの結果に応じてプロトコル固有のレスポンスを返す責任を持ちます。

サービスロジックのほぼすべては、ビジネスドメインを表す domain model に存在します。これらのオブジェクトのうち、services は複数のドメイン活動をまたいで調整し、repositories はドメインエンティティの集合を扱い、多くの場合は永続化に支えられています。

あるサービスが別のサービスを協力相手として持つ場合、外部サービスと通信するためのロジックが必要になります。gateway はリモートサービスとのメッセージの受け渡しをカプセル化し、リクエストとレスポンスをドメインオブジェクトとの間で整形します。おそらく、基盤となるプロトコルを理解するクライアントを使い、リクエスト・レスポンスのサイクルを処理します。

ごく単純な場合や、サービスが他のサービスの所有するリソースをまたぐ aggregator として振る舞う場合を除けば、マイクロサービスはリクエスト間でドメインのオブジェクトを永続化できる必要があります。通常これは、永続化要件の複雑さに応じて、オブジェクトリレーショナルマッピングまたはより軽量な data mapper によって実現されます。

多くの場合、このロジックは、ドメインの repositories から利用される専用オブジェクト群にカプセル化されます。

### ネットワークと外部データストア

マイクロサービスはネットワーク越しに互いに接続し、「外部」のデータストアも利用します。

マイクロサービスは、関連する各モジュールの間でメッセージを渡しながらリクエストを処理し、レスポンスを形成します。特定のリクエストでは、services、gateways、repositories とのやり取りが必要になることがあるため、モジュール間の接続はゆるやかに定義されます。

自動テストは、こうした通信のそれぞれを、**可能な限り細かい粒度**でカバーすべきです。そうすることで、各テストが焦点の絞られた高速なフィードバックサイクルを提供できます。

リソースはリクエストを受け取り、妥当性を確認したうえで、そのリクエストの処理を開始するためにドメインを呼び出します。

ビジネストランザクションを完了するために多くのモジュールを調整しなければならない場合、リソースは service に委譲します。そうでなければ、関連するモジュールと直接やり取りします。

外部サービスへの接続はネットワーク境界を越えるため、特別な注意が必要です。システムはリモートコンポーネントの停止に耐えられるべきです。gateways には、そのようなエラーケースを扱うロジックが含まれます。

通常、外部サービスとの通信は、API の過度な呼び出しとレイテンシを避けるため、同等のインプロセス通信よりも粗い粒度になります。

同様に、外部データストアとの通信にも異なる設計上の考慮が必要です。サービスは外部サービスよりもデータストアに論理的に強く結合していることが多いものの、データストアもネットワーク境界の向こう側にあり、レイテンシと停止リスクを伴います。

ネットワーク分断の存在は、採用するテストのスタイルに影響します。これらのモジュールのテストは実行時間が長くなることがあり、チームの制御外の理由で失敗することもあります。

### 複数サービスからなるシステム

複数のサービスは、システムとして協調し、ビジネス上価値のある機能を提供します。

通常、1つのチームは1つ以上のマイクロサービスの守り手として振る舞います。これらのサービスは、より大きなビジネスリクエストを処理するためにメッセージを交換します。交換フォーマットとしては現在 JSON が最も一般的ですが、代替手段はいくつもあり、その中では XML が最もよく使われています。

場合によっては、同期的な point-to-point 方式よりも、非同期の publish-subscribe 通信機構のほうがユースケースに適していることがあります。Atom syndication format は、マイクロサービス間の pub-sub を実装する軽量な手段として、ますます人気が高まっています。

ビジネスリクエストがネットワーク分断によって隔てられた複数コンポーネントにまたがるため、システム内で起こり得る故障モードを考えることが重要です。timeouts、circuit breakers、bulkheads といった技術は、コンポーネント停止にもかかわらず、システム全体の稼働時間を保つ助けになります。

大規模なシステムでは、多くの場合、複数のチームがそれぞれ異なる bounded contexts に責任を持ちます。

外部サービスに関するテスト上の関心事は、自チームの管理下にあるサービスに関する関心事とは異なることがあります。外部チームのサービスについては、そのインターフェースや可用性に関して保証できることが少ないからです。

### 単体テスト

単体テストは、アプリケーション内のテスト可能な最小のソフトウェア片を実行し、それが期待どおりに振る舞うかを判断します。

テスト対象の単位の大きさは厳密には定義されていません。しかし、単体テストは通常、クラス単位、または関連する小さなクラス群を中心に書かれます。テスト対象の単位が小さいほど、その単位の分岐の複雑さは低くなるため、単体テストで振る舞いを表現しやすくなります。

単体テストを書くのが難しい場合、それはしばしば、モジュールをより独立した、より一貫性のある部分に分割して個別にテストすべきだという兆候になります。したがって単体テストは、有用なテスト戦略であると同時に、特にテスト駆動開発と組み合わせたときに、**強力な設計ツール**でもあります。

単体テストでは、テスト対象の単位を協力オブジェクトから分離するかどうかに基づく重要な違いがあります。

Sociable unit testing は、状態の変化を観察することでモジュールの振る舞いをテストすることに焦点を当てます。これは、テスト対象の単位をブラックボックスとして扱い、そのインターフェースだけを通してテストします。

Solitary unit testing は、オブジェクトとその依存先との相互作用や協調に注目します。依存先は test doubles に置き換えられます。

これらのスタイルは競合するものではなく、異なるテスト上の問題を解くために、同じコードベースで頻繁に併用されます。

### マイクロサービス内の単体テスト

どちらのスタイルの単体テストも、マイクロサービスの内部で重要な役割を果たします。

サービスは一般に、豊かなドメインを、配管コードと調整コードが取り囲む形になります。

ドメインロジックは、多くの場合、複雑な計算や状態遷移の集まりとして現れます。この種のロジックは状態に強く基づくため、単位を分離しようとしても得られる価値はほとんどありません。つまり、可能な限り、テスト対象の単位のすべての協力オブジェクトには実際のドメインオブジェクトを使うべきです。

配管コードでは、テスト対象の単位を外部モジュールから分離することと、状態変化に対してテストすることの両方を行うのは難しくなります。そのため、test doubles を使うほうが効果的です。

このレベルの単体テストの目的は、外部依存へのリクエストを生成したり、外部依存からのレスポンスをマッピングしたりするために使われるロジックを検証することであり、統合された形で通信を検証することではありません。したがって、協力オブジェクトに test doubles を使うことで、リクエスト・レスポンスのサイクルを信頼性高く、繰り返し可能な形で制御できます。

このレベルの単体テストは、統合テストよりも高速なフィードバックを提供し、例外的な状況で外部依存がそうするように doubles に応答させることで、エラー条件を強制できます。

調整ロジックがより気にするのは、モジュール内部の複雑なロジックよりも、モジュール間で渡されるメッセージです。test doubles を使うことで、渡されるメッセージの詳細を検証し、レスポンスを stub して、モジュール内の通信の流れをテストから指定できます。

ある調整ロジックがあまりにも多くの doubles を必要とするなら、それは通常、何らかの概念を抽出して分離してテストすべきだという良い指標です。

サービスのサイズが小さくなるほど、複雑なドメインロジックに対する配管コードと調整コードの比率は高まります。同様に、異なる技術への adapter や他サービス上の aggregator のように、配管コードと調整コードだけで構成されるサービスもあります。

そのような場合、包括的な単体テストは見合わないかもしれません。コンポーネントテストのような別レベルのテストのほうが、より大きな価値を提供できます。

単体テスト、そしてテスト一般の意図は、テスト対象の単位の振る舞いを制約することです。不幸な副作用として、ときにはテストが実装まで制約してしまうことがあります。これは、mock ベースのアプローチへの過度な依存として現れることがよくあります。

単体テストが提供する価値と、その保守コストや実装をどれだけ制約するかを、常に問い直すことが重要です。そうすることで、テストスイートを小さく、焦点の合った、価値の高いものに保てます。

### 単体テストだけでは不十分

単体テストだけでは、システムの振る舞いについて保証を与えられません。

ここまでで、システムの中核となる各モジュールについて、分離された状態での十分なカバレッジは得られています。しかし、それらのモジュールが連携して完全なサービスを形成するときのカバレッジや、リモート依存との相互作用に関するカバレッジはありません。

各モジュールが協力オブジェクトと正しく相互作用することを検証するには、より粗い粒度のテストが必要です。

### 統合テスト

統合テストは、コンポーネント間の通信経路と相互作用を検証し、インターフェースの欠陥を検出します。

統合テストは、複数のモジュールを集めてサブシステムとしてテストし、それらが意図どおりに協調して、より大きな振る舞いを実現することを検証します。統合テストは、サブシステム内の通信経路を実行し、各モジュールが仲間とどう相互作用するかについて誤った前提を持っていないかを確認します。

これは、実際の協力オブジェクトを使う場合であっても、テスト対象の単位の振る舞いを細かくテストすることを目的とする単体テストとは対照的です。コンポーネントやモジュールを統合するテストはどの粒度でも書けますが、マイクロサービスアーキテクチャでは通常、統合コードの層と、それが統合しようとしている外部コンポーネントとの相互作用を検証するために使われます。

この種の統合テストが有用な外部コンポーネントの例には、他のマイクロサービス、データストア、キャッシュがあります。

### データストアと外部コンポーネントの統合

データストアや外部コンポーネントとの統合は、統合テストによる高速なフィードバックの恩恵を受けます。

外部コンポーネントと相互作用するモジュールの自動テストを書くときの目的は、その外部コンポーネントを受け入れテストすることではなく、モジュールが十分に通信できることを検証することです。したがって、この種のテストは、統合モジュールを通る基本的な成功パスとエラーパスをカバーすることを目指すべきです。

gateway の統合テストにより、HTTP ヘッダーの不足、不正な SSL 処理、リクエスト・レスポンスボディの不一致といったプロトコルレベルのエラーを、可能な限り細かいテスト粒度で洗い出せます。

例外的な状況で、サービスと利用しているプロトコルクライアントが期待どおりに応答することを確かめるため、特殊なエラー処理もテストすべきです。

外部コンポーネントからタイムアウトや遅いレスポンスのような異常な振る舞いを引き起こすのが難しい場合があります。この場合、事前に定めた形で失敗するよう設定できる、外部コンポーネントの stub 版をテストハーネスとして使うと有益です。

外部コンポーネントに対してテストする場合、テストが特定のデータの存在に依存するため、状態管理が難しくなることがあります。この問題を軽減する1つの方法は、代表的で害のない固定データセットを、すべての環境で必ず利用できるよう合意しておくことです。

永続化の統合テストは、コードが前提としているスキーマがデータストアに存在するスキーマと一致していることを保証します。

ORM を使っている場合、これらのテストは、ツールに設定されたマッピングが返ってくる結果セットと互換性を持つことへの信頼も与えます。

現代の ORM は、キャッシュや必要なときだけ flush する点で非常に洗練されています。データが完全に往復することを確実にするために、事前条件、アクション、アサーションの間でトランザクションが閉じられるようにテストを構成することが重要です。

ほとんどのデータストアはネットワーク分断の向こう側にあるため、タイムアウトやネットワーク障害の影響も受けます。統合テストは、統合モジュールがこれらの障害を適切に扱うことを検証しようとすべきです。

このスタイルのテストは、統合モジュールに含まれるロジックをリファクタリングまたは拡張するときに高速なフィードバックを提供します。しかし、失敗する理由も複数あります。統合モジュールのロジックが退行した場合もあれば、外部コンポーネントが利用不能になったり契約を破ったりした場合もあります。

この問題を軽減するには、必要なときに高速なフィードバックを得るための統合テストは少数にとどめ、統合境界の両側を包括的に検証するための追加カバレッジを単体テストと契約テストで提供します。また、外部の停止が開発を妨げないように、CI ビルドパイプラインで統合テストを分離することも理にかなう場合があります。

### より粗いテストが必要になる

マイクロサービスに対するより粗い粒度のテストがなければ、ビジネス要件が満たされているという確信はありません。

単体テストと統合テストにより、マイクロサービスを構成する個々のモジュールに含まれるロジックの正しさには信頼を持てます。

しかし、より粗い粒度のテストスイートがなければ、マイクロサービスが全体として連携し、ビジネス要件を満たすことは確信できません。

これは完全に統合されたエンドツーエンドテストでも達成できますが、外部依存から分離してマイクロサービスをテストすることで、より正確なテストフィードバックと、より短いテスト実行時間を得られます。

### コンポーネントテスト

コンポーネントテストは、実行されるソフトウェアの範囲をシステムの一部に限定し、内部コードインターフェースを通じてシステムを操作し、test doubles を使ってテスト対象コードを他のコンポーネントから分離します。

コンポーネントとは、より大きなシステムの中にある、十分にカプセル化され、一貫性があり、独立して置き換えられる任意の部分です。

そのようなコンポーネントを分離してテストすることには、いくつもの利点があります。範囲を単一コンポーネントに限定することで、そのコンポーネントにカプセル化された振る舞いを徹底的に受け入れテストしつつ、broad stack の同等テストよりも速く実行されるテストを維持できます。

test doubles を使ってコンポーネントを仲間から分離することで、仲間が持つかもしれない複雑な振る舞いを避けられます。また、コンポーネントのために制御されたテスト環境を提供し、該当するエラーケースを繰り返し可能な形で引き起こすことにも役立ちます。

マイクロサービスアーキテクチャでは、コンポーネントとはサービスそのものです。この粒度でテストを書くことで、API の契約は利用者の視点からテストによって駆動されます。サービスの分離は、外部の協力相手を test doubles に置き換え、内部 API エンドポイントを使ってサービスを調査または設定することで実現します。

この種のテストの実装にはいくつかの選択肢があります。テストはサービスと同じプロセスで実行すべきでしょうか、それともネットワーク越しの別プロセスで実行すべきでしょうか。test doubles はサービスの内部に置くべきでしょうか、それとも外部に置いてネットワーク越しに到達するべきでしょうか。実際のデータストアを使うべきでしょうか、それともインメモリの代替に置き換えるべきでしょうか。次のセクションで、これをさらに説明します。

### インプロセスのコンポーネントテスト

インプロセスのコンポーネントテストは、動く部品を最小化しながら、包括的なテストを可能にします。

インメモリの test doubles とデータストアを使って、完全なマイクロサービスをメモリ内でインスタンス化することで、ネットワークにまったく触れないコンポーネントテストを書くことができます。

これにより、テスト実行時間を短くでき、動く部品の数を最小化してビルドの複雑さを減らせます。

ただしこれは、テスト対象の成果物をテスト目的で変更し、「test」モードで起動できるようにしなければならないことも意味します。依存性注入フレームワークは、起動時に提供される設定に基づいてアプリケーションの配線を変えることで、これを実現する助けになります。

テストは内部インターフェースを通じてマイクロサービスと通信し、リクエストを dispatch し、レスポンスを取得できるようにします。これを実現するために独自の shim が使われることがよくありますが、JVM ベースのマイクロサービス向けの inproctester や、.NET ベースのマイクロサービス向けの plasma のような、あらかじめ用意されたライブラリもいくつか存在します。

このようにして、インプロセスのコンポーネントテストは、実際のネットワーク相互作用に伴う追加オーバーヘッドを発生させずに、サービスに対して実際の HTTP リクエストを実行する場合にできるだけ近づけます。

マイクロサービスを外部サービスから分離するために、gateways は実際のプロトコルレベルのクライアントではなく test doubles を使うよう設定できます。内部リソースを使えば、特定のリクエストが一致したときに事前定義されたレスポンスを返すよう、これらの test doubles をプログラムできます。

これらの test doubles は、外部の協力相手がオフラインである、応答が遅い、不正な形式のレスポンスを返すといった、コンポーネントを通る unhappy paths を模倣するためにも使えます。これにより、エラー条件を制御された繰り返し可能な方法でテストできます。

外部データストアをインメモリ実装に置き換えることで、テスト性能を大きく改善できます。これは実際のデータストアをテスト境界から外すことになりますが、永続化の統合テストが十分なカバレッジを提供します。

場合によっては、採用している永続化機構が十分に単純で、軽量な独自実装を使えることがあります。あるいは、cassandra や elasticsearch のように組み込み実装を提供するデータストアもあります。H2 database engine のように、外部データストアをインメモリで模倣するツールもあります。

インプロセスの受け入れテストを書くときに、test doubles やセットアップデータを直接設定することも可能ですが、すべてのリクエストを特権的な内部リソース経由にすることで、サービスをよりブラックボックスとしてテストできます。これにより、永続化技術や外部サービス通信を変更しても、コンポーネントテストスイートに影響を与えずに済みます。

### 内部リソース

内部リソースは、テスト以外にも役立ちます。

奇妙に思えるかもしれませんが、内部制御をリソースとして公開することは、テスト以外にも、監視、保守、デバッグなど多くの場合に有用です。RESTful API の一貫性により、そのようなリソースと相互作用するためのツールはすでに多く存在し、全体の運用上の複雑さを減らす助けになります。

通常公開される内部リソースの種類には、ログ、feature flags、データベースコマンド、システムメトリクスがあります。多くのマイクロサービスには、サービスとその依存先の健全性、主要トランザクションのタイミング、設定パラメータの詳細を提供する health check リソースも含まれます。単純な ping リソースも、ロードバランシングの助けとして有用です。

これらのリソースは、持っている制御権や公開する情報の点でより特権的であるため、多くの場合、独自の認証を必要とするか、ネットワークレベルでロックダウンされます。内部制御を構成する API の部分を URL 命名規約で名前空間化したり、別のネットワークポートで公開したりすることで、ファイアウォールレベルでアクセスを制限できます。

### アウトオブプロセスのコンポーネントテスト

アウトオブプロセスのコンポーネントテストは、完全にデプロイされた成果物を実行し、stubbing の複雑さをテストハーネスへ押し出します。

別プロセスとしてデプロイされたマイクロサービスに対してコンポーネントテストを実行すると、より多くの層と統合点を実行できます。すべての相互作用が実際のネットワーク呼び出しを使うため、デプロイ成果物は変更せずに済み、テスト固有のロジックは不要です。

このアプローチでは、複雑さはテストハーネスに押し出されます。テストハーネスは外部 stubs の起動と停止、ネットワークポートと設定の調整に責任を持ちます。

ネットワーク相互作用と実際のデータストアの利用により、テスト実行時間は長くなる可能性があります。しかし、マイクロサービスが複雑な統合、永続化、起動ロジックを持つ場合には、アウトオブプロセスのアプローチのほうが適しているかもしれません。

マイクロサービスが別プロセスのポートで listen しているため、アウトオブプロセスのコンポーネントテストは、振る舞いの検証に加えて、マイクロサービスが正しいネットワーク設定を持ち、ネットワークリクエストを処理できることも検証します。

同様に、client と persistence のモジュールは、別プロセスの外部依存と統合された状態で実行されます。テストハーネスは起動時にマイクロサービスを設定し、テスト用依存先の正しい URL を指すようにします。

外部サービス stubs には、いくつかの種類があります。API 経由で動的にプログラムされるもの、手作りの fixture データを使うもの、実際の外部サービスへのリクエストとレスポンスを記録して再生する record-replay 機構を使うものがあります。

例となるツールには、動的 stub と fixture ベースの stub の両方をサポートする moco、stubby4j、mountebank、そして record-replay スタイルの stubbing を可能にする vcr があります。

外部サービスに多くの協力相手がある場合、そのサービス固有の custom stub を構築すると、利用者が stub を自分で管理せずに済むため、価値があるかもしれません。

このレベルのテストは、サービスの振る舞いをビジネスの言葉で記述するため、Gherkin のようなビジネスが読める DSL を、Cucumber や specflow といったツールで表現すると有益なことがよくあります。

### テスト戦略の組み合わせ

テスト戦略を組み合わせることで、高いテストカバレッジにつながります。

単体テスト、統合テスト、コンポーネントテストを組み合わせることで、マイクロサービスを構成するモジュールの高いカバレッジを達成でき、マイクロサービスが必要なビジネスロジックを正しく実装していることを確信できます。

それでも、ごく単純なユースケースを除けば、多くのマイクロサービスが連携してより大きなビジネスプロセスを満たさない限り、ビジネス価値は実現されません。このテスト方式の中には、外部依存が期待された契約を満たしていることや、マイクロサービス群が正しく協調してエンドツーエンドのビジネスフローを提供していることを保証するテストがまだありません。

これを補うのが、外部依存の契約テストと、システム全体に対するより粗い粒度のエンドツーエンドテストです。

### 契約テスト

統合契約テストは、外部サービスの境界で行うテストであり、外部サービスが利用側サービスの期待する契約を満たしていることを検証します。

ある利用者が、コンポーネントの振る舞いを使うためにそのインターフェースに結合すると、その間に契約が形成されます。この契約は、入力データ構造と出力データ構造、副作用、性能、並行性特性への期待から成ります。

コンポーネントの各利用者は、自分の要件に基づいて異なる契約を形成します。コンポーネントが時間とともに変化するなら、各利用者の契約が引き続き満たされることが重要です。

統合契約テストは、コンポーネントが契約を満たすことを明示的に検証する仕組みを提供します。

関係するコンポーネントがマイクロサービスである場合、インターフェースは各サービスが公開する public API です。各利用側サービスの保守者は、提供側サービスのうち実際に使っている側面だけを検証する独立したテストスイートを書きます。

これらのテストはコンポーネントテストではありません。サービスの振る舞いを深くテストするのではなく、サービス呼び出しの入力と出力が必要な属性を含んでいること、レスポンスレイテンシとスループットが許容範囲内にあることをテストします。

理想的には、各利用側チームが書いた契約テストスイートはパッケージ化され、提供側サービスのビルドパイプラインで実行できるようにします。そうすれば、提供側サービスの保守者は、自分たちの変更が利用者に与える影響を知ることができます。

### 利用者契約の合計

すべての consumer contract tests の合計が、サービス全体の契約を定義します。

契約テストは外部サービスの利用者に信頼を与えますが、そのサービスの保守者にとってはさらに価値があります。サービスのすべての利用者から契約テストスイートを受け取ることで、利用者に影響を与えないという確信を持って、そのサービスに変更を加えられます。

3つのフィールド、すなわち identifier、name、age を持つリソースを公開しているサービスを考えます。このサービスは3つの異なる利用者に採用されており、それぞれがリソースの異なる部分に結合しています。

Consumer A は identifier と name のフィールドだけに結合しています。そのため、対応する契約テストスイートはリソースのレスポンスがそれらのフィールドを含むことをアサートします。age フィールドについては何もアサートしません。

Consumer B は identifier と age のフィールドに結合しているため、契約テストはそれらが存在することをアサートしますが、name フィールドについては何もアサートしません。

Consumer C は3つすべてのフィールドを必要としており、それらがすべて存在することをアサートする契約テストスイートを持っています。

新しい利用者が API を採用し、first name と last name の両方を必要とする場合、保守者は name フィールドを deprecated にし、名前の構成要素を持つ複合オブジェクトを含む別フィールドを導入することを選ぶかもしれません。

古い name フィールドを削除するために何が必要かを見るには、保守者はレスポンスからそれを削除し、どの契約テストが失敗するかを確認できます。この場合、Consumer A と Consumer C に deprecated であることを知らせる必要があるとわかります。Consumer A と Consumer C が新しいフィールドへ移行したあと、deprecated されたフィールドを削除できます。

これを有効にするためには、producer と consumer の両方が、メッセージのシリアライズとデシリアライズの際に Postel's Law に従い、自分たちにとって重要でないフィールドを無視すべきです。

このアプローチは Parallel Change の一例です。そこでは、どの利用者の契約も壊さずに、一定期間をかけて API を変更できます。

契約テストスイートは、新しいサービスが定義されるときにも価値があります。利用者は、サービスから必要なものを表現するテストスイートを構築することで、API 設計を駆動できます。

これらの consumer driven contracts は、サービス構築を担当するチームとの議論の材料になると同時に、API の準備状況を示す自動テストにもなります。

Pact、Pacto、Janus など、契約テストを書く助けとなるツールがいくつも存在します。

### エンドツーエンドテスト

エンドツーエンドテストは、システム全体を端から端までテストし、システムが外部要件を満たし、目的を達成していることを検証します。

他の種類のテストとは対照的に、エンドツーエンドテストの意図は、使用されているコンポーネントアーキテクチャに関係なく、システム全体がビジネス目標を満たしていることを検証することです。

これを達成するために、システムはブラックボックスとして扱われ、テストは完全にデプロイされたシステムのできるだけ多くを実行します。その操作は、GUI やサービス API のような公開インターフェースを通じて行われます。

エンドツーエンドテストはより business facing であるため、ドメインの言葉でテストケースを表現する、ビジネスが読める DSL を利用することがよくあります。

マイクロサービスアーキテクチャは、同じ振る舞いに対してより多くの動く部品を含むため、エンドツーエンドテストはサービス間の隙間をカバーすることで価値を提供します。これにより、サービス間で受け渡されるメッセージの正しさへの追加の信頼が得られるだけでなく、firewalls、proxies、load-balancers といった追加のネットワークインフラが正しく設定されていることも保証できます。

エンドツーエンドテストは、マイクロサービスアーキテクチャが時間とともに進化することも可能にします。問題ドメインについて学びが増えるにつれて、サービスは分割されたり統合されたりする可能性があります。エンドツーエンドテストは、そのような大規模なアーキテクチャリファクタリングの間も、システムが提供するビジネス機能が損なわれていないという信頼を与えます。

### エンドツーエンドテストの境界

エンドツーエンドテストのテスト境界は、他の種類のテストよりもはるかに大きくなります。

目的は完全に統合されたシステムの振る舞いをテストすることなので、エンドツーエンドテストは可能な限り粗い粒度で相互作用します。

システムが直接的なユーザー操作を必要とする場合、その相互作用は1つ以上のマイクロサービスが公開する GUI を通じて行われるかもしれません。この場合、Selenium WebDriver のようなツールが、GUI を駆動してシステム内の特定のユースケースを引き起こす助けになります。

headless なシステムでは、エンドツーエンドテストは HTTP クライアントを使い、public API を通じてマイクロサービスを直接操作します。

このようにして、システムの正しさは、テスト境界が形作る周縁での状態変化やイベントを観察することで確認されます。

一部のシステムは十分に小さく、単一チームがすべての構成コンポーネントを所有できます。しかし多くの場合、システムは成長し、外部で管理される1つ以上のマイクロサービスに依存するようになります。

通常、これらの外部サービスはエンドツーエンドのテスト境界に含まれます。しかし、まれにそれらを除外することを選ぶ場合もあります。

外部サービスがサードパーティによって管理されている場合、繰り返し可能で副作用のない形でエンドツーエンドテストを書くことができないかもしれません。同様に、一部のサービスは信頼性の問題を抱え、チームの制御外の理由でエンドツーエンドテストを失敗させることがあります。

このような場合、外部サービスを stub することが有益です。エンドツーエンドの信頼の一部は失われますが、テストスイートの安定性は得られます。

### エンドツーエンドテストの難しさ

エンドツーエンドテストを書き、保守することは非常に難しくなり得ます。

エンドツーエンドテストは、ここまでに説明した他の戦略よりもはるかに多くの動く部品を含むため、そのぶん失敗する理由も多くなります。GUI 内であれ、サービス間の非同期バックエンドプロセスによるものであれ、エンドツーエンドテストはシステム内の非同期性も考慮しなければならない場合があります。これらの要因は、flakiness、過剰なテスト実行時間、テストスイートの追加保守コストにつながります。

以下のガイドラインは、エンドツーエンドテストの追加の複雑さを管理する助けになります。

#### 1. エンドツーエンドテストはできるだけ少なく書く

低いレベルのテストによって高い信頼を得られるのであれば、エンドツーエンドテストの役割は、すべてが結びついていて、マイクロサービス間に高レベルの不一致がないことを確認することです。

したがって、このレベルでビジネス要件を包括的にテストするのは無駄です。時間と保守の点でエンドツーエンドテストが高価であることを考えれば、なおさらです。

エンドツーエンドテストスイートを小さく保つうえでうまく機能する戦略の1つは、time budget を適用することです。これは、チームがテストスイートの実行を待てる時間の量です。スイートが成長し、実行時間が time budget を超え始めた場合、割り当てられた時間内に収めるため、最も価値の低いテストを削除します。time budget は時間単位ではなく、分単位であるべきです。

#### 2. ペルソナとユーザージャーニーに集中する

エンドツーエンドスイートのすべてのテストを価値あるものにするには、システム利用者の personas と、その利用者がシステムを通じてたどる journeys を中心にモデル化します。これにより、利用者が最も価値を置くシステム部分への信頼が得られ、それ以外のカバレッジは他の種類のテストに任せられます。

Gauge や Concordion のようなツールは、ビジネスが読める DSL を通じて journeys を表現する助けになります。

#### 3. 端を賢く選ぶ

特定の外部サービスや GUI がテストスイートの flakiness の主な原因である場合、そのコンポーネントを除外するようにテスト境界を再定義すると役立つことがあります。この場合、完全なエンドツーエンドカバレッジは、スイートの信頼性と引き換えにされます。他の形のテストが、その不安定なコンポーネントを別の手段で検証している限り、これは許容できます。

#### 4. 繰り返し可能性のために infrastructure-as-code に頼る

Snowflake environments も非決定性の源になり得ます。特に、エンドツーエンドテスト以外にも使われている場合はそうです。

マイクロサービスアーキテクチャの追加のデプロイ複雑性を管理するうえで大いに役立つ infrastructure-as-code を採用していれば、再現可能な形で環境をその場で構築できます。

エンドツーエンドテストスイートを実行するたびに新しい環境を構築することで、信頼性を改善できるだけでなく、デプロイロジックのテストとしても機能します。

#### 5. テストをデータ非依存にする

エンドツーエンドテストにおける難しさの一般的な原因は、データ管理です。既存データに依存すると、環境内でデータが変更され、蓄積されるにつれて失敗する可能性が生まれます。私はこれを false-failures と呼びます。つまり、その失敗はソフトウェアの欠陥を示すものではありません。

エンドツーエンドテストが依存するデータを自動管理すると、false-failures の可能性を減らせます。サービスが public API または internal API を通じて自分の所有するエンティティの構築をサポートしていれば、エンドツーエンドテストは実行前に自分の世界を定義できます。外部からの構築を許さないサービスについては、canned data をデータベースレベルで取り込めます。

このスタイルのテストを書くことに伴う難しさのため、一部のチームはエンドツーエンドテストを完全に避け、徹底した本番監視と、本番環境に対する直接のテストを選びます。

Synthetic transactions、すなわち本番システムに対して実際のトランザクションを実行する偽のユーザーは、典型的な監視技術を補完し、本番環境の健全性についてより多くの洞察を提供できます。さらに、主要なビジネスメトリクスが許容範囲を外れたときにアラートを出すことで、本番の問題を素早く特定する助けになります。

### マイクロサービスが増やす選択肢

マイクロサービスアーキテクチャは、どこで、どのようにテストするかについて、より多くの選択肢を提供します。

システムを小さく明確に定義されたサービスへ分割することで、以前は隠れていた追加の境界が露出します。これらの境界は、採用できるテストの種類とレベルという点で、機会と柔軟性を提供します。

場合によっては、マイクロサービスが複雑な要件を持つ中核的なビジネスプロセスをカプセル化していることがあります。このプロセスの重要性により、ここで説明したすべてのテスト戦略の範囲を含むような、非常に包括的なテストが必要になるかもしれません。別の場合には、マイクロサービスが実験的であったり、ビジネス上それほど重要でなかったり、寿命が短かったりするかもしれません。必要なテストレベルは低くなり、いくつかの戦略だけが意味を持つこともあります。

この意思決定プロセスはモノリシックアーキテクチャでも可能ですが、明確でよく定義された境界が加わることで、システムのコンポーネントを見て、それらを分離して扱うことが容易になります。

マイクロサービスアーキテクチャではテストの選択肢が増えますが、保守が難しく実行に時間のかかる肥大化した大きなテストスイートによってテストの価値を下げないために、**テストピラミッドの原則**に従うことはなお重要です。

### テストピラミッド

テストピラミッドは、異なる種類のテストの間でバランスを保つ助けになります。

一般に、テストの粒度が粗くなるほど、壊れやすく、実行に時間がかかり、書くことも保守することも難しくなります。この追加のコストは、そのようなテストが、より細かく焦点の絞られたテストよりも自然に多くの動く部品を含むことに由来します。

test pyramid の概念は、各粒度で書くべきテストの相対的な数を考えるための単純な方法です。ピラミッドの段を上がるにつれて、テストのスコープは広がり、書くべきテストの数は減っていきます。

ピラミッドの頂点には exploratory testing があります。これは、スクリプト化されたテストの一部として考慮されていなかった方法で、手動によりシステムを探索するものです。exploratory testing によって、チームはシステムについて学び、自動テストを教育し、改善できます。

テストピラミッドのガイドラインに従うことで、保守と実行に費用のかかる、肥大化した大きなテストスイートによってテストの価値を下げることを避けられます。

### まとめ

単体テスト: アプリケーション内のテスト可能な最小のソフトウェア片を実行し、それらが期待どおりに振る舞うかを判断します。

統合テスト: コンポーネント間の通信経路と相互作用を検証し、インターフェースの欠陥を検出します。

コンポーネントテスト: 実行されるソフトウェアの範囲をシステムの一部に限定し、内部コードインターフェースを通じてシステムを操作し、test doubles を使ってテスト対象コードを他のコンポーネントから分離します。

契約テスト: 外部サービスの境界での相互作用を検証し、外部サービスが利用側サービスの期待する契約を満たしていることをアサートします。

エンドツーエンドテスト: システム全体を端から端までテストし、システムが外部要件を満たし、目的を達成していることを検証します。
