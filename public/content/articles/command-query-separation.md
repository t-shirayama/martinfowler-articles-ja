# Command Query Separation

## 要約

Command Query Separationは、オブジェクトのメソッドを「結果を返し状態を変えないquery」と「状態を変えるが値を返さないcommand」に分ける考え方です。状態を変えないメソッドを明確にできると、呼び出し順序や利用箇所に対してより大きな自信を持てます。

Fowlerはこの原則を有用だとしつつ、stackの`pop`のような例外も認めています。言語が状態変更を支援して示せると望ましいものの、観測可能な状態だけが問題になるため、自動検出は難しいと述べています。

## 読むときの観点

- queryとcommandを戻り値の有無だけでなく、副作用の有無として読む。
- 状態を変えないqueryは、並べ替えや再利用がしやすい。
- `next`や`pop`のような便利な慣用句は、原則との緊張を生む。
- C++の`const`のように、言語機能で意図を示す方法に注目する。

## 原文の翻訳

Command Query Separationという用語は、Bertrand Meyerが著書『Object Oriented Software Construction』で作ったものです。この本は、オブジェクト指向の初期において最も影響力のあるOO本の1つでした。影響を与えたのは初版の方です。第2版もよい本ですが、持ち上げる前にジムで数か月鍛える必要があるでしょう。

基本的な考えは、オブジェクトのメソッドを2つの明確に分かれたカテゴリへ分けるべきだ、というものです。

- Query: 結果を返し、システムの観測可能な状態を変えない。つまり副作用がない。
- Command: システムの状態を変えるが、値を返さない。

`command`という語は他の文脈でも広く使われるため、私はこれらを`modifier`と呼ぶ方を好みます。`mutator`という用語も見かけます。

この原則で本当に価値がある考えは、状態を変えるメソッドと変えないメソッドをはっきり分けられると、とても便利だという点です。queryは多くの状況でより安心して使えます。どこに導入してもよく、順序を変えてもよいからです。一方、modifierにはもっと注意が必要です。

この原則では、戻り値の型が違いを示す手がかりになる、という考え方があります。たいていの場合うまく働くので、よい規約です。Javaでコレクションを反復する慣用句を考えてみてください。`next`メソッドは、コレクション内の次の項目を返すと同時に、iteratorを前へ進めます。個人的には、`advance`メソッドと`current`メソッドを分けるスタイルの方がずっと好みです。

MeyerはCommand Query Separationを絶対的に使うことを好みますが、例外もあります。stackからpopする操作は、状態を変えるqueryのよい例です。Meyerはこのメソッドを避けられると正しく言っていますが、これは便利な慣用句です。だから私は、この原則に従えるときは従いますが、**popを得るためなら原則を破る用意があります**。

言語そのものがこの考えを支援してくれるとよいでしょう。状態を変えるメソッドを検出する言語、あるいは少なくともプログラマがそれを印付けできる言語は想像できます。言語がそれを自動検出できない理由の1つは、状態を変えないというルールが、実際にはシステムのObservable Stateにだけ適用されるからです。プログラマによる印付けの方が現実的に思えますが、まれです。私が実際に見た唯一の例は、C++の`const`修飾子です。

私は長年C++を使っていないので、実務でどれほど有用かを評価するのは難しいところです。私の感覚では、優れたC++プログラマは`const`を多用し、それを好んでいます。
