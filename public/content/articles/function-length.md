# Function Length

## 要約

関数の長さを考えるとき、本当に重要なのは行数そのものではなく、コードの意図と実装の距離です。コード片を見て「何をしているのか」を理解するのに労力がいるなら、そのコードを関数に切り出し、その「何」を名前にするほうが読みやすくなります。

短い関数は、よい名前が付いていると、コードを自己文書化しやすくします。大きめの関数は物語のように読め、読者は必要に応じて詳細な関数へ潜っていけます。

## 読むときの観点

- 関数の長さを、画面に収まるかどうかではなく、意図と実装の距離で考える。
- Extract Functionは、再利用のためだけでなく、意図を名前で表すためにも使う。
- 一行だけの関数でも、意図を明らかにする名前があれば価値がある。
- 短い関数は、名前付けの質に強く依存する。

## 原文の翻訳

私のキャリアの中で、関数はどれくらいの長さであるべきかについて、たくさんの議論を聞いてきました。これは、より重要な問い、つまり「いつコードを独立した関数に包み込むべきか」の代替指標です。こうした指針の中には、長さに基づくものもありました。たとえば、関数は画面に収まるより大きくすべきではない、というものです。再利用に基づくものもありました。複数回使われるコードは独立した関数に置くべきだが、一度しか使われないコードはインラインのままにすべきだ、という考え方です。しかし、私にとって最も筋が通っている議論は、**意図と実装の分離**です。あるコード片を見て、それが「何」をしているのかを理解するために労力を使わなければならないなら、そのコードを関数に切り出し、その「何」にちなんだ名前を関数に付けるべきです。そうすれば、次にそのコードを読むとき、関数の目的がすぐに目に飛び込んできます。そしてたいていの場合、その関数が目的をどう達成しているのか、つまり関数本体については気にしなくてよくなります。

この原則を受け入れてから、私はとても小さな関数を書く習慣を身につけました。典型的には、ほんの数行の長さです。六行を超える関数は、私には少し匂い始めますし、一行だけの関数を持つことも珍しくありません。大きさは重要ではない、ということを強く実感したのは、Kent BeckがオリジナルのSmalltalkシステムから見せてくれた例でした。当時のSmalltalkは白黒のシステム上で動いていました。テキストやグラフィックを強調表示したいときは、表示を反転させます。Smalltalkのグラフィックスクラスには、このための`highlight`というメソッドがあり、その実装は単に`reverse`メソッドを呼ぶだけでした。メソッド名のほうが実装より長かったのです。しかし、それは問題ではありませんでした。コードの意図と実装の間には、大きな距離があったからです。

短い関数を心配する人もいます。関数呼び出しの性能コストを気にするからです。私が若かったころは、それが要因になることもありました。しかし今では、そうしたことは非常にまれです。最適化コンパイラは、キャッシュしやすい短い関数のほうがうまく働くこともよくあります。いつものように、[性能最適化についての一般的な指針](https://martinfowler.com/ieeeSoftware/yetOptimization.pdf)こそが重要です。あとから関数をインライン化する必要が出てくることもありますが、多くの場合、小さな関数は別の高速化の方法を示してくれます。リストに`isEmpty`メソッドを持たせることに反対する人たちのことを覚えています。一般的なイディオムは`aList.length == 0`を使うことだからです。しかしここでは、意図を明らかにする名前を関数に付けることが、性能の改善にもつながるかもしれません。コレクションが空かどうかを判断するほうが、その長さを求めるより速いことがあるからです。

このような小さな関数がうまく働くのは、**名前がよい場合だけ**です。だから、名前付けには十分に注意を払う必要があります。これは練習を要しますが、うまくできるようになると、このアプローチはコードを驚くほど自己文書化されたものにできます。より大きな粒度の関数は物語のように読めるようになり、読者は必要に応じて、より詳しく知りたい関数へ潜っていくことを選べます。
