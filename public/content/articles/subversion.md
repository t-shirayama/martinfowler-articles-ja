# Subversion

## 要約

Subversion は CVS の後継として広まったオープンソースのバージョン管理システムです。CVS の大きな弱点だった、コミットの原子性やファイル・ディレクトリ名の変更への対応を改善し、当時の実務で信頼できる選択肢になりました。

この記事では、Subversion を個人の作業環境同期やチーム開発に使う利点を述べつつ、バージョン管理がソフトウェア開発以外の領域にも広がる可能性を示しています。

## 読むときの観点

- Subversion が CVS のどの弱点を解消しようとしたのか。
- バージョン管理を「開発者だけの道具」と見なさない視点。
- 差分、マージ、共同作業をアプリケーションがもっと自然に扱えるようになる可能性。

## 原文の翻訳

Subversion はオープンソースのバージョン管理システムであり、本質的には CVS の後継にあたります。CVS の最大の問題を修正し、**原子的なコミット**や、ファイルとディレクトリのリネームへの対応といった機能を導入しています。私は数年にわたって使ってきましたが、とても堅牢だと感じています。

同僚の Mike Mason は、Subversion について Pragmatic Programmer の本を書いています。詳しい本はオンラインでも読めますが、Mike の本は Subversion の概要と使い方を学ぶためのすばらしい入門書です。たくさん買ってくれれば、彼の給料を上げずに済みます。

私が Subversion を使う大きな用途のひとつは、複数のデスクトップ環境を扱うことです。作業ファイルをすべて Subversion に置き、update と commit を使って同期しています。

同僚たちとの非公式な会話をもとにすると、Subversion、略して SVN は、Perforce を除く商用ツールよりよいシステムだと思います。もし別のものを使っているなら、乗り換えを検討してもよいでしょう。

私は日常的にバージョン管理を使っているので、これはコンピュータ利用のもっと多くの領域に広がりうるものだと考えています。ソフトウェア開発者以外でバージョン管理を使う人は、ほとんどいません。しかし開発者が知っているように、バージョン管理は共同作業のためのすばらしい仕組みであり、複数の人がひとつのソフトウェアシステムに一緒に取り組めるようにしてくれます。

では、バージョン管理がもっと広く使われると、どんな利点があるでしょうか。多くの人が使っているアプリケーションには、差分やマージの機能がほとんどありません。もしアプリケーションがバージョン管理をもっと意識するようになれば、そこからもっと面白いことが起こるはずです。堅牢で使いやすいオープンソースの道具が、バージョン管理の利用をより広い層に広げてくれることを期待しています。結局のところ、私たちはもうディスク容量不足に悩む時代ではないのです。
