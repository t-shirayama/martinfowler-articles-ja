# On the Diverse And Fantastical Shapes of Testing

## 要約

テストピラミッド、テストハニカム、テストトロフィーといった図は、テストの種類ごとにどれくらい力をかけるべきかを示そうとするものです。しかし議論の多くは、「unit test」や「integration test」という言葉の意味が人によって違うために混乱します。

Martin Fowler は、単体テストという語が歴史的にも実践上も幅を持って使われてきたことを整理し、孤立したテストと協調的なテストを区別します。そのうえで、どの形が正しいかを争うより、境界が明確で、速く、信頼でき、役に立つ理由で失敗するテストを書くことのほうが重要だと述べます。

## 読むときの観点

- 「unit test」と「integration test」が、相手の文脈で何を指しているのかを確認する。
- テストピラミッド批判が、実際には過剰なモック利用への批判になっていないかを見る。
- テストの分類よりも、テストが有用なフィードバックを返せるかを重視する。

## 原文の翻訳

最近、Twitter などで、チームがテスト作業をどのように分けるべきかという話題が再び盛り上がっている。特に Tim Bray は、自動テストを真剣に扱うべきだと説得力のある主張をしている。私の文章を読んできた人なら分かるように、私はこの点に大いに賛成している。

彼の記事では、テストハニカムやテストトロフィーのような図が取り上げられている。これらは、古くから知られているテストピラミッドへの反応として出てきたものだ。

これらの図が示そうとしているのは、さまざまな種類のテストにどれだけの労力を割くべきか、特に unit test と、より広い範囲のテストとのバランスである。ピラミッドは、大部分のテストを unit test として行うべきだと主張する。一方、ハニカムやトロフィーは、unit test は比較的少なくし、integration test に多くの比重を置くべきだと言う。

この議論について私が抱く二番目に大きな問題は、**unit test と integration test の違いが何を意味するのかがはっきりしていない**ため、議論が不透明になることだ。

「unit test」と「integration test」という言葉は、ソフトウェア用語の中でもとりわけ曖昧である。私が最初に理解したとき、それらは主に組織上の区分だった。大規模なウォーターフォール型プロジェクトでは、ある開発者または小さなチームが、数か月かけて一まとまりのコードを作る。そのまとまりを、隣接する部分からある程度切り離して扱える概念的な単位と考える。

実装が終わると、その単位を unit testing team に渡し、彼らが単独でテストする。その後、その単位を周辺のコードと統合し、より大きな部分、場合によってはシステム全体に対して integration test を行う。ここでの違いは、unit test が自分たちのコードを隔離してテストするのに対し、integration test は別々に開発されたコード同士が一緒に正しく動くかを見ることだった。

今日、多くの人が unit test に出会ったのは、Kent Beck が Extreme Programming の一部として広めた xUnit 系のテスティングツールを通じてである。Kent は「unit test」を、開発者が日々の作業の一部として書くテストという意味で使った。

この最初の定式化では、「unit test」は専任のテストチームではなくプログラマが書くテストを指していた。C3 で私たちが unit test を書いていたときは、単一クラスの振る舞いに注目することが多かった。ただし、そのオブジェクトが必要とする依存オブジェクトも含めたテストフィクスチャを用意し、そのメソッドを実行できるようにした。

依存オブジェクトも実際に動くが、そのテストでは他のコードは正しく動くものと仮定する。通常、それらのコードにもそれぞれのテストがあった。

この「unit test」という使い方については、かなり議論があった。あるテスト専門家は、Kent の用法を強く批判した。では unit test をどう定義するのかと尋ねると、彼は研修の最初の午前中だけで 24 種類の定義を扱う、といった趣旨の答えをした。

XP に影響を受けた unit testing の初期から、「unit test」という語を好まず、「microtest」や「programmer test」といった名前を提案する人もいた。

多くの人にとって、問題の中心は依存オブジェクトだった。たとえば注文オブジェクトをテストするとき、その注文が顧客オブジェクトと協調しているなら、注文のテストは顧客オブジェクトのバグによって失敗する可能性がある。これを避けるため、協調するオブジェクトを mock、stub、その他の test double に置き換えるスタイルが生まれた。私はこの違いを、**協調的な unit test と孤立した unit test** と呼ぶと便利だと考えている。

この区別には、XP の unit testing 実践における二つの流派、私が classic と mockist と呼ぶものの発展も絡んでいる。Classic な XP unit testing は、私たちが当初使っていた協調的なアプローチに従う。一方、mockist スタイルは、孤立したテストを好む。

ピラミッドとハニカムの議論に戻ると、ハニカムなどを支持する人たちの話を読むと、彼らはしばしばモックの使いすぎと、それが引き起こす問題を批判しているように聞こえる。そこから推測すると、彼らが「unit test」と呼ぶものは、私なら孤立した unit test と呼ぶものに限られている。同じように、彼らのいう integration test は、私なら協調的な unit test と呼ぶものにかなり近い。

そう考えると、ピラミッド対ハニカムの議論はかみ合わなくなる。私が聞いてきたテストピラミッドの説明では、unit test には協調的なものも孤立したものも含まれるからだ。

この意味の混乱は、integration test の定義によってさらに濁る。integration test という語の曖昧さに比べれば、unit test はまだ厳密に見えるほどである。ここでの教訓は、誰かがテスト分類について語り始めたら、**その人がその言葉で何を意味しているのかを掘り下げるべきだ**ということだ。おそらく、その人はあなたが直前に読んだ別の人と同じ意味では使っていない。

注意深く読んでいるなら、私が unit test と integration test の不明瞭さを、この議論についての二番目の問題だと言ったことに気づくだろう。最大の問題は、どの種類のテストを何パーセント書くべきかという議論が、しばしば本質から注意をそらすことだ。

本当に注目すべきなのは、明確な境界を示し、速く、信頼でき、有用な理由でだけ失敗する、表現力のあるテストを書くことである。図の形をめぐる議論よりも、そこに集中するほうが価値がある。
