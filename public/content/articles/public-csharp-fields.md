# Public Csharp Fields

## 要約

C# の property は、`getX` や `setX` のような慣習を言語機能として自然に支える便利な仕組みです。
一見すると、field と property は同じ記法で扱えるため、単純な公開データは public field にしておき、必要になったら property に置き換えればよさそうに見えます。

しかし reflection 上では field と property が違うものとして表現されるため、reflection を使うツールが壊れる可能性があります。
そのため、公開データ値には結局 property を書く必要がある、という短い嘆きの文章です。

## 読むときの観点

- C# property が Uniform Access に近い記法を提供する点を見る。
- ソースコード上の見た目と reflection 上の表現が違うことに注目する。
- カプセル化だけでなく、周辺ツールとの互換性が API 設計を左右する例として読む。

## 原文の翻訳

私が初めて C# に触れたとき、properties という考え方は最初から気に入りました。C++ や Java の `getX`、`setX` という慣習は、私にはいつも少しばかげて見えました。`obj.X = other.X` と書くほうがずっと自然です。get メソッドと set メソッドを持つ property を提供することで、よくある慣習が、言語に自然に支えられた機能になります。

もう1つ気に入ったのは、言語の観点から見ると、properties と fields が同じように見えることです。したがって、単に読み書きするだけの field `x` があるなら、それを field として宣言するだけで済みます。私はそれがカプセル化を破ることを心配しません。後でもっと凝ったことをしたくなれば、property に置き換えればよいからです。これにより、ばかげたアクセサ関数を大量に書かずに済みます。

残念ながら、話はそれほど単純ではありません。fields と properties は、プログラムテキスト上では同じようにアクセスできますが、reflection では別のものとして表現されます。その結果、reflection を使うさまざまなツールは、fields と properties を切り替えると壊れてしまいます。

そのため今では、アクセス可能なデータ値のために、ばかげたアクセサ関数を書かなければなりません。やれやれ。あるいは Ruby を使うことです。
