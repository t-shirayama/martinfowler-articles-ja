# Interview with Kent Beck and Martin Fowler

## 要約

この記事は、Kent Beck 氏と Martin Fowler 氏が Extreme Programming、特に XP における計画づくりについて語った 2001年のインタビューです。`Planning Extreme Programming` の刊行時期に、XP がどのように生まれ、どのような開発規模に向くのか、従来型の計画やプロセスとどう違うのかが問われています。

XP の計画は、将来を固定的に予測するためのものではなく、状況が変わったときに更新できる現在の見積もりとして説明されています。Fowler 氏は、計画を複雑な管理ツールの成果物にせず、実際に作業する人たちが詳細を考えることを重視しています。Beck 氏も、計画書そのものよりも、計画する行為が価値を生むという点を強調しています。

インタビュー全体では、XP は小さなチームで書かれた方法でありながら、その原則の多くはより大きなチームにも応用できるとされています。また、テスト、リファクタリング、コミュニケーション、継続的な統合といった技術的実践は、ビジネス側がスコープや優先順位を変えてもチームが壊れないようにする準備として位置づけられています。

## 読むときの観点

- XP を、単なる軽量プロセスではなく、計画・技術的実践・ビジネス判断を結びつける方法として読む。
- 計画を「予測」ではなく「現在の見通し」として扱う考え方に注目する。
- 小規模チーム向けに書かれた XP の原則が、より大きな文脈でどこまで使えるかを見る。
- PSP や RUP との比較では、XP が中央集権的な制御よりもチーム全体のふるまいを重視している点を見る。
- 2001年当時の XP ブームを、期待と誇張の両方を含む歴史的文脈として読む。

## 原文の翻訳

インタビューは、XP が「極限」の名を持つスポーツのように、興奮と危険の両方を伴うものだという導入から始まります。Kent Beck 氏と Martin Fowler 氏は、XP の世界についての質問に答えながら、計画、チームの規模、既存の方法論との違い、今後の見通しを語っています。

XP の起源については、Fowler 氏が冗談を交えて答えています。その後、Beck 氏は、最初にチームを率いたときにはテストやレビューなど自分が筋がよいと思うことを少しずつ求め、次の機会には重要だと思う実践を強く押し上げ、それ以外を削った経験を語っています。XP は、そのような実践を高い強度で組み合わせたものとして説明されています。

XP がどのような規模の開発に向くかについて、Fowler 氏は、書かれている形の XP は小さなチームにもっとも向いており、十数人を超えるチームにそのまま勧めるわけではないと述べています。一方で、倍ほどの規模のプロジェクトでも XP が有効に使われた経験があり、計画に関する考え方を含む多くの原則は大きなチームにも役立つとしています。Beck 氏は、何百人規模に効くとは実例を見るまで言わないとしつつ、時間や専門性をまたぐ広がりには強い関心を示しています。

XP の計画が従来型の計画と違う点について、Fowler 氏は三つの点を挙げています。第一に、計画は複雑なプロジェクト管理ソフトを使いこなす専門家だけのものではなく、単純であるべきだということです。第二に、詳細な計画は実際に作業をする人たちが立てるということです。第三に、計画は未来の予言ではなく、現時点での見積もりであり、状況が変われば変えなければならないということです。**計画の価値は、固定された計画書ではなく、計画し続ける活動にあります。**

XP が世界中で広がった理由について、Beck 氏は、XP がどの文化にとっても等しく難しいからだと見ています。国や組織文化によって、難しさを感じる場所と安心できる場所は異なります。Fowler 氏は、長い間、現場には混沌とした「作って直す」やり方と、作業そのものより重く見える複雑な方法論の二択があり、XP はその間に必要とされていた実用的な均衡を与えたと説明しています。

二人が XP の計画について本を書いた理由は、XP をどう実行するかについて、さらに材料が必要だと感じたからです。計画はその中心的な話題でした。Fowler 氏は、以前は単独で本を書くことが多かったものの、Beck 氏との共同執筆は非常にうまく進み、互いの文章を大胆に直せるほど文体が近かったと振り返っています。Beck 氏も、二人の書き方が軽く親しみやすく、それでいて正確であることを目指していたと述べています。

従来の方法論との比較では、Beck 氏は PSP のような数値化と予測可能性を強く志向する考え方と XP を対比しています。XP では、品質や保守性のような性質を、中央で細かく制御するのではなく、単純なルールに従ってチーム全体が動くことから生じるものとして捉えます。Fowler 氏は RUP について、あまりに多くのものを受け入れられるため、XP さえ RUP の一形態として表現できてしまうほど曖昧だと見ています。

XP のほかの実践と計画の関係について、Fowler 氏は、計画は複雑な仕事を進めるために欠かせない活動だと述べています。Beck 氏は、XP の技術的側面は、ビジネスの視点を持つ人にスコープと優先順位の制御を渡すための準備だと説明しています。テスト、リファクタリング、コミュニケーション、統合に力を入れることで、ビジネス側が週明けに焦点を大きく変えても、チームが崩れずに対応できるようになります。

本を書く背景として、Fowler 氏は、成功したプロジェクトと失敗したプロジェクトの両方から多くを持ち込んだと述べています。二人は、最初に XP プロジェクトとして知られる C3 プロジェクトで一緒に働いており、そこから計画の中核的な考えが生まれました。さらに、その後の XP プロジェクトや XP コミュニティからの経験も本に反映されています。

読者に持ち帰ってほしいこととして、Fowler 氏は、毎日少しずつ計画すること、計画は変わるものだと期待すること、それでも計画する活動自体を大切にすることを挙げています。Beck 氏は、開発者には正直に見積もり、実績を追跡し、ビジネス上の問題にぶつかったら助けを求めることを促しています。顧客には、見積もり結果が気に入らないときに見積もりを変えるのではなく、何を先に作るかを創造的に考えるよう促しています。

将来について、Fowler 氏は XP がさらに多くの人やプロジェクトに広がり、他のプロセスにも影響を与え続けると見ています。Beck 氏は、XP は急速に成長し、期待だけでなく誇張や商売気もまとい、成功するプロジェクトも失敗するプロジェクトも出るだろうと述べています。このインタビューは、XP を万能薬としてではなく、計画と技術的規律を通じて現場の判断力を高める実践として読むと理解しやすくなります。
