# Self Testing Code

## 要約

Self-Testing Codeは、機能するソフトウェアと合わせて包括的な自動テストを書く実践です。
うまく行われていれば、単一のコマンドでテストを実行でき、コードに潜む重要な欠陥をテストが照らし出すと信頼できます。

この考えはContinuous IntegrationとContinuous Deliveryの土台です。
本当の価値は、本番バグを減らすことだけではなく、システムを安全に変更できるという自信をチームへ与えることにあります。

## 読むときの観点

- Self-Testing CodeはTDDそのものではなく、自動テストを持つ状態を指す。
- テストは「バグ検出器」として、変更直後に問題を知らせる。
- バグ修正では、先にバグを露出するテストを書く姿勢が重要になる。
- テスト文化、refactoring、Technical Debtの関係に注目する。

## 原文の翻訳

Self-Testing Codeは、著者が『Refactoring』で使った名前であり、機能するソフトウェアとあわせて包括的な自動テストを書く実践を指します。うまく行われていれば、単一のコマンドでテストを実行でき、そのテストがコードに潜むバグを明らかにしてくれると信頼できます。

著者がこの考えに初めて出会ったのは、OOPSLAのカンファレンスで「Beddara」Dave Thomasの話を聞いたときでした。彼は、すべてのオブジェクトは自分自身をテストできるべきだと語っていました。著者は、コマンドを入力するとソフトウェアシステム全体が自己テストを行う姿を思い浮かべました。昔、コンピュータを起動するとハードウェアのメモリテストが走ったのと似た感覚です。

その後、著者は自分のプロジェクトでこの方法を試し、その利点に満足しました。数年後、Kent Beckと仕事をしたとき、Kentも同じことを、著者よりずっと洗練された形で行っていると知りました。これはKent BeckとErich GammaがJUnitを作る少し前のことで、JUnitはSelf-Testing Codeと、その姉妹であるTest Driven Developmentの考え方と実践を支える基盤になりました。

Self-Testing Codeがあるとは、コードベースに対して一連の自動テストを実行でき、テストが通ればコードに重大な欠陥はないと信頼できる状態を意味します。別の見方をすると、ソフトウェアシステムを作るのと同時に、システム内の欠陥を見つけるバグ検出器を作っていることになります。チームの誰かが誤ってバグを入れると、その検出器が反応します。

テストスイートを頻繁に、少なくとも1日に数回実行すれば、そのようなバグを導入後すぐに見つけられます。すると最近の変更だけを見ればよくなり、原因を見つけるのがずっと簡単になります。動くコードと、それを動き続けさせるテストがなければ、プログラミングのひと区切りは完了していません。著者たちの態度は、重要なコードでテストがないものは壊れているとみなす、というものです。

Self-Testing CodeはContinuous Integrationの重要な一部です。著者は、Self-Testing Codeがなければ本当の意味でContinuous Integrationをしているとは言えないと述べます。Continuous Integrationの柱であるため、Continuous Deliveryにも必要な要素です。

Self-Testing Codeの明らかな利点は、本番ソフトウェアへ入り込むバグの数を大幅に減らせることです。その中心にあるのは、開発者がコードとテストを自然に一緒に考えるテスト文化を作ることです。

しかし最大の利点は、本番バグを避けることだけではありません。最大の利点は、システムを変更する自信です。古いコードベースは恐ろしい場所になりがちです。開発者は、動いているコードを変更することを恐れます。バグ修正でさえ危険です。直すバグより多くのバグを作るかもしれないからです。そのような状況では、新機能の追加がひどく遅くなるだけでなく、refactoringも怖くなります。結果としてTechnical Debtが増え、変更するほど次の変更が怖くなる悪循環に入ります。

Self-Testing Codeがあると、状況は変わります。人々は、コードをきれいにするための小さな修正を安全に行えると信じられます。ミスをした場合、あるいはミスをするときには、バグ検出器が反応し、すぐに回復して作業を続けられるからです。この安全網があれば、コードをよい状態に保つ時間を使えます。その結果、新機能追加が少しずつ速くなる好循環に入れます。

このような利点はTest Driven Developmentとの関係で語られることが多いですが、TDDとSelf-Testing Codeは分けて考えると役に立ちます。TDDはSelf-Testing Codeを生み出す利点を持つ特定の実践です。著者はTDDを支持していますが、コードを書いた後にテストを書くことでもSelf-Testing Codeは作れます。ただし、テストがあり、それが通るまでは作業が完了したとは言えません。Self-Testing Codeで重要なのは、テストをどう手に入れたかではなく、**テストが存在すること**です。

Self-Testing Codeを実践するチームの重要な振る舞いのひとつは、本番バグへの反応です。通常、このようなチームはまずそのバグを露出するテストを書き、その後で修正に取りかかります。多くの場合、そのテストを書く作業は、範囲を少しずつ狭める一連のテストになり、最終的にバグを引き起こすUnit Testへ到達します。これはデバッグ技法として有用であり、修正後に同じバグが戻らないようにするためにも不可欠です。チームはそのバグをきっかけに、似たような不足テストがないか探すことも多いでしょう。バグはコードの失敗であるだけでなく、テストの網の失敗でもある、という態度が必要です。

近年はSelf-Testingにもうひとつの次元が加わり、本番監視がより重視されるようになっています。Continuous Deliveryにより、ソフトウェアの新しいバージョンを素早く本番へ展開できます。この状況では、チームは本番でバグを発見し、修正版をデプロイするか、最後に正常だったバージョンへ戻すことで素早く対処することに力を入れます。
