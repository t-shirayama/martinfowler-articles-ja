# Refactoring: This class is too large

## 要約

この記事は、実在する乱れたコードベースから、大きくなりすぎた `ReconciliationIntro` クラスを小さなクラスへ分けていくリファクタリングの記録です。理想的なサンプルではなく、実際の制約や不完全さを含んだ例として、少しずつ安全に変更する進め方を示しています。

中心にある考えは、クラスの行数そのものよりも、責務が混ざり合って理解しづらくなっていることです。メソッドのまとまりを見つけ、依存関係を図にし、テストを支えにしながら、`FileLoader`、`BudgetingMonthService`、`Communicator`、`PathSetter`、`DebugModeSwitcher` へ責務を移していきます。

## 読むときの観点

- 大きいクラスを、まず責務のまとまりとして観察する。
- 移動やリネームと、振る舞いの変更を別コミットに分ける。
- テストを壊さないように、ごく小さな変更を積み重ねる。
- リファクタリング中に見つけた別の問題へ飛びつかず、今の目的に集中する。
- 抽出したクラスを、元のクラスに過度に依存させない。

## 原文の翻訳

実在する、欠点のあるコードベースからのリファクタリング例。

これはリファクタリングの話です。TDD の red-green-refactor サイクルの3番目にあるもので、いつもやっていることのはずです。少なくとも、やっていないときを除けば。

TDD は Test-Driven Development の略で、すべてのコード単位がテストされ、そのテストがシステムの振る舞いを記述するようにする技法です。そのために、コードを書く前にテストを書き、そのテストを通るようにコードを書きます。TDD についてはもっと多くを語れますが、この記事の主題ではありません。

私には、リファクタリングを怠ってきたせいで手に負えなくなったコードベースがあります。そこで、少しずつ整え直してきました。この記事では、**大きすぎるクラスを小さくする**ところを扱います。

### 言語と進め方

私は普段、クライアント、プロジェクト、手元の問題に合う言語でコードを書きます。ただ、キャリアの大半は .NET の世界で過ごしてきたので、C# は慣れた場所であり、最初に選びがちな言語です。この記事の例は C# ですが、コード断片は C# を知らなくても追えるくらい単純にしてあります。

私は TDD が好きで、ここで示すコードも大部分はそのやり方で書かれています。ただし、20年エンジニアをしてきた今でも、悪い習慣はかなり残っていますし、いつも正しくできるわけではありません。

### 問題の概要

話は退屈な家事から始まります。私は `Reconciliate` という個人用の会計ソフトを書きました。コマンドラインで動き、次のことをします。

1. 自分で管理しているカンマ区切りデータを読み込む。
2. 最近の銀行取引とクレジットカード取引の記録を読み込む。
3. 中央のスプレッドシートに基づく月次・年次の予測取引を読み込む。
4. 未照合の既存データを読み込む。
5. 銀行やカード会社から来た第三者データを読み込む。
6. 第三者データを自分のデータと照合する。
7. すべてを中央のスプレッドシートへ書き戻す。

直したいバグも、追加したい機能もありました。しかし典型的な作業時間は、いちばん下の子を寝かしつけたあと、夜遅くにノートPCへ向かい、自分の寝る時間まで少ししか残っていない時間です。しかも前回そのコードを見たのは数週間前ということもよくありました。そんな状況では、「このコードが散らかっているのは分かっているけれど、今それを理解して直す時間はない」と考えるのはひどく簡単です。

もちろん、これは理想的ではありません。

特に頭痛の種になっていたのが `ReconciliationIntro` クラスでした。膨れ上がり、入り組み、頭の中に収まりません。Dan Terhorst-North のいう “Fits In My Head” から外れていたのです。これが悪いフィードバックループを生んでいました。「このコードは考えるのが難しい。だからリファクタリングには、いま持っている以上の時間と気力が必要になる。だからさらに放置する。すると小さな変更でさえ、現状を理解して変更箇所を決めるだけで時間がかかる」。

たとえば、もう1枚クレジットカードを扱えるようにしたいとします。多くの場所では、多態性と Strategy パターンを使って、それぞれのカード固有の振る舞いをきれいに閉じ込めています。しかし `ReconciliationIntro` クラスだけは、設計が悪く、文脈の分離があいまいなため、このままカードを追加すると、すでに膨れたクラスをさらに悪化させることになります。銀行入金、銀行出金、カード1、カード2という4種類のデータに対して、重複した4つのコード経路があり、カード3を足すと同じアンチパターンを繰り返すだけです。

全体の狙いは、このコードを Strategy パターンを使ったより汎用的なコードへ置き換えることです。ただし、その前に4つの問題があります。

1. 1つの大きな `ReconciliationIntro` クラスが、責務を抱えすぎている。
2. private にネストしたコードが多く、公開インターフェイスがないため単体テストしづらい。
3. `ReconciliationIntro` の中に、大きすぎて多くをやりすぎているメソッドが1つある。
4. `ReconciliationIntro` の中に、詳細だけが異なる同じパターンのメソッドがいくつもある。

私はこれらをこの順で直すつもりです。この準備的なリファクタリングによって、各カードや口座の振る舞いを簡単にカプセル化できる地点まで持っていきます。Kent Beck の言葉どおり、「変更を簡単にしてから、簡単になった変更を行う」のです。

この記事が扱うのは、この最初の問題、つまり **このクラスが大きすぎる** という問題です。

### なぜリファクタリングするのか

`ReconciliationIntro` クラスについて簡単に考えられないのは、責務が多すぎるからです。もともとはソフトウェアの玄関のようなものとして設計され、いくつかのメッセージを表示して、主要な仕事をするクラスをインスタンス化するだけでした。しかし時間の経過とともに、ほかのコードがどんどん入り込んできました。もう1枚クレジットカードを追加しやすくしたいので、まずはこの大きなクラスを小さなクラスに分けます。

利点はいくつもあります。メソッド群を別々のクラスへ移すことで、明確な文脈を作り、密結合を減らせます。変更や修正をしたいときにどこへ行けばよいか分かり、修正時には頭の中に収まる小さく独立した部分だけを扱えば済みます。また、状態の操作も小さな文脈の中に閉じるので、システムの別の領域が思わぬ状態変更をしていないか心配する必要が減ります。

そもそもこの状態を避けられたでしょうか。もちろん避けられました。時間がないことに絶望するのではなく、問題箇所に出会うたびに小さな継続的変更を加えられたはずです。red-green-refactor サイクルの refactor 段階はそのためにあり、その習慣はいつでも取り戻せます。

ただ、プレッシャーの中でコードを書いていると、いつも正しい選択ができる状態にあるとは限りません。コードはすぐに硬く汚れ、リファクタリングが手に負えないもののように感じられます。そういうときこそ、どんな状態のコードであっても、**小さな変更で少しずつ良くする** 価値があります。

### どう進めるか

私にとって、リファクタリングでも新しいコードを書くときでも最も重要な原則は、ごく小さなステップで進むことです。目的はいくつかあります。

- すべてのステップで、コードがコンパイルでき、テストが実行できるようにする。
- ある変更でテストが落ちたら、すぐに直せるようにする。小さな変更と小さなコミットにしておけば、どの変更が原因か分かり、調べる量も少なくて済む。
- 自分がいまどこにいるかを頭の中で追えるようにする。単純なリファクタリングのつもりが、元の意図を超えた影響を持つことはよくある。そこで、コンパイル可能でテストが通る状態を保つ習慣がないと、迷い込んで抜け出しにくくなる。

このやり方には、リファクタリング対象がテストで覆われていることが必要です。今回はリファクタリングを始める前にそれを済ませています。ただし将来のリファクタリングの目的の1つは、コードをさらにテストしやすくすることでもあります。

Martin Fowler の『Refactoring』は、基本的なリファクタリング原則を示す推奨文献です。そこでも、小さなステップで動き、小さなコミットごとにビルドし、テストを走らせる価値が強調されています。

これは実在するコードベースであり、散らかったコードベースです。したがってコードはたくさんあります。もっと知りたい人のために、本文中で何度もコードへリンクしていますが、説明は意図的に高いレベルにとどめています。リンクを追わなくても読めるようにしています。

### 1. メソッドを並べ替える

まず `ReconciliationIntro` クラスのすべてのメソッドを、`region` を使って意味のあるグループに並べ替えます。これは commit `f2d9932` です。

図1: メソッドを region に並べ替えた後の `ReconciliationIntro`

#### 原則: 移動やリネームは、編集と別にコミットする

このようにコードを移動するときは、コードをまったく編集せず、別の作業、別のコミットとして行うことが重要です。メソッドを並べ替えると、差分には追加・削除されたコードの塊がたくさん出ます。移動と同時にコードを変えてしまうと、ある行が削除されたのか移動されたのか、移動された行が同じなのか違うのかが分かりにくくなります。Dan Terhorst-North の言葉を借りれば、「変更は構造的か振る舞い的かのどちらかであるべきで、両方であってはならない」のです。

図2: コードを移動したコミットがどう見えるか

メソッドを妥当そうな文脈にグループ化したので、次はそれらを別クラスへ取り出したくなります。どこから始めるべきでしょうか。ファイル読み込みコードは重複が最も多く、いちばん痛みを生んでいます。最終的にはより汎用的にしたいのですが、まずは雑音なしで見られるように取り出します。新しい `FileLoader` クラスを抽出します。ほかのグループも後で別クラスになりますが、そちらはずっと単純なので、この記事では主にファイル読み込みコードに焦点を当てます。

#### 省略したメソッド名

メソッド、パブ、子どもに至るまで、名前を付けることは悪名高い難事です。私の冗長さは、パブで会ったときに聞く長話と同じくらい、メソッド名にも現れます。リファクタリング前は、コードの重複によって、メソッド名が文脈の違いを表すために使われていました。その結果、とても長い名前になっていました。巨大な図を避けるため、多くの図では次のように略しています。

- `BBO`: `bank_and_bank_out`
- `CC1`: `cred_card1_and_cred_card1_in_out`
- `CC2`: `cred_card2_and_cred_card2_in_out`
- `Merge_bd`: `Merge_bespoke_data`
- `BM`: `budgeting_months`
- `ccds`: `credit_card_debits`
- `wpf`: `with_pending_file`

### 2. ファイル読み込みメソッドの関係を分析する

ここまでで行ったのは、同じクラス内でコードを動かしただけです。コミット前にビルドし、すべてのテストを走らせましたが、よほど不器用なことをしていなければ、この前のコミットは些細なもののはずです。次に行うことは、もう少し慎重に考える必要があります。

region の1つから、新しい `FileLoader` クラスへ取り出したいメソッドを見つけました。しかし本当にうまくいくでしょうか。隠れた依存はないでしょうか。それを知るために、移動したいメソッドどうしの関係を描き出します。移動対象のメソッドは図では青で示されています。

図3: 移動対象のファイル読み込みメソッド

すぐに分かるのは、完全に自己完結した文脈ではないということです。青いメソッドのいくつかは、別クラスに残したい黒や緑のメソッドへ呼び戻しています。対処法はいくつかあり、後で説明します。ここで、実行計画を定義できる場所まで来ました。

### 実行計画

完了時には、元の大きな `ReconciliationIntro` クラスは、削ぎ落とされた本体と5つの新しいクラスに分かれます。ただし region とクラスが一対一に対応するわけではありません。後のステップ7では、最初のグループ分けをさらに細かい境界へ洗練するからです。

図4: 実行計画

全体の目的は、この大きなクラスをより小さなクラスへ分けることです。ステップ1と2で、region と図を使ってコードの別々の領域を見つけ、一部のメソッドの関係を分析しました。それによって、この作業を小さなステップでどう進められるかを考える材料が得られました。

計画は次のようになります。状況によって同じ計画にはならないかもしれませんが、進め方に迷うときの悪くないテンプレートです。優先すべきことは、安全で増分的な変更ができるように準備することです。

1. メソッドを並べ替える。
2. ファイル読み込みメソッドの関係を分析する。
3. 親クラスに残るメソッドを、移動するメソッドから呼べるように修正する。
4. 新しい `FileLoader` クラスのために、既存テストを `FileLoaderTests` へ移す。
5. 新しい `FileLoader` クラスを作り、まず2つのメソッドだけを移す。
6. そのほかのメソッドを `FileLoader` へ移す。
7. ファイル読み込み以外の領域も、さらにクラスへ抽出する。

新しい `FileLoader` クラスは、各種のカンマ区切りデータを読み込み、照合の準備としてそれらをマージする責務を持ちます。現在いちばん痛いのはこのファイル読み込みコードなので、ここを最も詳しく説明します。

### 3. 残るメソッドを修正する

ファイル読み込みメソッドから呼ばれているメソッドとして、`Set_path` と `Recursively_ask_for_budgeting_months` の2つを見つけました。

図5: ファイル読み込みメソッドから呼ばれているメソッド

これらがファイル読み込みクラスと強く結合していないなら、対応は2つあります。新しい `FileLoader` クラスから呼び戻せるように public にするか、呼び出し元をファイル読み込みコードから外して、`ReconciliationIntro` の別メソッドへ移すかです。

`Recursively_ask_for_budgeting_months` には前者を、`Set_path` には後者を使います。`Recursively_ask_for_budgeting_months` は最終的には別クラスの public メソッドになりますが、今は外部から呼べることを確認したいだけです。実際にはすでに public でした。テストのためにそうされていたのです。これはそれ自体が臭いであり、別クラスの公開インターフェイスに属するほうがよいという合図です。

一方 `Set_path` は内部のパス変数を変更します。そこで、別に呼び出して、その結果のデータをパラメータとしてファイル読み込みメソッドに渡します。

#### 原則: 一度に一つのことをする。集中する

`Set_path` を編集している途中で、ほかにも変えたい点が2つ見つかりました。まず、`Set_path` は `_path` メンバ変数を変更する副作用を持っています。本当は自己完結して、新しいパスを返すだけのほうがよいでしょう。次に、`Set_path` は2箇所から呼ばれており、呼び出し順もあまり自然ではなく、重複もあります。今すぐ直したくなります。

しかし私は別のリファクタリングの途中です。そこで Trello にメモし、後回しにします。ここで脱線すると、コードを中途半端な状態にして、自分が何をしているか分からなくなるかもしれません。**集中は大事**です。

このステップでは、`Create_pending_csvs` にパラメータを追加し、最初は既定値を付けてコンパイルを保ちます。次に `Set_path` を別に呼び、その結果の `_path` を `Create_pending_csvs` へ渡します。さらに `Create_pending_csvs` の中から `Set_path` 呼び出しを取り除き、渡された値を使います。最後に既定値を外し、すべての呼び出し元が値を渡すことを強制します。この順に進めることで、常にコードをコンパイル可能に保ちます。

### 4. 新しい `FileLoader` クラスを覆うテストを作る

最初に移すメソッドは `Bank_and_bank_out__Merge_bespoke_data_with_pending_file` です。まず、そのメソッドを覆っている既存テストを、これから作る `FileLoader` 用の新しいテストクラスへコピーします。このメソッドにはすでにテストが1つあり、新しい direct debit データを pending ファイルへ正しくマージすることを確認しています。

ここでしているのは、新しいテストを書くことではなく、既存テストを移すことです。TDD ではコードの前にテストを書くという考えに慣れているため、コードに触るたびに新しいテストが必要だと思われることがあります。しかしリファクタリングでは、理想的には既存の機能がすでにテストで覆われています。機能を変えないのですから、既存テストを使って振る舞いが保たれていることを確認します。

このテストを見直すには良いタイミングです。しばらく前に書いたものなので、意味が通るかすぐに分かるはずです。テストは明確で読みやすく、システムの振る舞いの文書として働くべきです。まず気づいたのは、`Assert_direct_debit_details_are_correct` というアサートメソッド名が不十分なことでした。「正しい」とは何でしょうか。そこで読みやすく書き直しました。

その後、新しいテストクラスへテストと関連する private ヘルパーをコピーします。これらのテストはいずれ新しい `FileLoader` クラスを対象にしますが、新しいテストクラスが必要なものをすべて持っていると確信するまでは、古い `ReconciliationIntro` クラスを対象にします。すべてが安全に移るまでは、テストコードは重複します。

#### 自動リファクタリング

Resharper のようなツールがない場合、注意深く扱う必要がある自動化可能なリファクタリングはたくさんあります。Resharper があれば、大きなコードベース全体に呼び出し元を持つメソッド名も、素早く自信を持って変更できます。しかしそのようなツールがなければ、一時的なラッパーメソッドを作るなど、別の進め方が必要になることがあります。

私はまず元のテストと同じファイルに新しいテストクラスを作ります。コピー対象を見やすくするためです。その後、Resharper と Visual Studio を使ってすべてを新しいファイルへ移します。

図7: `BBO` テストの移動

### 5. 新しい `FileLoader` クラスを作り、2つのメソッドを移す

テストクラスが動くようになったので、新しい `FileLoader` クラスを作れます。

メソッドの呼び出しツリーで最も下にある、つまり葉にあたるのが `Bank_and_bank_out__Add_most_recent_credit_card_direct_debits` です。これは private メソッドで、独立したテストはなく、public な呼び出し元を通じてテストされています。そこで、このメソッドと呼び出し元の `Bank_and_bank_out__Merge_bespoke_data_with_pending_file` の両方を移します。これらが新しいクラスへ移る最初の2つのメソッドです。

ここでも、テストを赤にしないよう、コードが常にビルドできるよう、小さなステップで進めます。各ステップの後に、ビルドとテストを確認します。

1. `FileLoaderTests` は作成済みだが、まだ `ReconciliationIntro` にあるコードをテストしている。
2. 新しい `FileLoader` クラスを作る。private のファイル読み込みメソッドを新しいクラスへコピーし、呼び出し元が移るまで待機させる。ただし一時的に public にする必要がある。
3. `ReconciliationIntro` で新しい `FileLoader` インスタンスを作り、古い private メソッドの代わりに新しい public メソッドを呼ぶ。古い private メソッドは削除できる。
4. 元の呼び出し元メソッドを新しいクラスへコピーする。この時点では重複している。
5. テスト対象を `ReconciliationIntro` インスタンスから新しい `FileLoader` インスタンスへ変える。private メソッドもコピー済みなので、テストは通る。
6. `ReconciliationIntro` から、元の呼び出し元を直接呼べるようにする。
7. 一時的に public にしたメソッドを private に戻し、古い呼び出し元と古いテストクラスを削除する。

### 6. 残りのメソッドを `FileLoader` へ移す

これで残りのメソッドを移せます。最も単純なものから1つずつ扱い、メソッドどうしの依存やメンバデータへの依存に注意します。考えるべきことは、名前を変えるか、パラメータリストを新しいオブジェクトに置き換えるか、不要なパラメータはないか、内部の呼び出し先が状態を変更していないか、などです。

ツリーの下のほうにあるものを移す前にインライン化することもできましたが、そうすると public メソッドとして呼んでいるテストを壊してしまいます。そこで、それらは単独で移します。呼び出しツリーの外側の葉から始め、1つずつ進めます。

各メソッドでは、次の手順を使います。

- 移動先クラスにメソッドのコピーを作り、元のメソッドは残す。新しいメソッドは public にする。
- 元のメソッドの代わりに新しいメソッドを呼ぶ。
- 既存のテストを新しいテストクラスへコピーし、新しいコードをテストするようにする。
- 古いメソッドと古いテストを削除する。

最初の2つのメソッドではこの手順をすでに行いました。残りのメソッドでも同じ行動を繰り返します。今回はすべての小さなステップを個別にコミットしてはいませんが、同じ一連の手順を踏んでいます。各ステップの後で、意図的にテストを失敗させたときを除き、コードがビルドでき、テストが通ることを確認します。

一部のテストは以前にリファクタリング済みなので、同じパターンに従うテストにも同じ変更を繰り返せます。テストがないために移動が非常に単純なメソッドもあります。そもそもこのリファクタリングをしている理由の1つは、そのコードをもっとテストしやすくすることです。

#### メンバ変数について

4つの load メソッド、たとえば `Load_bank_and_bank_in` は、`ReconciliationIntro` の2つのメンバ変数 `_input_output` と `_spreadsheet_factory` を使っています。したがって、これらのメソッドを `FileLoader` へ追加するとき、同じ変数を `FileLoader` のメンバとしても導入し、コンストラクタで注入する必要があります。`_input_output` はすでに注入されていますが、`_spreadsheet_factory` を導入するときは最初に既定値を与えます。こうしておけばコードはビルドでき、新しいメソッドへ切り替える前に変更できます。

関連部分だけを単純化すると、最初は次のような形です。

```csharp
internal class ReconciliationIntro
{
    private readonly IInputOutput _input_output;

    public ReconciliationInterface Load_correct_files()
    {
        Load_bank_and_bank_in();
    }

    public ReconciliationInterface Load_bank_and_bank_in()
    {
        var file_loader = new FileLoader(_input_output);
        file_loader.Load_bank_and_bank_in();
    }
}

internal class FileLoader
{
    private readonly IInputOutput _input_output;

    public FileLoader(IInputOutput input_output)
    {
        _input_output = input_output;
    }

    public ReconciliationInterface Load_bank_and_bank_in()
    {
        _input_output.Output_line("Some text");
    }
}
```

次に、`ISpreadsheetRepoFactory` を既定値付きで追加します。これならまだコンパイルできます。

```csharp
internal class FileLoader
{
    private readonly IInputOutput _input_output;
    private readonly ISpreadsheetRepoFactory _spreadsheet_factory;

    public FileLoader(
        IInputOutput input_output,
        ISpreadsheetRepoFactory spreadsheet_factory = null)
    {
        _input_output = input_output;
        _spreadsheet_factory = spreadsheet_factory;
    }

    public ReconciliationInterface Load_bank_and_bank_in()
    {
        var pending_file_io = new FileIO<BankRecord>(_spreadsheet_factory);
        _input_output.Output_line("Some text");
    }
}
```

最後に、`ReconciliationIntro` から必要なメンバ変数を `FileLoader` へ注入します。

```csharp
internal class ReconciliationIntro
{
    private readonly IInputOutput _input_output;
    private readonly ISpreadsheetRepoFactory _spreadsheet_factory;

    public ReconciliationInterface Load_correct_files()
    {
        var file_loader = new FileLoader(_input_output, _spreadsheet_factory);
        file_loader.Load_bank_and_bank_in();
    }
}
```

### 7. さらに新しいクラスを抽出する

このリファクタリングの主な動機は、ファイル読み込みコードの4つのデータ種別について、重複を取り除き Strategy パターンを導入することです。そのためコミットを見ると、`FileLoader` を抽出したあと、私は別の作業へ進んでいます。しかしこの記事を書くために戻ってきたとき、まだ膨れたままの `ReconciliationIntro` を残すことに耐えられず、残りのクラス抽出も進めました。

最初に region を追加したとき、`Get budgeting months` の機能が明確な文脈を作っていることに気づいていました。そこで、これらのメソッドを新しい `BudgetingMonthService` クラスへ抽出します。公開入口が1つしかないため、これはとても素早く単純にできます。

`ReconciliationIntro` はまだ大きすぎます。しかしメソッドは互いに呼び合っており、コードに再び慣れてきた今では、残っている2つの region、`User Instructions and Input` と `Debug Spreadsheet Operations` が最善の分割だとは思えなくなりました。考える助けとして、スプレッドシートで呼び出し階層をざっと図示します。

### 振り返り

「でも、そのコードがその状態になったのは、夜遅くに空き時間をつまみ食いしながら一人で書いていたからでしょう。規律ある、すばらしいエンジニアリング実践を持つチームなら、そんなコードベースにはならないはずです」と言いたくなるかもしれません。しかし、完璧なチームはほとんどありません。多くのチームはどこかで時間の圧力により近道をします。優れたチームが、別の恵まれなかったチームのコードを扱わなければならないこともあります。理由は何であれ、多くのソフトウェアエンジニアはいつか、手に負えなくなったコードをリファクタリングする場面に出会うでしょう。

自分のコードベースを人目にさらすのは不安でした。明らかに完璧からは遠いからです。しかし、これが現実です。最高のプログラマでさえ、どこかのハードドライブに問題のあるコードを抱えている可能性があります。私は自分を最高のプログラマに数えていませんが、それでも同じことです。

私のクラスは大きすぎました。望ましくないコーディング習慣を身につけてしまっており、新しい機能を追加する前に立ち止まって良くしたかったのです。

大きすぎるクラスを直すために、私は次のことをしました。

- 新しいクラスを見つけやすくするため、メソッドを意味のあるまとまりに並べ替えた。
- ファイル読み込みメソッドの関係を分析し、コードの残りと最小限の結合で新しい `FileLoader` クラスを作る方法を考えた。
- 親クラスに残るメソッドを、必要なら移動するメソッドから呼べるように修正した。
- 新しい `FileLoaderTests` クラスを作り、`FileLoader` 用のカバリングテストを用意した。
- 新しい `FileLoader` クラスを作り、2つのメソッドを移した。
- 残りのメソッドを `FileLoader` へ移した。
- さらに新しいクラスを抽出した。

私は小さなステップで進み、各ステップでコンパイルし、テストを走らせました。今では、いくつかの小さなクラスの機能を呼び出す、ずっと小さなクラスがあります。そしてそれぞれが、以前より頭の中に収まりやすくなりました。

### 次は何か

この記事はリファクタリングの途中で終わります。次のステップへ進む前の関連するコードの状態を見たい場合は、commit `6103f0b` をチェックアウトする必要があります。また、この記事のステップ7の大部分は、そのコミットの後に行っています。

commit `6103f0b` の時点で、ファイル読み込みコードは `FileLoader` クラスへ取り出され、主な課題がよりはっきり見えるようになっています。次の4つのメソッドが明らかに問題です。

- `Load_bank_and_bank_in`
- `Load_bank_and_bank_out`
- `Load_cred_card1_and_cred_card1_in_out`
- `Load_cred_card2_and_cred_card2_in_out`

これらには次の問題があります。

- 重複が多く、一見すると同じに見える。
- 長すぎる。
- 内部でオブジェクトを作り、それらを互いに強く依存した形で渡していて、テストできない。

これらはすべて解決したい問題です。しかしさらにリファクタリングする前に、まずこれらのメソッドの周りにテストが本当に必要です。次にするのはそれです。できれば続編の記事として書きたいところですが、そういう約束をするのはやめておく知恵を学びました。今のところは読者への演習としておきます。
