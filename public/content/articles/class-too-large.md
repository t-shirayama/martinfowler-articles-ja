# Refactoring: This class is too large

## 要約

大きすぎるクラスは、多くの責務を抱え、変更理由が増え、理解しづらくなります。
ただし、単純に行数だけで判断するのではなく、どの責務が混ざっているのかを見極める必要があります。

この記事は、大きなクラスを前にしたときに、どこから手を付けるかを考える材料になります。
抽出する単位を探し、小さく安全に分けることが、設計改善の実践になります。

## 読むときの観点

- サイズではなく責務の混在を見る。
- 変更理由が複数ある場所を探す。
- 抽出後の名前が自然かを確認する。
- 一度に大きく分けず、観察しながら進める。

## 原文の翻訳

この記事は、大きすぎるクラスをどう分割するかを扱います。大きなクラスは、多くの場合、複数の責務を抱えています。問題は行数そのものではなく、**変更理由が複数混ざっていること**です。

分割の出発点は、クラスの中にあるデータと振る舞いのまとまりを見つけることです。メソッド同士が同じフィールドを使っているか、ある変更のときに一緒に変わる部分はどこかを観察します。そこから、新しいクラスや値オブジェクトへ責務を移せます。

読むときは、機械的に小さくするのではなく、概念として自然な境界を探すのが大切です。大きいクラスを分ける目的は、単にサイズを減らすことではなく、**理解しやすく、変更しやすい単位を作ること**です。
