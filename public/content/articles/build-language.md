# Build Language

## 要約

`ant`や`make`は、ビルドの進め方を記述するための言語です。
単純なビルドなら、タスクと依存関係の列として表現できますが、大きなシステムでは条件分岐や抽象化が必要になり、より一般的なプログラミング言語の力が欲しくなります。

この記事は、XMLでビルドを記述するよりも、Rakeのように通常のプログラミング言語の上にビルド用の慣習と支援を載せる方が、複雑なビルドに向いていると述べます。

## 読むときの観点

- ビルドスクリプトも一種のプログラムとして読む。
- 単純な依存関係だけなら専用記法で足りるが、複雑になると一般言語の構造が必要になる。
- XMLは階層データの記述には使えるが、ビルド言語として最良とは限らない。
- RakeはRuby内部DSLとして、タスクと依存関係を通常のプログラム内で表す。

## 原文の翻訳

Bruce Eckelが最近書いた`ant`と`make`についての記事をきっかけに、ビルド言語についての考えを共有したくなりました。`ant`も`make`も、ビルドがどのように起きるかを指定します。つまり、ビルドを記述するための言語です。どちらも広く使われており、成功してきました。しかし、どちらも限界にぶつかります。大きなシステムでは、別のプログラムから`ant`や`make`のファイルを生成している人をよく見かけます。

私はBruceに同意していると思います。理由はこうです。単純なビルドは、タスクと依存関係の列として簡単に表現できます。そのようなビルドでは、`ant`や`make`の機能はうまく働きます。しかし、より複雑なビルドには条件付きのロジックが必要です。そしてそれには、より一般的なプログラミング言語の構成要素が必要になります。`ant`や`make`が苦しくなるのはそこです。

`Ant`は、ファイル形式としてXMLを使う判断をしました。当時、私はそれを良い選択だと思っていました。初期の頃、私はMatt Foemmelと大きなプロジェクトで作業しており、彼は`ant`に似たシステムを作っていました。まだ`ant`は存在せず、私たちにはそれが必要だったのです。彼も言語としてXMLを選びました。XMLは階層データを記述するには許容できる方法であり、階層構造はビルドスクリプトの要求に合っているように思えたからです。

しかし私たちはその後、プログラミング言語の方が進むべき道だと結論しました。これはXMLの良い使い方ではありません。

私はRubyでかなりプログラミングをするので、自然と`Rake`を試し始めました。Ruby版の`make`です。`rakefile`のおもしろいところは、それが普通のRubyプログラムであり、タスクと依存関係を宣言するためのいくつかの慣習と支援を備えていることです。私はまだ小さなタスクでしか使っていませんが、今のところとても心地よく感じています。

`Rake`は本格的なプログラミング言語の上にあるので、大きなビルドにもよく合うだろうと期待しています。すでに、ループ、サブルーチン、リストの収集といったことをビルドファイルで行うのに便利だと分かっています。**複雑なビルドには、依存関係の宣言だけでなく、プログラミング言語としての表現力が必要になる**のです。
