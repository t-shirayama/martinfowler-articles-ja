# Use Cases And Stories

## 要約

Use CaseとXPのUser Storyは、どちらも要求を整理するための手段ですが、目的が異なります。Use Caseは、ユーザーがシステムを使って目的を達成する流れを物語として捉えます。一方、Storyはリリース計画やイテレーション計画で扱いやすい単位に要求を分割します。

そのため、両者は一対一に対応するとは限りません。小さなUse Caseが1つのStoryになることもあれば、Use Case内のシナリオや手順がStoryになることもあります。チームによっては両方を使い、別のチームはStoryだけ、またはUse Caseに実装時期を注記する形で進めます。

## 読むときの観点

- Use CaseとStoryを、同じものの粒度違いとしてではなく、目的の違いとして読む。
- Use Caseはユーザーの目的とシステム利用の流れ、Storyは計画と見積もりの単位に注目する。
- Storyはイテレーション内で実装できる大きさまで分割されるため、Use Caseより細かくなりやすい。
- 実務では、両方を必ず作るよりも、チームの計画方法に合わせて使い分ける。

## 原文の翻訳

UseCaseとXPのUserStoryの違いは何でしょうか。

これはよくある質問ですが、一般に合意された答えがあるわけではありません。XPコミュニティの多くの人は、storyをuse caseの単純化された形だと考えています。私も以前はそう考えていましたが、今は違う見方をしています。

Use caseとstoryは、どちらも要求を整理する方法であるという点では似ています。しかし、整理する目的が違います。Use caseは、ユーザーがどのようにシステムに関わり、どのようにシステムを使うかを物語として形づくるために、要求を整理します。そのため、use caseはユーザーの目標と、システムとのやり取りがその目標をどう満たすかに焦点を当てます。XPのstory、そしてしばしばfeatureと呼ばれる類似のものは、計画のために要求をまとまりへ分割します。

Storyは、XPのリリース計画プロセスの一部として見積もれるようになるまで、明示的に分割されます。要求の使い方が異なるため、よいuse caseとよいstoryを判断するための経験則も異なります。**両者は要求を整理する道具ではあっても、同じ目的の道具ではありません。**

両者の対応関係は複雑です。Storyは、通常はより細かい粒度になります。イテレーション、つまりXPでは1週間か2週間の期間内に、完全に作り切れる必要があるからです。小さなuse caseが、そのまま1つのstoryに対応することもあります。しかし、storyがuse case内の1つ以上のシナリオに対応することもあれば、use case内の1つ以上の手順に対応することもあります。新しい資産の減価償却方法をポップアップリストに追加するようなstoryは、use caseの物語にはそもそも現れないかもしれません。

両方を行う必要があるのでしょうか。多くのことと同じく、理論上は必要ですが、実務上はそうとは限りません。あるチームは、初期段階でuse caseを使って物語としての全体像を作り、その後、計画のためにstoryへ分割するかもしれません。別のチームは、直接storyから始めます。また別のチームは、use caseだけを使い、どのfeatureをいつ実施するかがわかるようにuse caseのテキストへ注記するかもしれません。**大切なのは、成果物の名前ではなく、要求を何のために整理しているかです。**
