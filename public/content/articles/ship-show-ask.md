# Ship / Show / Ask

## 要約

Ship / Show / Ask は、プルリクエストの利点を活かしながら、変更を素早く本流へ流し続けるためのブランチ戦略です。変更ごとに、レビューを待たずに mainline へ入れる `Ship`、プルリクエストを開いて共有しつつすぐにマージする `Show`、マージ前に議論やフィードバックを求める `Ask` を選びます。

この考え方は、すべての変更に同じ承認プロセスを課すのではなく、変更の性質やチームの信頼度に合わせて会話の量を調整するためのものです。継続的インテグレーションの速度と、プルリクエストが生むレビュー・学習・対話の場を両立させることを狙います。

## 読むときの観点

- すべての変更をレビュー待ちにすることが、本当に品質や信頼の向上につながっているか。
- チームの変更は、実際には `Ship`、`Show`、`Ask` のどれに近い扱いをされているか。
- 承認を必須にする代わりに、mainline を常にリリース可能に保つ技術とチームの信頼をどう育てるか。
- プルリクエストを会話の唯一の場にせず、早い段階から相談・共有する文化をどう作るか。

## 原文の翻訳

プルリクエストは、多くのソフトウェアチームに広く採用されてきました。プルリクエストを好む人もいれば、ブランチを作らず、チームがいつも変更を統合していた継続的インテグレーションの時代を懐かしむ人もいます。

ある意味で、プルリクエストは状況を大きく変えました。コードホスティングツールはすばらしいコードレビュー機能を提供しています。テスト実行、コード品質のチェック、本格的なプレビュー環境のデプロイまで、プルリクエスト上で動作するサービスを提供する SaaS もたくさんあります。

しかし、コードを貢献する主な方法としてプルリクエストが採用されたことで、問題も生まれました。継続的インテグレーションをしていたときに持っていた「すぐ出荷できる」意識が、いくらか失われてしまったのです。作業中の機能は統合を遅らせることで邪魔にならないようにされ、その結果、継続的インテグレーションが解決しようとしていた低頻度統合の落とし穴に陥ります。

プルリクエストが放置されて古くなることもあります。レビューを待つあいだ、何に取り組めばよいかわからなくなることもあります。「せっかくここにいるのだから、これもやっておこう」と考えるうちに、プルリクエストが膨れ上がることもあります。

レビューしなければならないプルリクエストの数に疲れて、コードについて話さなくなることもあります。注意を払うのをやめ、ただ「Approve」を押したり、「よさそうです」と言ったりするだけになります。

### Ship / Show / Ask の紹介

私が自分のチームで使ってきた、ソフトウェアのブランチに関するアプローチがあります。とてもよく機能したので、ここで紹介します。

変更を加えるたびに、`Ship`、`Show`、`Ask` の3つの選択肢から1つを選びます。

### Ship

これは継続的インテグレーションに最も近い感覚です。変更したいものがあるなら、それを mainline に直接加えます。このやり方では、自分の変更を本番へ運ぶために誰かを待つことはありません。コードレビューを依頼することもありません。面倒はありません。通常の継続的インテグレーションの技法で安全性を確保しながら、ただ変更します。

次のような場合にうまく機能します。

- 確立済みのパターンに沿って機能を追加した。
- 特に珍しくないバグを修正した。
- ドキュメントを更新した。
- フィードバックを受けて自分のコードを改善した。

### Show

ここでは、継続的インテグレーションの考え方を保ちながら、プルリクエストがもたらすよいところも活用します。変更をブランチで作り、プルリクエストを開き、それから誰かを待たずにマージします。自動チェック、たとえばテスト、コードカバレッジ、プレビュー環境などは待つべきですが、変更をライブにするために誰かのフィードバックを待つことはしません。

そうすることで、変更を素早くライブにしつつ、フィードバックと会話の場を作れます。チームはあなたのプルリクエストの通知を受け、あなたが何をしたのかをレビューできます。アプローチやコードについてフィードバックできます。質問もできます。あなたがしたことから学ぶこともできます。

次のような場合にうまく機能します。

- このコードをどう良くできるか、ぜひフィードバックがほしい。
- 私が使った新しいアプローチやパターンを見てほしい。
- X をリファクタリングしたので、今はこう見える。
- 面白いバグだった。どう直したか見てほしい。

### Ask

ここでは立ち止まります。変更をブランチで作り、プルリクエストを開き、マージする前にフィードバックを待ちます。正しいアプローチを取れているか自信がないのかもしれません。満足していないコードがあるものの、どう改善すればよいかわからないのかもしれません。実験をして、みんながどう思うか知りたいのかもしれません。

現代のコードレビューツールは、この種の会話にとてもよい場を提供します。チーム全体でプルリクエストを見ながら議論することさえできます。

次のような場合にうまく機能します。

- これはうまく動くだろうか。
- この新しいアプローチをどう感じるか。
- これをもっと良くするために助けがほしい。
- 今日はここまで。明日マージする。

### ルール

- コードレビュー、あるいは「承認」は、プルリクエストをマージするための必須条件にしない。
- 人は自分のプルリクエストを自分でマージできる。こうすると、その変更が `Show` なのか `Ask` なのかを自分で制御でき、いつライブにするかも決められる。
- mainline をリリース可能に保つために役立つ、継続的インテグレーションと継続的デリバリーの技法を使わなければならない。Feature Toggle はその一例である。
- ブランチは長く生かさず、頻繁に mainline に rebase する。

### 会話

プルリクエストは変更について話す便利な方法になり得ますが、落とし穴もあります。最も魅力的なアンチパターンは、プルリクエストがほかの会話の方法を置き換えられる、という考え方です。

ブランチに関するよくある問題のひとつは、人々がアプローチについて話し合わずに決めてしまうことです。プルリクエストが開かれるころには、最適とは言えないかもしれない解決策に時間が投資されています。レビューする人は選ばれた解決策の影響を受け、別のアプローチを提案しにくくなります。変更セットが大きく、ブランチの寿命が長いほど、この問題は悪化します。作業を始める前にチームと話してください。そうすれば、よりよいアイデアを得て、手戻りを避けられます。

`Show` や `Ask` の方法はプルリクエストだけではない、ということを覚えておきましょう。通話に入る、誰かの席に行く。早く、頻繁に作業を見せる。早く、頻繁に助けやフィードバックを求める。一緒にタスクに取り組む。

プルリクエストを開かないことも、コードについて会話しなくてよい理由にはなりません。チームがよいフィードバック文化を保ち、自分たちの考えや学びについて話し合うことは重要です。

### バランス

選択肢は3つあります。では、どれをより多く選ぶべきでしょうか。

場合によります。どのチームにも、その時点での独自のバランスがあると思います。

確立されたパターンで機能を届けているときは、より多く `Ship` するでしょう。チーム内の信頼が高く、品質基準を共有できている場合も、より多く `Ship` するでしょう。

しかし、まだ互いを知っている途中だったり、全員が新しいことに取り組んでいたりするなら、会話の必要性は大きくなります。そのため、より多く `Show` や `Ask` をすることになります。ジュニアエンジニアはしばしば `Show` や `Ask` をするかもしれません。シニアエンジニアは多くを `Ship` しながら、ときどき新しい技法や、みんなに試してほしいリファクタリングを `Show` するかもしれません。

柔軟性があまりないチームもあります。規制の厳しい業界では、すべての変更に承認やレビューのプロセスが必要になることがあります。ブランチを使うかどうかにかかわらず、それを実装する方法はいろいろありますが、ここでは立ち入りません。

### 自分のチームはこのアプローチを採用すべきか

あなたのチームは、すでに採用しています。

自分のチームの働き方を考えてみると、Ship / Show / Ask の何らかのバランスで動いていることに気づくはずです。私が見てきたほとんどのチームは、「主に Ship」か「主に Ask」のどちらかに入ります。

### 主に Ship している場合

ほとんどブランチを作らず、すべてのコミットが mainline に直接入るなら、あなたは「主に Ship」しています。その場合は、少し `Show` することが役に立つかどうか考えてみてください。

プルリクエストモデルがこれほど広まった大きな理由は、リモートファーストで非同期なチームを支えるからです。自分の作業の面白い部分を明示的に `Show` することで、ほかの人が学び、会話に含まれていると感じやすくなります。これは、特にリモートで働いている人や、勤務時間が異なる人がいる場合に有効です。

また、特に十分に話さないチームでは、常に mainline にコミットしていると、問題のある変更が何週間も経ってからようやく見つかることがあります。そのころには詳細がぼやけていて、有益な会話をするのが難しくなっています。チームメンバーに `Show` のアプローチを促すと、進めながらコードについてより多く話せます。

ペアプログラミングは、チーム内の継続的なコミュニケーションを促す有効な技法のひとつです。

### 主に Ask している場合

チームがほとんどの変更でプルリクエストを開いているなら、あなたは「主に Ask」しています。プルリクエストは品質とフィードバックのための便利な道具ですが、スケールの問題があります。承認を待つという避けられない事実は、時間がかかるということです。フィードバック待ちの変更が多すぎると、フィードバックの質が下がるか、進捗が遅くなります。もっと `Show` を試せば、両方のよいところを得られます。

多くの `Ask` に頼っている理由は、信頼の問題かもしれません。「すべての変更は承認されなければならない」や「すべてのプルリクエストには2人のレビュアーが必要」といったポリシーはよくありますが、それは開発チームへの信頼が不足していることを示しています。

これは問題です。承認ステップは応急処置にすぎず、根本にある信頼の問題を直してはくれません。少し多く `Show` することで、開発パイプラインの圧力をいくらか下げられます。そのうえで、トレーニング、チームでの議論、アンサンブルプログラミングのように、信頼を築く活動に力を向けてください。開発者が `Ask` ではなく `Show` するたびに、その人がチームとの信頼を築く機会になります。

多くの `Ask` に頼っている別の理由は、変更を安全に mainline へ置く方法がないことかもしれません。この場合、mainline をリリース可能に保つ技法を学び、実装する必要があります。それまでのあいだ、より多く `Show` することは、安全な変更を本番へ運ぶ障壁を下げる方法になります。障壁が下がれば、それはチームメンバーへの動機づけにもなります。変更を安全にする方法を見つけられれば、より早くライブにできるからです。

### 結論

では、Ship / Show / Ask とは何でしょうか。根本的には、2つのものです。

第一に、両方のよいところを得るための工夫です。フィードバックを待たずに自分のプルリクエストをマージし、その後で届くフィードバックにはきちんと注意を払います。

第二に、ブランチ戦略をより包摂的で動的に見る方法です。Ship / Show / Ask は、各チームのアプローチが「常に Ship」と「常に Ask」のあいだのどこかにあることを思い出させます。そして、変更をひとつずつ独立に考え、自分に問いかけるよう促します。これは `Ship` すべきか、`Show` すべきか、それとも `Ask` すべきか。
