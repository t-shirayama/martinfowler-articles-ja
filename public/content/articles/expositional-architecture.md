# Expositional Architecture

## 要約

Expositional Architecture は、実際のシステムのアーキテクチャを、学びや発想の材料として紹介するための Fowler の呼び名です。
それは標準や best practice ではなく、ショーホームや見本庭園のように、自分の状況に合う要素を見つけるための例です。

この記事は、成功した点だけでなく失敗した点や意見が割れる点も、アーキテクチャを学ぶうえで重要だと述べています。

## 読むときの観点

- 例を best practice としてではなく、設計判断を考える材料として読む。
- architecture を「重要なもの」として広く捉える Fowler の立場を押さえる。
- 成功例だけでなく、うまくいかなかった要素からも学ぶ姿勢を見る。
- 実際のシステムを扱うため、チームや文脈に依存する語彙や名称が調整されている点に注意する。

## 原文の翻訳

ソフトウェアシステムについての理解を深めるうえでの問題のひとつは、私たちが十分な数の例を見ていないことです。多くの専門分野では、人々はすでに行われたことを見ることで学びます。例は、刺激になり、よいアイデアの源になり、困難への警告にもなります。長い間、ソフトウェアではこのように学ぶことがずっと難しかったのです。

私はいつも、システムがどのように組み立てられているかを見るのを楽しんできました。これからは、そうする時間をもっと取り、私が見る興味深いアーキテクチャのいくつかを共有したいと思っています。私はそうしたアーキテクチャを **Expositional Architecture** と考えています。ショーホームや見本庭園のようなものだからです。それらは自分の目的にぴったり合うものではないかもしれませんが、真似したくなる要素を含んでいます。

この形式の記事の一覧については、expositional architectures タグを参照してください。

私は普段、A-word、つまり architecture という言葉を避けています。とても滑りやすい言葉だからです。この場合には、Ralph Johnson による私の好きな architecture の定義、「重要なもの、それが何であれ」に従います。つまり、私自身の判断と、その開発に関わったチームの判断の両方に基づいて、そのシステムの設計のどこが興味深いと思うかを語る、ということです。

私が expositional という言葉を使うのは、これらのアーキテクチャが興味深いアイデアの源であり、何らかの best practice として意図されたものではないことを強調するためです。そもそも私は、何らかの標準として設定されるアーキテクチャにはとても慎重です。具体的なシステムを作るときには、注意を払うべき変数が非常に多いからです。たとえば、多くの人はスケーラブルなアーキテクチャの重要性を強調します。この場合、彼らはたいてい、大量の負荷を処理できることを意味しています。しかし有用なシステムの多くは、高い負荷を受けることのない内部システムです。そうしたシステムは、別のドライバーを支えるように設計されるべきです。

成功しなかった要素を強調することも、しばしば有用です。私たちは、うまくいったものを見るだけで学ぶわけではありません。うまくいかなかったものは、魅力的に見える道を避けるためのよい手がかりになることがよくあります。また、あるアーキテクチャ上の特徴が、チームメンバーの一部には非常に好まれ、別のメンバーには嫌われることもよくあります。その好意と嫌悪を動かしているものを理解すれば、それが自分のアーキテクチャ上の美意識に合うかどうかの印象を得られます。

私は Expositional Architecture の記事をそれほど頻繁には書かないでしょう。書くには多くの時間と労力がかかるからです。システムを十分に理解し、それがどう動いているか、どこが興味深い部分なのかを感じ取るには時間がかかります。また、開発チームが物事を説明し、私が何かを完全に取り違えていないか確認するための時間と労力も必要です。

私が説明するアーキテクチャの多くは、Thoughtworks のプロジェクトから来るだろうと思っています。そのほうが私にとって近づきやすいからです。しかし、これは絶対的なルールではありません。時間があるときに何かが目に留まれば、TW 以外のプロジェクトについても喜んで話します。実際、私が最初に書いた Expositional Architecture は LMAX で、Thoughtworks の取り組みではありませんでした。ただし、そこで働いていた元 ThoughtWorker が私の連絡先でした。

私が書く Expositional Architecture は、少なくともひとつの実際のシステムに基づきます。私は、本番運用されてから1年ほど経ったものを好みます。開発中にはよく見えたものでも、しばらく本番で動かしてみると違って見えることが多いからです。ただし、それは絶対的なルールではなく、好みです。

アーキテクチャを説明するときには、参照しやすいようにそのアーキテクチャに名前を付けます。しかし通常、この名前は私の説明のために作ったものです。実際のシステム名は、クライアントの文脈と強く結びつきすぎているからです。同じように、私はここで書いている他の記事と一貫する語彙でシステムを説明します。その語彙は、開発チームが実際に使っている用語とは異なるかもしれません。
