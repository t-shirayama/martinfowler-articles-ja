# Structured-Prompt-Driven Development (SPDD)

## 要約

原文タイトル: Structured-Prompt-Driven Development (SPDD)  
原文URL: https://www.martinfowler.com/articles/structured-prompt-driven/  
著者: Wei Zhang, Jessie Jie Xia  
作成日: 2026年4月28日  
翻訳状態: 日本語訳・要約  
このページは非公式の日本語訳・要約であり、Martin Fowler 氏、martinfowler.com、Thoughtworks の公式翻訳ではありません。

SPDD は、LLM に渡すプロンプトをその場限りのチャットではなく、レビュー可能で、バージョン管理され、再利用できる成果物として扱う開発方法です。個々の開発者が AI で速くコードを書く段階から、チームや組織が AI による変更を統制し、意図と実装を同期させながら届ける段階へ移るための方法として説明されています。

中心にあるのは REASONS Canvas と呼ばれる構造化プロンプトの枠組みです。Requirements、Entities、Approach、Structure、Operations、Norms、Safeguards によって、要求、ドメイン、設計方針、実装手順、標準、制約を一つのプロンプト資産にまとめます。**現実とコードがずれたら、まずプロンプトを直し、その後でコードを直す**というワークフローが重要なルールです。

記事では、LLM 利用料金を計算する請求エンジンに、モデル別料金と複数プランの課金ロジックを追加する例を通じて、要求整理、分析、Canvas 作成、コード生成、テスト生成、レビューの流れを説明しています。最後に、SPDD が効果を発揮する場面、向かない場面、開発者に求められる三つのスキルを整理しています。

## 読むときの観点

- SPDD は「もっと速くコードを出す」ためだけではなく、AI 生成物をチームで統制するための方法だと捉える。
- プロンプトを、実装意図と設計判断を残す一次成果物として扱う点を見る。
- REASONS Canvas が、要求から実装手順、標準、禁止事項までをどのように一つの構造に集めるかを確認する。
- コード生成前の抽象化、合意形成、反復レビューが、AI 時代の開発者の主要な仕事として位置づけられている点に注目する。
- SPDD は論理や制約が重い領域に向き、短命の作業や美的判断が中心の作業には重すぎる可能性がある。

## 原文の翻訳

LLM を使ったプログラミング支援は、個々の開発者には大きな価値を示してきました。しかし、個人が速くコードを書けることは、そのままチーム全体のスループット向上にはつながりません。あいまいな要求が素早くコードになれば、誤解も素早く広がります。レビューはより多くの変更を扱う必要があり、統合やテストの問題も増えます。生成されたからといって、要求や設計に合っているとは限りません。

この問題に対して、Thoughtworks の内部 IT チームは Structured Prompt-Driven Development、すなわち SPDD と呼ぶ方法を作りました。SPDD の狙いは、AI 支援を個人の効率化から、品質を犠牲にせず拡張できる組織的な能力へ変えることです。

SPDD では、プロンプトを一級のデリバリー成果物として扱います。場当たり的な会話に頼るのではなく、プロンプトをバージョン管理し、レビューし、再利用し、時間とともに改善します。構造化プロンプトには、要求、ドメイン用語、設計意図、制約、タスク分解が含まれます。LLM はその定義された境界の中でコードを生成するため、出力はより予測しやすく、検証しやすくなります。

SPDD の第一の構成要素は REASONS Canvas です。これは、プロンプトを作るための七つの部品からなる構造です。R は Requirements、つまり解くべき問題と完了条件です。E は Entities、ドメインのエンティティと関係です。A は Approach、要求を満たすための戦略です。S は Structure、変更がシステムのどこに入るか、どのコンポーネントや依存関係に関わるかです。O は Operations、抽象的な戦略をテスト可能な実装手順へ分解する部分です。N は Norms、命名、観測性、防御的コーディングなどの横断的な標準です。最後の S は Safeguards、性能、セキュリティ、不変条件など、破ってはいけない境界です。

この Canvas は、コード生成の前に意図と境界をそろえます。仕様が一つの構造化プロンプトに集まるため、レビュー担当者は散らばったチャットログや部分的な差分ではなく、一つの成果物を見て判断できます。さらに、同じ構造を繰り返し使うことで、プロンプトはチームで統制しやすくなります。ドメイン知識や設計判断がプロンプトに蓄積されるため、個人の知見も反復を通じてチーム資産になっていきます。

第二の構成要素は SPDD のワークフローです。このワークフローは、プロンプトをコードと同じ規律の中に置きます。コミット履歴、レビュー、品質ゲートの対象にするのです。重要な原則は、**現実と実装がずれたら、まずプロンプトを修正し、それからコードを更新する**ことです。これにより、レビューは単にバグを探す作業から、意図が正しいかを確認する作業へ近づきます。

SPDD は Spec-Driven Development と同じように、コード生成の前に仕様を書くことから始めます。ただし SPDD は、その仕様をどのように作り、レビューし、コードと同期させ続けるかに焦点を置きます。プロンプトは一度生成して捨てるものではなく、意図の記録として保守されます。要求から実装仕様へ進む中で、REASONS Canvas はシステムが何をするかだけでなく、選ばれた設計方針、構造、標準、制約も記録します。

記事の例では、既存の請求エンジンに機能を追加します。このシステムは、LLM の利用トークン数から料金を計算する小さなバックエンドです。拡張では、`POST /api/usage` に `modelId` を追加し、モデルごとの料金を扱えるようにします。Standard プランでは月間クォータを維持しつつ、超過分をモデル別単価で計算します。Premium プランではクォータを設けず、prompt tokens と completion tokens を別々の単価で課金します。将来の課金モデルを追加しやすいように、Strategy や Factory のような拡張可能な設計も求められます。

最初のステップでは、大きな要求をユーザーストーリーに分割し、最終的には例に適した簡潔なストーリーへ統合します。ここで重要なのは、実装詳細を急いで決めることではなく、背景、ビジネス価値、スコープ、受け入れ条件をそろえることです。受け入れ条件は、無効なリクエスト、Standard プランのモデル別超過課金、Premium プランの分割課金という三つの観点で整理されます。

次のステップでは、開発者がストーリーを分析します。新しい必須フィールドは `modelId` であり、これがモデル別単価を選ぶ鍵になります。Standard プランでは既存のクォータ処理を保ち、超過分だけモデル別に計算します。Premium プランでは全トークンが課金対象になり、prompt と completion を別計算します。スコープ外として、顧客 CRUD、過去請求の照会、サブスクリプション管理、モデル管理などは扱いません。

その後、分析結果から REASONS Canvas を作成します。Canvas は、ビジネス上の意図を開発者が読める設計方針へ変換し、さらに LLM が実装できる手順へ落とし込みます。ここでは、課金計算の責務を分けること、プランごとの計算を分離すること、入力検証を維持すること、既存の回帰を壊さないことなどが明示されます。単なるプロンプトではなく、レビュー可能な実装計画として扱われます。

コード生成の段階では、LLM は Canvas に書かれた Operations、Norms、Safeguards に従ってタスクごとに変更を行います。生成されたコードはそのまま信頼するのではなく、レビュー、修正、テストを通じて検証します。さらに、ユニットテストや API テストの生成もワークフローに含められます。テストは、通常ケース、境界条件、エラーケースを明確にし、Canvas に書かれた意図が実装で保たれているかを確かめる役割を持ちます。

この例が示す成果は、単なる機能追加ではありません。変更の意図がプロンプトに残り、設計判断が追跡可能になり、レビューの焦点がコードの細部だけでなく意図と境界に移ります。成功したパターンは、次の機能追加で再利用できる資産になります。SPDD の価値は、コード生成そのものよりも、**AI に任せる範囲を明確に定め、その結果をチームで検証できる形にすること**にあります。

SPDD を使いこなすには、三つのスキルが重要です。第一は abstraction first、つまりコード生成の前に、どんなオブジェクトが存在し、どう協調し、どこに境界があるかを明確にする力です。第二は alignment、実装前に何をするか、何をしないか、どの標準や制約に従うかを合意する力です。第三は iterative review、AI の出力を一回限りの草案ではなく、制御された反復プロセスの中で育てる力です。

SPDD がよく合うのは、標準化された反復的な業務ロジック、規制やセキュリティなど強い制約がある領域、複数人での監査可能な開発、複数サービスにまたがる整合性が必要なリファクタリングです。一方で、緊急のホットフィックス、短命のスクリプト、探索的なスパイク、ビジネスルールが曖昧な領域、純粋に美的判断が中心の作業には向きにくいとされています。

SPDD には投資も必要です。設計を先に考えるというマインドセット、抽象化やモデリングに強い開発者、プロンプトを安定して運用するための自動化が求められます。そのため、現時点では熟練者向けに見える面もあります。記事では、今後は個人の職人技に依存するよりも、組織レベルの資産システムによって SPDD を扱いやすくしていく方向が示されています。

結論として、SPDD は「最短でコードを生成する方法」ではありません。要求、抽象化、境界、実装手順、レビューをそろえ、AI が動く空間を明確にする方法です。論理が重く、長期的な保守性や監査可能性が重要な領域では、正しい変更を自信を持って届けるための有力な手段になります。
