# Pair Programming

## 要約

Pair Programming は、2人の開発者が一緒にコードを書くソフトウェア開発のプラクティスです。多くの場合、同じ画面や同じキーボードを共有し、変更の一歩一歩について話し合いながら進めます。

一見すると2人で1つの作業をするのは無駄に見えるかもしれません。しかし、プログラミングの難しさは単にキーボードを打つことではなく、コードの理解を積み上げ、どう変えるべきかを判断することにあります。この記事は、Pair Programming を効果的に行うには技術だけでなく、集中した協働に慣れるためのスキルも必要だと述べています。

## 読むときの観点

- Pair Programming を「2人で同じタイピングをすること」ではなく、理解と判断を共同で進める方法として読む。
- 効率の議論では、入力速度ではなく、誤った分析や手戻りを減らせるかに注目する。
- 集中的な社会的接触が合わない人もいる一方で、経験後に評価が変わる人もいる点を押さえる。
- うまく行うには学習が必要で、雑に導入すると逆効果にもなりうる。

## 原文の翻訳

Pair Programming は、開発者が2人一組で作業するソフトウェア開発の実践である。重要なコードは2人のプログラマーによって書かれ、典型的には横に並んで座り、1つのモニターを見ながら、場合によっては1つのキーボードを共有して進める。コードを追加するとき、2人はそれぞれの手順について話し合う。

最初に Pair Programming を見た多くの人は、2人をこのように使うのは無駄だと感じる。しかし、それが無駄になるのは、プログラミングで一番難しい部分がタイピングだと考える場合だけである。実際のプログラミングは、コードがどう動いているかの理解を積み上げ、どう変更するのが最善かを判断し続ける作業である。

2人で一緒に考えると、そのような問題に対して別々に作業するより前に進めることがある。Fowler は、すぐ近くにもう一つの頭脳があることで洞察を得るのが速くなり、何時間も浪費しかねない分析の袋小路に入りにくくなると述べている。**価値は打鍵の分担ではなく、理解と判断を共有することにある**。

Pair Programming は、集中を要する協働的な活動である。そのため、このような濃い社会的接触が合わない人も多いだろう。一方で、内向的な人を含め、最初は嫌いになると思っていた人が、適切に経験してみると好きになった例も多い。

Pair Programming はスキルであり、下手に行うと機能しない。うまく学ぶ最良の方法は、すでに上手にできる人と組むことである。それが難しい場合は、経験あるペアプログラマーによる詳しい記事を読み、よくある誤解についての補足記事も参照するとよい。
