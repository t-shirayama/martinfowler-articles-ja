# Contract Test

## 要約

外部サービスをテストダブルで置き換えると、自分たちのテストは速く安定します。しかし、そのダブルが外部サービスの実際の契約を正しく表しているかは別問題です。Contract Test は、テストダブルに対して想定している呼び出しが、外部サービスでも同じ契約として成立するかを確認するためのテストです。

通常のビルドのたびに必ず走らせるというより、外部サービス側の変更リズムに合わせて定期的に実行し、失敗したらダブルや自分たちのコード、あるいは提供側との認識を更新するきっかけにします。よりよい形としては、Consumer-Driven Contracts に進み、提供側が利用側の契約テストを自分たちのパイプラインで実行できるようにします。

## 読むときの観点

- テストダブルを使う利点と、その正確性をどう担保するかを分けて考える。
- Contract Test の失敗は、単なるビルド失敗ではなく、契約変更に対する調整の合図として読む。
- 外部サービスの実データそのものではなく、呼び出しの契約や応答の形式を確認する点に注目する。

## 原文の翻訳

`TestDouble` を使うもっとも一般的な場面のひとつは、外部サービスと通信するときです。典型的には、そのようなサービスは別のチームが保守しており、遅くて信頼しにくいネットワーク越しにあり、サービス自体も信頼しきれないことがあります。だからこそテストダブルは便利です。自分たちのテストが遅くなったり不安定になったりするのを防いでくれます。しかし、ダブルに対してテストする場合、常に問題になるのは、そのダブルが本当に外部サービスを正確に表しているのか、そして外部サービスが契約を変更したら何が起こるのか、という点です。

これに対処するよい方法は、自分たちのテストは引き続きダブルに対して実行しつつ、それとは別に Contract Test のセットを定期的に実行することです。これらのテストは、テストダブルに対して行うすべての呼び出しが、外部サービスに対して行った場合と同じ結果を返すかを確認します。**Contract Test の失敗は、サービス契約の変更に合わせてテストダブルを更新し、おそらく自分たちのコードも見直す必要があることを示します。**

これらのテストは、通常のデプロイメントパイプラインの一部として実行する必要はありません。通常のパイプラインは自分たちのコード変更のリズムに基づきますが、これらのテストは外部サービスの変更リズムに基づくべきものです。多くの場合、1日に1回実行するだけで十分です。

Contract Test の失敗は、通常のテスト失敗と同じようにビルドを壊す必要があるとは限りません。ただし、整合性を取り戻すための作業を起こすべきです。それは、外部サービスとの整合性を回復するためにテストやコードを更新することかもしれません。同じくらいあり得るのは、外部サービスを管理している人たちと会話し、変更について話し合い、その変更が他のアプリケーションにどのような影響を与えているかを知らせることです。

この外部サービス提供者とのコミュニケーションは、そのサービスが本番アプリケーションの一部として使われている場合にはさらに重要になります。このような場合、契約変更が本番アプリケーションを壊し、緊急修正や提供側チームとの急ぎの会話を引き起こすことがあります。

契約が予期せず壊れる可能性を下げるには、Consumer Driven Contracts のアプローチへ進むのが有用です。そのためには、自分たちの Contract Test のコピーを提供側チームに渡し、提供側のビルドパイプラインの一部として実行できるようにするとよいでしょう。

このように外部サービスをテストする場合、通常はその外部サービスのテスト用インスタンスに対して行うのが最善です。場合によっては本番インスタンスを叩かざるを得ないこともあります。そのときは、何が起きているのかを提供側に伝え、テストが行うことに特に注意する必要があります。

Contract Test は外部サービス呼び出しの契約を確認しますが、必ずしも正確なデータそのものを確認するわけではありません。スタブは、ある特定の日付時点の応答をスナップショットとして持つことがよくあります。この場合に重要なのは実際のデータではなくデータの形式です。したがって Contract Test は、実際のデータが変わっていても、形式が同じであることを確認する必要があります。

こうしたテストダブルを作る最良の方法のひとつは、`SelfInitializingFake` を使うことです。
