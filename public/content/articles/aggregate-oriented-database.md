# Aggregate Oriented Database

## 要約

Aggregate Oriented Databaseは、key-value、document、column-familyのように、関連するデータのまとまりを保存の基本単位にするデータベースです。DDDのAggregateに近い単位でデータを扱うため、アプリケーションのプログラミングモデルやクラスタ上での分散戦略と相性がよい、という利点があります。

一方で、集約とは別の切り口でデータを分析したい場合には扱いにくくなります。明確な集約を操作する場面ではaggregate-oriented databaseを使い、異なる見方でデータを切り分けたい場面ではrelational databaseやgraph databaseを使う、というPolyglot Persistenceの考え方につながります。

## 読むときの観点

- NoSQLの分類を、個別製品名ではなく保存単位の違いとして見る。
- DDDのAggregateが、クラスタ分散やshardingの単位としても働く点に注目する。
- プログラミングのしやすさと、別視点での分析しやすさのトレードオフを意識する。
- map-reduceやmaterialized viewが、集約構造をまたぐ読み方を補う仕組みとして語られている点を見る。

## 原文の翻訳

『NoSQL Distilled』に取り組み始めたとき、すぐに浮かんだ話題のひとつは、NoSQLデータベースがリレーショナルモデルとは異なるデータモデルを使うということでした。多くの資料では、データモデルを少なくともkey-value、document、column-family、graphの4つのグループに分けています。このリストを見ると、最初の3つには大きな共通点があります。どれも、密接に関連したデータの豊かな構造を、保存の基本単位として持っているのです。key-value storeではvalue、document storeではdocument、column-family storeではcolumn familyがそれに当たります。DDDの用語で言えば、このデータのまとまりは**Aggregate**です。

NoSQLデータベースの台頭は、主にGoogleやAmazonで使われるような大規模クラスタ上で、データを効率よく保存したいという要求から生まれました。リレーショナルデータベースはクラスタを前提に設計されたものではなかったため、人々は代替手段を探しました。Aggregateを基本単位として保存することは、クラスタ上で動かすには理にかなっています。Aggregateは、一緒にアクセスされると期待される大きなデータの塊なので、shardingのような分散戦略にとって自然な単位になります。

Aggregateは、アプリケーションプログラマにとっても意味があります。画面いっぱいの情報を取り込み、それをリレーショナルデータベースに保存するなら、保存前にその情報を行へ分解しなければなりません。Aggregateであればマッピングはずっと単純になります。NoSQLデータベースの初期採用者の多くが、より扱いやすいプログラミングモデルだと報告する理由はここにあります。

プログラミングモデルと分散モデルがかみ合うことには、大きな価値があります。アプリケーションプログラマがどのようにデータをまとめているかをデータベースが知っていれば、その知識をクラスタ全体の性能向上に使えるからです。

ただし、重要な弱点もあります。この方法は、データアクセスがAggregateとそろっているときにはとてもうまく機能します。しかし、別の見方でデータを眺めたい場合はどうでしょうか。注文入力では、注文をAggregateとして保存するのが自然です。しかし、商品の売上分析はAggregate構造を横断します。データベースでAggregate構造を使わない利点は、さまざまな利用者のために、データを異なる切り口で切ったり組み替えたりできることです。

Aggregate-oriented storeでmap-reduceがよく語られるのはこのためです。map-reduceはクラスタ上で動かすのに向いたプログラミングパターンです。map-reduce jobは、異なる読み手のためにデータを別のまとまりへ再編成できます。多くの人がmaterialized viewと呼ぶものです。ただし、これはリレーショナルモデルを使うよりも手間がかかります。

これはPolyglot Persistenceを支持する論点の一部です。明確なAggregateを操作しているとき、特にクラスタ上で動かしているときにはaggregate-oriented databaseを使います。一方で、同じデータをさまざまな方法で操作したいときには、relational databaseやgraph databaseを使うのがよいのです。
