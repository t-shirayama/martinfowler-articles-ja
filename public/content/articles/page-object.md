# Page Object

## 要約

Page Object は、Web ページやページ内の断片を、テストから扱いやすいアプリケーション固有の API で包むパターンです。
テストが HTML 要素や UI ウィジェットの細部へ直接依存しないようにし、UI 変更に対してテストを壊れにくくします。

この記事では、Page Object をページ単位ではなく利用者に意味のある画面要素単位で作ること、Page Object 自体には原則としてアサーションを持たせないこと、そして UI の詳細を隠すカプセル化の例として捉えることが説明されています。

## 読むときの観点

- Page Object が隠すべきものは HTML 構造やウィジェットの細部だと見る。
- 「ページ」ではなく、ユーザーに意味のある画面要素をモデル化する。
- Page Object とテスト DSL、アサーション責務の境界に注目する。
- UI を通したテストが必要な場面と、UI からロジックを逃がす設計の関係を考える。

## 原文の翻訳

Web ページに対するテストを書くときには、そのページ内の要素を参照して、リンクをクリックしたり、何が表示されているかを判断したりする必要があります。しかし、HTML 要素を直接操作するテストを書くと、UI の変更に対してテストが壊れやすくなります。page object は、HTML ページ、またはその断片を、アプリケーション固有の API で包みます。これにより、HTML の中を探し回らずにページ要素を操作できます。

page object の基本的な経験則は、人間ができることや見られることを、ソフトウェアのクライアントにもできるようにすることです。また、プログラムしやすく、ウィンドウ内の下層のウィジェットを隠すインターフェースを提供すべきです。したがって、テキストフィールドにアクセスするには文字列を受け取り返すアクセサメソッドを用意し、チェックボックスには boolean を使い、ボタンは行動を表すメソッド名で表現するのがよいでしょう。

page object は、GUI コントロールそのものの中にあるデータを見つけて操作するために必要な仕組みをカプセル化すべきです。よい経験則は、具体的なコントロールを変更することを想像することです。その場合でも、page object のインターフェースは変わるべきではありません。

“page” object という言葉にもかかわらず、これらのオブジェクトは通常、ページごとに1つ作るものではなく、ページ上の重要な要素ごとに作るものです。ここには、“page object” という名前が誤解を招くという議論があります。1ページにつき1つの page object だけを持つべきだと思わせるからです。“panel object” のような名前のほうがよかったかもしれません。しかし “page object” という用語が定着しました。これは、名前付けが難しいことのもうひとつの例でもあります。たとえば複数のアルバムを表示するページなら、いくつかの album page object を含む album list page object があるでしょう。おそらく header page object と footer page object もあるでしょう。とはいえ、複雑な UI の階層の一部は、UI を構造化するためだけに存在しています。そのような合成構造は、page object によって露出されるべきではありません。

経験則は、**アプリケーションの利用者にとって意味のあるページ構造をモデル化する**ことです。

同じように、別のページへ移動する場合、最初の page object は新しいページのための別の page object を返すべきです。一般に、page object の操作は基本型、たとえば文字列や日付、または別の page object を返すべきです。ナビゲーションのようなことに応じて page object が別の page object を作る責任を持つ、という助言はよくあります。ただし、実践者の中には、page object が汎用的なブラウザコンテキストを返し、テストがそのコンテキストの上にどの page object を構築するかをテストの流れに基づいて制御する、というやり方を好む人もいます。特に条件分岐する流れではそうです。

その好みの根拠は、テストスクリプトが次に期待されるページを知っており、その知識を page object 自身の中に重複させる必要はない、という点にあります。静的型付け言語を使っている場合、この好みはさらに強くなります。ページ遷移が型シグネチャに表れることが多いからです。

page object 自体にアサーションを含めるべきか、それともテストスクリプトがアサーションを行うためのデータを提供するだけにすべきかについては意見が分かれます。page object にアサーションを含めることを支持する人は、これによってテストスクリプトでのアサーションの重複を避けやすくなり、よりよいエラーメッセージを出しやすくなり、より Tell, Don't Ask らしい API を支えられると言います。

アサーションを持たない page object を支持する人は、アサーションを含めると、ページデータへのアクセスを提供する責務とアサーションロジックが混ざり、肥大化した page object につながると言います。

私は、page object にはアサーションを持たせないほうを好みます。一般的なアサーションのためのアサーションライブラリを用意すれば、重複を避けられると思いますし、それによってよい診断情報も出しやすくなります。私のように通常はアサーションなしのスタイルを好む人にとっても、問題のないアサーションの形が1つあります。それは、特定のテストが調べている事柄ではなく、その時点でのページやアプリケーションの不変条件を確認するアサーションです。

page object はテストでよく使われますが、**それ自体がアサーションを行うべきではありません**。その責務は、下層のページの状態へアクセスする手段を提供することです。アサーションロジックを実行するのは、テスト側のクライアントの役割です。

ここでは HTML の観点からこのパターンを説明しましたが、同じパターンはどの UI 技術にも同じように適用できます。Java Swing UI の詳細を隠すためにこのパターンが効果的に使われているのを見たことがありますし、他のほとんどすべての UI フレームワークでも広く使われているはずです。

並行性の問題も、page object がカプセル化できる別の話題です。これには、ユーザーには非同期に見えない非同期操作の非同期性を隠すことが含まれるかもしれません。また、UI スレッドとワーカースレッドの間で振る舞いを割り当てることを気にしなければならない UI フレームワークで、スレッドに関する問題をカプセル化することも含まれるかもしれません。

page object はテストで最もよく使われますが、アプリケーションの上にスクリプティングインターフェースを提供するためにも使えます。通常は、スクリプティングインターフェースを UI の下に置くのが最善です。そのほうが、たいてい複雑さが少なく、速いからです。しかし、アプリケーションが UI にあまりにも多くの振る舞いを入れてしまっている場合には、page object を使うことが苦しい状況をいくらかましにする最善策になるかもしれません。ただし、可能ならそのロジックを移す方向を探してください。そのほうが、スクリプティングにとっても UI の長期的な健全性にとってもよいからです。

Cucumber や内部 DSL のような Domain Specific Language を何らかの形で使ってテストを書くことはよくあります。そうするなら、テスト用 DSL を page object の上に重ねるのが最善です。DSL 文を page object への呼び出しへ変換するパーサを持つ、という形です。

テストメソッドの中に WebDriver API があるなら、それはやり方を間違えています。これは Simon Stewart の言葉です。

Presentation Model、Supervising Controller、Passive View のように UI 要素からロジックを移すことを狙うパターンは、UI を通してテストする有用性を下げます。そのため、page object の必要性も下がります。

page object はカプセル化の古典的な例です。UI 構造やウィジェットの詳細を、他のコンポーネント、つまりテストから隠します。開発するときには、このような状況を探すのがよい設計原則です。「ソフトウェアの残りの部分から、どんな詳細を隠せるだろうか」と自問してください。どのカプセル化にも言えるように、これには2つの利点があります。すでに強調したように、UI を操作するロジックを1箇所に閉じ込めれば、システム内の他のコンポーネントに影響を与えずにそこで変更できます。

もうひとつの重要な利点は、クライアント、つまりテストコードが理解しやすくなることです。そこにあるロジックがテストの意図についてのものになり、UI の詳細で散らからないからです。

私は最初、このパターンを Window Driver という名前で説明しました。しかしその後、“page object” という用語が Selenium の Web テストフレームワークによって広まり、一般的に使われる名前になりました。

Selenium の wiki は page object の利用を強く勧めており、どのように使うべきかについて助言を提供しています。また、アサーションを持たない page object を支持しています。

あるチームは、ソフトウェアアップグレード後に Selenium テストスイートの2つの版を更新する時間を測定しました。page object を使った版は最初のテストケースでは少し時間がかかりましたが、残りのテストではずっと速くなりました。詳しくは Leotta らの “Improving test suites maintainability with the page object pattern”, ICSTW 2013 を参照してください。
