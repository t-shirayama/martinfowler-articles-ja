# Incremental Migration

## 要約

Incremental Migrationは、データ移行を本番直前の一度きりの作業として扱わず、短いiterationごとに少しずつ実行・拡張していく考え方です。データ移行は軽く見られがちですが、実際には汚れたデータ、捨てる前提の低品質な移行コード、本番直前の圧力が重なり、大きなリスクになります。

この記事では、痛い作業ほど頻繁に行うというagileの考え方をデータ移行に適用します。早く小さく移行を繰り返すことでリスクとストレスを下げるだけでなく、domain expertとの会話に実データを使えるようになる点が大きな利点だと述べています。

## 読むときの観点

- データ移行は「一度だけ動く捨てコード」と見なすほど危険になる。
- 既存データの汚さは、抽象的な設計議論では見えにくい。
- incremental migrationは、リリース直前のpanicを減らすリスク管理である。
- 実データを使うことで、domain expertとの会話が具体的になる。

## 原文の翻訳

どんな専門職にも、ふだん忘れられがちで無視されることが多いのに、ちょうど悪いタイミングで噛みついてくる活動があります。ソフトウェア開発におけるその1つがdata migrationです。

多くの新しいソフトウェアプロジェクトでは、どこか別の場所に存在していたデータを、新しいシステムが稼働した後に移す必要があります。システム置き換えでは古いデータをすべて移す必要があるかもしれませんし、新機能のために別システムからデータを読み込むこともあります。

この作業は真剣に扱われないことがよくあります。結局のところ、少しデータを読み、いくらか変換し、新しいシステムへ読み込むだけだと思われるからです。さらに、そのコードは一度しか実行しないので、速くしたりきれいにしたりする意味はないと考えられます。移行が終われば、そのコードは安全に捨てられるというわけです。

そしてもちろん、プロジェクトの最後まで心配する必要はありません。新システムの本番稼働直前に移行を実行したいだけだからです。

私の読者は、少なくともソフトウェア記事の好みからして見識が高いので、ここで苦笑している姿が目に浮かびます。data migrationはwhiteboard上の抽象の安全圏から見ると簡単そうに見えますが、実際には足をすくう厄介な細部でいっぱいです。

- 既存データが多少汚れているとは思っていても、実際の汚さにはたいてい誰もが驚かされる。その結果、作業全体は本来よりはるかに複雑になることが多い。
- 一度きりで捨てるコードなので、migration codeにはあまり設計努力が払われない。DesignPayoffLineを下回ると想定されるからだ。しかしその想定はしばしば間違っており、特に前の点がある場合はそうである。
- 思っていたより難しく膨れ上がる作業は、それだけで楽しくない。それを出荷直前まで放置すると、問題に大きな契約金を渡しているようなものになる。

agileの文脈で私が好んで使う短い言葉があります。痛いなら、もっと頻繁にやる、というものです。表面的には非論理的なので覚えやすく、その中には本当の真実があります。多くの難しい活動は、より頻繁に行うことで、ずっと扱いやすくできます。XPerはこの原則をtesting、integration、design、planningへ適用することでよく知られています。したがって、それがdata migrationへ適用されても驚くことではありません。

私がこれを初めて見たのは、同僚のJosh Mackenzieが、ほどほどの規模のプロジェクトで行ったときでした。1年で12人ほどの開発者が関わり、直近に2度の失敗を経験していたプロジェクトです。彼は2週間ごとのiterationでデータを移行することにしました。各iterationで、チームは構築中の新機能を支えるために追加すべきデータを考え、live systemからその追加データを移すようdata migration systemを更新しました。

こうしたことではよくあるように、実際には人々が恐れていたほど不可能ではありませんでした。結果としてリスクとストレスが下がり、この選択には価値がありました。チームは明らかな利点を評価しました。それは、本番直前に慌てふためく事態が明確に少なかった、ということです。

しかし最も興味深い利点は、彼らが予想していなかったものでした。incremental migrationは、domain expertとのコミュニケーションを大きく改善しました。通常、domain expertとuse caseについて話すときは、架空のscenarioを作ります。incremental data migrationを使うことで、チームは実例を使う習慣を持つようになりました。そのほうがdomain expertにとってずっと関係づけやすかったのです。

さらに、開発側がdomain expertにbuildを見せるとき、その中にはlive dataのcopyが含まれていました。その結果、domain expertは最近出くわした難しいcaseで新システムがどう動くかを調べられました。特に扱いにくい状況は、test environmentへ簡単にcopyできました。

コミュニケーション改善がなくても、incremental migrationを行う価値はあります。実施するなら、**domain expertと実データを使って話せる機会**を活かす準備をしておくべきです。
