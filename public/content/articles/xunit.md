# Xunit

## 要約

Xunit は、ソフトウェア開発者の間で広く知られるようになった一連のテストフレームワーク群の総称です。名前は、最初に広く知られるようになった JUnit から派生しています。

その源流は Smalltalk にあり、Kent Beck が自動テストを日常的な開発の中心に置くために、小さく使いやすいテスト実行の仕組みを作っていたことにさかのぼります。JUnit はその考えを Java の世界へ広め、Extreme Programming や Test Driven Development の普及を支える重要な役割を果たしました。

## 読むときの観点

- Xunit は単一の製品名ではなく、JUnit から広がったテストフレームワークの系譜として読む。
- Smalltalk での素朴な実践が、JUnit を通じて多くの言語に移植されていった流れに注目する。
- 小さく単純な道具が、開発文化やテストへの態度を大きく変えた点を押さえる。

## 原文の翻訳

`Xunit` は、ソフトウェア開発者の間で広く知られるようになった一群のテストフレームワークに付けられた総称です。この名前は、その中で最初に広く知られた `JUnit` から派生しています。

こうしたフレームワークの起源は、実は Smalltalk にあります。Kent Beck は、ソフトウェア開発の中心に自動テストを置くことを強く支持していました。彼自身や彼の顧客がそれを実践できるように、ユニットテストを整理して実行するための簡単なフレームワークを作っていました。重視されていたのは、プログラマが普段使っている Smalltalk 環境の中でテストを定義し、テスト全体でも一部でも素早く実行できるようにすることでした。

Kent とその周囲の人たちは、Smalltalk IDE の中で、システムに変更を加えるたびにユニットテストを実行していました。編集とテストを素早く繰り返す開発サイクルです。

私はそのころ Kent と出会いました。自分でも同じようなものを作っていましたが、Kent のフレームワークには、ばかばかしいほどの単純さと、必要な機能だけがちょうどよく入っているという、よい組み合わせがありました。要するに、彼のほうが私よりうまく作っていたので、私はそれを使うことにしました。特に C3 プロジェクトでそのフレームワークを使い、そこで Ron Jeffries もそれに触れることになりました。

ここで「それ」と言っていますが、厳密には少し誤解を招く表現です。Kent Beck による単一の Smalltalk ユニットテストフレームワークがあったわけではありません。Kent は、人々が自分たちの環境を自分たちで制御することを望んでいました。そのため、各チームが自分たちでフレームワークを作ることを好んでいました。作るのに数時間しかかからず、そうすることでチームは自分たちの状況に合わせて気軽に変更できました。つまりそれは、実質的には `Seedwork` でした。

それでも、この考えは Smalltalk コミュニティの外ではまだ知られていませんでした。したがって、このアイデアをより広く広めた功績は JUnit にあると言ってよいでしょう。JUnit は、チューリッヒから 1997 年の OOPSLA が開かれるアトランタへ向かう飛行機の中で生まれました。Kent は Erich Gamma と同じ便に乗っていました。長いフライトで、二人の開発者がほかに何をするでしょうか。もちろんプログラミングです。JUnit の最初の版はそこで作られ、ペアプログラミングで進められ、しかもテストファーストで作られました。ある種のメタ循環的な楽しさがあります。私はその話を聞いてコピーを求めたので、最初期のアルファユーザーの一人になりました。

私は気に入らないところを自由に変え、その一部を Kent と Erich に送り返しました。もし、assertion message が Java の慣習どおり任意引数を末尾に置くのではなく、最初の引数になっている理由を誰のせいにすべきか気になっているなら、そのあたりに関係があります。

JUnit は、赤と緑のバーによる進行表示も導入しました。C3 では、テストがひとつでも失敗するとウィンドウ全体を赤にし、すべて通れば緑にしていました。統合作業をしているとき、中央のビルドマシン上のウィンドウを見れば状態がすぐ分かりました。JUnit はこれを進行バーとして導入し、その結果、ソフトウェア開発者の語彙に新しい言葉を加えました。

JUnit は急速に広まりました。そして、Extreme Programming と Test Driven Development の成長を支えるうえで不可欠な存在になりました。私はこの十年ほどで、テストに対する態度が大きく変わるのを見てきましたが、その中で JUnit は大きな役割を果たしたと思います。**小さく単純だったからこそ、人々は学びやすく、使いやすかった**のです。また、ほかの人たちが拡張したり、ツールに統合したりしやすい形でもありました。

JUnit が人気を得るにつれて、ほかの言語にも同じようなものが求められるようになりました。Michael Feathers が C++ 向けに `CppUnit` を作っていたことを覚えています。おそらく、それが最初期の移植のひとつだったでしょう。その後、大量の移植版が続き、ほとんどすべての言語に少なくともひとつは JUnit の移植版があるようになりました。Smalltalk にも、実際のフレームワークとして「逆移植」されることになったのは、ある意味では避けられなかったのだと思います。

移植版の出来には違いがあります。元の JUnit をほとんど行単位で作り直し、対象言語への配慮があまりないものもあります。初期の `NUnit` には、Visual Age for Java の特殊な扱いに由来する `isVAforJava` メソッドさえありました。一方で、より洗練されたものもあります。NUnit 2.0 は C# の attributes を活用している点を Anders Hejlsberg に評価されました。その経験は、Java が annotations を発展させるにつれて Java コミュニティや JUnit 自身にも戻ってきました。
