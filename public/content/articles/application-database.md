# Application Database

## 要約

Application Databaseは、ひとつのアプリケーションだけが制御しアクセスするデータベースです。単一のアプリケーションの都合に合わせてスキーマを設計できるため、Integration Databaseより具体的で理解しやすく、複雑さも抑えやすくなります。

このスタイルの大きな利点は、利用箇所が単一アプリケーションに閉じ込められているため変更しやすいことです。共有が必要な場合は、データベースを直接共有するのではなく、サービスや読み取り専用のReporting Databaseを通じて提供できます。

## 読むときの観点

- Application Databaseを、Integration Databaseとの対比で理解する。
- データベースの所有権が、変更容易性やスキーマ設計に与える影響を見る。
- 他アプリケーションとのデータ共有を、直接DB共有ではなくサービス提供として考える。
- SOAで語られるautonomous applicationの前提として読む。

## 原文の翻訳

Application Databaseという言葉を、単一のアプリケーションによって制御され、アクセスされるデータベースという意味で使います。これはIntegration Databaseと対になるものです。データベースにアクセスするのがひとつのアプリケーションだけなら、そのアプリケーションの要求を満たしやすいように、データベースを具体的に定義できます。その結果、Integration Databaseよりも理解しやすく、複雑さの少ないスキーマになることがよくあります。

他のアプリケーションとデータを共有するために、制御側のアプリケーションはサービスを提供できます。また、より広い範囲の読み取り専用利用のためにReporting Databaseを提供することもあります。

Application Databaseの大きな利点のひとつは、すべての利用が単一アプリケーションにカプセル化されているため、変更しやすいことです。**本番投入後であっても、evolutionary database designやdatabase refactoringを使って、Application Databaseの設計に大きな変更を加えられます。**

Application Databaseのスキーマは、通常、そのアプリケーションチーム自身が設計し管理するのが最もよい形です。多くの場合、経験豊富なデータベース専門家がアプリケーションチームの一員として加わります。このデータベース専門家は、データベースをアプリケーション全体の要求に近い状態に保つため、他のアプリケーション開発者と非常に密に働く必要があります。

Service Oriented Architectureについて議論するとき、autonomous applicationという言葉がよく使われます。この言葉は、そのアプリケーションのデータがApplication Databaseに保存されていることを示唆しているように見えます。
