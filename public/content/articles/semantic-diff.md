# Semantic Diff

## 要約

通常の diff は、2つの成果物の違いを文字列やバイナリの変更として示します。Semantic Diff は、単なる変更結果ではなく、その変更が何を意味するのかを理解する差分です。

## 読むときの観点

- テキスト差分が「何が変わったか」は示せても、「なぜそう変わったか」までは示しにくい点を見る。
- リファクタリングのように意味のある操作を、差分やマージの道具が理解できると何が変わるかを考える。
- バージョン管理ツールがプログラム構造を理解する可能性として読む。

## 原文の翻訳

多くのバージョン管理システムは、成果物のバージョン間の変更を使い、それを理解することに頼っています。これは、Unix でその変更を生成するコマンド名から、よく diff と呼ばれます。テキストファイルやバイナリファイルについては、よい diff やマージのアルゴリズムがあります。しかし、こうした diff の問題は、かなり単純だということです。2つの成果物のバージョンを見て、一方からもう一方へ移るための単純な方法を生成するだけです。

**Semantic Diff は、変更の結果だけでなく、変更の目的を理解します。**

たとえば、あるツールで Extract Method リファクタリングを実行してクラスを変更し、それが2つのバージョンの間の唯一の変更だったとします。現在のツールは、プログラムのテキスト上の変更を見ますが、私がリファクタリングを行ったことは知りません。そのため、2つのバージョンの diff を調べると、変更箇所は示せますが、リファクタリングが行われたことを強調するような形では示せません。これは、ツールが私の意図を実際に知っている場合に比べて、マージをより扱いにくくすることもあります。

この考え方には、一般に受け入れられている用語があるかもしれません。もしそうなら、ぜひ知らせてください。
