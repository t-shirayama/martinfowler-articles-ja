# The New Methodology

## 要約

The New Methodology は、2000年前後に広がり始めた軽量なソフトウェア開発方法論、後に agile と呼ばれる考え方を、従来型の重量級プロセスとの対比で説明した記事です。計画、文書、工程の統制を中心にした方法論に対し、新しい方法論は適応性、人間中心、短いフィードバック、動くソフトウェアを重視します。

この記事は、アジャイルを単なる「プロセスを減らす運動」としてではなく、ソフトウェア開発の不確実性に合った別の制御方法として描きます。予測可能性を高めるために詳細な計画を増やすのではなく、変化を前提にして、短いサイクルで学びながら進むことが重要だと述べています。

後半では、Extreme Programming、Scrum、Crystal、Context Driven Testing、Lean Development、Rational Unified Process など、当時の代表的な潮流を概観します。最後に、アジャイルは誰にでも自動的に合うものではないが、混沌とした code and fix よりは規律を与え、重量級プロセスより導入しやすい場合が多いと論じます。

## 読むときの観点

- この記事はアジャイル普及初期の文脈で書かれており、現在の定番用語とは少し違う表現がある。
- 「計画をしない」のではなく、不確実性の高い開発に合う形で計画と制御を組み替える話として読む。
- 方法論の軽さは、規律の欠如ではなく、チームが実際に守れるプロセスにするための設計だと捉える。
- 人間中心という主張を、開発者の好き勝手ではなく、ソフトウェア開発が人間の判断と協働に依存するという前提として読む。
- 各手法の説明は歴史的なスナップショットであり、現在の実践や名称とは異なる部分がある。

## 原文の翻訳

ソフトウェア開発の世界では長い間、方法論というと、分析、設計、実装、テストを順番に進め、大量の文書と承認で制御する重いプロセスを指すことが多くありました。こうした方法論は、計画を立て、計画から外れないように管理することを重視します。しかし実際のソフトウェア開発では、要求も技術理解も進行中に変わります。問題は、計画が不要かどうかではなく、変化が避けられない仕事をどのように制御するかです。

新しい方法論は、この不確実性を正面から扱います。将来を詳細に予測して固定するよりも、短いサイクルで動くソフトウェアを作り、そこから学び、進路を調整します。**適応性は、無計画ではなく別種の規律**です。変化を失敗と見なすのではなく、より正しい理解に近づくための情報として使います。

従来型の方法論は、ソフトウェア開発をある程度予測可能な工学作業として扱いがちでした。建築や土木のように、設計が先に完了し、その後に建設が行われるという比喩がよく使われます。しかしソフトウェアでは、設計と構築の境界が薄く、実装しながら設計を学ぶことが多くあります。コードを書くこと自体が設計活動であり、コンパイル、実行、テストを通じて新しい理解が得られます。

この見方では、プロセスの目的も変わります。詳細な事前仕様で変化を抑え込むのではなく、変化に耐えられるチームとコードを作ります。短い反復、継続的なフィードバック、顧客との密な協働、テスト、リファクタリング、頻繁な統合が重要になります。進捗は文書の完成ではなく、実際に動くソフトウェアで測られます。

新しい方法論は、人間中心でもあります。ソフトウェア開発の成果は、手順書だけではなく、関係者の会話、判断、学習、信頼に大きく左右されます。そのため、プロセスは人間を交換可能な部品として扱うのではなく、チームが状況に応じて判断できる余地を持つ必要があります。これは自由放任ではありません。むしろ、チームが自分たちの働き方を観察し、改善し続けることを求めます。

Extreme Programming は、この新しい方法論の代表例として紹介されます。XP は短い反復、顧客との協働、テスト、リファクタリング、継続的インテグレーション、ペアプログラミングなどを組み合わせ、アジャイルな原則を実際に遂行するための具体的な技術を与えました。初期のアジャイル論が理念に寄りがちだった中で、XP は「それをどう実現するか」に強い答えを出した点で大きな影響を持ちました。

Scrum は、主にプロジェクト管理の側面に焦点を当てます。開発を短い反復に分け、日々のミーティングを通じて状況を見える化し、チームがスプリントごとに成果を届けられるようにします。Scrum はエンジニアリング実践そのものには比較的踏み込まないため、XP の技術的実践と組み合わせて使われることが多いと説明されています。

Crystal は、チームの大きさや失敗の重大性に応じてプロセスを調整する一群の方法論です。すべてのチームに同じプロセスを当てはめるのではなく、必要最小限のプロセスで成功できる形を探ります。ここで重要なのは habitability、つまり開発者がそのプロセスの中で実際に生きて働けるかという観点です。効率だけでなく、人間が継続できることを重視します。

Context Driven Testing は、テストの世界における同様の動きとして紹介されます。従来のテスト観では、事前に書かれた仕様への適合を確認する役割が強調されがちでした。しかしアジャイルな環境では、仕様自体が学習によって変わります。テスターは固定された文書の番人ではなく、状況に応じてリスクを見つけ、チームの学習を助ける役割を担います。

Lean Development は、製造業のリーン生産、とくにトヨタ生産方式から着想を得た考え方です。著者は安易な類推には慎重であるべきだとしつつも、リーンの考え方がアジャイルに多くの有用な発想をもたらしたと述べます。無駄の削減、流れの改善、学習、価値への集中といった観点は、ソフトウェア開発にも強い示唆を与えます。

Rational Unified Process は、オブジェクト指向コミュニティから生まれたよく知られたプロセスです。RUP は単一の手順というより、多くの実践を含むプロセスフレームワークです。そのため、チームは自分たちのプロジェクトに合う development case を定義する必要があります。ユースケース駆動、反復型、アーキテクチャ中心という特徴を持ちますが、非常に柔軟であるがゆえに、厳格なウォーターフォールからアジャイルに近い運用まで、何でも RUP と呼べてしまう危うさも指摘されています。

アジャイルを採用すべきかどうかは、単純な問いではありません。著者は、アジャイルは誰にでも無条件に合うわけではないとしつつ、現在の多くの現場で見られる code and fix よりは規律を与え、重量級プロセスよりは導入しやすい場合が多いと見ています。軽量であることは、プロセスに慣れていない組織にとって重要な利点になります。

導入の出発点は、アジャイルを試すのに適したプロジェクトを選ぶことです。アジャイルは根本的に人間中心なので、まずはその働き方を試したいチームから始める必要があります。嫌がる人々にアジャイルを押しつけることは、アジャイルの考え方そのものと矛盾します。顧客側にも協働の意思があることが望ましいですが、最初は消極的だった顧客が、数か月の経験を通じて考えを変えることもあります。

この記事の結論は、アジャイルが万能薬だということではありません。重要なのは、ソフトウェア開発が予測困難で、人間の協働に大きく依存する活動であることを認め、その性質に合ったプロセスを選ぶことです。新しい方法論は、重いプロセスを単に小さくしたものではなく、変化、学習、人間、動くソフトウェアを中心に据えた別の開発観を示しています。
