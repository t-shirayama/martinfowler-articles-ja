# Yagni

## 要約

Yagni は「You Aren't Gonna Need It」の略で、将来必要になると推測している機能を、今は作らないという考え方です。

将来のための機能は、不要になる可能性があるだけでなく、価値提供の遅れ、保守する複雑さ、後で間違った形だったと気づく技術的負債を生みます。
ただし Yagni は、コードベースを健康に保つ努力を否定するものではありません。むしろ、変更しやすいコードがあるからこそ成り立つ実践です。

## 読むときの観点

- 「将来必要そうな機能」と「変更しやすくするための設計努力」を区別する。
- presumptive feature が持つ、構築コスト、遅延コスト、保有コスト、修復コストを見る。
- Yagni はリファクタリングや self-testing code と対立しない点に注目する。
- 小さな抽象化や拡張ポイントにも、Yagni の判断が積み重なることを読む。

## 原文の翻訳

Yagni はもともと、「You Aren't Gonna Need It」を表す頭字語です。これは Extreme Programming から来たマントラで、アジャイルなソフトウェアチームで一般的によく使われます。将来ソフトウェアに必要になると私たちが推測している何らかの能力は、今は作るべきではない、なぜなら「それは必要にならない」からだ、という主張です。

Yagni は、XP の Simple Design という実践を指すための言葉です。これは『The White Book』初版に由来し、第2版では関連する考え方として「incremental design」に言及しています。XP の多くの要素と同じように、これは90年代後半に広く信じられていたソフトウェアエンジニアリングの原則のいくつかと鋭く対照をなしていました。当時は、ソフトウェア開発を慎重に事前計画することが強く推し進められていました。

この言い回しの起源は、C3 プロジェクトでの Kent Beck と Chet Hendrickson の初期の会話です。Chet は、システムがまもなく必要とする一連の能力を Kent に持ちかけました。Kent はその一つひとつに対して「you aren't going to need it」と答えました。Chet は学びが早く、すぐに yagni を適用する機会を見つける能力で知られるようになりました。「yagni」は頭字語として生まれましたが、私は今では普通の単語として私たちの語彙に入ったと感じているので、大文字は使いません。

私が、Minas Tirith で海運業向けの保険を売るスタートアップと働いていると想像してみましょう。彼らのソフトウェアシステムは、主に二つのコンポーネントに分かれています。一つは価格設定、もう一つは販売です。依存関係のため、関連する価格設定ソフトウェアが完成するまでは、販売ソフトウェアを有用に作ることができません。

現在、チームは価格設定コンポーネントを更新し、嵐のリスクを扱えるようにしています。彼らは6か月後には、海賊行為のリスクに対する価格設定にも対応する必要があると知っています。今ちょうど価格設定エンジンに取り組んでいるため、海賊行為の価格設定のための presumptive feature を今作ることを検討します。そうすれば、販売ソフトウェアに取りかかる前に価格設定サービスが完成するからです。

この記事では、まだ利用可能にされていない機能を支えるコードを、presumptive feature と呼びます。

Yagni はこれに反対します。海賊行為の価格設定は6か月間必要ないのだから、必要になるまで作るべきではない、と言います。したがって、このソフトウェアを作るのに2か月かかると思うなら、あと4か月は着手すべきではありません。もちろん、スケジュールリスクや販売コンポーネントの更新に対するバッファはここでは脇に置いています。

Yagni を支持する第一の論拠は、今はこの presumptive feature が必要だと思っていても、私たちが間違っている可能性が高いということです。結局のところ、アジャイル手法の文脈には、変化する要求を歓迎するという受け入れがあります。計画駆動型の要求分析の達人なら、これは要求分析が十分でなかったからであり、もっと時間と労力をかけるべきだったと反論するかもしれません。

私は、必要なものを事前に見極めることがどれほど難しく高くつくかを指摘して反論します。仮にそれができたとしても、Gondor 海軍が海賊を一掃してしまい、ビジネスモデル全体が崩れるような予想外の出来事に見舞われることはあります。

この場合、presumptive feature には明らかなコストがあります。構築コストです。今や役に立たなくなった機能を分析し、プログラミングし、テストするために費やしたすべての努力です。

しかし、必要性に対する私たちの理解が完全に正しく、Gondor 海軍が海賊を一掃しなかったとしましょう。この幸せな場合でさえ、presumptive feature を作ることには二つの深刻なコストがあります。第一のコストは、価値提供の遅れです。海賊行為の価格設定ソフトウェアに労力を使ったために、別の機能を作れませんでした。

代わりに、気象リスク向けの販売ソフトウェアを作ることに力を使っていれば、嵐リスクの機能を完全な形で本番に出し、2か月早く収益を生み出せたはずです。Presumptive feature によるこの遅延コストは、嵐保険から得られたはずの2か月分の収益です。

人々が presumptive feature を作る一般的な理由は、後で作るより今作ったほうが安いと考えるからです。しかし、そのコスト比較は、少なくとも遅延コストと比べて行わなければなりません。できれば、不要な機能を作っている確率も織り込むべきです。その確率は少なくとも 3分の2 です。

この 3分の2 という数字は Kohavi らによるものです。彼らは Microsoft の製品で作られ、デプロイされた機能の価値を分析し、慎重な事前分析があっても、改善しようとしていた指標を実際に改善したものは 3分の1 だけだったと見出しました。

多くの場合、人々は「今作る」ことと「後で作る」ことの比較コストを十分に考え抜きません。私がこの状況で開発者をメンタリングするときに使う一つの方法は、後で必要になったときにその能力を導入するには、どんなリファクタリングが必要になるかを想像してもらうことです。しばしば、その思考実験だけで、後で追加しても大きく高くつくわけではないと納得してもらえます。

そうした想像から生まれる別の結果として、今簡単にでき、複雑さをほとんど増やさず、後のコストを大きく下げるものを追加することがあります。エラーメッセージをインラインのリテラルではなく lookup table にするのは、その例です。単純でありながら、後で翻訳対応をしやすくします。

Jeremy Miller は、使われたことのない拡張ポイントは単なる無駄な努力ではなく、邪魔になる可能性も高い、と注意を促しています。

遅延コストは、成功した presumptive feature が課すコストの一つです。しかし、もう一つに保有コストがあります。Presumptive feature のコードはソフトウェアに複雑さを加えます。この複雑さによって、そのソフトウェアを変更しデバッグするのが難しくなり、ほかの機能のコストが増えます。海賊価格設定機能がソフトウェアに余分な複雑さを持ち込むことで、嵐保険の販売コンポーネントを作るのに数週間余計にかかるかもしれません。

その2週間は二重に効きます。機能を作る追加コストに加え、本番に出すのに時間がかかったことによる追加の遅延コストです。海賊保険ソフトウェアが役に立ち始めるまでに作られるすべての機能に対して、私たちは保有コストを負います。もし海賊価格設定ソフトウェアが一度も必要にならなければ、それを削除するまでに作られるすべての機能に対して保有コストを負い、さらに削除コストも負うことになります。

ここまで私は、presumptive feature を二つの分類、つまり成功したものと失敗したものに分けてきました。当然ながら、実際にはそこには連続的な幅があります。そしてその幅の中で、強調しておく価値のある一点があります。正しい機能を間違った形で作ってしまう場合です。開発チームは、利用者についてもコードベースについても、常に学んでいます。使っているツールについても学び、そのツールも定期的にアップグレードされます。コードがどのように協調して動くかについても学びます。

これらすべてが意味するのは、6か月前にコード化した機能が、今ならそうすべきだとわかっているやり方で作られていなかった、と気づくことがよくあるということです。その場合、Technical Debt を蓄積したことになり、その機能を修復するコストや、その難しさを回避し続ける継続的なコストを考えなければなりません。

したがって、presumptive feature には三つの分類があり、yagni を無視したときには四種類のコストが生じることになります。

私の保険の例は、比較的利用者に見える機能について話しています。しかし同じ議論は、将来の柔軟性を支える抽象化にも当てはまります。嵐リスク計算機を作るとき、後で海賊行為やその他のリスクを支えるために、抽象化やパラメータ化を今入れようと考えるかもしれません。Yagni は、それをしないように言います。他の価格設定機能が必要にならないかもしれませんし、必要になったとしても、そのとき実際に学ぶことと、今考えている必要な抽象化は一致しないかもしれないからです。

これは、すべての抽象化を避けろという意味ではありません。しかし、現在の要求に対するコードの理解を難しくする抽象化は、**有罪と推定される**という意味です。

Yagni は大きな機能で最も目立ちますが、小さなことでもより頻繁に見られます。最近、私はコード行の一部をハイライトできるコードを書きました。このために、ハイライトするコードを正規表現で指定できるようにしています。ここで見える問題の一つは、正規表現全体がハイライトされるため、ハイライトしたい範囲より大きな部分にマッチさせる必要がある場合に対応できないことです。

正規表現の中にグループを使い、グループが存在する場合にはそのグループだけをハイライトするようにすれば解決できると思っています。しかし、ハイライト対象より大きな範囲にマッチする正規表現をまだ使う必要がありません。だから、この場合に対応するようハイライト用のコードを拡張していません。実際に必要になるまで、そうしないつもりです。同じ理由で、実際に使う準備ができるまで、フィールドやメソッドも追加しません。

このような小さな yagni の判断は、プロジェクト計画のレーダーには映りません。開発者として、まもなく必要になると確信している抽象化を追加するために1時間を使うのは簡単です。しかし、上で述べたすべての論拠はそこにも当てはまります。そして多くの小さな yagni の判断は、コードベースの複雑さを大きく減らしつつ、より緊急に必要な機能の提供を速めることにつながります。

Yagni がなぜ重要なのかを理解したところで、yagni についてよくある混乱を掘り下げられます。Yagni が適用されるのは、presumptive feature を支えるためにソフトウェアへ組み込まれる能力だけです。ソフトウェアを変更しやすくする努力には適用されません。Yagni は、コードが変更しやすい場合にだけ実行可能な戦略です。したがって、リファクタリングに努力を費やすことは yagni 違反ではありません。リファクタリングはコードをよりしなやかにするからです。

Self Testing Code や Continuous Delivery のような実践にも、同じ理屈が当てはまります。これらは evolutionary design の enabling practices です。それらがなければ、yagni は有益な実践から呪いへ変わります。しかし、しなやかなコードベースがあるなら、yagni はその柔軟性を強化します。Yagni には、evolutionary design によって可能になり、同時にそれを可能にもするという興味深い性質があります。

Yagni は、コードベースの健全性をないがしろにする正当化ではありません。**Yagni は変更しやすいコードを必要とし、またそれを可能にします**。

また私は、yagni が適用されるのは、後になるまで利用しない余分な複雑さを今導入する場合だけだと主張します。将来の必要のために何かをしても、それが実際にはソフトウェアの複雑さを増やさないなら、yagni を持ち出す理由はありません。

とはいえ、yagni を適用したために問題が起き、以前の変更ならずっと安く済んだはずの高価な変更に直面する場合もあります。難しいのは、そうした場合を事前に見つけるのが難しく、yagni が労力を節約した場合よりも記憶に残りやすいことです。これは availability bias の帰結です。私の感覚では、yagni の失敗は比較的まれであり、そのコストは yagni が成功したときの利益によって容易に上回られます。
