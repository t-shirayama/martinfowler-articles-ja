# Catalog of Patterns of Distributed Systems

## 要約

分散システムでは、複数のデータコピーを同期しながら、ノード障害やネットワーク遅延による不整合にも対処しなければなりません。
この記事は、データ保存、メッセージング、システム管理、計算基盤などで繰り返し現れる問題と解決策を、分散システムのパターン集として整理したカタログです。

このページにあるのは、各パターンの短い概要と、O'Reilly のオンライン書籍にある該当章へのリンクです。
個々のパターンの詳細な説明ではなく、全体像をつかむための目次として読むと役立ちます。

## 読むときの観点

- パターンを単体で覚えるより、複製、順序づけ、合意、ログ、パーティション、リクエスト処理のどの問題に効くかを見る。
- 一貫性を強める工夫と、レイテンシやスループットを改善する工夫の違いを意識する。
- 「どのノードが何を知っているか」「どの順序で判断できるか」を追う。
- 書籍本文への入口として、知らないパターン名を拾っていく。

## 原文の翻訳

分散システムをプログラムすることには、特有の難しさがあります。
多くの場合、データのコピーを複数持ち、それらを同期させる必要があります。
しかし、処理ノードが常に正しく動くとは期待できず、ネットワーク遅延も容易に不整合を引き起こします。
それにもかかわらず、多くの組織は、データ保存、メッセージング、システム管理、計算能力を担う中核的な分散ソフトウェアに依存しています。
こうしたシステムは共通の問題に直面し、**似た解決策**でそれらを解いています。

2020年に、私はこれらの解決策をパターンとして集め始め、作成しながらこのサイトで公開してきました。
2023年には、それらが『Patterns of Distributed Systems』という書籍として出版されました。
このサイトでは現在、各パターンの短い要約と、O'Reilly.com のオンライン電子書籍にある該当章へのディープリンクを掲載しています。

### Clock-Bound Wait

クラスタ内のノード間にある時刻の不確かさを覆うだけの時間を、値の読み書きの前に待ちます。
そうすることで、クラスタ内のノード間で値を正しく順序づけられるようにします。

### Consistent Core

より強い一貫性を提供する小さなクラスタを維持します。
これにより、大きなデータクラスタは、クォーラムにもとづくアルゴリズムを実装しなくてもサーバ活動を調整できます。

### Emergent Leader

クラスタ内での古さにもとづいてノードを順序づけます。
これにより、ノードは明示的な選挙を実行しなくてもリーダーを選べます。

### Fixed Partitions

パーティション数を固定しておきます。
クラスタのサイズが変わっても、データからパーティションへの対応が変わらないようにするためです。

### Follower Reads

読み取りリクエストをフォロワーから提供します。
これにより、スループットを高め、レイテンシを下げます。

### Generation Clock

サーバの世代を示す、単調増加する数値です。

### Gossip Dissemination

ランダムに選んだノードを使って情報を伝播させます。
ネットワークをあふれさせることなく、クラスタ内のすべてのノードへ情報が届くようにするためです。

### HeartBeat

他のすべてのサーバへ定期的にメッセージを送ることで、サーバが利用可能であることを示します。

### High-Water Mark

最後に成功した複製を示す、write-ahead log 内のインデックスです。

### Hybrid Clock

システム時刻と論理時刻を組み合わせます。
これにより、バージョンを日付時刻として扱いながら、順序づけられるようにします。

### Idempotent Receiver

クライアントからのリクエストを一意に識別します。
クライアントが再試行したとき、重複したリクエストを無視できるようにするためです。

### Key-Range Partitions

ソートされたキー範囲でデータを分割します。
範囲クエリを効率よく扱うためです。

### Lamport Clock

値のバージョンとして論理タイムスタンプを使います。
これにより、サーバ間で値を順序づけられるようにします。

### Leader and Followers

サーバ群にまたがる複製を調整するために、単一のサーバを置きます。

### Lease

クラスタノードが活動を調整するために、期限付きのリースを使います。

### Low-Water Mark

write-ahead log のどの部分を破棄できるかを示すインデックスです。

### Majority Quorum

2つのサーバグループが独立して判断してしまうのを避けるため、すべての判断に過半数を要求します。

### Paxos

ノードが切断される場合でも安全に合意へ到達するために、2つの合意形成フェーズを使います。

### Replicated Log

すべてのクラスタノードへ複製される write-ahead log を使って、複数ノードの状態を同期させます。

### Request Batch

複数のリクエストをまとめ、ネットワークを最適に利用します。

### Request Pipeline

前のリクエストの応答を待たずに、同じ接続上で複数のリクエストを送ります。
これによりレイテンシを改善します。

### Request Waiting List

他のクラスタノードからの応答にもとづいて、応答条件が満たされた後に返答が必要となるクライアントリクエストを追跡します。

### Segmented Log

ログを単一の大きなファイルにせず、複数の小さなファイルへ分割します。
運用を容易にするためです。

### Single-Socket Channel

単一の TCP 接続を使って、サーバへ送られるリクエストの順序を保ちます。

### Singular Update Queue

リクエストを非同期に処理する単一スレッドを使います。
呼び出し元をブロックせずに順序を維持するためです。

### State Watch

サーバ上の特定の値が変化したときに、クライアントへ通知します。

### Two-Phase Commit

複数ノード上のリソースを、ひとつの原子的な操作として更新します。

### Version Vector

クラスタノードごとに1つのカウンタを持つ一覧を維持します。
並行更新を検出するためです。

### Versioned Value

値への更新をすべて新しいバージョンとして保存します。
過去の値を読めるようにするためです。

### Write-Ahead Log

ストレージのデータ構造をディスクへフラッシュしなくても耐久性を保証します。
すべての状態変更を、追記専用ログへのコマンドとして永続化するためです。
