# Using ChatGPT as a technical writing assistant

## 要約

この記事は、Mike Mason が ChatGPT を技術文書作成の補助として試した経験をまとめたものです。対象は、Thoughtworks Technology Radar の blip、技術業界のマクロトレンド記事、非技術者向けの説明文などです。機密情報や顧客情報を扱わない範囲に限定し、公開前提の文章作成でどこまで役立つかを検証しています。

結論は、ChatGPT は初稿作りや追加観点の提案には有用だが、正確さ、深さ、文体の抑制には人間の専門的な編集が必要だというものです。特に、AI が楽観的で宣伝文句のような文章を出しやすいこと、文脈の制限で前提を失うこと、もっと詳しくするよう頼んでも同じ内容に留まりがちなことが問題として挙げられています。

## 読むときの観点

- ChatGPT を「完成原稿を出す道具」ではなく、初稿と発想の補助として見ている点に注目する。
- Webhook や Giskard の例では、AI の出力がそれらしく見えても専門家レビューが不可欠であることを見る。
- 非技術者向けの説明では、文体変換や要約がかなり有効に働く場面がある。
- AI の文章は前向きで一般論に寄りやすく、批判的な文脈では編集が必要になる。
- 最終的な責任は著者にあり、AI の出力を自分の名前で出せる品質まで見直す必要がある。

## 原文の翻訳

著者は、専門的な場面で ChatGPT を使うにあたり、まず知的財産や顧客情報の扱いに注意した。上司と法務担当者に相談し、顧客に関わる用途では使わず、公開を前提とした技術文書の作成に絞った。試したかったのは、Technology Radar の短い blip、技術業界のマクロトレンド記事、そして Radar の内容をビジネス読者にも読める形へ書き換える作業だった。

Radar の blip では、ChatGPT はある程度それらしい文章を出した。たとえば、ChatOps の webhook を設定値ではなく credential として扱うべきだという注意喚起では、背景説明や問題の整理は役に立った。一方で、最後に不要な行動喚起を加えるなど、Radar の文体としては外す部分もあった。Giskard の blip では、さらに問題がはっきりした。出力は自信ありげで前向きすぎ、機械学習の実務に詳しい人から見ると理解の浅い文章に見えた。**それらしい文章と正しい文章は同じではない**。

マクロトレンド記事の支援では、ChatGPT はメモをもとに数段落の文章を作ることができた。ただし、全体に明るく大げさな調子になりやすく、導入、説明、まとめという定型構造を毎回作りがちだった。そのため、冒頭や結論を削ったり、著者自身の言葉に近づけたりする編集が必要だった。

非技術者向けに Radar の内容を書き換える用途では、かなり良い結果が得られた。アクセシビリティのような技術的な話題を、「それは何か」と「ビジネス上の文脈」に分けて説明させると、読みやすい文章が出た。ここでは、複雑な内容を短く整理する能力が時間の節約につながった。

著者が見た ChatGPT の強みは、関連する知識や似た議論をすばやく呼び出せること、初稿を作って作業の足場にできること、足りない節や観点を提案できることだった。弱みは、正確さと深さに限界があること、楽観的すぎる文体になりやすいこと、文脈を忘れること、同じ出力の周辺から抜け出せなくなることだった。

出力を良くするには、文脈と文体を明示するプロンプトを用意し、小さな単位で反復的に作業するのがよい。長い文章を一気に任せるより、構成を考える会話と各セクションを書く会話を分けたほうが扱いやすい。また、AI を単なる部下のように扱うより、協働相手として丁寧に文脈を与えるほうが良い出力を得やすい。

一方で、AI に頼りすぎる危険もある。便利な補助が急に使えなくなると、思った以上に作業が止まることがある。AI の出力は、事実、含意、抜け落ちたニュアンスを必ず確認しなければならない。編集には時間がかかるので、生産性向上の効果を過大評価しないほうがよい。最終的には、記事に名前を載せる著者自身が責任を持つ。

著者は、ChatGPT を使った実験を成功と見ている。ただし、単純に時間が大きく節約されたというより、出力を出発点にして作業を前に進められたという意味での成功である。LLM は、ストーリー作成、受け入れテスト、ドキュメント作成など、ソフトウェア開発周辺の多くの作業で初稿や追加観点を出す助けになりうる。とはいえ、**人間の専門家が出力を読み、判断し、責任を引き受けること**は残る。
