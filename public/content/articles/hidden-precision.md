# Hidden Precision

## 要約

Hidden Precision は、データがこちらの想定より細かい精度を持っていることで起きるバグを指します。
日付だけを扱っているつもりが実際には時刻まで含む instant を扱っていたり、浮動小数点数が見えない誤差を持っていたりすると、比較や範囲判定が直感と違う結果になります。

この記事は、日付・時刻のモデル化や金額の表現では、見えない精度に注意し、用途に合った型や表現を選ぶべきだと説明しています。

## 読むときの観点

- 「より精密ならよい」とは限らず、期待していない精度がバグになる点を見る。
- 日付と instant を混同したとき、境界条件がどう崩れるかを確認する。
- 浮動小数点数で金額やしきい値比較を扱う危険を読む。
- 型名や生成関数の名前が、モデルの誤解を招いていないか考える。

## 原文の翻訳

データを扱っていると、そのデータが私の想定よりも精密であることがあります。精密であることはよいことだから、精度が高いほどよいのではないかと思うかもしれません。しかし、**隠れた精度は微妙なバグにつながる**ことがあります。

```javascript
const validityStart = new Date("2016-10-01");   // JavaScript
const validityEnd = new Date("2016-11-08");
const isWithinValidity = aDate => (aDate >= validityStart && aDate <= validityEnd);
const applicationTime = new Date("2016-11-08 08:00");

assert.notOk(isWithinValidity(applicationTime));  // 望んでいた結果ではない
```

上のコードで起きているのは、開始日と終了日を指定して、両端を含む日付範囲を作ろうとした、ということです。しかし実際には日付を指定したのではなく、時点を指定していました。そのため、終了日を 11 月 8 日として印を付けているのではなく、11 月 8 日の 00:00 を終了時刻として印を付けていることになります。その結果、11 月 8 日の中の時刻は、真夜中を除いて、本来は含めたい日付範囲の外に出てしまいます。

隠れた精度は日付でよく起きる問題です。残念なことに、この例のように実際には instant を返す日付生成関数がよくあるからです。これは名前付けのまずさの例であり、日付と時刻のモデル化全般のまずさの例でもあります。

日付は隠れた精度の問題を示すよい例ですが、もうひとつの原因は浮動小数点数です。

```javascript
const tenCharges = [
  0.10, 0.10, 0.10, 0.10, 0.10,
  0.10, 0.10, 0.10, 0.10, 0.10,
];
const discountThreshold = 1.00;
const totalCharge = tenCharges.reduce((acc, each) => acc += each);
assert.ok(totalCharge < discountThreshold);   // 望んでいた結果ではない
```

私がいま実行したところ、ログ出力では `totalCharge` は `0.9999999999999999` でした。これは、浮動小数点数が多くの値を正確には表せないためで、その小さく見えない精度が都合の悪いときに現れることがあるからです。

ここから得られる結論のひとつは、**金額を浮動小数点数で表すことには非常に慎重であるべき**だということです。セントのような小数の通貨単位があるなら、通常はその最小単位を整数で表すのが最善です。たとえば €5.00 を 500 として、できれば money type の中で表します。より一般的な結論は、比較に関して浮動小数点数は扱いにくい、ということです。テストフレームワークの assert が比較の精度を常に持っているのもそのためです。
