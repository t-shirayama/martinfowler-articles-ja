# Foundation Platform

## 要約

Foundation Platform は、その上にアプリケーションを作る前に、先に共通基盤として作られるプラットフォームです。理論上は安定した API を先に用意できるように見えます。

しかし Fowler は、この進め方は実践ではうまくいかないことが多いと述べています。実際のアプリケーションの必要を事前に理解するのは難しく、必要以上の機能や、実際には合わない機能を作り込みがちだからです。

## 読むときの観点

- アプリケーションより先にプラットフォームを作るリスクを見る。
- 安定 API を早く作りたいという理屈と、実際の要求理解の難しさを分けて考える。
- 必要以上の機能を持つ基盤になりやすい点に注意する。
- Harvested Platform との対比で読む。

## 原文の翻訳

Foundation Platform とは、その上に作られるどのアプリケーションよりも前に構築されるプラットフォームです。考え方としては、プラットフォームを必要とするさまざまなアプリケーションのニーズを分析し、それからプラットフォームを作ります。プラットフォームが完成したら、その上にアプリケーションを作ります。要点は、アプリケーションの作業を始める前に、プラットフォームは本当に安定した API を持っている必要がある、ということです。そうでなければ、プラットフォームへの変更はアプリケーションへの波及効果のために管理しにくくなります。

これは理論上はもっともらしく聞こえますが、私が見てきた限り、実践ではいつもうまくいきません。問題は、プラットフォームの本当のニーズを理解するのがとても難しいことです。その結果、プラットフォームは実際に必要なものよりはるかに多くの機能を持つことになります。多くの場合、その機能はアプリケーションが本当に必要としているものとうまく合いません。

これを Harvested Platform と対比してください。

私はもともとこの項目を FoundationFramework という名前で公開しました。しかし私たちの語彙は変化し、以前なら `framework` と呼んでいたところで、今では `platform` という言葉を使っています。とはいえ、**核になる考え方は同じ**です。
