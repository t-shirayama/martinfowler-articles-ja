# In Memory Test Database

## 要約

In Memory Test Databaseは、ディスクを使わず主記憶上で動くデータベースです。プロセスの開始とともに作られ、プロセスの終了とともに消えるembedded databaseとして使われることが多く、特にテストの高速化に有効です。

この記事は、SQL互換のインメモリDBを使う方法と、Repositoryでデータアクセスを抽象化してメモリ上のデータ構造に差し替える方法を対比しています。テスト速度だけでなく、イベントログをsystem of recordにしてメモリ上の状態を再構築するような設計にも可能性があると述べています。

## 読むときの観点

- インメモリDBは永続化しないからこそ、テストで強い効果を発揮する。
- SQL互換DBを使う方法は便利だが、本番DBとの差異に注意が必要。
- Repositoryを使う方法は、SQL以外のデータ源も同じ形で差し替えやすい。
- インメモリDBの利点が、Repository不在という設計上の匂いを隠していないかを考える。

## 原文の翻訳

インメモリデータベースとは、ディスクに触れず、完全に主記憶上で動作するデータベースです。多くの場合、embedded databaseとして動きます。つまり、プロセスの開始時に作られ、そのプロセス内に埋め込まれて動作し、プロセスの終了時に破棄されます。

多くの人はデータベースを、大きくてディスク中心のものだと考えます。しかし、インメモリデータベースの小さいながら活発な世界も存在します。何らかの管理されたデータへ高速にアクセスする必要があり、そのデータが変化しない、あるいは再構築できるため永続化を必要としないアプリケーションがあります。ルーターのrouting tableやEventPosterを想像するとよいでしょう。

伝統的なデータベースシステムを開発している人にとっても、インメモリデータベースは役に立ちます。特にテストでは有用です。enterprise applicationを開発していると、テストスイートの実行時にデータベースへアクセスするテストが大きな時間の浪費になることがあります。インメモリデータベースへ切り替えると、桁違いの効果でbuild timeを劇的に短縮できることがあります。多くのThoughtWorkerはgreen barをしばらく見ないと落ち着かなくなるので、これは大きな違いを生みます。

テスト用にインメモリデータベースへ向かう道は、大きく2つあるようです。1つ目はSQLのインメモリデータベースライブラリを使うことです。Javaの世界ではHSQLDBがよく使われているようです。ほかではSQLiteやFirebirdの名前も挙がります。これらのよいところは、通常のSQLで問い合わせできることです。ただし、対象となる本番データベースとまったく同じ方言や機能を持っているとは限りません。

似たことは、ファイルベースのデータベースをram disk上で動かすことでも実現できます。これにより、テスト環境と本番環境の配置をより近く保てます。

もう1つの道は、すべてのデータベースアクセスをRepositoryの背後に抽象化することです。そうすれば、データベースを通常のメモリ上データ構造へ差し替えられます。多くの場合、オブジェクトグラフへの入口に対するhash tableの集まりだけで十分です。Repositoryアプローチの強みの1つは、SQLではないデータソースにも一貫したアクセス方法を与え、stub化できることです。つまりobject-relational mapping systemもRepositoryの内側に隠されます。

実際、SQLインメモリデータベースの利用を積極的に嫌う人もいます。そうした人たちは、SQLやobject-relational mapperのコードがdomain modelのあちこちへ広がることを助長すると考えています。SQLをインメモリで動かすと、遅いアクセスの痛みの多くは取り除けますが、**Repositoryがないという匂いを消臭剤で覆い隠す**ように働くこともあります。

ここまではテストが主な動機ですが、インメモリデータベースには今後さらに多くの可能性があると思います。現在のメモリ容量なら、多くのapplication databaseをメモリへ読み込めます。アプリケーション状態へのすべての変更をevent logとして保持する方式を使えば、インメモリデータベースを、ログ適用結果のcacheとして扱えます。必要に応じて再構築し、snapshotを取れるのです。

このようなスタイルは、多数のreaderと少数のwriterがいる場合に、非常にscalableで高性能になりえます。私は、高性能アプリケーションにインメモリデータベースを使っている事例にいくつか出会いました。ここでの違いは、テストではopen-sourceのものが好まれる一方で、こうした経験はnicheな商用データベースで語られることが多い点です。

Prevaylerは、この種のアプローチで大きな注目を集めました。試した知人たちは、インメモリオブジェクトとの強い結合とmigration toolの不足が深刻な問題を引き起こしたと言っていました。しかし、変更ログをsystem of recordとして永続化するアプローチは、将来探究する価値のある豊かな領域だと思います。

### フォローアップ

このページを書いた後、興味深いメールをいくつか受け取ったので、その論点を共有します。

ある読者は、SQLが得意でオブジェクトがそれほど得意ではない作業にインメモリデータベースを使うのが好きだと言っていました。SQLがオブジェクトや手続き型コードよりもはるかに優雅に問題を解ける場合は確かにあります。ただし通常、SQLで考えることを好む開発者は少数派だと私は感じています。

同僚のSteve Sparksは、ある最近のプロジェクトについて教えてくれました。そのプロジェクトでは、テスト時に最初の呼び出しでlive databaseからデータを取得し、そのデータをファイルに保存してインメモリRepositoryの初期化に使っていました。以後の問い合わせはデータベースに行かなくて済みます。私はC3プロジェクトで初めてこのやり方を見ました。そこではSQL query stringをkeyにしたhash tableへデータを保持していました。値がなければDB2へ行き、結果をcacheしていました。

Steven Gravesは、私の最初の記事がインメモリデータベースの一般的な使い道を十分に扱っていなかったと指摘しました。その結果、私は一部を書き直し、記事のタイトルも変更しました。
