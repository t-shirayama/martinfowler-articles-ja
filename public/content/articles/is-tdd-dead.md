# Is TDD Dead?

## 要約

`Is TDD Dead?` は、David Heinemeier Hansson の「TDD is dead」という挑発的な問題提起をきっかけに、Kent Beck、David Heinemeier Hansson、Martin Fowler が TDD とソフトウェア設計について議論した連続対話です。

議論の中心は、TDD そのものの是非ではなく、自己テスト可能なコード、設計への影響、フィードバック、QA、テストのコスト、そして実践をどのように状況に合わせるかです。三者は立場の違いを保ちながらも、自己テスト可能なコードの価値、TDD が有効な文脈の存在、そして技法を盲信せず考えて使う必要性では大きく一致しています。

## 読むときの観点

- TDD と自己テスト可能なコードを区別して読む
- モックや過度な分離が本当に TDD に由来するのか、設計判断に由来するのかを見る
- テストは信頼を得る手段であり、設計やQA、運用フィードバックと組み合わせて考える
- 「TDDは死んだ」という表現を、技法の否定ではなく実践の再点検として読む

## 原文の翻訳

このページは、Kent Beck、David Heinemeier Hansson、Martin Fowler による一連の公開対話をまとめたものです。テーマは Test-Driven Development、つまり TDD と、それがソフトウェア設計に与える影響です。動画と音声に加えて、公開ページには各回の議事録が掲載されています。以下は、その公開議事録で確認できる内容に基づく日本語での整理です。

## どこから始まったか

この対話は、David の RailsConf keynote と、その後のブログ記事から始まりました。David は Rails コミュニティにおける TDD とユニットテストへの不満を表明し、最初の記事では「TDD is dead」と宣言しました。

その後、Martin が David の続きの記事に誤字修正を送り、David が講演や記事について Martin の考えを聞きたいと返しました。二人は Skype で一時間ほど議論し、David は Kent とも同じように話しました。Kent は三人で対話を続け、それを公開してはどうかと提案しました。

このシリーズには厳密な計画はありませんでした。おおまかには、30分ほどの対話を半ダース程度行うという考えでした。目的は、互いの経験と見方を理解することでした。初期の回では質問を受け付けず、最後の回でいくつか質問に答えています。

## 1. TDD と信頼

最初の回では、TDD の流れに対する三者の経験と、TDD と自己テスト可能なコードがしばしば混同されることが話題になります。

David は、TDD とユニットテストをめぐる三つの問題を挙げます。定義の混乱、モックを使ってアーキテクチャを駆動することで起きるテスト誘発の設計損傷、そして red/green/refactor のサイクルが自分にはうまく働かなかったことです。

Kent は、TDD の起源を Smalltalk での試行錯誤に求め、自分の性格には TDD がよく合ったと説明します。彼にとって、プログラマーは自分のコードが動くという信頼を持てるべきであり、TDD はそこへ到達するひとつの方法です。ただし唯一の方法ではありません。

David は、テストがないまま完了とは言えないという考えには賛成します。しかし、そこに至る手段として TDD が常に適切だとは考えていません。人によって頭の働き方は違い、向いている技法も言語も違うからです。彼が嫌うのは、TDD が自己テスト可能なコードから得られる信頼と混同されることです。

Kent は、TDD に向くコードでは楽しい流れに入れるが、そうでない部分もあると話します。TDD ではない部分でも、回帰テストと短いフィードバックループは使います。二つのスタイルを混ぜることに問題はなく、クラシックとジャズの両方を演奏するようなものだと見ています。

Martin は、ここには二つの用語上の混乱があると整理します。第一に、David の批判は「TDD では重いモックを使わなければならない」という前提に基づいているが、それは必須ではありません。第二に、自己テスト可能なコードと TDD は違います。**TDD は自己テスト可能なコードを得るためのひとつの方法であって、それ自体と同義ではありません**。

## 2. テスト誘発の設計損傷

二回目では、TDD が設計を傷つけることがあるのか、その損傷は本当に損傷なのか、そして設計が損なわれているかどうかをどう判断するのかが議論されます。

David は、多くのモックを使った TDD から生まれると彼が見ているアーキテクチャ例を取り上げます。アプリケーションの各層が分離され、コントローラを実際のモデル、データベース、リクエスト/レスポンスのサイクルから切り離してテストできます。David にとって重要なのは、その具体例そのものではなく、孤立したテストを容易にするために必要になる不要な間接化と複雑さです。

Kent は、テスト誘発の損傷を TDD のせいにするのは、車で悪い場所へ行って車を責めるようなものだと返します。問題は TDD ではなく、そうした間接化がどんな状況で価値を持ち、どんなコストを伴うのかを理解することだと考えます。David は、TDD に乗ると特定の方向へ促され、テストをひとつずつ足すうちに怪物のような構造へ進むことがあると反論します。

Kent は、それは一つひとつの設計判断の積み重ねだと見ます。TDD は設計に進化的な圧力を与えますが、テストが覆う粒度については人によって好みが違います。

議論は、凝集度と結合度、外部依存の扱い、インメモリの保存先とWebサービスを同じようには扱えないこと、境界はある程度漏れることへ広がります。Martin は、ここで本質的なのは TDD というより、Rails などの環境から隔離したいという欲求ではないかと見ます。Kent は、テストしにくさは設計上の洞察が必要な兆候であることが多いと述べます。

この回の結論は、次にプログラミング中のフィードバックをどう求めるか、そのトレードオフを扱うというものでした。

## 3. フィードバックとQA

三回目では、プログラミング中に得るフィードバックの種類と、開発者へのフィードバックを提供するQAの役割が議論されます。

Kent は、理想的にはプログラミング上の判断について即時で誤りのないフィードバックが欲しいと言います。しかしそれは不可能なので、どこまで妥協するかが問題になります。彼は、フィードバックの頻度、赤/緑の信号の正確さ、支払えるオーバーヘッド、ソフトウェアの寿命という制約を挙げます。

Martin は、フィードバックで知りたいことを三つに分けます。ソフトウェアが利用者にとって役に立っているか、何かを壊していないか、そしてコードベースが今後も素早く作れる健康さを保っているかです。

David は、TDD の成功がQAの軽視につながったと問題提起します。TDD を導入した多くの組織はQAをなくし、開発者が過信するようになったと見ています。古いQAとの敵対的な関係は壊れていましたが、振り子が逆に振れすぎたという見方です。高品質なソフトウェアを作るには、自分ではない誰かに試してもらうことが重要だと彼は言います。

Kent は、旧来のQAとの関係が機能不全だったことを認めつつ、問題は「何と比べるか」だと述べます。有効なQAがあるならQAなしは劣りますが、機能不全なQAとの関係よりはQAなしの方がよい場合もあります。Martin は、Thoughtworks では多くのプロジェクトにQAがいること、そして90年代からの大きな変化は敵対関係と手動のスクリプトテストを減らしたことだと補足します。

David は、開発者が顧客サポートやオンコールから切り離されることも問題だと言います。Kent も、オンコールは書かなかったテストを教えてくれるフィードバックループだと見ます。**テストの緑は有用な信号ですが、利用者や本番環境からのフィードバックを置き換えるものではありません**。

## 4. テストのコスト

四回目では、テストやTDDの不利な点、つまりテストしすぎることがあるのか、機能するコードよりテストを重んじすぎる問題があるのかが扱われます。

David は、トレードオフを語るには欠点を理解しなければならないと述べます。欠点がなければトレードオフは存在しないからです。彼は、TDD が人を特定の方向へ促すこと、そして過剰テストの危険を挙げます。

「本番コードを一行書く前に失敗するテストを書くべきだ」という考えは一見妥当に見えますが、場合によっては本番コード一行に対してテストコードが四行あるような状態を生みます。振る舞いを変えるとき、変更すべきコードが増えるわけです。

Kent は「場合による」と答えます。JUnit では厳格に test-first を行い、結果に満足していました。だから TDD が常に過剰テストを招くとは考えていません。同時に、あるテストがどんな独自のカバレッジを提供しているのか、いわば delta coverage を考えるべきだとも述べます。何も新しいものを買っていないテストは、コミュニケーション上の価値がない限り削除してよいのです。

Martin は、テストが足りない状態と多すぎる状態の目安を示します。自信を持ってコードを変更できないなら、テストが足りないか十分によくありません。一方で、コードを変えるたびに本番コードよりテストを変える方が大変だと感じるなら、多すぎる兆候です。目指すのはちょうどよい領域であり、それはチームがどんな間違いをしやすく、どんな間違いが問題になるかを経験から知ることで見えてきます。

後半では、テストが機能コードより重要視されすぎる問題が話題になります。David は、かつて文書がコードより重要だと考えられたように、今はテストが機能コードより大事だと扱われることを懸念します。Martin は、テストは目的のための手段であり、利用者を支えることよりテスト環境にエネルギーを注ぐのは悪い動きだと同意します。

設計品質は数値にしにくいため、テスト速度、カバレッジ、比率のような測りやすいものに注意が奪われます。しかし、そうした数値は低い優先度のものを過度に重要に見せることがあります。**テストは設計と利用者価値を支えるための手段であり、目的そのものではありません**。

## 5. 質問に答える

最後の回では、視聴者からの質問に答えます。TDD を使ったオープンソースの例、TDD の使い方を変える条件、経験の浅い開発者にとってのTDDなどが扱われます。

David は、TDDが設計損傷を起こした、あるいはうまく使われた実アプリケーションの良い例は少ないと述べます。オープンソースの貢献者は、公開フレームワークやライブラリの外で非公開のアプリケーションを扱うことが多いためです。そのため、議論はしばしば互いに違う文脈を前提に進み、実際よりも意見が食い違って見えます。

Kent は、JUnit はTDDを厳格に使ってうまくいった例だと言います。ただし、JUnit は明確なインターフェースがあり、TDD に向いた領域です。この議論で扱っているのは、もっと別種のアプリケーションです。

TDD が不要になる条件について、Kent は TDD が信頼を与えること、問題を小さく分けて進められることを挙げます。難しいからといって手放すつもりはありません。Martin は、TDD が時代遅れになるかどうかではなく、文脈によって適用可能性が違うことが重要だと述べます。自分のWebサイトのツールチェーンでは回帰テストを持ち、少しずつ進めているものの、TDD はあまり当てはまりません。一方、infodeck のコードでは TDD が有効な振る舞いも多くありました。

David も似た経験を語ります。TDD からテストに入門し、気に入ってすべてに適用しようとしましたが、MVC Webアプリケーションの多くの領域では気持ちよく使えないと徐々に気づきました。それでも、TDD をやめることは自己テスト可能なコードを諦めることではありません。彼にとって、TDD の価値はそこにありました。

Martin は、TDD やどんな技法でも、まず試し、使いすぎ、そこから自分に合う形へ落ち着くのがよいと言います。TDD は自己テスト可能なコードへの入口にもなります。

経験の浅い開発者については、Martin は TDD が小さな単位で進むことや、インターフェースと実装を分けて考えることを促すと述べます。ただし、よい設計には経験が必要であり、TDD がそれを保証するわけではありません。経験の浅い人は十分にリファクタリングしないことが多く、最適でない設計につながります。それでも、その人がTDDなしで作ったものと比べれば、よい出発点になることがあります。

David は、TDD が自分にとって補助輪として役立ったと認めます。一方で、議論がそこから十分に進まなかったことを問題視します。彼が「TDD is dead」と言ったのは、TDDそのものではなく、現在広まっている変異した形を指しており、第一原理に戻る必要があるという意図でした。

三者の結論は、意外なほど一致しています。自己テスト可能なコードには大きな価値があり、TDD はある文脈で有効です。どれほど多くの文脈に当てはまるかでは意見が分かれるかもしれませんが、最終的には、ソフトウェア開発に関わるなら自分で考える必要があります。自分とチームに効く実践を積み上げ、どんな技法も盲目的に採用してはいけません。

試し、使い、時には使いすぎ、そこから自分たちに合うものを見つける必要があります。ソフトウェア開発は完全に体系化された科学ではないため、私たちは自分たちの経験と向き合いながら進むしかありません。
