# The LMAX Architecture

## 要約

LMAX は、低レイテンシで大量の注文を処理する小売向け金融取引プラットフォームです。
この記事は、その中核が単一スレッドの Business Logic Processor と Event Sourcing、そして入出力を支える Disruptor によって構成されていることを説明しています。

高性能化のために明示的な並行処理へ進むのではなく、ビジネスロジックを単一スレッドに集約し、I/O 周辺だけを並行化するという設計判断が中心です。
キュー、ロック、CPU キャッシュ、メカニカルシンパシーへの考察を通じて、性能と設計の単純さを両立するアーキテクチャの考え方を学べます。

## 読むときの観点

- Business Logic Processor がなぜ単一スレッドでよいのかを、性能だけでなく複雑さの削減として読む。
- Event Sourcing がインメモリ処理、フェイルオーバー、診断をどう支えているかを見る。
- Disruptor が従来のキューと何を違えることでキャッシュ効率を高めているかを押さえる。
- このアーキテクチャが向く領域と、向かない可能性がある領域を切り分ける。

## 原文の翻訳

LMAX は新しい小売向け金融取引プラットフォームである。そのため、多数の取引を低レイテンシで処理しなければならない。このシステムは JVM プラットフォーム上に構築され、単一スレッドで1秒あたり600万件の注文を扱える Business Logic Processor を中心にしている。Business Logic Processor は Event Sourcing を使い、完全にメモリ内で動作する。Business Logic Processor の周囲には Disruptor が配置されている。Disruptor は、ロックを必要としないキューのネットワークを実装する並行処理コンポーネントである。

設計過程でチームは、キューを使った近年の高性能並行処理モデルの方向性が、現代の CPU 設計と根本的に相性が悪いと結論づけた。

ここ数年、「フリーランチは終わった」[^1] という話をよく耳にする。個々の CPU 速度が上がり続けることを期待することはできない。したがって、高速なコードを書くには、複数プロセッサを並行ソフトウェアで明示的に使う必要がある。これはうれしい知らせではない。並行コードを書くのはとても難しいからだ。ロックやセマフォは推論しにくく、テストもしにくい。つまり、私たちはドメインの問題を解くよりも、コンピュータを満足させることに多くの時間を費やしている。

Actor や Software Transactional Memory など、さまざまな並行処理モデルはこれを簡単にしようとしている。しかし、それでもバグや複雑さを持ち込む負担は残る。

だから私は、昨年3月の QCon London で LMAX の話を聞いたとき強く興味を持った。LMAX は新しい小売向け金融取引プラットフォームである。そのビジネス上の革新は、小売向けのプラットフォームである点だ。つまり、誰でもさまざまな金融デリバティブ商品を取引できる[^2]。このような取引プラットフォームには、非常に低いレイテンシが必要である。市場は急速に動いているため、取引は素早く処理されなければならない。小売向けプラットフォームでは、多数の人に対してこれを行う必要があるため、さらに複雑さが加わる。

その結果、より多くのユーザーがいて、多数の取引があり、それらすべてを素早く処理しなければならない[^3]。

マルチコアを前提に考える流れを踏まえると、このような厳しい性能要求からは、明示的な並行プログラミングモデルが自然に思い浮かぶ。実際、それが彼らの出発点でもあった。しかし QCon で人々の注意を引いたのは、彼らがそこへ行き着かなかったことだった。実のところ彼らは、プラットフォームのすべてのビジネスロジック、すなわち、すべての顧客のすべての市場におけるすべての取引を、単一スレッドで処理することにした。コモディティハードウェア上で1秒あたり600万件の注文を処理するスレッドである[^4]。

低レイテンシで大量のトランザクションを処理し、しかも並行コードの複雑さを持ち込まない。これを掘り下げずにいられるだろうか。幸いなことに、LMAX が他の金融企業と違うもうひとつの点は、技術上の判断について喜んで話してくれることだ。LMAX が本番稼働してしばらく経った今、この興味深い設計を探ってみる時期になった。

### 全体構造

図1: LMAX のアーキテクチャを3つの塊で表したもの

最上位で見ると、このアーキテクチャは3つの部分からなる。

- Business Logic Processor[^5]
- input disruptor
- output disruptors

その名前が示すとおり、Business Logic Processor はアプリケーション内のすべてのビジネスロジックを処理する。先に述べたように、これはメソッド呼び出しに反応し、出力イベントを生成する単一スレッドの Java プログラムとして行われる。したがって、JVM そのもの以外に実行用のプラットフォームフレームワークを必要としない単純な Java プログラムであり、テスト環境で簡単に実行できる。

Business Logic Processor はテスト用の単純な環境で実行できるが、本番環境で動かすには、もう少し複雑な振り付けが必要になる。入力メッセージはネットワークゲートウェイから取り出され、アンマーシャリングされ、複製され、ジャーナルに記録される必要がある。出力メッセージはネットワーク向けにマーシャリングされる必要がある。これらの作業は input disruptor と output disruptor が処理する。

Business Logic Processor と違い、これらは並行コンポーネントである。遅く、互いに独立した I/O 操作を含むからだ。これらは LMAX のために特別に設計・構築されたが、全体アーキテクチャと同様に、他の場所にも適用できる。

### Business Logic Processor

#### すべてをメモリ内に置く

Business Logic Processor は入力メッセージを順番に受け取り、メソッド呼び出しの形で処理し、それに対してビジネスロジックを実行し、出力イベントを発行する。これは完全にメモリ内で動作し、データベースやその他の永続ストアは存在しない。すべてのデータをメモリ内に置くことには、重要な利点がふたつある。第一に速い。アクセスのために遅い I/O を提供するデータベースがなく、処理がすべて順番に行われるため、実行すべきトランザクション処理もない。

第二の利点は、プログラミングが単純になることだ。オブジェクト/リレーショナルマッピングが不要になる。すべてのコードを Java のオブジェクトモデルで書けるため、データベースへのマッピングのために妥協する必要がない。

インメモリ構造を使うことには重要な帰結がある。すべてがクラッシュしたらどうなるのか。どれほど回復力の高いシステムでも、誰かが電源を抜くことには弱い。この対処の中心にあるのが Event Sourcing である。これは、Business Logic Processor の現在の状態を、入力イベントを処理することによって完全に導出できるという意味だ。

入力イベントストリームが耐久性のあるストアに保存されている限り、それは input disruptor の仕事のひとつだが、イベントを再生することでビジネスロジックエンジンの現在状態をいつでも再作成できる。

これを理解するよい方法は、バージョン管理システムを考えることだ。バージョン管理システムはコミットの列であり、任意の時点で、それらのコミットを適用することで作業コピーを構築できる。VCS は Business Logic Processor より複雑である。ブランチをサポートしなければならないからだ。一方、Business Logic Processor は単純な列である。

したがって理論上は、すべてのイベントを再処理することで、Business Logic Processor の状態をいつでも再構築できる。しかし実際には、立ち上げが必要になったときにそれでは時間がかかりすぎる。そこで、バージョン管理システムと同じように、LMAX は Business Logic Processor の状態のスナップショットを作成し、そのスナップショットから復元できる。活動が少ない時間帯に、毎晩スナップショットを取得している。

Business Logic Processor の再起動は速い。JVM の再起動、最近のスナップショットの読み込み、1日分のジャーナルの再生を含む完全な再起動は、1分未満で終わる。

スナップショットは新しい Business Logic Processor の起動を速くするが、午後2時に Business Logic Processor がクラッシュした場合には十分な速さではない。そのため LMAX は複数の Business Logic Processor を常時動かしている[^6]。各入力イベントは複数の Processor によって処理されるが、ひとつを除いて出力は無視される。稼働中の Processor が失敗した場合、システムは別のものへ切り替える。このフェイルオーバー能力も、Event Sourcing を使う利点のひとつである。

レプリカへ Event Sourcing することで、Processor 間の切り替えは数マイクロ秒で行える。毎晩スナップショットを取るだけでなく、Business Logic Processor も毎晩再起動している。複製により、これをダウンタイムなしで行えるため、24時間365日取引を処理し続けられる。

Event Sourcing の背景については、私のサイトに数年前からあるドラフトパターンを参照してほしい。その記事は、LMAX が使っている利点よりも時間的な関係の扱いに重点を置いているが、中核となる考え方は説明している。

Event Sourcing が価値を持つのは、Processor を完全にメモリ内で動かせるようにするからだ。しかし、診断においてもかなり大きな利点がある。予期しない振る舞いが発生した場合、チームはイベント列を開発環境へコピーし、そこで再生する。これにより、ほとんどの環境よりもはるかに簡単に、何が起きたかを調べられる。

この診断能力はビジネス診断にも広がる。リスク管理など、注文処理には不要だが大きな計算を必要とするビジネスタスクがある。例として、現在の取引ポジションに基づき、リスクプロファイル上位20人の顧客リストを得ることが挙げられる。チームは、複製したドメインモデルを立ち上げ、そこで計算を行うことでこれを扱っている。そうすれば、コアの注文処理を妨げない。

これらの分析用ドメインモデルは、異なるデータモデルを持ち、異なるデータセットをメモリ内に保持し、異なるマシンで実行できる。

#### 性能を調整する

ここまで、Business Logic Processor の速さの鍵は、すべてを順番に、メモリ内で行うことだと説明してきた。これを行うだけで、そして本当に愚かなことをしなければ、開発者は1万 TPS を処理できるコードを書ける[^7]。その後、よいコードの単純な要素に集中すれば、これを10万 TPS の範囲まで引き上げられることを見つけた。

必要なのは、うまく分割されたコードと小さなメソッドである。基本的には、これにより HotSpot がよりよく最適化でき、CPU が実行中のコードをより効率よくキャッシュできる。

さらに桁をひとつ上げるには、もう少し工夫が必要だった。LMAX チームがそこへ到達するうえで役立つと見つけたことはいくつかある。そのひとつは、キャッシュに優しく、ガベージに注意した Java コレクションのカスタム実装を書くことだった[^8]。例として、特別に書かれた配列ベースの Map 実装である `LongToObjectHashMap` で、プリミティブな Java の long を hashmap のキーとして使うことが挙げられる。

一般に彼らは、データ構造の選択が大きな違いを生むことが多いと見つけている。ほとんどのプログラマは、その文脈にどの実装が適切かを考えるのではなく、前回使った List をそのままつかむだけである[^9]。

最高水準の性能へ到達するためのもうひとつの技法は、性能テストに注意を向けることだ。私は長い間、人々が性能を改善する技法について多くを語る一方で、本当に違いを生むひとつのことはテストすることだと気づいてきた。

優れたプログラマでさえ、最終的には間違っている性能上の議論を組み立てるのがとても上手い。だから最高のプログラマは、推測よりもプロファイラとテストケースを好む[^10]。LMAX チームはまた、性能テストにおいてもテストを先に書くことが非常に効果的な規律であると見つけている。

#### プログラミングモデル

この処理スタイルは、ビジネスロジックの書き方と編成にいくつかの制約を持ち込む。最初の制約は、外部サービスとのやり取りをすべて取り出さなければならないことだ。外部サービス呼び出しは遅く、単一スレッドでは注文処理マシン全体を止めてしまう。そのため、ビジネスロジック内で外部サービスを呼び出すことはできない。

代わりに、そのやり取りを出力イベントで終え、別の入力イベントがそれを再開するのを待つ必要がある。

説明のために、LMAX ではない単純な例を使おう。クレジットカードでジェリービーンズを注文すると想像してほしい。単純な小売システムなら、注文情報を受け取り、クレジットカード検証サービスでカード番号を確認し、それから注文を確定する。これをすべてひとつの操作の中で行う。

あなたの注文を処理するスレッドは、クレジットカードの確認を待つ間ブロックされる。しかし、そのブロックはユーザーにとってそれほど長くはないだろうし、サーバーは待っている間、いつでも別のスレッドを Processor 上で動かせる。

LMAX アーキテクチャでは、この操作をふたつに分ける。最初の操作は注文情報を取得し、クレジットカード会社へイベント、つまり「クレジットカード検証が要求された」というイベントを出力して終わる。Business Logic Processor はその後、入力イベントストリーム内で「クレジットカードが検証された」イベントを受け取るまで、他の顧客のイベント処理を続ける。そのイベントを処理するとき、その注文に対する確定作業を行う。

この種のイベント駆動で非同期なスタイルで作業することは、いくぶん珍しい。ただし、非同期を使ってアプリケーションの応答性を改善すること自体はよく知られた技法である。また、リモートアプリケーションで起こりうるさまざまなことについて、より明示的に考えざるを得なくなるため、ビジネスプロセスの回復力も高まる。

プログラミングモデルの第二の特徴は、エラー処理にある。セッションとデータベーストランザクションという従来モデルは、有用なエラー処理能力を提供する。何か問題が起きた場合、それまでのやり取りで起きたことをすべて捨てるのは簡単だ。セッションデータは一時的なものであり、複雑な作業の途中ならユーザーに多少の苛立ちを与える代償はあるが、破棄できる。データベース側でエラーが起きた場合は、トランザクションをロールバックできる。

LMAX のインメモリ構造は入力イベントをまたいで永続するため、エラーがあった場合、そのメモリを不整合な状態のままにしないことが重要である。しかし、自動的なロールバック機能はない。その結果、LMAX チームは、インメモリの永続状態を変更する前に、入力イベントが完全に妥当であることを保証することへ多くの注意を払っている。彼らは、この種の問題を本番に入る前にあぶり出すうえで、テストが重要な道具だと見つけている。

### Input Disruptor と Output Disruptor

ビジネスロジックは単一スレッドで発生するが、ビジネスオブジェクトのメソッドを呼び出せるようになるまでに、いくつもの作業がある。処理の元となる入力は、メッセージの形でネットワークから届く。このメッセージは、Business Logic Processor が使いやすい形へアンマーシャリングされる必要がある。Event Sourcing はすべての入力イベントの耐久性あるジャーナルを保つことに依存しているため、各入力メッセージは耐久性あるストアへジャーナル記録されなければならない。

最後に、このアーキテクチャは Business Logic Processor のクラスターに依存しているため、入力メッセージをこのクラスター全体へ複製しなければならない。同様に出力側では、出力イベントをネットワーク送信用にマーシャリングする必要がある。

図2: input disruptor が行う活動。UML アクティビティ図の記法を用いている

replicator と journaler は I/O を伴うため、比較的遅い。そもそも Business Logic Processor の中心的な考えは、I/O を一切行わないことにある。また、これら3つの作業は比較的独立している。Business Logic Processor がメッセージに取り組む前にすべて終わっている必要はあるが、どの順序で実行してもよい。したがって、各取引が後続の取引に対する市場を変える Business Logic Processor とは違い、ここには並行性が自然に合う。

この並行性を扱うために、LMAX チームは特別な並行処理コンポーネントを開発し、それを Disruptor と呼んだ[^11]。

LMAX チームは、Disruptor のソースコードをオープンソースライセンスで公開している。

大ざっぱに言えば、Disruptor はキューのマルチキャストグラフのように考えられる。producer がオブジェクトをそこへ置くと、それらは別々の下流キューを通じて、並列に消費するすべての consumer へ送られる。しかし中を見ると、このキューのネットワークは実際には単一のデータ構造、つまりリングバッファであることがわかる。各 producer と consumer は、バッファ内で現在どのスロットに取り組んでいるかを示す sequence counter を持つ。

各 producer/consumer は自分の sequence counter へ書き込むが、他者の sequence counter を読むことはできる。このようにして producer は、consumer の counter を読み、自分が書き込みたいスロットが利用可能であることを、counter にロックをかけずに確認できる。同様に consumer は、他の consumer がそのメッセージを処理し終えるまで待つ必要がある場合、counter を監視することで、完了後に一度だけ処理することを保証できる。

図3: input disruptor は1つの producer と4つの consumer を調整する

Output disruptor も似ているが、マーシャリングと出力のために、順番に動く2つの consumer だけを持つ[^12]。出力イベントはいくつかの topic に整理されており、関心を持つ受信者だけにメッセージを送れるようになっている。各 topic はそれぞれ自分の disruptor を持つ。

ここで説明した disruptor は、1つの producer と複数の consumer というスタイルで使われているが、これは disruptor 設計上の制限ではない。Disruptor は複数 producer でも動作できる。この場合でもロックは必要ない[^13]。

Disruptor 設計の利点のひとつは、consumer が問題に遭遇して遅れた場合でも、素早く追いつきやすいことだ。unmarshaler がスロット15の処理中に問題を起こし、receiver がスロット31にいるときに戻ってきた場合、スロット16から30までのデータをまとめて読み取って追いつくことができる。Disruptor からのこの一括読み取りにより、遅れている consumer が素早く追いつきやすくなり、全体のレイテンシが下がる。

ここでは journaler、replicator、unmarshaler をそれぞれ1つずつとして説明してきた。実際に LMAX が行っているのもそうである。しかし設計上は、これらのコンポーネントを複数動かすこともできる。journaler を2つ動かした場合、一方が偶数スロットを取り、もう一方が奇数スロットを取る。必要になれば、これにより I/O 操作をさらに並行化できる。

リングバッファは大きい。入力バッファには2000万スロット、各出力バッファには400万スロットがある。sequence counter は64ビットの long 整数であり、リングスロットが循環しても単調に増え続ける[^14]。バッファサイズは2の累乗に設定されているため、コンパイラは sequence counter の番号からスロット番号へ写像する剰余演算を効率よく行える。システムの他の部分と同様に、disruptor も夜間に bounce される。

この bounce は主に、取引中に高価なガベージコレクションイベントが起きる可能性を下げるため、メモリを消す目的で行われる。私は、緊急時にどう行うかを練習できるので、定期的に再起動するのはよい習慣だとも思っている。

journaler の仕事は、何か問題が起きたときに再生できるよう、すべてのイベントを耐久性ある形で保存することである。LMAX はこれにデータベースを使わず、ファイルシステムだけを使う。イベントをディスクへストリーミングする。現代的に言えば、機械式ディスクはランダムアクセスにはひどく遅いが、ストリーミングには非常に速い。したがって「ディスクは新しいテープだ」という標語になる[^15]。

先に、LMAX は高速なフェイルオーバーを支えるため、システムの複数コピーをクラスターで動かしていると述べた。replicator はこれらのノードの同期を保つ。LMAX 内のすべての通信は IP マルチキャストを使うため、クライアントはどの IP アドレスが leader node かを知る必要がない。入力イベントを直接 listen し、replicator を実行するのは leader node だけである。replicator は入力イベントを follower node へブロードキャストする。

leader node が落ちた場合、その heartbeat がないことに気づかれ、別のノードが leader になり、入力イベントの処理を始め、自分の replicator を起動する。各ノードは自分自身の input disruptor を持つため、自分自身の journal を持ち、自分自身で unmarshaling を行う。

IP マルチキャストを使っていても、複製は依然として必要である。IP メッセージは、異なるノードで異なる順序に到着することがあるからだ。leader node は、その後の処理のために決定的な順序を提供する。

unmarshaler は、ネットワーク上のイベントデータを、Business Logic Processor 上の振る舞いを呼び出すために使える Java オブジェクトへ変換する。したがって、他の consumer と違い、リングバッファ内のデータを変更して、このアンマーシャリング済みオブジェクトを保存する必要がある。ここでのルールは、consumer はリングバッファへ書き込むことを許されるが、書き込み可能な各フィールドには、それへ書くことを許された並行 consumer がひとつだけでなければならない、というものだ。これにより、writer をひとつだけにするという原則が保たれる[^16]。

図4: disruptor を展開した LMAX アーキテクチャ

Disruptor は、LMAX システムの外でも使える汎用コンポーネントである。通常、金融企業は自社システムについて非常に秘密主義であり、事業の核心に関係しない項目についてさえ口を閉ざす。LMAX は全体アーキテクチャについてオープンであっただけでなく、Disruptor のコードもオープンソース化した。これは私をとてもうれしくさせる行為である。これにより、他の組織が Disruptor を使えるようになるだけでなく、その並行性の性質について、より多くのテストが行われるようにもなる。

### キューとメカニカルシンパシーの欠如

LMAX アーキテクチャが人々の注意を引いたのは、ほとんどの人が考えている高性能システムへのアプローチとは大きく違っているからだ。ここまでは、それがどう動くかを話してきたが、なぜこのように開発されたのかはあまり深く掘り下げていない。この話はそれ自体として興味深い。なぜなら、このアーキテクチャは突然現れたわけではないからだ。チームがこの形に落ち着くまでには、より従来型の代替案を長く試し、それらがどこで欠陥を持つかを理解する時間が必要だった。

今日の多くのビジネスシステムは、トランザクションデータベースによって調整される複数のアクティブセッションに依存する中核アーキテクチャを持っている。LMAX チームはこのアプローチに慣れており、それが LMAX ではうまくいかないと確信していた。この評価は、LMAX を設立した親会社である Betfair の経験に基づいていた。Betfair は、人々がスポーツイベントに賭けられる賭けサイトである。

そこでは非常に大量のトラフィックが、激しい競合とともに扱われる。スポーツ賭博は特定のイベントの周辺で突発的に集中しがちだからだ。これを動かすために、彼らは非常に高負荷なデータベース導入のひとつを持ち、それを機能させるために多くの不自然なことをしなければならなかった。この経験から、Betfair の性能を維持することがどれほど難しいかを知っており、取引サイトが必要とする非常に低いレイテンシでは、この種のアーキテクチャは機能しないと確信していた。

その結果、別のアプローチを見つけなければならなかった。

彼らの最初のアプローチは、最近多くの人が言っていること、つまり高性能を得るには明示的な並行性を使う必要があるという考えに従うことだった。このシナリオでは、注文を複数スレッドで並列に処理できるようにすることを意味する。しかし、並行処理ではよくあるように、難しさは、これらのスレッドが互いに通信しなければならないところから来る。注文を処理すると市場条件が変わり、その条件を伝達する必要がある。

彼らが初期に探ったアプローチは、Actor モデルとその親戚である SEDA だった。Actor モデルは、自分自身のスレッドを持つ独立したアクティブオブジェクトに依存し、それらがキューを介して互いに通信する。ロックプリミティブに基づく何かをしようとするより、この種の並行処理モデルのほうが扱いやすいと感じる人は多い。

チームは Actor モデルを使って取引所のプロトタイプを作り、性能テストを行った。そこで見つけたのは、Processor がアプリケーションの本当のロジックを行うよりも、キューの管理に多くの時間を費やしていることだった。キューアクセスがボトルネックだった。

このように性能を押し上げようとすると、現代のハードウェアがどのように構成されているかを考慮することが重要になってくる。Martin Thompson が好んで使う言葉は「mechanical sympathy」である。この言葉はレーシングカーの運転に由来し、ドライバーが車に対する本能的な感覚を持っていて、そこから車の能力を最大限引き出す方法を感じ取れることを表す。多くのプログラマ、そして正直に言えば私もその一人だが、プログラミングがハードウェアとどう相互作用するかについて、あまり mechanical sympathy を持っていない。

さらに悪いことに、多くのプログラマは自分に mechanical sympathy があると思っている。しかしそれは、現在では何年も時代遅れになった、ハードウェアが昔どう動いていたかという概念の上に築かれている。

現代の CPU でレイテンシに影響を与える支配的な要因のひとつは、CPU がメモリとどう相互作用するかである。今日、メインメモリへ行くことは、CPU の尺度では非常に遅い操作だ。CPU は複数段のキャッシュを持ち、それぞれがメインメモリより大幅に速い。したがって速度を上げるには、コードとデータをそれらのキャッシュへ入れたい。

ある水準では、Actor モデルはここで助けになる。Actor は、コードとデータをひとまとまりにする自分自身のオブジェクトだと考えられる。これはキャッシュの自然な単位である。しかし Actor は通信しなければならず、その通信はキューを通じて行われる。LMAX チームが観察したのは、キャッシュを妨げているのがそのキューだということだった。

説明はこうだ。キューに何らかのデータを置くには、そのキューへ書き込む必要がある。同様に、キューからデータを取り出すには、取り除くためにキューへ書き込む必要がある。これは書き込み競合である。複数のクライアントが同じデータ構造へ書き込む必要があるかもしれない。書き込み競合に対処するため、キューはしばしばロックを使う。しかしロックが使われると、カーネルへのコンテキストスイッチを引き起こす可能性がある。これが起きると、関係する Processor はキャッシュ内のデータを失う可能性が高い。

彼らが到達した結論は、最良のキャッシュ動作を得るには、任意のメモリ位置へ書き込むコアがひとつだけである設計が必要だということだった[^17]。複数の reader は問題ない。Processor はキャッシュ間で特別な高速リンクを使うことが多い。しかしキューは、one-writer 原則に反する。

この分析から、LMAX チームはいくつかの結論に至った。第一に、one-writer 制約に断固として従う Disruptor の設計へつながった。第二に、並行性管理から解放された単一スレッドがどれほど速く動けるのかを問う、単一スレッドのビジネスロジックアプローチを探るという考えへつながった。

単一スレッドで作業する本質は、ひとつのスレッドをひとつのコアで走らせ、キャッシュを温め、可能な限り多くのメモリアクセスをメインメモリではなくキャッシュへ向かわせることにある。これは、コードと作業中のデータセットの両方が、できるだけ一貫してアクセスされる必要があることを意味する。また、コードとデータを一緒に持つ小さなオブジェクトを保つことで、それらをひとまとまりとしてキャッシュ間で入れ替えられるようになり、キャッシュ管理が単純になり、やはり性能が向上する。

LMAX アーキテクチャへ至る道の重要な部分は、性能テストの利用だった。Actor ベースのアプローチを検討し、捨てることになったのは、プロトタイプを構築して性能テストを行ったからである。同様に、さまざまなコンポーネントの性能を改善する多くの段階も、性能テストによって可能になった。

mechanical sympathy は非常に価値がある。どのような改善が可能かについて仮説を立てる助けになり、後退ではなく前進へ導いてくれる。しかし最終的には、テストこそが説得力のある証拠を与える。

ただし、このスタイルの性能テストは、よく理解された話題ではない。LMAX チームは、意味のある性能テストを考え出すことは、本番コードを開発することより難しい場合が多いと繰り返し強調している。ここでも、正しいテストを開発するには mechanical sympathy が重要である。低レベルの並行処理コンポーネントをテストする場合、CPU のキャッシュ動作を考慮しないなら意味がない。

特に重要な教訓のひとつは、性能テストが実際のコンポーネントが行っていることを本当に測定できるほど速いことを保証するため、null component に対するテストを書くことである。速いテストコードを書くことは、速い本番コードを書くことと同じくらい難しい。測ろうとしているコンポーネントと同じくらいテストが速くなければ、誤った結果を得るのはあまりに簡単である。

### このアーキテクチャを使うべきか

一見すると、このアーキテクチャは非常に小さなニッチ向けに見える。何しろ、それを生んだ推進力は、多数の複雑なトランザクションを非常に低いレイテンシで実行できるようにすることだった。ほとんどのアプリケーションは600万 TPS で動く必要はない。

しかし、このアプリケーションについて私が魅了されるのは、多くのソフトウェアプロジェクトを悩ませるプログラミング上の複雑さの多くを取り除く設計に、彼らが行き着いたことだ。トランザクションデータベースを取り囲む並行セッションという従来モデルは、面倒ごとから自由ではない。通常、データベースとの関係には無視できない労力が入る。オブジェクト/リレーショナルマッピングツールは、データベースを扱う痛みの多くを助けてくれるが、すべてを解決するわけではない。

エンタープライズアプリケーションの性能チューニングの多くは、SQL をあれこれいじることを伴う。

今日では、私たち年寄りがディスク容量として得られたよりも多くのメインメモリを、サーバーへ積める。ますます多くのアプリケーションが、自分たちの作業セット全体をメインメモリに置けるようになっている。これにより、複雑さと遅さの源をひとつ取り除ける。Event Sourcing はインメモリシステムの耐久性問題を解く方法を提供し、すべてを単一スレッドで実行することは並行性の問題を解く。

LMAX の経験は、必要な性能が数百万 TPS 未満である限り、十分な性能上の余裕があることを示唆している。

ここには、CQRS への関心の高まりとのかなりの重なりがある。Event Sourcing されたインメモリ Processor は、CQRS システムの command 側に自然に合う。ただし、LMAX チームは現在 CQRS を使っていない。

では、この道へ進むべきでないことを示すものは何だろうか。このようなあまり知られていない技法では、これはいつも難しい問いである。専門職として、その境界を探るにはもっと時間が必要だからだ。ただし出発点として、このアーキテクチャを後押しする特徴を考えることはできる。

ひとつの特徴は、これがつながりのあるドメインであり、ひとつのトランザクションを処理することが、後続のトランザクションの処理方法を常に変える可能性を持つ点である。互いにより独立したトランザクションであれば、調整の必要は少なくなるため、別々の Processor を並列に動かすことがより魅力的になる。

LMAX は、イベントが世界をどう変えるかの結果を把握することに集中している。多くのサイトは、既存の情報ストアを取り出し、その情報のさまざまな組み合わせを、見つけられる限り多くの閲覧者へ描画することに重点を置いている。たとえば、どんなメディアサイトでも思い浮かべればよい。ここでのアーキテクチャ上の課題は、しばしばキャッシュを正しく扱うことに集中する。

LMAX のもうひとつの特徴は、これがバックエンドシステムだという点である。そのため、インタラクティブモードで動く何かにどれほど適用できるかを考えるのは妥当だ。Web アプリケーションは、リクエストに反応するサーバーシステムに私たちをますます慣れさせており、この側面はこのアーキテクチャによく合う。このアーキテクチャがそのような多くのシステムより先へ進んでいる点は、非同期通信を徹底して使うことにあり、その結果、先に概説したようなプログラミングモデルの変化が生じる。

これらの変化には、ほとんどのチームが慣れるまで時間がかかるだろう。多くの人は同期的な言葉でプログラミングを考えがちであり、非同期を扱うことに慣れていない。それでも、非同期通信が応答性にとって不可欠な道具であることは、長い間ずっと真実だった。JavaScript の世界で AJAX や node.js とともに非同期通信がより広く使われることで、このスタイルを調べる人が増えるかどうかは興味深い。

LMAX チームは、非同期スタイルに慣れるには少し時間がかかったものの、すぐに自然になり、しばしばより簡単になったと見つけている。特にエラー処理は、このアプローチのもとではずっと扱いやすかった。

LMAX チームは、調整役としてのトランザクションデータベースの日々は確実に残り少ないと感じている。この種のアーキテクチャを使えばソフトウェアをより簡単に書け、しかもより速く動くという事実は、従来の中央データベースを正当化する理由の多くを取り除く。

私としては、これは非常にわくわくする話である。私の目標の多くは、複雑なドメインをモデル化するソフトウェアに集中することにある。このようなアーキテクチャは優れた関心の分離を提供し、人々が Domain-Driven Design に集中し、プラットフォーム上の複雑さの多くを十分に分離しておけるようにする。ドメインオブジェクトとデータベースの密結合は、いつも苛立ちの種だった。このようなアプローチは、そこから抜け出す道を示唆している。
