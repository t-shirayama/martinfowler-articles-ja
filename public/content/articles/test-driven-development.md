# Test Driven Development

## 要約

Test-Driven Development（TDD）は、テストを書くことによってソフトウェア開発を導く技法です。
Kent Beck が Extreme Programming の一部として発展させたもので、次に追加したい機能のテストを書き、そのテストを通すコードを書き、最後にコードをきれいに整える、という短いサイクルを繰り返します。

TDD の中心は Red-Green-Refactor ですが、最初にテストケースの一覧を書き出し、そこから順に取り組むことも重要です。
テストを先に考えることで、Self-Testing Code を得やすくなるだけでなく、実装より先にインターフェイスを考えるようになります。

## 読むときの観点

- TDD を「テストを書く技法」だけでなく、設計を進めるための開発サイクルとして読む。
- Red-Green-Refactor のうち、特に Refactor を省くと何が失われるかに注目する。
- テストケースの順序づけが、設計上の重要点へ早く到達するための技術である点を見る。
- Test-First Programming、Self-Testing Code、インターフェイス設計の関係を整理する。

## 原文の翻訳

Test-Driven Development（TDD）は、テストを書くことによってソフトウェア開発を導く技法です。これは Kent Beck によって、1990年代後半に Extreme Programming の一部として発展しました。本質的には、次の三つの単純なステップを繰り返します。

- 追加したい次の小さな機能に対するテストを書く。
- そのテストが通るまで、機能コードを書く。
- 新しいコードと既存のコードの両方をリファクタリングし、よく構造化された状態にする。

この三つのステップは、しばしば Red-Green-Refactor と要約され、プロセスの中心にあります。ただし、それに加えて重要な最初のステップがあります。まずテストケースの一覧を書き出すことです。そのうえで、その中から一つを選び、Red-Green-Refactor を適用します。それが終わったら、次のテストを選びます。テストを適切な順番に並べることは一つの技術です。設計上の重要な点へすばやく到達できるようなテストを選びたいのです。作業中に新しいテストを思いついたら、その都度一覧へ追加していきます。

テストを先に書くこと、XP 第2版が Test-First Programming と呼ぶものには、主に二つの利点があります。もっとも明らかな利点は、Self-Testing Code を得る方法になることです。なぜなら、テストを通すことへの応答としてだけ、機能コードを書くことになるからです。もう一つの利点は、テストを先に考えることで、コードのインターフェイスを先に考えざるを得なくなることです。**インターフェイスとクラスの使い方へ焦点を当てること**は、インターフェイスと実装を分ける助けになります。これはよい設計の重要な要素であり、多くのプログラマが苦労するところです。

TDD を台無しにするもっともよくあるやり方として私が耳にするのは、三つ目のステップをなおざりにすることです。コードをきれいに保つためのリファクタリングは、このプロセスの重要な一部です。そうしなければ、私たちはコード断片の雑然とした寄せ集めに行き着くだけです。もっとも、それらには少なくともテストがあるので、設計の失敗としてはたいていの場合より痛みは小さい結果になります。

### さらに読む

TDD の標準的な進め方については、Kent による要約が、オンラインで読める重要なまとめです。

より深く知るには、Kent Beck の本『Test-Driven Development』を読むとよいでしょう。

James Shore の『The Art of Agile Development』の該当章も、堅実な説明です。そこでは、TDD が効果的なアジャイル開発の他の部分とどのようにつながるかも示されています。James はまた、『Let's Play TDD』というスクリーンキャストのシリーズも作っています。
