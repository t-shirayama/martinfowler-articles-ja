# Diff Debugging

## 要約

Diff Debugging は、回帰バグを調べるときに、そのバグを生んだ変更を探す調査方法です。
原因となった変更を見ることで、バグの場所や直し方について強い手がかりが得られます。

この方法を効果的に使うには、バージョン管理、再現可能なビルド、小さなコミットが重要です。
Git では `git bisect` によって多くの作業を自動化できます。

## 読むときの観点

- 回帰バグでは「どの変更で壊れたか」を知ることが大きな手がかりになる。
- Diff Debugging は二分探索で原因コミットを絞り込む。
- 再現可能なビルドがないと、古い版を検証しにくい。
- 小さく頻繁なコミットは、原因を見つけた後の理解を助ける。
- テストでバグを再現できれば、`git bisect` をさらに強力に使える。

## 原文の翻訳

回帰バグとは、以前から存在していた機能に新しく現れたバグである。こうしたバグを追うときは、どのソフトウェア変更がバグを発生させたのかを突き止めることが非常に役立つ。その変更を見ることで、バグがどこにあり、どう直せばよいかについて貴重な手がかりが得られる。この調査方法に広く知られた名前はあまりないが、Diff Debugging と呼ぶことができる。

Diff Debugging は、コードがバージョン管理されている場合にだけ機能する。幸い、今日ではそれが普通になっている。ただし、効果的に使うにはさらに条件がある。古いバージョンを容易に実行できるよう、再現可能なビルドが必要である。また、高頻度の統合によってコミットが小さく保たれていると、とても役に立つ。原因コミットが見つかったとき、何が起きたのかを絞り込みやすいからだ。

原因コミットを探すには、まず過去のどこかにバグがない版を見つける。それを最後に良かった版とし、現在の版を最初に悪い版とする。次に、その2点の中間にあるコミットを調べ、そこにバグがあるかを確認する。バグがあればそのコミットを最初に悪い版にし、なければ最後に良かった版にする。

この作業を繰り返すと、半区間探索、あるいは二分探索によって原因コミットにたどり着ける。Git を使っているなら、`git bisect` コマンドがこの作業の多くを自動化してくれる。バグの存在を示すテストを書けるなら、`git bisect` にそのテストを使わせることで、原因コミットの発見全体を自動化できる。

Diff Debugging は、普段のプログラミングセッションの中でも役に立つ。数分かかる遅いテストがある場合、しばらくは関連するテストの一部だけを実行しながら作業することがある。その間、緑になったテスト実行ごとにコミットしておけば、後で遅いテストが失敗したときに Diff Debugging を使える。

ここに、極めて頻繁にコミットする価値がある。長期的な履歴では小さすぎるため後で squash したくなるようなコミットでも、作業中の原因特定には役立つ。IDE の中には、バージョン管理のコミットより細かいローカル履歴を自動的に保持し、このやり方をさらに楽にしてくれるものもある。

Diff Debugging の本質は、バグを直接推測する前に、**バグを生んだ差分を特定すること**にある。差分が小さければ小さいほど、調査は速く、確かなものになる。
