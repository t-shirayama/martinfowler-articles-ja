# Domain Driven Design

## 要約

ドメイン駆動設計は、複雑な業務領域を理解し、その知識をソフトウェアのモデルに反映するための考え方です。
技術的な層分けだけではなく、業務の言葉、モデル、境界、チームの会話を重視します。

DDDを読むときは、パターン集としてではなく、複雑な領域で認識を揃えるための設計アプローチとして捉えるのがよさそうです。
実装パターンに進む前に、ユビキタス言語や境界づけられたコンテキストの意味を押さえると理解しやすくなります。

## 読むときの観点

- 業務の言葉をコードやモデルに反映する。
- 技術構造より先に、ドメイン理解を深める。
- 境界づけられたコンテキストでモデルの範囲を分ける。
- DDDを重い儀式ではなく、複雑さに向き合う方法として見る。

## 原文の翻訳

ドメイン駆動設計は、ドメインのプロセスやルールを深く理解したドメインモデルをプログラムすることを開発の中心に置く、ソフトウェア開発のアプローチです。
この名前は、Eric Evans が 2003 年に出した本に由来します。
その本では、このアプローチがパターンのカタログを通じて説明されています。
その後、実践者のコミュニティがこの考えをさらに発展させ、さまざまな本やトレーニングコースが生まれました。
このアプローチは、**整理すべき複雑で乱雑なロジックが多いドメイン**に特に適しています。

Eric Evans の 2003 年の本は、真剣なソフトウェア開発者にとって必読です。

ソフトウェアシステムは、ドメインについてよく作り込まれたモデルにもとづく必要がある、という考えは、少なくとも私がこの業界にいるあいだずっと存在してきました。
私はこの考え方の多くを Jim Odell から学びました。
彼は、データモデリング、情報工学、オブジェクト指向分析とともに、このような考え方を発展させました。
1980 年代から 1990 年代にかけて、データベースとオブジェクト指向のコミュニティで行われた多くの仕事において、基礎にあるドメインを表現することは重要な部分でした。

Eric Evans がその本を通じて大きく貢献したのは、このアプローチについて話すための語彙を発展させたことです。
当時の議論を支配していたさまざまなモデリング記法を越えて、重要な概念要素を特定しました。
その中心にあったのは、複雑なドメインのためにソフトウェアを開発するには、私たちが作るソフトウェアシステムの中へドメイン用語を埋め込む**ユビキタス言語**を作る必要がある、という考えでした。

多くの人がそのようなモデルを開発することについて語っていましたが、それらは紙の上で行われるだけで、たいていは事前に完了するものだと期待されていました。
DDD は、それらをソフトウェアの中で行い、ソフトウェアプロダクトの寿命を通じて進化させることを強調します。
Eric は Extreme Programming の強い支持者であり、ドメイン駆動設計をエクストリームプログラミングのアプローチに自然に含まれるものと見ています。
これは、私が知る XP 実践者の多くも共有している見方です。

この本は、オブジェクトを Entity、Value Object、Service Object に分類する考え方を導入しました。
私はこれを Evans Classification と呼んでいます。
また、集約の概念も特定しました。
これらは、プログラミング言語や図式的な記法のどちらからも抜け落ちていた、オブジェクトについて考えるうえでの重要な穴を埋めていると感じました。
DDD の特に重要な部分は、**大きなドメインを境界づけられたコンテキストのネットワークとしてどう組織するか**という Strategic Design の考え方です。
それまで、この問題に説得力のある形で取り組んだ人を私は見たことがありませんでした。

Eric の背景はオブジェクト指向コミュニティに根ざしていますが、ドメイン駆動設計の中核にある考え方は概念的なものです。
したがって、どのようなプログラミングアプローチにもよく当てはまります。
これは特に、戦略的設計の側面について強く言えます。

### さらに読む

Eric の元の本は読みづらい本として知られています。
しかし私は正直に言って、その努力に十分報いる本であり、真剣な開発者の本棚に置かれるべき一冊だと考えています。
Vaughn Vernon の 2013 年の本は、次の一歩として適しています。
後から発展した考え方を説明し、特に戦略的設計の側面に焦点を当てています。
