# Interrogatory LLM

## 要約

Interrogatory LLM は、LLM に人間をインタビューさせ、その結果から別の作業に使うための文脈文書を作らせる使い方です。複雑な機能設計や仕様整理では、LLM に与えるべき背景情報が多くなります。その文脈を人間が最初から書く代わりに、LLM が必要な質問を重ねて情報を引き出す、という発想です。

Fowler はこの方法を、文書を作る場面だけでなく、既存文書の妥当性を専門家に確認する場面にも広げています。書くことが得意でない人から知識を取り出し、他の人が読める形にするための手段としても使えるかもしれない、という見方です。

## 読むときの観点

- LLM に「答えを出させる」のではなく、必要な文脈を集めるために質問させる使い方として読む。
- 一度に多くの質問をさせず、会話を細かく進めることの意味に注目する。
- 文書作成だけでなく、専門家レビューの代替または補助として使える点を見る。
- AI らしい文章の癖が残っても、知識が共有可能な形になる価値と比較して考える。

## 原文の翻訳

LLM に複雑な作業をさせる必要があるとき、私たちは多くの文脈を与えなければならないことがよくあります。新しい機能の設計を考えるには、その機能をユーザーにどう見せたいのか、どのように実装すべきかのガイドライン、参照すべき外部システムの情報などが必要です。こうした情報は、Markdown で数ページになることもあります。いちばん分かりやすい方法は、人間がこの文脈を書くことです。しかし別の方法として、人間にインタビューしたあとで、LLM にこの文脈を書かせることもできます。

そのためのやり方は、LLM に私を問いただすよう促すことです。適切な文脈を作るために必要な質問を、すべて私に尋ねさせます。必要な情報の多くは私から与えられますし、LLM が自分で判断できない場合に参照すべき他の情報源を伝えることもできます。完了したら、次のステップを実行する別のセッション、場合によっては別のモデルのために、文脈レポートを作れます。

このアプローチについて、私が最初にまとまった説明を見たのは Harper Reed のブログでした。彼の方法で印象的なのは、LLM に一度に一つの質問だけをさせるよう強く求めている点です。私が試したときには、これを頻繁に思い出させる必要がありました。

Interrogatory LLM の別の使い方は、ソフトウェア仕様のようにドメイン知識を捉えた文書を与え、その文書が正確かどうかを判断するために、人間の専門家へインタビューさせることです。これは、人間の専門家に文書を読んでレビューしてもらう代替になります。レビューは難しいと感じる人が多いので、特に文書の出来がよくない場合には、LLM との会話のほうが実り多いかもしれません。

もちろん、この二つを組み合わせることもできます。一つの interrogatory LLM で文書を作り、別の interrogatory LLM で他の専門家と一緒にその文書をレビューする、という使い方です。

ここまでの話は、LLM の特定の利用目的に向けて、LLM に文脈を作成または評価させるものでした。しかし、この技法はもっと広く適用できます。私は自然に書く人間になりました。考えるうえで、書くプロセスが不可欠だと感じる人間です。何かを本当に理解するには、それについて書く必要があります。しかし人によって違いがあります。多くの人にとって、書くことは難しく、しばしば非常に難しいものです。誰かの頭の中にある情報を、他の人間が利用できる形に取り出す必要があるとき、これは現実的な問題になります。

そうした人たちにとっては、自分で文書を書くより、LLM にインタビューしてもらうほうが楽かもしれません。もちろん出来上がりには、私のような人間が身震いするような AI 文章の匂いが残るでしょう。それでも、急いで書いたせいで情報が失われたり、そもそも何も書かれなかったりするよりは、**情報そのものが残ることのほうが重要**です。
