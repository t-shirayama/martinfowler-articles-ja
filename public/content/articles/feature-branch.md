# Feature Branch

## 要約

Feature Branch は、新しい機能に取り組み始めるときにブランチを作り、その機能の作業をブランチ上で完了させてから共通コードベースへ統合するブランチパターンです。
オープンソース開発では特によく使われます。

作業を隔離できる一方で、統合のリスクは後ろに延期されます。
機能の完成に数週間から数か月かかる場合、問題の早期発見やリファクタリングを妨げることがあります。

## 読むときの観点

- Feature Branch は、未完成の機能を共通コードベースから隔離する。
- 隔離は安全に見えるが、統合リスクを後ろへ送る。
- 作業中に mainline の変更を取り込んでも、自分の変更は完成まで共有されない。
- 短期間で終わる機能なら問題は小さくなりやすい。
- 長命な Feature Branch は、統合遅れとリファクタリング抑制を招きやすい。

## 原文の翻訳

Feature Branch は、開発者が新しい機能に着手するときにブランチを開き、その機能に関する作業をすべてそのブランチ上で行う、ソースコードのブランチパターンである。機能が完了した時点で、変更をチームの他の作業と統合する。

作業中、開発者はチームの他のメンバーによって確認済みの変更を自分のブランチに取り込むことがある。これは、機能が完成したときの統合作業を減らすためである。しかし、自分の変更を共通コードベースに入れるのは完成後である。その結果、別々の Feature Branch で作業している2人は、どちらかが共通コードベースへマージするまで互いの作業を統合しない。

Feature Branch は人気のある手法であり、特にオープンソース開発に適している。機能に関する作業を完了までチームの共通コードベースから離しておけるため、マージに伴うリスクをその時点まで延期できる。

しかし、この隔離は問題の早期発見を妨げる。より深刻なのは、リファクタリングを抑制しやすいことだ。リファクタリングが不足すると、コードベースの健全性は大きく損なわれることがある。

Feature Branch の影響は、機能を完成させるまでにどれくらい時間がかかるかに大きく左右される。通常1日か2日で機能を完了できるチームであれば、十分に頻繁に統合できるため、統合遅れの問題を避けやすい。数週間、あるいは数か月かけて機能を完成させるチームでは、これらの困難により多く直面する。

したがって、Feature Branch を評価するときの焦点は、ブランチを使うかどうかだけではない。重要なのは、**統合がどれだけ遅れるか**である。統合が遅れるほど、衝突、設計のずれ、隠れた不具合、リファクタリング不足のリスクは大きくなる。
