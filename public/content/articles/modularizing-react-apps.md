# Modularizing React Applications with Established UI Patterns

## 要約

この記事は、Reactを「アプリケーション全体の形」ではなく、主にViewを構築するためのライブラリとして捉え直し、フロントエンドにも古典的なUI設計パターンやレイヤリングを適用する考え方を説明している。

例として、注文画面の支払い機能を段階的にリファクタリングしていく。最初は1つのReactコンポーネントにデータ取得、変換、状態管理、表示ロジックが混ざっている。そこから、表示用コンポーネント、custom hook、ドメインモデル、外部サービスへのGatewayのような責務へ分けることで、変更しやすく、再利用しやすい構造へ近づけていく。

中心にある主張は、Reactアプリケーションであっても、View、状態、ドメインロジック、データアクセスを区別することは有効だという点である。コンポーネントやhookにすべてを押し込むと、初期には速く見えても、国別の支払いルールや寄付機能のような要求が増えたときに、変更箇所が散らばり、理解と修正の負荷が高くなる。

## 読むときの観点

- Reactをアプリケーションの中心ではなく、View層の実装手段として捉える。
- コンポーネント、hook、モデル、データアクセスの責務がどこで混ざっているかを見る。
- 小さなサンプルで問題が見えにくくても、要求追加によってどのように構造上の弱さが表面化するかを追う。
- `Extract Function`、`Extract Class`、`Replace Conditional with Polymorphism`のようなリファクタリングが、フロントエンドにも自然に使えることを確認する。
- Viewから切り離したロジックが、別のViewライブラリ、CLI、バックエンドなどでも再利用できる可能性に注目する。

### ReactはViewのためのライブラリ

Reactは、UIを構築するためのJavaScriptライブラリである。にもかかわらず、実際のプロジェクトでは、データ取得、レスポンスの変換、状態管理、画面遷移、業務ルールまでがReactコンポーネントやhookの中に詰め込まれがちである。

小さな画面や試作であれば、それでも大きな問題には見えない。しかし、フロントエンドに置かれる業務ロジックが増えるほど、**すべてをコンポーネントに入れる構造は理解と変更のコストを押し上げる**。フロントエンドも通常のソフトウェアであり、関心の分離や既存の設計パターンは同じように役に立つ。

### 現実のReactアプリケーション

Reactの魅力は、データからDOMを作る関数のようにUIを考えられる単純さにある。しかし現実のアプリケーションには、ネットワークリクエスト、ルーティング、ローカルストレージ、キャッシュ、外部サービス連携、認証、ログ、性能調整など、多くの要素が存在する。

これらを1つのコンポーネントへまとめると、読む人は表示、通信、変換、業務判断のあいだを何度も行き来しなければならない。そこで、View、Model、Dataのようなレイヤに分け、1つずつ考えられるようにする。これはデスクトップGUIの時代から使われてきた考え方であり、Reactを使うフロントエンドにも適用できる。

### Reactアプリケーションの成長

最初の段階では、アプリケーション全体が1つのコンポーネントに収まっているかもしれない。HTMLに近い見た目のコードに、少しだけ状態や条件分岐が加わっている程度である。

画面が大きくなると、HTML構造に合わせて複数のコンポーネントへ分けるようになる。これはViewを理解しやすくするうえで効果がある。ただし、データ取得、形式変換、送信用データの組み立て、複雑な内部状態が残っていると、コンポーネントはなお重くなる。

次に、状態管理や副作用をcustom hookへ移す。hookは状態とその変化にまつわる処理を共有しやすくするが、hookの中にも純粋な計算や業務ルールが残ることがある。その場合は、ドメインオブジェクトやモデルとしてさらに切り出す。こうして、表示に依存しないロジックが徐々に明確な場所を持つようになる。

最終的には、View、hook、ドメインモデル、データアクセスのような層が見えてくる。重要なのは、最初から大きな構造を押しつけることではなく、**責務の違いが見えた時点で適切な場所へ分けていくこと**である。

### 支払い機能の導入

記事では、オンライン注文アプリケーションの支払いセクションを例にする。ユーザーは商品を選び、支払い方法を選択する。支払い方法はサーバー側で設定され、国によって表示される選択肢が変わる。設定が返らなければ、現金払いを既定として扱う。

最初の実装では、`Payment`コンポーネントの中に、支払い方法の取得、APIレスポンスのローカル形式への変換、ラジオボタンの描画、支払いボタンの表示がすべて含まれている。簡単なチュートリアルではよく見かける形だが、1つの場所に多くの関心が混ざっている。

この段階での問題は、変更のために複数の文脈を同時に理解しなければならないことだ。ネットワークリクエストの初期化、データ変換、支払い方法の描画、`Payment`自体の表示構造が同じ関数に並んでいるため、読む人の頭の中で切り替えが増える。

### Viewと非Viewの分離

最初の改善として、支払い方法の取得と状態管理を`usePaymentMethods`のようなcustom hookへ切り出す。これにより、`Payment`コンポーネントは支払い方法を受け取り、それを描画することに近づく。

さらに、支払い方法の一覧表示は独立した概念なので、`PaymentMethods`のような子コンポーネントへ分ける。こうすると、`Payment`は画面全体の構成を示し、支払い方法の具体的な描画は別の小さな部品に任せられる。

この時点でも、APIレスポンスをローカルな`PaymentMethod`へ変換する処理がhookに残っている。変換処理はReactの状態管理そのものではなく、業務上のデータ整形である。そこで、`convertPaymentMethods`のような純粋関数として外に出す。これにより、変換ロジックをテストしやすくなり、Viewからもhookからも独立させられる。

### データモデリング

支払い方法の変換では、外部サービスのデータ形式と、画面が必要とするデータ形式が異なる。外部の形をそのままViewに持ち込むと、API変更の影響が画面全体へ広がりやすい。

そこで、アプリケーション内部で使うモデルを定義し、外部データから内部モデルへの変換を明示する。これは単なる型の置き換えではなく、外部システムとの境界を作る作業でもある。内部モデルが安定していれば、外部APIの変更を局所化しやすくなる。

この分割の利点は、コンポーネントが表示に集中できること、hookが状態や副作用に集中できること、変換ロジックが独立してテスト可能になることにある。

### 寄付機能の追加

次に、新しい要求として、支払い時に端数を切り上げ、その差額を慈善団体へ寄付できる機能が加わる。ユーザーはチェックボックスで寄付に同意し、同意した場合は合計金額が切り上げられる。

この要求を素朴に実装すると、`Payment`コンポーネントには支払い方法の表示に加えて、寄付同意の状態、切り上げ計算、寄付額の表示、ボタン文言の変更が入ってくる。せっかく分けた責務が、再び1つのコンポーネントに集まり始める。

ここで、寄付同意の状態と切り上げ計算を`useRoundUp`のようなhookへ切り出す。hookは元の金額を受け取り、同意状態に応じた合計金額と寄付額を返す。View側は、その結果を使ってチェックボックスとボタンを描画するだけになる。

Reactのhookは、Viewの背後にある小さな状態機械として考えることもできる。UIイベントがhookの状態を変え、その結果として新しい表示が作られる。**コンポーネントをできるだけ表示関数に近づけ、状態の変化をhookへ寄せる**ことで、読みやすさと再利用性が上がる。

### 国別ルールとShotgun Surgery

事業が他国へ広がると、切り上げルールが国ごとに変わる。日本では最小単位が小さすぎるため百単位に切り上げたい、デンマークでは十単位にしたい、といった要求が出る。

単純には、`countryCode`を`Payment`から`useRoundUp`へ渡し、国コードによる条件分岐を追加できる。しかしこの方法では、切り上げ計算、通貨記号、チェックボックスの文言、ボタン表示など、複数の場所に同じような国別分岐が広がる。

これはShotgun Surgeryのにおいである。新しい国を追加するたびに、多くのファイルや関数を同時に直す必要があり、テストが不十分ならミスも起きやすい。

### ポリモーフィズムによる整理

散らばった条件分岐を減らす方法として、国ごとの支払い戦略をオブジェクトにまとめる。国によって違うのは、通貨記号、切り上げ金額、寄付額の計算である。これらを`PaymentStrategy`や`CountryPayment`のようなインターフェースまたはクラスに寄せる。

すると、`useRoundUp`は国コードを直接知らなくてよくなる。代わりに、渡された戦略オブジェクトに切り上げと寄付額の計算を依頼する。`Payment`コンポーネントも、表示用の文言やボタンラベルを作るときに同じ戦略を使える。

この構造では、国別の違いが1つの概念にまとまる。新しい国のルールを追加するときも、Viewやhookに条件分岐を増やすのではなく、戦略を追加または差し替える方向にできる。

### ネットワーククライアントの抽出

さらに分離を進めると、`usePaymentMethods`の中に残っているデータ取得処理もReact固有ではないことが見えてくる。現時点では短いコードでも、エラーハンドリング、リトライ、キャッシュ、ロード状態などが加わると、hookはすぐに膨らむ。

そこで、外部APIへのアクセスとレスポンス変換をまとめる小さなクライアントやGatewayを用意する。hookはそのクライアントを呼び出して、結果をReactの状態へ反映するだけになる。

このGatewayは、外部システムとの境界をカプセル化する。外部APIの形が変わっても、変更を1か所に閉じ込めやすい。View、hook、モデル、データアクセスの責務がそれぞれ明確になり、React以外の場所でも再利用できるコードが増える。

### レイヤを持つ利点

このように層を分けると、保守性、モジュール性、読みやすさ、拡張性が上がる。問題が起きたとき、どこを見ればよいかを絞り込みやすくなる。新しい機能を追加するときも、システム全体を触らずに済む可能性が高まる。

また、ドメインロジックが純粋なJavaScriptまたはTypeScriptとしてViewから独立していれば、将来Viewライブラリを変える場合にも影響を抑えられる。実際に置き換える機会は少ないかもしれないが、Viewから切り離されたロジックは、それ自体が設計上の余裕になる。

### 結論

Reactを使っているからといって、フロントエンドを特別な種類のソフトウェアとして扱う必要はない。従来のUIアプリケーションで有効だった設計原則やパターンは、ReactをView層として使うアプリケーションにも当てはまる。

重要なのは、UIをできるだけ薄く保ち、業務ロジックを支えるモデル層へ、データアクセスを別の層へ移すことだ。そうすれば、開発者は一度に1つの関心へ集中できる。結果として、既存コードへの変更は以前より扱いやすくなり、再利用できる部分も増えていく。
