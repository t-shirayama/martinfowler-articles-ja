# Pair Programming Misconceptions

## 要約

Pair Programming には、アジャイルなら必ず行うもの、XPなら強制されるもの、生産性を半分にするもの、といった誤解があります。
この記事は、それらを一つずつほどきながら、ペアプログラミングは処方箋ではなく、チームが有効性を確かめながら取り入れるプラクティスだと説明しています。

特に重要なのは、ペアプログラミングを正しく試さないまま判断しないことです。
単に誰かの肩越しに長時間眺めるだけではなく、経験者から助言を受け、設計、レビュー、知識共有への効果を見ながら評価する必要があります。

## 読むときの観点

- Pair Programming をアジャイルやXPの「必須条件」としてではなく、チームの有効性を高めるための選択肢として読む。
- 生産性をタイピング量だけで測る見方が、設計や欠陥予防、コード理解の共有を見落としていないか考える。
- 「自分には合わない」と決める前に、よいコーチングを受けて本来の形を試すことの意味を見る。
- 単純作業に見えるコードが、実は抽象化の不足を示していないかという視点に注目する。

## 原文の翻訳

Pair Programming についてよくある誤解をいくつか挙げます。

### アジャイルプロセスをやるならペアプログラミングをしなければならない

これはまったくの誤りです。「アジャイル」は非常に広い言葉であり、価値と原則、とりわけ Manifesto for Agile Software Development によって定義されています。このマニフェストはペアプログラミングに触れておらず、ほとんどのアジャイル手法も、それを自分たちのアプローチの一部にはしていません。

ペアプログラミングはXPのプラクティスなので、アジャイルコミュニティに大きな影響を与えてきました。その結果、アジャイルのプラクティス、つまりアジャイルプロジェクトに関わる人たちがよく使うプラクティスとして言及されることがよくあります。しかしそれは観察であって、処方ではありません。

### Extreme Programming はペアプログラミングを強制する

これはずっと微妙な問題です。ペアプログラミングはXPのプラクティスの一つであり、XPの始まり以来そうでした。ここでの微妙さは、XPをやっていると主張するチームにとって、XPのプラクティスが必須なのかどうかです。これは一見したよりもずっと難しい問いです。XPは、どのアジャイル手法とも同じく、チームが自分たちのプロセスを選ぶことを期待します。Kent Beck は『Extreme Programming Explained』で、プラクティスとは「XPチームが日々やっているのを目にするようなこと」だと述べています。

私は、ペアプログラミングはXPチームでは普通に見られるものだと言うでしょう。しかし、ペアプログラミングをしていないチームはXPチームを名乗れない、とまでは言いません。私の知る多くのXPerにとって、あるチームがXPかどうかという問いは面白いものではない、という点も指摘しておきます。**本当の問題は、そのチームが効果的かどうかです**。

私がペアプログラミングを強く勧めるとしても、せいぜい、XPのやり方を学びたいならペアプログラミングを試し、それが自分たちに合うか見てみるべきだ、という程度です。

### 自分がペア作業を好きにならないことはわかっているので、試す必要はない

この発言の問題は、多くの人がペアプログラミングに驚かされてきたことです。嫌いになるだろうと思いながら試してみたところ、とても気に入った、という人たちがいます。

さらに話を複雑にするのは、多くの人がペア作業をまずい形で試してしまい、それが誤った印象を与えることです。隅のキューブで誰かの肩越しに何時間も受け身で眺めることは、ペアプログラミングではありません。自分が本物を評価しているのだと確信できるように、きちんとコーチできる人についてもらってください。

### ペアプログラミングは開発者の生産性を半分にする

これに対する私の軽い答えは、「プログラミングで一番難しい部分がタイピングなら、その通りでしょう」です。

ペアプログラミングの支持者が支持しているのは、ペアは実際には別々に働く2人の開発者よりも生産的だと考えているからです。これは、ペア作業が継続的な議論とレビューをもたらすためです。よりよい設計を思いつき、間違いを減らし、より多くの人がコードに詳しくなります。こうしたことがすべて、タイピングする人数が少なくなることを埋め合わせます。

もちろん、私たちは生産性を測れないので、確かなことはわかりません。私の考えでは、試してみたうえで、ペア作業がある場合とない場合のどちらで自分たちがより効果的だと感じるかをチームが振り返るべきです。新しいプラクティスでは何でもそうですが、Improvement Ravine を越えられるだけの十分な時間を取るようにしてください。

### 複雑なコードでなければペア作業をする価値はなく、決まりきったコードには利点がない

これには一理あると思います。ペア作業は、設計を改善し、間違いを最小化するためのものです。書くのが簡単な決まりきったコードには、ペア作業が違いを生む機会はほとんどありません。

ただし、例外があります。退屈で決まりきったコードを書いていることは、においです。退屈で繰り返しの多いコードを書いているなら、たいていそれは、書くべき定型コードの量を劇的に減らす重要な抽象化を見逃しているしるしです。**ペア作業は、その抽象化を見つける助けになります**。
