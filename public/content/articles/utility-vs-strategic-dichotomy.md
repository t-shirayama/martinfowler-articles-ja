# Utility Vs Strategic Dichotomy

## 要約

IT やソフトウェアは、すべてが同じ性質を持つわけではありません。給与計算のように「確実に動けばよい」utility なものもあれば、事業上の差別化を支える strategic なものもあります。

この記事は、両者を同じやり方で扱うと失敗しやすいと述べています。utility では低コストと破局の回避が重要であり、strategic では競合より早く動けることが重要です。また、この区別は Bimodal IT とは異なり、システムの層ではなく、その背後にある事業活動が差別化要因かどうかで考えるべきだと説明しています。

## 読むときの観点

- ソフトウェアを、技術の種類ではなく事業上の役割で分類している点に注目する。
- utility と strategic では、予算、チーム編成、リスク、スピードの意味が変わる。
- strategic と呼べるものは少数であり、多くを strategic 扱いすると焦点がぼやける。
- パッケージ導入とカスタム開発の判断を、差別化要因の有無から読む。
- Bimodal IT への批判は、速さと品質をトレードオフとみなす発想への批判として読む。

## 原文の翻訳

私のキャリアを通じて繰り返し目にしてきたテーマの一つに、ソフトウェア開発の性質と重要性があります。数年前、ある見込み顧客が私たちの営業担当者に「ソフトウェアは下水管のようなものだ。確実に動いてほしいし、細かいことは知りたくない」と言いました。これは Nicholas Carr が「IT Doesn't Matter」で語った種類の考え方です。一方で、私たちは多くの企業で、IT が事業にとってより明確な戦略的実現手段となり、新しい市場への参入や市場シェアの大幅な拡大を可能にする仕事もしてきました。では、IT は下水管のような utility なのでしょうか。それとも strategic asset なのでしょうか。

私の見方では、それはシステムによってどちらにもなりえます。utility な IT 取り組みの典型例は給与計算です。誰もが必要としますが、多くの人にとっては「ただ動いてほしい」ものです。

では、utility project と strategic project を分ける要因は何でしょうか。私の考えでは、それは根底にある事業機能が差別化要因かどうかに尽きます。その機能をどう行うかが、競合より優れている理由の重要な一部であるなら、その機能を支えるソフトウェアは可能な限り良いものである必要があります。Ross Pettit が言うように、「これは技術の性質によって IT を分けることではなく、その技術が受け入れ先の事業に何をするかによって分けること」です。

この二分法について最も重要なのは、ソフトウェアプロジェクトには二種類があり、**まったく違う扱いをしなければならない**と理解することです。strategic な取り組みに人を配置し、運営し、予算をつける方法は、utility project のやり方とはまったく異なります。あまりにも多くの場合、人々は一方に良いことはもう一方にも良いと仮定し、その結果として問題が必ず起きます。

もう一つの帰結は、strategic なプロジェクトは少数しかないということです。80/20 ルールが当てはまりますが、実際には 95/5 に近いかもしれません。この二分法そのものを認識しないことが最もよくありますが、同時に、あまりに多くのプロジェクトを strategic だと考えてしまうこともよくあります。

これらの取り組みが異なる最も重要な点の一つは、リスクがどこにあるかです。utility project で最大のリスクは、何らかの破局的な誤りです。下水管が壊れたり、給与支払いに失敗したりしてほしくはありません。そのため、それが起きないように十分な注意を払う必要がありますが、それ以外ではコストをできるだけ低くしたいものです。しかし strategic project で最大のリスクは、競合より先に何かをしないことです。そのため、素早く反応できる必要があります。何かをしないことによる機会費用は、ソフトウェア開発そのものの費用よりはるかに大きいので、コストはずっと小さな問題になります。

この二分法は固定的なものではありません。strategic だった事業活動が、時間がたつにつれて utility になることがあります。頻度は低いものの、企業がその活動を差別化要因にする方法を見つければ、utility が strategic になることもあります。Apple はパーソナルコンピュータのデザインで、これに近いことをしました。

この二分法が役立つ場面の一つは、カスタムソフトウェアを作るか、パッケージを導入するかを決めるときです。utility の定義は差別化要因がないことなので、明らかな選択はパッケージを使うことです。strategic function については、競合と同じソフトウェアを使いたくはありません。それは差別化する能力を損なうからです。

多くの場合、人々はこの点を理解して utility function のためにパッケージを買いますが、その後でそれをカスタマイズするために巨額の費用を使います。これも同じくらい無駄です。私の見方では、utility function ではパッケージを買い、そのソフトウェアに合わせて事業プロセスを調整します。通常これは政治的に実現しにくいので、回避策としては低いグレードのソフトウェアチームに取り組ませます。破局を避けるだけの注意は払いますが、それ以外では高グレードのチームは必要ありません。

この二分法が影響を及ぼすもう一つの場面は、agile methods の役割です。多くのアジャイル実践者は strategic な考え方から来ており、アジャイルを特徴づける柔軟性と市場投入までの速さは strategic project にとって重要です。しかし utility project では、アジャイルの利点はそれほど重要ではありません。utility function に agile approach を使うのが誤った選択かどうかは分かりませんが、それがそれほど重要ではないことは確かです。

多くの分類と同じように、その間には多くの灰色の領域があります。それでもこれは、コントラストを強めて、より二値的な考え方を強制する強い理由がある、まれなケースの一つだと思います。この投稿の草稿について議論したとき、Ross はこうコメントしました。「灰色の濃淡」は、物事を間違ったカテゴリに積み上げる口実を与える。本当は utility であるものが、実際の utility として扱われるのではなく、膨らんだ重要性を与えられてしまう。strategic の入れ物に入るものを最小にする方向へ傾けて、二値の判断を強制することは、IT の取り組みにしばしば欠けている焦点を与える助けになるでしょう。

Ross はさらに、utility work と strategic work の両方に責任を持つ単一の IT 部門は存在すべきではない、とまで主張します。両者に必要な mindset と management attitudes は、あまりにも違うからです。それは、倉庫を設計する人たちに美術館の設計も期待するようなものです。

### Bimodal IT

最近、一部のコンサルティング会社が Bimodal IT、あるいは Two Speed IT という考え方を広めています。一見すると、これは Utility/Strategic dichotomy と同じように見えますが、実際にはかなり違う考え方です。Bimodal IT は、間違った方向へ向かう道でもあります。

第一の違いは、Bimodal IT が根底にある事業活動ではなく、IT システムを層によって分離することに基づいている点です。Bimodal IT は、素早く変化する front-end systems of engagement と、遅いが信頼性の高い backend systems of record を分けます。しかし strategic business function で素早く革新しようとしているなら、通常は front-end と back-end の両方のシステムで素早い変更が必要になります。

Bimodal IT についての私の二つ目の不満は、この分離を推進する理由の一つに、素早く変化する systems of engagement には本質的に欠陥が多く、速度を得るためにその欠陥を許容する、という考え方があることです。しかし Thoughtworks や他の先進的なアジャイル組織での十年以上の経験は、この TradableQualityHypothesis が誤ったトレードオフであることを示しています。通常、短い cycle time を持つ agile approaches を導入すると、本番環境の欠陥も一桁減ることが分かります。実際、このように欠陥を減らさなければ、それほど素早く cycle することはできません。高品質、つまり低い欠陥数は、短い cycle-time を可能にする重要な要因なのです。

### さらに探るには

- Marc McNeil は、プロジェクトをトラクター、原子力発電所、そして最先端として語っています。
- Neal Ford は、SOA のような integration efforts は strategic にはなれないと指摘しています。
- Jez Humble は、bimodal IT model の三つの欠陥を説明しています。
