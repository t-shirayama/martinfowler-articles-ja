# The Learning Loop and LLMs

## 要約

ソフトウェア開発は、設計を済ませてから実装者が組み立てる「組立ライン」ではない。コードを書き、動かし、結果から学ぶことで設計そのものが育つ。LLM は環境構築や試行の敷居を下げ、意図をさまざまな記法へ翻訳する力を持つが、学習そのものを肩代わりするわけではない。

LLM がすぐに動くコードを出してくれるほど、開発者は理解を飛ばしてしまう危険がある。長く保守できるソフトウェアを作るには、観察し、試し、思い出して別の文脈に適用する学習ループを自分で回し続ける必要がある。

## 読むときの観点

この記事は、Unmesh Joshi による martinfowler.com の記事「The Learning Loop and LLMs」の非公式な日本語訳・要約です。原文は https://www.martinfowler.com/articles/llm-learning-loop.html です。

LLM を「コードを書く機械」として見るよりも、「学習と設計のループを加速する道具」として見ると、この記事の主張が読みやすくなります。特に、速く作れることと深く理解していることを混同しない、という点が中心です。

## 原文の翻訳

ソフトウェア開発は、それを組立ラインに変えられるという考えに、ずっと抵抗してきた。道具がより賢く、速く、高機能になっても、本質的な行為は変わらない。私たちは、やってみることで学ぶ。

### 組立ラインはソフトウェア開発の比喩として適切ではない

成熟した工学分野の多くでは、プロセスは明快だ。少数の専門家がシステムを設計し、専門性の低い作業者がその計画を実行する。設計と実装を分けられるのは、物理法則が安定して予測可能であり、構築のパターンを反復できるからだ。

ソフトウェアはそうではない。もちろん、自動化できる反復的な部分はある。しかし、実装の前に設計を完了できるという前提そのものが成り立たない。ソフトウェアでは、**設計は実装を通じて現れる**。

正しい設計を理解する前に、コードを書かなければならないことがよくある。コードから得られるフィードバックこそが、私たちの主要な道しるべだ。その多くは、孤立して行えるものではない。ソフトウェア作りには、開発者、プロダクトオーナー、利用者、その他のステークホルダーの絶え間ない相互作用が含まれる。それぞれが固有の洞察を持ち寄る。プロセスはこの動的な性質を反映しなければならない。コードを書く人は単なる「実装者」ではない。正しい設計を発見する中心にいる。

### LLM は組立ラインの比喩を再び持ち込んでいる

アジャイルの実践は、20年以上前にこのことを認識した。アジャイルから学んだことを忘れてはならない。今日、大規模言語モデル、つまり LLM の台頭によって、設計構造をよく考え終えたあとに、コード生成を孤立した作業として行う誘惑が再び生まれている。しかし、その見方はソフトウェア開発の本質を見落としている。

### LLM をブレインストーミングの相手として慎重に使うことを学んだ

私は最近、自分の本で説明しているパターンに基づいて、分散システムを構築するためのフレームワークを開発した。その過程で LLM をかなり試した。LLM は、ブレインストーミング、命名、ボイラープレート生成に役立った。しかし同じくらい頻繁に、微妙に間違っていたり、より深い意図とずれていたりするコードも生成した。大きな部分を捨てて最初からやり直さなければならないこともあった。

最終的に私は、LLM をより慎重に使うことを学んだ。自律的な構築者としてではなく、アイデアを出すためのブレインストーミングの相手として使う、ということだ。その経験を通じて、ソフトウェア開発の性質について考えるようになった。特に重要なのは、ソフトウェアを書くことは根本的に学習の行為であり、LLM エージェントが手元にあるからといって、学習する必要から逃れられるわけではない、という点である。

### LLM は実験の敷居を下げる

意味のある作業を始める前には、重要な一歩がある。動き出すための準備だ。環境を整えること、依存関係をインストールすること、適切なコンパイラやインタプリタを選ぶこと、バージョンの不一致を解決すること、実行時ライブラリをつなぎ込むことは、ときに最も苛立たしく、同時に必要不可欠な最初の障壁になる。

「Hello, World」プログラムが伝説的であることには理由がある。それは単なる伝統ではない。想像が実行と出会う瞬間を示している。

最初に出力が成功すると、ループが閉じる。道具が整い、システムが反応し、コードを通じて考えられるようになる。この準備段階で、LLM はもっとも輝く。初期のビルドファイルを下書きする、適切なフラグを見つける、依存関係のバージョンを提案する、プロジェクトを立ち上げる小さなスニペットを生成するなど、最初の摩擦を乗り越える助けとして非常に有用だ。LLM は出発線の摩擦を取り除き、実験の敷居を下げる。

しかし、「Hello, World」のコードがコンパイルされて動いたあと、本当の仕事が始まる。

### 私たちの仕事には根本的な学習ループがある

私たちが行う仕事の性質を考えると、継続的な学習が仕事を動かすエンジンであることは明らかだ。単純なテキストエディタから最先端の AI まで、どのような道具が手元にあっても、深く長続きする知識を築く道筋は、飛ばすことのできない基本的で手を動かすパターンに従う。このプロセスは、単純だが強力なサイクルに分けられる。

### 観察し、理解する

これが出発点だ。チュートリアルを見る、ドキュメントを読む、既存のコードを調べるといった形で、新しい情報を取り入れる。何かがどのように動くはずかについて、基本的なメンタルマップを作っている段階である。

### 実験し、試す

次に、受け身の観察から能動的な参加へ移らなければならない。新しいプログラミング技法について読むだけではない。自分でコードを書く。変えてみる。壊そうとしてみる。そして何が起きるかを見る。これは、抽象的な考えが自分の頭の中で現実的で具体的に感じられ始める、重要な「手を動かす」段階だ。

### 思い出し、適用する

これがもっとも重要な段階であり、本当の学習が証明される場所だ。新しい課題に直面し、以前に学んだことを能動的に思い出し、別の文脈に適用しなければならない瞬間である。「前に似た問題を見たことがある。ここではあの解法を使える」と考える場所だ。知識を取り出して使うこの行為が、断片的な情報を持続する技能へ変える。

### AI は学習を自動化できない

だからこそ、道具はあなたの代わりに学習できない。AI は完璧な解を数秒で生成できるかもしれない。しかし、自分で作る苦労から得られる経験を与えることはできない。小さな失敗や「ああ、そうか」という瞬間は、学習の不可欠な特徴であり、自動化によって取り除くべきバグではない。

学習に近道はない。

### 誰もが固有の方法で学習サイクルを進む

この学習サイクルは、人によって異なる。何かを試し、何がうまくいくかを見て、フィードバックに基づいて調整する継続的なループである。ある方法は自分にしっくりくるし、別の方法はそうではない。真の専門性は、この絶え間ない適応を通じて自分に合うものを見つけることで築かれ、技能を本当に自分のものにする。

### アジャイル手法は学習の重要性を理解している

学習のこの根本的な性質と、私たちの仕事における重要性こそが、効果的なソフトウェア開発手法が現在の姿に進化してきた理由である。イテレーション、ペアプログラミング、スタンドアップミーティング、レトロスペクティブ、TDD、継続的インテグレーション、継続的デリバリー、DevOps について語るのは、私たちがアジャイル陣営にいるからだけではない。これらの技法が、学習の根本的な性質と、それが仕事の中で持つ重要性を認めているからだ。

### 学習の必要性が、高水準のコード再利用を難しくしてきた

逆に、職業上の仕事における継続的学習の役割は、ソフトウェア開発における長年の課題の一つを説明する。高水準のコード再利用が限定的にしか成功してこなかったことだ。文脈に応じた学習が根本的に必要であるため、長く追い求められてきた高水準のコード「再利用」はつかみにくい目標のままだった。その成功は、データ構造や Web クライアントのように、明確に定義された普遍的な問題を解く技術ライブラリやフレームワークにおおむね限られている。

その水準を超えると、再利用はうまくいかなくなる。ほとんどのソフトウェア上の課題は、学び取り内面化しなければならない固有のビジネス文脈に深く埋め込まれているからだ。

### ローコードは速度を与えるが、学習なしではその速度は続かない

ここで、「スターターキット」や「ローコードプラットフォーム」がもたらす速度の幻想に行き着く。これらは標準的なユースケースに対して強力な初速を与えるが、その速度には代償がある。私たちが使う既製のコンポーネントは、本質的には文脈を圧縮した束である。無数の設計判断、トレードオフ、教訓がその中に隠れている。

それらを使うことで、私たちは機能を得るが、学習は得ない。採用した複雑な仕組みについて、内面化された知識はゼロのままになる。その結果、作業を終えるために費やす時間が急増し、生産性が急落することがある。

### 小さな変更に見えるものが、時間を食い尽くす穴になる

これは、飽和状態にあるソフトウェアシステムの性能グラフにとても似ている。そこでは「膝」が現れ、それを超えるとレイテンシが指数関数的に増え、スループットが急激に落ちる。要求が既製の解決策からほんの少しでも外れた瞬間、最初の高速化は消え去る。コンポーネントがどう動くかについて深い文脈を持たない開発者は、ブラックボックスに向き合うことになる。

小さな変更に見えるものが行き止まりになったり、時間を食い尽くす穴になったりして、最初の数日で節約したはずの時間をすぐに飲み込んでしまう。

### LLM は一時的な速度を増幅し、専門性の発達を損なう

大規模言語モデルは、この動きを何倍にも増幅する。私たちは今、速度が二桁向上し、コストが下がるという急進的な生産性向上の主張に囲まれている。しかし、私たちの仕事の基礎にある性質を認めなければ、これらの指標は罠になる。

真の専門性は、深い文脈を築くために知識を学び、適用することで作られる。この道のりなしに既製の解を提供する道具には、隠れた危険がある。一見完璧なコードを電光石火で提供することで、LLM は保守の崖の究極の形を示す。長期的に堅牢で保守可能なシステムを築くために不可欠な学習を迂回する、魅力的な近道である。

### LLM はすべての道具への自然言語インターフェイスを提供する

では、なぜ LLM にこれほど大きな期待が寄せられるのだろうか。大規模言語モデルのもっとも注目すべき強みの一つは、ソフトウェア開発にある多くの言語を橋渡しする能力だ。仕事のそれぞれの部分には固有の方言が必要になる。ビルドファイルには Gradle や Maven の構文があり、`vmstat` や `iostat` のような Linux 性能ツールには構造化された出力があり、SVG グラフィックは XML ベースのマークアップに従う。さらに Python、Java、JavaScript などの汎用言語も数多くある。

ここに、独自の API、DSL、設定ファイルを持つ無数のツールやフレームワークが加わる。LLM は、人間の意図とこれらの専門化された言語のあいだの翻訳者として働ける。「2本の曲線の SVG を作って」「複数モジュール用の Gradle ビルドファイルを書いて」「この `vmstat` の出力から CPU 使用率を説明して」といったことを平易な英語で述べると、適切な構文のコードを数秒で生成できる。これは非常に大きな能力だ。

それは参入障壁を下げ、摩擦を取り除き、これまでになく速く始める助けになる。しかし、この翻訳の流暢さは学習と同じではない。自然言語で意図を表現し、動くコードを受け取る能力は、それぞれの言語の設計、制約、トレードオフを学ぶことから得られる深い理解を置き換えない。これらの専門化された記法には、数十年にわたる工学上の知恵が埋め込まれている。

それらを学ぶことで、私たちは変更について推論できるようになる。自信を持ってシステムを修正し、拡張し、進化させられる。LLM は探索を滑らかにするが、成熟はより深い理解から生まれる。

### LLM による意図からコードへの流暢な翻訳は、学習と同じではない

大規模言語モデルは大きなレバレッジを与えてくれる。しかし、それが機能するのは、私たちが学習と理解に焦点を当てる場合だけだ。LLM は、アイデアを探索し、準備を整え、多くの専門化された言語をまたいで意図をコードに翻訳することを容易にする。

しかし本当の能力、つまり変化に応答する能力は、どれだけ速くコードを生成できるかからではなく、形作っているシステムをどれだけ深く理解しているかから生まれる。道具はますます賢くなる。だが、**学習ループの性質は変わらない**。

長く残るソフトウェアを作り続けるには、学習の性質を認めなければならない。それを忘れれば、私たちはいつも保守の崖に立つことになる。
