# Code Smell

## 要約

Code Smellは、システムの深い問題に対応していることが多い、表面上の兆候を指します。匂いはそれ自体が必ず悪いわけではなく、調べるべき場所を知らせる合図です。

長すぎるメソッドやデータだけを持つクラスのような匂いは、経験の浅い人にも見つけやすいという利点があります。チームで少しずつ匂いを探し、上級者と話しながらリファクタリングを学ぶ入口にもなります。

## 読むときの観点

- smellは問題そのものではなく、問題の可能性を示す兆候として読む。
- すぐ見つかることと、深く調べる必要があることの両方を意識する。
- Data Classのような匂いを、振る舞いを移すきっかけとして見る。
- チーム教育のために、匂いをひとつずつ扱う方法に注目する。

## 原文の翻訳

著作権上の理由により、このファイルには原文本文の全文翻訳を収録していません。以下は、記事の主張を日本語で読むための要旨です。

Code Smellとは、システムの中により深い問題があるかもしれないことを示す、表面上の兆候です。この言葉は、Fowlerの『Refactoring』をKent Beckが手伝っていたときに生まれました。

この定義には微妙な点があります。第一に、smellは素早く見つけられるものです。長いメソッドは典型例で、コードを眺めただけで違和感を覚えます。第二に、smellは必ずしも問題を意味しません。長いメソッドでも問題がない場合はあります。だからこそ、smellを見つけたら、そこに本当に問題があるかをさらに調べる必要があります。

よいsmellは、見つけやすく、しかも多くの場合に興味深い問題へ導いてくれるものです。Data Class、つまりデータだけを持ち振る舞いを持たないクラスは、その好例です。そのクラスに本来どんな振る舞いがあるべきかを考え、リファクタリングによって振る舞いを移していくきっかけになります。

smellのよいところは、経験の浅い人でも見つけやすい点です。問題の本質を評価したり修正したりする力がまだなくても、兆候を見つけてチームの上級者に相談できます。リード開発者が「今週のsmell」を決め、チームで探して話し合うやり方は、プログラマとしての判断力を少しずつ育てる方法になります。

Code Smellは、コードを責めるための言葉ではありません。**調べる価値のある場所を知らせる合図**として扱うと、リファクタリングと学習の入口になります。
