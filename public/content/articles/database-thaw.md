# Database Thaw

## 要約

Database Thawは、リレーショナルデータベースが長く当然視されてきた状況に、非リレーショナルな選択肢が再び現れ始めたことを「解凍」として捉える記事です。QConでの講演や代替データベースの話題をきっかけに、Drizzle、CouchDB、Gemstone、Neo4Jなどが取り上げられています。

Fowlerは、リレーショナルデータベースの支配はデータ管理そのものよりも、統合の役割によって強められてきたと見ます。統合の中心がSQLや共有データベースからHTTPやサービスへ移るなら、アプリケーションは自分の用途に合ったData Modelを選びやすくなります。

## 読むときの観点

- リレーショナルデータベースの支配を、技術的優位だけでなく統合方式として見る。
- Shared Database Integrationが、SQLの重要性を押し上げたという説明に注目する。
- HTTPやWeb Servicesによる統合が、Application Databaseの選択肢を広げる点を見る。
- 非リレーショナル技術の台頭を、RDBMSの消滅ではなく選択肢の増加として読む。

## 原文の翻訳

数年前、プログラミング言語の世界の人々が、Javaによって起きた言語の「Nuclear Winter」について話しているのを聞きました。誰もがJavaの計算モデルに収束してしまい、当時のC#はほとんど模倣以上のものとは見られていなかったため、プログラミング言語における創造性が消えてしまった、という感覚でした。その感覚は今では薄れつつあります。しかし、もっと重要な雪解けが始まっているのかもしれません。データベースについての考え方をめぐる、より長く深い凍結の雪解けです。

私がソフトウェア開発の仕事を始めたころ、リレーショナルデータベースを広めていた人たちと一緒に働いていました。私はオブジェクト指向の世界で彼らに出会いました。当時、多くの人はOO databaseがデータベースの次の進化段階になると考えていました。今では分かるように、そうはなりませんでした。現在ではリレーショナルデータベースが非常に深く根づいており、ほとんどのプロジェクトは最初からRDBMSを前提にします。

先週のQConでは、この前提に疑問を投げかける講演の流れが強くありました。特に印象に残ったのはTim Brayの基調講演で、彼はいくつものデータ管理の側面をたどり、その中で興味深いプロジェクトをいくつか取り上げました。

- Drizzleはリレーショナルデータベースの一種ですが、現代のリレーショナル製品が持つ多くの仕組みを避けています。私はこれをRISC RDBMSのようなものだと考えています。リレーショナル機能セットの最小限だけを支えるものです。
- CouchDBは、分散key-value pairモデルへの多くの試みのひとつです。非常に単純なデータモデルで、実際にはhashmap以上のものではありませんが、この種のアプローチは高トラフィックのWebサイトでかなり人気を集めています。
- Gemstoneはobject database勢のひとつでした。GemstoneとSmalltalkの組み合わせは、私には非常に強力な開発環境に見えました。多くの後継より優れていました。Gemstoneは今もニッチな存在として残っていますが、Maglevによってさらに足場を得るかもしれません。Maglevは、そのアプローチ、つまりデータベースと仮想マシンの融合に近いものをRubyの世界へ持ち込むプロジェクトです。

この講演に加えて、Kresten Krab Thorupが主催する代替データベースのトラック全体もありました。そこで触れられた追加のツールのひとつがNeo4Jです。graph、あるいはnetwork databaseのツールで、Jim Webberから珍しいほどの称賛を得ていました。

こうした製品について自然に湧く問いは、なぜODBMSが失敗したのに、これらは成功しうるのかということです。リレーショナルの支配を解かすほど、環境の何が変わったのでしょうか。リレーショナルがこれほど支配的だった理由については多くの仮説があります。私の考えでは、その支配はデータ管理における役割というより、統合における役割によるところが大きいです。

今日の多くの組織では、統合の主要なパターンはShared Database Integrationです。複数のアプリケーションが共通データベースを使うことで統合されます。こうしたIntegration Databaseがある場合、すべてのアプリケーションが共有データへ容易にアクセスできることが重要です。そのため、SQLの役割が非常に重要になります。**ほぼ標準の問い合わせ言語としてのSQLの役割が、リレーショナルデータベースの支配の中心にありました。**

データベース領域が温まり始めているのは、統合に対する代替手段が存在するようになったからです。特にWeb Servicesの台頭があります。さまざまな旗印のもとで、アプリケーション同士がHTTP上でテキスト、多くはXMLのdocumentを受け渡すことで会話する流れが広がっています。Webは、インターネットでもイントラネットでも、この統合方式をSQL以上に広く行き渡らせました。これはよいことです。私は、複数のアプリケーションが共通データベースを通じて密結合する方法を好んだことがありません。これ以上大きなカプセル化の破れはありません。

統合プロトコルをSQLからHTTPに切り替えると、データベースをIntegration DatabaseからApplication Databaseへ変えることができます。この変化は大きなものです。第一段階として、Ruby on Railsが採ったような、より単純なobject-relational mappingのアプローチを支えます。さらに、リレーショナルデータモデルの万力のような支配を壊します。HTTPで統合するなら、アプリケーションが自分のデータをどのように保存しているかはもはや重要ではありません。その結果、アプリケーションは自分の必要に合ったData Modelを選べるようになります。

これは、リレーショナルデータベースが消えるという意味ではないと思います。結局のところ、リレーショナルデータベースが適切な選択である状況は多くあります。しかし、アプリケーション開発者が、自分たちの要求にとって正しい選択肢は何かを考えるべき時代になったということです。非リレーショナルのプロジェクトが人気と成熟度を増すにつれて、ますます多くの人が別の選択肢を選ぶようになるでしょう。
