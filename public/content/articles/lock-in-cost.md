# Lock In Cost

## 要約

Lock-in Costは、単に移行コストだけで測るべきものではありません。特定ベンダーのサービスを使うことで得られる市場投入の速さや機会獲得も考えると、ロックインの判断は「移行コスト - 機会獲得」として見直せます。

技術的な移植性は望ましいものの、ビジネス上の価値を無視して完全なポータビリティだけを追うと、かえって機会を逃すことがあります。重要なのは、移行コストだけを見るのではなく、システムを適応しやすくして移行コストを下げることです。

## 読むときの観点

- ロックインを技術問題だけでなく、ビジネス判断として見る。
- 移行コストと機会獲得の両方を式に入れる。
- 完全なベンダー非依存を目指すコストと、市場投入の速さを比べる。
- クラウド利用では、抽象化しすぎた汎用的な使い方の限界も考える。

## 原文の翻訳

最近のクライアントとの仕事で、私はserverless architectureがぴったり合うと予見していました。しかし、serverless architectureを採用するという考えは、vendor lock-inへの恐れから、クライアントにはうまく受け入れられませんでした。小売業者にとっては興味深い時期でした。AWSに留まることは、別の小売業でもあるAmazonに競争上の優位を与えることを意味するかもしれないからです。競合を支援したくないという考えから、私のクライアントは、私たちが選ぶソリューションが他のクラウドベンダーへ完全に移植可能であることを確かめたいと考えていました。

技術的な観点から見れば、システムをあるプラットフォームから別のプラットフォームへ移せる能力を持つことは、間違いなく望ましいことです。コンテナ化が登場した今、特定のプラットフォームに縛られたいと思う人がいるでしょうか。別のプラットフォームへ移ると決めたときに、高いロックインコストをビジネス側へ示したいとは思いません。したがって、このシナリオが起きたときの移行コストをできるだけ低くする必要があります。現在の理解でロックインコストを単純な式にするなら、次のようになります。

```text
Lock-in cost = Migration cost (?)
```

この式は、技術的な観点だけから見ているなら正しいものです。しかし、ビジネスの観点を見落としてはいけません。私たちが届ける技術的ソリューションは、常にビジネス上の問題を解決するために設計されます。多くの場合、特定の技術を採用するとビジネスは利益を得ます。重要な利益の1つは、市場投入までの時間を短くできることです。市場投入の速さは、機会獲得として式にできます。

```text
Lock-in cost = Migration cost - Opportunity gain
```

Opportunity gainを測ることは、とても困難です。未知の未知を扱うことになるからです。Migration costは分析し、筋道立てて考えられます。対照的に、Opportunity gainはそれほど簡単には分析できません。あるプラットフォームから別のプラットフォームへどう移行するかは理論化し分析できます。しかし、競合の市場機会をつかむことで得られる利益をどう計算するのでしょうか。技術とビジネスの両方の観点を組み合わせ、意思決定プロセスを全体として見ることで、あなたが行うロックインの判断は利益につながるかもしれません。

event-driven architectureを構築する例を見てみましょう。そのアーキテクチャでは、分散メッセージングシステムを選ぶ必要があります。すでにAWSをプラットフォームとして選んでいるなら、Kinesisのようなベンダー固有サービスを選択できます。これらのサービスは完全にマネージドで、すぐに動かせるため、機会獲得をもたらします。Kafkaのようなベンダー非依存のシステムと比べると、こうしたベンダー固有サービスは、より高い移行コストを発生させます。しかし、自前で分散メッセージングシステムを立ち上げるには、特にそのようなプラットフォームを構築した経験がまだない場合、堅牢化し本番対応にするまでにより多くの時間がかかります。意思決定を移行コストだけから見るのではなく、システムをより適応しやすくすることで、**移行コストをどう下げられるか**に注目してください。特にこのクラウド利用の例では、generic cloud usageという慣行を避けるよう勧める理由と似ています。
