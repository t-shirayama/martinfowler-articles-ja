# Business Facing Test

## 要約

Business Facing Test は、顧客、ユーザー、ビジネスアナリストなど、開発チームの非プログラマーとのコミュニケーションを助けるためのテストです。自動化される場合は、システム内部のコンポーネント構成ではなく、ドメインに沿った言葉でシステムの振る舞いを表します。

受け入れ条件として使われることが多く、テストが通ることは、顧客が期待する機能をシステムが提供していることを示します。一方で、ビジネス向けのテストには探索的テストやユーザビリティテストのような手動テストも重要な役割を持ちます。

## 読むときの観点

- テストを、単なる検証手段ではなく、ビジネス側との会話を支える表現として見る。
- Broad Stack Test として実装されがちな理由と、Component Test として実装する利点を分けて考える。
- 自動テストだけでなく、探索的テストやユーザビリティテストもテストポートフォリオに含めて捉える。

## 原文の翻訳

Business Facing Test とは、顧客、ユーザー、ビジネスアナリストなど、開発チームのうちプログラミングをしないメンバーとのコミュニケーションを助けるために使われるテストである。

自動化された場合、それらのテストはシステムそのもののコンポーネントアーキテクチャを無視し、ドメインに沿った言葉でシステムを記述する。Business Facing Test は受け入れ条件として使われることが多く、そうしたテストが通ることは、**顧客が期待する機能をシステムが提供している**ことを示す。

自動化された Business Facing Test は、何らかの形の Domain Specific Language で表されることが多い。これは、プログラマーではない人とのコミュニケーションを助けると同時に、プログラマーがコードの細部から一歩離れて考えるための仕組みにもなる。Cucumber や Twist のようなツールは、こうした DSL の設計を助け、それをテスト対象システムに結びつける仕組みを提供する。

Business Facing Test は、ユーザー視点の表現で書かれるため、テスト対象システムをブラックボックスとして扱う発想になりやすく、Broad Stack Test として実装されることが多い。しかし、Business Facing Test を Component Test として実装することには大きな利点がある。そうすることで、保守しやすく、実行も速くなることが多い。

私は自動テストの大きな支持者だが、Business Facing Test において手動テストが重要な役割を果たすことも認識しておく必要がある。探索的テストやユーザビリティテストのような技法は、本質的に手動の活動であり、**バランスの取れたテストポートフォリオ**に欠かせない部分である。

Story Test と User Journey Test は、Business Facing Test の一般的な二つの形である。Business Facing Test という用語は、Brian Marick のテスト四象限に由来する。
