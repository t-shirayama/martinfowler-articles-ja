# Data Models

## 要約

Data Modelの議論は、リレーショナルモデルが勝利したかのように見られがちですが、Fowlerはそれほど単純ではないと述べます。データベースの世界ではSQLに近いリレーショナルモデルが支配的である一方、メモリ上のデータ構造ではnetwork modelが広く使われています。

XMLの普及やメモリ価格の低下によって、階層型モデルやインメモリ中心のデータベースも含め、どのData Modelがどの種類のデータに合うのかを改めて考える価値がある、という短い問題提起です。

## 読むときの観点

- Data Modelを、データベース製品ではなくデータの考え方として捉える。
- disk上のrelational modelと、memory上のnetwork modelの違いを見る。
- XMLがhierarchic data modelを再び目立たせた背景を考える。
- データの種類によって適したモデルが異なる、という結論に注目する。

## 原文の翻訳

私が初期に気に入っていた本のひとつに、TsichritzisとLochovskyによるData Modelsの本があります。その本では、データについて考えるためのさまざまなモデルが論じられていました。特に当時よく議論されていた3つのモデル、Relational Data Model、Hierarchic Data Model、Network Data Modelが取り上げられていました。

最近では、異なるData Modelの長所と短所について人々があまり話していないように思います。Data Modelについて語るのはデータベース関係者であることが多く、たいていの人は、その競争はとうの昔にリレーショナルモデルが勝ったと考えているようです。

しかし私は、それが本当に正しかったとは思いません。そして状況はますます面白くなっています。

データベースの世界でリレーショナルモデルが支配的であることは疑いありません。最近出会う業務アプリケーションのほとんどはSQLデータベースを使っています。一般の人々にとっては、それでリレーショナルと十分に近いと言えます。しかし、インメモリのデータ構造を見ると別の世界が見えてきます。そこではnetwork modelが支配的です。実際、人々はdisk上のrelational modelをmemory上のnetwork modelへ変換するために、しばしば多くの労力を払っています。これがAnemic Domain Modelが広く使われる理由のひとつだと私は考えています。

これは興味深いことです。私は何度も、なぜわざわざ関係をrecordへ変換するために苦労するのか、と人々に尋ねてきました。返ってくる答えは、ほとんどいつも同じところに行き着きます。**多くの開発者にとって、network modelはrelational modelより扱いやすい**ということです。もちろん常にそうだとは限りませんが、recordを好む人が多数派だとは思います。

これは、SQLがデータベースではうまく機能する一方で、インメモリ処理には同等のものがないことと関係しているのかもしれません。ADO.NETで興味深いと感じる点のひとつはrelationalに考えることですが、ここでも多くの人がdatasetをrelationalなスタイルで扱いたがらないのを見ます。

現在のData Modelの状況に着実に挑戦しているもうひとつの要素は、XMLの台頭です。XPathやXQueryのようなXMLベースの技術は、hierarchic data structureへアクセスする標準的な方法を提供します。多くの点で、XMLがそのデータの標準的なテキスト表現を提供することは、このhierarchic dataを問い合わせ操作する標準的な方法があることに比べれば、おまけのようなものです。

さらに根本的な技術変化が、Data Modelの揺り戻しを生むと思います。メモリ容量が増え、価格が下がるにつれて、ほとんどのデータベースを完全にメモリ上に置ける地点へ近づいています。そこに変更を永続化する仕組みを組み合わせれば、性能についての前提が根本的に異なる、まったく別種のデータベースが生まれます。Prevaylerはこの考え方の例です。彼らの性能数値がどの程度妥当なのかは分かりませんが、たとえ2桁ほど外れていたとしても、なお印象的です。

どのData Modelが意味を持つのかについての前提を払い落とし、こうしたモデルの基礎について再び考え始める時期なのかもしれません。私の感覚では、異なる種類のデータには異なる種類のモデルがよく合います。relational modelは表形式のデータには完璧ですが、『テンペスト』を保存したい場合にはひどく向きません。だからこそ、異なるData Model、それを使う技術、そしてどのモデルがどの種類のデータに向いているのかを知っておくことには意味があります。
