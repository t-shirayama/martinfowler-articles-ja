# What Is Code?

## 要約

LLMによってコード生成が安く速くなると、コードの価値は「機械への命令を書くこと」だけでは説明できなくなります。この記事は、コードには機械への命令である側面と、問題領域を理解するための概念モデルである側面があると説明します。

重要なのは、ドメインの語彙を技術的な語彙に写し取り、名前、境界、関係、制約としてコードに刻むことです。LLMは既存の語彙やパターンを扱うのが得意ですが、チームが意味を理解しないまま語彙だけを増やすと cognitive debt が蓄積します。

## 読むときの観点

- コードを「生成物」ではなく、ドメイン理解を共有するための概念モデルとして読む。
- 良い名前や抽象化は、最初から与えられるものではなく、反復とリファクタリングの中で発見されるものとして捉える。
- LLMとの協働では、プロンプトだけでなく、コードベースに埋め込まれた語彙そのものが重要な文脈になる。
- cognitive debt は、生成されたコードの量ではなく、導入された概念をチームが理解していないことから生じる。

## 原文の翻訳

コードとは何でしょうか。高いレベルでは、この問いへの答えは明らかに見えます。コードとは開発者が書くものであり、機械に何をすべきかを伝える、プログラミング言語で表現された命令です。長年、コードを書くとは、一語ずつ入力していくことを意味していました。進歩は、コードをどれだけ効率よく作り、コンパイルし、テストし、デプロイできるかで測られてきました。

現代のLLMによって、私たちはコードを作るためにすべての語を入力する必要がなくなりました。大量の実行可能なコードを、高レベルな説明から生成できるようになっています。これは、より深い問いを突きつけます。コードを作ることが安くなるなら、コードの何に価値が残るのでしょうか。

### コードの2つの側面

コードは常に、別々でありながら絡み合った2つの目的を果たしてきました。

第一に、コードは機械への命令の集まりです。計算を指示し、データを動かし、ストレージとやり取りし、実行を調整します。LLMの時代にコモディティ化されつつあるのは、この部分です。

第二に、コードは問題領域の概念モデルです。これは「設計」の側面です。よく設計されたコードベースは、機械への命令を含むだけではありません。人間やツールが推論するための概念も含んでいます。

私たちがコーディングと呼ぶ活動は、この2つの側面が出会う場所です。私たちは、システムを理解するための概念、名前、境界、関係を形作っているのです。

### 概念モデルと語彙

概念モデルを明示することは、コーディングのより深い側面です。それは、システムが扱うべきドメインとユースケースによって駆動されます。どのドメインにも、確立されたプロセスや実践があり、さらに重要なものとして共有された語彙があります。概念モデルが見える形になるのは、その語彙を通じてです。

語彙は通常、特定の言語や主題で使われる言葉の集合として理解されます。私がこの記事を書けるのは、英語の語彙を知っているからです。読者がこの記事を読めるのは、私とその語彙を共有しているからです。

しかし、この記事を理解するには、英語を知っているだけでは足りません。この記事はソフトウェア開発についてのものです。ソフトウェア開発は広く成熟した分野であり、独自の技術語彙を持っています。私が abstraction という言葉を使うとき、単に英語の単語を使っているのではありません。固有の意味、歴史、含意を持つソフトウェア開発の概念を指しています。ソフトウェア開発に詳しくない読者は、その単語を表面的には理解できるかもしれませんが、この文脈でのより深い意味を見落とすでしょう。こうした、確立された語彙を持つ成熟した領域をドメインと呼びます。

これは、すべての本格的なドメインに当てはまります。コミュニケーションは共有語彙に依存しています。相手が人であれ、フレームワークであれ、LLMであれ、私たちが使う言葉は、受け手が理解し行動に移せる概念に対応していなければなりません。

### コードの中の語彙

よく設計されたコードベースは、ある語彙の表現です。では、コードの中の語彙はどこから来るのでしょうか。ここに、ソフトウェア開発の独自性がよく表れます。ソフトウェア開発は、さまざまなドメインの交差点で行われます。一方には銀行、金融、小売、在庫、医療、保険などのドメインがあります。もう一方には、Web、インフラ、AI、データエンジニアリングなどのドメインがあります。

Web開発をする人は、Webアーキテクチャ、Webメソッドの意味、GETが持つ普遍的なキャッシュ可能性、その意味論がもたらす含意をよく理解している必要があります。それを知らない人は、複雑なシステムをうまく設計できません。他のドメインでも同じです。**語彙は単なるラベルの集まりではなく、意味、制約、設計上の帰結を運びます。**

小売ドメインを考えてみましょう。小売のためのコードを書くとき、私たちは顧客、商品、注文、出荷、支払いなどについて話します。小売ドメインでWeb開発をするとき、コードには小売ドメインをWebドメインへ対応づける概念が含まれます。たとえば Catalog はリソースであり、それに対して GET、POST、PUT、DELETE といったHTTPメソッドで操作できます。コードを書く人は、その両方の語彙に通じている必要があります。

あるドメインのためにコーディングすることは、根本的には翻訳の行為です。開発者は、ドメイン語彙を技術ドメインの語彙へ写し取ります。その過程で、プログラミング言語が提供する構成要素を使って、新しい語彙も作られます。ログ、リポジトリ、クォーラム、トランザクションといった概念があり、money のような固有の概念もあります。概念は型になり、関係はインターフェイスになり、ルールは不変条件になり、ワークフローは合成になります。

変数の正確な名前、メソッドの境界、クラスの階層は、少しずつ発見されます。正しい抽象化は、最初から明らかでないことがよくあります。現実の制約に照らしてコードを絶えず形作り、リファクタリングしていく中で、ようやく姿を現します。うまく使われると、コードは少しずつ、ドメインそのものを読みやすく、非常に具体的に表現したものになります。

技術ドメインでは、基礎となる実装パターンを提供するフレームワークやライブラリがよく見つかります。フレームワークやライブラリは、コード化された語彙です。もっとも一般的な利用パターンを捉えています。Web、統合、関連する関心事を含むエンタープライズアプリケーションを構築するために Spring Framework のようなエコシステムが存在するのはそのためです。異なるプログラミング言語はそれぞれの味わいを持ち、固有の設計制約を伴います。それらはフレームワークやライブラリにも反映され、コード化されます。

### Bounded Context とローカルな語彙

ドメインに安定した反復構造があり、広く共有された意味論がある場合、フレームワークはうまく機能します。しかし「オンライン小売」や「証券取引所」のようなものは違います。それらは単なる技術スタックではありません。普遍的な高レベルフレームワークが存在しない主な理由は、その語彙がドメインのすべての実例で十分に安定していないからです。普遍的な抽象化を見つけようとすると、役に立たないほど汎用的になるか、広く適用するには偏りすぎたものになります。

中核となるビジネスモデルに近づくほど、抽象化はローカルに発見されなければなりません。これが、Domain-Driven Design における bounded context の考え方がとても重要な理由です。bounded context は、特定の語彙とモデルが有効である境界を示します。同じ言葉が異なる文脈では違う意味を持つことがあり、それぞれの文脈には独自の抽象化、ルール、言語が必要です。

では、こうしたローカルな抽象化と語彙をどう作るのでしょうか。その多くは、コードを書き、それを振り返る反復的なセッションを通じて作られます。TDDのような技法は、語彙を反復的に発展させるうえで優れています。モデルと振る舞いの間に継続的なフィードバックを強いることで、適切な名前、抽象化、境界を発見する助けになります。

コーディングは孤立しては起こりません。ドメイン専門家、ユーザー、開発者の密接な協働が必要です。この協働は、ローカルな抽象化と語彙を築くために欠かせません。

これは、アジャイルソフトウェア開発の教訓に直接つながります。個人と対話、顧客との協調、動くソフトウェア、変化への対応を重視することは、単なるプロセス上の助言ではありません。フィードバックを通じて語彙を発見し、洗練していく方法なのです。

Domain-Driven Design は、ubiquitous language という考え方でこれをより明示します。これは、開発者とドメイン専門家によって発展させられ、動くソフトウェアに対して継続的に検証される共有言語です。

### 思考の道具としてのプログラミング言語

コードを通じて語彙を築くには、生成されたコードを受け身でレビューするだけではなく、コードを書き直し、形を変えることに能動的に関わる必要があります。コードについて深く考える行為そのものが、コードを書くことに能動的に関わっているときにだけ起こることがよくあります。

プログラミング言語、その構成要素、制約は、それ自体が思考の道具になります。異なるプログラミング言語が提供する設計制約は、私たちの考え方を形作ります。Goのチャネルと軽量スレッド、Javaのオブジェクト指向モデル、Rustの所有権モデルは、それぞれ特定の形で構造、境界、トレードオフを見るように促します。その意味で、プログラミング言語は設計を表現する助けになるだけではありません。**設計を発見する助けにもなります。**

最近、非同期プログラミングの例のために、独自の Future 実装を設計する必要がありました。Future API の重要な側面の1つは、一連のアクションを表現できるように合成を設計することです。

```java
var future1 = action1();
future1.thenCompose(val1 -> action2(val1))
       .thenCompose(val2 -> action3(val2))
```

このAPIをうまく実装するには、関数型プログラミングの概念と語彙を知っていることが重要です。それらの概念を知らないと、実装も使い方もぎこちないものになります。

ときには、プログラミング言語の構文が冗長すぎて、解法の背後にある構造を隠してしまうことがあります。たとえば最近、ワークショップ用の snapshot isolation 実装に取り組んでいたとき、本質的な要件を平易な英語で説明すると少し曖昧で、Javaコードにすると冗長すぎました。TLA+のような、より制約の強い形式仕様が役立ったでしょう。しかし、1ページの擬似形式仕様を書くだけでも大きな助けになりました。

```text
Begin(T, coord):
  R(T) := HLC(coord).now()
  writeSet(T) := {}

Read(T, N, key):
  N.HLC.tick(R(T)) // HLC advanced. So any write or commit after this is guaranteed to be at a higher ts
  return latest committed version of key with ts <= R(T)

Write(T, N, key, value):
  N.HLC.tick(R(T))
  if LatestCommittedVersion(key).ts > R(T):
    abort T
  place provisional intent for (key, value)
  writeSet(T) := writeSet(T) union {key}
```

この擬似形式仕様は、私の思考を明確にし、その後の議論、実装、テストによる検証のよい土台になりました。

### LLMとの作業

「語彙を築くこととしてのコーディング」を考えると、LLMに対して重要な含意があります。LLMは、大量のテキストとコードに含まれる語彙で訓練されています。名前、API、ライブラリ、フレームワーク、イディオム、設計パターン、実装構造の間に繰り返し現れる関係を学習します。Controller、Repository、Reducer、ConsensusModule、TransactionLog といった言葉を見るとき、それらの名前は単なるラベルではありません。既知のコード構造や期待される振る舞いとの関連を伴っています。

LLMと作業するときに語彙が重要なのはそのためです。プロンプトが曖昧で一貫しない言葉を使っていれば、モデルは私たちが意図する設計を推測しなければなりません。コードベースが不明瞭な名前や一貫しない概念を使っていれば、モデルには従うべき安定した構造がほとんどありません。しかし語彙が正確で一貫しており、コードに体現されていれば、LLMは私たちの意図を有用な実装へより確実に対応づけられます。

### Cognitive Debt

これは、LLM支援コーディングに特有の危険も説明します。cognitive debt です。cognitive debt は、言葉、抽象化、構造が、それを扱う人々に十分理解されないまま使われると蓄積します。LLMはこのリスクを増幅します。もっともらしいコードを大量に、非常に速く生成できるからです。生成されたコードには、controller、repository、reducer、factory、transaction、scheduler など、見慣れた構造が含まれるかもしれません。コードはコンパイルできるかもしれません。基本的なテストに通ることさえあるかもしれません。

しかし、チームがそれらの構造の背後にある概念モデルを理解していないなら、コードベースは共有理解なしに語彙を獲得したことになります。問題は、LLMがコードを生成したことではありません。問題は、**開発者が理解を築くより速く、コードが語彙を導入したこと**です。そのギャップが、cognitive debt の主要な原因の1つです。

### 共有された概念モデルとしてのコード

「Designing Abstractions with LLMs」で論じたように、コードを書くことには、抽象化を発見することと適用することという、深く絡み合った2つの活動があります。抽象化を発見するところで、私たちは語彙を発展させています。強い語彙が築かれると、それは共有された概念モデルを表します。そのモデルが存在すれば、コーディングの多くは、その概念モデルを使ってユースケースを構築することになります。

ここで、よいライブラリやよい基盤コードが力を発揮します。私たちは、プログラミング言語や環境の複雑さを隠し、築いてきた語彙に対して、より英語に近いインターフェイスを与えようとします。典型的には、DSLを構築し、その語彙や抽象化を自然言語に近い形で使えるようにします。

LLMはこれが非常に得意です。抽象化語彙に対する自然言語インターフェイスを提供します。最もよい点は、抽象化語彙の背後に実行可能なコードがあれば、そのコード自体が、LLMの誤りを修正するための優れたガードレールとして働くことです。よい抽象化、実行可能な振る舞い、テスト、型、不変条件はすべて、モデルを制約し、その出力をより有用にします。

私たちは語彙を使って外部DSLを作ることができます。LLMは、外部DSLの上にある自然言語インターフェイスとして非常によく機能します。私は PlantUML のようなツールで、LLMを大いに活用してきました。これは驚くことではありません。LLMは設計上、語彙同士を対応づけることに最もよく働くからです。

その意味で、LLMの時代には強い基盤コードがさらに重要になります。システムの語彙がよく形成されると、コーディングは生の構文を作ることよりも、よく発達した概念言語を使って信頼できるソフトウェアを構築することになります。

### ハーネスと文脈としてのコードそのもの

context engineering や harness engineering についての多くの議論では、コードをブラックボックスとして扱い、コードが正しく生成される責任は、プロンプトで適切な文脈を与えたり、仕様、テスト、静的検証などを備えた適切なハーネスを構築したりすることで、外部から管理されるものと考えます。

しかし、明確に定義された語彙を形成する抽象化を備えた、よく構造化されたコードは、それ自体がハーネスと文脈の最も重要な部分になります。私は、よく設計されたコードがLLMとうまく連携する例を繰り返し見てきました。さらに重要なのは、コードが安定した抽象化と明確な意味論を持って作られていると、どのLLMモデルを使うかを選ぶ自由がある程度得られ、プロンプトの正確さについてあまり心配しなくてよくなることです。コード構造と付随するテストそのものが、LLMの出力を信頼でき有用なものにする文脈とハーネスを提供します。

### 結論

コーディングの役割は消えつつあるわけではありません。しかし、変わりつつあります。

LLMによってコード生成が安くなるにつれて、命令を機械的に書く行為は中心ではなくなっていきます。より重要になるのは、概念モデルを明示し、適切な語彙を発見し、その語彙を反復、ドメイン専門知識、フィードバックを通じて洗練することです。だからこそ、プログラミング言語は今も深く重要なのです。私たちは、生成されたコードを受け身でレビューするだけの存在になるべきではありません。コードを書く行為そのものが、私たちの思考の一部です。

コードは今でも機械への命令です。しかし、それは理解のモデルでもあります。LLMの時代には、その第二の役割がさらに重要になります。コーディングの未来は、より多くのコードをより速く書くことだけではありません。人間とLLMの双方がその上で働ける、よりよい概念モデル、よりよい語彙、よりよい基盤を築くことなのです。
