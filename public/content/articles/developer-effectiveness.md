# Maximizing Developer Effectiveness

## 要約

この記事は、開発者の生産性を個人の出力量で測るのではなく、組織が開発者にどれだけ効果的な作業環境を提供できているかで捉えるべきだと述べています。新しい技術やプロセスを導入しても、複雑さや認知負荷、待ち時間が増えれば、開発者の効果性は下がります。

著者は、開発者が日々使うフィードバックループ、特に1日に何十回も何百回も回す micro-feedback loop を短く、単純で、価値のあるものにすることが重要だと説明します。Spotify や Etsy の例を通じて、開発者体験、プラットフォーム思考、軽量なガバナンスが、組織全体の効果性を高める手段として紹介されています。

## 読むときの観点

- 開発者個人の「生産性」ではなく、組織が摩擦をどれだけ減らしているかに注目する。
- フィードバックループの長さ、分かりやすさ、開発者にとっての有用性が、リードタイムやデプロイ頻度にどうつながるかを見る。
- micro-feedback loop のような小さな遅延が、集中や統合の遅れとして積み重なる点を読む。
- Developer Experience を、社内向けの技術能力をプロダクトとして扱う考え方として捉える。
- ガバナンスを承認プロセスではなく、ガードレールと自動化として短いフィードバックループに変える視点を見る。

## 原文の翻訳

私は、変革の最中にあるエンジニアリング組織を支援することがよくあります。これはたいてい、技術の変革であると同時に文化の変革でもあります。たとえば、こうした組織は、独立したチームを持ち DevOps のアプローチを採用できるように、中心的なモノリシックシステムをマイクロサービスへ分割しようとしているかもしれません。また、市場からのフィードバックやシグナルにより速く反応するため、アジャイルやプロダクトの技法を改善したいと考えているかもしれません。

こうした取り組みは、変革の旅のどこかで何度も失敗してきました。マネージャーは遅延や予算超過に不満を持ち、技術者はあらゆる方向から来る障害を解消しようとして苦労します。生産性は低すぎます。チームは数多くの依存関係、認知負荷、新しいツールやプロセスに関する知識不足によって身動きが取れなくなります。最新技術について経営陣に約束された成果は、十分な速さでは実現しません。

こうした状況を調べると、問題の主な理由の一つは、エンジニアリング組織が開発者に効果的な作業環境を提供することを怠っている点にあります。変革の中で、新しいプロセス、新しいツール、新しい技術をあまりにも多く導入し、その結果、日々の作業に複雑さと摩擦を増やしてしまっています。

私はさまざまな種類の企業と仕事をしています。デジタルトランスフォーメーションの初期段階にある企業も、その途中にある企業も、最初から DevOps 文化を採用している企業もあります。その経験から、開発者の効果性が高い企業と低い企業の間には、アプローチに明確な違いがあると分かりました。

もっとも説明しやすいのは、開発者の一日を見ることです。

### 効果性の高い環境での一日

開発者は次のように一日を過ごします。

- チームのプロジェクト管理ツールを確認し、スタンドアップに参加する。そこで何に取り組むべきかが明確になっている。
- 開発環境は、開発環境と本番環境に合ったライブラリで自動更新されており、CI/CD パイプラインは green であることを確認する。
- 最新のコードを取得し、小さなコード変更を行い、ローカル環境へのデプロイとユニットテストの実行ですばやく検証する。
- 機能のために別チームのビジネス能力に依存している。開発者ポータルからドキュメントと API 仕様を見つけられる。まだ疑問があるので、そのチームの Slack ルームに入り、サポート担当の開発者からすぐに助けを得る。
- 数時間、中断されずに自分のタスクに集中する。
- 休憩を取り、コーヒーを飲み、散歩し、同僚と卓球をする。
- コード変更をコミットする。変更は複数の自動チェックを通過し、本番へデプロイされる。ビジネス指標と運用指標を監視しながら、本番ユーザーへ段階的にリリースする。

この開発者は一日の中で漸進的な進歩を得られ、満足して帰宅します。

### 効果性の低い環境での一日

開発者は次のように一日を過ごします。

- 朝からすぐに、本番の問題に関する複数のアラートに対応しなければならない。
- システム横断の集約ログがないため、エラーレポートを見つけるために複数のログシステムと監視システムを確認する。
- 運用担当者と電話でやり取りし、そのアラートが false positive だったと判断する。
- 以前完了した機能について、アーキテクチャ、セキュリティ、ガバナンスの各グループからの返答を待たなければならない。
- 一日が多くの会議で分断される。その多くはステータス報告の会議である。
- 以前の機能がレビュー担当者に承認されたので、それを別ブランチへ移す。すると、サイロ化した QA チームが管理する、ほとんど常に red である長い夜間 E2E テストスイートが起動する。
- 別チームの API に依存しているが、最新のドキュメントを見つけられない。そのため、代わりに相手チームのプロジェクトマネージャーと話して問い合わせようとする。答えを得るためのチケットには数日かかるため、現在のタスクはブロックされる。

話を続けることもできます。しかし結局、この開発者はあまり成果を上げられず、いらだち、やる気を失って帰ることになります。

### 開発者の効果性

効果的であるとは何を意味するのでしょうか。開発者にとっては、顧客に最大の価値を届けることです。自分のエネルギーと革新性を、会社の目標に向けて最善の形で使えることです。効果的な環境は、有用で高品質なソフトウェアを本番に出しやすくし、それを運用しやすくします。その結果、開発者は不要な複雑さ、軽薄な作業の揺れ、長い遅延に対処しなくて済み、価値を生むタスクに集中できます。

効果性の低い環境の例では、すべてが本来より長くかかります。開発者としての一日は、終わりのないブロッカーと官僚主義でできています。問題は一つだけではありません。たくさんあります。これは千の小さな傷による死に似ています。小さな非効率が複利のように効き、ゆっくりと生産性を壊していきます。非効率の感覚は、エンジニアリングだけでなく組織全体へ広がります。エンジニアは無力感を覚え、生産的でなくなります。

さらに悪いことに、彼らはそれを受け入れてしまいます。その働き方が、開発とはこういうものだと定義する日常になります。開発者は learned helplessness を経験します。

一方、高い効果性を持つ環境を提供する組織には、勢いの感覚があります。すべてが簡単で効率的であり、開発者が出会う摩擦は少なくなります。彼らはより多くの時間を価値の創出に使えます。デジタルトランスフォーメーションに取り組む企業にとって最も作るのが難しいのは、こうした摩擦の少ない環境と、それを支える、絶えず改善したいという意欲と能力を育てる文化です。

組織は開発者の生産性を測る方法を探します。よくあるアンチパターンは、コード行数や機能の出力量を見ること、あるいはパフォーマンスの低い開発者を見つけることに焦点を置きすぎることです。会話を反転させ、**組織が効果的なエンジニアリング環境をどう提供しているか**に焦点を当てるほうがよいでしょう。生産的であることは開発者を動機づけます。摩擦がなければ、彼らは創造的に考え、自分の力を発揮する時間を持てます。

私の経験では、組織がこれをしなければ、最高のエンジニアは去っていきます。強い技術人材を採用したい優れた革新的なデジタル企業がたくさんある中で、開発者が効果性の低い環境で働く理由はありません。

開発者の効果性を最適化した企業の例を見てみましょう。

### ケーススタディ: Spotify

Spotify は、開発者の効果性をよりよく理解するために、エンジニアを対象としたユーザーリサーチを行いました。この調査から、二つの重要な発見が得られました。

1. 内部ツールの断片化。Spotify の内部インフラとツールは、小さく孤立した「島」として作られており、エンジニアにコンテキストスイッチと認知負荷を生んでいた。
2. 発見しにくさ。Spotify には技術情報を見つけるための中心的な場所がなかった。情報があちこちに分散していたため、エンジニアは情報をどこから探し始めればよいかさえ分からなかった。

Spotify の Developer Experience チームは、これらの問題を負のフライホイールとして説明しています。つまり、開発者があまりにも多くの未知に直面し、多くの判断を孤立した状態で迫られ、それが断片化と重複作業をさらに増幅し、最終的にプロダクトのエンドツーエンドのデリバリー時間を損なう悪循環です。

この複雑さを緩和するため、Spotify は Backstage を開発しました。これはプラグインアーキテクチャを持つオープンソースの開発者ポータルであり、すべてのインフラプロダクトを一つの場所で公開できるようにし、一貫した開発者体験と、エンジニアが必要な情報を見つける出発点を提供します。

### どこから始めるか

私が効果性の高い環境として説明しているものは、DevOps 文化、継続的デリバリー、プロダクト思考を完全に受け入れた会社で働く感覚です。多くの企業がこの環境を目指す旅の途中にあるのは、とても自然なことです。彼らは Accelerate や State of DevOps report を読んでおり、どのような組織を作ろうとしているのかを知っています。

4 key metrics、つまり lead time、deployment frequency、MTTR、change fail percentage は、DevOps のパフォーマンスを測る優れた指標です。

DevOps の指標を見る一つの方法は、それらを lagging indicator と捉えることです。現在地を理解し、改善のために会社が行うべき具体的なことを探る作業が必要な時を示すには有用です。理想的には、より行動に移しやすい、低レベルの効果性の leading metric を特定したいところです。それらは上位の指標と相関し、積み上がっていきます。

これは、開発者満足度に関するサーベイなど、他の調査ソースとも組み合わせるべきです。

改善のために使うべき優れた助言、プラクティス、ツール、プロセスは圧倒的な量があります。何をすべきかを知るのはとても難しいものです。私の調査では、いくつかの重要な開発者フィードバックループがあることが分かりました。私は、これらのループを最適化し、速く単純にすることへ焦点を当てることを勧めます。フィードバックループの長さ、制約、結果として得られる成果を測ります。

新しいツールや技法が導入されたとき、これらの指標は、開発者の効果性がどの程度改善されたか、少なくとも悪化していないかを明確に示せます。

### フィードバックループ

私が特定した重要なループは次のとおりです。

| フィードバックループ | 低い効果性 | 高い効果性 |
| --- | --- | --- |
| ローカルのコード変更が動くことを検証する | 2分 | 5-15秒（技術選択による） |
| 欠陥の根本原因を見つける | 4-7日 | 1日 |
| コンポーネントが他のコンポーネントと統合されることを検証する | 3日-2週間 | 2時間 |
| 変更が非機能要件を満たすことを検証する | 3か月 | 1日-1週間（変更範囲による） |
| 新しいチームで生産的になる | 2か月 | 4週間 |
| 社内の技術的な問い合わせに答えを得る | 1-2週間 | 30分 |
| 新しいサービスを本番で立ち上げる | 2-4か月 | 3日 |
| 変更が顧客にとって有用だったかを検証する | 6か月、または検証されない | 1-4週間（変更範囲による） |

これらの指標は、私が可能だと観察してきたものに基づいています。すべての企業がすべてのフィードバックループで高い効果性の枠に入る必要はありません。しかし、意思決定を導く具体的な目標にはなります。エンジニアリング組織は、自分たち固有の文脈の中で調査を行い、どのサイクルと指標が技術戦略にとって重要かを見極めるべきです。

フィードバックループを最適化するためにどの技法が適用されたのか、また企業がそこへ至るまでにどのような旅をしてきたのかを見ることは有用です。そうしたケーススタディは、自分の組織に適用できる多くのアイデアを与えてくれます。

開発者が開発中にフィードバックループをどう使うかを単純化して見ると、開発者は作業が仕様や期待される標準を満たしているかを、途中の複数の地点で検証していることが分かります。重要な観察点は次のとおりです。

- フィードバックループが短いほど、開発者はそれをより頻繁に実行する。
- 結果が開発者にとって価値あるものだと見なされ、単なる官僚的な負担でないなら、開発者はより頻繁に実行し、その結果に基づいて行動する。
- より早く、より頻繁に検証すれば、後の手戻りが減る。
- 結果を解釈しやすいフィードバックループは、往復のコミュニケーションと認知負荷を減らす。

組織がこれらを実現できないと、問題はすぐに複合化します。開発者には多くの無駄な努力が発生します。待つ、探す、結果を理解しようとする時間として現れます。それが積み重なり、プロダクト開発に大きな遅延を引き起こします。そしてそれは、4 key metrics、特に deployment frequency と lead time の低いスコアとして現れます。

### micro-feedback loop の導入

私が観察してきたところでは、基本を確実に押さえる必要があります。開発者が1日に10回、100回、あるいは200回行うことです。私はこれを micro-feedback loop と呼んでいます。バグ修正中にユニットテストを実行することかもしれません。コード変更がローカル環境や開発環境に反映されるのを見ることかもしれません。環境内のデータを更新することかもしれません。開発者は権限を与えられれば自然に最適化しますが、micro-feedback loop が放置されていることがよくあります。

これらのループは意図的に短いものなので、非常に小さな時間単位を扱うことになります。

なぜそんな小さな問題に焦点を当てなければならないのかを、マネジメントに説明するのは難しいものです。2分かかるコンパイル段階を15秒にするために、なぜ投資しなければならないのでしょうか。これは多くの作業を必要とするかもしれません。おそらく、システムを独立したコンポーネントへ分離する必要もあるでしょう。2日かかっているものを最適化するほうが、取り組む価値のあることとしてはるかに理解しやすいものです。

しかし、その2分はすぐに積み上がり、1日に100分を超えることがあります。こうした小さな停止は、文脈と集中を失う機会になります。開発者が気を散らし、メールを開くかコーヒーを取りに行こうと判断するには十分な長さです。いったん気が散り、フロー状態から外れると、高い生産性の状態に戻るまで最大23分かかることを示す研究があります。

エンジニアが休憩を取ったり、時々頭を切り替えたりすべきでないと言っているわけではありません。しかし、それは意図的に行うべきであり、環境に強制されるべきではありません。

実際には、開発者はこうした非活動の瞬間を有用なことで埋めて補おうとします。二つのタスクを同時に持ち、切り替えるかもしれません。変更をまとめることで、コンパイル頻度を下げるかもしれません。私の調査では、どちらもコードの統合と開発時間の遅れにつながります。

どこまで最適化すべきでしょうか。いつ十分だと言えるのでしょうか。ある変更を15秒まで短縮できたとして、さらに3秒にできると考えているとします。その投資に価値はあるでしょうか。それは変更の難しさと、それがもたらす影響によります。10チームを速くするツールや能力を開発できるなら、価値があるかもしれません。ここで、個別チームの最適化ではなく、プラットフォーム思考が関わってきます。

分散システムは特に難しい課題です。システムを複数のデプロイ可能な単位、通常はマイクロサービスに分割することには、多くの妥当な理由があります。しかし分散システムは、開発者の効果性を含め、多くのことを難しくもします。チームがチームの自律性や実行時性能を最適化する一方で、速いフィードバックループを維持する投資をしないために、開発者の効果性を犠牲にしてしまうことがあります。

これは、私の会社が非常によく出会う状況です。

### 組織の効果性

効果性の高い組織は、エンジニアリング組織を、効果性とフィードバックループを最適化するように設計しています。リーダーシップは時間をかけて、開発者がこれらのフィードバックループを漸進的に改善する権限を持てる文化を作ります。

それは、技術と、開発チームから摩擦を取り除くことがビジネスに不可欠である、というリーダーシップの認識から始まります。これはいくつかの形で現れます。

技術リーダーは、効果性を継続的に測定し、見直します。効果性の高い組織は、4 key metrics と自分たちの文脈にとって重要な他のデータポイントを追跡し、データに基づく意思決定を行うためのフレームワークを作っています。この文化は経営層から始まり、組織の残りの部分へ伝えられます。

指標に加えて、日々その環境で働く individual contributor の声を聞くための開かれた場を作ります。開発者は自分たちが直面している問題を知っており、それを解決するための多くのアイデアを持っています。

この情報に基づき、エンジニアリングマネージャーは投資の優先順位を決められます。大きな問題には、開発者体験の悪さに対処するための相応に大きなモダナイゼーションプログラムが必要かもしれません。しかし多くの場合、より重要なのは、チームが継続的改善を行えるように権限を与えることです。

重要な原則は、Developer Experience を受け入れることです。これに焦点を当てたチームの作業プログラムを見ることは珍しくありません。Developer Experience とは、**技術的な能力をエンドユーザー向けプロダクト開発と同じアプローチで構築すること**を意味します。同じリサーチ、優先順位付け、アウトカムベースの思考、利用者からのフィードバックの仕組みを適用するのです。

開発者を動機づけるために、効果性の高い組織は権限を分散します。つまり、開発者が日々の生活を改善できる能力を持つべきだということです。チームが漸進的な技術改善を行い、技術的負債を管理するための方針があります。開発者とプロダクトマネジメントの間には、データに基づく健全な議論があるべきです。

効果性の高い組織は、開発者が革新できる能力も提供します。チームに明確な目標とボトルネックについての明確な理解があれば、開発者は問題解決に創造性を発揮できます。こうした組織は、最良のアイデアが浮かび上がる仕組みも作り、その後、データを使って何が最善かを評価し、そこへさらに投資します。

コミットメント、測定、権限委譲の後に来るのがスケーリングです。

一定の組織規模になると、規模の経済によって効率を作る必要が出てきます。組織は、プラットフォーム思考を適用することでこれを行います。つまり、効果性の改善に特化した内部プラットフォームを作るのです。開発者の効果性を改善する技術的能力を構築するエンジニアリングチームへ投資します。他の開発チームを自分たちの消費者と見なし、提供するサービスをプロダクトとして扱います。

これらのチームには技術プロダクトマネージャーがいて、利用チームへ与えている影響に関連する成功指標を持ちます。たとえば observability に焦点を当てたプラットフォーム能力チームは、監視、ログ、アラート、トレーシングの能力を作り、チームがサービスの健全性を簡単に監視し、自分たちのプロダクトの問題をデバッグできるようにします。

ガバナンスの必要性は依然として優先事項です。しかし、効果性の高い組織ではその適用方法が大きく異なるため、これは汚い言葉として見られる必要はありません。彼らは中央集権的なプロセスから、軽量なアプローチへ移ります。長い承認プロセスではなく、ガードレールを設定して伝え、チームを正しい方向へ nudging することです。ガバナンスは、次のように実装されると、効果性において重要な役割を持てます。

- 明確なエンジニアリング目標。
- チームやサービスが互いにやり取りする方法の指定。
- 有用なピアレビューの促進。
- ベストプラクティスをプラットフォーム能力へ組み込むこと。
- architecture fitness function による制御の自動化。

本質的に、効果的な組織はガバナンスのフィードバックループを短くします。これについては、今後の記事でさらに掘り下げます。

### ケーススタディ: Etsy

Etsy は DevOps ムーブメントの先駆者の一つでした。そのリーダーたちは、速く進むことは技術戦略であると同時にビジネス戦略でもあるという信念を持ち、開発者の効果性を文化に埋め込むよう取り組んできました。彼らは、価値あるプロダクトを速く安全に本番へ出す能力を積極的に測定し、障害や遅さを解消するために技術投資を調整します。

Etsy の重要な指標の一つは lead time であり、これは測定、監視され、オフィス中にリアルタイムで表示されています。lead time が特定の重要なしきい値を超えると、リリースエンジニアリングチームがそれを妥当な水準まで下げるために動きます。CTO の Mike Fisher は、Etsy のエンジニアがすばやく前進することを「恐れない」と語っています。新しいことを試すためのセーフティネットがあるからです。

ソフトウェアを速くデプロイすることは、話の半分にすぎません。本当に効果的であるためには、そのソフトウェアが消費者にとって価値あるものでなければなりません。Etsy はこれを、データ駆動のアプローチで行います。各機能には測定可能な KPI があります。

コード変更は一連のチェックを通り、開発者はその変更が性能、可用性、失敗率などに関する Etsy の SLA を満たしていると確信できます。変更が本番に出ると、Etsy の実験プラットフォームがユーザー行動の指標を取得できます。チームはその指標を使ってプロダクトを反復し、関連する KPI とユーザー満足度を最適化します。最終的にその変更に価値がないと証明されれば、技術的負債を避けるために取り除かれます。

Etsy には、開発者体験を優先する現在の取り組みがあります。それには四つの柱があります。

### Developer Experience の四つの柱

Help me craft products は、プロダクトエンジニアが仕事をするために適切な抽象化、ライブラリ、足場を持てるようにするものです。

Help me develop, test, and deploy は、プロダクトエンジニア、特に開発環境そのもの、IDE や linter、ユニットテストや統合テストのパターンとランナー、デプロイのツールとプロセスに焦点を当てます。

Help me build with data は、データサイエンティストと機械学習エンジニアに焦点を当て、データエンジニアリングのエコシステム全体が、直感的で、テストしやすく、デプロイしやすい形に整えられていることを確かめます。

Help me reduce toil は、オンコールエンジニアに焦点を当て、適切な水準の自動化を備えた本番システムを構築し、runbook とドキュメントがアクセスしやすく最新であり、toil 的な活動の削減を追跡して優先順位付けできるようにします。

この方針は、Etsy のリーダーシップが開発者に対して本当にコミットしていることを表しています。彼らは、4 key metrics を含む指標を追跡し、開発者の net promoter score、つまり NPS を把握するために月次サーベイを行うことで、自分たちの効果性を継続的に確認しています。

### 結論

この記事の冒頭では、開発者の効果性の重要性と、それが開発者の幸福度と生産性へ与える影響について述べました。私は、単なるツールや技法ではなく、開発者が達成しようとするアウトカムに焦点を当てました。さらに考察を進めると、開発者がプロダクトを開発する際によく使う一連のフィードバックループが見えてきます。

また、micro-feedback loop の非効率がどのように積み重なり、4 key metrics やプロダクト開発速度といったより大きな指標に影響するかについても述べました。さらに、Developer Experience を原則として重視すること、そしてプラットフォーム思考がスケール時の効率と効果性を最大化する助けになることを強調しました。

このシリーズの続きでは、開発者の効果性と個々のフィードバックループを、ケーススタディを通じてさらに深く説明します。そこでは、組織がどのようにしてこれらの数値と結果としてのアウトカムを達成できたのかについて、具体的な詳細を示します。また、ローカルなレベルとグローバルなレベルでこれらの最適化を可能にする組織構造とプロセスについても説明します。
