# Enterprise Application

## 要約

Enterprise Application は、医療記録、外国為替取引、給与計算、リース会計のように、長く残るデータと複雑な業務ルールを扱うアプリケーションを指すために Fowler が使った言葉です。
公式な定義はありませんが、永続データ、同時アクセス、多数の画面、バッチ処理、既存システムとの統合、文脈ごとに異なる概念、複雑な business logic といった特徴を共有します。

この記事は、「enterprise」という語が大企業向けという響きを帯びてしまった後でも、Fowler の用法では小さな組織やスタートアップの重要な業務アプリケーションも含まれると説明しています。

## 読むときの観点

- Enterprise Application を企業規模ではなく、扱うデータと業務上の責任から捉える。
- 永続データ、トランザクション、画面、バッチ、統合が複雑さを作る様子を見る。
- 同じ「顧客」のような言葉でも、組織や文脈によって意味がずれる点に注意する。
- business logic がしばしば business illogic になる理由を、現実の取引や例外条件から考える。

## 原文の翻訳

今世紀の初めごろ、私は『Patterns of Enterprise Application Architecture』という本に取り組んでいました。その本を書くときに困ったことのひとつは、題名をどうするか、もっと言えば、私が扱っている種類のソフトウェアシステムを何と呼ぶかでした。私は、自分のソフトウェア開発の経験がいつも特定の形のソフトウェアに集中してきたことを意識していました。たとえば医療記録、外国為替取引、給与計算、リース会計のようなものです。

これらは、プリンター内部の組み込みソフトウェア、ゲーム、飛行制御ソフトウェア、電話交換機とは大きく異なります。私はこの種のシステムを説明する名前が必要で、最終的に **Enterprise Application** という言葉に落ち着きました。

私がよく言わなければならないことですが、この言葉に正式な定義はありません。しかし Enterprise Application には、いくつか共通する特徴があります。

Enterprise Application は、通常、大量の永続データを持ちます。そのデータは、たいてい何らかのデータベース管理システムで管理されます。多くの場合、そのデータベースはリレーショナルですが、NoSQL の選択肢も増えています。このデータはたいてい、それを処理するアプリケーションよりも長く残り、価値も高くなります。

このデータは、同時にアクセスされ、操作されます。数は大きく変わります。社内アプリケーションなら利用者は数十人程度かもしれませんが、顧客向けの Web アプリケーションなら数万人になってもおかしくありません。同時実行性が高いにもかかわらず、多くの Enterprise Application 開発者は、critical region、race condition、その他の古典的な並行プログラミングの要素について、それほど多く考えません。その代わり、データベースや専用のトランザクション管理ツールが管理する **transaction** の上に思考を組み立てます。

これだけ多くのデータがあるため、Enterprise Application にはそれを扱う多数のユーザーインターフェース画面があります。通常、同じデータが異なる文脈で異なる方法により操作されます。ユーザーは常用する人から時々使う人まで幅広いため、インターフェースは慣れの度合いの違いに合わせなければなりません。また、忘れられがちなオフライン、つまりバッチ処理もかなりあります。

まったく新しい Enterprise Application を作っているとしても、それを孤立して作ることはありません。むしろ、他の Enterprise Application と統合する必要があります。それらのシステムは幅広いチームによって作られています。多くの顧客へ販売するベンダーが作ったものもあれば、自分の組織のためだけに内部で作られたものもあります。そうしたアプリケーションは何十年にもわたり、実にさまざまな技術で書かれてきました。その中には、あなたの母親に聞かなければならないようなものもあります。

扱わなければならない統合の仕組みもたくさんあります。ファイル交換、共有データベース、メッセージングミドルウェアなどです。ときおり、こうした通信技術をすべて整理しようとする試みが現れますが、完全には成功せず、後にはさらに複雑さが残ります。

異なるアプリケーションが同じデータにアクセスしている場合でさえ、それらの間にはかなりの概念上の不協和があります。顧客という言葉は、営業組織にとっては、技術サポートにとっての意味とはかなり違うかもしれません。同じように聞こえるエンティティが、文脈ごとに異なるフィールドを持ちます。さらに悪い場合には、同じ名前のフィールドが異なる意味を持ちます。

そして、いわゆる business logic があります。オペレーティングシステムを書いているときは、全体を論理的に保とうとし、ソフトウェアを単純で信頼できるものにするための簡略化を見つけて実装しようとします。しかし業務ルールは、そのままの姿で与えられます。それを変えたいなら、67回の会議と3人の副社長の退任が必要になります。業務ルールはたいてい、奇妙な条件が行き当たりばったりに並び、驚くような形で相互作用します。

その狂気には、もっともな理由があります。個々のルールは、営業担当者が特別な一回限りの条件を付けることで、ある特定の取引をまとめられた事例だからです。それを千回やると、多くの Enterprise Application の中心にある複雑な business **illogic** が生まれます。

Enterprise Application は大きいことも小さいこともあります。議論は大規模で複雑なアプリケーションに集中しがちですが、素早く作る必要がある小さなアプリケーションにも課題があります。大きなシステムは、うまくいかないと大きな音を立てます。しかし小さなシステムの累積的な影響も、企業の健全性に驚くほどの影響を与えることがあります。

物事に名前を付けるのはいつも難しいものです。最小限の単語数で、読者の頭の中に適切な含意を呼び起こし、その定義が何を意味するかを絶えず思い出させなくてもよいようにしなければなりません。全体として、私は自分の選択にかなり満足してきました。しかし本を書き終えてから、enterprise という言葉は、私の用法とは少し合わない含意を持つようになりました。

本を書いて以降に出てきた問題のひとつは、enterprise が今ではたいてい、大規模で歴史のある会社を意味することです。人々は、Facebook、Etsy、あるいはカスタム T シャツを作る100人規模の会社ではなく、G.E. や Siemens を思い浮かべます。しかし上の私の定義に従えば、小さなスタートアップでさえ、私なら Enterprise Application と呼ぶソフトウェアに依存しています。

そのため、Ruby on Rails のコミュニティが enterprise を侮蔑語として使うようになったとしても、私は Ruby on Rails を Enterprise Application を作るためのフレームワークと呼び、BaseCamp を Enterprise Application の典型例と呼ぶでしょう。ただし、私がそう言ったことを DHH には言わないでください。さもないと、彼は私を車のボンネットの飾りにしてしまうでしょう。

enterprise をめぐるこうした含意のために、別の用語が必要なのではないかと考えることがあります。P of EAA を書いていたとき、私の作業用タイトルは「Information Systems Architecture」でした。しかし information systems には、古い技術という望ましくない含意があると感じました。思い切り復古的に「data processing」と呼ぶこともできるでしょうが、全体としては、Enterprise Application は、私が思いつけるどの言葉よりもまだよい用語に見えます。

この記事は、『P of EAA』の序文にある Enterprise Application の定義をもとにしたものです。
