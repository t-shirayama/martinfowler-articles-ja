# Data Mesh Accelerate Workshop

## 要約

この記事は、Data Mesh への変革を始める組織が、技術先行や長すぎる事前設計に偏らず、ビジネス目標と技術的な取り組みをそろえるための Data Mesh Accelerate workshop を紹介しています。ワークショップは、現在地の把握、望ましい将来像の共有、ユースケースとデータプロダクトの発見を通じて、関係者の初期アラインメントを作ることを目的としています。

サンプルアジェンダでは、Data Mesh の四原則、Nirvana statement、4 key metrics、OKR、ユースケース探索、データプロダクト発見、レビューとチェックアウトまでを扱います。Data Mesh を一度に大きく設計するのではなく、短いフィードバックループとプロダクト思考で進めるための実践的な入口として読めます。

## 読むときの観点

- Data Mesh を技術導入だけでなく、組織とビジネス目標の変革として捉える。
- ワークショップが「何を作るか」より前に、目標、課題、望ましい状態をそろえる場として設計されている点を見る。
- Use Case と Data Product の関係を、アウトカムから具体的な構成要素へつなぐ流れとして読む。
- 4 key metrics や OKR が、進捗と価値を測る共通言語として使われている点に注目する。
- remote でも対面でも実施できるように、活動の目的と手順が分離されている点を見る。

## 原文の翻訳

技術的でグローバル化した世界で生き残ろうとする組織は、データを効果的に扱わなければなりません。多くの組織が big data と AI に投資している中で、このデータを効果的に管理する必要性は、これまで以上に重要になっています。Data Mesh は、企業が価値ある洞察を生み出し、データ駆動組織になるという約束を果たす助けになります。

Data Mesh についてさらに知るには、このサイトにある Zhamak Dehghani の二つの基礎的な記事を読むとよいでしょう。

- How to Move Beyond a Monolithic Data Lake to a Distributed Data Mesh
- Data Mesh Principles and Logical Architecture

Data Mesh のパラダイムは、四つの原則に基づいています。

- データ所有権とアーキテクチャの、ドメイン指向の分散。
- プロダクトとして提供される、ドメイン指向のデータ。
- 自律的でドメイン指向のデータチームを可能にする、プラットフォームとしてのセルフサービス型データインフラ。
- エコシステムと相互運用性を可能にする federated governance。

Data Mesh は、分析データと業務運営の二分法を取り除くための、分散型の社会技術的アプローチです。残念ながら、多くの社会技術的アプローチと同じように、多くの組織は Data Mesh 変革を成功させるために本当に必要な目標と戦略をそろえることに苦労します。Data Mesh への変革は難しいものです。それは、チームの組織のされ方、仕事の優先順位の付け方、最新のデータ関連技術の進歩をどう適用するかにおけるパラダイムシフトを表しています。

これは組織全体の人々に影響する、広範な変化です。

私たちは、さまざまな組織でいくつもの Data Mesh 変革に関わってきました。また、世界中でさらに多くの変革に関わってきた Thoughtworker たちと意見を交わし、学んできました。

Data Mesh 変革に向き合う中で、私たちは二つのよくある課題を見てきました。技術偏重と、big design upfront です。

一つのアプローチでは、チームは自分たちの仕事をより高いレベルの目標とそろえないまま、データプロダクトを作り始めます。このアプローチ、つまり技術偏重は、Data Mesh とは技術者と、データを収集し、変換し、提供する新しい方法のことだという真剣な考えに基づいています。Data Mesh の主な目標は、データの価値を大規模に解き放つことです。技術はその重要な一部ですが、ビジネス目標とのアラインメントは成功に不可欠です。

ビジネスの戦略目標と十分にそろわないまま技術側から始めるのは、行き先を知らずに休暇の荷造りをするようなものです。そうしてはいけません。旅を始める前に、初期のアラインメントを確実に作ってください。

もう一つのよくあるアプローチは、Data Mesh の実装を始める前に深い分析を行うことです。これは Big Design Up Front であり、分析と設計に長い期間、しばしば数か月を費やします。その結果として、先に進む道筋について多くの図と計画が作られます。

図や計画は素晴らしいものです。しかし価値は、それらを現実にし、ユーザーとビジネスに価値を届けることから生まれます。Data Mesh の重要な概念はプロダクト思考です。そこでは、ユーザーのニーズと実際の利用に基づいてプロダクトを漸進的に進化させます。Data Mesh 変革全体への私たちのアプローチも、この進化的なアプローチに従い、短いフィードバックループを持つ必要があります。

### Data Mesh Accelerate Workshop

Thoughtworks での私たちの答えは、Data Mesh Accelerate と呼ぶプロセスです。初期の方向性を定め、理解を築き、戦略的アラインメントを作るためのよい活動の並びを組み立てました。これは、始めるために十分な初期分析と設計を提供し、技術とビジネスのアラインメントのよいバランスを与えます。

Data Mesh Accelerate workshop は、コラボレーション、現在状態の把握、望ましい姿のマッピングに焦点を当てた一連の活動を提供します。主な目的は、ステークホルダー間の初期アラインメントを作り、グループがどのような Data Mesh のアウトカムを目指しているのかを理解し、データプロダクトを特定し、設計し、構築するプロセスを探索することです。

ワークショップを終えると、ステークホルダーは Data Mesh の概念、これからの旅がどのようなものか、次のステップについて共有理解を持つべきです。Data Mesh workshop は、組織の Data Mesh 変革における最初の一歩を表しています。先には多くのステップがありますが、最初の一歩を成功させるために時間をかけることは、その後の旅に大きく報いると私たちは見てきました。

### accelerate workshop の位置づけ

Data Mesh Accelerate Workshop は、Data Mesh 変革の一部です。Data Mesh 変革を始めるには、まず取り組む適切なドメインを選びます。次に、ビジョンからユースケースへ進みます。これが accelerate です。その後、そのユースケースを支えるために必要なデータプロダクト、プラットフォーム、組織変更を特定します。これは discovery と inception です。そして実際にデータプロダクトを構築し、それらを Data Mesh に追加します。

Data Mesh Accelerate workshop が合うのは、成功する Data Mesh 変革を始めるために、ドメインのステークホルダー、つまりビジネスと技術の人々が関心を持ち、数時間を確保できるときです。これは、初期の道筋について全員をそろえる手段を提供します。

### Data Mesh accelerate workshop のサンプルアジェンダ

### Kick-off

週は kick-off で始まり、集中的なセッションの連続を経て、レビューで終わります。kick-off では、その週のアジェンダと期待値を扱います。ワークショップから得られた結果はレビューで提示されます。

#### 手順

1. 組織の主なスポンサーに、この取り組みの重要性について話してワークショップを開いてもらう。
2. ワークショップ全体の意図、アジェンダ、各活動について短く説明する。ここは ground rule に注意を向ける機会でもある。
3. 全員に手短に自己紹介してもらう。名前と役割に加え、好きな食べ物や休暇で行きたい場所など、何か別のことを共有してもらうのもよい。

kick-off は、ワークショップの目的とこれからの活動について明確な理解を持って終わるべきです。参加者の中には初めて一緒に仕事をする人もいるかもしれません。そのため、人々が互いにつながる余地を作るというヒントが重要になります。

携帯電話をオフにする、他の視点に開かれていることを期待する、といった明確な ground rule を設定すると、参加者は自分に何が期待されているかを理解し、ワークショップ全体が円滑に進みやすくなります。

### Data Mesh の四原則とこれからの課題

Data Mesh の四原則の活動には、二つの主な目標があります。一つは、Data Mesh の四原則、つまり Domain Ownership、Data as a Product、Self-serve Data Platform、Federated Computational Governance について共有理解を作ることです。もう一つは、組織とグループの現在の文脈で、それぞれを実装する際の課題についてオープンな会話を促すことです。

#### 手順

1. 四原則について短く紹介する。
2. 参加者に、各原則の適用可能性への脅威を共有してもらう。「そこへ到達するうえで、私たちの前にはどのような課題があるか」
3. これからの課題について会話する。

Data Mesh とその原則について、全員の理解レベルを近づけることが重要です。Data Mesh をよく理解している人が、少なくとも次の内容を扱う短いプレゼンテーションを参加者に行うべきです。

- Data Mesh の「なぜ」。どのような問題を解決しようとしているのか。
- Data Mesh の四原則。それぞれの理由と詳細。

情報を共有した後、参加者が原則について明確化し、共有し、質問する時間を与えます。

参加者には、過去の経験、懸念、そして四原則それぞれについてこれからどこに課題があると見ているかを共有してもらいます。課題について話すことで、グループは、特にデータによって駆動される変革の初期に、より注意を払うべき領域について最初の感覚を得られます。

「明日からこの原則を適用し始めるとしたら、どのような課題に直面するだろうか」

参加者が自分の視点を共有している間、グループの他の人には、その視点を自分の視点と比較してもらいます。そのために、何が同じだったか、何が違ったか、何に驚いたかという三つの質問を使います。これらの質問は、グループ内の異なる視点を比較対照し、共通点と blind spot を明らかにする優れた方法です。

組織が直面する課題と参加者が挙げる pain point を探索することで、現在の文脈を理解し、その後の議論を適切に枠づけられます。Data Mesh はデータ組織が経験する一般的な失敗パターンに対処することを目指しているため、自分たちの組織が直面している具体的な課題を理解することは、出発点として優れています。

### Data Mesh Nirvana

この活動の目的は、理想的な将来状態を明確に表す nirvana statement を作ることで、Data Mesh の文脈で「よい状態」とは何かを理解することです。この活動は、グループが先を見通し、現在の pain point の一部を内省する機会を提供します。

「Nirvana とは特別な場所であり、完璧な最終状態です。多くの場合は到達できませんが、目指すことが重要です。この人々のグループと組織文脈において nirvana が何であるかをそろえ、理解することはとても重要です。」

#### 手順

1. 参加者を小さなグループに分ける。
2. 各グループに nirvana statement を作ってもらい、それを全体に発表してもらう。
3. 組み合わせて、共有された nirvana statement を作る。

参加者を小さなグループに分けると、活発な議論を促進しやすくなります。小さなグループで作業すると、人々の参加は増えます。私たちは、4-6人のグループが活発な議論をしやすくすることを見てきました。

各グループに同じ指示を与えます。Data Mesh 変革にとって理想的な将来状態について、短い文を書くことです。「Our Nirvana is...」というプロンプトを書いた付箋は、この議論を始める助けになります。

グループが nirvana statement を作ったら、再び全体に戻り、各グループから一人のボランティアにその statement を読み上げ、背後にある考えを共有してもらいます。

最後のタスクは、異なる statement を一つの共有された nirvana statement にまとめることです。議論とコラボレーションを刺激するために、私たちは fishbowl アプローチを勧めます。このプロセスの終わりには、グループ全員が望ましい目標を表していると同意する、一つの nirvana statement を持つべきです。

この単純な問いが、参加者を大いに巻き込み、そしてそろえていく様子を見るのはとても興味深いものです。私たちの nirvana は何か。グループがそれに合意すると、そこへ向かうステップを明確にするのはずっと容易になります。

何か大きなものを達成するための選択肢、行動、取り組み、実行タスクについて話す前に、**高いレベルの目標にそろうことが不可欠**です。

### 4 Key Metrics

4 key metrics の活動には、二つの主な目標があります。一つは、4 key metrics の現在状態について会話を促すことです。もう一つは、チームがどこへ向かおうとしているかを見えるようにすることです。

書籍 Accelerate と関連研究は、優れた組織パフォーマンスとソフトウェアデリバリーパフォーマンスを達成するうえで、4 key metrics が重要であることを示しています。4 key metrics は、Lead time、Deploy frequency、Mean Time to Restore、つまり MTTR、Change fail percentage です。

> 私をどのように測るかを教えてくれれば、私がどのように振る舞うかを教えよう。
>
> -- Eli Goldratt

#### 手順

1. 4 key metrics の詳細を共有し、その重要性について議論する。
2. 参加者に、これら4つの指標それぞれについて、現在自分たちはどこにいると考えるかを共有してもらう。
3. 4 key metrics に関する将来の期待について議論する。

まず、参加者が 4 key metrics の使い方と背後にある研究に慣れるように、4 key metrics を紹介します。

ワークショップ参加者の中には、異なる領域と異なる役割の代表者がいるべきです。この多様な人々のグループは、各指標について現在地を完全に理解するために必要です。

たとえば、最初の指標である lead time から始めます。参加者に、Lead time の指標に照らして現在自分たちはどこにいると考えるかを共有してもらいます。

- 6か月超。
- 1-6か月。
- 1週間-1か月。
- 1日-1週間。
- 1日未満。
- 1時間未満。

ドットが集まったら、参加者にそのドットの背後にある物語を共有してもらいます。特に外れ値は、チームの経験について貴重な洞察を与えてくれます。

活動を締めくくるには、これらの指標を今後1年でどう改善できると参加者が考えているかについて会話を促すべきです。グループはまだ、そこへどう到達するかを正確に決めているわけではありません。それでも、会話を開き、前にある努力に注意を向けることが重要です。この会話を始めるには、こう言うことができます。「1年後を考えたとき、これら4つの指標それぞれについて、私たちはどこまで行けると思いますか」

指標は、望ましいアウトカムと進捗の測定方法について重要な会話をもたらします。それだけでなく、現在状態と望ましい将来状態、そして望ましい状態がどれほど遠いかについての最初の会話に、全員の注意を向けさせます。この活動は、共通の目標を持つだけでなく、似た測定可能なレンズを通して進捗を見ることで、技術とビジネスを結びつけます。

### Objectives and Key Results

この活動の目標は、組織の主要な objectives についてそろい、成功へ向けた測定可能な進捗について会話を始めることです。この会話には多くの方法があります。この記事では OKR、つまり Objectives and Key Results を選びましたが、Lean Value Tree を作ることも、ビジネスビジョンを日々の仕事につなげる有効なアプローチです。もちろん、あなたの組織がすでに目標や objective のフレームワークを使っているなら、前回の objectives を持ち込むことで時間を節約できるかもしれません。

Objectives and Key Results、つまり OKR は、グループが goals、つまり Objectives を定義し、その後 Key Results という定量的指標でアウトカムを追跡するための目標設定フレームワークです。OKR は Intel で1970年代から使われており、Google、Oracle、Twitter、LinkedIn、Dropbox など多くの有名な組織が採用してきました。

Objectives は「どこへ行きたいのか」、Key Results は「そこへ向かっているかどうかをどう知るのか」です。

#### 手順

- 組織の高レベルの objectives を特定する。
- それらの objectives へ向けた進捗を測れる key results をブレインストーミングする。
- グループとして OKR を反復する。

OKR をうまく書くのは難しいことがあります。objectives は本当に価値あるアウトカムでしょうか、それともアウトプットでしょうか。Key Results は、Objectives を実現するために必要なものを本当に捉えているでしょうか。参加者に OKR を洗練する時間を与えることで、よくある誤りを避けます。

Objectives と Key Results を特定することは、これからの旅をより具体的にします。先へ進み、構築しようとするデータプロダクトを定義し始めるとき、これらの OKR を念頭に置くことで、アウトカムとアウトプットの間の隔たりを埋めやすくなります。

### Use Case を探索する

前のステップで Objectives を特定したら、次はその目標を実際にどう実現できるかに注意を向けます。Use case は、顧客やユーザーにとって望ましいアウトカムを翻訳したものです。分析データの文脈では、use case は通常、次のどちらかです。

Efficiency gains。

例: 効率を最適化するためにサービス利用を予測する。

または、

Experience improvements。

例: ユーザーの過去の購入に基づいてサービスを推薦する。

この活動は、グループが Data Mesh Use Case をブレインストーミングし、そろえる瞬間です。議論に焦点を当てるため、次の Use Case テンプレートを提供します。

私たちは、<この Use Case> が <この GOAL> の達成に役立つと考える。

#### 手順

- 参加者をグループに分け、上のテンプレートを使って use case をブレインストーミングしてもらう。
- 各グループに use case を全体へ発表してもらい、その後で議論とクラスタリングを行う。
- Use Case を洗練し、選択する。

この活動は、Data Mesh Use Case の定義と議論から始めるべきです。そうすることで、ブレインストーミングはより生産的になります。この段階では、後で絞り込むため、できるだけ多くのアイデアを集めたいところです。

グループが戻ってきたら、各グループのボランティアに、特定した use case を発表してもらい、より広い議論へ開きます。これは、似た use case を組み合わせる、あるいはクラスタリングする機会であり、分析的でない use case を除外する機会でもあります。

分析的 use case の集合ができたら、どれを次の活動へ進めるかを決める必要があります。use case が少数であれば、そのすべてを次の活動へ進めてもよいでしょう。ただし多すぎる場合は、優先順位を付けるべきです。優先順位付けの技法はかなり多くあります。あなたの文脈に合うものを選んでください。

「作れば人は来る」は、多くの変革の取り組みが陥る落とし穴です。そこで、提示したテンプレートを通じて Use Case を前に特定した Objectives と結びつけておくことで、作業が高レベルの取り組みのアウトカムとそろっていることを確実にします。

### Data Product の発見

Data Mesh は、摩擦を取り除き、価値を届け、データユーザーを本当に喜ばせるために Product Thinking を適用します。Data Product は Data Mesh の architectural quantum であり、「高い機能的凝集を持って独立にデプロイでき、その機能に必要なすべての構造要素を含む、アーキテクチャの最小単位」です。これは Evolutionary Architectures からの定義です。

これらの Data Product は Data Mesh の構成要素を表します。この活動では、上で特定した Use Case を満たす助けになる Data Product を特定します。一つ以上の Data Product の相互作用が、一つの Use Case を満たします。

前の活動で少数の Use Case を得たら、Use Case と Data Product の関係を理解し、提案し、マッピングし始める必要があります。

#### 手順

1. 選択した Use Case のための Data Product を特定する。
2. 各 Data Product の source と consumer を特定する。
3. Data Product 間の接続をマッピングする。

「この Use Case に対応するために Data Product を雇うとしたら、その仕事は何だろうか」

Jobs to be done、つまり JTBD は、Clayton Christensen が作った、顧客と、新しいプロダクトやサービスを採用する動機を理解するためのフレームワークです。Data Product を定義するために、私たちは同じアプローチに従うことを勧めます。上の単純な問いは、参加者が自分たちのデータをうまく使う方法について議論を始める助けになります。

Data Product の job を特定したら、実際にその job を遂行するにはどのデータが必要かを尋ねることで、少し深掘りできます。これには、既存の master data や operational system、手作業のデータ入力、あるいは他の Data Product さえ含まれるかもしれません。

Data Product の source を特定したら、このデータを実際に消費することに関心を持つのは誰かを尋ねるべきです。これには、特定の人、役割、他のデータシステム、他の Data Product が含まれるかもしれません。

Data Mesh のビジョンの一部は、再利用可能で相互運用可能な Data Product の mesh です。この活動の最後の部分では、Data Product が互いにどう相互作用するかをマッピングします。Data Product が別の Data Product から消費する場所があれば、ボード上にそのリンクを描きます。この活動の終わりには、Data Product interaction map が見え始めるはずです。

Data Product、その source、その consumer、そしてそれらの接続を特定することは、成功する Data Mesh 変革に不可欠な部分です。ワークショップのこの段階で、グループは最初の数個の Use Case について、パズルの非常に重要なピースをつなげているはずです。

### Wrap up and review

ワークショップを wrap up することは、体験に区切りの感覚を与え、進捗をレビューし、次のステップに関する会話を始めるために重要です。最後に短い要約を行い、次のステップについて文脈を提供すると、accelerate workshop はずっと完成度の高いものに感じられます。

ワークショップは、takeaway point と、ワークショップのさまざまな活動から得られたつながりを要約して締めくくるべきです。レビューで扱いたいトピックには、次のようなものがあります。

#### 手順

- ワークショップ中に特定した goals、use cases、data products。
- 興味深い観察、学び、推奨事項。
- 次のステップの詳細。

ワークショップのアウトカムに関心を持つ外部ステークホルダーを、このセッションへ招くことも役立つかもしれません。

最後の活動が終わったら、Data Mesh 変革の開始を楽しみにしながら、ワークショップを締めくくることができます。
