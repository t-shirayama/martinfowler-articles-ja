# Uniform Access Principle

## 要約

Uniform Access Principleは、モジュールが提供するサービスを、保存された値なのか計算された値なのかを漏らさない同じ記法で使えるようにすべきだ、という原則です。クライアントは、たとえば`age`がフィールドなのか計算結果なのかを知る必要がありません。

この原則はカプセル化、とくにdata hidingの重要な一部です。多くの言語は完全には支援していないため、JavaやC++の`getAge`のようなアクセサ命名規約で近づけることになりますが、保存値と計算値で命名を分けて原則を壊さないことが大切です。

## 読むときの観点

- クライアントに実装方式を知らせない、というカプセル化の観点から読む。
- 「保存された値」と「計算された値」を、同じ表記で扱えることの利点を見る。
- 遅い計算をどう知らせるかは例外的な問題であり、測定に基づいて判断する。
- read-onlyかどうかと、保存値か計算値かは別の問題として分けて考える。

## 原文の翻訳

> モジュールが提供するすべてのサービスは、それが保存によって実装されているのか、計算によって実装されているのかを明かさない、一様な記法で利用できるべきである。
>
> -- Bertrand Meyer

Bertrand Meyerは、非常に影響力のある著書『Object-Oriented Software Construction』でこの原則を作り出しました。

この原則の本質的な点は、personオブジェクトがあり、そのageを尋ねるなら、そのageがオブジェクトに保存されたフィールドであっても、計算された値であっても、同じ記法を使うべきだということです。これは実質的に、personのクライアントは、ageが保存されているのか計算されているのかを**知るべきでも気にするべきでもない**という意味になります。

これによりpersonオブジェクトは、保存と計算の間を簡単に切り替える柔軟性を得られます。また、たいていはクライアントにとって不要な関心も取り除けます。これはカプセル化の重要な一部です。少なくとも、カプセル化におけるdata hidingの側面ではそうです。

これはオブジェクト指向プログラミングの基本的な機能ですが、驚くほど少数のOO言語しか本当に従っていません。もちろんEiffelは例外です。ほとんどの言語は規約でそれを模倣しており、そのためJavaやC++のプログラムでは`getAge`のようなアクセサを使う習慣があります。

言語がどう振る舞うにせよ、プログラマがこの原則を破らないことが重要です。uniform accessを損なう規約を見かけることはよくあります。その例として、データアクセサは`getAge`と名付け、計算アクセサは`calcAge`と名付ける、というものがあります。私はどちらの場合でも、アクセサには**同じ付け方で名前を付ける**ようにしています。

uniform accessへの反論のひとつは、データアクセスは速い一方で計算は遅い場合があり、メソッドの応答が遅いかどうかをクライアントプログラマに示すことは有用だ、というものです。私はこれを特別な場合として扱う方を好みます。とりわけ遅い計算がある場合を除き、uniform access principleに従います。そしてその計算は、遅いと予想されるだけでなく、遅いと測定されたものであるべきです。その場合でも、caching accessorのように、探索できる他の選択肢があります。

uniform accessはgetterとsetterの両方に適用されますが、計算アクセサはしばしばread-onlyです。プロパティがread-onlyかどうかは、それが保存されたものか計算されたものかとは**別の問題**です。

### 注

この原則の表現は第2版からのものです。第1版では、MeyerはこれをUniform Reference Principleと呼んでいました。第2版でUniform Access Principleへ変更し、引用した定義を示しました。
