# The Agile Fluency Model

## 要約

Agile Fluency Model は、アジャイルなチームがどのような能力を身につけると、どのような組織的な利益を得られるのかを整理するモデルです。成熟度モデルのように「上に行くほどよい」と見るのではなく、組織がどの利益を必要とし、そのためにどの投資を引き受けるかを選ぶためのものです。

モデルは、Focusing、Delivering、Optimizing、Strengthening という4つのゾーンで説明されます。Focusing はビジネス価値への集中、Delivering は市場のリズムに合わせた安定的なリリース、Optimizing はよりよいプロダクト判断、Strengthening は組織全体を強くする学習と意思決定を扱います。

この記事の中心は、アジャイルの成果はチームの努力だけでは生まれないという点です。チームは練習によって fluency を高めますが、その練習を可能にする時間、権限、組織構造、マネジメントの支援がなければ、期待した利益は得られません。

## 読むときの観点

- Agile Fluency Model を「全チームが最上位を目指す階段」ではなく、必要な成果と投資を選ぶための地図として読む。
- fluency は個人の技量ではなく、チームとして圧力下でも自然に発揮できる習慣だと捉える。
- 各ゾーンの利益だけでなく、その利益を得るために組織が何を変える必要があるかを見る。
- アジャイルの失敗をチームの未熟さだけに帰さず、組織的な制約や支援不足にも目を向ける。
- Strengthening は確立済みの標準というより、組織設計とマネジメントの先端的な探求として読む。

## 原文の翻訳

アジャイル手法は主流になりましたが、その普及は問題も生みました。多くの組織では、期待したほどの成果が得られていないと感じられています。Agile Fluency Model は、アジャイルの考え方からどのような成果を期待できるのか、そのために何を投資すべきなのか、成果が出ないときにどこを見るべきなのかを考えるための枠組みです。

このモデルでは、チームが学習を進めるにつれて4つのゾーンを経験すると考えます。Focusing のチームはビジネス価値を生み、Delivering のチームは市場のリズムに合わせて出荷し、Optimizing のチームは市場をリードし、Strengthening のチームは組織全体を強くします。ここでいう proficiency は観察できる行動であり、fluency はその行動を、余裕のあるときだけでなくプレッシャーの下でも自然に発揮できる状態です。**fluency は個人ではなくチームの性質**として扱われます。

このモデルを成熟度モデルとして読むと誤解します。遠いゾーンほど常に優れているわけではありません。それぞれのゾーンには価値があり、同時にコストと難しさがあります。必要以上の投資を強いると、組織の反発を招き、アジャイルそのものへの信頼を損ねることもあります。どのゾーンが適切かは、組織の状況、プロダクトの性質、長期保守の有無、市場変化への向き合い方によって変わります。

fluency を得るには、知識を学ぶだけでは足りません。技法は研修で学べても、自然に使えるようになるには、数か月から数年にわたる意図的な練習が必要です。チームは複数の proficiency を並行して伸ばし、前進と停滞を繰り返します。組織は、チームに fluency を期待するだけでなく、その旅に必要な時間、支援、構造を提供しなければなりません。支援が不足すると、進歩が遅れるだけでなく、離職や冷笑的な文化を生む危険があります。

Focusing ゾーンは、アジャイルの基礎にあたります。チームは個人作業の集まりではなく、共有された目標に向かう協働チームとして働きます。技術的な作業項目だけでなく、スポンサー、顧客、利用者にとっての価値で計画と進捗を語ります。Scrum、Kanban、Extreme Programming の非技術的な部分、ユーザーストーリー、バックログ、レトロスペクティブ、タイムボックス、タスクボードなどが、このゾーンに向かう実践として使われます。

Focusing の主な利益は、透明性、達成感、整合性です。少なくとも月に一度、チームが何に取り組み、それがビジネス価値の観点でどう進んでいるかを示せることが重要な目安になります。このゾーンに到達するには、メンバーをチームに専任させ、ビジネス価値を理解する代表者を関与させ、個人競争や分断された配置を減らし、マネージャーが個人の作業割り当てではなく作業システムを支援する必要があります。

Delivering ゾーンでは、チームは価値に集中するだけでなく、市場が受け入れられる頻度で実際にリリースできるようになります。重要なのは、単にリリースした経験があることではなく、ビジネスが望むときに、低いリスクと小さいコストで最新の成果を出せることです。**Delivering はアジャイルの持続可能性**を表します。自動テスト、継続的インテグレーション、リファクタリング、DevOps 的な運用との連携など、幅広い技術的能力への投資が必要になります。

Delivering の利益は、欠陥の少なさと高い生産性です。短期的には技術習得のために速度が落ちることがありますが、長期的には変更しやすく、リリースしやすいソフトウェアを作れるようになります。技術的負債が多い場合、このゾーンへの到達には長い時間がかかります。組織は、学習の時間を認め、品質を犠牲にした短期的な圧力を避け、技術的卓越を支える必要があります。

Optimizing ゾーンでは、チームは市場や顧客の理解をもとに、より価値の高いプロダクト判断を行います。ビジネスの専門家や対象領域の専門家が、外部の依頼者としてではなく、チームの一員として関わります。チームは仮説を立て、実験し、必要なら短い予告で計画を大きく変えられるように仕事を組み立てます。ここでは、計画に従ったかではなく、ビジネス成果を生んだかが重視されます。

Optimizing には、組織構造への大きな変更が伴います。チームに予算、計画、結果への責任を持たせ、マネージャーは日々の戦術的判断から、方向づけや組織横断の調整へ役割を変えます。この投資は現状の権限構造を揺さぶるため、すべての組織に合うわけではありません。しかし市場変化を自らリードしたい組織にとっては、大きな見返りがあります。

Strengthening ゾーンは、アジャイルの将来像として説明されます。このゾーンのチームは、自分たちのプロダクトや市場だけでなく、より大きな組織システムの中での役割を理解し、そのシステムをより成功させるよう働きます。チーム間で学びを広げ、他チームを助け、複数チームからの洞察を組織の意思決定に流し込みます。

このゾーンはまだ探索的です。組織設計、複雑性理論、Open Space、Sociocracy、Holacracy などの考え方が参照されます。多くの組織にとっては、まず遠くから観察する対象かもしれません。ただし、チームの自己選択や戦略的な Open Space のように、Strengthening に向かう実践の一部は、完全な fluency を目指さない組織にも役立つ可能性があります。

Agile Fluency Model は現実を単純化したモデルですが、チームの能力、組織の期待、必要な投資を話し合うための有用な言葉を提供します。成果が出ないときは、チームを責める前に、複数のチームが同じ proficiency でつまずいていないかを確認します。同じ問題が広く現れているなら、それは個人や単一チームではなく、組織レベルの制約かもしれません。

結論として、このモデルは「どのアジャイル手法を使うべきか」を決める表ではありません。どの利益を望むのか、そのためにどの投資を引き受けるのか、そして組織がどの fluency を本当に可能にしているのかを考えるための会話の道具です。**より多くが常によいのではなく、必要な成果に見合う選択と支援が重要**なのです。
