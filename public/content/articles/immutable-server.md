# Immutable Server

## 要約

Immutable Serverは、一度デプロイしたサーバーをその場で変更せず、更新が必要になったら新しいイメージから作り直して置き換える考え方です。
Configuration SynchronizationやPhoenix Serverの発想をさらに進め、稼働中のインスタンスに手を入れる余地を減らします。

狙いは、構成のずれを抑え、テスト済みのベースイメージから同じ状態のサーバーを繰り返し作れるようにすることです。
ただし、ログ、データベース、ランタイム状態など、サーバーの外へ逃がすべきデータの扱いをあわせて設計する必要があります。

## 読むときの観点

- Phoenix Serverとの違いは、稼働後の構成変更を避ける点にある。
- ベースイメージ、外部設定、永続データの境界を分けて読む。
- 「サーバーを直す」のではなく「サーバーを置き換える」運用モデルとして捉える。
- Immutableにできない状態を、どこへ移すかが設計上の焦点になる。

## 原文の翻訳

CFEngine、Puppet、Chefのような自動構成ツールを使うと、サーバーのあるべき設定を記述し、新しいマシンや既存のマシンをその状態へ近づけられます。これは壊れやすいSnowflake Serverを避ける助けになります。また、必要になれば破棄して再構築できるPhoenix Serverも作れます。Immutable Serverはこの流れの自然な到達点であり、**一度デプロイされたら変更されず、新しい更新済みインスタンスで置き換えられるサーバー**です。

自動構成ツールは、長く動いているサーバーへ構成を繰り返し適用し、最新の仕様に同期させるためによく使われます。理論上は、サーバーを無期限に動かし続けながら、完全に一貫した最新状態へ保てるように見えます。しかし実際には、サーバー構成のすべてを完全に管理することは難しく、構成のずれや稼働中サーバーへの予期しない変更が入り込む余地が残ります。

そこでPhoenix Serverが役に立ちます。サーバーを定期的に破棄し、ベースイメージから再構築すれば、サーバーの要素は既知の状態へ戻ります。細かな構成仕様を過剰に維持しなくても、実行環境をリセットできるのです。

Phoenix Serverを使い始めると、構成管理の焦点は稼働中サーバーからベースイメージへ移ります。修正、変更、更新は稼働中のシステムではなくベースイメージに加え、そのイメージを自動テストへ通します。新しいサーバーを作るのは、その手順に合格してからです。

つまりPhoenix Serverの完全な状態は、ベースイメージ、自動構成管理、そしてデータの組み合わせで作られます。このため、自動構成管理だけでサーバーの100%を制御しなければならないという圧力は下がります。

しかし、短い寿命のサーバーに対して構成管理を何度も適用し続ける価値は小さくなります。むしろ、稼働中のシステムに変更を入れることはリスクを増やします。この考えは自然にImmutable Serverへつながります。

よくテストされたベースイメージからサーバーインスタンスを起動した後は、構成管理ツールを走らせるべきではありません。それは未テストの変更をインスタンスに入れる機会を作るからです。必要な変更はベースイメージへ加え、テストし、展開します。変更前のサーバーは破棄して置き換えます。

これはContinuous IntegrationやContinuous Deliveryの実践と同じ考え方です。ソフトウェアでは、あるバージョンのアプリケーションを一度だけデプロイ可能な成果物へまとめ、すべての環境で同じものを動かすほうが安全です。Immutable Serverでは、ベースイメージへの変更を単位にして、そこから作られたすべてのインスタンスが一貫しているとわかるようにします。

サーバーロールごとの主な違いは、サーバーの外から与えられる設定値にあります。多くの仮想化基盤やクラウド基盤では、新しいインスタンスを作るときにメタデータを設定し、稼働中のサーバーから読み取れるようにできます。Zookeeperのような中央レジストリから設定値を取得することもあります。Immutable Serverでは、インスタンスごとの設定項目の数と範囲を小さく保ち、可能ならそれらも自動テストへ通すのが望ましいでしょう。

### データの扱い

サーバーを使い捨てにできても、その上にあるデータまで使い捨てにできるとは限りません。Phoenix ServerやImmutable Serverを実装するときは、サーバーを破棄して作り直しても残すべきデータは何か、追加サーバーでスケールさせるために複製すべきデータは何かを考える必要があります。

実行時には必要ないが価値のあるデータは、インスタンスの外へ送れます。たとえばログファイルを中央のsyslogサーバーへ送る方法があります。共有ファイルシステムを使えば、複数のサーバーから同じファイルを参照できます。クラウドでは、EBSのようなマウント可能なストレージを新しいサーバーへ付け替えたり、複製してクラスタの拡張に使ったりできます。データベースについては、RDSのような管理サービスに任せることもあります。

Immutable Serverの本質は、稼働中サーバーを修理することではありません。**変更をテスト済みイメージへ閉じ込め、サーバーを置き換えることで一貫性を保つ**ことです。
