# Agentic Programming

## 要約

ソフトウェア開発では、開発者が IDE に直接コードを打ち込むのではなく、LLM に作業を依頼し、その結果をレビューする場面が増えています。Fowler はこの働き方を「agentic programming」と呼び、プログラミングの性質を大きく変えるものとして捉えています。

Agentic programming では、人間はコードへの責任を持ち続けます。vibe coding のようにコードの存在を忘れてしまうのではなく、生成されたコードを読み、テスト結果を見て、必要なら細かく指示し直します。また、IDE 内の高度なコード補完とも異なり、LLM エージェントがソースツリーを直接操作し、ファイルを作成・変更し、テストを実行しながら作業を進めます。

この変化は、今後のプログラマーに必要な活動や技能を問い直します。Fowler は、LLM を取り巻くガイドやセンサーを整える harness engineering が重要になり、同時に、ドメイン理解やユーザー・顧客との協働がさらに大切になると見ています。

## 読むときの観点

- Agentic programming を、単なるコード補完やチャットによる相談ではなく、LLM が作業対象のソースツリーを扱う開発形態として読む。
- 人間がコードを見ない vibe coding との違いを押さえる。
- 人間の役割が「打ち込む人」から「エージェントの作業を方向づけ、検証する人」へ移る点に注目する。
- 保存されたガイドライン、テスト、ログ、レビューなど、LLM の周辺にある仕組みを開発能力の一部として見る。
- ドメイン理解やユーザーとの反復的な協働が、AI 時代でも弱まらず、むしろ重要になる理由を考える。

## 原文の翻訳

ソフトウェア開発者が IDE にコードを直接入力する場面は、以前より少なくなりつつあります。代わりに、LLM へプロンプトを与えてコードを書かせ、その結果を人間がレビューする働き方が広がっています。これはプログラミングの性質を大きく変えるものです。人間はコードを生成する LLM エージェントを監督する立場になりますが、ソフトウェアが何をし、どのように動くかについての責任は引き続き人間にあります。

Fowler は、agentic programming を vibe coding と区別したいと述べます。vibe coding では、人間はコードを見ず、コードが存在することすら忘れてしまいます。一方、agentic programming では、人間はコードに関心を持ち続け、多くの場合は詳細にレビューします。また、IDE の中で LLM が高度なコード補完として参加する使い方とも異なります。

この働き方には、まだ広く定着した名前がありません。ただし「agentic」という形容詞はよく使われており、agentic programming、agentic software engineering、agentic coding などの言い方があります。Fowler は現時点では、**agentic programming** という呼び方に傾いています。

執筆時点では、agentic なツールは主にターミナル環境で動きます。プログラマーはプロンプトを出し、そのプロンプトには、LLM が従うべきガイドラインを説明する保存済み文書が含まれることもよくあります。LLM はソースツリーを直接操作し、ファイルを作成・変更し、コードを実行し、テスト結果を評価しながら、しばしば長い時間にわたって開発作業を続けます。

作業が終わると、人間はエージェントの成果を評価します。コードレビューを行い、テスト結果を確認し、その他の観測手段から得られる出力も見ます。つまり、人間の仕事は消えるのではなく、生成された結果を方向づけ、検証し、責任を引き受ける形へ移ります。

この変化は、将来のプログラマーにどのような活動が必要になるのか、どのような技能が求められるのかという大きな問いを投げかけます。現時点では、LLM の周囲にあるガイドやセンサーを整える **harness engineering** が中心的になりそうです。加えて、プログラマーが扱うドメインを理解し、ユーザーや顧客と協働しながら、製品を反復的に定義し構築する力の重要性も高まります。
