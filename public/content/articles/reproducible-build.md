# Reproducible Build

## 要約

Reproducible Buildは、過去の任意の時点のソースから、当時と同じ方法でソフトウェアをビルドし直せることを指します。
Continuous Integrationでは当然の前提として扱われがちですが、過去リリースの障害調査や修正では重要な能力です。

ホスト環境へ頻繁にリリースしている場合でも、再現可能なビルドは役に立ちます。
再現できないバグの調査、回帰の特定、欠けているテストの発見に使えるからです。

## 読むときの観点

- 「今のheadで直せるか」だけではなく、「過去のリリースを再現できるか」を見る。
- 再現不能なバグへの対応では、リリース版を再構築できることが理解を助ける。
- 回帰調査では、過去ビルドを二分探索のように使える。
- バグ修正と同時に、欠けていたテストを追加する姿勢が重要になる。

## 原文の翻訳

Continuous Integrationを支持する人々の一般的な前提のひとつは、ビルドが再現可能であるべきだということです。つまり、作業しているシステムの過去の任意のバージョンを取り出し、当時とまったく同じ方法でソースからビルドできるべきだ、ということです。

著者がよく参照するビルドプロセスの資料では、これは主要な実践として明示されていないことがあります。おそらく、それが基礎的な前提だからでしょう。あまりに当然と考えられているため、説明する必要がないと思われているのです。

再現可能なビルドを持つ大きな理由のひとつは、まだ使われている過去リリースの問題に対処できるようにすることです。1年前に顧客へソフトウェアをリリースし、今になって重大なバグが報告されたなら、そのソフトウェアを再作成して修正を届けられることが重要です。

では、毎週ホスト環境へリリースしている場合はどうでしょうか。さらに、しっかりしたContinuous Deliveryプロセスがあり、バグ修正は次のリリースまで待つか、非常に重要なら早期リリースできるとします。この場合でも、Reproducible Buildは必要でしょうか。

バグ報告を受け、headでそのバグを再現し、headで修正し、次のリリースを待つかすぐにリリースする、という流れだけで済むなら不要かもしれません。しかし、それでも再現可能なビルドがとても役に立つ場合があります。

まず、バグ報告を受けても再現できない場合です。そのとき、単に「直ったことにする」とは言いたくありません。著者なら、まず本当にそのバグを理解しているかを確かめたいと考えます。そのためには、リリースされたバージョンのソフトウェアをチェックアウトし、ビルドし、その上でバグを再現できるかを確認したいはずです。バグの再現に自信を持つには、ビルドも再現できる必要があります。

さらに、最近の開発の中で偶然そのバグが直っていたと確信できる場合でも、少なくともひとつテストが欠けていたはずです。そのテストを書き、現在は成功し、リリースされたビルドでは失敗することを確認したくなります。

もうひとつのケースはregressionです。顧客が「以前はなかったバグが今はある」と連絡してきたとします。この種のバグは、長い間隠れていて、ある条件で突然姿を現すことがあります。たとえば月初が月曜日に当たるときだけ起きるかもしれません。いずれにせよ、2か月前には動いていたと思うソフトウェアが、今はバグを持っている状況です。

ここでReproducible Buildがあれば、Diff Debuggingを使えます。顧客は2か月前には問題がなかったとかなり確信しています。当時のビルドが20000で、現在が28000だとします。そこでビルド20000をチェックアウトし、バグがあるかを見ます。なければ次に24000を試します。それでもなければ26000を試します。やがて、そのバグが最初に現れたrevision 26543を特定できます。現代のバージョン管理システムには、この作業を助ける機能があります。そこまでわかれば、revision 26543とその親との差分を見ることで、バグを見つけやすくなります。

Reproducible Buildは、単に古いソフトウェアを再び作るためだけのものではありません。**過去の振る舞いを確かめ、現在の修正に必要なテストを見つけ、問題が入り込んだ場所を絞るための土台**です。
