# Activity Oriented

## 要約

Activity Oriented は、分析、UX、開発、テストといった活動ごとにチームを分ける組織の作り方です。専門性の標準化や人材活用の効率化には魅力がありますが、各チームが自分たちの活動だけを最適化し、価値あるソフトウェアを届けるという全体像から離れやすくなります。

この記事は、ソフトウェア開発では活動別の分業よりも、成果に責任を持つ Outcome Oriented なチームのほうが適していると述べています。フィードバックを速くし、バッチサイズを小さくし、エンドツーエンドの応答性を高めるには、必要な活動をできるだけ一つのチーム内で行えるようにすることが重要です。

## 読むときの観点

- 活動別組織の利点が、ソフトウェア全体の価値提供と衝突する場面を見る。
- 開発チーム、テストチーム、プロダクト管理チームが別々の指標で動くと何が起きるかを考える。
- バッチサイズ、フィードバックループ、サイクルタイムへの影響に注目する。
- 専門性そのものではなく、専門性に沿って組織を分けることが問題だという区別を読む。
- Communities of Practice が、活動別組織に戻らず専門性を育てる手段として語られている点を見る。

## 原文の翻訳

重要なソフトウェア開発の取り組みでは、分析、ユーザー体験設計、開発、テストなど、いくつもの異なる活動が必要になります。Activity-oriented なチームは、こうした活動を軸に組織されます。つまり、ユーザー体験設計、開発、テストなどに専任のチームを置くのです。Activity-orientation は多くの利点を約束しますが、ソフトウェア開発ではたいてい、OutcomeOriented なチームで進めるほうがうまくいきます。

伝統的に、大規模な IT 部門を持つ大企業、つまり Enterprise IT では、マトリクス型の IT 組織、あるいは機能別組織から集めた activity-oriented なチームの束で、IT 開発プロジェクトを実行してきました。マトリクスの実線側、たとえば開発、テストなどの VP が率いる組織は、たいてい活動の境界に沿っていて、点線側のプロジェクト組織やプログラム組織へ「リソース」を貸し出します。よくある正当化の理由は次のようなものです。

- すべての開発者が単一の組織、つまりマトリクスの一つの腕に属していれば、開発における慣習や技法の標準化に役立つ。テストなどでも同じである。
- すべての開発者に長く続く開発・エンジニアリングのマネージャーがいれば、メンタリング、トレーニング、能力育成全般に役立つ。テストなどでも同じである。
- 互換可能だと見なされる開発者やテスターなどのプールからプロジェクトへ人を配置することで、人材の稼働率を最大化し、結果としてコスト効率を高められる。

しかし、activity-oriented なチームは、自分たちの活動を最適化しがちであり、役に立つソフトウェアを届けるという大きな絵を最適化しにくくなります。これは、何に責任を持たされ、どのように測られるかの結果です。開発者だけのチームが velocity だけで測られることは珍しくありません。スコープを届けることだけを任されているなら、そのスコープが解決するはずだった問題を本当に解決するのかまでは考えないでしょう。たとえ考えたとしても、仕様を決める責任だけを持つ別の activity-oriented チームであるプロダクト管理チームに止められるかもしれません。

活動で組織すると、チーム間で引き渡される作業のバッチサイズを小さくしにくくなります。別個のテストチームは、開発チームからストーリーを一つずつ受け取ろうとはしません。むしろ、リリース一回分のストーリー、少なくとも機能単位のすべてのストーリーをまとめてテストしたがります。これによりフィードバックループは長くなり、エンドツーエンドのサイクルタイムは増え、全体としての応答性は損なわれます。

高速な IT には、意欲あるチームが必要です。自律性は、チームにとって重要な動機づけ要因です。しかし activity-oriented な組織に与えられる自律性には限界があります。なぜなら、そうしたチームは自律性を、自分たちの活動領域の最適化に使いがちだからです。

Activity-oriented な組織の変種として、極端に専門化されたチームがあります。たとえば次のような形で現れます。

- ツールやスキルを中心にしたチーム。たとえば WebSphere Portal Server チームや BizTalk チーム。
- アーキテクチャ層ごとのチーム。たとえばプレゼンテーション層チーム、ミドルウェアチーム、データ層チーム。

こうしたチームには問題があります。焦点が狭く、全体像よりもチームの成果を最適化しがちだからです。もちろん、専門家を必要とするツールもあります。しかし、それは専門家を別チームに隔離する理由にはなりません。**問題は専門化そのものではなく、専門化の線に沿って組織すること**です。

### よりよく機能するもの

ソフトウェア開発は、反復的な設計プロセスです。本当の反復を実現し、速いフィードバックの価値を得るには、必要な活動を、できるだけ一つのチーム、つまり共通のレポートラインを持つチームの中で行う必要があります。多くのインターネット企業や独立系ソフトウェアベンダーは、すでにこのように運営しています。

Sriram の本では、アジャイルのアプローチを使って効果的な IT 組織を作る方法が、さらに詳しく扱われています。

Activity-oriented な組織は、稼働率を最大化するという観点からは魅力的に見えるかもしれません。しかしそれは、価値の付加とエンドツーエンドの応答性を最大化する妨げになります。今日のビジネス環境では、ソフトウェア開発のような資本的支出に関わるものでは特に、コスト効率よりも市場への応答性、つまり時間効率のほうが重要です。さらに、実線のレポートラインは、最も重要な目標である応答性と価値の付加に合わせるべきです。

レポート構造が活動に沿っていない状況でも能力を効果的に育てるには、Communities of Practice を設けるとよいでしょう。そこには、メンタリングできる力を持ち、コミュニティイベントの開催、本の購入、トレーニングの手配に使える予算を持つ、専門家のコミュニティリードを置きます。こうしたコミュニティは、標準化をやりすぎることなく、慣習や技法をそろえる助けにもなります。安定したレポート先のマネージャーについては、それが問題になるのはプロジェクト中心の IT の場合だけです。BusinessCapabilityCentric な構成であれば、安定したレポート先のマネージャーを持つことができます。
