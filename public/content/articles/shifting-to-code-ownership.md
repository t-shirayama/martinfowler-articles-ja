# Shifting To Code Ownership

## 要約

Shifting To Code Ownershipは、collective code ownershipを好むチームでも、状況によってはweak code ownershipへ寄せる判断が有効になり得ることを示す短い事例です。
規律の弱い変更が重要な領域に入り続けると、品質を重視する開発者がその修正に追われ、速度と士気の両方が損なわれます。

ただし、所有を強めるだけでは副作用もあります。
pair switching、ownerとの協働、再び自由に触れるための道筋など、制約を補うpracticeが必要になります。

## 読むときの観点

- code ownershipの形は信条ではなく、チームの状況に合わせて調整するものとして読む。
- collective ownershipを弱める判断が、なぜ速度と士気の改善につながったのかを見る。
- 「誰が触れるか」を制限するときに、学習や復帰の経路をどう残すかに注目する。
- agileやXPのpracticeも、local considerationsで調整されるべきだという結論を押さえる。

## 原文の翻訳

最近のCodeOwnershipの記事で、著者はcode ownershipの問題をどのように考えているかを説明しました。著者のソフトウェア開発の友人にはextreme programmerが多く、collective code ownershipを好む傾向があります。しかし、この種のpracticeは絶対的なものではなく、常にlocal considerationsによって調整されるべきです。著者の同僚のひとりが、次のような話を送ってくれました。これは、たとえXPの強い支持者であってもpracticeを変える必要がある場合を示す、よい例だと著者は考えました。その同僚は自分のチームについて話しているため、匿名を希望しています。

彼は、2人ほどの開発者による規律の弱いprogrammingに対抗するため、チームを「collective」モデルから「weak」モデルへ移したと言います。かなり率直なfeedbackと組み合わせた結果、velocityは向上しました。主要なコード領域をいま「所有」しているprogrammerたちが、標準に届かないコードを絶えず手直ししなくてよくなったからです。一方、それらの重要領域で標準に届かない作業をしていた人たちは、bug huntingやriskの低いcode changeのような仕事をするようになり、それによって他の人たちがさらに解放されました。

moraleにも正味の改善がありました。標準に届かない成果を出していた人たち以外は、彼らのcheck-inを毎回問題がないか監視し、見逃した問題を直さなければならないことに不満を募らせていたからです。この変更は、quality、TDD、non-speculationなどを真剣に扱う人々に報いるものになりました。

しかし、それを釣り合わせるために追加のpracticeやpolicyも必要でした。

- より頻繁なpair switching。実際のpolicyでは、コードのどの部分であっても作業はできます。ただし、自分が「free play」を持つ領域以外であれば、そこにfree playを持つ人とpairを組むか、少なくとも事前にその人たちを通じて自分の考えを十分に検討してもらう必要があります。
- 戻る道はownerを通ります。ownerが、あなたのコードは十分な水準に達すると安心できれば、その領域で再び自由にtaskを取れるようになります。
- 状況が改善しない場合は、さらに踏み込んだ対応が必要になります。

その同僚にとって、これは非常に学びの多い経験でした。以前にはここまでやる必要がなく、「厳しい役」を演じることに本当に気が進まなかったからです。enabling practiceではなくdirecting practiceを導入するのは彼にとってとても大変でしたが、それ以降、状況は大きく改善しました。

このようなlocal adaptationは、extreme programmingにとっても、あらゆるagile methodにとっても不可欠な部分です。ほかの条件が同じなら、その同僚はいまでもcollective code ownershipを好んでいます。しかし、**すべての条件が同じであることはめったにありません**。
