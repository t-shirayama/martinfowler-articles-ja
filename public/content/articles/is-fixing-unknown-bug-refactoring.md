# Is Fixing An Unknown Bug Refactoring

## 要約

`Introduce Null Object`のようなリファクタリングは、`null`参照で例外が起きていた箇所にデフォルトの振る舞いを与えることがあります。その結果、開発者が気づいていなかったバグが直ってしまうなら、それはまだリファクタリングと呼べるのか、という問いを扱っています。

Fowlerの答えは、おおむね「呼べる」です。重要なのは、プログラムの`observable behavior`、つまり外から見て保存したい振る舞いが変わったかどうかです。知らず、また保存する意図もなかったバグの振る舞いは、リファクタリングが守るべき振る舞いとは見なさなくてよい、という考え方です。

## 読むときの観点

- リファクタリングで守るべきなのは、すべての実行結果ではなく、意図して観察される振る舞いである。
- `Introduce Null Object`は便利だが、`null`チェックや例外に依存したコードを見落とすと影響が出る。
- 手作業のリファクタリングでは判断を入れられるが、ツールによるリファクタリングはより保守的になる。
- reflectionや文字列からのメソッド呼び出しのように、ツールでも完全には追跡しにくい呼び出しがある。

## 原文の翻訳

Przemyslaw Pokrywkaが提示した興味深い問題があります。リファクタリング本に載っている`Introduce Null Object`は、とても有用なリファクタリングです。しかし、このリファクタリングは振る舞いを変えることがあります。メソッドが`null`を返し、その`null`に対してメソッドを呼ぶと、通常はnull pointer exceptionが起きます。一方でNull Objectを使うと、何らかのデフォルトの振る舞いが返ります。

多くのリファクタリングは、ある意味では振る舞いを変えます。たとえば`Form Template Method`を適用すれば、プログラムの動き方は変わります。そこで重要になるのは、それが`Definition Of Refactoring`でいう**observable behaviorを変えるのか**という点です。つまり、プログラムが本質的に行うことを変えているのか、という問いです。

`Introduce Null Object`では、返された参照を扱う場所、とくに`null`かどうかを調べている場所を、プログラム全体から探す必要があります。このため、このリファクタリングは少し扱いが難しくなります。

問題の面白いところは、バグのある箇所を見落とした場合です。どこかで`null`参照に対してメソッドを呼んでいたとします。リファクタリング前なら例外が起き、ここではその例外を開発者が知らず、最上位のハンドラまで上がっていたと仮定します。リファクタリング後にはデフォルトの振る舞いが起き、それが結果としてバグを直してしまうかもしれません。

知らないバグを直してしまった場合でも、それはリファクタリングでしょうか。Fowlerは「そうだ」と考えます。知らなかった、あるいは保存するほど重要だと考えていなかったバグの振る舞いは、観察可能な振る舞いとは言いにくいからです。たとえバグの存在を知っていたとしても、その振る舞いを保ちたいと思っていなかったなら、リファクタリングと呼んでよい、という立場です。

ただし、これは境界にある事例です。Fowler自身も、考えを変える可能性や、さらに細かな例外を検討する余地があるとしています。

この話は、手作業のリファクタリングとツール主導のリファクタリングの違いも示しています。手作業では、このような判断を人間が下せます。一方でツールは、たいていもっと慎重である必要があります。それでもツールが常に振る舞いを保存できるわけではありません。単純な`Rename Method`でさえ、ファイルから読み込んだ名前を使ってreflectionで呼び出されていれば、リファクタリングによって壊れることがあります。
