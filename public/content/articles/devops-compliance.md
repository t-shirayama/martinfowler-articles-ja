# Compliance in a DevOps Culture

## 要約

規制対象の企業では、DevOps の速度とコンプライアンスの統制が対立するように見えがちです。
しかし、コンプライアンスを後工程の監査や承認ゲートとして扱うほど、遅く、弱く、責任の所在が曖昧な仕組みになりやすくなります。

この記事は、コンプライアンスをソフトウェア開発ライフサイクルに組み込み、証跡、権限、監査、セキュリティチェックを自動化し、プロダクトチームが統制を所有できるようにすることで、DevOps 文化と規制要求を両立できると説明します。

## 読むときの観点

- コンプライアンスを「遅くするための承認」ではなく、信頼できる変更のための設計制約として読む。
- 分離職務、監査証跡、変更承認を、手作業ではなく自動化されたパイプラインの一部として捉える。
- 統制を中央部門が抱え込むほど、実際のリスク管理が弱くなる可能性を見る。
- 文化、プロセス、ツールの三つがそろって初めて、継続的デリバリーとコンプライアンスが両立する点に注目する。

## 原文の翻訳

「DevOps」と「コンプライアンス」は、同じ会話に置くと相性が悪いように聞こえる言葉です。DevOps は高速なフィードバック、自動化、継続的デリバリー、チームの自律を強調します。一方、コンプライアンスは、統制、承認、証跡、分離職務を思い起こさせます。規制産業では、この2つはしばしば衝突するものとして扱われます。

しかし、私たちが目指すべき姿は衝突ではありません。コンプライアンスの目的は、組織が安全で説明可能な方法で働いていることを示すことです。DevOps の目的は、変更を小さく、頻繁に、安全に、測定可能にすることです。この2つは本来、同じ方向を向けます。**良い DevOps は、コンプライアンスを弱めるのではなく、より検証しやすくする**ことができます。

問題は、多くの組織でコンプライアンスが古い開発モデルに結びついていることです。年に数回の大きなリリースを前提に、事前承認、手作業のチェックリスト、会議体、別チームによる検査が積み重なります。このやり方は、変更が少なく遅い世界ではどうにか機能しました。しかし、変更が日々、場合によっては1日に何度も行われる世界では、同じ仕組みはボトルネックになるだけでなく、実際のリスク把握も難しくします。

### コンプライアンスは文化の問題である

DevOps 文化では、プロダクトチームが作ったものを本番で運用し、結果に責任を持ちます。コンプライアンスも同じ考え方に近づける必要があります。中央の監査部門やリスク部門が最後に確認するのではなく、プロダクトチームが日々の作業の中で統制を実行し、その結果を証跡として残せるようにします。

これは、統制をなくすという意味ではありません。むしろ、統制を実際に機能する場所へ移すということです。設計、コード、ビルド、テスト、デプロイ、監視の各段階に、必要なチェックを組み込みます。チームは「監査のために作業を止める」のではなく、「普段の作業が監査可能である」状態を作ります。

この変化には信頼が必要です。コンプライアンス担当者は、開発チームが統制を軽視しているわけではないことを理解する必要があります。開発チームは、コンプライアンス担当者が単に遅らせたいのではなく、会社と顧客を守ろうとしていることを理解する必要があります。互いを早い段階から巻き込み、制約を共有し、統制を自動化する方法を一緒に設計することが重要です。

### 後工程の承認から、組み込み統制へ

伝統的なコンプライアンスでは、変更を本番へ出す前に、別の人や委員会が承認することがよくあります。この考え方の背景には、変更者と承認者を分けることで不正や誤りを防ぐという発想があります。しかし、承認者が変更の中身を十分に理解していなければ、その承認は形式的になります。多くの承認は、実質的には「誰かがボタンを押した」という証跡でしかありません。

DevOps 環境では、変更は小さく頻繁です。そのため、すべての変更を人間の会議で審査するのは現実的ではありません。代わりに、変更が満たすべき条件を明確にし、それをパイプラインで機械的に検証します。コードレビューが行われたか、自動テストが通ったか、セキュリティスキャンが成功したか、インフラ変更が承認されたテンプレートに基づいているか、誰がどの変更をいつ行ったか。こうした事実を自動的に記録します。

この方法では、承認はなくなるのではなく、より具体的で検証可能になります。人間は個々の小さな変更を機械的に承認する代わりに、統制の設計、例外の扱い、重大なリスクの判断に集中できます。

### 分離職務を再考する

分離職務は、コンプライアンスで最もよく議論されるテーマの1つです。開発者が自分のコードを本番へデプロイできるなら、統制が失われるのではないかと心配されます。しかし、重要なのは「誰がボタンを押したか」だけではありません。誰が変更を作り、誰がレビューし、どの自動検証が通り、どの環境でどう実行されたかです。

手作業のデプロイでは、運用担当者が開発者の作った手順書を読んで実行することがあります。この場合、ボタンを押す人は分かれていますが、運用担当者が変更の意味を完全に理解しているとは限りません。手順の転記ミスも起こります。これに対し、自動化されたデプロイでは、変更はバージョン管理され、同じパイプラインを通り、すべての実行結果が記録されます。

したがって、分離職務は物理的な手作業の分離ではなく、権限、レビュー、監査証跡、検証の組み合わせとして設計できます。たとえば、1人の開発者が自分だけで本番変更を完了できないように、プルリクエストレビューを必須にし、承認されたパイプライン以外からの本番変更を禁止し、特権操作をログに残します。

### 監査証跡を自動的に集める

コンプライアンス作業で大きな負担になるのは、後から証跡を集めることです。誰が何を承認したのか、どのテストが実行されたのか、どの成果物が本番に入ったのかを、スプレッドシートやチケットシステムから探し回ります。このような作業は時間がかかり、誤りや抜け漏れも起こりやすいものです。

DevOps の実践は、この問題を大きく改善できます。ソース管理、CI/CD、アーティファクトリポジトリ、インフラ自動化、監視システムをつなげれば、変更の履歴は自然に記録されます。コミット、レビュー、ビルド、テスト、スキャン、デプロイ、ロールバック、インシデント対応が、時系列に沿った証跡になります。

監査のために別の文書を作るのではなく、実際にシステムを変更した流れそのものが証跡になります。これは監査人にとっても有益です。サンプルされた変更について、手作業の説明ではなく、実際のパイプライン実行結果を確認できるからです。

### コントロールをコード化する

Infrastructure as Code や Policy as Code は、コンプライアンスを DevOps 文化へ組み込むための強力な手段です。環境設定、ネットワークルール、アクセス権、暗号化設定、ログ保管期間などをコードとして管理すれば、変更はレビュー可能になり、テスト可能になり、再現可能になります。

たとえば、特定の種類のデータを扱うサービスは暗号化されたストレージだけを使える、インターネットから直接アクセスできるリソースは特定のタグと承認済み設定を持つ必要がある、本番データベースには多要素認証を通った特定ロールだけがアクセスできる、というルールを自動検査できます。

このような統制は、人がチェックリストで確認するよりも一貫して適用されます。例外が必要な場合も、例外そのものを明示的に記録し、期限や責任者を持たせられます。**統制をコード化すると、統制は口約束ではなく実行される仕組み**になります。

### セキュリティとコンプライアンスを早く入れる

DevOps 文化では、問題を早く見つけるほど修正コストが小さくなると考えます。コンプライアンスも同じです。リリース直前に監査やセキュリティレビューを行うと、問題が見つかったときの選択肢は少なく、手戻りは大きくなります。だから、セキュリティ担当者やコンプライアンス担当者を早い段階からチームの会話へ入れる必要があります。

これは、すべてのチームに専任の監査人を置くという意味ではありません。むしろ、共通のパターン、テンプレート、ガードレール、セルフサービスの助言を提供し、チームが自分で判断できるようにします。必要なときには専門家が相談に乗り、リスクの高い領域ではより深く関与します。

DevSecOps という言葉が示すように、セキュリティは開発と運用の間に追加される別工程ではありません。設計、実装、テスト、デプロイ、運用の中へ組み込まれます。コンプライアンスも同じ方向へ進むべきです。

### 例外を管理する

どれほどよく設計された統制でも、例外は必要になります。緊急修正、既存システムの制約、まだ自動化されていない領域、規制解釈が未確定のケースなどです。重要なのは、例外を隠さないことです。

例外は明示的に記録し、なぜ必要だったのか、誰が承認したのか、いつまで有効なのか、恒久対応は何かを追跡します。例外が多すぎるなら、それはチームが怠けているサインではなく、統制の設計が現実に合っていないサインかもしれません。例外の分析は、統制を改善する貴重なフィードバックになります。

### 監査人との関係を変える

DevOps 文化における監査は、最後に欠点を見つける活動から、統制の設計と運用が期待通りに機能しているかを継続的に確認する活動へ変わります。監査人には、パイプライン、バージョン管理、ログ、権限モデルを理解してもらう必要があります。開発チームには、監査人が何を証明しようとしているのかを理解してもらう必要があります。

監査人を早い段階で巻き込むと、どの証跡が必要か、どの統制が信頼できるか、どの部分に人間の判断が必要かを一緒に決められます。監査の準備のために数週間の作業を止めるのではなく、普段から監査可能な状態を作ることができます。

### 継続的コンプライアンス

DevOps における理想は、リリース可能な状態を常に保つことです。同じように、コンプライアンスでも、監査直前に慌てて整えるのではなく、常に説明可能な状態を保つことを目指せます。これを継続的コンプライアンスと呼べます。

継続的コンプライアンスでは、統制の状態をダッシュボードで見えるようにし、失敗した統制を早く検知し、修正します。依存ライブラリの脆弱性、未承認の権限変更、暗号化されていないストレージ、期限切れの例外、失敗したバックアップテストなどが、通常の運用指標と同じように扱われます。

これは、コンプライアンスを完全に自動化できるという意味ではありません。リスク判断、規制解釈、倫理的判断には人間が必要です。しかし、人間が注意を向けるべきところを、手作業の証跡集めから、本当に判断が必要な領域へ移せます。

### まとめ

コンプライアンスは、DevOps の敵である必要はありません。むしろ、DevOps の実践は、コンプライアンスをより強く、より一貫し、より説明可能にできます。そのためには、コンプライアンスを後工程のゲートとして扱うのをやめ、開発と運用のライフサイクル全体へ組み込む必要があります。

チームが統制を所有し、パイプラインが証跡を集め、ポリシーがコードとして表現され、監査人が早い段階から関わるとき、組織は速さと安全性のどちらかを選ばなくて済みます。規制対象の環境でも、継続的デリバリーと説明責任は両立できます。
