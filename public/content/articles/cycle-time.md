# Cycle Time

## 要約

Cycle Time は、ソフトウェアの新しい機能がアイデアから本番環境で動くところまで進むのにかかる時間を表す指標です。
アジャイル開発では、小さな機能に分け、待ち時間を減らすことで、この時間を短くしようとします。

ただし、どこを開始点とし、どこを終了点とするかはチームや文脈によって揺れます。
そのため、Cycle Time を比較するときは、同じ測り方をしているかを先に確認する必要があります。

## 読むときの観点

- Cycle Time は単なる速度指標ではなく、フィードバックを早めるための見方である。
- 開始点と終了点の定義が違うと、数値の意味も変わる。
- Lead Time と Cycle Time の使い分けは業界で安定していない。
- Value Stream Map を使うと、作業時間と待ち時間を分けて見やすい。
- 機能を小さくすることと Slack を増やすことが、短縮の有効な手段になりうる。

## 原文の翻訳

Cycle Time とは、ソフトウェアシステムの新しい機能が、アイデアの段階から本番環境でユーザーの役に立つところまで進むのにかかる時間である。アジャイルの文脈では、この時間をできるだけ短くしようとする。小さな機能を定義して実装し、開発プロセスの待ち時間を減らすことで、より早く学習できるからだ。

アジャイル開発の重要な特徴は、作業を分析、実装、テストといった活動別に分けるウォーターフォール型から、機能の小さなまとまりごとに進める反復型へ移ることにある。小さな機能をユーザーの前に出すことで、**実際の利用から得られるフィードバックループ**が生まれる。

このフィードバックが速いほど、開発プロセスもプロダクトの価値も改善しやすくなる。そのため、機能がワークフロー全体を通って本番に届くまでの速さが重視される。Cycle Time は、その速さを表すための言葉である。

しかし、測定には難しさがある。終了点は比較的わかりやすく、本番環境でユーザーに価値を届け始めた時点と考えられることが多い。それでも、Canary Release の最初の対象ユーザーに届いた時点なのか、全ユーザーへ展開された時点なのか、アプリストアの承認待ちを含めるのかといった曖昧さが残る。

開始点はさらに揺れやすい。最初のコミットを開始点にすることもあれば、事前分析を含める考え方もある。顧客が最初にその機能を思いついた時点まで戻る人もいるが、優先度が低く、しばらくトリアージ領域に置かれていたアイデアまで同じように扱うと、別の意味を持つ数値になる。

Lead Time という言葉も、Cycle Time と同義に使われたり、開始点の違いで区別されたりする。ただし、その区別には一貫性がない。したがって、両方の言葉が出てきたら、まずその人や組織が何を開始点と終了点にしているのかを確認するのがよい。

Cycle Time の異なる範囲には、それぞれ利点がある。同じ状況で複数の範囲を使い、たとえば「最初のコミットからの Cycle Time」と「アイデアからの Cycle Time」のように形容語を添えて区別すると、違いを明確にしやすい。

つまり、Cycle Time は有用だが滑りやすい概念である。チーム間で比較するなら、同じ定義で測っているという確信が必要だ。それでも、Cycle Time を考え、それを短くしようとすることには価値がある。

実践上は、アイデアから本番までのすべてのステップを Value Stream Map として描くとよい。各ステップで実際に作業している時間と、その間の待ち時間を見える化すれば、流れを改善する手がかりが得られる。よく効く介入としては、機能の粒度を小さくすること、そして直感に反するように見えても **Slack を増やすこと**が挙げられる。

仕事の流れを理解し、改善する努力には価値がある。アイデアが早く本番へ届くほど、新機能の恩恵を早く得られ、働き方を改善するためのフィードバックも早く得られる。
