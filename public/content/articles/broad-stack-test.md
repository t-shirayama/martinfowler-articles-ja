# Broad Stack Test

## 要約

Broad Stack Test は、大きなアプリケーションの大部分をまとめて動かすテストです。エンドツーエンドテストやフルスタックテストと呼ばれることもあります。限定された一部だけを対象にする Component Test とは対照的ですが、両者の境界は明確な線ではなく連続的です。

アプリケーション全体に近い形で動かせるため、コンポーネント間の相互作用に起因するバグを見つけやすい一方で、遅く、壊れやすく、保守しにくくなりがちです。そのため、テストピラミッドでは Broad Stack Test を少数に抑えることが推奨されます。

## 読むときの観点

- Broad Stack Test と Component Test の違いを、二分法ではなく連続体として捉える。
- UI から操作するか、サービスインターフェースや UI 直下から操作するかで、範囲と扱いやすさが変わる。
- リモートシステムを含めると遅く壊れやすくなるため、test double と contract test の組み合わせを検討する。

## 原文の翻訳

Broad Stack Test とは、大きなアプリケーションのほとんどの部分を実行するテストである。しばしば end-to-end test や full-stack test と呼ばれる。これは、システムの明確に定義された一部分だけを実行する Component Test と対比される。

Broad Stack Test と Component Test の違いは、明確な境界線ではなく連続体である。スタックの完全さから一部が欠ける領域の一つは、テストがアプリケーションをどのように操作するかにある。

Broad Stack Test は、Selenium や Sahi のようなツールを使って Web アプリケーションをテストする場合のように、UI 経由でアプリケーションを操作することが多い。しかし、UI のすぐ下で操作する Subcutaneous Test でも、ソフトウェアの残りの大部分を実行しているなら Broad Stack Test になりうる。さらに範囲を限定して、サービスインターフェース経由でアプリケーションを実行するテストも、サーバ側の Broad Stack Test と考えられる。

スタック全体を覆わないもう一つの領域は、リモートシステムとの接続である。私を含め多くの人は、リモートシステムを呼び出すテストは不必要に遅く、壊れやすいと考えている。通常は、そうしたリモートシステムには Test Double を使い、その double が正しいかを Contract Test で確認するほうがよい。

Broad Stack Test の利点は、アプリケーションの各部品を接続した状態で実行できることにある。そのため、Component Test では見つけにくい、コンポーネント間の相互作用に起因するバグを見つけられる。

しかし Broad Stack Test は、Component Test よりも保守が難しく、実行も遅くなりがちである。したがって Test Pyramid は、**Broad Stack Test は少数に抑えるべきだ**と示唆している。
