# Getter Eradicator

## 要約

Getter を一律に悪とみなして取り除こうとする姿勢は、カプセル化を守ろうとする動機から生まれます。
しかし、カプセル化の本質は単にデータを隠すことではなく、変わりうる設計判断を隠すことにあります。

この記事は、getter を減らすこと自体ではなく、データを取り出して外側で手続き的に処理していないかを見るための文章です。
getter は危険信号になることがありますが、オブジェクト同士が協調するために正当にデータを受け渡す場面もあります。

## 読むときの観点

- getter の有無だけでなく、振る舞いがどこに置かれているかを見る。
- 「Tell, Don't Ask」がどの場面で効くのかを考える。
- Data Class が振る舞いを失っていないか疑う。
- カプセル化を「データ隠蔽」だけでなく「変化しうる設計判断の隠蔽」として読む。

## 原文の翻訳

Getter Eradicator たちは、getter メソッドを見たときに口の左端がぴくりと動くので見分けがつきます。戦斧をすばやく引き抜き、またひとつの getter がクラスから容赦なく切り落とされると、男らしい Getter Eradicator の足元でそのクラスは感謝の歓喜に身を震わせる、というわけです。

まあ、英国ビールに戻ったことが少し効きすぎているのかもしれません。とはいえ、Chris の穏やかなからかいは、私の小さなこだわりを突いてきました。クラスに getter メソッドを持たせるのは避けるべきだ、そういうものはカプセル化違反だ、と言う人たちにはよく出会ってきました。Allen Holub の記事は、その代表的なもののひとつです。

一般的な正当化は、getter がカプセル化を破るというものです。もし `overs`、`runs`、`wickets` というフィールドを持つ `bowler` クラスがあるなら、getter、つまり `getOvers`、`getRuns`、`getWickets` を追加するのは、フィールドを public にするのと大して変わらない、というわけです。

この議論には一理ありますし、私も本当に必要になるまではアクセサを書かないように勧めています。しかしこの議論には、カプセル化の要点を見失う危険もあります。私にとってカプセル化の要点は、実のところデータを隠すことではなく、特に将来変わるかもしれない領域で、**設計上の判断を隠すこと**にあります。内部データ表現はその一例ですが、唯一の例ではなく、常に最良の例でもありません。外部データストアと通信するためのプロトコルは、カプセル化のよい例です。そこではデータ表現よりも、そのストアへ送るメッセージのほうが重要です。カプセル化について考えるときは、「データを公開しているか」ではなく、「どの変動要素を、なぜ隠しているのか」と問うほうがよいと思います。Craig Larman は、この話題について私のためによいコラムを書いてくれました。

getter eradicator たちが集まるときの合言葉は、たいていカプセル化の防衛です。しかし、彼らの本当の動機は、もっと筋が通っていて実用的なものだと思います。オブジェクト指向言語で書かれたコードの中には、設計としては手続き型になっているものが大量にあります。現代の言語がオブジェクトに支配されているという意味では、オブジェクト指向コミュニティは「勝った」のかもしれません。しかし、オブジェクト指向プログラミングが広く使われるようになったという意味では、まだ勝っていません。その結果、本来ならオブジェクト自身に置いたほうがよい振る舞いを実行するために、オブジェクトからデータを引き出す手続きが今でもよく見られます。これは Pragmatic Programmers の原則である **「Tell, Don't Ask」**に反しています。こうした手続き型プログラミングは getter があるからこそ可能なので、getter をなくせと人に言うことは、振る舞いを適切な場所へ移す後押しになります。

この動機には大いに共感します。ただし、getter を避けよとだけ言うのは、かなり鈍い道具だとも思います。オブジェクト同士がデータを交換して協調しなければならない場面は多く、その場合には getter が本当に必要になるからです。

単純な経験則を探すなら、私が好むのは Kent Beck から最初に聞いたものです。同じオブジェクトに対して、あるコードが複数のメソッドを呼び出している場合には常に注意する、というものです。これはアクセサでも起こりますし、もっと妥当なコマンドでも起こります。あるオブジェクトから2つのデータを尋ねているなら、計算しようとしているデータそのものを求める1つの依頼に置き換えられないでしょうか。あるオブジェクトに2つのことを命じているなら、1つのコマンドに置き換えられないでしょうか。もちろん、そうできない場合もたくさんありますが、**その問いを立てる価値は常にあります**。

もうひとつのよい警告サインは Data Class です。つまり、フィールドとアクセサだけを持つクラスです。これはほとんど常に問題の兆候です。振る舞いが空っぽだからです。そうしたクラスを見たら、いつも疑ってかかるべきです。そのデータを使っているのは誰かを探し、その振る舞いの一部をオブジェクトへ移せないか見てください。このような場合、「この getter をなくせるだろうか」と自問するのは役に立ちます。たとえなくせなくても、その問いによって振る舞いをよい方向へ動かせることがあります。

オブジェクト間で振る舞いをどう配分するかは、オブジェクト指向設計の本質です。したがって、どんな設計でもそうであるように、厳密で即座に適用できる規則があるわけではなく、トレードオフの判断があります。Craig Larman が「Information Expert」と呼ぶように、データと同じクラスに振る舞いを置くことは、最初に検討すべき選択肢です。しかし、それが唯一の道ではありません。レイヤリングがそれに優先することはよくありますし、Gang of Four のパターンの多くは、特定の必要に応じてデータと振る舞いを分けています。よい経験則は、**一緒に変わるものは一緒に置く**ということです。データと、それを使う振る舞いは一緒に変わることが多いですが、それ以上に重要な別のまとまりが見えてくることもよくあります。
