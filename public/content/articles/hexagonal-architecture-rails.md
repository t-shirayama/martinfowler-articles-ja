# A Conversation with Badri Janakiraman about Hexagonal Rails

## 要約

Martin Fowler 氏と Badri Janakiraman 氏による、Ruby on Rails における Hexagonal Architecture の役割についての動画対談ページです。1本目では、Active Record と Data Mapper のトレードオフ、特にデータベースを hexagon の外側に置くかどうかが扱われます。2本目では、Rails をひとつのプラットフォームとして受け入れるのか、それとも部品の集まりとして選択的に使うのかが議論されます。

中心にあるのは、Hexagonal Architecture を機械的な形としてではなく、何を内側に置き、何を外側として隔離するのかを考えるための判断枠組みとして読むことです。

## 読むときの観点

- データベースを常に外部コンポーネントと決めつけず、アプリケーションの性質に応じて判断する。
- 複雑なドメインロジックや制御できないデータベースがあるとき、Data Mapper が有利になる理由を見る。
- Rails の sweet spot にいるなら、フレームワークから過度に隔離しようとするコストも考える。
- 隔離すべき対象はプラットフォームそのものではなく、外部システムや中核でない機能である、という区別に注目する。

## 対応範囲

原文ページは動画2本と、それぞれの概要・Minutes を掲載しています。会話全体の transcript は公開ページ上で確認できないため、ここではページ本文、各動画の概要、公開されている Minutes を翻訳しています。

## 原文の翻訳

最近、Kent Beck、David Heinemeier Hansson と「TDD は死んだのか」について話してきたことから、ほかにも多くの会話が生まれました。そのひとつが、Thoughtworks の senior developer のひとりである同僚の Badri Janakiraman とのやり取りです。Badri は10年以上の経験を持ち、Ruby on Rails だけでなく、Java や .NET など、ほかの開発スタックでも仕事をしてきました。

ここ6年ほど、彼は私たちの製品部門である Studios で働いており、その時間の多くを、長く続いている Rails プロジェクトであるプロジェクトコラボレーションツール Mingle に費やしてきました。

私たちの議論はとても興味深かったので、それを動画として記録すると面白いだろうと思いました。そこで時間を取り、ビデオチャットを行いました。Ruby on Rails における Hexagonal Architecture の役割について、興味深い視点を得られるはずですし、「Is TDD Dead」の議論に対する有用な補足にもなると思います。

### 1: Active Record or Data Mapper?

2014年6月5日。

動画。

私たちは Hexagonal Architecture の考え方を紹介し、Badri は、データベースを hexagon の外側に押し出すために Active Record を使うのか Data Mapper を使うのか、そのトレードオフを説明します。

### Minutes

Badri は、Alistair Cockburn がもともと示した Hexagonal Architecture の考え方を説明するところから始めました。このアーキテクチャの主な考え方は、ドメインが外部コンポーネントに依存しない自己完結した実体である、というものです。これにより、そうしたコンポーネントから独立してアプリケーションを考え、テストできるようになります。

そこから、何が外部で、何が内部なのかという問いが生まれます。Rails における Hexagonal Architecture の説明では、たいていデータベースを外部として扱いますが、それは当然の前提ではありません。元の記事では、さまざまな形の UI を通じてアプリケーションを駆動することも取り上げています。

Martin は、Hexagonal Architecture と Layered Architecture の関係を描きました。よくあるレイヤリングでは、UI、domain、data source のような層を持ち、依存関係は UI から domain、domain から data source へと流れます。Hexagonal Architecture は、domain と data source の間の依存関係を変え、依存が UI から domain へ、そして data source から domain へ向かうようにします。

Martin が『Patterns of Enterprise Application Architecture』を書いたとき、彼はこのアプローチを Data Mapper パターンとして説明しました。これは、ドメインをデータソースと結びつける Active Record と対比されるものです。

Martin は Badri に、Data Mapper と Active Record の違いをどう特徴づけるかを尋ねました。Badri の答えは、構築する必要があるアプリケーションの種類次第だ、というものでした。キャリアの初期に、彼はリースや保険など、複雑なビジネスロジックを持つアプリケーションを作っていました。ドメインにそのような複雑さがある場合は、ドメインをデータベース上の関心事から分離する必要があります。

ここ6年ほど、彼は Rails を使って、もっと単純なドメインロジックを持つ製品アプリケーションに取り組んできました。たとえば Mingle は、カードウォールのメタファを使ってバックログを管理するプロジェクトコラボレーションツールです。Thoughtworks の哲学では、ツールがチームのプロセスを制約すべきではないため、ユーザー定義フィールドを必要とするデータの柔軟性が多く求められます。

ユーザー定義フィールドに対する通常のアプローチはあまりうまく機能しないため、私たちはカスタムフィールドを追加するためにリレーショナルスキーマを動的に変更しています。これは非常にうまく機能しており、ドメインオブジェクトをデータベースに結びつけ、データベースをアプリケーションの中心へ持ち込んでいます。Martin は、ここでの主題は、**ドメインモデルとデータソースを独立して進化させたいときに Data Mapper を使う**ということだ、とまとめました。

Badri はさらに、Rails 以前の仕事では、アプリケーション開発者がデータベース構造を完全には制御できないケースがあったと述べました。そうした場合、Data Mapper はより魅力的になります。Data Mapper はデータベースからの隔離をより多く与えてくれますが、完全な隔離ではありません。特定の必要に応じて、同じデータを異なるドメインモデルオブジェクトで読み込むなど、データ管理上の関心事を考慮する必要は残ります。

Badri は、Mingle がいくつかの異なるデータベース、つまり Postgres、Oracle、そして一時期は MySQL で動くことを説明しました。単体テストがデータベースにアクセスするという事実は、これらのデータベース間の移植性に大いに役立ちます。これは、そのためには隔離が必要だという一般的な考えとは反対です。

Martin は、Data Mapper と Active Record のトレードオフを次のようにまとめました。Data Mapper を使う理由は、統合データベースを使っている場合などデータベースを制御できないこと、または複雑なドメインロジックがあることです。こうした力が働いていないなら、Active Record のほうが単純であり、したがってより良い選択です。ここでは、データベースの大きさは要因ではないことも指摘しました。Mingle は5万行規模のアプリケーションで、100を超えるデータベーステーブルを持っていますが、それでも Active Record でうまく機能しています。

Martin は、テストの実行にどれくらい時間がかかるのかを尋ねました。Badri は、複数のブラウザとデータベースで並列にテストを走らせて、およそ10分程度だと答えました。チームは、10分の commit suite であれば flow を保つのに十分速いと考えています。彼らは pending head にコミットしています。

### 2: Rails: Platform or Suite of Components?

2014年6月12日。

動画。

Ruby on Rails のような豊かなフレームワークを使うとき、それをプラットフォームとして扱うことも、部品の集まりとして扱うこともできます。Badri はその違いについて話し、私たちはその判断に伴うトレードオフを議論します。

### Minutes

この会話は、Martin が Rails やそれに似たフレームワークに対する姿勢を尋ねるところから始まりました。欠点も含めて受け入れ、プラットフォームとして扱うべきでしょうか。それとも、好きな部分を選び、気に入らない部分は使わない、部品の集まりとして扱うべきでしょうか。Badri は、Rails はモジュール化の面でかなり進んできたものの、現時点ではコンポーネント全体を入れ替えるより、Rails の一部を使わないようにするほうが容易だと答えました。

たとえば、メールを送受信する必要がないアプリケーションでは、Action Mailer を外すことができます。標準的な Ruby アプリケーションに Action Support を取り込み、それが提供する多くの便利メソッドや core extension を利用することもできます。とはいえ、たとえば Rails の request-response dispatch cycle を、Node.js のような evented system に簡単に置き換えることはできません。これは、Rails が非常に特定の種類のアプリケーションを支えるように作られているからです。

Martin は、Rails は Basecamp を作るためのフレームワークだという決まり文句に触れて笑いました。

Badri はそれに対し、Rails の sweet spot は、リレーショナルデータストアと会話する、深く Web 的なアプリケーションだと述べました。言い換えると、アプリケーションのユーザー数には限りがあり、人間が Web UI を通じて使う場合でも、ほかの機械が REST API を通じてアプリケーションとやり取りする場合でも、Web が偶然の要素ではないようなアプリケーションです。Badri はさらに、問題は人々がフレームワークの採用をあまりに早く決めてしまうことにあるのかもしれない、と推測しました。

フレームワークは、多くの便利さと引き換えに、多くの設計上の選択肢を取り去ります。だからこそ、まず自分たちがアプリケーションをどのように設計したいのかを理解し、その設計上の選択を妨げないフレームワークを選ぶべきかもしれません。最初にフレームワークを選び、それがシステムを望むように設計する邪魔になる、という順番ではありません。

Martin は、フレームワークを使い、アプリケーションをある方法で設計するという選択は、かなり前に行われているかもしれないと答えました。とはいえ、フレームワークの設計者が別の道を選ぶこともあり、その結果、自分たちが行き詰まることもあります。Martin はここまでの会話を、フレームワークの sweet spot にいるなら、将来の変化に備えて最初から自分たちを隔離する労力をかけるより、**フレームワークをプラットフォームとして受け入れるほうが理にかなう**、とまとめました。

Badri はこのまとめに同意し、追加した層が設計上の思考を導く助けになると感じる人もいると述べました。ただし、彼自身は Rails をプラットフォームとして使う側に寄る傾向があります。

フレームワークをプラットフォームとして使うという彼自身の好みは、過去にフレームワークを迂回しようとした経験から来ています。.NET の rich-client framework の例を使い、彼は、双方向データバインディングを使わないようにしようとしたアプリケーションでの経験を思い出しました。チームの一部メンバーがそのアプリケーションをどうテストすべきだと考えていたかの影響で、データバインディングの仕組みを使わずに humble view を作ろうとしたのです。

その結果、かなり大きな混乱が起きました。フレームワークはそのように使われるように設計されておらず、私たちはあまりにも多くのコードを書き、フレームワークの大きな部分を再発明することになったからです。これは、フレームワークの流れに逆らいたいチームが理解しておくべきコストです。そのコストが見合う場合もあるかもしれませんが、多くの場合に非常に大きなコストがあるという事実は避けられません。

彼はさらに、フレームワークに大きな変更を加えたいと感じるなら、チームは best of breed のライブラリを選び、アラカルトなアプリケーションを構築することを望むかもしれない、と付け加えました。Martin は、重要なのは、必要に応じて、フレームワークを Platform として使うのか Suite of Components として使うのかを情報に基づいて選ぶことであり、どちらか一方を盲目的に全面採用することではない、と繰り返しました。その際には、どちらのアプローチのコストと利点も考慮する必要があります。

Martin は、フレームワークが特定の設計上の選択を代わりに行ってくれることに価値はあるのか、それによってチームの新しいメンバーがより早く立ち上がれるのか、と尋ねました。Badri は、これは Java 世界の Spring なども含め、フレームワーク一般の利点だと答えました。

彼は、フレームワークを使う人々が従う共通の利用パターンのほうが、自分にはより重要に感じられたと述べました。たとえば Enterprise Java の世界では、2001年頃、ほぼすべてのアプリケーションが、トランザクション境界をどこに置き、どのように実装するかについて独自の実装を持っていました。彼は、関わるアプリケーションごとに、それを何度も学ばなければなりませんでした。

一方 Rails では、すべての request-response cycle を包む filter の中でトランザクションを開始し、ロールバックまたはコミットするという共通パターンがあります。この種の学びは、ひとつの Rails コードベースから次の Rails コードベースへ持ち運べるため有用です。

Martin は、人々がフレームワークをその sweet spot ではない場所に押し込まないよう呼びかけました。Badri は、この点はまったく妥当だとしたうえで、Rails はフレームワークが意見を持たない場所では、さまざまな設計上の選択に対して確かに扉を大きく開いている、と付け加えました。彼は、関わったプロジェクトからいくつかの例を挙げました。

最初の例は、バージョン管理システムからコミットデータを取得し、それをアプリケーション内のほかのデータの文脈で表示するというものです。この機能がなくてもアプリケーションが動く必要があり、また複数のバージョン管理システムからのデータをアプリケーションに取り込む必要があったため、彼らは、この情報がシステムに入る入口を定義する port を設計しました。

この port は、その port に到着する各種データに対して適切な adapter を使い、すべての incoming data を適応させる単一の場所になりました。

2つ目の例は双方向通信です。彼らは Jabber のチャットルームボットを作り、チームのチャットルーム内のメッセージのうち、作業中のストーリーに関連するものを取り込み、バックログを管理するアプリケーション内のそのストーリーに discussion thread として添付しました。

この場合も、この機能はアプリケーションの中核ではなく、個々のチームが必要に応じて有効にも無効にもできる必要がありました。そのため、チャットルームとコアアプリケーションの間のすべての通信を扱う gateway を実装しました。コアアプリケーションは、この機能が設定されているかどうかをまったく意識しません。これにより、ドメインロジックの中心に対して十分な隔離を得られました。

Martin はこれら2つの例を、Rails を使うことは隔離を完全に避けることではない、とまとめました。彼が強調するのは、プラットフォームから自分を隔離したいわけではないが、外部の部品からは隔離したいという点です。アプリケーションのどの部分が外部で、どの部分が内部なのかという区別は、私たちの会話を最初の Hexagonal Architecture パターンへ一周させて戻してくれます。
