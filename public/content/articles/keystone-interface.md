# Keystone Interface

## 要約

Keystone Interfaceは、大きな機能を小さく統合しながら、利用者に見える最後の入口だけを後で差し込むための考え方です。バックエンドや内部連携は段階的に実装して本番へ入れ、UIやAPIなど利用者側のインターフェイスを最後の「要石」として加えることで、長命なfeature branchを避けやすくします。

この手法がうまく働くには、隠れているコードも十分にテストできる構造が必要です。UI越しだけでなく、subcutaneous testsや低いレイヤのテストで確認できるアーキテクチャが前提になります。

## 読むときの観点

- ユーザーに見える入口を最後に足すことで、未完成機能を本番コードに統合できる点を見る。
- Feature Flagsとの違いと、トグルをUI側に閉じ込める利点を考える。
- UIを最後に作る危険を避けるため、薄い縦スライスでの開発と組み合わせる必要がある。
- UIだけでなく、APIなど他の消費者向けインターフェイスにも応用できる。

## 原文の翻訳

ソフトウェア開発チームは、できるだけ頻繁に作業を統合すると、ずっと楽に進められることに気づきます。また、本番へ頻繁にリリースすることにも価値があります。しかしチームは、半分だけできた機能をユーザーに見せたいわけではありません。この緊張を扱うための有用な技法は、バックエンドのコードをすべて作って統合し、ただしユーザーインターフェイスは作らない、というものです。機能は統合してテストできますが、UIは最後まで控えておき、要石のように最後に加えて機能を完成させ、ユーザーに見えるようにします。

この技法の単純な例として、顧客に急ぎ注文の選択肢を与えることを考えます。その注文には、顧客の居住地や、その地域でどの配送会社が営業しているかに応じた価格付けが必要です。対象となる商品の性質は、倉庫でのピッキング方法にも影響します。特定の顧客だけが急ぎ注文を利用できるかもしれず、その条件も配送先、時期、注文された商品の種類に左右されるかもしれません。

全体として、これはかなりの量のビジネスロジックです。特に、倉庫、カタログ、カスタマーサービスの各システムとの厄介な連携が絡みます。この作業には数週間かかるかもしれませんが、その間にも他の機能は数日ごとにリリースする必要があります。一方、ユーザーから見れば、急ぎ注文は注文フォーム上のチェックボックスにすぎません。

このチェックボックスをkeystoneとして使うなら、チームは数回の本番リリースにまたがって、基盤となるビジネスロジックと社内システムへのインターフェイスを開発します。ユーザーは、こうした潜在コードの存在に気づきません。最後の段階でだけ、keystoneであるチェックボックスを見えるようにすればよく、それは比較的短時間でできます。こうすれば、**すべての潜在コードを統合済みの本番システムの一部にできる**ため、長命なfeature branchに伴う問題を減らせます。

潜在コードは、もし有効になっていた場合と同じ程度の確信を持てるようにテストする必要があります。これは、テストの大半をユーザーインターフェイス越しに行わなくてよいよう、システムのアーキテクチャが作られていれば可能です。Unit TestsやTest Pyramidのより低い層のテストは、この方法で簡単に実行できるはずです。Broad Stack Testsでさえ、それをSubcutaneous Testsにする仕組みがあれば実行できます。場合によってはUI自体にかなりの振る舞いが含まれますが、見えるUIをHumble Objectにできる設計なら、それもテストできます。

すべてのアプリケーションがsubcutaneousな方法で広くテストできるよう作られているわけではありません。しかし、keystoneを使う能力がなくても、そのための努力には価値があります。UI越しに走るテストは、どれほど優れた自動化ツールがあっても、常に準備が面倒です。より多くのテストをsubcutaneous testsや低いレベルのテスト、特にunit testsへ移すことで、Deployment Pipelinesを大きく高速化し、Continuous Deliveryを可能にできます。

もちろん、ほとんどのUIはチェックボックス以上のものになります。ただし、keystoneにする作業量はそこまで大きくないこともよくあります。Webアプリでは、複雑な機能が独立したWebページになることが多く、そのページは完全に作ってテストでき、keystoneは単なるリンクになります。デスクトップアプリでは、複数の画面があり、それらを見えるようにするメニュー項目がkeystoneになるかもしれません。

とはいえ、UIを単純なkeystoneとしてまとめられない場合もあります。その場合はFeature Flagsを使う時です。それでも、keystoneとして考えることは有用です。feature toggleをUIにだけ適用するようにできるからです。これにより、バックエンドコードのあちこちへトグル地点を散らさずに済み、トグル適用の複雑さを減らし、単純なトグル機構を使えるようにし、時が来たら削除しやすくなります。

UIを最後に開発することには一般的な危険があります。バックエンドコードが、後で作られたUIとうまく合わない形で設計されてしまうかもしれません。あるいは、UIに必要な注意が遅くまで払われず、反復が不足して貧弱なユーザー体験につながるかもしれません。そうした理由から、keystoneアプローチは、小さくても完全に動く機能を素早くリリースする薄い縦スライスでプロダクトを作る、全体的な進め方の中で最もよく機能します。

ここではユーザーインターフェイスを例にしましたが、もちろん同じアプローチはAPIのような他のインターフェイスにも使えます。消費者向けのインターフェイスを最後に作り、それを単純に保つことで、大きな機能であっても小さな塊として作り、統合できます。

Dark Launchingは、この変種です。新機能が作られたら呼び出されますが、結果はユーザーに表示されません。これはバックエンドシステムへの影響を測るために行われ、変更によっては有用です。すべて問題なければ、keystoneを追加できます。
