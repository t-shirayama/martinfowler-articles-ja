# Using an Agile Software Process with Offshore Development

## 要約

Martin Fowler 氏が、Thoughtworks の Bangalore 拠点などでの経験をもとに、オフショア開発でアジャイルな進め方を使えるのかを検討した記事です。アジャイルは対面での密なコミュニケーションを重視する一方、オフショア開発は距離、時差、文化、ビジネス文脈の不足によってコミュニケーションを難しくします。

記事の結論は単純な賛否ではありません。オフショア開発では余分なコストとリスクが増えるものの、継続的インテグレーション、相互訪問、短いイテレーション、定期的なビルド、複数のコミュニケーション手段を組み合わせれば、アジャイルな進め方は成立しうると述べています。ただし、低い単価だけを見て費用削減を期待するのは危険で、**距離によるコミュニケーション損失をコストとして見る**必要があります。

## 対応範囲

このページは全文翻訳ではなく、日本語訳・要約です。原文の主な論点、教訓、注意点を整理し、付録的な改訂履歴や謝辞は扱っていません。

## 読むときの観点

- オフショア開発の問題を、単なる地理的距離ではなく、信頼、時差、文化、ビジネス文脈の伝達不足として読む。
- 継続的インテグレーションや短いイテレーションを、分散チームの統制ではなくフィードバックを速くする仕組みとして見る。
- 「安い人件費」ではなく、採用できる人材、追加される調整コスト、作るべき機能を誤るリスクを含めて判断する。
- 2006年時点の記事であることを踏まえつつ、分散チームやリモート協働に今も残る論点を拾う。

## 原文の翻訳

### オフショア開発でアジャイルなソフトウェアプロセスを使う

Thoughtworks は数年間、北米やヨーロッパの開発プロジェクトを支えるため、インドの Bangalore に拠点を運営してきました。従来のオフショア開発は計画駆動の方法論に基づくことが多いのに対し、Thoughtworks はアジャイルの立場をとっています。この記事では、オフショアでアジャイル開発を行った経験と、そこから得た教訓を述べています。

アジャイルな方法論の基本には、開発に関わる人々のコミュニケーションを重視する考え方があります。特に対面での会話は、情報を伝えるうえで効率的で効果的だとされています。ところがオフショア開発では、開発者は物理的に遠く離れ、時差もあります。そのため、アジャイルの前提とオフショア開発は一見すると相性が悪く見えます。

Fowler 氏が問うているのは、オフショアという状況でもアジャイルの技法を使えるのか、そして計画駆動の方法と比べてどうなのか、という点です。経験から見ると、アジャイルな進め方はオフショアでも機能しうるものの、得られる利点については慎重に考える必要があります。

### 得られた教訓

複数拠点で開発すると、インターフェースを注意深く定義していても統合問題が起きやすくなります。そこで重要になるのが継続的インテグレーションです。全員が単一のコードベースに対して作業し、ビルドとテストを継続的に実行することで、拠点がどこにあってもメインラインに近い状態を保てます。**統合問題は早く発見されれば、見つけにくくなる前に直せる**からです。

対面のやりとりを完全に再現することはできませんが、人を動かすことは大きな助けになります。各拠点から相手側へ ambassador を送り、技術面や要求面の理解を補います。こうした人は、公式な情報だけでなく、プロジェクトで自然に流れる雑多な文脈も伝えます。小さな噂や背景情報の中に、後で重要になるものが含まれていることがあるためです。

短期の訪問も信頼作りに役立ちます。訪問の目的は、単に作業を進めることではなく、関係を作ることです。予定を詰め込みすぎると、肝心の人間同士の接点が失われます。食事や街の案内のような非公式な時間も、分散チームでは重要な活動になります。

アジャイル導入で難しいのは、プロセスよりも文化の変化です。指示命令型の組織では、実際に作業する人が判断し、質問し、問題を早く表に出すことが難しくなります。オフショアの文脈では、上位者への遠慮や受け身の合意が問題を増幅することがあります。そのため、西側のチームは、相手が黙って同意しているように見えるときほど注意を払う必要があります。

共通情報を置く場所として、Fowler 氏は wiki を有用だと述べています。ストーリーカード、設計指針、ビルド手順、進捗メモなど、参照が必要な情報を置く場所です。wiki は構造が固定されすぎていないため、チームが成長に合わせて情報の置き方を進化させられます。ただし、放置すると散らかるため、手入れする役割も必要です。

距離が広がるほど、要求の伝達には少し多くの形式が必要になります。成熟した XP チームでは、受け入れテストを要求理解の手段として使うことがあります。顧客が短い説明を書き、リモート側のアナリストやテスターがテストスクリプトに落とし込むことで、曖昧な点を質問しやすくなります。テストを書くこと自体が、要求を理解する作業になります。

定期的なビルドは、機能面のフィードバックを早めます。顧客が前日の作業結果を動かして確認できれば、誤解を早く修正できます。この仕組みを成立させるには、両拠点で環境をそろえ、問題を再現できるようにしておくことが欠かせません。完成を待ってから見せるよりも、途中の機能を早く見てもらう方が、分散開発では価値があります。

短いステータスミーティングも、リモートの調整には有効です。ただし、時差は大きな制約になります。片方の拠点だけに不便な時間を押しつけると、協働はうまくいきません。祝日や地域ごとの生活文脈も共有されていないことが多いため、予定を立てるときには相手側の事情を意識する必要があります。

イテレーション計画ミーティングは、リモート拠点に合わせて調整します。事前に顧客がストーリーの説明を書き、質問をメールなどで扱い、開発チームが細かなタスクに分解しておくと、通話時間を短くできます。リモート会議は負荷が高いため、実際の会議では未解決の論点に集中するのが効果的です。

大きな既存コードベースをリモートチームへ移す場合、最初はバグ修正から始めるとよい場合があります。バグ修正は新機能開発よりもコードを読む時間が長く、既存システムに慣れる入口になります。ただし、経験豊かな人にとっては単なる雑務のように見える危険もあるため、目的を明確にする必要があります。

チーム分割は、分析、設計、実装、テストといった活動別ではなく、機能やモジュール別に行う方がよいとされています。活動別に分けるとウォーターフォール的な分断が強まり、リモート側が文脈を持てなくなります。リモート側にも分析や設計を担える力を育て、開発者の近くにビジネス理解を持つ人を置くことが重要です。

オフショアでは、同じ場所にいる場合より多くの文書が必要になります。これはアジャイルが文書を重視しないから不要という意味ではありません。対面で済ませられない分、文書化はオフショアを選ぶことによって発生するコストの一部になります。重要なのは、文書だけに頼るのではなく、動画、メーリングリスト、チャット、通話など複数の手段を早めに整えることです。

### コストと利点

オフショア開発の利点として、低い単価が注目されがちです。しかし、単価はコストの一部にすぎず、投資対効果全体で見る必要があります。優れた開発者間の生産性差は給与差より大きいこともあり、オフショアによる単価差が追加の調整コストやリスクを上回るとは限りません。

最大の影響はコミュニケーションです。距離と時差は、要求の誤解を増やし、間違った機能を作る可能性を高めます。ドキュメント中心の組織では、もともと直接対話の利点をあまり使っていないため、オフショアによる追加ダメージが見えにくいかもしれません。一方、アジャイルは直接的な接触によってコミュニケーションを改善しようとするため、オフショアの痛みを強く感じます。

24時間開発で市場投入を早めるという主張について、Fowler 氏は懐疑的です。人を追加するなら、コミュニケーションの難しさを小さくできる場所で追加する方が効率的だからです。ただし、優秀な人材を十分に採用できない場合、オフショアチームは安さではなく人材へのアクセスという意味で価値を持ちます。

### オフショアとアジャイルのこれから

オフショア開発の長所と短所を完全に理解するには、まだ時間が必要だと Fowler 氏は述べています。単価差と同じだけのコスト削減が得られると考えるのは誤りです。一方で、インドなどの拠点にも北米やヨーロッパと同じくらい優秀な開発者を採用できるという経験も示されています。

オフショア開発の弱点は、文化とビジネスとの距離にあります。アジャイル開発は密なコミュニケーションと開かれた文化を必要とするため、オフショアではその痛みをより強く受けます。それでも Fowler 氏は、アジャイルな方法で受ける痛みは、計画駆動の方法そのものよりは小さいと見ています。
