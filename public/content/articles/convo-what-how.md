# Conversation: LLMs and the what/how loop

## 要約

この対話では、ソフトウェア開発の中心的な課題を「変化に耐えるシステムを作ること」と捉える。要件をコードへ直線的に翻訳することが問題なのではなく、ドメイン上の「何を」実現したいのかと、プログラミング言語や計算モデルで「どう」表現するのかを往復しながら、認知負荷を下げる抽象を育てることが重要だと論じている。

TDD はこの what/how ループを明示化する設計戦略であり、LLM はそのループの中で翻訳層や探索の相手として役立つ。ただし、LLM にただプロンプトしても、将来のシナリオに耐える構造は育たない。開発者自身が、書き、テストし、リファクタリングしながら、説明でき、変更できるモデルを作る必要がある。

## 読むときの観点

この記事は、Unmesh Joshi、Rebecca Parsons、Martin Fowler による martinfowler.com の対話記事「Conversation: LLMs and the what/how loop」の非公式な日本語訳・要約です。原文は https://www.martinfowler.com/articles/convo-what-how.html です。

会話の中で使われる “what” はソフトウェアが実現したい意図やドメイン上の意味、“how” はそれを計算可能な構造として表す方法を指しています。ただし記事の主張は、この二つを分離するのではなく、相互に形作るものとして扱う点にあります。

- 「what」を固定する: 先にテストを書くことで、実装の詳細に気を取られる前に、「what」の問い、つまり命名、入力、出力に答えることを自分に強制する。このとき一時的に、自分自身のコードのクライアントとして振る舞っている。

「what」と「how」のあいだを反復しているとき、意図を表現したりシナリオを記述したりすれば、LLM はすばやくコードを見せてくれる。生成されたコードはたいてい、テストを通す最初の版であることを覚えておくのが重要だ。私たちは依然として、プログラミングパラダイムと言語が提供する道具、そしてコミュニティのイディオムを使って、解決策を構造化する必要がある。

よい抽象は、意図を実装へ翻訳しやすくする。

抽象が安定し、プログラミングがほとんど宣言的な構文でそれらの抽象を使うことになると、LLM の言語表現の強みは特に有用になる。いくつかのプロンプトで動く実装を作れる。抽象がプロンプトにとって不可欠な語彙を形成するため、生成されたプログラムはより予測しやすく、レビューもしやすくなる。

「リクエストを配送タイムスタンプ順に priority queue へ並べる」や、「この顧客レコードを repository を使って保存する」といった指示は、よく理解されたイディオムに従うコードを生成しやすく、レビューと管理を容易にする。

ここには隠れた依存がある。LLM は成熟に依存している。言語が十分に成熟していない場合や、新しいパラダイムを探索している場合、LLM は十分に訓練されていない。こうした frontier では道具が乏しく、私たちは実質的に自力で進むことになる。この制約は重要な問いを投げかける。LLM が豊富な訓練データのある場所でしか働けないなら、AI の時代に新しい言語やパラダイムはどこから生まれるのだろうか。それらが二度と来なくても、私たちは気にするのだろうか。

### Martin

これは、作家の Will Wilkinson が言っていたことを思い出させる。「最良の文章は、なじみのあるものを異化し、新鮮にし、再び呼び出す。しかし LLM の出力は、設計上、なじみのあるものの平均化された焼き直しである」。私たちがありふれた、つまりなじみのある問題に苦しんでいるなら、よい解決策は必然的になじみのないものになる。

一方で、LLM が文章でもプログラミングでも、なじみのないアイデアを出せるとは思っている。本質的には、ランダムな創造によるものだ。LLM はなじみのないものを hallucinate している。しかし第三の手で言えば、LLM にはその新しい創造を発展させることはできない。必要な訓練データを持っていないからだ。人間は LLM を使って、奇妙だが可能性のあるものを hallucinate させ、それを発展させられる。

だからこそ、LLM に三つの異なる代替案を出すよう頼むことが有用な場合が多い。人間が最良のものを選んだり、三つすべてから部分を組み合わせて、よりよいものにしたりできるからだ。

### Unmesh

だから、理解を進化させ、安定した構造を作っているとき、単に LLM に「プロンプト」して仕事をさせることはできない。もちろん LLM を活用はする。しかし、安定した構造を作るために、私たち自身が反復的にコードを導かなければならない。Domain-Specific Languages の本に照らせば、これは Semantic Model を作る段階である。LLM はこの段階で強力な道具として扱うべきだが、開発者がモデルの主要な設計者であり続けなければならない。
