# Service Oriented Ambiguity

## 要約

SOA, Service Oriented Architecture は、人によって意味するものが大きく異なるため、問いに答えにくい言葉になっています。Web services の公開、アプリケーションが消えるような構成、標準化された通信構造、非同期メッセージングなど、互いに異なる考えが同じ言葉の下に集まっています。

この記事は、SOA という言葉を救おうとするよりも、そこに含まれる具体的な考えをそれぞれ独立して説明し、名前を付けるべきだと述べています。

## 読むときの観点

- SOA という言葉を聞いたとき、相手がどの意味で使っているのかを確認する。
- 同じ用語の下に、互いに両立しない設計思想が混在する危険を見る。
- 価値のある具体的な考えを、曖昧になった流行語から切り離して扱う。

## 原文の翻訳

Thoughtworks がうかつにも私を顧客の前に出すと、必ずと言ってよいほど聞かれる質問があります。「SOA, Service Oriented Architecture についてどう思いますか」というものです。これはほとんど答えようのない質問です。SOA は、人によってあまりにも多くの異なる意味を持っているからです。

- ある人にとって、SOA とは Web services を通じてソフトウェアを公開することです。この人々はさらに、さまざまな WS-* 標準を期待する人たちと、HTTP 上の XML なら何でもよい、場合によっては XML でなくてもよいと考える人たちに分かれます。
- ある人にとって、SOA はアプリケーションが消えていくようなアーキテクチャを意味します。代わりに、ビジネス機能とデータを提供する core services があり、それらが提供するものを集約して表示する UI aggregators が存在します。
- ある人にとって、SOA とは、通常は XML ベースの、何らかの標準構造を使ってシステム同士が通信できるようにすることです。最悪の形では、これは「山括弧付きの CORBA」です。より洗練された形では、組織のための標準的な backbone を考え出し、アプリケーションをそれに合わせて動かすことになります。この backbone は HTTP を含む場合もあれば、含まない場合もあります。
- ある人にとって、SOA は、ほとんどの場合に非同期メッセージングを使って、異なるシステム間でドキュメントを転送することを意味します。本質的には、高価な EAI ベンダーに lock-in されない EAI です。

SOA の良いところはデータとプロセスを分離することだと言う人もいれば、データとプロセスを組み合わせることだと言う人もいます。Web 標準を使うことだと言う人もいれば、Web 標準から独立していることだと言う人もいます。非同期であることだと言う人もいれば、同期であることだと言う人もいますし、同期かどうかは重要ではないと言う人もいます。

数年前、私は Microsoft PDC にいました。丸一日 SOA についての発表を聞き、最後には SOA の panel に出ました。私は笑いを取るつもりで、ほかに誰か SOA が一体何なのか理解している人はいるのか、と尋ねました。その後、ある人が、この曖昧さは Object Orientation でも起きたことだとコメントしました。それには一定の真実があります。OO が何を意味するかについても、いくつかの異なる見方がありましたし、今もあります。しかし、Object Ambiguity は **Service Oriented Ambiguity** ほど大きくはありません。

では、どうすればよいのでしょうか。まず、SOA 陣営の下には、どれほど多くの異なる、しかもほとんど互換性のない考えが含まれているのかを、常に覚えておく必要があります。これらは SOA とは独立に、適切に説明され、名前を付けられる必要があります。

私は、SOA は「components」や「architecture」と並ぶ、意味のない概念になってしまったと思っています。もはや救えません。だからこそ、実体のある具体的な考えには、独立した生を与える必要があります。
