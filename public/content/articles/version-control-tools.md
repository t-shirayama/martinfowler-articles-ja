# Version Control Tools

## 要約

バージョン管理ツールは、履歴を残すためだけでなく、チームが共同作業するための基盤です。Martin Fowler は、厳密な比較実験ではなく、自身とThoughtworks周辺での経験に基づいて、Subversion、Git、Mercurialを現実的に推奨しやすい選択肢として整理しています。

中心になる論点は、中央集権型か分散型か、GitかMercurialか、そして企業標準の制約があるときに複数のVCSをどう併用するかです。最終的には、道具の優劣だけでなく、継続的インテグレーションや小さな変更を頻繁に統合する作業習慣のほうが大きな意味を持つ、と結論づけています。

## 読むときの観点

- バージョン管理を、単なる履歴保存ではなくチーム作業の土台として読む。
- DVCSの柔軟性は、チームの規律や作業習慣と組み合わせて評価する。
- GitとMercurialの違いを、機能の豊富さと学びやすさのトレードオフとして見る。
- 企業標準のVCSとチーム内の効率的なVCSを併用する考え方に注目する。
- 最後の論点である「ツールよりプラクティス」を、記事全体の判断軸として読む。

## 原文の翻訳

ソフトウェア開発者とツールについて話していると、バージョン管理ツールは大きな話題になりがちです。まともな開発者ならバージョン管理を使う段階には到達しているはずで、いったんそうなると、そのツールは日々の仕事のかなり大きな部分を占めます。

バージョン管理ツールは、プロジェクトの履歴を保つためだけのものではありません。**チームが協力して作業するための基盤**でもあります。だからこそ、出来の悪いバージョン管理ツールに対する不満を耳にすることが多いのも当然です。

ThoughtworksのTechnology Radarでは、企業が評価対象にすべきバージョン管理ツールとして、Subversionと分散バージョン管理システム、つまりDVCSを挙げました。ここでは、Thoughtworks内部での多くの議論をもとに、バージョン管理ツールについての考えを補足します。

ただし、いくつか割り引いて読んでください。これは、Thoughtworksの同僚や社外の友人・知人との会話を、科学的とは言えない形で寄せ集めたものです。厳密なテストや構造化された比較をしたわけではありません。私の最近の個人的な経験は主にSubversionとMercurialですが、私の使い方は一般的な開発チームの使い方とはかなり違います。また、私の周囲の多くは、ラベルを嫌がる人も含めて、agileやXPに近い働き方を好みます。そのため、この記事に腹を立てる人も多いだろうと思いますが、その反応から良い記事が出てきて、あとでリンクできることを期待しています。

基本的には、広く支持されているバージョン管理システムは3つあります。Subversion、Git、Mercurialです。

## 推奨のしきい値を越えないもの

多くのツールは、推奨できるしきい値を越えません。理由は大きく2つあります。能力が足りない場合と、十分に見えてこない場合です。

能力不足としてよく不満を聞くものがあります。好まれるツールにも不満はありますが、しきい値の後ろにあるものは不満の割合が大きくなります。特に批判を集めるのは、IBMのClearCaseとMicrosoftのTFSです。これらがよく批判される理由の一つは、顧客企業の現場でとてもよく使われており、会社の方針で使用が義務づけられていることが多いからです。

こうした問題は、VCSの使い方に関する企業方針によってさらに悪化することがあります。作業を進めるたびに障害物になるような、奇妙なワークフローを課されたチームの話も聞きます。VCSはそのワークフローを実施させる道具なので、どうしてもその悪評を一緒に背負ってしまいます。

ここでは、能力不足のツールがどんな問題を抱えているかを詳しく述べるつもりはありません。それだけで別の記事になります。

## 分散型か中央集権型か

多くの場合、中央集権型と分散型のどちらを選ぶかは、開発チームがどれだけ熟練し、規律を持っているかに左右されます。分散型はワークフローに大きな柔軟性を与えますが、その柔軟性は、うまく使える成熟度がなければ危険にもなります。

Subversionは、単純な中央リポジトリのモデルを促し、大規模なブランチ運用をあまり勧めません。継続的インテグレーションを使う環境では、このモデルはかなりうまく合います。私の友人たちの多くはそのように働くので、Subversionは多くの環境で良い選択になります。

DVCSはワークフローをいろいろ組み立てられる柔軟性を持ちますが、私の知る多くの人は、それでも継続的インテグレーションで使う共有メインラインリポジトリという考え方を作業の中心に置いています。現代のVCSは、複数人の変更を魔法のようにマージできる道具を備えていますが、それでもマージしているのは基本的にテキストです。**意味的な整合性を得るには継続的インテグレーションが必要**です。そのため、DVCSを使うチームでも、中央のマスターリポジトリという考え方は残ることが多いのです。

Subversionには、分散型の親戚と比べて主に3つの弱点があります。

第一に、分散型ではリポジトリ全体のコピーがローカルディスクにあるため、リポジトリ操作はネットワーク越しの呼び出しを必要とせず、常に速くなります。ログを見る、古いリビジョンとの差分を見る、リポジトリ全体を使う作業をする、といった場面では違いがはっきり出ます。自宅ネットワークでも気づくほどなら、リポジトリが別大陸にある分散プロジェクトでは大きな問題になります。

第二に、リポジトリへのネットワーク接続から離れていても、分散型ならリポジトリと一緒に作業できます。飛行機の中で、作業の区切りをコミットし、履歴を眺め、リビジョンを比較できます。

第三の弱点は、純粋なツールの問題というより社会的な問題です。DVCSは、実験のための素早いブランチ作成を促します。

## GitかMercurialか

DVCSを選ぶなら、次はどちらにするかです。注目を集めているのはMercurialとGitなので、実際の選択はこの2つの間になると考えています。大まかに言えば、これは力と使いやすさの選択であり、そこにマインドシェアとGitHubの影が加わります。

Gitは、その力強さで好まれているように見えます。ファイル名変更が絡む場合でさえ、テキストのマージを自動的かつ正しく行う能力に熱狂する人は多いです。マージ能力を客観的に比較したテストを私は見ていませんが、主観的な評判はGitに傾いています。

一方で、多くの人にとってGit最大の弱点は、コマンドやメンタルモデルがしばしば謎めいていることでした。Gitに批判的な人は、設計から少しずつ振る舞いを推測していける発見しやすさが足りないと言います。支持者は、Gitが他のVCSとは違うメンタルモデルを使うため、Gitを理解するにはそれまでのVCSの知識を多少捨てる必要があるのだ、と説明します。理由はどうあれ、Gitは内部を学ぶことを楽しむ人により魅力的で、Mercurialはただバージョン管理をしたい人により訴えるように見えます。

GitHubの存在も重要です。Gitに懐疑的な人でさえ、GitHubをプロジェクト共同作業の場として非常に高く評価しています。Mercurial側の同等サービスであるBitbucketは、同じ種類の愛着を引き出してはいません。ただし、Google CodeやMicrosoftのCodePlexなど、差を縮めるかもしれないサイトもあります。

かつてGitはWindowsでうまく動かず、そのため勧めにくいものでした。しかし今では、Cygwinではなくmsysgitを使うなら状況は変わっています。Windowsでの比較において、msysgitはMercurialと比べて問題にならない程度には十分だ、というのが私たちの見方です。

一般に、GitはブランチをMercurialよりうまく扱うと見られています。特に、実験やチェックポイントのための短命なブランチではそうです。Mercurialは、高速な別リポジトリディレクトリのクローンやパッチキューといった別の仕組みを促しますが、Gitのブランチはより単純でよいモデルです。

Mercurialには、大きなバイナリファイルに関する問題があるようです。そうしたものは通常Subversionで管理するほうがよい、というのが私の一般的な提案です。ただし、別管理するほど多くない場合には、少数の大きなバイナリファイルがMercurialを苦しめるかもしれません。

## 複数のVCS

同時に複数のVCSを使うことには、しばしば価値があります。これはたいてい、チームが使いたいものより能力の低いVCSを、より広い組織上の理由で使わなければならない場合です。

よく遭遇するのは、会社標準として不十分なVCS、たいていClearCaseが定められている一方で、チームとしては効率よく働きたい、という状況です。この場合、日々のチーム作業には別のVCSを使い、必要なタイミングで会社標準のVCSにコミットするやり方でうまくいったことがあります。チーム内のVCSでは各人が1日に何度もコミットしますが、会社標準のVCSでは1、2週間に一度のコミットになります。企業側の管理者にとっても、そのほうが都合がよいことがよくあります。

この二重利用のシナリオは、git-svnでも一般的です。人々はローカルではGitを使い、共有Subversionシステムへコミットします。git-svnは、MercurialよりGitを選ぶ理由の一つでもあります。ローカルDVCSは、中央集権型VCSから離れた遠隔拠点で作業する場合に特に価値があります。ネットワーク断や帯域の問題が、中央リポジトリから遠い拠点を簡単に止めてしまうからです。

多くのチームは、この二重VCSの働き方から利益を得られます。とりわけ、企業VCSが多くの儀式的な手続きを強制している場合にはそうです。ローカル開発チームと企業側の管理者では、VCSに求める動機が違うことが多いため、二重VCSは双方を幸せにしやすいのです。

## 最後に

ここまでツールについて長々と話してきましたが、実際にはプラクティスやワークフローのほうが大きな違いを生むことがよくあります。ツールは良いプラクティスを使いやすくしてくれますが、最終的には、その環境で効果的な働き方をするかどうかは人間次第です。

私は、多くの小さな変更を、継続的インテグレーションによって素早く統合できるやり方を好みます。**CIなしで良いツールを使うより、CIありで悪いツールを使うほうがよい**と思っています。
