# Courtesy Implementation

## 要約

Courtesy Implementation は、あるクラス自身にとっては少し不自然に見える操作でも、より豊かなインターフェースにそろえることで全体のモデルを単純にできるなら、その簡単な実装を提供するという考え方です。

Composite パターンでは、葉のクラスにも共通操作を持たせることで、型判定の条件分岐を避けられます。局所的な違和感よりも、モデル全体の扱いやすさを優先する判断です。

## 読むときの観点

- クラス単体で自然かどうかだけでなく、構造全体を単純にするかを見る。
- 型判定の条件分岐を避けるために、葉にも同じメソッドを持たせる設計を考える。
- 現実世界の厳密な写像ではなく、目的に合うモデルを作っている点を意識する。
- Courtesy Implementation が、モデルを読みやすくするための妥協なのかを見極める。

## 原文の翻訳

クラスを書くとき、たいていはそのクラスの機能がそのクラスにとって意味を持つように努めます。しかし、クラスが本来なら自然には持たない、より豊かなインターフェースに合わせるために機能を追加するのが妥当な場合もあります。

最もよくある明白な例は、Composite パターンを使うときに現れます。コンテナの簡単な例を考えてみましょう。箱があり、その箱には別の箱と象を入れられます。仮想の象にはそういう利点があります。ある箱の中に象が何頭いるかを知りたいとします。そのとき、箱の中の箱の中の箱に入っている象も数えなければなりません。解決策はもちろん、単純な再帰です。

```ruby
# Ruby
class Node
end

class Box < Node
  def initialize
    @children = []
  end
  def << aNode
    @children << aNode
  end
  def num_elephants
    result = 0
    @children.each do |c|
      if c.kind_of? Elephant
        result += 1
      else
        result += c.num_elephants
      end
    end
    return result
  end
end

class Elephant < Node
end
```

ここで `num_elephants` の中にある `kind_of?` の判定は臭います。オブジェクトの型を調べる条件分岐には注意すべきだからです。一方で、別のやり方はあるのでしょうか。そもそもこの判定をしているのは、象は箱や象を入れられないので、象の中に何頭の象がいるかを尋ねる意味がないからです。象は何も包含できないのだから、象が何頭の象を含んでいるかを尋ねるのは、私たちの世界のモデルには合いません。

現実世界をモデル化していないと言ってもよいかもしれませんが、この例は少し気まぐれすぎて、その議論には向かなそうです。

しかし Composite パターンを使う人は、条件分岐を避けるためにメソッドを用意することがよくあります。つまり、次のようにします。

```ruby
class Node
  # 強い型付けの言語なら、ここで抽象的な
  # num_elephants を定義する
end

class Box < Node
  def initialize
    @children = []
  end
  def << aNode
    @children << aNode
  end
  def num_elephants
    result = 0
    @children.each do |c|
      result += c.num_elephants
    end
    return result
  end
end

class Elephant < Node
  def num_elephants
    return 1
  end
end
```

この種のことに強い違和感を持つ人は多いのですが、Composite 構造を走査するコードのロジックは大きく単純になります。私はこれを、階層内のノードとしての役割に対する礼儀として、葉クラスである象に単純な実装を提供してもらうことだと考えています。

私がよく引く類比は、数学における 0 乗の定義です。任意の数の 0 乗は 1 である、と定義されています。しかし直感的には、ある数を 0 回自分自身に掛けると 1 になる、というのは納得しにくいところがあります。なぜ 0 ではないのでしょう。しかしその定義にすると数学全体がうまく収まるので、私たちは疑念を脇に置いてその定義に従います。

モデルを作るとき、私たちは世界をどう認識したいかに合うようにモデルを設計しています。Courtesy Implementation は、**モデルを単純にするなら価値がある**のです。
