# Anemic Domain Model

## 要約

貧血ドメインモデルは、ドメインオブジェクトがデータ構造に近くなり、重要な業務ロジックがサービス層へ押し出されている設計を指します。
見た目にはドメインモデルのようでも、データと振る舞いが結びついていなければ、実質的には手続き型の設計になります。

この記事の主張は、サービス層を否定することではありません。
サービス層は薄く調整役にとどまり、検証、計算、業務ルールの中心は、振る舞いを持つドメインモデルに置くべきだという点が重要です。

## 読むときの観点

- ドメインオブジェクトがデータだけの入れ物になっていないかを見る。
- サービス層が業務ルールの主な置き場になっていないか確認する。
- レイヤリングと、ドメインロジックをドメイン層に置くことを混同しない。
- O/Rマッピングのコストに見合うだけのオブジェクト指向設計上の利点が得られているか考える。

## 原文の翻訳

これは、かなり長いあいだ存在しているアンチパターンのひとつですが、最近になって特に勢いを増しているように見えます。この件について Eric Evans と話していたところ、私たちは二人とも、このパターンが以前より広まっているようだと感じていました。きちんとした Domain Model を強く支持している私たちにとって、これはよいことではありません。

Anemic Domain Model の基本的な症状は、最初に見たときには本物のように見えることです。そこにはオブジェクトがあり、その多くはドメイン空間にある名詞から名前を付けられています。そして、それらのオブジェクトは、真のドメインモデルが持つような豊かな関係や構造で結びついています。落とし穴は振る舞いを見たときに現れます。それらのオブジェクトにはほとんど振る舞いがなく、**getter と setter の袋**と大差ないものになっているのです。

実際、こうしたモデルには、ドメインオブジェクトにはドメインロジックを一切置いてはいけない、という設計ルールが伴うこともよくあります。その代わりに、すべてのドメインロジックを受け持つ一群のサービスオブジェクトがあります。これらのサービスがすべての計算を行い、その結果でモデルオブジェクトを更新します。サービスはドメインモデルの上に位置し、**ドメインモデルをデータとして使う**のです。

このアンチパターンの根本的な恐ろしさは、オブジェクト指向設計の基本的な考え方、つまりデータとプロセスを一緒にまとめるという考え方にまったく反していることです。貧血ドメインモデルは、実際には手続き型スタイルの設計にすぎません。これは、私のようなオブジェクト指向寄りの人間、そして Eric が、Smalltalk の初期からずっと闘ってきた種類のものです。さらに悪いことに、多くの人は貧血状態のオブジェクトを本物のオブジェクトだと思い込んでしまい、**オブジェクト指向設計の要点を完全に見失ってしまいます**。

もちろん、オブジェクト指向の純粋主義を語るのは結構なことです。しかし私は、この貧血状態に反対するには、もっと根本的な論拠が必要だと感じています。要するに、貧血ドメインモデルの問題は、ドメインモデルのコストをすべて負担するのに、その利点をまったく得られないことです。主なコストは、データベースへマッピングする際の扱いにくさであり、通常は O/R マッピングの層全体を必要とします。複雑なロジックを整理するために強力なオブジェクト指向の技法を使う場合に限って、**このコストは見合うものになります**。

しかし、すべての振る舞いをサービスへ引き出してしまうと、本質的には Transaction Script になってしまい、ドメインモデルがもたらせる利点を失います。『Patterns of Enterprise Application Architecture』で述べたように、Domain Model は常に最善の道具であるとは限りません。

また、ドメインオブジェクトに振る舞いを入れることは、永続化やプレゼンテーションといった責務からドメインロジックを分離するためにレイヤリングを使う、堅実なアプローチと矛盾するものではない、という点も強調しておく価値があります。ドメインオブジェクトに入るべきロジックは、ドメインロジックです。つまり、検証、計算、ビジネスルールなど、呼び方は何であれ、**業務上の意味を持つロジック**です。

ドメインオブジェクトにデータソースやプレゼンテーションのロジックを置くべきだと主張できる場合もありますが、それは私がここで言っている貧血状態の話とは別の問題です。

この話で混乱が生じる原因のひとつは、多くのオブジェクト指向の専門家が、ドメインモデルの上に手続き的なサービスの層を置き、Service Layer を形成することを推奨している点にあります。しかし、それはドメインモデルから振る舞いを空にしてよいという主張ではありません。実際、Service Layer を支持する人たちは、**振る舞いの豊かなドメインモデルと組み合わせて** Service Layer を使います。

Eric Evans の優れた本『Domain-Driven Design』では、これらの層について次のように述べられています。

> Application Layer（彼の呼び方では Service Layer）は、ソフトウェアが行うべき仕事を定義し、表現力のあるドメインオブジェクトに問題を解かせる。この層が責任を持つタスクは、ビジネスにとって意味があるか、他システムのアプリケーション層とやり取りするために必要なものである。この層は薄く保たれる。ビジネスルールや知識は含まず、タスクを調整し、ひとつ下の層にあるドメインオブジェクトの協調へ作業を委譲するだけである。
>
> ビジネス状況を反映する状態は持たないが、ユーザーやプログラムにとってのタスクの進行状況を反映する状態を持つことはできる。
>
> Domain Layer（または Model Layer）は、ビジネスの概念、ビジネス状況に関する情報、ビジネスルールを表現する責任を持つ。ビジネス状況を反映する状態はここで制御され、使われる。ただし、それを保存する技術的な詳細はインフラストラクチャに委譲される。この層はビジネスソフトウェアの中心である。

ここでの要点は、Service Layer は薄いということです。主要なロジックはすべて Domain Layer にあります。彼は Service パターンの中でも、この点を繰り返しています。

> より一般的な誤りは、振る舞いを適切なオブジェクトに収めようとする努力をあまりに早く諦め、少しずつ手続き型プログラミングへ滑り落ちていくことである。

なぜこのアンチパターンがこれほど一般的なのか、私にはわかりません。適切なドメインモデルを実際に使った経験がない人が多いこと、特にデータ寄りの背景を持つ人たちにその傾向があることが理由ではないかと思います。技術によっては、この傾向を助長するものもあります。たとえば J2EE の Entity Beans がそうで、それが私が POJO のドメインモデルを好む理由のひとつです。

一般に、サービスの中に見つかる振る舞いが多ければ多いほど、ドメインモデルの利点を自分で奪っている可能性が高くなります。すべてのロジックがサービスにあるなら、**自分自身からその価値を完全に奪っている**のです。
