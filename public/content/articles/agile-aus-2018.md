# The State of Agile Software in 2018

## 要約

このページは、Martin Fowler 氏が Agile Australia 2018 で行った基調講演の書き起こしに基づく日本語訳・要約です。
アジャイルソフトウェア開発は広く普及した一方で、価値や原則を伴わない faux-agile が増えたことを問題として扱っています。

Fowler 氏が強調する課題は、チームへ画一的なプロセスを押しつける Agile Industrial Complex に対抗すること、technical excellence の重要性を取り戻すこと、そして一時的な project ではなく長く続く product を中心にチームを組織することです。
それでも、アジャイルコミュニティには学び、変わり、初期の宣言作成者が想像しなかった問題に取り組む力がある、と講演は締めくくられます。

## 対応範囲

このページは全文翻訳ではなく、原文の主要な論点を日本語で読むための要約と部分的な翻訳です。
原文は martinfowler.com の [The State of Agile Software in 2018](https://www.martinfowler.com/articles/agile-aus-2018.html) です。
この日本語文書は非公式のものであり、Martin Fowler 氏、martinfowler.com、Thoughtworks の公式翻訳ではありません。

## 読むときの観点

- agile という言葉の普及と、価値や原則の空洞化を分けて読む。
- チームが自分たちの働き方を選び、継続的に改善することを中心に置く。
- technical excellence を、単なる開発者向けの技術論ではなく、変化へ対応する力として見る。
- project から product への視点転換が、ビジネス側と技術側の関係をどう変えるかを見る。

## 原文の翻訳

表面的には、アジャイルソフトウェア開発の世界は明るく見えます。
アジャイルは主流になり、多くの組織で語られるようになりました。
しかし Fowler 氏は、その現実には不安があると述べます。
多くの現場で行われているものは、アジャイルの価値や原則を無視した faux-agile になっているからです。

かつての課題は、アジャイルを真面目に受け止めてもらうことでした。
ところが2018年時点では、状況が変わっています。
多くの組織が「すでにアジャイルをやっている」と言う一方で、実際に見ると、期待される価値観や実践とは大きく異なることがあります。
このため現在の課題は、アジャイルという名前だけを使い、基本原則に反するやり方に向き合うことだとされます。

最初の問題は、Fowler 氏が Agile Industrial Complex と呼ぶものです。
これは、チームの状況を見ずに、外から決められた方法を押しつける動きです。
アジャイルの中心には、process や tool よりも individuals and interactions を重視する考えがあります。
そのため、チームは自分たちのプロセスを選ぶだけでなく、作業を進めながらそれを変え続ける必要があります。
ここで重要なのは、**働き方はチーム自身が状況に合わせて進化させる**という点です。

ソフトウェア開発には、あらゆる現場にそのまま当てはまる単一の方法はありません。
エンタープライズアプリケーション、電話交換機、組み込みソフトウェアでは性質が違います。
同じ領域の中でも、legacy environment、チーム内の人間関係、組織上の制約は異なります。
それにもかかわらず、ある方法をすべての人に適用しようとすることは、アジャイルの基本精神に反します。

二つ目の問題は、technical excellence の重要性が十分に認識されていないことです。
アジャイルを語る場で、ソフトウェアを書く技術そのものが軽く扱われることがあります。
しかし、変化へ素早く対応するには、変更しやすいソフトウェアが必要です。
Refactoring はその中心にあります。
観察可能な振る舞いを変えずに内部構造を改善する小さな変更を積み重ねることで、新しい機能を入れやすくします。

この考え方は Yagni とも結びついています。
後から安全に変更できるなら、最初からあらゆる機能を入れる必要はありません。
不要な機能を先に作ると、ソフトウェアは膨らみ、理解しにくくなります。
ただし、この戦略は refactoring、testing、continuous integration、continuous delivery などの実践に支えられていなければ成り立ちません。
technical excellence とは、派手な技術選択ではなく、**適応し続けられるソフトウェアを作る能力**です。

三つ目の課題は、software project という見方を手放し、product-oriented な見方へ移ることです。
一時的に立ち上げて終わる project ではなく、長く続く product や business capability を中心にチームを組織します。
そのチームには、技術者だけでなく、ビジネス側の人たちも含まれます。
顧客経験のどの部分を良くするのかを軸に、チームの境界を考える必要があります。

Fowler 氏は、アジャイルを名乗る組織で、開発者が実際の利用者と話すことを許されていなかった例を挙げます。
これは、アジャイルの発想とは逆です。
Snowbird で名前を考えていたころ、Kent Beck 氏が「conversational」という候補を出したように、ソフトウェア開発には継続的な会話が必要です。
開発者、ビジネス側、利用者が、顧客体験を良くするために直接つながっていなければなりません。

講演の最後で Fowler 氏は、問題は多いものの悲観しすぎてはいない理由を述べます。
Agile Manifesto の作成者たちは、運動の将来に対して自分たちだけが特別な役割を持つべきだとは考えませんでした。
そのため、新しい人々が加わり、初期メンバーが想像していなかった課題にも取り組めるコミュニティになりました。
学び続け、変わり続けられることこそが、アジャイルコミュニティの大きな強みだとされています。
