# Lazy Initialization

## 要約

Lazy Initializationは、変数、オブジェクト指向では多くの場合クラスのfieldを、最初にアクセスされたときに初期化する技法です。計算に時間がかかる値を、実際に必要になるまで作らずに済ませたいときに役立ちます。

ただし、これは応答性を改善するための最適化です。実際の性能問題がないなら使うべきではありません。また、デバッグ中にfieldを見るだけで初期化が走り、通常利用では起きない状態変化が起きることがあります。

## 読むときの観点

- 値の計算を「必要になるまで遅らせる」最適化として読む。
- 初期化を早く終わらせたい場合と、後で遅延が出てもよい場合の違いを見る。
- デバッグ時に観察が状態変化を起こす危険に注意する。

## 原文の翻訳

Lazy Initializationは、変数、OOの文脈では通常クラスのfieldを、**最初にアクセスされたときに初期化する**技法です。典型的な形は、だいたい次のようになります。

```csharp
public FooClass Foo {
  get {
    if (_foo = null) _foo = calculateFoo();
    return _foo;
  }
}
```

Lazy Initializationは、fieldの値を計算するのに時間がかかり、実際にその値が必要になるまでは計算したくない場合に役立ちます。そのため、そのfieldが多くの文脈では必要にならない場合や、オブジェクトの初期化を素早く終わらせ、必要なら遅延を後回しにしたい場合によく有用です。

これは、lazy initialized valueをclientが必要としない状況で応答性を助けるための最適化技法だということを覚えておいてください。どんな最適化でもそうですが、解決すべき本当の性能問題がない限り、この技法を使うべきではありません。

特にLazy Initializationは、デバッグ上の悩みを引き起こすことがあります。デバッグ中にfieldを見ようとすると、通常の利用では起きない状態変化をシステムに起こしてしまうからです。ObservableStateは変わらなくても、実際の状態は変わります。その結果、print文を追加するとbugが消えたように見える状況につながることがあります。金曜の午後に胃を痛めるには、実にうってつけの展開です。
