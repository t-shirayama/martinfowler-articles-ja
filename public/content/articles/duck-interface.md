# Duck Interface

## 要約

Duck Interface は、Ruby の `Array` と Java の `List` の違いを入り口に、静的言語と動的言語で同じような機能が異なる役割を担うことを考える記事です。

動的言語では、ひとつのクラスが複数の役割を担いやすく、メソッド名もその文脈に合わせて増えがちです。ある言語のクラス設計を、別の言語の価値観だけで裁かないほうがよい、という慎重な結論につながります。

## 読むときの観点

- Ruby の `Array` と Java の `List` を単純な優劣ではなく、言語文化の違いとして読む。
- Queue や Stack の専用クラスを作るか、既存コレクションに役割を持たせるかを見る。
- 静的型付けではインターフェース、動的型付けでは Duck Typing が設計判断に影響する。
- API の大きさやメソッド名を、その言語の文脈で評価する。

## 原文の翻訳

私が素朴だったのかもしれませんが、Humane Interface についての投稿があれほどの議論を引き起こすとはまったく思っていませんでした。残念ながら、その多くは私が論じようとしていた根本的な点ではなく、Ruby の `Array` と Java の `List` の相対的な長所をめぐる議論になってしまいました。それでも、そこからいくつかよい会話の支流が生まれたと思います。

ただし、私の意図は、Ruby の `Array` のほうが優れているとか、Ruby のほうが優れていると言うことではなかった、と指摘しておくべきでしょう。どちらかが他方より優れているとは、より多くの文脈がなければ言えません。実のところ、私は Ruby の `Array` も Java の `List` も好きです。ただし両者はかなり違った設計をされています。どんなソフトウェアでもそうであるように、これらにも欠点はあります。今この瞬間の私が望むとおりにクラスを書く人などいないからです。しかし自分のコードをこれらと競わせたいとは思いません。要点は、どちらも有用であり、私はどちらもよく使っているということです。だから例として思い浮かんだのです。

こうした会話の流れのひとつから、`Array` と `List` の違いには、humane/minimal という哲学以外にも理由があることが見えてきました。その理由のひとつは、似た機能がふたつの言語で異なる役割を果たしていることに関係しています。

Ruby の `Array` には、一覧を見た人を少し戸惑わせるメソッドがいくつもあります。`push` と `pop` について、Elliotte は「誰かがスタックをリストの中へ押し込んだ」と言いました。さらに、リストのような `shift` と `unshift` もあります。これは正しく見えません。Elliotte はまた、「queue や stack を使っているなら、`List` クラスではなく `Queue` や `Stack` クラスを使うべきではないのか」と言っています。

これを読んで、記憶の奥のほうで何かがひっかかりました。私は『Smalltalk Best Practice Patterns』を取り出しました。オブジェクト指向を真剣に理解したい人なら誰もが読むべき、すばらしい本です。構文は少し変わっていますが。

> Smalltalk に来た多くの人が最初に書くオブジェクトのひとつが Stack です。Stack は、歌や物語や理論的プログラミング言語についての何百もの論文で語られてきた基本データ構造です。ところが、基本イメージのどこにも Stack クラスはありません。私は何度も Stack が書かれるのを見てきましたが、長く残ったためしはありません。
>
> -- Kent Beck

少なくとも当時の Smalltalker のやり方は、Ruby の `Array` に相当する Smalltalk の `OrderedCollection` を使うことでした。`push` や `pop` すらなく、Kent は代わりに `addLast:` と `removeLast:` を使う例を示していました。

Kent は stack や queue がない理由を説明していません。「なぜ Smalltalk に Stack がないのか。まあ、『そういうものだから』だ。`OrderedCollection` を使って stack を模倣するのが文化の一部なのだ」と述べています。

これについて自分がどう感じているのか、私にはよくわかりません。Kent の文章にも、はっきりしない感じが漂っています。queue のようなものを使うなら、`OrderedCollection new` と書くより `Stack new` と書くほうが理にかなっているように思えます。まして `addLast` や `removeLast` より、`push` や `pop` を使うほうが自然です。

この状況の一部は、静的言語と動的言語の違いに関係しているのではないかと思います。静的言語は、厳密な型インターフェースを通じてオブジェクトに話しかけることを好みます。動的言語では、クラスが複数の役割に合うことができます。つまり Duck Typing です。Java にも queue として二重の役割を果たすリスト、`LinkedList` があります。しかし普通はそれを別々のインターフェースを通じて使うので、共通の実装であることを意識しません。

Smalltalk 的な感覚は、`OrderedCollection` で欲しいものが手に入るのだから、なぜ別のクラスを作るのか、というものです。Ruby はこの反応を反映しつつ、その文脈で使う人のために意味のあるメソッド名を追加しているように見えます。公平を期すなら、Rubyist たちが `Array` を stack として使えることに実際に満足しているのか、それとも後悔を伴う遺産なのか、私にはまったくわかりません。

もうひとつの要因は、こうした構造の実装について言語が何を促すかです。Charles Miller が言ったように、「Java の設計は小さなインターフェースと、ヘルパークラス上の static メソッドとして提供されるユーティリティ関数を促す。Ruby の設計は、mixin されたユーティリティメソッドを持つ、より大きなクラスを促す」のです。

ここから導ける結論のひとつは、**ある言語のクラスの機能を、別の言語の価値観で判断することには慎重であるべき**だということかもしれません。`Array` は `List` に相当するのでしょうか。それとも `List` に collections パッケージ内のさまざまなインターフェースや実装を加えたものに相当するのでしょうか。あるいは、もっと複雑な何かなのでしょうか。リストに対して 78 種類のことができるという考えにたじろぐ人もいるかもしれませんが、Lisp を使う人なら、さらに多くを加えたいと思うのではないかと私は疑っています。

Ruby の `Array` にも難点はありますが、Java の collections よりも作業しやすいと認めざるを得ません。とはいえ、それが humane interface の指針によるものなのか、Ruby の構文上の支援によるものなのかは、私にはよくわかりません。

全体として、ここで誰に同意するのか、あるいはそれが本当にそれほど重要なのか、私にはよくわかりません。こうした議論の多くと同じように、いちばん面白いのは両方の見方を理解しようとすることだと思います。
