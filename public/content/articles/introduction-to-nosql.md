# Introduction to NoSQL

## 要約

この講演は、NoSQL データベースの入門として、NoSQL という言葉の由来、代表的なデータモデル、一貫性の問題、そして現代のデータ保存における位置づけを扱います。

NoSQL は、リレーショナルデータベースを単純に置き換えるものではありません。
むしろ、用途ごとに適したデータストアを選ぶ Polyglot Persistence の時代に、選択肢のひとつとして理解する必要があります。

## 読むときの観点

- NoSQL という名前が、明確な製品分類ではなく偶然の呼称から広がった点を見る。
- Aggregate-Oriented Database というまとまりで考える意義を押さえる。
- 一貫性、可用性、応答時間のトレードオフを技術だけでなくビジネス判断として読む。
- NoSQL とリレーショナルデータベースを、勝ち負けではなく使い分けとして捉える。

## 原文の翻訳

### 講演の位置づけ

これは NoSQL データベースへの入門講演です。
データベースの種類、一貫性に関する問題、そしてデータ保存において NoSQL がどのような役割を果たすのかを扱います。

NoSQL という言葉の起源は、Twitter 上のミートアップ用ハッシュタグでした。
しかしその言葉は、20年にわたるリレーショナルデータベースの支配に対する、もっとも本格的な挑戦を表すものへと変わっていきました。
名前が偶然に生まれたため、この領域には厳密な定義がほとんどなく、非常に幅広いものが含まれます。
それでも、その多くを **Aggregate-Oriented Database** という呼び方でまとめて考えることには意味があります。

NoSQL データベースは、一貫性について新しい問題を持ち込みます。
ただし、ACID トランザクションがある場合でさえ、アプリケーション側で同時更新を扱わなければならないことは珍しくありません。
多くの NoSQL データベースが分散データを扱えることは、一貫性の問題をさらに複雑にします。

この複雑さは、CAP 定理が示すように、一貫性と可用性、そして応答時間のトレードオフにつながります。
そのトレードオフは、最終的には **技術判断だけでなくビジネス判断**です。

NoSQL データベースは、現代のデータ需要に対する真剣な選択肢です。
しかし、それが唯一の選択肢であるわけではありません。
私たちは、個別のデータアクセス要件に基づいてデータ保存技術を選ぶ Polyglot Persistence の時代にいます。
