# Monolith First

## 要約

Monolith First は、新しいシステムを最初から microservices として作るのではなく、まず monolith として作り、境界や必要性が見えてから分割する戦略です。

Microservices には運用と設計の大きな追加コストがあり、サービス境界を早い段階で正しく引くのは難しいため、初期にはフィードバック速度と学習を優先するほうがよい、というのがこの記事の中心です。
ただし、反対意見もあり、特にチームが microservices の経験を十分に持つ場合や、置き換え対象で境界が安定している場合には判断が変わり得ます。

## 読むときの観点

- 成功例の多くが「大きくなった monolith を分割した」ものだという観察から読む。
- microservice premium が初期の学習速度に与える影響を見る。
- 境界を見つけることと、Bounded Context を定めることの関係を押さえる。
- Monolith First は「永遠に monolith にする」主張ではなく、学習してから分ける戦略として読む。

## 原文の翻訳

Microservices architecture を使っているチームの話を聞いていると、共通するパターンに気づきました。

1. 成功している microservices の話のほとんどは、大きくなりすぎた monolith から始まり、それが分割されたものです。
2. 最初から microservices システムとして作られたシステムについて聞いた事例のほとんどは、深刻な問題に陥っています。

このパターンから、私の同僚の多くは、たとえそのアプリケーションが十分に大きくなり、microservices にする価値があると確信していても、新しいプロジェクトを microservices で始めるべきではない、と主張するようになりました。

Microservices は有用なアーキテクチャです。しかし、その支持者でさえ、使うには大きな Microservice Premium がかかると言います。つまり、microservices が役に立つのは、より複雑なシステムに限られるということです。このプレミアムは、本質的にはサービス群を管理するコストであり、チームの速度を落とします。そのため、単純なアプリケーションでは monolith のほうが有利になります。

ここから、monolith-first strategy への強い論拠が生まれます。将来 microservices architecture の恩恵を受ける可能性が高いと思っていても、新しいアプリケーションはまず monolith として作るべきだ、という考え方です。

その第一の理由は、古典的な Yagni です。新しいアプリケーションを始めるとき、それが利用者にとって有用だとどれほど確信できるでしょうか。設計の悪い成功したソフトウェアシステムをスケールさせるのは難しいかもしれません。しかし、それでも逆の状態よりはましです。今では私たちも認識しているように、ソフトウェアのアイデアが有用かどうかを見極める最良の方法は、単純な版を作り、それがどれだけうまく機能するかを見ることです。

この最初の段階では、速度、したがってフィードバックのサイクルタイムを優先する必要があります。したがって、microservices のプレミアムは避けるべき足かせです。

Microservices で始めることの二つ目の問題は、サービス間に良く安定した境界を見つけられた場合にだけ、それがうまく機能することです。これは本質的に、適切な Bounded Context の集合を描く作業です。サービス間で機能をリファクタリングするのは、monolith の中で行うよりずっと難しくなります。しかし、よく知ったドメインで働く経験豊かなアーキテクトでさえ、最初に境界を正しく引くのは非常に難しいものです。

まず monolith を作ることで、microservices の設計がそこへ糖蜜の層を塗ってしまう前に、**正しい境界がどこにあるかを見極める**ことができます。また、より細かい粒度のサービスに必要な Microservice Prerequisites を育てる時間も得られます。

私は、monolith-first strategy を実行するいくつかの方法を聞いてきました。論理的な方法は、ソフトウェア内のモジュール性に注意を払い、API 境界とデータの保存方法の両方を意識して、monolith を慎重に設計することです。これをうまく行えば、microservices へ移行するのは比較的簡単です。ただし、このアプローチがうまくいったという十分な数の話を聞ければ、私はもっと安心できるでしょう。

任意のシステムを取り上げて microservices に分割できると仮定してはいけません。ほとんどのシステムは、モジュール間に依存関係を持ちすぎていて、筋の通った分割ができません。Monolith の分解を試みたものの、すぐに混乱へ陥った事例は数多く聞いています。段階的に microservices へ進む道が成功した事例もいくつか聞いていますが、それらは比較的良いモジュール設計から出発していました。

より一般的なアプローチは、monolith から始めて、周辺部から少しずつ microservices を剥がしていくことです。このようなやり方では、microservices architecture の中心にかなり大きな monolith が残ることがあります。ただし、新しい開発の大半は microservices 側で行われ、monolith は比較的静かな状態になります。

もう一つの一般的なアプローチは、monolith を完全に置き換えることです。これを誇るべきアプローチと見る人はほとんどいません。それでも、monolith を Sacrificial Architecture として作ることには利点があります。特に monolith によってすばやく市場に出られるなら、捨てるつもりの monolith を作ることを恐れる必要はありません。

私が出会った別の道は、最終的に目指すよりも大きな粒度のサービスを、まず数個だけ作ることです。そうした粗い粒度のサービスを使って、複数サービスで働くことに慣れます。同時に、その粗い粒度によって、サービス間リファクタリングの量が減るという利点も得られます。そして境界が安定してきたら、より細かい粒度のサービスへ分解します。

厳密にはこれを「duolith」と呼ぶべきなのかもしれません。しかしこのアプローチは、monolith-first strategy の本質に従っていると思います。つまり、知識を得るために粗い粒度から始め、後で分割するのです。

私の周囲の多くは monolith-first approach に傾いていますが、決して全員一致ではありません。反論は、microservices で始めることで、microservice 環境で開発するリズムに慣れられる、というものです。簡単に microservices へ分解できるほど十分にモジュール化された monolith を作るには、多くの、おそらく多すぎる規律が必要です。

Microservices で始めれば、全員が最初から分離された小さなチームで開発することに慣れます。また、チームがサービス境界で分かれていると、必要になったときに開発体制をスケールさせるのがずっと容易になります。これは特に、安定した境界を早い段階で見つけられる可能性が高い、システム置き換えの場合に実行しやすいでしょう。

証拠は乏しいものの、私は、チームに microservices システムを構築した相応の経験がない限り、microservices から始めるべきではないと感じています。

Monolith-first strategy を使うべきかどうかをどう判断するかについて、私はまだ十分な逸話を持っているとは感じていません。Microservices はまだ初期の段階にあり、学べる逸話も比較的少ないのです。ですから、この話題についての助言は、どれほど自信たっぷりに語られていても、**暫定的なもの**として受け止めるべきです。
