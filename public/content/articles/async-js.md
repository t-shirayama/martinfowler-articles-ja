# Testing Asynchronous JavaScript

## 要約

JavaScriptの非同期コードをテストするには特別なやり方が必要だと思われがちですが、この記事は、多くのコードは「本質的に非同期」なのではなく「非同期に使えるAPIを呼んでいる」だけだと説明します。その違いを理解すれば、テストではコールバックやPromiseを同期的に扱い、読みやすい単体テストを書けます。

後半では、コールバック中心の設計がテストを読みにくくすること、Promiseを使うと非同期処理の制御フローを値として扱いやすくなり、テストランナーやアサーションライブラリと組み合わせて簡潔なテストを書けることを示します。

## 読むときの観点

- 非同期対応のコードと、本質的に非同期なコードを区別する。
- テストでは依存先を差し替えて、非同期APIを同期的に振る舞わせられる場合がある。
- コールバックが呼ばれないバグを見逃さないため、テストの制御フローの終端を明示する。
- Promiseは、処理の呼び出しと結果の扱いを分離し、テストを読みやすくする。

## 原文の翻訳

JavaScript界隈では、非同期コードをテストするには通常の同期コードとは違う考え方が必要だ、という誤解がよくあります。しかし、多くの場合それは正しくありません。大切なのは、非同期的な振る舞いを支援するコードと、本質的に非同期なコードを区別することです。

たとえば、Ajax呼び出しを行い、完了時にコールバックを呼ぶコードを見ると、私たちはそれを非同期コードだと思いがちです。確かに、実際のAjax実装はネットワークI/Oを伴い、結果は後で返ってきます。しかし、コールバックを受け取るというAPIの形だけを見れば、それ自体が必ず非同期であるとは限りません。実装がコールバックをその場で呼べば、呼び出し全体は同期的に完了します。

つまり、**コールバックを使っていることと、本質的に非同期であることは同じではありません。** コールバックやPromiseを使うAPIは非同期に実装できますが、テストでは同じ形のAPIを同期的に振る舞わせることもできます。この区別が、非同期対応コードの単体テストを単純にします。

ユーザー情報をAjaxで取得し、JSONをUserオブジェクトに変換し、結果をコールバックに渡す関数を考えます。本番ではAjaxの完了後にコールバックが実行されます。しかし単体テストでは、Ajax関数を差し替えて、用意したJSONをすぐに完了コールバックへ渡すようにできます。そうすれば、ネットワークもイベントループの次の周回も待たずに、対象コードの変換ロジックを検証できます。

このテストの狙いは、非同期ライブラリそのものを検証することではありません。対象コードが、依存先から返ってきた値を正しく処理するかどうかを確認することです。Ajaxをスタブに置き換えて同期的に呼べるようにすることで、テストは単純な一連の流れとして実行できます。

ただし、このやり方には落とし穴があります。もし本番コードの中で、最終的に渡されたコールバックを呼ぶ行を誤って消してしまったらどうなるでしょうか。アサーションがコールバックの中に書かれている場合、そのコールバック自体が呼ばれないため、テストは何も検証せずに通ってしまう可能性があります。

この問題を避けるには、コールバックが呼ばれたことを明示的に確認する必要があります。単純にはフラグを使って、テスト後にそのフラグが立っていることを検証できます。Mochaを使うなら、done関数を受け取る非同期テストとして書く方法もあります。テストが期待した制御フローの終端に到達した時点でdoneを呼び、到達しなければテストは失敗します。

この仕組みは、テストが本当に複数回のイベントループをまたぐ場合にも使えます。ただ、コールバック中心のコードをテストすると、どうしても配管のようなコードが増えがちです。テストが読みにくい、書きにくいと感じるなら、それは対象コードの設計に匂いがあることを示しているかもしれません。

そこでPromiseが役に立ちます。Promiseは、コールバックのオブジェクト指向版のように考えられます。従来のコールバックでは、非同期関数を呼ぶときに、処理が終わった後に何をするかも同時に渡します。Promiseでは、非同期処理を依頼することと、その後の処理を登録することが分離されます。関数はPromiseオブジェクトを返し、呼び出し側はそのPromiseに後続処理をつなげます。

この分離により、制御フローがより宣言的になります。たとえば、Ajaxから返るPromiseにJSONをUserへ変換する関数をthenでつなげれば、値がパイプラインを流れながら変換されていく形になります。コールバックを直接渡すより、処理の流れを読みやすく表現できます。

Promise版の実装では、ユーザー情報を取得する関数はコールバックを受け取らず、Promiseを返します。Ajaxの結果をPromiseとして扱い、その後にパース関数を連結します。呼び出し側は返されたPromiseにさらに処理をつなげられます。これにより、対象関数は「非同期処理を開始すること」と「結果をどう扱うか」をきれいに分けられます。

テストも読みやすくなります。Ajaxを、あらかじめ解決済みのPromiseを返すスタブに置き換えます。次に対象関数を呼び、返ってきたPromiseに対して、最終的に得られるUserのfullNameが期待どおりかを確認します。概念としてはコールバック版のテストと同じですが、制御フローがPromiseとして表に出ているため、テストランナーやアサーションライブラリが扱いやすくなります。

たとえば、Promise対応のアサーションライブラリを使えば、「このPromiseは最終的にこのプロパティを持つ値へ解決される」といった形で期待を表現できます。さらに、Promiseをテストから返せるテストランナー拡張を使えば、doneを明示的に呼ばなくても、テストランナーがPromiseチェーンの完了を待ってくれます。

ここにPromiseの強みがあります。**非同期の制御フローを値として扱える**ため、テストコードはその制御フローをテストランナーへ渡せます。テスト自身が低レベルな待ち合わせを知る必要はなく、Promiseを返して、残りはフレームワークに任せられます。

もちろん、すべてのコードが単に非同期対応で済むわけではありません。本質的に非同期なコードもあります。たとえばsetTimeoutを呼んで明示的にイベントループの次の周回へ処理を送るコードや、ネイティブのノンブロッキングAPIを直接呼ぶコードです。この場合は、テストも実際に非同期として扱う必要があります。

しかし、通常のアプリケーションコードがそうした低レベルの非同期処理を直接書くことは、それほど多くありません。多くの場合、jQueryのようなライブラリやブラウザAPIに統合しているだけです。その統合部分は、テストで依存先を差し替えれば同期的に検証できます。

結論として、非同期JavaScriptのテストで重要なのは、非同期という言葉に引きずられすぎないことです。対象コードが本当にイベントループをまたぐ必要があるのか、それとも非同期に使えるAPIを呼んでいるだけなのかを見極めます。後者であれば、テストでは依存先を制御し、同期的な流れに平らにして、通常の単体テストとして検証できます。

Promiseを使うと、その構造はさらに明確になります。処理を呼び出すこと、結果を変換すること、テストランナーが完了を待つことを、それぞれ分けて扱えるからです。非同期コードをテストしやすくする最もよい方法は、テスト技法だけを工夫することではなく、非同期の制御フローを読みやすく、扱いやすい設計にすることです。
