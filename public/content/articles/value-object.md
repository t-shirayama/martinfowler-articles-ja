# Value Object

## 要約

値オブジェクトは、オブジェクトの同一性ではなく、保持している値によって等しさを判断する小さなオブジェクトです。座標、金額、期間のような概念を明示的な型として扱うことで、比較、検証、振る舞いの置き場所が明確になります。

値オブジェクトを安全に使うには、等価性の実装と不変性が重要です。参照を共有したまま変更できると、思わぬ場所に影響するエイリアシングバグを招くため、変更時には新しいオブジェクトを作る設計が勧められます。

## 読むときの観点

- 「同じ」と判断する基準が、参照なのか値なのかを意識する。
- 値オブジェクトを作るときは、等価性の実装と周辺ライブラリとの相性を見る。
- 不変性がエイリアシングバグを避けるための中心的な手段であることを押さえる。
- どの概念を値オブジェクトにするかは、文脈によって変わる。

## 原文の翻訳

プログラミングをしていると、何かを複合的なものとして表すと便利だと感じることがよくあります。2次元座標は `x` の値と `y` の値から成ります。金額は数値と通貨から成ります。日付範囲は開始日と終了日から成り、それら自体も年、月、日という複合的なものになりえます。

こうしたことをしていると、2つの複合オブジェクトが同じものかどうか、という問いにぶつかります。どちらもデカルト座標 `(2,3)` を表す2つの点オブジェクトがあるなら、それらを等しいものとして扱うのは理にかなっています。このように、プロパティの値、この例では `x` 座標と `y` 座標によって等しいとされるオブジェクトを、**値オブジェクト**と呼びます。

しかし、プログラミングで注意しておかないと、プログラムの中でその振る舞いが得られないことがあります。

たとえば、JavaScript で点を表したいとします。

```javascript
const p1 = {x: 2, y: 3};
const p2 = {x: 2, y: 3};
assert(p1 !== p2);  // NOT what I want
```

残念ながら、このテストは通ってしまいます。JavaScript は、オブジェクトの等価性を調べるときに、中に入っている値を無視して参照を見るからです。

多くの状況では、値ではなく参照を使うのが理にかなっています。大量の販売注文を読み込んで操作しているなら、それぞれの注文を1つの場所に読み込むのは自然です。その後、Alice の最新の注文が次の配送に含まれているかを確認する必要があるなら、Alice の注文のメモリ参照、つまり同一性を取り、その参照が配送に含まれる注文リストにあるかを見ればよいのです。このテストでは、その注文の中身を気にする必要はありません。

同じように、一意な注文番号に頼って、Alice の注文番号が配送リストに載っているかを見ることもあるでしょう。

したがって私は、オブジェクトを区別する方法に応じて、値オブジェクトと参照オブジェクトという2種類のオブジェクトを考えるのが有用だと思っています（注1）。それぞれのオブジェクトが等価性をどう扱うべきかを自分が理解し、その期待どおりに振る舞うようプログラムする必要があります。その方法は、使っているプログラミング言語によって変わります。

複合データをすべて値として扱う言語もあります。Clojure で単純な複合データを作ると、次のようになります。

```clojure
> (= {:x 2, :y 3} {:x 2, :y 3})
true
```

これは関数型のスタイル、つまりすべてを不変の値として扱うやり方です。

しかし、関数型言語を使っていなくても、値オブジェクトを作れることはよくあります。たとえば Java では、標準の点クラスは私が望むように振る舞います。

```java
assertEquals(new Point(2, 3), new Point(2, 3)); // Java
```

これが機能するのは、点クラスがデフォルトの `equals` メソッドを、値を調べるテストで上書きしているからです（注2、注3）。

JavaScript でも似たことはできます。

```javascript
class Point {
  constructor(x, y) {
    this.x = x;
    this.y = y;
  }
  equals (other) {
    return this.x === other.x && this.y === other.y;
  }
}

const p1 = new Point(2,3);
const p2 = new Point(2,3);
assert(p1.equals(p2));
```

ここでの JavaScript の問題は、私が定義したこの `equals` メソッドが、他の JavaScript ライブラリにとっては未知のものだという点です。

```javascript
const somePoints = [new Point(2,3)];
const p = new Point(2,3);
assert.isFalse(somePoints.includes(p)); // not what I want

//so I have to do this
assert(somePoints.some(i => i.equals(p)));
```

Java ではこれは問題になりません。`Object.equals` がコアライブラリで定義されており、他のすべてのライブラリも比較にそれを使うからです（`==` は通常、プリミティブにだけ使われます）。

値オブジェクトのよい結果の1つは、メモリ上の同じオブジェクトへの参照を持っているのか、等しい値を持つ別の参照を持っているのかを気にしなくてよくなることです。しかし注意していないと、その幸せな無関心が問題につながることがあります。少し Java の例で説明しましょう。

```java
Date retirementDate = new Date(Date.parse("Tue 1 Nov 2016"));

// this means we need a retirement party
Date partyDate = retirementDate;
// but that date is a Tuesday, let's party on the weekend
partyDate.setDate(5);

assertEquals(new Date(Date.parse("Sat 5 Nov 2016")), retirementDate);
// oops, now I have to work three more days :-(
```

これはエイリアシングバグの例です。ある場所で日付を変更したら、予想していなかった場所にまで影響が及んでしまいます（注4）。エイリアシングバグを避けるために、私は単純ですが重要な規則に従っています。**値オブジェクトは不変であるべき**です。パーティーの日付を変えたいなら、代わりに新しいオブジェクトを作ります。

```java
Date retirementDate = new Date(Date.parse("Tue 1 Nov 2016"));
Date partyDate = retirementDate;

// treat date as immutable
partyDate = new Date(Date.parse("Sat 5 Nov 2016"));

// and I still retire on Tuesday
assertEquals(new Date(Date.parse("Tue 1 Nov 2016")), retirementDate);
```

もちろん、値オブジェクトを実際に不変にしておけば、それを不変として扱うのはずっと簡単になります。オブジェクトでは、たいていは単に設定用のメソッドを用意しないことでこれを実現できます。先ほどの JavaScript クラスなら、次のようになります（注5）。

```javascript
class Point {
  constructor(x, y) {
    this._data = {x: x, y: y};
  }
  get x() {return this._data.x;}
  get y() {return this._data.y;}
  equals (other) {
    return this.x === other.x && this.y === other.y;
  }
}
```

不変性は、エイリアシングバグを避けるために私が最も好む技法ですが、代入が常にコピーを作るように保証することでも、それらを避けることは可能です。C# の `struct` のように、この能力を備えた言語もあります。

ある概念を参照オブジェクトとして扱うべきか、値オブジェクトとして扱うべきかは、文脈によって決まります。多くの状況では、郵便住所をテキストの単純な構造として扱い、値による等価性を持たせる価値があります。しかし、より高度な地図システムでは、郵便住所を精巧な階層モデルに結びつけるかもしれず、その場合は参照のほうが理にかないます。ほとんどのモデリング問題と同じように、**文脈が違えば解も違います**（注6）。

文字列のような一般的なプリミティブを、適切な値オブジェクトに置き換えるのは、しばしばよい考えです。電話番号を文字列として表すことはできますが、電話番号オブジェクトにすれば、変数やパラメータがより明示的になり、言語が対応していれば型チェックも効き、検証の自然な置き場所になり、適用できない振る舞いを避けられます。たとえば、整数の ID 番号に対して算術演算をしてしまうようなことを避けられます。

点、金額、範囲のような小さなオブジェクトは、値オブジェクトのよい例です。しかし、概念上の同一性を持たない、あるいはプログラム全体で参照を共有する必要がないなら、より大きな構造でも値オブジェクトとしてプログラムできることはよくあります。これは、不変性をデフォルトとする関数型言語により自然に合います（注7）。

値オブジェクト、とりわけ小さな値オブジェクトは、あまりにも些細で考える価値がないものとして見過ごされがちだと私は感じています。しかし、よい値オブジェクトの集合を見つけると、その上に豊かな振る舞いを作れるようになります。この感覚を試すには、範囲クラスを使ってみるとよいでしょう。より豊かな振る舞いを使うことで、開始属性と終了属性をめぐるさまざまな重複した細工を防いでくれます。

私は、こうしたドメイン固有の値オブジェクトがリファクタリングの焦点になり、システムを劇的に単純化できるコードベースによく出会います。そうした単純化は、何度か目にするまでは人々を驚かせることが多いものです。そしてそのころには、値オブジェクトはよい友人になっています。
### 参考文献

Vaughn Vernon の説明は、おそらく DDD の観点から見た値オブジェクトについての最も優れた詳細な議論です。彼は、値とエンティティのどちらにするかをどう判断するか、実装上のヒント、値オブジェクトを永続化するための技法を扱っています。

この用語は、2000年代初頭に広まり始めました。その時期に値オブジェクトについて語っている本として、PoEAA と DDD の2冊があります。また、Ward の Wiki でも興味深いやり取りがありました。

用語上の混乱の原因の1つは、世紀の変わり目ごろ、一部の J2EE 文献が「value object」という語を Data Transfer Object の意味で使っていたことです。その使い方は現在ではほとんど消えていますが、出会うことはあるかもしれません。

### 注

1. Domain-Driven Design では、Evans の分類が値オブジェクトとエンティティを対比しています。私はエンティティを参照オブジェクトの一般的な形の1つだと考えていますが、「エンティティ」という用語はドメインモデルの中だけで使います。一方で、参照オブジェクトと値オブジェクトの二分法は、すべてのコードにとって有用です。

2. 厳密には、これは `awt.Point` のスーパークラスである `awt.geom.Point2D` で行われています。

3. Java でのほとんどのオブジェクト比較は `equals` で行われます。これは少し厄介でもあります。等価演算子 `==` ではなく、`equals` を使うことを覚えておかなければならないからです。これは面倒ですが、`String` も同じように振る舞うため、Java プログラマーはすぐに慣れます。他のオブジェクト指向言語ではこれを避けられることがあります。Ruby は `==` 演算子を使いますが、それを上書きできます。

4. Java 8 より前の日付と時刻の仕組みにおける最悪の特徴をめぐっては、強力な候補がいくつもあります。しかし、私ならこれに票を入れます。ありがたいことに、今では Java 8 の `java.time` パッケージによって、そのほとんどを避けられます。

5. クライアントが `_data` プロパティを操作できるため、これは厳密には不変ではありません。しかし、十分に規律あるチームなら、実践上は不変にできます。チームが十分に規律を保てるか心配なら、`freeze` を使うかもしれません。実際、単純な JavaScript オブジェクトに `freeze` を使うだけでもよいのですが、私は宣言されたアクセサを持つクラスの明示性を好みます。

6. これについては、Evans の DDD 本でさらに議論されています。

7. 不変性は参照オブジェクトにとっても価値があります。販売注文が `get` リクエストの間に変わらないなら、それを不変にすることには価値がありますし、有用であれば安全にコピーできるようにもなります。しかし、一意な注文番号に基づいて等価性を判断しているなら、それによって販売注文が値オブジェクトになるわけではありません。
