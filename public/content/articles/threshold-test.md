# Threshold Test

## 要約

Threshold Test は、デプロイメントパイプラインに組み込まれたテストで、現在のビルドから得られる測定値をしきい値と比べ、許容範囲を超えたらビルドを失敗させます。性能劣化の早期発見や、しきい値を段階的に厳しくする改善活動に向いています。

## 読むときの観点

- しきい値が、性能や品質の劣化をどの時点で止める役割を持つか。
- ratcheting によって、しきい値をどう段階的に改善へつなげるか。
- しきい値を緩める判断を、暗黙の妥協ではなく明示的な意思決定にできるか。

## 原文の翻訳

Threshold test とは、Deployment Pipeline に挿入されるテストであり、何らかの測定可能な現象を監視するために、現在のビルドの値をしきい値と比較する。現在のビルドの値がそのしきい値を越えると、テストは失敗し、ビルドも失敗する。

Threshold test のよくある例は性能である。チームは代表的な操作の集合を選び、それらにかかる時間を測定する。そして、これらの操作が現在の値よりも意味のある程度長い時間を要した場合に失敗するよう、threshold test を設定する。このようなしきい値は、あるコミットが性能劣化を持ち込んだケースを見つけるのに便利である。データ量の多いアプリケーションでは、こうした劣化は、まずいクエリやインデックスの使い方の悪さから起こることが多い。

Threshold test があれば、この種の問題が入り込んだ最初の時点で見つけられる。失敗を引き起こしたのがこのコミットだと分かるため、修正もしやすくなる。問題を探す範囲を絞れるからである。さらに threshold test を維持しておくと、この種の問題がコードベースに積み上がるのを防げる。悪いクエリがすでにたくさんあると、その影響に隠れて新しい問題も入り込みやすくなる。

Threshold test は ratcheting と組み合わせて使うこともできる。これは、値を改善するために、時間をかけてしきい値を少しずつ厳しくしていくやり方である。性能を改善するコミットを追加するたびに、しきい値を下げる。これにより、時間をかけて性能を徐々に改善しやすくなる。Threshold test と ratcheting の組み合わせは、出発点がよくない状態で、改善プログラムを始めるときに特に役立つ。

Threshold test が失敗し、チームがしきい値は厳しすぎると判断して、しきい値を緩める場合もある。状況によっては、それが正しい判断であることもある。ここで threshold test が助けになるのは、**その判断を意識的なものにする**点である。

Threshold test は、メトリクスの適切な使い方における「絶対値よりも傾向の追跡を重視する」「より短い追跡期間を使う」という考え方に従う方法の一つである。横断的な要件をテストする一般的な助言については、『Continuous Delivery』第9章が参考になる。
