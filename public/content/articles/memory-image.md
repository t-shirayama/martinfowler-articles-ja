# Memory Image

## 要約

Memory Image は、イベントソーシングを使って永続的なイベントログを真実の源にし、アプリケーション状態そのものはメモリ上に保持する設計です。
プロセスが落ちても、イベントとスナップショットを再生して状態を復元できます。

このアプローチは高速で、データベースマッピングの複雑さを避けられます。
一方で、イベントの確実な保存、スナップショット、復元時間、並行性、イベント形式の進化や移行を慎重に扱う必要があります。

## 読むときの観点

- 永続化の対象がアプリケーション状態ではなくイベントである点を押さえる。
- データベースを使わないことによる単純化と、代わりに引き受ける責務を比較する。
- LMAX、Event Poster、Smalltalk の例がどの性質を示しているかを見る。
- 長期運用では、イベントログとモデルの進化をどう分離するかに注目する。

## 原文の翻訳

エンタープライズアプリケーションを始めるとき、最初期に出てくる問いのひとつは「データベースとどうやって話すか」である。最近なら、少し違う問いとして「どの種類のデータベースを使うべきか。リレーショナルか、それとも NOSQL データベースのどれかか」と聞くかもしれない。しかし、考えるべき別の問いがある。「そもそもデータベースを使うべきなのか」だ。

エンタープライズアプリケーションを特徴づけるもののひとつは、長期的なデータを保存する必要があることだ。そのため、人々は自然にデータベースへ手を伸ばす。結局のところ、データの永続化はデータベースが行う主なことのひとつである。Memory Image を使うことは、データベースを伴わない永続化への別の道である。

Memory Image の重要な要素は、イベントソーシングを使うことだ。これは基本的に、アプリケーション状態へのすべての変更をイベントとして捕捉し、そのイベントを永続ストアに記録することを意味する。さらに、これらのイベントを再生することでアプリケーション状態全体を再構築できることも意味する。つまり、**イベントが主要な永続化メカニズムになる**。

イベントソーシングを使うシステムの身近な例は、バージョン管理システムである。すべての変更はコミットとして捕捉され、空のディレクトリにコミットを再生することで、コードベースの現在の状態を再構築できる。もちろん実際には、すべてのイベントを再生するのは遅すぎる。そのためシステムは、アプリケーション状態のスナップショットを定期的に永続化する。再構築は、最新のスナップショットを読み込み、そのスナップショット以降のイベントを再生することで行われる。

イベントソーシングには、過去の状態を再構築できることを含め、多くの帰結がある。しかし Memory Image にとって重要な性質は、アプリケーション状態を最新の永続ストアに保つことを心配する必要がなくなる点だ。代わりに、アプリケーション状態をメインメモリに保持しておけばよい。プロセスがクラッシュした場合は、イベント（およびスナップショット）から再構築できる。

Memory Image を使うと高い性能を得られる。すべてがメモリ内で行われ、データベースシステムへの IO やリモート呼び出しがないからだ。おそらくさらに重要なのは、データベースマッピングのコードをなくせること、あるいはメモリ上の状態とデータベース状態の同期を心配しなくてよいことである。

その一方で、イベントを信頼できる形で保存し、処理できるようにしなければならない。スナップショットを保存し読み込むコードも書く必要があり、サービス品質を維持できるだけ速くシステムを復元する方法も考えなければならない。データベースは永続化だけでなく、トランザクション上の並行性も提供する。そのため、並行性について何をするのかを考える必要がある。

もうひとつ、かなり明らかな制約は、保持したいデータ量よりも多くのメモリを持っていなければならないことだ。メモリサイズが着実に増えるにつれて、これは以前ほど大きな制約ではなくなってきている[^1]。

Memory Image は、さまざまな種類のシステムで利用できる。私が出会った3つの例を挙げてみよう。

最も新しい例は LMAX である。LMAX は高性能な取引システムで、単一の JVM スレッド上で 600万 TPS を処理する。ここでは Memory Image の性能上の利点が明らかに大きな要因だが、彼らはプログラミングモデルの単純化も同じくらい重要だと考えた。すべてが単一スレッドに関することなので、並行性について心配する必要がない。可用性を高く保つために、Memory Image のコピーを複数実行している。ひとつが停止した場合でも、非常に高いトランザクションレートを保ちながら別のインスタンスへ切り替えられる。

数年前、私は Event Poster アーキテクチャを使ういくつかのシステムについて書いた。このスタイルは、分析目的で多くの UI にメモリ上のモデルへの読み取りアクセスを提供する。複数の UI があるということは複数のスレッドがあるということだが、書き込み側はひとつだけ、つまりイベントプロセッサだけである。このため並行性の問題は大きく単純化される。

最も古い例は、この名前の由来でもある Smalltalk 開発環境である。多くの開発ツールはファイルシステム上のテキストファイルに依存し、必要に応じてコンパイルまたは解釈する。Smalltalk は、すべてのソースコードとコンパイル済みメソッドを image の中に保持していた[^2]。実行したすべてのコマンドは変更ログに保存された。たいていの場合は image（スナップショット）を保存していたが、必要であれば、愚かなことをしてしまったときに安定した基点から変更ログを再生できた。

この種の多くの考え方と同じように、これは何度も使われ、再発明されてきたアプローチである[^3]。しかし主流として広く採用されることはなかった。永続データをデータベースに保持させる方法が、今もより一般的なアプローチであり続けている。

Memory Image について私が聞いたことのある問題のひとつは、移行に関するものだ。ソフトウェアシステムを構築するときはいつでも、そのシステムが変更をどう扱うかを理解することが重要である。Memory Image では、本質的な作業は、イベントログから Memory Image を継続して再構築できるようにすることだ。

ここでの落とし穴のひとつは、イベントログに使うシリアライズ構造が、イベント構造を変更したくなったときに進化をうまく扱えないものになっていることだ。特定のイベントクラスを作ってシリアライズすると、後でイベントクラスの構造を変更したときに、古いイベントを処理するのが難しくなるかもしれない。多くの場合、マップやリストのような汎用的なデータ構造でシリアライズするのが最善である。

また、イベントとモデル構造そのものとの間に、よい分離を保つことも重要である。イベントデータとモデルを内省して何らかの自動マッピングシステムを作りたくなるかもしれない。しかし、それはイベントとモデルを結合してしまうため、モデルを移行しつつ古いイベントを処理し続けることを難しくする。

ある時点では、イベントログそのものを古い形式から新しい形式へ移行する価値があるかもしれない。イベントログの移行は多くの場合、より面倒である。それでも、元のイベント構造からかなり遠くまで進化してきた場合には選択肢になりうる。

長い間、Memory Image を使うことへの大きな反論はサイズだった。しかし今では、ほとんどの汎用サーバーが、かつて私たちがディスクで持っていた以上のメモリを備えている。その結果、ほとんどのワーキングセットは安全にメモリ上へ保持できるようになった。私たちは数年前にこのことに気づいていたが、Memory Image はまだ比較的まれである。今は NOSQL の動きによって、人々が永続化の選択肢を考え直すようになっているので、このパターンが増えるかもしれないと私は思う。

[^1]: イベントに含まれるデータの一部だけが必要なら、メモリ使用量を減らせる可能性もある。必要に合うなら、同じイベントを、異なるデータの部分集合を扱う別々の Memory Image システムへ送ることができる。

[^2]: ここで Smalltalk について過去形を使っているのは、私がそれを使ってから長い時間がたっており、その間に振る舞いが変わっているかもしれないからである。Smalltalk は今も存在しているが、残念ながらニッチな環境にとどまっている。

[^3]: Prevayler プロジェクトを覚えている人もいるかもしれない。これはこのアプローチのオープンソース実装である。数年前には Java コミュニティで大きな話題になったが、その後はかなり静かになっている。そのコミュニティでは、このアプローチの総称として system prevalence という言葉を使っている。
