# Designed Inheritance

## 要約

Designed Inheritance は、継承を許す箇所を意図的に設計し、文書化し、それ以外は上書きできないようにするという考え方です。

継承はカプセル化を破りやすい密接な関係なので、ライブラリ利用者を危険から守るという利点があります。一方で Fowler は、完全に禁止するよりも、通常は避けるべきだと示しつつ必要なら突破できる仕組みのほうを好んでいます。

## 読むときの観点

- 継承を「拡張の自由」と見るか、「壊れやすい契約」と見るかを意識する。
- ライブラリ作者と利用者の能力差ではなく、詳細知識の差として議論している点を見る。
- `prohibit` ではなく `discourage` を好む、という立場の違いを押さえる。
- 呼び出し用のインターフェースと継承用のインターフェースを分けて考える。

## 原文の翻訳

オブジェクト指向の世界で長く続いている議論のひとつに、Open Inheritance と Designed Inheritance の対立があります。Designed Inheritance の原則は、おそらく Josh Bloch の「継承のために設計して文書化せよ。さもなくば継承を禁止せよ」という言葉に最もよく要約されています。このアプローチでは、どのメソッドを継承可能にするかを慎重に決め、それ以外は Seal して上書きされないようにします。

Designed Inheritance を支持する最も一般的な議論は、継承が非常に密接で、カプセル化を破る関係でもあるため、サブクラスがスーパークラスを実質的に壊しやすいというものです。たとえば、メソッドが呼ばれたときに必要な振る舞いを省いてしまうことがあります。

多くの開発者、とりわけ Enabling Attitude を持つ人たちは、この種の議論にあまり納得しません。私にとってより魅力的だった別の議論は、Elliotte Rusty Harold が XOM の設計原則について語っていたものです。ここでの要点は「API は専門家が非専門家のために書くものだ」ということです。ライブラリ作者は、そのライブラリが扱う技術に精通しているべきです。そして、その技術をライブラリ利用者にとって単純にするよう努めるべきです。

カプセル化とは秘密を隠すことです。したがってよいライブラリは、利用者がそのライブラリを呼び出しで使う場合でも継承で使う場合でも、あらゆる複雑さや危険箇所を隠すべきです。XOM では、私はライブラリクラスを安全に上書きして望むことを実現できます。それでもライブラリは、私がそうでなければ起こり得るさまざまな問題を気にしなくても、整形式の XML を生成することを保証してくれます。

この議論は、ライブラリ作者は賢く利用者は愚かだ、という含みを持つ通常の議論より、私にはずっと説得力があります。これは能力の話ではなく、詳細な知識の話です。私は XML のみっともない細部について、できるだけ無知でいたいのです。その種の詳細を理解する必要から解放されれば、自分が本当に達成したい作業に頭を使えます。

この説得力のある議論にもかかわらず、私の直感、そして私が enabling attitude を持っているという事実は、Open Inheritance を好む方向に傾きます。おそらく問題の核心は、継承して安全な領域を示すために使っている仕組みにあります。通常、私たちが持っているのは、クラスやメソッドを seal する能力だけです。両陣営をなだめる代替案として、seal を乗り越える能力があってもよいでしょう。そうすれば、設計されていないものを上書きするには、あえてそうする必要があります。

明示的に seal を開かなければ、コンパイラは通常の継承だけを許します。しかし seal を開く仕組みを使うなら、コンパイラは信頼をあなたに渡し、その結果に対する責任もあなたが負うことになります。だから私は、Josh Bloch の `prohibit` を `discourage` に置き換えるほうを好みます。

より広く見ると、呼び出しと継承の両方について、インターフェースをどう考えるかには、まだ改善の余地がたくさんあると思います。Consumer Driven Contracts のようなアイデアは、インターフェース定義とは何を意味するのかを考え直す助けになります。答えが何になるかはわかりませんが、よい答えは私たち全員を驚かせるものになるだろうと思います。
