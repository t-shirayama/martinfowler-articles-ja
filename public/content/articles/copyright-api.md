# APIs should not be copyrightable

## 要約

APIを著作権で囲い込むべきではない、という主張です。
ソフトウェアのインターフェイスは、実装を差し替えたり、別システムと接続したり、テスト用の代替実装を作ったりするための接続点です。
そこに著作権による制約を強くかけると、相互運用性、競争、利用者の利益が損なわれます。

一方で、この記事はソフトウェアに知的財産権が一切不要だと言っているわけではありません。
著作権は作り手の仕事を守る仕組みとして有用ですが、APIのような接続のための表現にまで広げると、ソフトウェア開発の基本的な活動を妨げる、というのが中心的な論点です。

## 読むときの観点

- APIを「実装」ではなく「接続点」として読む。
- 互換実装、アダプター、テストダブルが日常的な開発にどう必要かを考える。
- 著作権による保護と、過剰な制約が競争を妨げる危険のバランスを見る。
- 2021年の最高裁判断が、APIの著作権性そのものではなく Fair Use を理由にした点に注意する。

## 原文の翻訳

先月、Electronic Frontier Foundation（EFF）は米国最高裁判所にアミカス・ブリーフを提出し、API（Application Programming Interface）に著作権を認める下級審判決を見直すよう、判事たちに求めました。

私はそのブリーフに署名した77人のソフトウェア専門家のひとりです。ただし、その顔ぶれには Abelson と Sussman、Aho と Ullman、Josh Bloch、Fred Brooks、Vint Cerf、Peter Deutsch、Mitch Kapor、Alan Kay、Brian Kernighan、Barbara Liskov、Guido van Rossum、Bruce Schneier、Ken Thompson らが含まれており、少々気後れしています。

もとの訴訟は Oracle が Google を相手取って起こしたもので、Oracle は Java API の著作権を保有しており、Google が Android を作る際にそれを侵害した、と主張しました。私がこのブリーフを支持する理由は、この2つの巨大テクノロジー企業の争いの細部とは関係ありません。関心があるのは、知的財産法をソフトウェア、特に**ソフトウェアのインターフェイス**にどう適用すべきかという問いです。

私は、ソフトウェアの中に知的財産として扱うべきものなど何もない、と主張する立場ではありません。ソフトウェア特許は本質的に壊れていると思っていますが、著作権は、ソフトウェアの作者が自分の大きな労力の成果に何が起きるかについて、ある程度の制御を持つためのよい仕組みです。

ソフトウェアは、常に収入源として扱いにくいものでした。コピーするのがあまりにも簡単だからです。著作権は、少なくとも一部のコピーを制御するための法的な土台を与えます。このようなものがなければ、誰かが何かを作る仕事をしながら住宅ローンを払っていくことは、とても難しくなります。私たちはみな無料のものが好きですが、人々が自分の仕事から生計を立てる機会を得るのは公平なことだと思います。

しかし、どんな知的財産の仕組みも、この利点と、過剰な知的財産上の制約がさらなるイノベーションを妨げる危険とのバランスを取らなければなりません。それは、発明を拡張する場合にも、創作物を別の形で想像し直す場合にも当てはまります。そのため、特許制度や著作権制度には何らかの制限が組み込まれています。そのひとつは時間による制限です。特許も著作権も期限切れになります。ただし、ミッキーマウス不連続性がそれを脅かしています。

インターフェイスとは、もの同士を接続する方法です。物理世界の例として、交換レンズ式のカメラがあります。多くのカメラメーカーは、他社が自社カメラ向けのレンズを作ることを歓迎していません。しかし、そうした第三者企業はインターフェイスの仕組みをリバースエンジニアリングし、そのカメラに装着できるレンズを作れます。第三者部品の提供者では、このようなことが日常的に起きています。そして、こうした第三者は、主要メーカーが支援していない低価格や機能を提供するうえで大きな役割を果たしています。

私は Canon のカメラで Sigma のレンズを使っていました。当時 Canon は18-200mmレンズを作っていなかったからです。カメラ用の第三者製バッテリーも買ったことがあります。安かったからです。同じように、自動車を第三者製部品で修理したこともあります。コストを下げるため、あるいは自分のニーズに合うオーディオシステムを得るためです。

ソフトウェアのインターフェイスも、これとほとんど同じです。オープンなインターフェイスを拡張したり、インターフェイスをリバースエンジニアリングしたりできることは、ソフトウェアシステムの発展に大きな役割を果たしてきました。インターネットの成長にとって、オープンなインターフェイスは不可欠でした。HTTPサーバーを作ったり、それに接続したりするために、誰も著作権ライセンス料を払う必要はありません。

Unixスタイルのオペレーティングシステムの成長も、AT&T Unix のソースコードの多くには著作権があった一方で、インターフェイスには著作権がなかったという事実に大きく依存していました。これにより、BSDやLinuxのような派生システムはUnixのインターフェイスに従うことができました。その結果、Unix上に作られたプログラムが新しい実装とやり取りしやすくなり、こうしたオープンソースシステムが普及する助けになりました。

SMBとSambaの物語は、著作権で囲い込まれないAPIが競争を促したよい例です。Windowsが支配的なデスクトップOSになると、そのSMBプロトコルは単純なネットワークを支配しました。Windows以外のコンピューターが支配的なWindowsプラットフォームと効果的に通信したいなら、SMBと話す必要がありました。Microsoftは競合がこれを行うための文書を提供しませんでした。SMBと通信できないことが、競合に対する障壁になっていたからです。

しかし Andrew Tridgell はSMBの仕様を推測し、Unix向けの実装である Samba を作ることができました。Sambaを使うことで、Windows以外のコンピューターもネットワーク上で協調できるようになり、MacやLinuxを基盤にしたシステムとの競争が促されました。何年も前には、IBM BIOSでも同様の話があり、競合他社によってリバースエンジニアリングされました。

自由市場システムの力は競争から生まれます。近所のパン屋より安い、あるいはおいしいパンを焼く方法を見つけたなら、私はパン屋を始めて競争できます。時間がたてば、小さなパン屋でも大きなパン屋と競争できるようになるかもしれません。この仕組みが働くには、市場を支配する既存プレイヤーが、コストを下げたり品質を上げたりするイノベーションを持つ新規参入者を妨げる障壁を作れないよう、市場を構成することが重要です。

ソフトウェアでは、インターフェイスがこの点で重要な接点になります。インターフェイスをオープンに保つことで、より多くの機能と低いコストをもたらすイノベーションを促す、競争的なソフトウェア市場を育てられます。これを閉ざすと、互いに通信できない**非互換なコンピューターシステムの孤島**が生まれます。

このような非互換の孤島は、新しい競合にとって大きな障壁であり、その理由だけでもよくありません。しかし、利用者にとっても悪いものです。利用者は、互いに連携できるソフトウェアに価値を置きます。さまざまなソフトウェアベンダーが通信に関心を持たないとしても、他の提供者が入り込み、その隙間を埋めることを後押しすべきです。システム同士を結びつけるには、インターフェイスを安全に実装して通信経路を作れるように、オープンなインターフェイスが必要です。

物理世界では標準的なコネクターを重視しています。ソフトウェアの接続は、すべてを標準化するには多様すぎることがよくありますが、著作権法を使ってさらなる障害を追加すべきではありません。

インターフェイスを実装する必要性は、さらに深いところにもあります。プログラマーとして、私たちは仕事をするために、自分たちのコードベースの外で定義されたインターフェイスを実装しなければならないことがよくあります。あるライブラリを前提に書かれたソフトウェアを、別のライブラリでも動くように変更しなければならないことは一般的です。そのための有用な方法は、最初のライブラリのインターフェイスを実装し、その処理を2つ目のライブラリへ転送するアダプターを書くことです。

インターフェイスの実装はテストでも不可欠です。それにより、Test Doubleを作れるからです。

したがって、プログラムを適切に書く能力のために、ソフトウェアが連携して動くことを望む利用者のために、そして競争を促す自由市場を望む社会のために、**APIに著作権を使うべきではありません**。

### 追記: 最高裁の判断

この事件は最高裁まで進み、最高裁は2021年4月5日に Google 側の勝訴と判断しました。この判断では、APIに著作権を認められるかどうかには踏み込まず、GoogleによるAPIの再実装は Fair Use であると判断しました。

上の記事では Fair Use を考慮していませんでしたが、私には、それがこの状況を捉える有効な枠組みに見えます。APIをこのように扱うことで、APIに著作権を認めることから生じる問題を避けられます。

判決文冒頭の4ページの要約は、その論理をかなり明快に説明しています。
