# Deployment Pipeline

## 要約

Deployment Pipeline は、ビルドとテストを複数の段階に分け、早い段階で多くの問題を見つけつつ、後段でより高い信頼を得るための仕組みです。
速いフィードバックと包括的な検証の間にある緊張を扱う方法でもあります。

各段階は、時間をかけるほど信頼度を高めます。
本番デプロイは通常、パイプラインの最後の段階に位置づけられます。

## 読むときの観点

- Deployment Pipeline は Continuous Delivery の中心的な構成要素である。
- 早い段階は高速なフィードバック、後段は深い検証を担う。
- 段階には自動実行だけでなく、人の承認や手動確認が含まれる場合もある。
- パイプラインは技術的検査だけでなく、可視性、協働、監査証跡も提供する。
- 現在のデリバリープロセスをパイプラインとして描くと改善点を見つけやすい。

## 原文の翻訳

自動化されたビルドとテスト環境の課題のひとつは、速いフィードバックを得るためにビルドを速くしたい一方で、包括的なテストには時間がかかることだ。Deployment Pipeline は、この問題に対処するためにビルドを複数の段階へ分ける。

各段階は、通常、より多くの時間を支払う代わりに、より高い信頼を与える。初期段階は多くの問題を素早く見つけ、後段の段階は遅くなるが、より徹底した確認を行う。Deployment Pipeline は Continuous Delivery の中心的な要素である。

一般に、最初の段階ではコンパイルなどを行い、後続段階で使うバイナリを作る。後段には、自動化できないテストのような手動確認が含まれる場合もある。段階は自動で進むことも、人の承認を必要とすることもある。また、ビルドを速めるために複数のマシンで並列化されることもある。

本番環境へのデプロイは、通常、パイプラインの最後の段階に置かれる。ただし、パイプラインの役割は単にデプロイを自動化することだけではない。より広く言えば、Deployment Pipeline の仕事は、**本番で問題につながる変更を検出すること**である。

検出すべき問題には、性能、セキュリティ、ユーザビリティなどが含まれる。よいパイプラインは、ソフトウェアを届けるさまざまな関係者の協働を助け、変更がシステムの中をどう流れているかを全員に見えるようにし、十分な監査証跡も残す。

Continuous Delivery を導入するよい方法は、まず現在のデリバリープロセスを Deployment Pipeline としてモデル化することだ。そのうえで、ボトルネック、自動化できる箇所、協働が必要な接点を調べる。既存の流れを可視化することで、改善は抽象論ではなく具体的な作業になる。
