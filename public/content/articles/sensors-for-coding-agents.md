# Maintainability sensors for coding agents

## 要約

コーディングエージェントに保守しやすいコードを書かせるには、事前のガイドだけでなく、問題を観測して自己修正させるセンサーが重要になります。この記事は、AIで作り直した社内分析ダッシュボードを題材に、lint、依存関係ルール、結合度分析、AIによるモジュール性レビュー、テストスイートとmutation testingを、保守性のセンサーとして試した経験をまとめています。

ファイルや関数レベルでは、ESLintのような静的解析が想像以上に有効でした。一方、モジュール性や結合度のような横断的な設計品質は、数値だけでは判断しにくく、LLMによる意味解釈が必要でした。テストについては、AI生成テストのカバレッジだけを信じると危うく、mutation testing が回帰検出力の不足を見つける助けになります。

## 読むときの観点

- センサーを、エージェントが人間のレビュー前に自己修正するためのフィードバックとして読む。
- computational なセンサーと inferential なセンサーの得意分野の違いを見る。
- 静的解析のしきい値や警告抑制は、機械的に従わせるだけでなく、判断を残しレビュー可能にするものとして捉える。
- 結合度やモジュール性は、数値だけでなく文脈とトレードオフの解釈が必要だと意識する。
- AI生成テストの多さやカバレッジではなく、既存の振る舞いを壊したときに検出できるかに注目する。

## 原文の翻訳

コーディングエージェント利用者のための harness engineering についての最近の記事で、私はコーディングエージェントのハーネスを拡張するためのメンタルモデルを示しました。それは、良いエージェント出力の確率を高め、人間の目に届く前に問題を自己修正できるようにする、ガイドとセンサーのシステムです。この記事はその実践的な続編であり、コードベースを保守しやすく保つためのセンサーを使った経験を説明します。

コードベースで達成し監視したい次元はいくつもあります。機能的な正しさ、アーキテクチャ適合性、そして保守性です。ここでの保守性とは、コードベースを時間とともに簡単かつ低リスクに変更できるようにすること、つまり internal quality のことです。今日すばやく変更できるだけでなく、将来もそうできるようにしたいのです。

変更するたびに、あるいはAIに変更させるたびに、バグや適合性の劣化を持ち込むのではないかと心配したくありません。AI生成コードベースの保守性にひびが入り始めた最初の兆候として、私はよく、小さな調整なのに変更ファイル数が増えることを見ます。あるいは、以前は動いていたものが変更で壊れ始めることです。

いつものように、私の視点は主にデジタルプロダクトやエンタープライズソフトウェアのようなアプリケーションを書くことにあります。AIコーディングエージェントを使うリスクプロファイルや目的は、作るソフトウェアによって異なります。

内部品質の問題は、人間の開発者に影響するのと似た形でAIエージェントにも影響します。絡み合ったコードベースで作業するエージェントは、既存実装を探す場所を間違えたり、重複に気づかず不整合を作ったり、タスクに本来必要な以上の文脈を読み込まざるを得なくなったりします。

この記事では、コードベースの保守性について私たちとAIが振り返る助けになるさまざまなセンサーを試し、そこから学んだことを説明します。

### アプリケーション

私は、コミュニティマネージャー向けの社内分析ダッシュボードに取り組んでいます。これはチャットスペースの活動、エンゲージメント、人口統計データを複数のAPIから読み取り、Webフロントエンドで表示するものです。

例のアプリケーションは、Web UI、サービス層、外部APIから構成されています。技術スタックは TypeScript、NextJS、React です。バックエンドはAPIからデータを読み取り、結合します。このアプリケーション自体はしばらく前からありますが、今回の実験のために、AIを使って最初から作り直しました。

コード品質や保守性についてAIに伝えるガイド、たとえばMarkdownファイルはほとんど置きませんでした。センサーからのフィードバックだけに頼ったとき、どれくらいうまくできるかを見たかったからです。

### 使ったセンサーの全体像

本番までの経路に沿って設定したセンサーの全体像を示します。

コーディングセッション中には、エージェントのそばで継続的に動き、すばやいフィードバックを与えるセンサーを使いました。

- 型チェッカー（computational）
- ESLint（computational）
- 社内AppSecチームが指定するSASTツールである Semgrep（computational）
- 内部モジュール依存の構造ルールを確認する dependency-cruiser（computational）
- テストカバレッジを含むテストスイートの結果（computational。ただしテストスイート自体はAI生成なので、作成は inferential）
- incremental mutation testing（computational）
- pre-commit hook の一部として動く GitLeaks。エージェントがコミットしようとしたときにフィードバックを与えるので、これもセンサーとみなします（computational）

統合後のパイプラインでは、同じ computational なセンサーをCIでも再実行しました。セッション中のセンサーは開発中にエージェントへ早期フィードバックを与え、CIパイプラインはクリーンなインフラ上で、統合後の結果を確認します。

繰り返し実行するセンサーとしては、その場で起きるエラーではなく、時間とともに蓄積する drift を検出するため、遅い周期で動くものを用意しました。

- 社内アプリケーション向けAppSecチェックリストから派生したプロンプトによるセキュリティレビュー（inferential）
- 「ユーザー名をWebフロントエンドへ送ってはいけない」といった内容を記述したデータ取り扱いレビュー（inferential）
- 依存ライブラリの年齢や活動状況を先にスクリプトで取得し、それをもとにアップグレードや非推奨化などの推奨をAIがレポートする依存関係の鮮度レポート（computational と inferential）
- モジュール性と結合度のレビュー（computational と inferential）

この文脈を置いたうえで、最初のカテゴリのセンサーに入ります。

### 基本ハーネスとモデル

アプリケーションを構築する間、私は Cursor、Claude Code、OpenCode を混ぜて使いました。頻度はこの順です。デフォルトモデルはたいてい Claude Sonnet で、計画や分析の一部では Claude Opus を使い、実装タスクでは Cursor の composer-2 モデルをよく使いました。

### 静的コード解析: 基本的なlint

まず、このアプリケーションでESLintを使って学んだことから始めます。ESLintのような基本的なlintツールは、主に個々のファイルや関数のレベルで保守性リスクを対象にします。

私の経験では、AIの失敗モードのうち静的コード解析で最も取りやすいものは、関数引数の最大数、ファイル長、関数長、循環的複雑度です。しかし、これらはESLintのデフォルトプリセットでは有効になっておらず、先に上限値を設定する必要がありました。AI利用に向けた、よりよいプリセットを静的解析ツールが提供するようになることを期待しています。少し調べると、テストファイルの要求や構造化ログなど、既知のエージェント失敗モードを対象にしたESLintプラグインのルールセットを公開し始めている人たちもいます。

### 自己修正のためのガイダンス

センサーは、エージェントが自己修正できるようにフィードバックを与えるものです。理想的には、その自己修正のための追加文脈も与えたいところです。良い種類の prompt injection です。そのために、私はデフォルトメッセージの一部を上書きするカスタムESLintフォーマッターを作りました。もちろん、自然なことにAIの助けを借りました。

`no-explicit-any` 警告に対するガイダンスの例は次のようなものです。

```text
We want things to be typed to make it easier to avoid errors, especially for key concepts.
But we also want to avoid cluttering our codebase with unnecessary types. Make a judgment
call about this. If you choose to not introduce a type, suppress it with:
// eslint-disable-next-line @typescript-eslint/no-explicit-any -- (give reason why)
```

### 警告管理は扱いやすくなったのか

静的コード解析は長く存在してきましたが、チームはそれを設定していても、一貫して使わないことがよくありました。その理由の1つは、管理コストです。この分析を効果的に使うには、チームが「きれいな家」を保つ必要があります。そうでなければ、メトリクスはノイズになるだけです。特に上の `no-explicit-any` のような警告は扱いが難しいものです。常に修正したいわけではなく、場合によるからです。

それらを1つずつ抑制するのは、いつも面倒で、コード上のノイズのように感じられました。

コーディングエージェントによって、私たちは今、そのきれいなベースラインを得られるかもしれません。上のガイダンステキストでは、エージェントは判断するよう求められ、コード中で警告を抑制することも許されています。これにより、抑制は管理可能で、見える形になり、レビュー可能になります。

最大行数や許容される循環的複雑度のようなしきい値については、特定の場合にリファクタリングが不要または不可能だと考えるなら、しきい値を少し上げてもよいとlintメッセージでエージェントに伝えました。これはしきい値を永遠に抑制するのではなく、上げるだけなので、将来さらに悪化すれば再びルールが発火します。二択の「抑制するか従うか」を強制せずに、制約を保てます。

### 観察

AIが作った例外、つまり抑制された警告や引き上げられたしきい値を見ることは、コードレビューを始めるよい出発点になりました。

AIは循環的複雑度のしきい値を上げる判断を頻繁にしましたが、さらに促すとよいリファクタリングを提案しました。しきい値を上げたのはこのカテゴリだけでした。後で、この項目には自己修正ガイダンスを置いておらず、しきい値の引き上げは絶対的な例外であるべきだという明示的な指示がなかったことに気づきました。これは、カスタムlintメッセージが実際に大きな違いを生みうることを示しています。

ルールをコードの場所によって違う扱いにしたいこともあります。たとえば `no-console`、つまりAIが `console.log` を使うのを注意するルールです。バックエンドでは、代わりにloggerコンポーネントを使ってほしい。フロントエンドでは、直接ログ出力を使わないでほしいかもしれませんし、少なくとも別のログコンポーネントを使う必要があるかもしれません。これも自己修正ガイダンスの力を示す例であり、AIが意味的な判断と解析警告の管理を助けられる場所です。

ルール同士のトレードオフの例にも注意していました。これまでに見た唯一の例は、`max-lines` と `max-lines-per-function` のルールによって生まれたものです。このセンサーフィードバックの結果として、AIが有用なリファクタリングを行い、小さな関数やコンポーネントへ分解するのをかなり見ました。しかしReactフロントエンドでは、より小さなコンポーネントの連鎖を通して値を渡す結果、非常に多くのプロパティを持つコンポーネントが増えるという心配な傾向も見えています。

このようなトレードオフの間でAIがどれくらい一貫した判断をできるかについては、まだ有用な観察は得られていません。

### 主な学び

全体として、静的解析でかなり多くのことをカバーできることに、私はよい意味で驚きました。これまで静的解析がやや使われてこなかった理由と、何が変わったのかを、何度も自分に思い出させる必要がありました。それは費用対効果のバランスです。AIによってカスタムスクリプトやルールを作ることがはるかに安くなったので、コストは下がりました。

利益も増えています。解析結果は、私が自分でコードを書くときにはそれほど起こらない多くの衛生要因について、最初の感触を得る助けになります。よくあるAIのミスを先に片付けられるのです。

ただし、これが誤った安心感や品質の錯覚につながらないかとも考えています。こうしたlinterが過去にそれほど使われてこなかったもう1つの理由は、限界があるからです。私たちは、品質の単純化された指標として使うことに慎重でした。静的解析では捉えられない、より意味的な品質の側面はたくさんあります。AIがそうしたギャップをツールと組んで十分に埋められるかは、まだ分かりません。

新しいルールセットを有効にするたびに、コード内の新しい「問題」が見つかることにも気づきました。それはいつも、無関係なものと本当に重要なものの混合でした。そのため、エージェントへのフィードバック過多によって、過剰設計のリファクタリングの渦に送り込んでしまわないか心配しています。

### 静的コード解析: 依存関係ルール

基本的なlintは、主にファイルや関数内の品質と複雑さに焦点を当てます。次に私は、ファイルやモジュール境界をまたぐ保守性の懸念について、私とエージェントにフィードバックを与えられるセンサーを調べ始めました。この領域の解析ツールは、歴史的には基本的なlintよりさらに使われてきませんでした。

コードベース内でよいモジュール性を保つために、私たちとAIを助けるセンサーの可能性を学ぶため、私は3つのものを試しました。

- 依存関係ルール（deterministic）
- 結合度分析（deterministic と inferential）
- モジュール性レビュー（inferential）

まず依存関係ルールから始めます。実装の途中あたりで、私はエージェントと一緒に、このアプリケーションのレイヤー化されたモジュール構造を考えました。そして、そのレイヤーを強制する dependency-cruiser のルールを書く手伝いを頼みました。

たとえば、あるルールは `clients` フォルダー内のコードが `services` フォルダーから何もimportしないことを強制します。

```js
{
  name: "clients-no-services",
  comment:
    "API clients must not depend on the orchestration layer above them. " + LAYERS,
  severity: "error",
  from: { path: "^server/clients/", pathNot: "/__tests__/" },
  to: { path: "^server/services/" },
}
```

ESLintメッセージと同じように、エラーメッセージも少し拡張し、レイヤーの考え方全体を振り返る自己修正ガイダンスにしました。

```text
ERROR  clients-no-services
  API clients must not depend on the orchestration layer above them.
  [Layers: routes -> services -> clients + domain; Services orchestrate: fetch data via clients, compute via domain -- no I/O, no SDKs, no knowledge of data fetching.]
```

AIなしでは、私はこれらのルールをすばやく整えられなかったでしょう。このツールの設定構文には大きな初期コストがあり、AIがそのコストをほぼ完全に吸収してくれました。

導入後、エージェントは数回ルールに違反しましたが、`dependency-cruiser` のフィードバックに基づいて自己修正しました。そのため、フォルダーの概念を保つ助けになりました。

同じアプローチで、フロントエンドにおけるReact hooksの構造に関する規約も導入しました。

AIがこの構造の外に新しいフォルダーを作り始めたときにどう検出するかも考える必要がありました。そこで、新しいファイルはすべて事前定義されたフォルダー構造のどこかに置かなければならない、というルールを作りました。

依存関係ルールを導入した時点で、コードのフォルダー構造はすでに少し場当たり的になっていました。ルールがエージェントにその整理を促し、その後もレイヤーを強制し続けるのを見られました。そのため、コード構造をMarkdownガイドに書く代替として、かなり有用だと感じました。ただし、この種のツールは import、ファイル名、フォルダー構造で表現できることに限られます。

### 静的コード解析: 結合度データ

次に、コードベースから典型的な結合度メトリクス、つまりファイルごとのincoming/outgoing importとcallの数を抽出する実験をしました。

既存ツールは使わず、typescript compilerを使ってそれらのメトリクスを作るアプリケーションをコーディングエージェントに書かせました。実験の一部として最大限柔軟に試せるようにしたかったからです。私はそれに2つのインターフェイスを追加させました。人間である私が見るための、さまざまな可視化を備えたWebインターフェイス。そして、コーディングエージェントにメトリクスを提供できるCLIです。

人間向けの可視化の多くは、dependency structure matrix など、よく知られた概念です。私はそれらの解釈が面倒だと感じました。vibe coded されたもので改善の余地は確かにありましたが、それ以上にデータの性質によるところが大きいと思います。データがかなり細かく、それを解釈し、より高いレベルのよい実践へ対応づけるには、多くの文脈と経験が必要です。

そのため、AIに変更されたコードベースをレビューするとき、人間の認知負荷を下げるうえで、この種のツールはまだそれほど役に立たないのではないかと感じています。

AI向けには、このカスタムCLI（`coupling-analyser`）へのアクセスをエージェントに与え、データに基づくレポートを作らせました。重大な問題をどう改善するかの提案も含めました。

プロンプトでは、`npx coupling-analyser` の実際のCLI出力に根ざして、TypeScriptコードベースのモジュール性と結合度品質に関するMarkdownレポートを作るよう求めました。推測だけでなく、CLIを実行して標準出力を取得し、ホットスポット、循環、相互依存、fan-in/fan-out、変更リスクなどを、具体的なモジュールIDや数値とともに要約するよう指示しました。

このLLM主導の分析は、私が可視化図を見れば見つけただろう結合のホットスポットを、より消化しやすい形式で示しました。決定的なツールの結果に分析を根ざさせることで、信頼感も高まりました。エージェントがコードベースを自力でスキャンして結合問題を見つけるより、おそらく時間とトークンも少なく済んだでしょう。

しかし、このデータに基づいてLLMが見つけたものは、かなり物足りないものでした。使ったモデルは Claude Opus 4.7 です。

1つの大きな問題として、必要なコンポーネントをすべて初期化するfactoryを挙げました。しかし私は、そのfactoryを軽量なdependency injection frameworkのように振る舞うコンポーネントとして意図的に導入していました。

もう1つは、フロントエンドとバックエンドで共有される `zod` schema を、LLMが「god module」と呼んだことです。しかしこれは、バックエンドとフロントエンドの間に明示的な契約を作る一般的なパターンです。しかも私の場合のように、バックエンドとフロントエンドが一緒に進化し、同じリポジトリにあるなら、それほど問題ではありません。

正当なパターンが高結合のハブとして現れる場合、将来の分析ではそれらを抑制する方法が必要です。そうしなければ、さらにノイズを生みます。

少し興味深かった発見は、domainフォルダー内の `index.ts` が `./domain` 内のすべてのファイルを無差別に公開し、多くの場所からimportされていることでした。これもレイヤーの明示的な契約を作る一般的なパターンではありますが、利点と欠点があり、このコードベースに適しているかを調べる価値はあります。

これらの例は、基本的なlint以上に、良いか悪いかに明確な定義があるわけではなく、何が適切かが重要であることを示しています。そして、どの結合が適切かは、コードベースの生のcall/importグラフだけでなく、多くの文脈に依存します。この小さな実験に基づく限り、この種の結合度データが単独でAIに有用だという印象はありません。

このデータのより実用的な使い方として考えられるのは、コードレビューのリスクトリアージです。AIによるコード変更をレビューするとき、変更されたファイルの影響半径を知るのは有用そうです。たとえば10以上の呼び出し元を持つファイルが変更されたなら、より注意を払えます。あるいは、AIレビューエージェントが、どこにトークンを使うべきかを優先づけるためにデータを使えるかもしれません。

### 静的コード解析: AIによるモジュール性レビュー

結合度データ実験の結果が物足りなかった理由はいくつか考えられます。何を分析すべきかについてのプロンプトがあまり具体的でなかったのかもしれません。結合度データがAIにとって有用でないのかもしれません。あるいは、結合度データだけでは浅すぎて、コード全体の文脈が足りなかったのかもしれません。

そこで最後に、私は完全に inferential な方向へ進み、Vlad Khononov の「Modularity Skills」を使ってコードベース設計を分析し、モジュール性の問題を見つけました。これは非常に実りの多いものでした。将来の変更リスクを明らかに下げるリファクタリングについて、多くの興味深い手がかりを与えてくれました。

私はこのスキルを2回目にも実行し、結合度分析CLIへのアクセスを与えました。AIは主にデータから確認を得ましたが、追加の発見はほとんどありませんでした。むしろ、CLIが見落としている多くのことを指摘しました。また、最初の分析の文脈なしに実行した2回目の分析では、1回目が見つけなかった別の問題も浮かび上がりました。重要なときには、LLMベースの分析を複数回実行して、より全体像を得る価値があるという有用なリマインダーです。

結果のハイライトをいくつか挙げます。モデルは結合度分析と同じ Claude Opus 4.7 です。

重複したrouteコードがありました。3つのバックエンドエンドポイントがそれぞれ独自のrouteファイルを持ち、その実装はほぼ同じでした。そのため、バックエンドAPIの一般原則、たとえばrequest IDの導入、エラーハンドリングやロギング方針の変更などを行う場合、複数ファイルを変更しなければなりません。3つ目のエンドポイントを導入したばかりだったので、まだ抽象化されていなかったのは妥当だと思います。しかし私の経験では、AIエージェントはコードを3回目、4回目に繰り返しても、明示的に促されなければ自分からリファクタリングを始めることはあまりありません。コピー&ペーストで満足してしまいます。

バックエンド呼び出しの不整合もありました。言い換えると、意味的な重複の別の形です。アプリケーションには、同じパラメータ群、つまり選択されたチャットスペースと分析対象の日付範囲でバックエンドを呼ぶ必要があるページが3つあります。そのうち2つは同じhookと一般的なアプローチを使っていました。しかしAIが3つ目のページを導入したとき、そこから逸脱し、似た振る舞いを独自の方法で再実装しました。これはたとえばエラーハンドリングの不整合につながり、バックエンドAPIの原則が変わったときに再び複数ファイルを変更する必要を生みます。

中心的な引数の扱いも非効率でした。すべてのページは、チャットスペースIDと日付範囲をバックエンドへ渡します。ユーザーが日付範囲を指定する方法を変えたとき、AIが40を超える多くのファイルを変更しなければならなかったことに私はすでに気づいていました。何かがおかしいことは分かっており、分析はそれを確認しました。「リクエストパラメータがすべてのレベルで繰り返されている」という問題です。推奨は、これらのパラメータを包むオブジェクトを導入することでした。AIはある意味でそれをすでに行っていましたが、そのオブジェクトの利用を最後まで徹底しておらず、一貫しない混乱になっていました。

責務が誤った場所に置かれている問題もありました。レビューは、本来モジュールを配線するだけの責任を持つはずのfactory内に、認証コードが少しあることを見つけました。ユーザーが認証されていないときにmock dataへフォールバックする実装です。このような予期しない場所にあると、新しいrouteが追加されたときに見落とされるリスクがあります。

許容できる高import数の「ハブ」については、よりよく解釈されました。前の結合度分析が見つけた「god classes」を覚えているでしょうか。モジュール性スキルもそれらに気づきましたが、どちらの場合も、このアプリケーションの文脈では目的があることをうまく指摘しました。これは、スキルのプロンプトがよかったからか、あるいはこの分析が実際にコードの中身を読んだのに対し、もう一方には結合度データだけに頼るよう求めたからだと思います。

主な学びは次のとおりです。

- `dependency-cruiser` のような依存関係パーサーは、基本的なフォルダー構造や依存方向を強制するライブセンサーとして有効ですが、到達できる範囲には限界があります。
- AIによるモジュール性レビューは「garbage collection」のよい例であり、強力なプロンプトを与えるとうまく働きました。実際の結合度データに根ざさせても、大きな違いは見えませんでした。これをコミット内の変更ファイルに適用し、より早くパイプライン内で使う方法を見つけられるとよいでしょう。ただし、私はまだそこまでは試していません。
- 私は、コードベースの大半を構築した後に、自分自身ではこの種のレビューを適用せずにモジュール性レビューを実行しました。その結果、将来のリスクを高めていたであろう、かなり懸念すべき妥当な発見がありました。人間のレビューと結合度の専門性、さらにこれらの追加AIレビューがなければ、エージェントは確実に inadvertent technical debt を積み増していたことを示しています。

全体として、コードベース設計とモジュール性は、computational なセンサーだけではあまり助けにならない関心事のようです。意味的な解釈を加え、トレードオフを考慮するにはAIが必要です。

### 回帰センサーとしてのテストスイート

テストには多くの目的があります。設計について考え、それを導く助けになり、アプリケーションの望ましい振る舞いを文書化し、回帰を検出する助けにもなります。つまり、変更によって既存機能を壊したときに教えてくれます。効果的な回帰テストはコードベースの保守性に大きな役割を果たし、変更をはるかに安全にします。

したがって保守性センサーの文脈では、このセクションは回帰センサーとしてのテストスイートの役割について述べます。

既存のテストが失敗したとき、私たちは問いを立てなければなりません。「偶然何かを壊したので実装を変える必要があるのか。それとも意図的に振る舞いを変えているので、この新しい仕様に合わせてテストを変える必要があるのか」。失敗するテストは、AIにまさにその問いを尋ねる機会を与えます。もちろん、AIが常に正しい判断をするとは限りません。しかしよいテストスイートは、AIが望ましい既存の振る舞いを壊す確率を下げます。

私のチャット分析アプリケーションでは、手動テストとテストカバレッジの確認以外にはあまり監督せず、時間をかけてすべてのテストをエージェントに書かせました。後から回帰検出の効果を分析するため、完全にAI生成のテストスイートを持ちたかったのです。

AIがレビューなしでテストを生成するアプローチには、主に2つのリスクがあります。

- カバレッジはテスト効果の十分な指標ではありません。
- テストが誤った振る舞いをテストしているかもしれません。これはテスト効果を確認するよりはるかに難しい問題で、別の機会の話です。この記事はテスト効果だけに焦点を当てます。つまり、コードが望ましい振る舞いを実装していると仮定したとき、壊れるコードを捕まえるテストがあるかどうかです。

使える道具には何があるでしょうか。

- Coverage: コードのどの部分がテストで実行されたかを追跡し、テストから見えている部分と見えていない部分を示します。
- Property-based testing: 手作りの例ではなく、定義した性質から多くの入力組み合わせを生成し、不足している論理的テストケースを見つけられます。
- Fuzz testing: 予期しない入力や不正な入力をシステムへ投げ込み、入力耐性に関する不足テストケースを見つけられます。
- Mutation testing: 小さなコード変異を導入し、テストスイートがそれを捕まえるか確認することで、不足しているassertionを見つけられます。

私のアプリケーションでは、ユースケースに property-based testing と fuzz testing があまり合わなかったため、coverage と mutation testing を使いました。

### Mutation Testing

mutation testing がassertionのギャップを見つける助けになることを示すため、私のコードベースから小さな例を挙げます。mutation testing結果の分析中、エージェントは私のために図を作りました。

`mappers.ts` ファイルは statement coverage 100%、branch coverage 75% と報告されていました。しかし実際にはunit testがなく、私が使ったmutation testingツールである Stryker は13個の survivor を報告しました。つまり、Strykerが13個のコード変異を入れた後も、テストスイートはgreenのままだったということです。この場合にカバレッジが高かったのは、コードベースに大きなacceptance testがあり、最終的にこれらの関数を呼んでいたからです。**カバレッジはある行が実行されたことを教えてくれますが、その影響が検証されたことは教えてくれません。**

この小さなmapperヘルパー関数 `dvpToSchema` が将来変更されると、UIのデータグラフ表示を壊す可能性があります。

AIはmutationのホットスポットを分析し、テスト品質をどこで高めるべきかの優先計画を作るうえで非常に役立ちました。

Strykerは巨大なJSONファイルに結果を書きます。分析を助け、誤ってコンテキストウィンドウを詰まらせないようにするため、私はエージェントがStrykerの結果を効率よく問い合わせられるカスタムスクリプトを生成しました。これは、AIを助けるためにAIが私を助けてくれた、多くの例の1つです。

```text
Query a Stryker mutation-testing JSON report from the command line.
Usage:
python query_stryker.py <report.json> <command> [options]

Commands:
  summary Overall status totals, mutation scores, thresholds.
  files Per-file breakdown, default sorted by mutation score asc.
  hotspots Lines with the most survivors / no-coverage mutants.
  tests Test effectiveness: weak, unused, or top-killer tests.
```

現在は、end-to-end型のacceptance testが増える傾向があるように見えます。冒頭で述べたように、AIはテスト生成が非常に得意になったため、開発者がAIに多くのテストを生成させ、あまりレビューしないことがかなり普通になっています。特にunit testのレビューはとても面倒です。

私は、それらをまったく見ないのが良いことだと言っているわけではありません。しかし、すべてのテストを人間がレビューし続けるのが持続可能だと考えるのは現実的ではなく、人々が実際にそうするとも思えない、という現実は認めます。AIコーディングの未来にふさわしい testing pyramid、ice cream cone、muffin の形を探す中で、approved scenarios のような技法が人気を集めています。

上で示したように、acceptance test はカバレッジを高めますが、多くの場合assertionはそれほど厚くありません。その結果、テスト効果について誤った安心感を与えます。mutation testing は、そのギャップを監視する助けになります。

もちろん、mutation testing には実践上の制限があります。かなりリソースを消費します。私の設定では、他のセンサーのように継続的には実行せず、incremental run を手動で起動しました。

### 結論と未解決の問い

computational なセンサーが最も印象的だったのは、ファイルと関数のレベルでした。モジュール性や結合度のようなファイル横断の関心事は別の話でした。生データ自体は非常にノイズが多く、LLMによる意味解釈、つまり inferential なセンサーなしにはあまり有用ではありませんでした。しかし、よいプロンプトを使うと、その出力と助言には非常に感心しました。また、経験レベルの異なる人に向けて、この情報をさまざまな形で提示できる可能性にも感心しました。

実験ではまだ見ていないものの、今後より問題になりうると疑っているのは、センサー同士の衝突です。`max-lines` と `max-lines-per-function` のルールは、いくらか緊張関係の兆候を示しました。より小さな関数へのリファクタリングが、複雑さをコンポーネントのプロパティ連鎖へ押し込んだのです。おそらく、似たようなトレードオフはもっと潜んでいます。それが時間とともに問題になるのか、なるならどう問題になるのかを見るのは興味深いでしょう。

このアプリケーションでは、センサーの効果をより純粋に見るため、ガイドはまったく気にしませんでした。ガイドとセンサーのバランスがどう進化するのかに興味があります。センサーの集合に自信を持てるようになったら、どのガイドを削除できるのでしょうか。センサーは、より弱いモデルの利用を現実的にするのでしょうか。ガイドとセンサーの一貫性をどう保つのでしょうか。そして、それらをまとめて保守しやすくする方法を見つけられるのでしょうか。

回帰テストの領域では、テストの大半をAIに任せる判断をするとき、mutation testing がどれほど重要になるかについて、私は本当に目を開かれました。そしてもう一度強調したいのは、テストの正しさについては、まったく別の議論が必要だということです。

これらのセンサーの一部は、成果物の品質に対する私の信頼を実際に高めます。しかし、人間を完全にループの外へ出す魔法の解決策ではありません。それでも私は、computational と inferential の両方のセンサーを相棒にすることで、レビュー体験と信頼レベルが改善されたことを確かに経験しました。
