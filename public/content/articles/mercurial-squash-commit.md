# Mercurial Squash Commit

## 要約

Mercurial で複数の作業コミットをひとつの変更セットにまとめる手順を説明する記事です。ファイル移動を含む作業では履歴追跡が壊れやすいため、Mercurial Queues と git 形式の diff を使って、安全に squash する流れを示しています。

## 読むときの観点

- Mercurial では複数リポジトリを使い分ける作業スタイルが自然な前提になっている
- 履歴を書き換える前に、元の作業履歴を別リポジトリとして残しておく
- ファイル移動を含む変更では、`hg log -f` で移動前の履歴を追えるか確認する
- バージョン管理は監査や協調作業だけでなく、コードベースの変化を語る履歴としても使われる

## 原文の翻訳

最近 Mercurial でいくつかのコミットを squash する作業を少し試したので、同じことを探している人の役に立つかもしれないと思い、手順を書いておく。このやり方が最善かどうかはわからないが、自分にはかなりうまく機能した。

```sh
hg clone base working
# base の tip はリビジョン 73
cd working
# 作業しながらコミットする
cd ..
hg clone working squash
cd squash
hg qimport -r 74:tip
hg qgoto 74.diff
hg qfold $(hg qunapp)
hg qfinish -a
cd ../base
hg pull ../squash
```

私がやっていた基本的な作業は、ファイルやフォルダをかなり大きく移動することだった。作業中にチェックポイントを置きたかったので、いくつかの段階に分けてコミットしたかった。しかし、バージョン履歴にはひとつのコミットとして残したかった。git なら rebase でもっと簡単にできるらしい。**単一のコミットにまとめると、何が起きたのかを理解しやすくなる**。ファイル移動はリポジトリログを読むのをややこしくしがちなので、なおさらだ。

ファイル移動は手順そのものも複雑にする。何度か、移動の追跡能力を失ってしまうためにうまくいかない手順に行き当たった。移動したファイルについて `hg log -f` を実行し、移動前の元のコミットがいつ何だったのかを見られる状態にしておきたい。

最初に、mq 拡張、つまり Mercurial Queues を有効にし、diff を git 形式に設定する必要があった。git 形式の diff は、ファイル移動を正しく追跡する助けになる。

```ini
# ~/.hgrc
[extensions]
mq=

[diff]
git=true
```

このように Mercurial を使うときは、複数のリポジトリを使うのが一般的な作業方法のようだ。Mercurial は、git や svn ならブランチを使うような場面で、別々のリポジトリを使うことを促す。この点については議論もあるが、それが Mercurial らしい作業スタイルだ。この例では、元のリポジトリを `base` と呼ぶ。

最初のステップは、`base` を作業用リポジトリに clone することだった。

```sh
hg clone base working
```

この時点で、`base` と `working` の tip はリビジョン 73 だった。そこでファイル移動を行い、途中でいくつかのチェックポイント用リビジョンを作った。

```sh
cd working
hg mv foo1 newdir/foo1
.. more hg mv ..
hg ci -m "moving around"
.. more hg mv ..
hg ci -m "moving around"
.. more hg mv and hg ci..
cd ..
```

作業が終わった時点で、最後のリビジョンは 80 になっていた。

これらをひとつのコミットに squash するため、さらに別のリポジトリを clone した。

```sh
hg clone working squash
```

ここで clone しておくことが重要だ。これから履歴を編集するため、うまくいったと確信できるまで元の履歴を手元に残しておきたかったからだ。次に、そのリポジトリへ移動する。

```sh
cd squash
```

ここで、自分が作った各リビジョンのコミットを Mercurial の patch queue 機構のパッチに変換した。

```sh
hg qimport -r 74:tip
```

最初の変更を現在のパッチにする。

```sh
hg qgoto 74.diff
```

すべてのパッチをひとつのパッチに畳み込む。

```sh
hg qfold $(hg qunapp)
```

この畳み込まれたパッチのコミットメッセージは、個々のコミットメッセージをつなげたものになる。きれいなコミットとして残すには、単一のメッセージにしたかった。

```sh
hg qrefresh -m "reorganized files"
```

その後、パッチを通常のコミットに変換した。

```sh
hg qfinish -a
```

これで、その作業全体を含む単一のコミットができた。内容に問題がないか確認し、特に移動したファイルで `hg log -f` を試して、履歴がまだ残っていることを確かめた。すべて問題ないと判断できたら、その単一の変更セットを `base` リポジトリに pull した。

```sh
cd ../base
hg pull ../squash
```

バージョン管理システムへの関心が年月とともにどう変わってきたかを見るのは興味深い。初期の主な目的、というより唯一の目的は監査だった。古いリビジョンへ安全に戻れるようにし、主に問題の診断に使うためだ。やがて関心は、人々の協調作業をどう可能にするかへ移った。これは監査の必要を置き換えたのではなく、その上に積み重なった。

今ではさらに、**コードベースがどのように変化してきたかを語る物語として履歴を使う**ことに注意が向けられている。だからこそ、このような履歴を書き換えるコマンドが欲しくなる。この必要も、前の二つの必要の上に積み重なるものだが、新しい能力と新しい緊張関係を持ち込む。
