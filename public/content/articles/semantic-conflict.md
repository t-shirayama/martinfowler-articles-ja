# Semantic Conflict

## 要約

Semantic Conflictは、テキスト上は安全にマージできるのに、プログラムの意味や振る舞いが壊れる衝突です。
現代のバージョン管理ツールはrenameを含むテキストのマージをうまく扱えますが、変更の意図までは理解できません。

Feature Branchingで問題になるのは、単なるテキスト衝突だけではありません。
Self-Testing Codeと頻繁なmergeは、Semantic Conflictを早く見つけ、影響を小さくするための重要な戦略です。

## 読むときの観点

- 「マージできた」ことと「正しく動く」ことを分けて考える。
- renameや副作用の分離の例から、意味のずれがどう起きるかを見る。
- ツールの進歩はテキスト衝突を減らすが、Semantic Conflictを消すわけではない。
- CIや短いbranchが、意味のずれを早く見つける理由を押さえる。

## 原文の翻訳

著者と同僚がFeature Branchについて話すのを聞いた人なら、彼らがそのパターンをあまり好んでいないことを知っているでしょう。反対する大きな理由のひとつは、branchを作るのは簡単だが、mergeは難しいという観察です。ときどき、現代のVersion Control Toolはmergeを十分に簡単にしているので、Feature Branchingには価値があるという主張を耳にします。

確かに現代のツールは、昔よりずっと上手にmergeできます。その力のよい例がrename越しのmergeです。たとえば著者が`lorem.rb`の中身を変更し、Jezがそのファイル名を`ipsum.rb`へ変更した場合でも、ツールは適切にmergeできます。

これはよいことですが、解決しているのはtextual conflictだけです。Semantic Conflictには役立ちません。Semantic Conflictとは、Jezと著者がそれぞれ変更を行い、テキスト上は安全にmergeできるのに、プログラムの振る舞いが変わってしまう状況を指します。

最も単純な例は、関数名の変更です。たとえば著者が、`clcBl`というメソッドは`calculateBill`という名前のほうが扱いやすいと考えたとします。現代のrefactoring toolを使えばこれは簡単です。Shift+F6を押し、新しい名前を入力すれば、ツールがすべての呼び出し元を変更します。

しかし、Jezが自分のfeature branchでこのメソッドへの呼び出しを追加していたら問題が起きます。2つの変更をmergeすると、テキスト上のmergeはうまくいきます。しかしプログラムは同じようには動きません。

メソッド名の変更は単純な例であり、静的型付き言語ではコンパイルに失敗するため見つけやすいものです。しかし、もっと微妙なSemantic Conflictはたくさんあります。`calculateBill`メソッドを見ていた著者が、そのメソッドは請求額を計算するだけでなく、会計システムへ会計エントリも送っていると気づいたとします。著者はその副作用が気に入らず、`notifyAccounting`という別メソッドへ取り出します。そして`calculateBill`のすべての呼び出し元を見つけ、`notifyAccounting`の呼び出しを追加します。しかしJezは自分のbranchでその変更を知りません。

ここでの第一のポイントは、ツールがどれほど強力でも、守ってくれるのはtextual conflictからだけだということです。特に厄介なのは、Semantic Conflictのほうが見つけにくく、修正もしにくいことです。

Semantic Conflictを自動的に解決することはできません。いつかツールがその一部を扱えるようになるかもしれませんが、厄介なものは残り続けるでしょう。少なくとも、コンピュータが私たちの意図を読んで自動的に推論できるようになるまではそうです。それでも、Semantic Conflictへの対処を大きく助ける戦略はいくつかあります。

ひとつめはSelf-Testing Codeです。テストは、コードの意味についてテストが期待している見方と、実際のコードの振る舞いが一致しているかを調べる探針のように働きます。Jezが呼び出しているコードについて、ある振る舞いを期待し、そのためのテストを持っていれば、統合時にそのテストが壊れます。もちろん完璧な対応ではありません。テストは決して完全にはなりません。それでも実際には多くのSemantic Conflictを捕まえます。また、衝突を見つけた後の修正までは助けませんが、見つけること自体が戦いの大きな部分です。

もうひとつの助けは、より頻繁にmergeすることです。Jezが著者の変更を数日後ではなく数時間後に知るなら、困難はずっと小さくなります。古い意味に基づいて多くのコードを積み上げる前に気づけるからです。これが、著者たちがContinuous Integrationを強く支持する理由です。

Feature Branchingをツールが十分に扱いやすくしたと主張する人々には、いくつかのグループがあります。ひとつは「enterprise grade」のVCSを売る人々です。著者たちはそこにはあまり関心を持ちません。もうひとつはDVCS、つまりDistributed Version Control Systemの支持者です。こちらには少し注意が必要です。人々はしばしば、DVCSがFeature Branchingを簡単にすることを根拠にDVCSを正当化します。しかしそれはSemantic Conflictの問題を見落としています。

DVCSを使うよい理由はたくさんあります。だからこそ、よいツールを問題のある技法と結びつける必要はありません。Semantic Conflictの教訓は、**テキストとしてmergeできることを、意味として統合できたことと取り違えない**ことです。
