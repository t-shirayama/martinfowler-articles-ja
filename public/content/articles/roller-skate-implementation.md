# Roller Skate Implementation

## 要約

アジャイル開発では、最初からすべての機能をそろえるのではなく、小さな機能セットで早く本番投入することが重要です。早く市場に出せれば、そのぶん早くビジネス価値を得られます。

この記事では、新商品のソフトウェア対応を段階的に進めた例が紹介されています。最初は電話番号を載せた静的ページ、次にWebフォームからFAXを送る仕組み、最後にバックエンド連携へ進むという形です。洗練されていなくても、価値を早く届けるための実装の分割が主題です。

## 読むときの観点

- 最初のリリースに必要な機能を、完成形から逆算しすぎずに考える。
- 一時的な人手や運用で補う判断が、早期の市場投入にどう効くかを見る。
- 「きれいな実装」と「早く価値を出す実装」のトレードオフに注目する。

## 原文の翻訳

アジャイル開発の重要な性質の一つは、小さな機能の一部だけでシステムを本番稼働させる方法を見つけることです。私たちは、ソフトウェアがもたらすビジネス価値のためにソフトウェアを作ります。早く本番稼働させるほど、そのビジネス価値の少なくとも一部を早く得られます。

同僚のDave Leigh-Fellowsが、この種の考え方について、私のお気に入りの例を話してくれました。彼がある証券会社で働いていたときの話です。その会社には、市場に出したい新しい種類の商品がありました。その商品を完全にソフトウェアで支えるなら、顧客がWebページに入力し、その内容からバックエンドシステムに必要な取引を発生させる、という形になります。しかしDaveは、それより早く商品を市場に出す方法を考えました。

- バージョン1は、その商品を説明し、電話番号を掲載する静的なWebページでした。一時スタッフが顧客と話し、情報をバックエンドシステムに入力しました。
- バージョン2は、顧客が入力したデータを取得するWebフォームでした。ただし、このバージョンでは、そのデータをバックエンドシステムへ読み込みませんでした。代わりに、WebフォームがFAXを生成しました。彼らはさらに一時スタッフを雇い、FAX機から注文を受け取り、バックエンドシステムへ情報をキー入力する人たちへ渡しました。FAX機が少し離れた場所にあったため、ここでローラースケートが登場しました。
- バージョン3では、Webフォームをバックエンドシステムへ直接つなぎました。

最初の二つのバージョンは、これまで考え出された中でもっとも優雅な解決策ではなかったかもしれません。しかし、**商品をずっと早く市場に出す**ことはできました。ローラースケートを使う反復開発の例を、私はほかに見たことがありません。ただ、それは必要がないからというより、想像力が足りないからかもしれません。
