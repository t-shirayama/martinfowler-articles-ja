# Craftmanship And The Crevasse

## 要約

ソフトウェアクラフトマンシップの議論は、アジャイルの中で軽視されがちな技術的実践を再び前面に出すという意味で価値があります。しかし、技術だけに焦点を合わせすぎると、開発者と顧客の間にある深い溝をさらに広げてしまう危険があります。

この記事の中心は、プログラマの理想像を「上手にコードを書く人」だけに閉じないことです。ドメイン専門家と話し、顧客が成果を出すためにソフトウェアがどう役立つかを考える姿勢も、同じくらい重要だと述べています。

## 読むときの観点

- クラフトマンシップという比喩そのものではなく、運動が何に反応しているのかに注目する。
- Scrum や Lean が技術的実践を軽視しがちな一方で、クラフトマンシップも顧客との関係性を軽視しうる、という対称性を読む。
- Extreme Programming が技術的スキルと顧客との関係を両立しようとしていた点に注目する。
- プログラマの中心に置くべきものはソフトウェア自体ではなく、ソフトウェアが届ける利益だという結論を意識する。

## 原文の翻訳

Daniel Terhorst-North の最近のソフトウェアクラフトマンシップに関するブログ記事は、多くのブログ上の議論を巻き起こしました。興味があれば下で要約しています。その中には多くの論点がありますが、彼の主題のひとつが特に私に響いたので、この記事を書いています。

ただしその話に入る前に、ひとつ脇へどけておきたい要素があります。私は以前から、ソフトウェア開発の比喩をめぐる議論は退屈だと感じてきました。MetaphoricQuestioning に意味がある場面はありますが、ソフトウェア開発が craft なのか、art なのか、trade なのか、あるいはデザートのトッピングなのかには、根本的に興味がありません。

私にとって重要なのは、クラフトマンシップという比喩ではなく、この数年で立ち上がってきたように見える運動の性質です。外部から見た私の印象では、ソフトウェアクラフトマンシップのコミュニティを活気づけている主な力は、アジャイル運動の変化への反応です。アジャイルウイルスの初期には、支配的な系統は Extreme Programming であり、そこでは技術的実践について多くが語られていました。

現在のアジャイルで支配的な系統は Scrum と Lean です。これらはプログラミングそのものにはあまり関心を払いません。そのため、自分を主にプログラマだと認識している人たちは、自分の人生の大きな部分がアジャイルの世界ではもはや重要ではない、と感じています。

Daniel Terhost-North、別名 tastapod。

したがって、ソフトウェアクラフトマンシップの世界は、プログラミングを再び前面かつ中心に置ける場所です。そこでは、テスト、関数型言語の学び方と使い方、良い設計の原則などについて話せます。マネジメントや分析の問題は、弱ってしまったアジャイルコミュニティに任せておけばよい、というわけです。この点には、私も共感するところが多くあります。私は DesignStaminaHypothesis を受け入れています。つまり、ソフトウェアを効果的に開発したいなら、良い技術的実践に注意を払う必要がある、という考えです。

ですから、こうした問題に注意を向ける運動は重要です。しかしそこには危険もあります。技術的な問題に焦点を当てすぎることで、クラフトマンシップ運動が、顧客とのコミュニケーションという同じくらい重要な役割を過小評価してしまう危険です。

Kent の仕事で私がとても好きな点のひとつは、顧客との関係と、取引のこちら側をきちんと果たすために必要なスキルとの間に、本当のバランスがあることです。AgileManifestoMeeting で彼が、Extreme Programming の主な目的はソフトウェア開発者と顧客の間の分断を癒すことだ、と言っていたのを覚えています。

Dan と私が「破滅の口を開けたクレバス」と表現したこの分断は、ソフトウェア開発における最も重要な問題のひとつです。

このクレバスの責任の大きな部分は、プログラマと顧客はあまりに違う生き物なのでコミュニケーションできない、そして試みるべきでもない、という考えに基づいた組織上の習慣にあります。しかし多くのプログラマもまた、喜んでそのクレバスを広げる側に回っているように見えます。数年前、Eric Evans の観察に私は強く印象づけられました。開発者はシニアになるほど技術的な問題に注目しがちで、自分が取り組んでいるドメインを理解することにはあまり関わらなくなる、というものです。

Domain-Driven Design は、まさにそれを変えようとするものです。とはいえ、リポジトリで dependency injection をどう使うか、といった話題の議論にしばしば沈み込んでしまうようにも見えます。私自身のキャリアも、この点では影響を受けています。一般的なソフトウェア開発の論者として活動するようになるにつれ、ドメインモデリングに取り組めなくなりました。ドメインモデリングは、ずっと私がソフトウェア開発で一番好きな部分だったにもかかわらずです。

したがって、Scrum と Lean が技術的スキルを軽視することでこの問題を悪化させるのと同じように、クラフトマンシップもまた、関係性のスキルを軽視することでクレバスを悪化させるのではないかと私は恐れています。私にとって理想のプログラマとは、プログラミングの craft に熟練しているだけでなく、ドメイン専門家とコミュニケーションしながらドメインについて学ぶことに意欲を持ち、顧客が自分たちの仕事で力を発揮できるようにソフトウェアが役立つ最善の方法を見つけることに参加できる人です。

Dan の言葉を言い換えるなら、**プログラマの世界の中心にソフトウェアを置くべきではありません**。プログラマは、そのソフトウェアが届けるはずの利益に焦点を合わせるべきです。
