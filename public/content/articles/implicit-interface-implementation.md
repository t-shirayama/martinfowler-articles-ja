# Implicit Interface Implementation

## 要約

Implicit Interface Implementation は、あるクラスの public な暗黙インターフェースを、実装継承なしに別クラスが実装できたらどうなるか、という考え方です。
Java や C# ではクラスの public メソッド群は暗黙のインターフェースとして存在しますが、それを `implements Customer` のように実装することはできません。

この記事は、置き換え可能性、テストダブル、Interface Implementation Pair への不満、動的言語との違いを通じて、明示的な interface がない場合の不便さを論じます。
特にテストで、実装継承ではなく再実装を差し込みたい場面が中心です。

## 読むときの観点

- クラスが持つ「暗黙の public interface」と、言語の明示的な interface 型を区別する。
- 実装継承ではなく interface 継承だけが欲しい場面を見る。
- テストダブルを差し込むとき、明示的な interface がないことがどんな制約になるかを考える。
- 動的型付け言語では同じ問題がどう消えるのかを読む。

## 原文の翻訳

Java と C# は、どちらも純粋な interface 型について同じモデルを共有しています。`interface Mailable` のように純粋な interface を宣言し、Java であれば `class Customer implements Mailable` のように、それを実装すると宣言できます。クラスは任意の数の純粋な interface を実装できます。このモデルが見落としていることのひとつは、クラスがあるところには常に暗黙の interface がある、という点です。

public な暗黙 interface `Customer` とは、`Customer` で宣言されたすべての public メンバーのことです。この暗黙 interface は、私が見たことのあるすべてのオブジェクト指向言語に存在します。Java も C# もできないことのひとつは、この暗黙 interface を実装することです。つまり、`class ValuedCustomer implements Customer` と書くことはできません。

暗黙 interface を実装するとは、どういう意味でしょうか。基本的には、`ValuedCustomer` クラスが `Customer` の public interface に宣言されているすべてのメソッドを実装しているが、`Customer` の実装、つまり public メソッド本体、非 public メソッド、データは受け継がない、と型システムに伝えることです。言い換えれば、**実装継承ではなく interface 継承**を行うということです。

これは、`Customer` を、customer のすべての public メソッドを含む interface に変更し、その interface を実装する `CustomerImpl` クラスを用意することと同等です。

なぜこれが役に立つのでしょうか。昔の Java、現在のコレクションフレームワークができる前の時代に経験した例があります。私たちは `Vector` クラスを自分たちの実装で置き換えたいと思っていました。しかし `Vector` はクラスであり、私たちにできるのはサブクラス化だけだったため、それはできませんでした。ライブラリが自由な置き換えを可能にする interface を提供していないために、このような事態に出くわすことがあります。この機能がなければ、私たちは行き詰まります。

これは最近では特にテストで問題になります。何かを stub に差し替えたい場面はたくさんありますが、interface がない限り、それが難しかったり不可能だったりします。また、置き換えをテストで支えることだけが理由で、純粋な interface 型を定義することにもつながります。Interface Implementation Pair はよく使われるアプローチですが、私たちの多くは好んでいません。Implicit Interface Implementation は、もっとすっきりしたアプローチになるでしょう。

では、なぜ言語はこれを許さないのでしょうか。正直なところ、私にはわかりません。私は言語設計者ではないからです。かつて Anders Hejlsberg にこの件を尋ねる機会がありました。彼の返答は、メンバーを明示的に `virtual` と宣言した場合にだけオーバーライドを許したいという彼の好みに近いものでした。基本的には、サブクラス、あるいはこの場合は実装者が、スーパークラスを壊してしまうことへの懸念です。これは、サブクラス化をどう使うかという、はるかに広い話題に触れています。ただし、これは夕食の席での短い会話にすぎなかったので、私たちが本当に議論を掘り下げられたとは思っていません。

これを書いた後で、昔の同僚 Mike Rettig が、この考え方の問題のひとつを指摘してくれました。クラスは実際には複数の暗黙 interface を定義している、という点です。たとえば Java では、`Customer` クラスは実際には public、protected、package、private の4つの暗黙 interface を定義しています。あるオブジェクトが `Customer` と協調するとき、これらの interface のどれを使うかは場合によります。`Customer` の別インスタンスは private 機能を使えます。暗黙 interface を実装したいなら、すべてを実装するのか、どこまで実装するのかを定義しなければなりません。型システムがそれを追跡するのがどれほど難しいのかは、私にはわかりません。

もちろん、私が挙げた例はたいてい、public な暗黙 interface だけで十分にうまくいきます。したがって実践上は、それで十分かもしれません。

Ian Griffiths は、この問題はクラスと interface を混ぜることに関係しているのではないかと指摘しました。Microsoft の COM 技術では、両者の間に明確な分離がありました。「COM でオブジェクトを使いたいなら、interface 経由で使わなければならない。したがって、常に自分自身の実装を作ることができる」というわけです。これは VB 6 ではかなり透過的に扱われ、COM interface は背後で生成できました。

この問題は動的型付け言語では起こりません。別のクラスの interface を実装したいなら、同じメソッドを実装し、それが必要な場所でそのオブジェクトを使えばよいだけです。すべてのメソッドを実装する必要はありません。注目している特定の相互作用で使われるものだけを実装すればよいのです。これはテストではとてもうまく機能する仕組みで、Smalltalker たちはしばらくの間これを Imposter パターンと呼んでいました。Java でこの種のことをするために dynamic proxy を使うのもかなり一般的ですが、私は Implicit Interface Implementation のほうが意図を伝えやすいと感じています。

これは何か重要なのでしょうか。主にはテストの問題のように見えます。再実装を使えないと Test Double を差し込むのがずっと難しくなりますし、スーパークラスがデータベースへの本物の接続を求める場合、サブクラス化ではうまくいかないこともよくあります。これはテストだけの問題かもしれません。Robert Conley は、VB6 の再実装能力をテストでよく使うが、本番コードで必要になったことはない、と話してくれました。
